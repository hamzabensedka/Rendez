  "content": "# Planity Clone — System Architecture\n\n## 1. Architecture Overview\n\nThis document describes the complete system architecture for the Planity Clone mobile application. The system follows a **clean architecture** approach with clear separation of concerns across frontend, backend, and infrastructure layers.\n\n### 1.1 Architectural Principles\n\n- **Separation of Concerns**: Each module has a single responsibility.\n- **Domain-Driven Design**: Business logic is isolated in domain modules.\n- **API-First**: Backend exposes a RESTful API consumed by the mobile app.\n- **Offline-First**: Mobile app works offline with local data sync.\n- **Scalability**: Horizontal scaling for backend services via Docker Compose.\n- **Testability**: Unit and integration tests for all layers.\n\n### 1.2 Technology Stack\n\n| Layer | Technology |\n|-------|------------|\n| Mobile App | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated |\n| Backend API | NestJS, TypeScript, Prisma ORM |\n| Database | PostgreSQL with PostGIS extension |\n| Cache & Session | Redis |\n| Build & CI/CD | Nx, pnpm, GitHub Actions, EAS Build, Docker Compose |\n| Testing | Jest (unit, integration) |\n| Auth | Supabase Auth |\n\n## 2. Service Boundaries\n\n### 3.1 Mobile App (Expo/React Native)\n\n**Responsibilities**:\n- UI rendering and user interaction\n- Local state management via TanStack Query\n- Offline data caching\n- Navigation via Expo Router\n- Biometric authentication\n- Push notification handling\n\n**Key Modules**:\n- `app/` — Expo Router pages for each screen\n- `components/` — Reusable UI components (buttons, cards, modals, bottom sheets)\n- `hooks/` — Custom hooks for data fetching, location, auth\n- `services/` — API client, storage service, auth service\n- `state/` — TanStack Query client configuration and query hooks\n\n### 3.2 Backend API (NestJS)\n\n**Responsibilities**:\n- Business logic execution\n- Data validation and persistence\n- Authentication and authorization\n- Slot computation and availability management\n- Notification dispatching\n\n**Module Boundaries**:\n\n| Module | Domain | Key Entities |\n|--------|--------|--------------|\n| Auth | User authentication | User, Session, OTP |\n| Users | User profile management | User, Address |\n| Businesses | Business profiles | Business, BusinessHour, Staff |\n| Services | Service catalog | Service, ServiceCategory |\n| Appointments | Booking management | Appointment, AppointmentService |\n| Reviews | Customer feedback | Review, ReviewImage |\n| Categories | Service categorization | Category, SubCategory |\n| Availability | Slot computation | Slot, AvailabilityRule |\n| Notifications | Push, email, SMS | Notification, NotificationTemplate |\n\n### 3.3 Shared Package\n\n**Responsibilities**:\n- Type definitions shared between frontend and backend\n- Enum definitions (BookingStatus, PaymentStatus, NotificationType)\n- Constants (API endpoints, error codes)\n\n## 4. Data Flow\n\n### 4.1 Authentication Flow\n\n```\nMobile App                    Backend API                    Supabase Auth\n    |                              |                              |\n    |-- Login request ------------>|                              |\n    |                              |-- Auth request ------------->|\n    |                              |<-- JWT + refresh token -----|\n    |<-- JWT + refresh token ------|                              |\n    |                              |                              |\n    |-- Store tokens securely      |                              |\n    |-- (SecureStore)              |                              |\n```\n\n### 4.2 Booking Flow\n\n```\nMobile App                    Backend API                    Database\n    |                              |                              |\n    |-- Get availability -------->|                              |\n    |                              |-- Query slots ------------->|\n    |                              |<-- Available slots ---------|\n    |<-- Slots -------------------|                              |\n    |                              |                              |\n    |-- Create appointment ------>|                              |\n    |                              |-- Validate & insert ------->|\n    |                              |<-- Appointment created -----|\n    |<-- Confirmation ------------|                              |\n```\n\n### 4.3 Offline Sync\n\n- TanStack Query caches API responses locally\n- When offline, app uses cached data\n- On reconnect, mutations are retried automatically\n- Conflict resolution uses last-write-wins strategy\n\n## 5. API Design\n\n### 6.1 Base URL\n\n- Development: `http://localhost:3000/api/v1`\n- Production: `https://api.planityclone.com/api/v1`\n\n### 6.2 Endpoints\n\n| Method | Endpoint | Description |\n|--------|----------|-------------|\n| POST | /auth/register | Register new user |\n| POST | /auth/login | Login |\n| POST | /auth/refresh | Refresh token |\n| POST | /auth/logout | Logout |\n| GET | /businesses | List businesses (with filters) |\n| GET | /businesses/:id | Business detail |\n| GET | /businesses/:id/services | Services for business |\n| GET | /businesses/:id/availability | Available slots |\n| POST | /appointments | Create booking |\n| GET | /appointments | User's appointments |\n| PATCH | /appointments/:id | Update/cancel booking |\n| GET | /categories | List categories |\n| GET | /reviews?businessId= | Business reviews |\n| POST | /reviews | Create review |\n\n### 6.3 Response Format\n\n```json\n{\n  \"success\": true,\n  \"data\": { ... },\n  \"meta\": {\n    \"page\": 1,\n    \"limit\": 20,\n    \"total\": 100\n  },\n  \"error\": null\n}\n```\n\n## 7. Caching Strategy (Redis)\n\n### 7.1 Cache Layers\n\n- **Session Cache**: Store JWT refresh tokens with TTL\n- **Availability Cache**: Precomputed slots for next 7 days, invalidated on schedule change\n- **Business List Cache**: Search results cached for 5 minutes\n- **Category Cache**: Static categories cached for 1 hour\n\n### 7.2 Invalidation\n\n- On appointment creation, invalidate availability cache for that business\n- On business update, invalidate business list cache\n- On category change, invalidate category cache\n\n## 8. Deployment & CI/CD\n\n### 8.1 Docker Compose\n\n```yaml\nversion: '3.8'\nservices:\n  postgres:\n    image: postgis/postgis:15-3.3\n    environment:\n      POSTGRES_DB: planity\n      POSTGRES_USER: planity\n      POSTGRES_PASSWORD: secret\n    volumes:\n      - postgres_data:/var/lib/postgresql/data\n    ports:\n      - \"5432:5432\"\n\n  redis:\n    image: redis:7-alpine\n    ports:\n      - \"6379:6379\"\n\n  api:\n    build: ./apps/api\n    ports:\n      - \"3000:3000\"\n    environment:\n      DATABASE_URL: postgresql://planity:secret@postgres:5432/planity\n      REDIS_URL: redis://redis:6379\n    depends_on:\n      - postgres\n      - redis\n\nvolumes:\n  postgres_data:\n```\n\n### 8.2 GitHub Actions\n\n- **CI Pipeline**:\n  - Lint and type-check all packages\n  - Run unit tests (Jest)\n  - Run integration tests\n  - Build mobile app (EAS Build for preview)\n- **CD Pipeline**:\n  - Deploy backend to staging/production\n  - Submit mobile app to TestFlight/Play Console\n\n## 9. Testing Strategy\n\n### 9.1 Mobile App\n\n- **Unit Tests**: Jest for hooks, utils, services\n- **Component Tests**: React Native Testing Library\n- **Integration Tests**: Detox for E2E flows (booking, search)\n\n### 9.2 Backend\n\n- **Unit Tests**: Jest for services, controllers, guards\n- **Integration Tests**: Supertest for API endpoints with test database\n- **E2E Tests**: Full flow with Docker Compose\n\n## 10. Performance Considerations\n\n- **API Response Time**: Target < 200ms for 95th percentile\n- **Database Indexing**: Indexes on business location, appointment start_time, user email\n- **Lazy Loading**: Images and heavy components in mobile app\n- **Code Splitting**: Expo Router lazy loads screens\n- **Animation**: Use React Native Reanimated for 60fps animations\n\n## 11. Security\n\n- **Authentication**: JWT with short-lived access tokens (15 min) and long-lived refresh tokens (7 days)\n- **Authorization**: Role-based access control (RBAC) for admin endpoints\n- **Data Encryption**: HTTPS everywhere, passwords hashed with bcrypt\n- **Input Validation**: Zod schemas on both frontend and backend\n- **Rate Limiting**: Redis-based rate limiting on auth endpoints\n\n## 12. Monitoring & Logging\n\n- **Backend**: NestJS logger with structured JSON logs\n- **Mobile**: Sentry for crash reporting\n- **Database**: PostgreSQL slow query log\n- **Metrics**: Prometheus + Grafana for API performance\n\n## 13. Conclusion\n\nThis architecture provides a scalable, maintainable foundation for the Planity Clone application. The separation into mobile app, backend API, and shared packages ensures clear ownership and independent deployability. The use of modern tooling (Expo, NestJS, Prisma, TanStack Query) accelerates development while maintaining high code quality.",
