Service Boundaries & Responsibilities\n\n### 5.1 Auth Service\n- **Responsibilities:** User registration, login, OAuth, JWT token management, password reset, session handling.\n- **Data:** Users table, refresh tokens table.\n- **Dependencies:** Supabase Auth (for OAuth providers), Redis (blacklisted tokens).\n\n### 5.2 User Service\n- **Responsibilities:** Profile management (CRUD), preferences, notification settings, address management.\n- **Data:** Users table (extended profile), addresses table.\n- **Dependencies:** Auth service (user identity).\n\n### 5.3 Business Service\n- **Responsibilities:** Business registration, profile management, hours, photos, amenities, staff management.\n- **Data:** Businesses, business_hours, business_photos, amenities, staff tables.\n- **Dependencies:** User service (provider ownership).\n\n### 5.4 Service Catalog Service\n- **Responsibilities:** Category hierarchy, service definitions, pricing, add-ons, staff-service associations.\n- **Data:** Categories, services, service_variations, staff_services tables.\n- **Dependencies:** Business service.\n\n### 5.5 Booking Service\n- **Responsibilities:** Slot availability calculation, booking creation, rescheduling, cancellation, hold management.\n- **Data:** Bookings, booking_holds, availability_slots (materialized view or computed).\n- **Dependencies:** Business service, Service Catalog, Payment service.\n\n### 5.6 Search Service\n- **Responsibilities:** Full-text search, geospatial queries, auto-complete, filtering, sorting.\n- **Data:** Search index (PostgreSQL tsvector + PostGIS), Redis cache for popular queries.\n- **Dependencies:** Business service, Service Catalog.\n\n### 5.7 Payment Service\n- **Responsibilities:** Payment processing (Stripe/PayPal), refunds, payout to providers, invoice generation.\n- **Data:** Payments, transactions, payout_records tables.\n- **Dependencies:** Booking service, User service.\n\n### 5.8 Notification Service\n- **Responsibilities:** Push notifications (Expo Push), email (SendGrid), SMS (Twilio), in-app notifications.\n- **Data:** Notification_logs, push_tokens tables.\n- **Dependencies:** Booking service, User service.\n\n### 5.9 Review Service\n- **Responsibilities:** Rating and review submission, moderation, aggregate calculations.\n- **Data:** Reviews, review_reports tables.\n- **Dependencies:** Booking service (verified bookings only).\n\n### 5.10 Admin Service\n- **Responsibilities:** Platform analytics, user management, business verification, content moderation.\n- **Data:** Admin logs, analytics snapshots.\n- **Dependencies:** All other services (read-only for most).\n\n## 6. API Design (RESTful)\n\n### 7.1 Authentication\n- `POST /api/auth/register` — Register with email/password\n- `POST /api/auth/login` — Login, returns JWT + refresh token\n- `POST /api/auth/refresh` — Refresh access token\n- `POST /api/auth/logout` — Invalidate refresh token\n- `POST /api/auth/forgot-password` — Send reset email\n- `POST /api/auth/reset-password` — Reset with token\n- `GET /api/auth/oauth/:provider` — OAuth redirect\n\n### 7.2 Users\n- `GET /api/users/me` — Get current user profile\n- `PATCH /api/users/me` — Update profile\n- `GET /api/users/me/bookings` — List user bookings (paginated)\n- `GET /api/users/me/favorites` — List favorite businesses\n- `POST /api/users/me/favorites/:businessId` — Add/remove favorite\n\n### 7.3 Businesses\n- `GET /api/businesses` — Search businesses (query params: q, lat, lng, radius, category, etc.)\n- `GET /api/businesses/:id` — Get business details\n- `POST /api/businesses` — Create business (provider only)\n- `PATCH /api/businesses/:id` — Update business (owner only)\n- `GET /api/businesses/:id/services` — List services\n- `GET /api/businesses/:id/staff` — List staff\n- `GET /api/businesses/:id/availability` — Get available slots (query: date, staffId)\n\n### 7.4 Bookings\n- `POST /api/bookings` — Create booking (with slot hold)\n- `GET /api/bookings/:id` — Get booking details\n- `PATCH /api/bookings/:id` — Reschedule or cancel\n- `DELETE /api/bookings/:id` — Cancel booking\n\n### 7.5 Payments\n- `POST /api/payments/create-intent` — Create payment intent (Stripe)\n- `POST /api/payments/confirm` — Confirm payment (webhook)\n- `GET /api/payments/:id` — Get payment status\n\n### 7.6 Reviews\n- `GET /api/businesses/:id/reviews` — List reviews\n- `POST /api/reviews` — Create review (after booking)\n- `PATCH /api/reviews/:id` — Update review\n- `DELETE /api/reviews/:id` — Delete review\n\n## 8. Data Flow for Key Scenarios\n\n### 8.1 Booking Flow\n1. User searches for a service → `GET /api/businesses?q=haircut` returns results.\n2. User selects business → `GET /api/businesses/:id` loads detail.\n3. User picks service and staff → `GET /api/businesses/:id/availability?date=2024-01-15&staffId=123` returns slots.\n4. User selects slot → App calls `POST /api/bookings` with service, staff, startTime.\n5. Backend creates a booking hold (10 min TTL in Redis), returns booking ID.\n6. User proceeds to payment → `POST /api/payments/create-intent` with booking ID.\n7. Payment confirmed → Webhook updates booking status to CONFIRMED, releases hold, triggers notification.\n8. User receives push notification and email confirmation.\n\n### 8.2 Search with Geospatial\n1. App sends `GET /api/businesses?lat=40.7128&lng=-74.0060&radius=5&q=hair`.\n2. Backend uses PostGIS `ST_DWithin` to filter businesses within radius.\n3. Full-text search on business name and service names using `tsvector`.\n4. Results sorted by relevance (distance + text match).\n5. Response includes paginated list with next available slot (computed from availability).\n\n## 9. Caching Strategy\n\n| Data | Cache Location | TTL | Invalidation |\n|------|----------------|-----|--------------|\n| Business details | Redis | 5 min | On business update |\n| Service categories | Redis | 1 hour | On category change |\n| Search results | Redis (keyed by query) | 2 min | On new booking or business update |\n| User session | Redis | 30 days | On logout |\n| Availability slots | Redis (computed) | 1 min | On booking creation/cancellation |\n| Static assets (images) | CDN (CloudFront) | 1 day | On upload |\n\n## 10. Security Considerations\n\n- **Authentication:** JWT with short-lived access tokens (15 min) and long-lived refresh tokens (30 days) stored in secure HTTP-only cookies or SecureStore.\n- **Authorization:** Role-based access control (RBAC) using NestJS guards. Providers can only modify their own businesses.\n- **Rate Limiting:** 100 requests/min per user, 20 requests/min for auth endpoints (Redis-based).\n- **Data Validation:** Zod schemas on both client and server for type safety.\n- **SQL Injection:** Prevented by Prisma ORM parameterized queries.\n- **CORS:** Restricted to known origins.\n- **HTTPS:** Enforced in production.\n- **Payment Security:** Stripe Elements for PCI compliance; no raw card data stored.\n\n## 11.
