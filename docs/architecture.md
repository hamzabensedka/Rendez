Monorepo Structure\nWe use Nx with pnpm workspaces to manage the monorepo. The root contains configuration for TypeScript, ESLint, Prettier, and Docker Compose. The `apps` directory holds the three frontend applications and the backend API. The `packages` directory contains shared libraries.\n\n```\nplanity/\nв”ңв”Җв”Җ apps/\nв”Ӯ   в”ңв”Җв”Җ mobile/                  # Expo React Native app\nв”Ӯ   в”ңв”Җв”Җ provider-portal/         # Next.js web app for business owners\nв”Ӯ   в”ңв”Җв”Җ admin-dashboard/         # Next.js web app for admins\nв”Ӯ   в””в”Җв”Җ api/                     # NestJS backend\nв”ңв”Җв”Җ packages/\nв”Ӯ   в”ңв”Җв”Җ shared-types/            # TypeScript interfaces & enums\nв”Ӯ   в”ңв”Җв”Җ ui/                      # Shared React Native component library\nв”Ӯ   в”ңв”Җв”Җ config/                  # ESLint, TS configs\nв”Ӯ   в””в”Җв”Җ utils/                   # Common utilities (date, formatting)\nв”ңв”Җв”Җ tools/                       # Nx generators, scripts\nв”ңв”Җв”Җ docker-compose.yml\nв”ңв”Җв”Җ nx.json\nв”ңв”Җв”Җ pnpm-workspace.yaml\nв”ңв”Җв”Җ tsconfig.base.json\nв””в”Җв”Җ package.json\n```\n\n## 4. Backend Architecture (NestJS)\nThe backend is a single NestJS application organized into feature modules. Each module encapsulates its own domain logic, Prisma service, and controllers. This modular monolith allows clear boundaries and easy extraction if needed.\n\n### 4.1 Module Boundaries\n- **AuthModule**: Handles user registration, login, social auth, JWT issuance, and token refresh. Integrates with Supabase Auth for identity management.\n- **UserModule**: Profile management, notification preferences, payment methods.\n- **BusinessModule**: CRUD for businesses, services, staff, working hours, categories.\n- **BookingModule**: Appointment booking flow, slot computation, rescheduling, cancellation, optimistic concurrency control.\n- **ReviewModule**: Customer reviews, ratings, moderation, provider responses.\n- **PaymentModule**: Integration with Stripe, handling payment intents, refunds, and saving payment methods.\n- **NotificationModule**: Push notifications, email, SMS via adapters. Uses Supabase Realtime for in-app notifications.\n- **SearchModule**: Full-text search and geospatial queries using PostgreSQL/PostGIS.\n- **AdminModule**: Business approval, user management, analytics dashboards.\n\n### 4.2 Data Layer\n- **Prisma ORM** with PostgreSQL is the primary database. PostGIS extension enables location-based queries (e.g., find businesses within radius).\n- **Redis** is used for:\n  - Caching frequently accessed data (business details, service lists).\n  - Session store for rate limiting and temporary locks during booking.\n  - Queue for background jobs (email, notifications).\n- **Supabase** provides:\n  - Authentication (Supabase Auth) with social logins and phone OTP.\n  - File storage for user avatars, business photos, review images.\n  - Realtime subscriptions for live booking updates and notifications.\n\n### 4.3 Slot Computation Engine\nA dedicated service within BookingModule calculates available time slots. It considers:\n- Business regular hours and breaks.\n- Staff individual schedules (overrides).\n- Service duration and buffer times.\n- Existing bookings (with status confirmed/pending).\n- Holidays and special hours.\nThe algorithm generates slots at configurable intervals (default 15 min) and uses Redis locks to prevent double-booking during the confirmation window.\n\n## 5. Mobile App Architecture (Expo)\nThe mobile app targets iOS and Android via Expo managed workflow. It uses Expo Router for file-based navigation and TanStack React Query for server state management.\n\n### 5.1 Folder Structure\n```\napps/mobile/\nв”ңв”Җв”Җ app/                        # Expo Router pages\nв”Ӯ   в”ңв”Җв”Җ (auth)/                 # Auth screens (login, signup)\nв”Ӯ   в”ңв”Җв”Җ (tabs)/                 # Main tab navigator\nв”Ӯ   в”Ӯ   в”ңв”Җв”Җ home.tsx\nв”Ӯ   в”Ӯ   в”ңв”Җв”Җ search.tsx\nв”Ӯ   в”Ӯ   в”ңв”Җв”Җ bookings.tsx\nв”Ӯ   в”Ӯ   в””в”Җв”Җ profile.tsx\nв”Ӯ   в”ңв”Җв”Җ business/[id].tsx       # Business detail\nв”Ӯ   в”ңв”Җв”Җ booking/                # Booking flow steps\nв”Ӯ   в””в”Җв”Җ ...\nв”ңв”Җв”Җ components/                 # Screen-specific components\nв”ңв”Җв”Җ hooks/                      # Custom hooks (useAuth, useBookings)\nв”ңв”Җв”Җ services/                   # API client functions (React Query)\nв”ңв”Җв”Җ stores/                     # Zustand stores for UI state\nв”ңв”Җв”Җ utils/                      # Helpers\nв”ңв”Җв”Җ constants/                  # Theme, config\nв””в”Җв”Җ app.json                    # Expo config\n```\n\n### 5.2 State Management\n- **Server State**: TanStack React Query handles all API data fetching, caching, and mutations. Optimistic updates are used for favorites and booking confirmations.\n- **Client State**: Zustand stores manage UI state (e.g., selected filters, map view toggle).\n- **Auth State**: Supabase client SDK manages session and user object; a custom hook exposes it.\n\n### 5.3 Navigation\nExpo Router provides typed, file-based routing. Deep linking is configured for booking confirmations and password resets. The tab navigator includes Home, Search, Bookings, and Profile. Modals are presented for filters and booking steps.\n\n### 5.4 UI & Animations\n- The `packages/ui` library provides reusable components (Button, Card, Calendar, TimeSlotPicker, etc.) built with React Native core and styled using a design token system.\n- React Native Reanimated powers smooth transitions, map interactions, and the booking flow step animations.\n- Dark mode is supported via a theme context that switches design tokens.\n\n## 6. Web Portals (Provider & Admin)\nBoth portals are built with Next.js (App Router) for SSR and SEO where needed. They share the same UI components from `packages/ui` adapted for web (using React Native Web or a parallel web component library). For simplicity, we use a separate web component library within `packages/ui-web` that mirrors the mobile design system.\n\n### 6.1 Provider Portal\nAllows business owners to:\n- Manage business profile, services, staff, working hours.\n- View and manage bookings (calendar view, list).\n- Respond to reviews.\n- View basic analytics.\n\n### 6.2 Admin Dashboard\nProvides super admin capabilities:\n- Approve/reject new businesses.\n- Manage categories, users, and platform settings.\n- View global analytics and reports.\n- Moderate reviews.\n\nBoth portals consume the same NestJS API as the mobile app, with role-based access control (RBAC) enforced via guards.\n\n## 7. Shared Packages\n- **shared-types**: All TypeScript interfaces for API requests/responses, enums (BookingStatus, UserRole), and utility types. This package is the single source of truth for data contracts.\n- **ui**: React Native component library with design tokens (colors, spacing, typography). Components are built with accessibility in mind.\n- **config**: Shared ESLint, Prettier, and TypeScript configurations.\n- **utils**: Date formatting, price calculation, validation helpers used across apps.\n\n## 8. Data Flow & API Design\n### 8.1 API Style\nRESTful API with JSON payloads. Endpoints are versioned under `/api/v1/`. NestJS controllers handle validation via class-validator and transform via class-transformer. Swagger documentation is auto-generated.\n\n### 8.2 Typical Booking Flow\n1. Mobile app fetches business details and available slots via `GET /businesses/:id/availability?date=...&serviceId=...`.\n2. User selects slot and confirms; app sends `POST /bookings` with service, staff, start time.\n3. Backend acquires a Redis lock for that slot, validates availability again, creates a pending booking, and initiates payment if required.\n4. On payment success, booking status becomes confirmed; otherwise, it's cancelled and lock released.\n5. Confirmation is returned, and a push notification is scheduled.\n\n### 8.3 Real-time Updates\nSupabase Realtime is used for:\n- Live availability changes (when a slot is taken).\n- Booking status updates (confirmed, cancelled).\n- New review notifications for providers.\nThe mobile app subscribes to relevant channels using the Supabase client.\n\n## 9. Database Schema (High-Level)\nThe Prisma schema defines the following main models:\n- **User**: id, email, phone, role (CUSTOMER, PROVIDER, ADMIN), profile info.\n- **Business**: id, ownerId (User), name, description, address, location (PostGIS point), photos, workingHours (JSON), approvalStatus.\n- **Service**: id, businessId, name, duration, price, bufferBefore, bufferAfter, categoryId.\n- **Staff**: id, businessId, name, photo, schedule (JSON).\n- **Booking**: id, customerId, businessId, serviceId, staffId, startTime, endTime, status, paymentId.\n- **Review**: id, bookingId, customerId, businessId, rating, text, photos, response.\n- **Category**: id, name, parentId, icon.\n- **PaymentMethod**: id, userId, token (from Stripe), type, last4.\n\nPostGIS is used on the `Business.location` column for distance-based queries. Indexes are created on frequently queried fields.\n\n## 10. Caching & Performance\n- **Redis Cache**: Business details, service lists, and category trees are cached with TTL. Cache invalidation occurs on updates via NestJS interceptors.\n- **Query Optimization**: Prisma queries use select to fetch only needed fields. Pagination uses cursor-based approach for infinite scroll.\n- **Image Optimization**: Supabase Storage serves images with on-the-fly resizing via transformation URLs.\n- **Mobile**: React Query provides stale-while-revalidate strategy, reducing network calls. List items use `FlashList` for performance.\n\n## 11. Authentication & Authorization\n- **Authentication**: Supabase Auth handles sign-up, login, social OAuth, and phone OTP. The mobile app uses Supabase client SDK. The backend validates Supabase JWT tokens using a custom NestJS guard that fetches the user from Supabase.\n- **Authorization**: Role-based guards check user role (Customer, Provider, Admin) on each endpoint. Business ownership is verified for provider actions.\n\n## 12. Real-time & Notifications\n- **Push Notifications**: Expo Push Notifications service for mobile. Backend queues notification jobs in Redis (BullMQ) and processes them via Firebase Cloud Messaging (FCM) / APNs.\n- **Email/SMS**: Handled by adapters (SendGrid, Twilio) triggered from the notification module.\n- **In-app**: Supabase Realtime channels for live booking updates and chat (future).\n\n## 13. Deployment & CI/CD\n- **Mobile**: EAS Build creates development, preview, and production builds. Over-the-air updates via EAS Update.\n- **Backend & Web Portals**: Dockerized using Docker Compose for local dev. Production deployment on a cloud platform (AWS ECS, Fly.io, or Railway) with a CI/CD pipeline via GitHub Actions.\n- **GitHub Actions Workflow**:\n  - Lint, type-check, and test on every PR.\n  - Build and push Docker images on merge to main.\n  - Deploy to staging/production environments.\n  - Run EAS Build for mobile on tag creation.\n- **Database**: PostgreSQL managed service (Supabase or AWS RDS) with automated backups. Redis managed (Upstash or ElastiCache).\n\n## 14. Testing Strategy\n- **Unit Tests**: Jest for all packages and apps. NestJS modules tested with mocked dependencies. React Native components tested with React Native Testing Library.\n- **Integration Tests**: API endpoints tested with supertest, using a test database.\n- **E2E Tests**: Detox for mobile critical flows (booking, auth). Playwright for web portals.\n- **Contract Testing**: (Optional) Pact to ensure API compatibility between frontend and backend.\n\n## 15.
