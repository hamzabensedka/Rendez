  "content": "# Planity Clone Architecture\n\n## 1. Overview\nPlanity Clone is a mobile-first beauty & wellness booking platform. The system is designed as a monorepo with clear separation between mobile apps (customer, provider, admin) and backend services. It leverages Expo (React Native) for cross-platform mobile delivery, NestJS for a modular backend, PostgreSQL with PostGIS for geospatial queries, Redis for caching and real-time data, and Prisma as the ORM. The architecture emphasizes scalability, maintainability, and a clean separation of concerns.\n\n## 2. Technology Stack\n- **Mobile**: Expo SDK 50+, React Native, TypeScript, Expo Router (file-based routing), TanStack React Query (server state), React Native Reanimated (animations), React Native Maps, Expo Notifications.\n- **Backend**: NestJS (Node.js), TypeScript, Prisma ORM, PostgreSQL 15, PostGIS 3, Redis 7, BullMQ (job queues), Socket.IO (optional real-time).\n- **Infrastructure**: Nx monorepo, pnpm workspaces, Docker Compose (local dev), GitHub Actions (CI/CD), Supabase (auth, storage, real-time optional), EAS Build (mobile CI/CD), Jest (testing).\n- **Shared**: TypeScript types, validation schemas (Zod), design tokens, UI components (shared package).\n\n## 3. Monorepo Structure\nUsing Nx with pnpm workspaces, the repository is organized into apps, libs, and tools.\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile-customer/          # Expo app for customers\n│   ├── mobile-provider/          # Expo app for providers\n│   ├── admin-dashboard/          # Web (React) admin dashboard\n│   └── api/                      # NestJS backend\n├── libs/\n│   ├── shared-types/             # TypeScript interfaces & enums\n│   ├── shared-ui/                # Reusable UI components (React Native)\n│   ├── shared-utils/             # Helpers, formatters, validators\n│   ├── api-client/               # Generated API client (axios/fetch wrapper)\n│   └── design-tokens/            # Colors, typography, spacing\n├── tools/\n│   ├── eslint-config/\n│   ├── jest-config/\n│   └── scripts/\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n├── package.json\n└── README.md\n```\n\n## 4. Backend Architecture (NestJS)\nThe backend is a single NestJS application organized into modules following domain-driven design. It exposes RESTful APIs (with potential GraphQL for admin) and uses BullMQ for background jobs.\n\n### 4.1 Module Boundaries\n- **AuthModule**: Registration, login, social OAuth, JWT issuance, refresh tokens, email verification, password reset. Integrates with Supabase Auth or custom implementation.\n- **UserModule**: Profile management, roles (customer, provider, admin), admin user CRUD.\n- **BusinessModule**: Business CRUD, categories, staff, services, opening hours, amenities. Provider onboarding and admin approval.\n- **BookingModule**: Appointment creation, slot computation, extras, status management, cancellation/rescheduling.\n- **AvailabilityModule**: Real-time slot calculation based on staff schedules, service durations, existing bookings, breaks. Uses Redis for caching computed slots.\n- **PaymentModule**: Integration with Stripe/PayPal, payment intent creation, webhook handling, refunds.\n- **ReviewModule**: Ratings and reviews for businesses.\n- **NotificationModule**: Push notifications (Expo), email (SendGrid), SMS (Twilio). Triggered by events.\n- **SearchModule**: Full-text search (PostgreSQL tsvector or Elasticsearch), geospatial search (PostGIS), autocomplete.\n- **AdminModule**: Dashboard metrics, user/business management, category management, system config.\n- **CommonModule**: Guards (JWT, Roles), interceptors (logging, transformation), pipes (validation), filters (exception handling).\n\n### 4.2 API Design\n- RESTful endpoints with versioning (`/api/v1/...`).\n- Request/response validation using Zod (shared with frontend).\n- Pagination, filtering, sorting standardized via query parameters.\n- Error responses follow RFC 7807 Problem Details.\n\n### 4.3 Database Schema (Prisma)\nKey models:\n- **User**: id, email, passwordHash, role, verified, profile info.\n- **Business**: id, ownerId, name, description, address, location (PostGIS point), coverImage, approvalStatus.\n- **Category**: id, name, icon, parentId (self-relation).\n- **Service**: id, businessId, categoryId, name, duration, price, description.\n- **Staff**: id, businessId, userId (optional), name, photo, specialties.\n- **Schedule**: id, staffId, dayOfWeek, startTime, endTime, breaks.\n- **Appointment**: id, customerId, businessId, serviceId, staffId, startTime, endTime, status, extras.\n- **Review**: id, customerId, businessId, rating, comment.\n- **Payment**: id, appointmentId, amount, status, stripePaymentIntentId.\n- **Notification**: id, userId, type, title, body, read.\n\nPostGIS extension used for `location` column (geometry(Point, 4326)) and spatial queries (ST_DWithin, ST_Distance).\n\n### 4.4 Caching Strategy (Redis)\n- **Slot cache**: Precomputed available slots for next N days per staff/service, invalidated on booking/cancellation.\n- **Session cache**: Refresh token blacklist/whitelist.\n- **Rate limiting**: API rate limits per user/IP.\n- **Search autocomplete**: Cached suggestions.\n- **Real-time features**: Optional presence/typing indicators via Redis pub/sub.\n\n### 4.5 Background Jobs (BullMQ)\n- **Slot recomputation**: Triggered after any booking change, recalculates availability for affected staff.\n- **Reminders**: Push notifications 24h and 1h before appointment.\n- **Email verification**: Send verification email on registration.\n- **Payment settlement**: Handle delayed capture or refunds.\n- **Data cleanup**: Archive old notifications, logs.\n\n### 4.6 Authentication & Authorization\n- JWT access tokens (15 min) and refresh tokens (30 days) stored securely (HTTP-only cookies for web, secure storage for mobile).\n- Role-based guards: `@Roles('customer')`, `@Roles('provider')`, `@Roles('admin')`.\n- Admin MFA via TOTP (speakeasy).\n- Social login: Google/Apple OAuth2, handled by backend exchanging code for tokens.\n\n## 5. Frontend Architecture (Mobile)\nThe mobile apps (customer and provider) share the same Expo project but use different entry points and configurations. The admin dashboard is a separate React web app (not detailed here).\n\n### 5.1 Expo Router & Navigation\n- File-based routing under `app/` directory.\n- Nested layouts for authenticated/unauthenticated flows.\n- Deep linking support for booking confirmations, password resets.\n\n### 5.2 State Management\n- **Server state**: TanStack React Query for all API data fetching, caching, and mutations. Optimistic updates for booking, favorites.\n- **Client state**: React Context for auth state, theme, and UI preferences.\n- **Form state**: React Hook Form + Zod validation (shared schemas).\n\n### 5.3 UI & Animations\n- Shared UI library (`libs/shared-ui`) with components built using React Native core and Reanimated for animations (transitions, micro-interactions).\n- Design tokens from `libs/design-tokens` ensure consistency.\n- Map integration: `react-native-maps` with custom markers and clustering.\n\n### 5.4 Key Flows\n- **Guest browsing**: Home screen with curated lists, search, business detail. Booking CTA triggers auth modal.\n- **Customer booking**: Step-by-step wizard (service → staff → date → time → extras → review → payment). Date/time selection uses cached slots from API.\n- **Provider management**: Dashboard with upcoming appointments, business editing, staff management, schedule configuration.\n\n## 6. Shared Packages\n- **shared-types**: All TypeScript interfaces, enums, DTOs. Single source of truth for API contracts.\n- **shared-ui**: Reusable components (Button, Card, Modal, Input, etc.) with variants, loading/error/empty states.\n- **shared-utils**: Date formatting, distance calculation, price formatting, validation helpers.\n- **api-client**: Auto-generated Axios instance with interceptors for auth token injection, error handling, and base URL configuration. Generated from OpenAPI spec or manually typed.\n\n## 7. Infrastructure & DevOps\n### 7.1 Local Development\n- Docker Compose spins up PostgreSQL (with PostGIS), Redis, and optional Supabase local stack.\n- Backend runs with `nx serve api`; mobile with `nx start mobile-customer`.\n- Shared libs built automatically via Nx dependency graph.\n\n### 7.2 CI/CD\n- **GitHub Actions**:\n  - Lint, type-check, unit tests (Jest) on every PR.\n  - Build and publish Docker image for API on merge to main.\n  - EAS Build triggered for mobile apps on main branch (staging) and tags (production).\n- **EAS Build**: Handles native builds, code signing, OTA updates via EAS Update.\n\n### 7.3 Deployment\n- **Backend**: Containerized NestJS app deployed to a cloud provider (AWS ECS, Fly.io, or Railway). PostgreSQL and Redis as managed services (Supabase for DB, Upstash for Redis).\n- **Mobile**: Distributed via App Store / Google Play using EAS Submit.\n- **Admin Dashboard**: Static site deployed to Vercel/Netlify.\n\n### 7.4 Testing Strategy\n- **Unit tests**: Jest for all shared libs, backend services, and frontend hooks/components.\n- **Integration tests**: Supertest for API endpoints, Prisma test database.\n- **E2E**: Detox or Maestro for critical mobile flows (booking, auth).\n- **Contract testing**: Ensure API client matches backend DTOs.\n\n## 8. Security Considerations\n- All communications over HTTPS.\n- Input validation and sanitization (Zod + class-validator).\n- Rate limiting on auth and sensitive endpoints.\n- SQL injection prevention via Prisma parameterized queries.\n- CORS configured strictly.\n- Secrets management via environment variables (never committed).\n- Regular dependency audits.\n\n## 9. Monitoring & Observability\n- Structured logging (Pino) with correlation IDs.\n- Error tracking (Sentry) for both backend and mobile.\n- Performance monitoring (OpenTelemetry) for API latency.\n- Health check endpoints for uptime monitoring.\n\n## 10. Future Considerations\n- Migration to microservices if team scales; current modular monolith is sufficient.\n- GraphQL gateway for admin dashboard to reduce over-fetching.\n- Real-time availability updates via WebSockets.\n- AI-powered recommendations for personalized home screen.\n\n---\n*Document maintained by the architecture team.
