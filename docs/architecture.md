  "content": "# Planity Clone Architecture Document\n\n## 1. Overview\nThis document defines the complete system architecture for Planity Clone – a mobile‑first beauty & wellness booking platform. The system consists of a consumer mobile app, a provider web portal, an admin dashboard, a backend API, and background workers. All components are built within an Nx monorepo using TypeScript, with heavy emphasis on clean separation of concerns, scalability, and maintainability.\n\n## 2. Architecture Principles\n- **Modular Monolith with Clear Boundaries:** Core business logic lives in a single NestJS application split into strictly separated modules (services). Each module aligns with a bounded context.\n- **Mobile‑First, API‑Driven:** The consumer experience is an Expo/React Native app consuming a REST API. Admin and provider dashboards are web apps that also consume the same API.\n- **Stateless Backend, Externalised State:** Session and cache state reside in Redis; persistent data in PostgreSQL + PostGIS.\n- **Eventual Consistency for Non‑Critical Workflows:** Bookings and availability use optimistic locking and Saga‑like compensation for payment failures.\n- **Testability:** Every layer is unit‑testable (Jest). Integration tests for API endpoints, E2E for critical user journeys.\n\n## 3. System Context\n```\n+-------------------+       +-----------------------+       +-----------------------+\n|   Mobile App      |       |   Provider Web App    |       |   Admin Web App       |\n| (Expo/React Native)|<--->| (React/Next.js)       |<--->| (React/Next.js)       |\n+-------------------+       +-----------------------+       +-----------------------+\n              \\                          |                          /\n               \\                         |                         /\n                v                        v                        v\n+-------------------------------------------------------------------+\n|                          API Gateway (NestJS)                      |\n|  - Auth, Rate Limiting, Request Validation, Proxy to services     |\n+-------------------------------------------------------------------+\n              |          |          |            |          |\n              v          v          v            v          v\n+--------+ +--------+ +--------+ +--------+ +--------+ +-----------+\n| Auth   | |Business| |Booking | |Payment | | Notif  | | Availability|\n| Service| |Service | |Service | |Service | |Service | | Service    |\n+--------+ +--------+ +--------+ +--------+ +--------+ +-----------+\n       |         |         |         |          |          |\n       v         v         v         v          v          v\n+------------------+  +-----------+  +-----------+  +------------+\n|   PostgreSQL     |  |   Redis   |  |  Supabase |  |  File Store|\n| (PostGIS)        |  | (cache,   |  | (Auth,    |  | (images)   |\n|                  |  |  queue)   |  |  Storage) |  |            |\n+------------------+  +-----------+  +-----------+  +------------+\n```\n\n## 4. Technology Stack\n- **Mobile/Consumer App:** Expo SDK 50+, React Native, TypeScript, Expo Router, React Query, Reanimated 3, React Native Maps, Stripe SDK\n- **Web Dashboards:** React, TanStack Router, same API\n- **Backend:** NestJS (REST + WebSocket for real‑time), Prisma ORM, PostgreSQL 15+ with PostGIS 3+, Redis 7 (caching & BullMQ queues), Stripe SDK\n- **Authentication:** Supabase Auth (email/password, Google, Apple), JWT RS256 tokens validated by backend\n- **Infrastructure:** Nx monorepo with pnpm workspaces, Docker Compose for local dev, GitHub Actions CI/CD, EAS Build for mobile deployments\n- **Testing:** Jest, React Testing Library, Supertest, Detox (E2E)\n\n## 5. Monorepo Folder Structure\n```\nplanity/\n├── apps/\n│   ├── mobile/                # Expo consumer app\n│   ├── api/                   # NestJS backend (gateway + services)\n│   ├── provider-web/          # Provider dashboard web app\n│   ├── admin-web/             # Admin dashboard web app\n│   └── worker/                # Background job runner (NestJS standalone)\n├── libs/\n│   ├── shared-types/          # TypeScript interfaces & DTOs\n│   ├── ui/                    # Shared UI components (React Native & Web)\n│   ├── api-client/            # Auto‑generated API client (axios)\n│   ├── utils/                 # Common helpers (date, geolocation, etc.)\n│   └── config/                # Shared configs (ESLint, Jest, Tailwind)\n├── tools/                     # Nx generators, scripts\n├── docker-compose.yml         # Local PostgreSQL + Redis + Supabase\n├── nx.json\n├── package.json\n└── pnpm-workspace.yaml\n```\n\n## 6. Service Boundaries & Modules\nThe backend is a modular monolith but designed for future extraction into separate deployables. Each bounded context maps to a NestJS module.\n\n### 6.1 Auth Service\n- **Responsibilities:** User/Provider/Admin registration, login, token refresh, social login handling, password reset, email verification.\n- **Implementation:** Integrates with Supabase Auth; the backend stores minimal user profile data in its own database linked by `supabaseUserId`. After successful Supabase authentication, the API issues a short‑lived session token (signed with own secret) for API access. Role‑based claims (`user`, `provider`, `admin`) are embedded.\n- **ACL:** A `RolesGuard` checks permissions on each endpoint.\n\n### 6.2 Business Service\n- **Responsibilities:** CRUD for businesses, services, categories, staff members, operating hours, gallery images.\n- **Database:** Uses PostGIS for geolocation queries. Stores business coordinates, radius, operating hours as JSONB.\n- **Category Management:** Categories are global (managed by admin) but a business can assign categories to its services.\n\n### 6.3 Availability Service\n- **Responsibilities:** Slot computation engine. Given a business, service(s), date, and optional staff, returns available time slots.\n- **Algorithm:** Combines business operating hours, staff schedules, existing bookings (reserved slots), and breaks. It runs on‑demand in the booking flow or via a cacheable endpoint. Results are stored in Redis with a TTL (30 seconds) to avoid repeated heavy computation during the booking flow.\n- **Concurrency:** Before confirming a booking, a Redis‑based distributed lock (with `SET NX PX`) is acquired on the specific slot to prevent double booking.\n\n### 6.4 Booking Service\n- **Responsibilities:** Orchestrates the booking transaction: validate slot, create booking record (with status `pending`), initiate payment, and upon payment confirmation mark as `confirmed`. If payment fails, cancel the pending booking and release the lock.\n- **Rescheduling/Cancellation:** Updates booking status, creates audit logs, handles cancellation policies (check 24h window).\n- **Multi‑service Bookings:** A booking can contain multiple services (line items). The availability engine computes contiguous slots that sum up to total duration.\n\n### 6.5 Payment Service\n- **Responsibilities:** Integration with Stripe. Creates PaymentIntent, handles webhooks for payment status updates, refunds, and disputes. Never stores raw card details.\n- **Flow:** Booking service requests a PaymentIntent; the client confirms on mobile via Stripe SDK; webhooks notify `PaymentService` which then emits an event `payment.confirmed` or `payment.failed` to the Booking Service.\n\n### 6.6 Notification Service\n- **Responsibilities:** Push notifications (Expo Push), email (SendGrid/Mailgun), in‑app notifications.\n- **Implementation:** A BullMQ queue processes notification jobs (e.g., booking confirmation, reminder 24h before appointment, cancellation notice). The worker app (`apps/worker`) consumes these queues.\n- **Preferences:** Users can toggle notification types in their profile.\n\n### 6.7 Appointment Management\n- **Responsibilities:** Serving appointment lists (upcoming, past) with filters, ability to add to calendar (.ics generation), rate/review linking.\n- **Database:** Reads booking records, joined with business/service data.\n\n### 6.8 Search & Discovery\n- **Responsibilities:** Full‑text search across businesses and services using PostgreSQL full‑text search (with `tsvector`) and geolocation filtering via PostGIS. Also provides autocomplete suggestions (cached in Redis).\n- **Index:** A materialised view or a dedicated search table is maintained.\n\n### 6.9 Favorites & User Profile\n- **Responsibilities:** Toggle favorites, list favorites, basic user profile updates. Favorites table links `userId` to `businessId`.\n\n### 6.10 Admin Service\n- **Responsibilities:** Manage categories, moderate reviews, view analytics, manage platform settings. Separate guard ensures only admin role.\n\n## 7. Data Flow – Booking Example\n1. Mobile app requests available slots from `GET /availability?businessId=X&serviceIds=Y&date=...`. Availability Service computes and returns slots.\n2. User selects a slot, app calls `POST /bookings`. Booking Service acquires lock, validates, creates booking with status `pending`, returns booking ID and payment intent secret.\n3. App uses Stripe SDK to confirm payment. On success, Stripe webhook hits `PaymentService` which verifies signature, updates payment status, and publishes event.\n4. Booking Service (subscriber) receives `payment.confirmed`, updates booking to `confirmed`, releases slot lock, and enqueues notification job.\n5. Notification Worker sends push notification and email.\n\n## 8. Database Design (Key Tables)\n**Users** `id, supabaseUserId, email, name, phone, role, createdAt`\n**Businesses** `id, ownerId, name, description, categoryIds[], location (PostGIS Point), address, phone, website, ratingAvg, totalReviews, operatingHours (JSONB), imageUrls[]`\n**Services** `id, businessId, name, description, durationMinutes, price, categoryId, staffIds[]`\n**Staff** `id, businessId, name, specialty, serviceIds[], schedule (JSONB)`\n**Bookings** `id, userId, businessId, totalDuration, totalPrice, status (ENUM: pending, confirmed, canceled, completed, no_show), startTime, endTime, cancellationReason, createdAt`\n**BookingServices** `bookingId, serviceId, staffId, price, duration`\n**AvailabilitySlots** (materialised in cache, not persisted) Derived from operating hours, staff schedules, existing bookings.\n**Reviews** `id, userId, businessId, bookingId, rating, comment, createdAt`\n**Favorites** `userId, businessId, createdAt` (unique constraint)\n**Categories** `id, name, icon, displayOrder`\n**Notifications** `id, userId, type, title, body, read, createdAt`\n**PaymentIntents** `id, bookingId, stripePaymentIntentId, amount, status, createdAt`\n\nAll tables use UUID primary keys. PostGIS geometry column for location. Indexes on foreign keys, full‑text search vectors, and geospatial GIST indexes.\n\n## 9. Caching Strategy (Redis)\n- **Slot Availability:** Cache slot response per `(businessId, date, serviceIds, staffId)` for 30s. Invalidate on any booking change.\n- **Business Search Autocomplete:** Sorted set of business names keyed by prefix; updated daily via job.\n- **User Session:** JWT blacklist (logout) stored as a set with TTL matching token expiration.\n- **Rate Limiting:** Per IP/User using Redis sliding window.\n- **Locks:** Slot locks with key `lock:slot:{businessId}:{staffId}:{startTime}` and TTL 30s.\n\n## 10. API Design\nRESTful JSON over HTTPS. Versioned via URL prefix `/api/v1`. CRUD follows standard patterns. Key endpoints:\n- `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`\n- `GET /businesses?lat=&lng=&radius=&category=&query=`\n- `GET /businesses/:id`\n- `GET /availability?businessId=&serviceIds=&date=&staffId=`\n- `POST /bookings`, `GET /bookings/:id`, `POST /bookings/:id/cancel`, `POST /bookings/:id/reschedule`\n- `POST /favorites`, `DELETE /favorites/:businessId`, `GET /favorites`\n- `POST /reviews`, `GET /businesses/:id/reviews`\n- Admin endpoints under `/admin/...`\n\nRequest/response validation using `class-validator` and `class-transformer`. OpenAPI/Swagger documentation auto‑generated.\n\n## 11. Security\n- **Authentication:** Supabase Auth issues RS256 JWT; the API verifies the token using the Supabase JWKS endpoint. After verification, the API issues its own session token (HS256) with short expiry (15 min). Refresh tokens are stored in an HTTP‑only cookie (for web) or secure storage (mobile).\n- **Authorization:** NestJS guards and decorators (`@Roles('user')`, `@BusinessOwner()`) enforce ownership access.\n- **Data Protection:** All communication over TLS. Sensitive data encrypted at rest (application‑level for payment details).\n- **Input Validation:** Strict DTO validation prevents injection attacks.\n- **Rate Limiting:** Configurable per endpoint using Redis.\n\n## 12. Background Jobs & Notifications\nHandled by `apps/worker` which runs a NestJS application with BullMQ processors. Queues:\n- `notifications` – push & email notifications.\n- `reminders` – scheduled jobs (cron‑like) to check upcoming appointments and send reminders.\n- `data-cleanup` – delete expired sessions, anonymise old data.\n- `search-index` – refresh full‑text search materialised view.\n\n## 13. Mobile Architecture Details\n- **Navigation:** Expo Router file‑based routing. Stacks: `(auth)`, `(tabs)` (Home, Search/Favorites, Bookings, Profile).\n- **State Management:** TanStack React Query for server state, Zustand for local UI state (filters, temporary booking selections).\n- **Location:** Expo Location to get user coordinates; ask permission with fallback to manual city entry.\n- **Maps:** React Native Maps with custom markers and clustering (`react-native-map-clustering`). Map view and list view toggle with shared filter state.\n- **Offline Support:** Cached API responses via React Query for read‑only data (business detail, categories). Booking requires connectivity.\n- **Payments:** Stripe React Native SDK.\n- **Push:** Expo Notifications with FCM/APNs.\n\n## 14. Testing Strategy\n- **Unit Tests:** Jest for services, utilities, and React components. Backend services extensively tested with mocked Prisma and Redis.\n- **Integration Tests:** Supertest for API endpoints against a test database (Docker‑based) with seeded data.\n- **E2E:** Detox for critical paths (registration, booking, cancellation) on iOS/Android simulators in CI.\n- **Load Testing:** k6 scripts for slot availability and booking concurrency.\n\n## 15. CI/CD & Deployment\n- **GitHub Actions:** Monorepo‑aware workflows using Nx affected commands. On PR: lint, typecheck, unit & integration tests, EAS build preview. On merge to main: build staging Docker image for API, deploy to staging environment, run E2E, then promote to production after manual approval.\n- **EAS Build:** Mobile app builds triggered from CI using `eas build --platform all --profile production`.\n- **Docker Compose:** Local development environment with PostgreSQL, Redis, Supabase (local instance using `supabase/cli`).\n\n## 16. Infrastructure & Hosting\n- **API & Workers:** Deployed as containers on a Kubernetes cluster (or AWS ECS/Fargate).\n- **Database:** Managed PostgreSQL (AWS RDS or Supabase managed) with PostGIS extension.\n- **Redis:** Elasticache or Redis Labs.\n- **Supabase:** Used for auth and file storage (images). API validates tokens and can be pointed to self‑hosted Supabase.\n- **Images:** Uploaded to Supabase Storage; URLs stored in DB.\n\n## 17. Monitoring & Observability\n- **Structured Logging:** Pino logger with JSON format, shipped to CloudWatch/Loki.\n- **Metrics:** Prometheus metrics exposed from API; Grafana dashboards for booking rates, slot lock wait times, error rates.\n- **Tracing:** OpenTelemetry (optional) for distributed tracing across services.\n\n## 18. Conclusion\nThe architecture leverages industry‑standard patterns and battle‑tested technologies to deliver a robust, scalable booking platform. The modular design allows teams to develop independently while maintaining clear contracts. Future evolution (e.g., splitting into microservices) is straightforward because bounded contexts are already isolated within the monolith.",
