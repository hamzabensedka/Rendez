Technology Stack\n- **Mobile:** Expo SDK 50+, React Native, TypeScript, Expo Router (file-based routing), TanStack React Query (server state), React Native Reanimated (animations), React Native Maps.\n- **Backend:** NestJS (Node.js), TypeScript, Prisma ORM, PostgreSQL 15 with PostGIS extension, Redis (caching, queues), BullMQ (background jobs).\n- **Monorepo & Tooling:** Nx workspace, pnpm workspaces, ESLint, Prettier, Husky.\n- **Infrastructure:** Docker Compose (local dev), GitHub Actions (CI/CD), Supabase (optional auth storage, but we use own JWT), EAS Build (mobile builds).\n- **Testing:** Jest (unit/integration), React Native Testing Library, Supertest (API e2e).\n\n## 4. Monorepo Structure (Nx + pnpm)\nThe repository is organised as an Nx integrated monorepo with pnpm for package management. Applications and libraries are clearly separated.\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                  # Expo React Native app\n│   └── api/                     # NestJS backend\n├── libs/\n│   ├── shared-types/            # TypeScript interfaces, DTOs, enums\n│   ├── shared-ui/               # Reusable React Native components, design tokens\n│   ├── shared-utils/            # Date helpers, formatters, validators\n│   └── shared-config/           # ESLint, Prettier, tsconfig bases\n├── tools/                       # Nx generators, scripts\n├── docker-compose.yml           # PostgreSQL, Redis, pgAdmin\n├── nx.json\n├── package.json\n├── pnpm-workspace.yaml\n└── tsconfig.base.json\n```\n\n## 5. Service Boundaries & Modules\n\n### 6.1 Auth Module\n- Handles sign-up, login (email/password, Google, Apple), token refresh, password reset.\n- Issues JWT access tokens (15 min) and refresh tokens (30 days) stored in Redis.\n- Role-based guards (Customer, Provider, Admin) using custom decorators.\n- Endpoints: `POST /auth/signup`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/forgot-password`, `POST /auth/reset-password`.\n\n### 6.2 User Module\n- Manages user profiles, preferences, notification settings.\n- Endpoints: `GET /users/me`, `PATCH /users/me`, `DELETE /users/me`.\n\n### 6.3 Business Module\n- CRUD for businesses, services, staff, working hours, buffer times.\n- Provider-only access for own business; admin can manage all.\n- Endpoints: `POST /businesses`, `GET /businesses/:id`, `PATCH /businesses/:id`, `GET /businesses/:id/services`, `POST /businesses/:id/services`, etc.\n\n### 6.4 Search Module\n- Full-text search on business name, service name, city, postal code using PostgreSQL `tsvector`.\n- Geospatial search with PostGIS: `ST_DWithin` for radius queries, ordering by distance.\n- Filters: category, price range, rating, availability today, open now.\n- Endpoints: `GET /search?q=...&lat=...&lng=...&radius=...&filters=...`.\n\n### 6.5 Availability Module\n- Core engine that computes available time slots for a given business, service, staff, and date.\n- Considers business working hours, staff shifts, service duration, buffer times, existing bookings, and manual blocks.\n- Uses Redis for caching computed slots (TTL: 5 minutes) to reduce database load.\n- Endpoints: `GET /availability/:businessId?serviceId=...&staffId=...&date=...`.\n\n### 6.6 Booking Module\n- Manages appointment lifecycle: create, confirm, cancel, reschedule.\n- Validates slot availability atomically using database transactions.\n- Triggers background jobs for confirmation emails, reminders.\n- Endpoints: `POST /bookings`, `GET /bookings`, `PATCH /bookings/:id/cancel`, `PATCH /bookings/:id/reschedule`.\n\n### 6.7 Review Module\n- Allows customers to leave ratings and reviews for completed appointments.\n- Moderation flagging for inappropriate content.\n- Endpoints: `POST /reviews`, `GET /businesses/:id/reviews`.\n\n### 6.8 Notification Module\n- Manages push notification tokens, email templates.\n- Processes jobs from BullMQ queues: `sendBookingConfirmation`, `sendReminder`, `sendCancellation`.\n- Integrates with Firebase Cloud Messaging (FCM) for push and a transactional email service (e.g., SendGrid).\n\n### 6.9 Admin Module\n- Super admin dashboard endpoints: user management, business approval, platform analytics.\n- Protected by Admin role guard.\n\n## 7. Database Design (Prisma + PostgreSQL + PostGIS)\n\n### 7.1 Core Tables\n- **User:** id, email, passwordHash, name, phone, avatarUrl, role (CUSTOMER, PROVIDER, ADMIN), createdAt.\n- **Business:** id, ownerId (FK User), name, description, address, location (PostGIS geometry point), phone, website, category, photos[], averageRating, isActive.\n- **Service:** id, businessId, name, description, duration (minutes), price, category, isActive.\n- **Staff:** id, businessId, name, photo, specialties[], isActive.\n- **StaffService:** staffId, serviceId (many-to-many).\n- **WorkingHours:** id, businessId, dayOfWeek (0-6), openTime, closeTime.\n- **StaffShift:** id, staffId, dayOfWeek, startTime, endTime.\n- **BlockedTime:** id, businessId (or staffId nullable), date, startTime, endTime, reason.\n- **Booking:** id, customerId, businessId, serviceId, staffId, startDateTime, endDateTime, status (PENDING, CONFIRMED, CANCELLED, COMPLETED), notes, createdAt.\n- **BookingService:** bookingId, serviceId (for multi-service bookings).\n- **Review:** id, bookingId, customerId, businessId, rating (1-5), comment, createdAt.\n- **Favorite:** userId, businessId (unique pair).\n- **NotificationToken:** userId, token, platform.\n\n### 7.2 Geospatial Indexing\n- Business.location uses `geometry(Point, 4326)` with a GiST index for fast nearest-neighbor queries.\n- Search queries combine `ST_DWithin` and `tsvector` for relevance.\n\n### 7.3 Caching Strategy (Redis)\n- **Availability slots:** Cache key `availability:{businessId}:{serviceId}:{staffId}:{date}` with JSON array of slots. Invalidate on booking creation/cancellation.\n- **Business details:** Cache for 10 minutes, invalidate on update.\n- **Search results:** Not cached due to high variability; rely on database performance.\n- **Session/refresh tokens:** Stored in Redis with TTL.\n\n## 8. API Design\n- **RESTful** with JSON payloads.\n- Versioning via URL prefix `/api/v1/`.\n- Consistent error format: `{ statusCode, message, errors? }`.\n- Pagination: cursor-based for search results (`?cursor=...&limit=20`).\n- Rate limiting using Redis (e.g., 100 requests/min per user).\n\n## 9. Authentication & Authorization Flow\n1. User logs in → server validates credentials → issues access token (JWT, 15 min) and refresh token (opaque, stored in Redis with 30-day TTL).\n2. Mobile app stores tokens securely (expo-secure-store).\n3. Every API request includes `Authorization: Bearer <access_token>`.\n4. On 401, the app uses the refresh token to obtain a new access token via `/auth/refresh`.\n5. Roles are embedded in JWT payload; `RolesGuard` checks required roles.\n\n## 10. Real-Time Availability Computation\n- **Algorithm:** For a given business, service, staff, and date:\n  1. Fetch business working hours and staff shifts for that day.\n  2. Generate potential slots based on service duration + buffer time.\n  3. Fetch existing confirmed bookings for that staff (and business-level blocks).\n  4. Exclude overlapping slots.\n  5. If no staff specified, aggregate across all staff providing the service.\n- **Optimisation:** Pre-compute and cache in Redis with a short TTL. Use database-level conflict checks during booking creation with `SELECT ... FOR UPDATE` to prevent double-booking.\n\n## 11. Background Jobs (BullMQ)\n- **Queues:** `notifications`, `reminders`, `data-cleanup`.\n- **Processors:**\n  - `sendBookingConfirmation`: sends push and email after booking.\n  - `sendReminder`: scheduled 24h and 1h before appointment.\n  - `sendCancellation`: notifies provider and customer.\n  - `cleanExpiredTokens`: removes expired refresh tokens from Redis.\n- Redis is used as the broker for BullMQ.\n\n## 12. Mobile App Architecture\n- **Routing:** Expo Router with file-based navigation. Authenticated routes are protected by a layout that checks auth state.\n- **State Management:**\n  - **Server state:** TanStack React Query for all API data (caching, pagination, optimistic updates).\n  - **Client state:** Zustand for global UI state (e.g., selected location, theme).\n- **API Client:** Axios instance with interceptors for token refresh and error handling.\n- **Animations:** React Native Reanimated for smooth transitions, booking confirmation animations, and map interactions.\n- **Maps:** `react-native-maps` with custom clustering.\n- **Offline Support:** React Query's `networkMode` allows stale data display; critical mutations queue when offline (future enhancement).\n\n## 13. Shared Packages & Design System\n- **Design Tokens:** Defined in `shared-ui/tokens` (colors, spacing, typography, shadows).\n- **Components:** Button, Input, Card, Avatar, Rating, etc., all themed and accessible.\n- **Validation:** Zod schemas shared between frontend and backend for DTOs.\n- **Types:** All interfaces and enums in `shared-types` ensure type safety across the stack.\n\n## 14. Deployment & CI/CD\n\n### 14.1 Local Development\n- Docker Compose spins up PostgreSQL (with PostGIS), Redis, and pgAdmin.\n- `pnpm dev` starts both mobile (Expo) and API (NestJS) with hot reload.\n\n### 14.2 CI/CD Pipeline (GitHub Actions)\n- **On push:** Lint, type-check, unit tests (Jest), build affected projects (Nx).\n- **On PR to main:** Integration tests, E2E API tests.\n- **On release tag:** Build and deploy:\n  - **Mobile:** EAS Build creates development/ production builds; OTA updates via EAS Update.\n  - **Backend:** Docker image built and pushed to container registry; deployed to cloud (e.g., AWS ECS, Railway).\n\n### 14.3 Infrastructure\n- **Database:** Managed PostgreSQL with PostGIS (e.g., Supabase, AWS RDS).\n- **Redis:** Managed instance (Upstash, ElastiCache).\n- **File Storage:** Supabase Storage or S3 for business photos.\n- **Push Notifications:** Firebase Cloud Messaging.\n\n## 15. Testing Strategy\n- **Unit Tests:** Jest for utilities, hooks, services (both mobile and API).\n- **Component Tests:** React Native Testing Library for UI components.\n- **API Integration Tests:** Supertest against NestJS with a test database.\n- **E2E Tests:** Detox (mobile) for critical flows (booking, auth).\n- **Performance Tests:** k6 for availability endpoint under load.\n\n## 16. Security Considerations\n- All endpoints use HTTPS.\n- Passwords hashed with bcrypt.\n- JWT secrets rotated periodically.\n- Input validation with Zod/Pipes.\n- Rate limiting on auth and public endpoints.\n- SQL injection prevented by Prisma.\n- CORS configured for mobile app origin.\n\n## 17. Scalability & Performance\n- **Database:** Read replicas for search-heavy queries; connection pooling with PgBouncer.\n- **Caching:** Redis reduces primary database load for availability and business details.\n- **Background Jobs:** Offload time-consuming tasks to BullMQ workers.\n- **Mobile:** React Query pagination and optimistic updates ensure smooth UX.\n- **Future:** Split backend into microservices if needed (e.g., separate search service), but monolith is sufficient for initial scale.\n\n## 18. Conclusion\nThis architecture provides a solid foundation for Planity Clone, balancing rapid development with production-grade scalability. The monorepo structure with shared types and design system ensures consistency, while the chosen technologies (Expo, NestJS, Prisma, Redis) offer a modern, type-safe stack. The clear separation of concerns and modular backend allow for future growth without major refactoring.",
