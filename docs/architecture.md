  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nThe Planity Clone is a monorepo built with Nx, using Expo (React Native) for the mobile client and NestJS for the API server. Data is persisted in PostgreSQL with PostGIS for geo‑queries, cached in Redis, and background jobs are handled by BullMQ. The stack follows clean‑architecture principles: presentation layer (Expo Router + React Query), domain layer (shared types & services), infrastructure layer (NestJS controllers, Prisma ORM, Redis client). CI/CD is orchestrated via GitHub Actions, with Docker Compose for local dev and EAS Build for OTA updates.\n\n## 2. Service Boundaries\n| Service | Responsibility | Tech |\n|---------|----------------|------|\n| **Mobile App** | UI, navigation, local state, offline cache, Reanimated animations | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated |\n| **API Gateway** | HTTP REST/GraphQL façade, auth, rate‑limiting, request validation | NestJS (Express adapter) |\n| **Business Service** | CRUD for businesses, categories, geo‑search, availability slots | NestJS + Prisma + PostGIS |\n| **Booking Service** | Appointment creation, rescheduling, cancellation, slot computation | NestJS + Prisma |\n| **User Service** | Authentication, profile, favorites, reviews | NestJS + Prisma + Redis (session store) |\n| **Payment Service** | Integration with payment provider (Stripe/PayPal) | NestJS |\n| **Notification Service** | Email/SMS/push notifications, background job processing | BullMQ + Redis |\n| **Admin Portal** | Admin dashboard for content & user management | NestJS (separate module) + optional React web (out of scope) |\n| **Worker** | Long‑running jobs (availability recomputation, reminder emails) | BullMQ workers (NestJS) |\n\n## 3. Data Flow\n1. User opens app → Expo Router renders screens.\n2. UI fetches data via React Query → calls API Gateway (`/api/*`).\n3. Gateway validates JWT, forwards to appropriate NestJS service.\n4. Service uses Prisma to interact with PostgreSQL (PostGIS for location queries).\n5. Frequently read data (business listings, categories) cached in Redis (TTL 5‑10 min).\n6. Mutations (booking, payment) trigger BullMQ jobs for notifications, slot recomputation, etc.\n7. Admin portal uses same API with role‑based guards.\n\n## 4. Key Architectural Decisions\n- **Clean Architecture Layers**: Presentation (mobile UI) → Application (React Query hooks, NestJS services) → Domain (shared-types, business logic) → Infrastructure (Prisma, Redis, BullMQ).\n- **State Management**: React Query handles server state; UI‑only state (modal visibility, form inputs) kept in React state or lightweight Zustand.\n- **Geo‑Search**: PostGIS `ST_DWithin` for radius queries; exposed via Business Service `findNearby`.\n- **Availability & Slot Computation**: Business defines recurring opening hours; slot generation runs as a BullMQ job on change, results stored in a materialized view or cached table for fast lookup.\n- **Authentication**: JWT issued by NestJS AuthService; mobile stores token in SecureStore; refresh token rotation via Redis.\n- **Payments**: Delegated to Stripe SDK on mobile (payment sheet) with webhook endpoint verifying events; API creates payment intents and records transactions.\n- **Notifications**: BullMQ queues for email (SendGrid), SMS (Twilio), and push (Expo Notifications). Workers run alongside API or as separate containers.\n- **Caching Strategy**: Read‑through cache for business listings; write‑through invalidation on mutation via Redis pub/sub or direct delete.\n- **Error Handling**: Central NestJS exception filter maps to standardized JSON error format; mobile uses React Query’s error boundary.\n- **Testing**: Unit tests with Jest for services & utils; integration tests with Supertest for API endpoints; mobile component testing with React Native Testing Library & Jest.\n- **CI/CD**: GitHub Actions runs lint, unit, e2e (Detox) on PR; on merge to main triggers EAS build (iOS/Android) and Docker image push; deployment to Kubernetes or ECS via Helm charts (out of scope).\n- **Dev Experience**: Docker Compose spins up PostgreSQL (with PostGIS extension), Redis, API, and optional Beekeeper Studio; hot reload via Expo dev client.\n\n## 6. Security Considerations\n- HTTPS enforced at API gateway (termination at load balancer or Nginx).\n- JWT short‑lived (15 min) with refresh token rotation stored HTTP‑only cookie (if web) or SecureStore.\n- Role‑based access control (RBAC) via NestJS guards (`@Roles('admin')`).\n- Input validation using class‑validator & DTOs.\n- Rate limiting via NestJS Throttler.\n- Secrets managed through environment variables; .env excluded; CI injects secrets.\n- Data encryption at rest (RDS encryption) and in‑process TLS.\n\n## 7. Scalability & Observability\n- Horizontal scaling of API containers behind a load balancer.\n- Redis cluster for caching and BullMQ.\n- Read replicas for PostgreSQL heavy read workloads (business search).\n- Logging via Winston (JSON format) shipped to ELK or CloudWatch.\n- Metrics via Prometheus endpoint (`@nestjs/terminus`) and Grafana dashboards.\n- Distributed tracing (OpenTelemetry) optional for future.\n\n## 8. Deployment Summary\n- **Local**: `docker compose up --build` → API at http://localhost:3000, Expo dev client connects via LAN.\n- **Staging**: GitHub Actions builds Docker images, pushes to registry, triggers Helm upgrade (or ECS task definition update).\n- **Production**: Same pipeline; EAS builds OTA updates for mobile; APK/IPA distributed via TestFlight/Google Play internal testing.\n- **Rollback**: Docker image tag immutability allows quick rollback; DB migrations are backward‑compatible.\n\n---\n*This document captures the high‑level architecture, service boundaries, and folder layout needed to start development of the Planity Clone using the specified stack.*\n",
