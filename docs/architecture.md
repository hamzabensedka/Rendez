The system is designed as a **monorepo** managed by **Nx** with **pnpm** workspaces, comprising a React Native mobile app (Expo), a NestJS API server, and shared libraries. The architecture prioritizes **separation of concerns**, **scalability**, **real-time capabilities**, and **developer experience** through strict typing and code sharing.\n\n### Key Architectural Decisions\n- **Monorepo with Nx**: Single repository for mobile app, API, and shared packages. Service Boundaries & Module Design\n\n### 4.1 API Modules (NestJS)\n\nEach module follows a consistent internal structure:\n```\nmodule-name/\nв”ңв”Җв”Җ module-name.module.ts\nв”ңв”Җв”Җ module-name.controller.ts\nв”ңв”Җв”Җ module-name.service.ts\nв”ңв”Җв”Җ module-name.repository.ts\nв”ңв”Җв”Җ dto/\nв”ңв”Җв”Җ entities/\nв””в”Җв”Җ tests/\n```\n\n#### Auth Module\n- **Responsibility**: Registration, login, token management, social auth, password reset.\n- **Integration**: Supabase Auth for identity; local DB for user profiles.\n- **Endpoints**: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/forgot-password`, `POST /auth/reset-password`.\n- **Dependencies**: User module, Email service.\n\n#### User Module\n- **Responsibility**: Profile CRUD, preferences, payment methods, account deletion.\n- **Endpoints**: `GET /users/me`, `PATCH /users/me`, `DELETE /users/me`, `GET /users/me/payment-methods`.\n- **Dependencies**: Auth module (for identity).\n\n#### Business Module\n- **Responsibility**: Business CRUD, search, filtering, geospatial queries.\n- **Endpoints**: `GET /businesses`, `GET /businesses/:id`, `POST /businesses` (provider), `PATCH /businesses/:id`, `GET /businesses/nearby`.\n- **Dependencies**: Category module, Review module, PostGIS.\n\n#### Service Module\n- **Responsibility**: Services offered by businesses (CRUD, listing).\n- **Endpoints**: `GET /businesses/:id/services`, `POST /businesses/:id/services`, `PATCH /services/:id`, `DELETE /services/:id`.\n- **Dependencies**: Business module.\n\n#### Availability Module\n- **Responsibility**: Slot computation engine. Considers business hours, staff schedules, existing bookings, buffer times, holidays.\n- **Endpoints**: `GET /availability/:businessId?date=&serviceId=&staffId=`.\n- **Dependencies**: Business module, Booking module, Redis (caching).\n- **Design**: Pure computation service; no persistence. Fetches constraints, computes available windows, caches result with short TTL.\n\n#### Booking Module\n- **Responsibility**: Appointment lifecycle (create, confirm, reschedule, cancel). Holds slot during payment.\n- **Endpoints**: `POST /bookings`, `GET /bookings`, `GET /bookings/:id`, `PATCH /bookings/:id`, `POST /bookings/:id/cancel`.\n- **Dependencies**: Availability module, Payment module, Notification module, Redis (slot holds).\n\n#### Review Module\n- **Responsibility**: Ratings and reviews for completed appointments.\n- **Endpoints**: `POST /bookings/:id/review`, `GET /businesses/:id/reviews`.\n- **Dependencies**: Booking module (verification).\n\n#### Payment Module\n- **Responsibility**: Payment processing via Stripe/PayPal, saved cards.\n- **Endpoints**: `POST /payments/intent`, `POST /payments/confirm`, `GET /payments/methods`.\n- **Dependencies**: Booking module, User module.\n\n#### Notification Module\n- **Responsibility**: Push notifications, email, SMS. Reminders, booking confirmations, marketing.\n- **Endpoints**: `POST /notifications/register-device`, `GET /notifications/preferences`, `PATCH /notifications/preferences`.\n- **Dependencies**: User module, Firebase Cloud Messaging.\n\n#### Category Module\n- **Responsibility**: Hierarchical category management.\n- **Endpoints**: `GET /categories`, `POST /categories` (admin), `PATCH /categories/:id`.\n- **Dependencies**: None.\n\n#### Upload Module\n- **Responsibility**: Image upload for businesses, users, and reviews.\n- **Endpoints**: `POST /upload`, `DELETE /upload/:key`.\n- **Dependencies**: Supabase Storage.\n\n---\n\n## 5. Data Architecture\n\n### 5.1 PostgreSQL + PostGIS (Primary Database)\n\n**Key Tables:**\n- `users` вҖ” id, supabase_uid, email, name, phone, role (CUSTOMER, PROVIDER, ADMIN), created_at\n- `businesses` вҖ” id, owner_id (FK users), name, description, address, location (PostGIS geography), phone, hours (JSONB), photos[], status (PENDING, ACTIVE, SUSPENDED)\n- `services` вҖ” id, business_id (FK), category_id (FK), name, duration_minutes, price, buffer_minutes\n- `staff` вҖ” id, business_id (FK), user_id (FK, nullable), name, photo, specialties[]\n- `staff_schedules` вҖ” id, staff_id (FK), day_of_week, start_time, end_time\n- `business_hours` вҖ” id, business_id (FK), day_of_week, start_time, end_time\n- `holidays` вҖ” id, business_id (FK, nullable), staff_id (FK, nullable), date, reason\n- `bookings` вҖ” id, customer_id (FK), business_id (FK), service_id (FK), staff_id (FK, nullable), start_time, end_time, status (PENDING, CONFIRMED, COMPLETED, CANCELLED), total_price\n- `reviews` вҖ” id, booking_id (FK, unique), rating (1-5), comment, created_at\n- `categories` вҖ” id, name, icon, parent_id (self-referential FK)\n- `favorites` вҖ” id, user_id (FK), business_id (FK), created_at (unique constraint on user+business)\n- `payment_methods` вҖ” id, user_id (FK), stripe_payment_method_id, last4, brand, is_default\n\n**Geospatial Indexing:**\n```sql\nCREATE INDEX idx_businesses_location ON businesses USING GIST (location);\n```\nEnables efficient `ST_DWithin` queries for \"nearby\" searches.\n\n### 5.2 Redis (Cache & Real-Time)\n\n**Use Cases:**\n- **Slot Availability Cache**: Key pattern `availability:{businessId}:{date}:{serviceId}:{staffId}`. TTL: 60 seconds. Invalidated on booking creation/cancellation.\n- **Slot Holds**: Key pattern `hold:{bookingId}`. TTL: 600 seconds (10 min payment window). Prevents double-booking.\n- **Session Store**: JWT refresh tokens with TTL.\n- **Rate Limiting**: Token bucket per user/IP.\n- **WebSocket Pub/Sub**: Channel `bookings:{businessId}` for real-time slot updates to providers.\n\n### 5.3 Supabase\n- **Auth**: Manages user identities, social logins, email verification. The API server validates Supabase JWTs.\n- **Storage**: Business photos, user avatars, review images. Bucket policies restrict access.\n\n---\n\n## 6.
