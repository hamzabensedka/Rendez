  "content": "# Planity Clone Architecture\n\n## Overview\nThe Planity Clone is a mobile-first application that connects users with local service businesses for discovery, booking, and management of appointments. The architecture follows a clean, modular approach with clear separation of concerns between the mobile client, API server, and shared libraries. The monorepo is managed by Nx, enabling incremental builds, shared tooling, and consistent TypeScript types across the stack.\n\n## Tech Stack\n- **Frontend**: Expo (SDK 49), React Native, TypeScript, Expo Router (file‑based navigation), TanStack React Query (data fetching & caching), React Native Reanimated (gestures & animations)\n- **Backend**: NestJS (Node.js framework), Prisma ORM, PostgreSQL (primary DB), PostGIS extension for geo‑spatial queries, Redis (caching & pub/sub), BullMQ (job queue)\n- **DevOps & Infra**: Nx (monorepo tooling), pnpm (workspace package manager), Docker Compose (local dev), GitHub Actions (CI/CD), Supabase (auth & storage fallback), EAS Build (Expo OTA & app store builds), Jest (unit & integration testing)\n\n## High‑Level Architecture\nThe system is split into three logical layers:\n1. **Presentation Layer** – Expo React Native app consuming the API via React Query.\n2. **Application Layer** – NestJS API exposing REST/GraphQL endpoints, handling business logic, validation, and orchestrating Prisma.\n3. **Data Layer** – PostgreSQL with PostGIS for geo‑search, Redis for caching & transient job data, and Supabase for optional auth/storage.\n\nCross‑cutting concerns (logging, error handling, monitoring) are implemented as NestJS interceptors and React Native middleware.\n\n## Service Boundaries\n### Frontend (Mobile)\n- **Apps/mobile** – Expo project.\n  - **src/**\n    - **app/** – Expo Router file‑based routes (tabs, stack).\n    - **components/** – Reusable UI pieces (buttons, cards, modals).\n    - **features/** – Feature‑scoped slices (auth, browse, booking, profile, favorites, reviews).\n    - **hooks/** – Custom React Query hooks wrapping API calls.\n    - **lib/** – API client (axios or fetch), Reanimated shared values, theme.\n    - **design-system/** – Shared component library (also published to libs/ui).\n    - **utils/** – Helpers (date, formatting, geolocation).\n    - **assets/** – Images, fonts, icons.\n    - **i18n/** – Localization files.\n    - **tests/** – Unit & e2e tests (Jest + React Native Testing Library).\n- **State Management** – React Query handles server state; React Context or Zustand for UI‑only state (e.g., modal visibility, theme).\n- **Animations** – Reanimated 2 for smooth transitions (map markers, modal presents).\n- **Offline** – Query persistence via `persistQueryClient` for limited offline browse.\n\n### Backend (API)\n- **Apps/api** – NestJS microservice (monolith for MVP).\n  - **src/**\n    - **main.ts** – Bootstrap.\n    - **app.module.ts** – Root module.\n    - **common/** – Guards, interceptors, pipes, exceptions, decorators.\n    - **auth/** – JWT strategy, Supabase integration, password reset flow.\n    - **users/** – CRUD, profile management.\n    - **businesses/** – Business CRUD, geo‑search (PostGIS), category filtering.\n    - **services/** – Service definitions tied to businesses.\n    - **appointments/** – Booking flow, slot computation, validation, conflict detection.\n    - **favorites/** – User‑business many‑to‑many.\n    - **reviews/** – Ratings, comments, business responses.\n    - **payments/** – Integration with Stripe/Paypal (webhook handling).\n    - **notifications/** – Email/SMS via Supabase Functions or external provider, triggered by BullMQ jobs.\n    - **jobs/** – BullMQ processors (email, payment confirmation, slot recomputation).\n    - **prisma/** – Prisma service wrapping PrismaClient.\n    - **generated/** – Prisma client output.\n    - **test/** – Unit, integration, and contract tests (Jest + SuperTest).\n- **API Style** – RESTful endpoints with versioning (`/api/v1/`); optional GraphQL via NestJS GraphQL module for admin dashboard.\n- **Validation** – class‑validator & DTOs.\n- **Caching** – Redis layer for frequent reads (business list, categories, slot caches). TTL‑based invalidation after write operations.\n- **Geo Search** – PostGIS `ST_DWithin` for radius queries; indexed on `location` geography column.\n\n### Infrastructure\n- **Docker Compose** – Spins up postgres (with PostGIS extension), redis, and api (NestJS) for local development.\n- **Supabase** – Provides auth (email/password, magic link) and object storage for user avatars, business photos; can be swapped with self‑hosted solution.\n- **EAS Build** – Handles OTA updates and production builds for iOS/Android.\n- **GitHub Actions** – Workflow for lint, test, build, and deploy (Docker images to ECR/GCR, Expo upload).\n- **Monitoring** – Sentry (frontend & backend), Loki/Prometheus via Docker sidecar (optional).\n\n## Folder Structure (Nx Monorepo)\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/          # Expo React Native app\n│   └── api/             # NestJS server\n├── libs/\n│   ├── ui/              # Shared design‑system components (React Native)\n│   ├── types/           # Shared TypeScript interfaces (e.g., User, Business, Appointment)\n│   ├── data-access/     # Thin wrapper around Prisma client (used by API)\n│   ├── utils/           # Date helpers, formatting, geolocation utils\n│   └── test-helpers/    # Custom matchers, test factories\n├── tools/\n│   └── scripts/         # DB migration, seed, dev utilities\n├── docker/\n│   └── compose.yml      # PostGIS‑enabled PostgreSQL, Redis, API\n├── .github/\n│   └── workflows/\n│       ├── ci.yml       # Lint, test, build\n│       └── cd.yml       # Deploy to staging/prod\n├── prisma/\n│   └── schema.prisma    # DB schema with PostGIS fields\n├── supabase/\n│   └── config.yaml      # Supabase project settings (optional)\n└── README.md\n```\n\n## Data Model (Prisma)\n```prisma\nmodel User {\n  id            String   @id @default(uuid())\n  email         String   @unique\n  name          String?\n  role          Role     @default(USER)\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n  appointments  Appointment[]\n  favorites     Favorite[]\n  reviews       Review[]\n}\n\nmodel Business {\n  id            String   @id @default(uuid())\n  name          String\n  description   String?\n  address       String\n  location      Geography @db.Geography(srid: 4326)   // PostGIS point\n  categoryId    String\n  category      Category @relation(fields: [categoryId], references: [id])\n  hours         BusinessHours[]\n  services      Service[]\n  reviews       Review[]\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\nmodel Category {\n  id   String @id @default(uuid())\n  name String\n  businesses Business[]\n}\n\nmodel Service {\n  id          String   @id @default(uuid())\n  name        String\n  durationInt Int      // minutes\n  price       Float\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  slots       Slot[]\n}\n\nmodel Appointment {\n  id        String   @id @default(uuid())\n  userId    String\n  user      User     @relation(fields: [userId], references: [id])\n  serviceId String\n  service   Service  @relation(fields: [serviceId], @id @default(uuid())\n  serviceId String\n  service   Service  @relation(fields: [serviceId], references: [id])\n  startTime DateTime\n  endTime   DateTime\n  isBooked  Boolean @default(false)\n}\n\nenum Role { USER ADMIN }\nenum AppointmentStatus { PENDING CONFIRMED CANCELLED COMPLETED }\n```\n\n## API Contracts (excerpt)\n- **GET /api/v1/businesses?lat=&lng=&radius=&category=&search=** – Returns paginated list with distance.\n- **POST /api/v1/auth/register** – {email, password, name} → {accessToken, refreshToken}.\n- **POST /api/v1/appointments** – {serviceId, startTime} → appointment.\n- **POST /api/v1/payments/webhook** – Stripe webhook handler.\n- **GET /api/v1/users/me/appointments** – User's upcoming/past appointments.\n\nAll responses follow `{ data, error?, meta? }` envelope.\n\n## State Management & Caching\n- **React Query** – `useBusinesses`, `useBusinessDetail`, `useAppointments`, etc., with `staleTime: 5 minutes`, `cacheTime: 10 minutes`.\n- **Query Invalidation** – After mutation (create appointment, favorite toggle) relevant queries are invalidated via `queryClient.invalidateQueries`.\n- **Redis** – Cache for business list per geo‑hash (TTL 2 min) and slot availability (TTL 30 sec). Implemented as a NestJS `CacheInterceptor`.\n\n## Real‑time & Notifications\n- **WebSocket** – Not required for MVP; notifications sent via push (Expo Notifications) triggered by BullMQ jobs after appointment creation or status change.\n- **BullMQ** – Jobs: `sendAppointmentConfirmation`, `sendReminder`, `processPayment`, `recalculateSlots`. Processed by separate workers (same API container, scalable).\n\n## Deployment\n- **Development** – `docker compose up` starts API, PostGIS, Redis; Expo dev client runs locally.\n- **Staging** – GitHub Actions builds Docker image, pushes to registry, deploys to Kubernetes (or ECS) with blue‑green.\n- **Production** – Same pipeline; Expo OTA updates via EAS Build; DB migrations run via Prisma migrate.\n\n## CI/CD\n```yaml\n# .github/workflows/ci.yml\nname: CI\non: [push, pull_request]\njobs:\n  lint-test:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v3\n      - uses: pnpm/action-setup@v2\n      - run: pnpm install --frozen-lockfile\n      - run: pnpm nx run-many --target=lint --all\n      - run: pnpm nx run-many --target=test --all\n  build:\n    needs: lint-test\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v3\n      - uses: pnpm/action-setup@v2\n      - run: pnpm install --frozen-lockfile\n      - run: pnpm nx run mobile:build --configuration=production\n      - run: pnpm nx run api:build --configuration=production\n```\n\n## Testing Strategy\n- **Unit** – Jest for utilities, React components, NestJS services.\n- **Integration** – SuperTest for API endpoints; React Native Testing Library with Jest for screens.\n- **E2E** – Detox for critical flows (login, search, booking).\n- **Contract** – Pact or manual schema validation to ensure frontend‑backend compatibility.\n\n## Security Considerations\n- Authentication via JWT (access token 15 min, refresh token 7 days) stored securely (SecureStore).\n- Passwords hashed with bcrypt.\n- Rate limiting on auth endpoints (NestJS Throttler).\n- Input validation and sanitization (class‑validator, Prisma).\n- CORS restricted to allowed origins.\n- Sensitive data (payment details) never stored; tokenized via Stripe.\n- Secrets managed via GitHub Actions secrets and Docker env files.\n\n## Scalability & Performance\n- Horizontal scaling of API containers behind a load balancer.\n- Read replicas for PostgreSQL for heavy geo‑search traffic.\n- Redis clustering for cache and job queue.\n- Indexes: PostGIS GIST on `location`, composite indexes on frequently queried fields (e.g., `businesses(categoryId, location)`).\n- Pagination with cursor‑based approach to avoid deep offsets.\n- Lazy loading of images via Expo `Image` component with caching.\n\n## Shared Types & Design System\n- `libs/types` contains DTO‑compatible interfaces (e.g., `BusinessDTO`, `AppointmentDTO`) generated from Prisma via `prisma-dto-generator` or hand‑written to ensure type safety across client and server.\n- `libs/ui` provides a theme‑able component library (Button, Input, Card, MapMarker) built with React Native Reanimated for smooth animations.\n\n## Conclusion\nThis architecture delivers a clean separation of concerns, leverages the power of Nx for monorepo efficiency, and uses proven technologies (Expo, NestJS, PostgreSQL/PostGIS, Redis) to satisfy the functional requirements while remaining maintainable, testable, and scalable.\n"
