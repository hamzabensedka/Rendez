  "content": "# Planity Clone Architecture\n\n## 1. Overview\nThe Planity Clone is a mobile-first application that connects users with local service businesses for booking appointments. The system follows a clean architecture with clear separation of concerns: mobile client (Expo/React Native), API server (NestJS with Prisma), admin portal, business owner portal, and background workers. Data is stored in PostgreSQL with PostGIS for location queries, and Redis is used for caching and pub/sub. The monorepo is managed with Nx and pnpm, enabling code sharing between mobile and server via shared types and design system. CI/CD is handled by GitHub Actions, with builds using EAS Build for mobile and Docker Compose for local development. Deployment targets include Supabase for hosted PostgreSQL (optional) and any container platform.\n\n## 2. Tech Stack\n- Mobile: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- Server: NestJS, TypeScript, Prisma ORM, PostgreSQL, PostGIS, Redis, BullMQ (for background jobs)\n- Tooling: Nx (monorepo), pnpm, Docker Compose, GitHub Actions, EAS Build, Jest\n- Optional Services: Supabase (hosted DB), Sentry (error tracking), etc.\n\n## 3. High-Level Architecture\n```\n[Mobile App] <-- HTTPS --> [API Gateway (NestJS)] <---> [PostgreSQL/PostGIS]\n                                   ^                ^\n                                   |                |\n                           [Redis Cache]   [Background Workers (BullMQ)]\n                                   |\n                           [Admin Portal (NestJS + React)]\n                           [Business Portal (NestJS + React)]\n```\n- Mobile app communicates exclusively with the NestJS API via REST or GraphQL (REST chosen for simplicity).\n- API handles authentication, business logic, validation, and delegates heavy computation to workers.\n- Workers process jobs such as slot computation, notifications, and periodic data sync.\n- Admin and Business portals are internal web apps built with NestJS serving SSR React (or Next.js) but can also be separate SPA; for simplicity they are NestJS applications serving static React builds.\n- Shared library (libs/shared) contains TypeScript interfaces, design system components (styled with React Native Web for web portals), and utility functions.\n- Data access layer uses Prisma; migrations managed via Nx workspace.\n\n## 4. Service Boundaries\n### 4.1 Mobile Client (apps/mobile)\n- Responsibilities: UI presentation, user interactions, local state, offline caching, deep linking via Expo Router.\n- Libraries: React Query for server state, Reanimated for gestures and animations.\n- Boundary: Does not contain business logic; all data mutations go through API calls.\n\n### 4.2 API Server (apps/api)\n- Responsibilities: Authentication (JWT), authorization, request validation, orchestrating domain services, exposing REST endpoints.\n- Uses Prisma for ORM, integrates with Redis for caching and rate limiting.\n- Delegates long-running tasks to BullMQ queues processed by workers.\n- Exposes OpenAPI spec for documentation.\n\n### 4.3 Background Workers (apps/worker)\n- Responsibilities: Processing BullMQ jobs: compute availability slots, send notifications (email/push/SMS), export reports, clean up stale data.\n- Shares same Prisma client and Redis connection as API.\n- No HTTP interface; only internal job processing.\n\n### 4.4 Admin Portal (apps/admin)\n- Responsibilities: CRUD operations on users, businesses, roles, view analytics, manage system settings.\n- Built with NestJS serving a React SPA (or Next.js) and uses same shared UI library.\n\n### 4.5 Business Owner Portal (apps/business)\n- Responsibilities: Business profile management, schedule setting, view and manage bookings, respond to reviews.\n- Similar tech stack to admin but with different routes and permissions.\n\n### 4.6 Shared Library (libs/shared)\n- Contains:\n  - TypeScript interfaces for DTOs, entities, API responses.\n  - Design system components (buttons, inputs, modals) built with React Native Web for cross-platform use.\n  - Utility functions (date helpers, geolocation, formatting).\n  - Constants (role enums, booking statuses).\n- Consumed by mobile, API (for validation), admin, business, and worker (for types).\n\n## 5. Data Model (Prisma Schema Highlights)\n```prisma\n// simplified\nmodel User {\n  id            String   @id @default(uuid())\n  email         String   @unique\n  passwordHash  String\n  role          Role     @default(CLIENT)\n  profile       Profile?\n  bookings      Booking[]\n  reviews       Review[]\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\nmodel Business {\n  id            String   @id @default(uuid())\n  name          String\n  description   String?\n  address       String\n  latitude      Float\n  longitude     Float\n  location      Point?   @db.Point\n  category      String\n  phone         String?\n  email         String?\n  schedule      ScheduleRelation?\n  services      Service[]\n  bookings      Booking[]\n  reviews       Review[]\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\nmodel Service {\n  id          String   @id @default(uuid())\n  name        String\n  durationInt Int      // minutes\n  price       Float\n  business    Business @relation(fields: [businessId], references: [id])\n  businessId  String\n  bookings    Booking[]\n}\n\nmodel Booking {\n  id            String   @id @default(uuid())\n  user          User     @relation(fields: [userId], references: [id])\n  business      Business @relation(fields: [businessId], references: [id])\n  service       Service  @relation(fields: [serviceId], references: [id])\n  startTime     DateTime\n  endTime       DateTime\n  status        BookingStatus\n  paymentId     String?\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\nmodel Review {\n  id          String   @id @default(uuid())\n  user        User     @relation(fields: [userId], references: [id])\n  business    Business @relation(fields: [businessId], references: [id])\n  rating      Int\n  comment     String?\n  createdAt   DateTime @default(now())\n}\n```\nEnums for Role, BookingStatus, etc. Indexes on location for PostGIS queries.\n\n## 7. Communication Protocols\n- Mobile ↔ API: HTTPS REST (JSON). Auth via Bearer JWT.\n- API ↔ Redis: Pub/Sub for real-time notifications (optional) and caching.\n- API ↔ Worker: BullMQ backed by Redis; jobs serialized as JSON.\n- Worker ↔ API: Reads/writes via Prisma; no direct HTTP.\n- Admin/Business Portals ↔ API: Same as mobile (they could also directly use Prisma if SSR, but we keep via API for consistency).\n- External Services: Payment gateway (Stripe) via API; email provider (SendGrid) via worker; push notifications via Expo push service (API calls to Expo).\n\n## 8. Deployment & DevOps\n### Local Development\n- `docker-compose.yml` spins up PostgreSQL (with PostGIS extension), Redis, and optionally Mailhog for email testing.\n- Nx run commands: `nx serve mobile`, `nx serve api`, `nx serve worker`, `nx serve admin`, `nx serve business`.\n- Hot reloading enabled for mobile via Expo dev client.\n\n### CI/CD (GitHub Actions)\n- On push to main:\n  - Lint, type-check, test (Jest) for all projects.\n  - Build Docker images for api, worker, admin, business.\n  - Push images to registry.\n  - Trigger EAS Build for mobile (iOS/Android) and upload to stores.\n  - Deploy images to Kubernetes or ECS via helm charts (not detailed here).\n- Secrets: DB credentials, Redis URL, JWT secret, payment keys stored in repo secrets.\n\n### Production Environment\n- Managed PostgreSQL (Supabase or self-hosted) with PostGIS enabled.\n- Redis Elasticache or similar.\n- API and workers run behind load balancer with autoscaling.\n- Mobile distributed via App Store/Play Store; updates via OTA (Expo) optionally.\n- Monitoring: Prometheus + Grafana, Loki for logs, Sentry for errors.\n\n## 9. Key Architectural Decisions\n- Clean Architecture: Separation of concerns between UI, application, domain, and infrastructure layers. Keep business logic in API/services, not in mobile.\n- Monorepo with Nx: Enforces shared types and design system, simplifies versioning, and allows atomic changes across client and server.\n- Expo Router: File‑based routing simplifies navigation and supports deep linking.\n- TanStack React Query: Handles server state, caching, background updates, and deduplication.\n- React Native Reanimated: Provides smooth gestures and animations for map interactions and booking flow.\n- Prisma + PostGIS: Strong ORM with spatial indexes for efficient location‑based queries.\n- BullMQ + Redis: Reliable background job processing with retry, prioritization, and monitoring.\n- Separate Worker App: Prevents API from being blocked by long‑running tasks; workers can be scaled independently.\n- Shared UI Library: Built with React Native Web so same components can be used in mobile and web portals, ensuring visual consistency.\n- Authentication: Stateless JWT with refresh token rotation stored in HTTP‑only cookies (or SecureStore on mobile) to mitigate XSS.\n- Testing: Unit tests with Jest for utilities, integration tests with Supertest for API endpoints, end‑to‑end tests with Detox for mobile critical flows.\n\n## 10. Future Extensions\n- GraphQL subscription layer for real‑time updates (e.g., booking changes).\n- Feature flags using LaunchDarkly.\n- AI‑driven recommendation engine for businesses.\n- Multi‑currency and tax calculation extensions.\n- Offline‑first capabilities with Expo SQLite and React Query persistence.\n\n## 11. Summary\nThis architecture provides a scalable, maintainable foundation for the Planity Clone, leveraging a modular monorepo, clean separation of concerns, and proven cloud‑native technologies to support rapid iteration and high availability.\n"
