Overview\nThe Planity Clone is a mobileвЂ‘first marketplace that lets users discover local service businesses, book appointments, and manage their schedules. Technology Stack\n- **Mobile**: Expo (SDK 50), React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated, ExpoвЂ‘SecureвЂ‘Store, ExpoвЂ‘Notifications.\n- **UI**: Shared design system built with React Native Paper / custom components, styled with StyleSheet and CSSвЂ‘inвЂ‘JS (twind).\n- **Backend**: NestJS (Node.js), Prisma ORM, PostgreSQL with PostGIS extension, Redis (caching & pub/sub), BullMQ for background jobs.\n- **DevOps**: Nx workspace, pnpm, Docker Compose for local dev, GitHub Actions for CI, EAS Build for OTA & store builds, Supabase for auth & storage (optional fallback), Jest + React Native Testing Library for unit/integration tests.\n- **Geospatial**: PostGIS for locationвЂ‘based queries; client uses ExpoвЂ‘Location and reactвЂ‘nativeвЂ‘maps.\n\n## 3. HighвЂ‘Level Architecture\n```\n[Mobile App] <--HTTPS/WS--> [API Gateway (NestJS)] <---> [PostgreSQL/PostGIS]\n                               |                     ^\n                               |                     |\n                               v                     |\n                            [Redis] <-> [BullMQ Workers]\n```\n- The mobile app communicates exclusively with the NestJS API via REST (JSON) and occasional WebSocket (via `@nestjs/websockets`) for realвЂ‘time updates (e.g., new booking notifications).\n- The API layer is split into **modules** corresponding to business domains (Auth, Business, Booking, Payment, Review, Admin, ProviderPortal).\n- Prisma generates a typeвЂ‘safe client; migrations are managed via `prisma migrate`.\n- Redis stores shortвЂ‘lived caches (search results, session tokens) and serves as the broker for BullMQ.\n- BullMQ workers run as separate NestJS services (or as `@nestjs/bull` processors) handling email/SMS notifications, payment webhook processing, and slot recomputation.\n- Docker Compose defines services: `api`, `db`, `redis`, `worker`.\n- In production, the API and workers run on containers (ECS/EKS or similar) behind an ALB; the mobile app fetches OTA updates via Expo EAS; static assets served from CDN.\n\n## 4. API Design (REST + WebSocket)\n- **Base URL**: `https://api.planityclone.com/v1`\n- **Versioning**: URL path (`/v1/...`); future versions via `/v2/`.\n- **Authentication**: Bearer JWT (`Authorization: Bearer <token>`); refresh token flow via `/auth/refresh`.\n- **Endpoints** (grouped by module):\n  - **Auth**: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/reset-request`, `POST /auth/reset-confirm`.\n  - **Business**: `GET /businesses` (search with query params `q`, `category`, `lat`, `lng`, `radius`), `GET /businesses/:id`, `POST /businesses` (provider only), `PUT /businesses/:id`, `DELETE /businesses/:id`.\n  - **Service**: nested under business: `GET /businesses/:businessId/services`, `POST /businesses/:businessId/services`.\n  - **Booking**: `GET /businesses/:businessId/slots?date=YYYY-MM-DD&serviceId=:id` (returns array of free slots), `POST /appointments` (create), `PATCH /appointments/:id` (reschedule/cancel), `GET /users/me/appointments`.\n  - **Payment**: `POST /payments/intents` (create Stripe intent), `POST /payments/webhook` (Stripe).\n  - **Review**: `GET /businesses/:businessId/reviews`, `POST /reviews`, `PATCH /reviews/:id` (business response).\n  - **Provider**: `GET /provider/dashboard`, `GET /provider/bookings`, `POST /provider/businesses`, etc.\n  - **Admin**: `GET /admin/users`, `GET /admin/businesses`, `PATCH /admin/users/:id/role`, etc.\n- **WebSocket**: Gateway at `/ws` with topics: `booking:${userId}`, `review:${businessId}`, `notification:${userId}`; uses `@nestjs/websockets` (Socket.io) for realвЂ‘time updates.\n- **Response Format**: `{ success: boolean, data?: any, error?: {code:string, message:string} }`.\n- **Validation**: DTOs with `class-validator`; global pipe for validation errors.\n- **Rate Limiting**: `@nestjs/throttler` (default 100 req/15min per IP).\n\n## 8. State Management & Data Fetching\n- **React Query** handles server state: queries for businesses, slots, appointments, reviews; mutations for create/update/delete.\n- Query keys are namespaced (e.g., `['businesses', {lat, lng, radius}]`).\n- Optimistic updates for booking creation/cancellation.\n- **Reanimated** used for gestureвЂ‘based UI (e.g., swipe to cancel appointment, animated map markers).\n- **AsyncStorage/SecureStore** for persisting refresh token and lightweight preferences.\n- **ExpoвЂ‘Notifications** for push notifications; token stored on user document.\n\n## 9. Caching & Pub/Sub\n- **Redis**: \n  - Cache layer for expensive search results (TTL 5вЂЇmin).\n  - Session store for refresh tokens (optional).\n  - Broker for BullMQ.\n- **Invalidation**: After business update, related search keys are purged via Redis `SCAN` pattern.\n\n## 10. Background Jobs (BullMQ)\n- Queues defined in `services/api/src/queues/`:\n  - `email`: sends confirmation, reminder, and promotional emails via SendGrid/SMTP.\n  - `slot-recalc`: runs nightly or after each booking to recompute and cache available slots for a business.\n  - `payment-webhook`: processes Stripe webhooks, updates Payment and Appointment status.\n- Workers are `@nestjs/bull` processors; concurrency tuned per queue.\n- Failed jobs move to a deadвЂ‘letter queue with alerting via email/SMS.\n\n## 11. DevOps & CI/CD\n- **Local**: `docker compose up` spins up `api`, `db`, `redis`.\n- **CI (GitHub Actions)**: \n  - Lint (ESLint, Prettier).\n  - Type checking (`tsc --noEmit`).\n  - Unit tests (Jest) for both mobile and API.\n  - Build Docker images and push to registry (on tag).\n- **EAS Build**: Triggered on git tag; produces OTA updates and store binaries (iOS/Android).\n- **Production**: \n  - API & workers deployed as Docker containers to a managed K8s cluster (or ECS Fargate).\n  - Autoscaling based on CPU/RAM and custom metrics (queue length).\n  - PostgreSQL managed service (e.g., Amazon RDS Aurora with PostGIS).\n  - Redis managed service (Elasticache).\n  - CDN (CloudFront) for static assets served from Expo build.\n- **Monitoring**: Prometheus + Grafana for API metrics; Sentry for error tracking; ExpoвЂ™s builtвЂ‘in analytics for app performance.\n\n## 12.
