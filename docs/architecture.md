Overview\n\nThis document describes the system architecture for the Planity Clone, a marketplace platform connecting customers with local service businesses. The architecture prioritizes clean separation of concerns, maintainability, scalability, and a great developer experience. The system is built using a monorepo managed by Nx and pnpm, with a React Native (Expo) mobile frontend and a NestJS backend.\n\n## 2. Technology Stack\n\n- **Frontend:** Expo (React Native), TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Backend:** NestJS, TypeScript, Prisma ORM, PostgreSQL, PostGIS, Redis\n- **Infrastructure:** Docker Compose, GitHub Actions, Supabase (Auth + Storage), EAS Build\n- **Testing:** Jest (unit + integration), React Native Testing Library\n- **Monorepo:** Nx, pnpm\n\n## 3. Folder Structure\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   ├── components/            # Shared UI components\n│   │   ├── features/              # Feature modules (auth, booking, search)\n│   │   ├── hooks/                 # Custom hooks\n│   │   ├── services/              # API client, query hooks\n│   │   ├── utils/                 # Helpers\n│   │   └── app.json\n│   └── api/                       # NestJS backend\n│       ├── src/\n│       │   ├── modules/           # Feature modules (auth, users, businesses, bookings)\n│       │   ├── common/            # Shared guards, interceptors, filters\n│       │   ├── config/            # Configuration files\n│       │   ├── database/          # Prisma schema, migrations, seeds\n│       │   └── main.ts\n│       ├── prisma/\n│       │   ├── schema.prisma\n│       │   └── migrations/\n│       └── test/\n├── packages/\n│   ├── shared-types/              # Shared TypeScript types/interfaces\n│   ├── ui-kit/                    # Reusable UI components (buttons, inputs, cards)\n│   └── utils/                     # Common utilities (date formatting, validation)\n├── tools/\n│   ├── generators/                # Nx generators for scaffolding\n│   └── scripts/                   # Build, deploy, seed scripts\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n├── tsconfig.base.json\n├── .github/\n│   └── workflows/\n│       ├── ci.yml\n│       └── deploy.yml\n└── docs/\n    └── architecture.md\n```\n\n## 4. Service Boundaries & Modules\n\n### 4.1 Backend (NestJS) Modules\n\n- **AuthModule** – Handles registration, login, OAuth, JWT, refresh tokens, biometric tokens, session management.\n- **UsersModule** – User profiles, addresses, preferences, account linking.\n- **BusinessesModule** – Business CRUD, categories, services, staff, availability, photos.\n- **BookingsModule** – Booking flow, slot management (pessimistic locking with Redis), cancellation, rescheduling, reminders.\n- **SearchModule** – Full-text search, geo-spatial queries (PostGIS), filtering, auto-complete.\n- **ReviewsModule** – Review CRUD, rating aggregation, owner responses.\n- **NotificationsModule** – Push notifications (Expo Push API), email (SendGrid), SMS (Twilio).\n- **AdminModule** – Admin dashboard, user management, platform analytics.\n- **PaymentsModule** – Payment processing (Stripe), pay-at-venue option, refunds.\n\n### 4.2 Frontend (Mobile) Feature Modules\n\n- **auth/** – Login, register, password reset, biometric, session handling.\n- **browse/** – Guest browse, search, map, category listing.\n- **business/** – Business detail, services, staff, reviews, gallery.\n- **booking/** – Multi-step booking flow, slot selection, add-ons, confirmation.\n- **appointments/** – Upcoming, past, cancelled appointments, reschedule, cancel.\n- **profile/** – User profile, settings, favorites, payment methods.\n- **admin/** – Admin screens (if applicable).\n\n## 5. Data Flow & Key Interactions\n\n### 5.1 Authentication Flow\n1. User enters credentials on mobile → Expo app sends request to `/auth/login`.\n2. NestJS AuthModule validates credentials, returns access token (15 min) and refresh token (7 days).\n3. Mobile stores tokens securely (expo-secure-store).\n4. TanStack React Query uses Axios interceptor to attach token and handle refresh.\n5. Biometric login: after first password login, device stores a biometric token (separate short-lived token) for quick re-auth.\n\n### 5.2 Booking Flow\n1. User selects service → mobile fetches available slots via `/bookings/slots?businessId=X&serviceId=Y&date=Z`.\n2. Backend queries PostgreSQL with PostGIS for business location and Prisma for availability.\n3. Slots are cached in Redis (TTL 2 minutes) to reduce DB load.\n4. User selects slot → mobile calls `/bookings/hold` → backend acquires Redis lock (10 min TTL) on slot.\n5. User completes booking → mobile calls `/bookings/confirm` → backend creates booking in DB, releases lock, sends confirmation push/email.\n6. If hold expires, Redis key is deleted, slot becomes available again.\n\n### 5.3 Search Flow\n1. User types in search bar → mobile debounces (300ms) then calls `/search?q=...&lat=...&lng=...&filters=...`.\n2. Backend SearchModule uses PostgreSQL full-text search (tsvector) and PostGIS for distance ordering.\n3. Results are cached in Redis (TTL 5 minutes) for popular queries.\n4. Auto-complete uses a separate endpoint with a smaller result set and shorter cache.\n\n## 6. Database Schema (Prisma)\n\nKey models:\n\n- **User** – id, email, passwordHash, name, phone, avatarUrl, createdAt, updatedAt.\n- **Business** – id, ownerId, name, description, categoryId, location (PostGIS point), address, phone, website, photos[], hours (JSON), rating, reviewCount.\n- **Category** – id, name, icon, parentId (self-referential).\n- **Service** – id, businessId, categoryId, name, description, duration, price, addOns[] (JSON).\n- **Staff** – id, businessId, userId (optional), name, photo, specialties[], rating.\n- **Availability** – id, staffId (nullable), dayOfWeek, startTime, endTime, recurrence (JSON).\n- **Booking** – id, userId, businessId, serviceId, staffId (nullable), startTime, endTime, status (pending/confirmed/cancelled/completed), totalPrice, addOns[], specialRequests, createdAt.\n- **Review** – id, bookingId, userId, businessId, rating, text, photos[], createdAt.\n- **Session** – id, userId, refreshToken, deviceInfo, expiresAt, createdAt.\n- **Notification** – id, userId, type, title, body, data (JSON), read, createdAt.\n\nPostGIS extension enables spatial queries on Business.location (geography type).\n\n## 7. Caching Strategy (Redis)\n\n- **Slot availability** – Key: `slots:{businessId}:{date}`, TTL: 2 min.\n- **Search results** – Key: `search:{queryHash}`, TTL: 5 min.\n- **Business detail** – Key: `business:{id}`, TTL: 10 min.\n- **User session** – Key: `session:{refreshToken}`, TTL: 7 days.\n- **Rate limiting** – Key: `ratelimit:{ip}:{endpoint}`, TTL: 15 min.\n- **Booking holds** – Key: `hold:{slotId}`, TTL: 10 min (pessimistic lock).\n\n## 8. API Design (RESTful)\n\n- `POST /auth/register` – Register with email/password.\n- `POST /auth/login` – Login, returns tokens.\n- `POST /auth/refresh` – Refresh access token.\n- `POST /auth/oauth/{provider}` – OAuth login.\n- `GET /businesses` – List businesses with filters, pagination.\n- `GET /businesses/:id` – Business detail.\n- `GET /businesses/:id/services` – Services list.\n- `GET /businesses/:id/staff` – Staff list.\n- `GET /businesses/:id/reviews` – Reviews with pagination.\n- `GET /search` – Full-text search with geo and filters.\n- `GET /search/autocomplete` – Auto-complete suggestions.\n- `GET /bookings/slots` – Available slots for a service/date.\n- `POST /bookings/hold` – Hold a slot (10 min).\n- `POST /bookings` – Confirm booking.\n- `GET /bookings/my` – User's appointments.\n- `PATCH /bookings/:id/cancel` – Cancel booking.\n- `PATCH /bookings/:id/reschedule` – Reschedule booking.\n- `POST /reviews` – Create review.\n- `GET /categories` – Category tree.\n- `POST /notifications/register` – Register push token.\n\n## 9. Security Considerations\n\n- JWT tokens with short expiry (15 min), refresh token rotation.\n- Rate limiting on auth endpoints (5 attempts/15 min).\n- Input validation (class-validator) on all endpoints.\n- SQL injection prevention via Prisma parameterized queries.\n- CORS restricted to mobile app origin.\n- HTTPS enforced.\n- Biometric tokens stored securely on device.\n- Booking holds use Redis with TTL to prevent deadlocks.\n\n## 10. Testing Strategy\n\n- **Unit tests** – Jest for individual functions/services.\n- **Integration tests** – NestJS testing module with test database (PostgreSQL container).\n- **E2E tests** – For critical flows (auth, booking) using supertest.\n- **Mobile tests** – React Native Testing Library for components, Detox for E2E.\n- **CI** – GitHub Actions runs lint, type-check, unit tests, integration tests on every PR.\n\n## 11. Deployment & Infrastructure\n\n- **Docker Compose** – Local development with PostgreSQL, Redis, and API.\n- **Supabase** – Managed Auth (optional), Storage for images.\n- **EAS Build** – Build and submit mobile app to App Store/Google Play.\n- **GitHub Actions** – CI/CD pipeline: lint → test → build → deploy.\n- **Environment variables** – Managed via .env files (local) and GitHub secrets (CI).\n\n## 12. Performance Targets\n\n- Search results <500ms (95th percentile).\n- Business detail page load <2s.\n- Booking confirmation within 5 seconds.\n- Slot availability query <200ms.\n- Map clustering handles 500+ markers.\n\n## 13. Monitoring & Observability\n\n- **Logging** – NestJS Logger, structured JSON logs.\n- **Metrics** – Prometheus (via @nestjs/prometheus) for request count, latency, error rate.\n- **Health checks** – `/health` endpoint for Docker health check.\n- **Error tracking** – Sentry for backend and mobile.\n\n## 14. Conclusion\n\nThis architecture provides a solid foundation for the Planity Clone, ensuring clean separation of concerns, scalability, and maintainability. The monorepo structure with Nx enables efficient code sharing and fast builds.
