The architecture follows clean architecture principles, with clear separation of concerns, modular monolith backend, and a mobile-first frontend built with Expo and React Native.\n\n## 2. Service Boundaries\n\n### 4.1 Mobile App (Frontend)\n\nThe mobile app is structured around feature modules, each containing screens, components, hooks, and services.\n\n**Key Modules:**\n- **Authentication**: Login, register, password reset, biometric auth\n- **Explore**: Search, filters, map view, business list\n- **Business Detail**: Business info, services, staff, reviews\n- **Booking**: Service selection, staff selection, date/time picker, confirmation\n- **Appointments**: Upcoming and past appointments management\n- **Profile**: User profile, settings, favorites\n- **Notifications**: Push notification handling and display\n\n### 4.2 Backend (NestJS Modules)\n\nEach backend module is a self-contained NestJS module with its own controller, service, DTOs, and Prisma integration.\n\n| Module | Responsibilities |\n|--------|------------------|\n| Auth | JWT generation, OAuth, password hashing, session management |\n| Users | CRUD, profile management, soft delete |\n| Businesses | CRUD, geolocation, hours management, verification |\n| Services | CRUD, categorization, pricing, duration |\n| Staff | CRUD, schedules, specialties, availability |\n| Bookings | Create, update, cancel, slot locking, conflict detection |\n| Reviews | CRUD, rating aggregation, moderation |\n| Categories | Hierarchical taxonomy management |\n| Search | Full-text search, autocomplete, geospatial queries |\n| Notifications | Push notification dispatch (FCM/APNs) |\n| Admin | Dashboard analytics, user management, dispute resolution |\n\n### 4.3 Shared Libraries\n\n- **shared-types**: TypeScript interfaces shared between mobile and backend (e.g., User, Business, Booking)\n- **api-client**: Auto-generated API client from OpenAPI spec using `openapi-typescript`\n- **ui-kit**: Reusable React Native components (Button, Input, Card, Modal, etc.)\n\n## 5. API Design\n\n### 7.1 RESTful Endpoints\n\n| Method | Endpoint | Description |\n|--------|----------|-------------|\n| POST | /auth/register | Register new user |\n| POST | /auth/login | Login with email/password |\n| POST | /auth/refresh | Refresh access token |\n| POST | /auth/logout | Invalidate refresh token |\n| POST | /auth/social | Social login (Google/Apple) |\n| POST | /auth/forgot-password | Send reset email |\n| POST | /auth/reset-password | Reset password with token |\n| GET | /users/me | Get current user profile |\n| PATCH | /users/me | Update profile |\n| GET | /businesses | List businesses (with filters) |\n| GET | /businesses/:id | Get business detail |\n| POST | /businesses | Create business (owner) |\n| PATCH | /businesses/:id | Update business |\n| GET | /businesses/:id/services | List services |\n| POST | /businesses/:id/services | Create service |\n| GET | /businesses/:id/staff | List staff |\n| GET | /businesses/:id/reviews | List reviews |\n| GET | /services/:id/slots | Get available slots |\n| POST | /bookings | Create booking |\n| GET | /bookings | List user bookings |\n| GET | /bookings/:id | Get booking detail |\n| PATCH | /bookings/:id | Update booking (cancel) |\n| GET | /search | Full-text search |\n| GET | /categories | List categories |\n| POST | /reviews | Create review |\n| POST | /favorites | Add favorite |\n| DELETE | /favorites/:businessId | Remove favorite |\n\n### 7.2 Response Format\n\nAll API responses follow a consistent envelope:\n\n```json\n{\n  \"success\": true,\n  \"data\": { ... },\n  \"meta\": {\n    \"page\": 1,\n    \"limit\": 20,\n    \"total\": 100\n  },\n  \"error\": null\n}\n```\n\nError responses:\n\n```json\n{\n  \"success\": false,\n  \"data\": null,\n  \"error\": {\n    \"code\": \"VALIDATION_ERROR\",\n    \"message\": \"Invalid email format\",\n    \"details\": [\n      { \"field\": \"email\", \"message\": \"must be a valid email\" }\n    ]\n  }\n}\n```\n\n## 8. Caching Strategy (Redis)\n\n| Cache Key | TTL | Purpose |\n|-----------|-----|---------|\n| `slot:{staffId}:{date}` | 5 min | Available slots for a staff member on a date |\n| `business:{id}` | 10 min | Business detail (invalidated on update) |\n| `search:{query}` | 2 min | Search results |\n| `autocomplete:{prefix}` | 1 min | Autocomplete suggestions |\n| `refresh:{userId}` | 7 days | Refresh token storage |\n| `rate_limit:{ip}` | 1 min | Rate limiting |\n| `booking_lock:{slotId}` | 10 min | Pessimistic lock during booking flow |\n\n## 9. Security Considerations\n\n- **JWT Tokens**: Access tokens stored in `expo-secure-store` (Keychain/Keystore)\n- **Password Hashing**: bcrypt with 12 rounds\n- **Rate Limiting**: 100 requests/minute per IP, 10 auth attempts/minute\n- **SQL Injection**: Prevented by Prisma parameterized queries\n- **XSS**: React Native handles via JS engine; sanitize HTML in reviews\n- **CORS**: Restricted to mobile app origin (custom scheme)\n- **API Keys**: Business owner endpoints require role-based access control (RBAC)\n- **Data Encryption**: HTTPS enforced; sensitive fields encrypted at rest\n\n## 10. Performance Targets\n\n| Metric | Target |\n|--------|--------|\n| API response time (p95) | < 300ms |\n| Search results | < 500ms |\n| Autocomplete | < 150ms |\n| Booking confirmation | < 1s |\n| Map load with 100 pins | < 2s |\n| App cold start | < 3s |\n| Image load (lazy) | < 1s per image |\n\n## 11. Run database migrations: `pnpm nx run backend:prisma:migrate`\n4. Start backend: `pnpm nx run backend:serve`\n5. Start mobile app: `pnpm nx run mobile:start`\n\n### 11.2 Testing\n\n- **Unit Tests**: Jest for backend services and mobile utilities\n- **Integration Tests**: Supertest for API endpoints\n- **E2E Tests**: Detox for mobile (future)\n- **CI Pipeline**: GitHub Actions runs lint, type-check, unit tests on every PR\n\n### 11.3 Deployment\n\n- **Mobile**: EAS Build for OTA updates and app store submissions\n- **Backend**: Docker container deployed to cloud (AWS ECS / GCP Cloud Run)\n- **Database**: Supabase managed PostgreSQL with PostGIS extension\n- **Redis**: Upstash or ElastiCache\n\n## 12. Monitoring and Observability\n\n- **Logging**: Winston (backend), `console.log` replaced with structured logging\n- **Metrics**: Prometheus + Grafana for API latency, error rates, booking volume\n- **Error Tracking**: Sentry for both backend and mobile\n- **Health Checks**: `/health` endpoint returns DB and Redis connectivity\n\n## 13.
