High-Level Overview\nThe Planity Clone is a full‑stack mobile‑first application consisting of:\n- **Mobile Client**: Expo (React Native) app using Expo Router, TanStack React Query, React Native Reanimated, TypeScript.\n- **API Server**: NestJS backend with Prisma ORM, PostgreSQL (with PostGIS extension), Redis for caching/pubsub, BullMQ for background jobs.\n- **Infrastructure**: Docker Compose for local development, GitHub Actions for CI/CD, EAS Build for Expo builds, Supabase for optional hosted DB/auth (can be swapped), Nx monorepo with pnpm workspace.\n- **Shared Libraries**: Type definitions, UI components, design system, utils, API clients.\n\n## 2. Monorepo Structure (Nx)\n```\nplanity-clone/\n├─ apps/\n│   ├─ mobile/          # Expo React Native app\n│   └─ api/             # NestJS server\n├─ libs/\n│   ├─ ui/              # Shared React Native components (design system)\n│   ├─ types/           # Shared TypeScript interfaces (DTOs, enums)\n│   ├─ api-client/      # Generated or hand‑written React Query hooks\n│   ├─ utils/           # Cross‑platform utilities (date, geo, auth helpers)\n│   ├─ config/          # Nx workspace config, eslint, prettier\n│   └─ test-helpers/    # Jest testing utilities\n├─ tools/\n│   ├─ generators/      # Custom Nx generators (component, service)\n│   └─ scripts/         # Dev ops scripts\n├─ docker/\n│   └─ compose.yml      # Docker Compose for dev (api, db, redis)\n├─ .github/\n│   └─ workflows/       # GitHub Actions CI/CD\n├─ nx.json\n├─ package.json         # root pnpm workspace\n└─ tsconfig.base.json\n```\n\n### 2.1 Apps\n- **mobile**: Entry point `apps/mobile/src/main.tsx`. Uses Expo Router for file‑based navigation. State management via React Query. Animations via Reanimated.\n- **api**: NestJS entry `apps/api/src/main.ts`. Modular structure: core modules (Auth, Business, Booking, Payment, Notification, Admin, ProviderPortal). Each module contains controllers, services, DTOs, entities, guards.\n\n### 2.2 Libs\n- **ui**: Design system built with React Native Reanimated gestures, styled with StyleSheet or nativewind. Exported as a local library for mobile.\n- **types**: Shared interfaces (e.g., `User`, `Business`, `Service`, `Appointment`, `Review`). Used by both mobile (via api-client) and NestJS (via Prisma‑generated types transformed).\n- **api-client**: Wrapper around `axios` or `fetch` that creates React Query hooks (`useBusinesses`, `useBookings`, etc.). Generated using NestJS OpenAPI plugin or manually.\n- **utils**: Geo helpers (haversine, PostGIS functions), date/time slot computation, auth token storage (expo-secure-store).\n- **config**: ESLint, Prettier, Jest, TypeScript configs shared across workspace.\n- **test-helpers**: Custom matchers, mock servers for React Native testing.\n\n## 3. Service Boundaries (Backend)\n\n| Module        | Responsibility                               | Key Entities                     | External Calls                |\n|---------------|----------------------------------------------|----------------------------------|------------------------------|\n| Auth          | User registration, login, social OAuth, password reset, JWT issuance | User, AuthToken                  | Redis (refresh token store), Email provider (via BullMQ) |\n| Business      | CRUD for businesses, location indexing, categories | Business, Category, BusinessHours | PostGIS (geo queries), Redis (cache listings) |\n| Service       | Service definitions tied to businesses       | Service, ServiceCategory         | – |\n| Availability  | Slot computation, schedule management        | Schedule, TimeSlot, Booking      | BullMQ (recompute on change) |\n| Booking       | Create, reschedule, cancel appointments      | Booking, PaymentTransaction      | Payment gateway (Stripe/PayPal), Notification service |\n| Payment       | Secure transaction handling                  | PaymentTransaction, Refund       | Payment gateway webhook |\n| Notification  | Send emails, push notifications              | NotificationLog                  | Expo Push Notifications, SMTP/BullMQ |\n| ProviderPortal| Business owner dashboard (listings, schedule) | Same as Business + Schedule      | – |\n| Admin         | Super‑user management, analytics             | User, Business, Booking          | – |\n| Core          | Shared guards, interceptors, validation pipes, exception filters | – | – |\n\nEach module follows NestJS best practices: `controllers` (HTTP), `services` (business logic), `repositories` (Prisma), `DTOs` (validation via class‑validator), `guards` (JWT, roles). Modules are lazy‑loaded and can be independently scaled.\n\n## 4. Data Layer\n- **Primary DB**: PostgreSQL with PostGIS extension for geographic queries (nearby businesses, distance sorting).\n- **ORM**: Prisma Client (generated from `prisma/schema.prisma`). Schema includes indexes on `location` (PostGIS `GEOGRAPHY(Point)`), composite indexes for availability lookups.\n- **Cache**: Redis used for:\n  - Session/refresh token storage.\n  - Caching business list queries (TTL 5 min).\n  - PubSub for triggering availability recomputation.\n- **Background Jobs**: BullMQ (Redis‑backed) for:\n  - Sending notifications (email/push).\n  - Recalculating available slots after schedule changes.\n  - Processing payment webhooks.\n  - Cleanup tasks (expired tokens, old logs).\n\n## 5. Mobile Client Architecture\n\n### 5.1 Navigation\n- Expo Router provides file‑based routing (`apps/mobile/app/(tabs)/...`). Auth guard via higher‑order component that checks `expo-secure-store` token.\n\n### 5.2 State Management\n- React Query (`@tanstack/react-query`) handles server state:\n  - Query keys: `['businesses', filters]`, `['business', id]`, `['bookings']`.\n  - Mutations for create/update/delete operations with optimistic updates.\n  - Background refetch intervals for availability.\n\n### 5.3 Animations & Gestures\n- React Native Reanimated 2 used for:\n  - Animated tab bar.\n  - Draggable map markers.\n  - Modal transitions.\n\n### 5.4 Design System\n- `libs/ui` exports components like `Button`, `Input`, `Card`, `MapMarker`, `Avatar`.\n- Theming via `ThemeContext` (light/dark) using `react-native-paper` or custom.\n\n### 5.5 Device Features\n- Expo Location for geolocation.\n- Expo Notifications for push.\n- Expo SecureStore for JWT.\n- Expo ImagePicker for profile pictures.\n\n### 5.6 Testing\n- Jest with `@testing-library/react-native`.\n- Mock React Query using `MockedProvider`.\n- Reanimated worklet tests using `react-native-reanimated/mock`.\n\n## 6. DevOps & CI/CD\n\n### 6.1 Local Development\n- `docker compose up -d` starts PostgreSQL (with PostGIS), Redis, and optionally MailHog.\n- Nx run commands:\n  - `nx serve mobile` (Expo dev client).\n  - `nx serve api` (NestJS watch mode).\n- Environment variables stored in `.env` files (not committed).\n\n### 6.2 Build & Release\n- **Mobile**: EAS Build configured via `eas.json`. Profiles: `development`, `preview`, `production`.\n- **API**: Dockerfile in `apps/api/Dockerfile`. CI builds image and pushes to registry (GHCR or Docker Hub).\n- **GitHub Actions** workflow (`.github/workflows/ci.yml`):\n  - Install pnpm, restore Nx cache.\n  - Run lint (`nx lint`).\n  - Run tests (`nx test mobile api`).\n  - Build mobile (EAS) and API Docker image.\n  - Deploy to staging (e.g., AWS ECS/Fargate or Fly.io) on push to `main`.\n\n### 6.3 Environment Variables\n- Shared via `nx` workspace `env/` folder or using `dotenv-cli`.\n- Secrets managed in GitHub Secrets for CI/EAS.\n\n### 6.4 Monitoring & Logging\n- API: NestJS logger integrated with Winston, logs shipped to stdout (captured by Docker).\n- Mobile: Expo logs + Sentry expo-sentry SDK for error tracking.\n- Metrics: Optional Prometheus exporter in NestJS.\n\n## 7. Security Considerations\n- Authentication: JWT (access token 15 min, refresh token 7 days) stored SecureStore; refresh token hashed in Redis.\n- Passwords: bcrypt hash.\n- CORS: Restrict to Expo origins.\n- Rate limiting: NestJS `ThrottlerGuard` per route.\n- Input validation: class-validator + class-transformer.\n- SQL injection: Prisma parameterized queries.\n- Data privacy: GDPR‑compliant deletion endpoints.\n- Payment: PCI‑DSS compliance via Stripe; no card data touches our servers.\n\n## 8. Scalability & Performance\n- **Read scaling**: Redis cache for business lists and popular queries; CDN for static assets (Expo web build).\n- **Write scaling**: NestJS microservices‑ready; each module can be extracted to separate service if needed.\n- **Geo queries**: PostGIS `ST_DWithin` with index on `location`.\n- **Slot computation**: Precompute availability slots nightly; incremental updates via BullMQ on schedule changes.\n- **Horizontal scaling**: Deploy API behind load balancer; sticky sessions not required due to JWT/Redis statelessness.\n\n## 9. Future Extensions\n- Web admin dashboard using Next.js (share `libs/types`, `libs/api-client`).\n- GraphQL layer (NestJS GraphQL) for flexible mobile queries.\n- Multi‑language support via i18next.\n- AI‑based recommendation service (separate Python service calling same DB).\n\n---\n*End of Architecture Document.*\n",
