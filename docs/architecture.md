  "content": "# Planity Clone System Architecture\n\n## 1. Overview\n\nPlanity Clone is a mobile-first appointment booking platform built with a modern cross-platform stack (Expo/React Native) and a robust NestJS backend. It connects customers with beauty and wellness businesses. The system supports guest browsing, customer bookings, business owner management, and an admin oversight dashboard. This document outlines the complete architecture, service boundaries, component design, and infrastructure decisions.\n\n## 2. Architecture Principles\n\n- **Separation of Concerns** ‚Äì Mobile UI, API logic, and data layers are strictly separated. Services are modular and independently testable.\n- **Performance First** ‚Äì Caching (React Query, Redis), background jobs (BullMQ), and optimized geospatial queries (PostGIS) ensure sub-300ms response times.\n- **Resilience & Atomicity** ‚Äì Booking flows use transactional writes and temporary slot holds to prevent double booking.\n- **Developer Experience** ‚Äì Monorepo with Nx and pnpm enables shared code, consistent tooling, and fast builds.\n- **Offline-Ready** ‚Äì The mobile app gracefully handles connectivity loss with stale-while-revalidate patterns.\n\n## 3. High-Level System Architecture\n\nThe system consists of four logical tiers:\n\n1. **Mobile Client** (Expo/React Native) ‚Äì Customer and business owner experiences.\n2. **Admin Web Client** (React) ‚Äì Internal dashboard for platform administrators.\n3. **API Gateway / BFF** (NestJS) ‚Äì REST API serving business logic, authentication, and real-time slot holds.\n4. **Infrastructure** ‚Äì PostgreSQL (with PostGIS), Redis, Supabase Auth & Storage, and BullMQ workers.\n\nData flows:\n\n- Mobile app authenticates via Supabase Auth and receives a JWT.\n- All API calls pass through the NestJS server, which validates the Supabase JWT and enforces role-based access.\n- Geospatial search queries use PostGIS extensions; Redis caches hot data and manages booking slot holds.\n- Background jobs (BullMQ) handle slot computation, reminders, and housekeeping.\n- Supabase Storage hosts user-uploaded images (business photos, avatars).\n\n## 4. Technology Stack\n\n| Layer | Technologies |\n|-------|--------------|\n| Mobile | Expo SDK 50, React Native, TypeScript, Expo Router (file-based routing) |\n| State & Data | TanStack React Query, React Context (auth), Zustand (local UI state) |\n| Animations | React Native Reanimated |\n| Backend | NestJS, TypeScript, Prisma ORM, PostgreSQL, PostGIS |\n| Caching & Queues | Redis, BullMQ |\n| Auth & Storage | Supabase (Auth, Storage) |\n| Monorepo | Nx, pnpm workspaces |\n| Testing | Jest, React Native Testing Library, Supertest (API) |\n| CI/CD | GitHub Actions, EAS Build (mobile), Docker Compose (local dev) |\n\n## 5. Monorepo Structure (Nx + pnpm)\n\nThe repository is an Nx integrated monorepo with pnpm as the package manager. Applications share code via libraries, enforcing module boundaries.\n\n### Top-Level Folder Scaffold\n\n```\nplanity/\n‚îú‚îÄ‚îÄ apps/\n‚îÇ   ‚îú‚îÄ‚îÄ mobile/                 # Expo app\n‚îÇ   ‚îú‚îÄ‚îÄ api/                    # NestJS backend\n‚îÇ   ‚îî‚îÄ‚îÄ admin-web/              # Admin dashboard (React + Vite)\n‚îú‚îÄ‚îÄ libs/\n‚îÇ   ‚îú‚îÄ‚îÄ shared-types/           # TypeScript interfaces & enums\n‚îÇ   ‚îú‚îÄ‚îÄ ui-kit/                 # Shared React Native components & design tokens\n‚îÇ   ‚îú‚îÄ‚îÄ utils/                  # Date helpers, formatters, etc.\n‚îÇ   ‚îî‚îÄ‚îÄ api-client/             # Auto-generated API client (OpenAPI)\n‚îú‚îÄ‚îÄ tools/                      # Nx generators, scripts\n‚îú‚îÄ‚îÄ docker-compose.yml          # Local infrastructure\n‚îú‚îÄ‚îÄ nx.json\n‚îú‚îÄ‚îÄ package.json\n‚îú‚îÄ‚îÄ pnpm-workspace.yaml\n‚îî‚îÄ‚îÄ jest.config.ts              # Root Jest config\n```\n\n## 6. Mobile Application Architecture\n\n### 6.1 Project Structure (`apps/mobile`)\n\n```\nsrc/\n‚îú‚îÄ‚îÄ app/                        # Expo Router file-based routes\n‚îÇ   ‚îú‚îÄ‚îÄ (auth)/                 # Auth screens (sign-in, sign-up, reset)\n‚îÇ   ‚îú‚îÄ‚îÄ (tabs)/                 # Main tabs: Home, Search, Bookings, Profile\n‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ index.tsx           # Home screen\n‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ search.tsx          # Search screen\n‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ bookings.tsx        # My Appointments\n‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ profile.tsx         # User profile\n‚îÇ   ‚îú‚îÄ‚îÄ business/               # Business detail, services, staff\n‚îÇ   ‚îú‚îÄ‚îÄ booking/                # Multi-step booking flow\n‚îÇ   ‚îî‚îÄ‚îÄ _layout.tsx             # Root layout with providers\n‚îú‚îÄ‚îÄ components/                 # Feature-specific components\n‚îÇ   ‚îú‚îÄ‚îÄ business/\n‚îÇ   ‚îú‚îÄ‚îÄ booking/\n‚îÇ   ‚îú‚îÄ‚îÄ ui/                     # Reusable primitives (Button, Card, etc.)\n‚îÇ   ‚îî‚îÄ‚îÄ map/                    # Map components (react-native-maps)\n‚îú‚îÄ‚îÄ hooks/                      # Custom hooks (useAuth, useSlots, useGeolocation)\n‚îú‚îÄ‚îÄ services/                   # API functions (react-query queries/mutations)\n‚îú‚îÄ‚îÄ stores/                     # Auth context, global filter state\n‚îú‚îÄ‚îÄ theme/                      # Colors, typography, spacing\n‚îú‚îÄ‚îÄ types/                      # App-specific types\n‚îî‚îÄ‚îÄ utils/                      # Local helpers\n```\n\n### 6.2 Routing & Navigation\n\nExpo Router provides file-based routing with nested layouts and deep linking support. The root `_layout.tsx` wraps the app in providers:\n\n- TanStack Query Client Provider\n- Auth Context Provider\n- Theme Provider\n\nProtected routes (e.g., `/booking`, `/profile`) redirect unauthenticated users to the sign-in screen. The booking flow can be resumed after authentication using a deep link with the saved state.\n\n### 6.3 State Management & Data Fetching\n\n- **Server State**: All API data (businesses, appointments, slots) is managed by TanStack React Query. Each feature defines query hooks (e.g., `useBusinesses`, `useTimeSlots`) with optimistic updates for mutations like booking creation.\n- **Auth & Session**: Supabase client handles auth state (current user, session). A React context provides the session object and login/logout/register methods.\n- **Local UI State**: Simple components use React `useState`; cross-screen state (e.g., search filters) uses Zustand stores with persistence.\n- **Offline Support**: React Query‚Äôs `networkMode: 'offlineFirst'` allows stale data to be shown when offline, with background refetch on connectivity.\n\n### 6.4 Authentication Flow\n\n- On launch, the app initializes Supabase and listens to `onAuthStateChange`.\n- Guest users can browse without a session.\n- When a protected action is triggered (e.g., ‚ÄúBook‚Äù), the app navigates to the auth stack. After successful sign-in, the pending action is stored and executed.\n- Business owner sign-up includes an additional step to create a business; the Supabase user ID is linked to a `BusinessOwner` record in the backend.\n\n### 6.5 Performance & Animations\n\n- List screens use `FlashList` for large datasets (virtualization).\n- Shared element transitions (e.g., business card to detail) are implemented with React Native Reanimated Layout Animations.\n- Map view clusters are handled on the backend to reduce client load.\n\n## 7. Backend Architecture (NestJS)\n\n### 7.1 Application Structure (`apps/api`)\n\n```\nsrc/\n‚îú‚îÄ‚îÄ main.ts                     # Bootstrap, CORS, global pipes\n‚îú‚îÄ‚îÄ app.module.ts               # Root module\n‚îú‚îÄ‚îÄ common/                     # Guards, interceptors, filters, decorators\n‚îú‚îÄ‚îÄ config/                     # Environment configuration (Zod)\n‚îú‚îÄ‚îÄ modules/\n‚îÇ   ‚îú‚îÄ‚îÄ auth/                   # JWT validation, Supabase integration, role guard\n‚îÇ   ‚îú‚îÄ‚îÄ user/                   # Customer & business owner profiles\n‚îÇ   ‚îú‚îÄ‚îÄ business/               # Business CRUD, search, geolocation\n‚îÇ   ‚îú‚îÄ‚îÄ service/                # Services, categories\n‚îÇ   ‚îú‚îÄ‚îÄ staff/                  # Staff management (business owner)\n‚îÇ   ‚îú‚îÄ‚îÄ booking/                # Booking flow, slot holds, availability\n‚îÇ   ‚îú‚îÄ‚îÄ payment/                # Stripe integration (webhooks, intents)\n‚îÇ   ‚îú‚îÄ‚îÄ notification/           # Email & push notifications (BullMQ consumers)\n‚îÇ   ‚îú‚îÄ‚îÄ review/                 # Reviews & ratings\n‚îÇ   ‚îú‚îÄ‚îÄ admin/                  # Admin dashboard endpoints\n‚îÇ   ‚îî‚îÄ‚îÄ upload/                 # Supabase Storage proxy\n‚îú‚îÄ‚îÄ prisma/\n‚îÇ   ‚îú‚îÄ‚îÄ schema.prisma           # Database models (includes PostGIS extensions)\n‚îÇ   ‚îî‚îÄ‚îÄ migrations/\n‚îú‚îÄ‚îÄ queue/                      # BullMQ producers and consumers\n‚îÇ   ‚îú‚îÄ‚îÄ slot-compute.queue.ts\n‚îÇ   ‚îú‚îÄ‚îÄ reminder.queue.ts\n‚îÇ   ‚îî‚îÄ‚îÄ payment-expiry.queue.ts\n‚îî‚îÄ‚îÄ test/                       # E2E and unit tests\n```\n\n### 7.2 Module Design & Service Boundaries\n\nEach module follows the NestJS module pattern: controller, service, repository (via Prisma). Clear boundaries:\n\n- **Auth Module**: A global guard validates Supabase JWT using `supabase/auth-helpers`. A role decorator restricts endpoints to `Customer`, `BusinessOwner`, or `Admin`. No local password storage; identity is external.\n- **User Module**: Manages user profiles (first name, last name, phone, avatar). Maps `authId` from Supabase to internal user records. `BusinessOwner` records are created upon business registration.\n- **Business Module**: Full business CRUD with geospatial search. The `search` endpoint uses PostGIS `ST_DWithin` and `ST_Distance` for proximity sorting, leveraging an index on `location`. Full‚Äëtext search on business name and services via PostgreSQL `tsvector`.\n- **Service & Category Module**: Hierarchical categories (admin‚Äëdefined) and services belonging to a business. Cached aggressively.\n- **Staff Module**: Manage staff members per business; visibility toggle to customers.\n- **Booking Module**: Core transactional logic.\n  - **Slot computation** (feature F11): Calculates available time slots based on business opening hours, staff schedules, service duration, and existing bookings. Results are cached in Redis (TTL 5 minutes). BullMQ processes batch re‚Äëcomputes.\n  - **Slot hold**: When a user enters the payment step, a Redis key (e.g., `slot:hold:<businessId>:<staffId>:<date>:<time>`) is set with a 5‚Äëminute expiration. The key is checked during booking creation to prevent conflicts.\n  - **Booking creation**: Prisma transaction ensures atomicity ‚Äì appointment creation + slot hold verification + payment record update.\n- **Payment Module**: Integrates with Stripe. Webhooks handle `payment_intent.succeeded` and `payment_intent.payment_failed`. On success, booking status is confirmed; on failure, the slot hold is released.\n- **Notification Module**: Consumes jobs from BullMQ queues to send emails (transactional, reminders) and push notifications (Expo push tokens). Templates are rendered server‚Äëside.\n- **Review Module**: Only confirmed/completed appointment customers can leave a review. Moderation endpoints for admin.\n- **Admin Module**: Analytics endpoints, business owner management, review moderation. Protected by `Admin` role guard.\n\n### 7.3 Database Design (PostgreSQL + PostGIS)\n\nPrisma schema with PostGIS extension enabled via migration. Key models:\n\n```prisma\nmodel User {\n  id        String   @id @default(uuid())\n  authId    String   @unique // Supabase user ID\n  email     String   @unique\n  firstName String?\n  lastName  String?\n  phone     String?\n  avatarUrl String?\n  role      UserRole\n  createdAt DateTime @default(now())\n  \n  customer      Customer?\n  businessOwner BusinessOwner?\n}\n\nmodel Business {\n  id          String   @id @default(uuid())\n  ownerId     String\n  owner       BusinessOwner @relation(fields: [ownerId], references: [id])\n  name        String\n  description String?\n  address     String\n  location    Unsupported(\"geometry(Point, 4326)\") @default(dbgenerated(\"''::geometry\"))\n  categoryIds String[]\n  phone       String?\n  photos      String[]\n  rating      Float    @default(0)\n  active      Boolean  @default(true)\n  createdAt   DateTime @default(now())\n  \n  services        Service[]\n  staff           Staff[]\n  openingHours    OpeningHours[]\n  appointments    Appointment[]\n  reviews         Review[]\n  \n  @@index([location], type: Gist)\n  @@index([name], type: Gin, with: [\"gin_trgm_ops\"]) // for trigram search\n}\n\nmodel Service {\n  id            String   @id @default(uuid())\n  businessId    String\n  business      Business @relation(fields: [businessId], references: [id])\n  name          String\n  description   String?\n  duration      Int      // in minutes\n  price         Float\n  categoryId    String?\n  isActive      Boolean  @default(true)\n  \n  @@index([categoryId])\n}\n\nmodel Appointment {\n  id            String      @id @default(uuid())\n  businessId    String\n  business      Business    @relation(fields: [businessId], references: [id])\n  customerId    String\n  customer      Customer    @relation(fields: [customerId], references: [id])\n  staffId       String?\n  staff         Staff?      @relation(fields: [staffId], references: [id])\n  serviceId     String\n  service       Service     @relation(fields: [serviceId], references: [id])\n  date          DateTime    @db.Date\n  startTime     String      // HH:mm\n  endTime       String      // HH:mm\n  status        AppointmentStatus\n  notes         String?\n  totalPrice    Float\n  paymentId     String?\n  createdAt     DateTime    @default(now())\n  \n  @@unique([staffId, date, startTime]) // prevent double booking\n}\n\n// Other models: Category, Staff, OpeningHours, Review, etc.\n```\n\n### 7.4 Caching & Real-Time Holds\n\n**Redis** used for:\n\n- **Slot availability cache**: Key `slots:<businessId>:<date>` stores JSON array of slots; refreshed by BullMQ after bookings.\n- **Slot holds**: `hold:<businessId>:<staffId>:<date>:<time>`, with TTL 5min.\n- **Session / rate limit**: For API protection.\n- **Category tree**: Pre-computed, rarely changes.\n\n### 7.5 Background Jobs (BullMQ)\n\n- **Slot compute**: Triggered on booking creation/cancellation, or schedule updates. Recalculates available slots and updates Redis.\n- **Reminder queue**: Scheduled jobs for 24h and 1h before appointment. Sends push/email notifications.\n- **Payment expiry**: Monitors incomplete payments and releases holds.\n\n### 7.6 API Design & Authentication\n\nAll API endpoints are prefixed with `/api/v1`. The NestJS server uses a global `SupabaseAuthGuard` that extracts the JWT from the `Authorization` header, validates it via Supabase public key, and attaches the verified user object (including `id`, `email`, `role`) to the request. Role‚Äëbased guards then restrict access.\n\nRequest/response validation uses `class-validator` and `zod` for complex shapes. OpenAPI (Swagger) is generated for documentation and client generation.\n\n### 7.7 File Storage\n\nImages are uploaded directly to Supabase Storage via pre-signed URLs generated by the `Upload` module, preventing server bottlenecks. The module also handles metadata (e.g., business photos array).\n\n## 8. Admin Web Client\n\nA lightweight React app (`apps/admin-web`) with role‚Äëbased authentication.
