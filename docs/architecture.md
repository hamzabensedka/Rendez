It is consumed by both mobile and api.\n\n## 5. Backend Architecture\n### 5.1 NestJS Module Design\nThe backend is structured as a modular monolith, each domain encapsulated in its own NestJS module with clear boundaries. Core modules:\n- **AuthModule**: Registration, login, social auth callbacks, token issuance/refresh, password reset. Uses Supabase Auth client or local JWT strategy with Redis-managed refresh tokens. Exposes `POST /auth/signup`, `POST /auth/login`, `POST /auth/refresh`, etc.\n- **UserModule**: Profile management (CRUD), avatar upload, notification preferences.\n- **BusinessModule**: Business CRUD, search (full-text + geospatial), category associations, gallery management. Endpoints for discover and business detail.\n- **ServiceCatalogModule**: Hierarchical service categories, services assignment to businesses. Read-heavy operations with caching.\n- **BookingModule**: Core booking flow (slot selection, booking creation, cancellation, rescheduling). Uses database transactions and Redis locks to prevent double-booking.\n- **AvailabilityModule**: Computes real-time availability for a service/staff/date combination. Called by BookingModule and SearchModule. Implements slot generation algorithm and caches results.\n- **ReviewModule**: Submission and listing of reviews per business, with moderation hooks.\n- **NotificationModule**: Templated push (via Firebase Cloud Messaging) and email (via SendGrid/Mailgun) notifications for booking confirmations, reminders, cancellations.\n- **PaymentModule**: Integration with Stripe for prepayment flows (P1). Handles payment intents and refunds.\n- **AdminModule**: Admin-only endpoints for managing businesses, users, disputes, and system settings. Guarded by role-based access.\n\nEach module follows a layered pattern: `controller` (route handling, DTO validation), `service` (business logic), `repository` (Prisma delegated via PrismaService), and `dto` (input/output validation using class-validator and class-transformer).\n\n### 5.2 Data Layer\n- **Prisma** ORM connects to PostgreSQL/PostGIS. The `schema.prisma` includes models for `User`, `Business`, `BusinessStaff`, `Service`, `Category`, `Booking`, `Review`, `WorkingHours`, `BookingSlot`, etc. `Business.location` is a `Geography(Point, 4326)` for PostGIS spatial queries.\n- **PostGIS** enables location-based searching (`ST_DWithin` for radius searches, `ST_Distance` for sorting by proximity).\n- **Redis** is used for:\n  - Caching search results and precomputed availability slots (TTL ~30sвҖ“2min).\n  - Storing refresh tokens and their rotation/invalidation (with TTL=refresh token lifetime).\n  - Rate limiting (using Redis-backed throttlers).\n  - Distributed locks for critical booking writes.\n\n### 5.3 API Design\n- RESTful endpoints prefixed with `/api/v1`.\n- Consistent error format: `{ statusCode, message, errors }`.\n- Pagination via `cursor` or `page/pageSize` (cursor for infinite scroll).\n- Search endpoint: `GET /businesses/search?q=hair&lat=...&lng=...&radius=10&category=...&sort=distance` вҶ’ returns paginated list with precomputed `nextAvailableSlot`.\n- Booking availability: `GET /availability?serviceId=...&date=...&staffId=optional` returns array of `{ startTime, endTime, staffId }`.\n- All mutations require valid JWT (except auth endpoints).\n\n### 5.4 Authentication & Security\n- **JWT Access Token**: Short-lived (15 minutes), signed with RS256 or HS256, sent in `Authorization: Bearer` header.\n- **Refresh Token**: Opaque token stored in Redis, associated with user ID. Rotation: each refresh issues a new pair and invalidates the previous refresh token.\n- **Supabase Auth Option**: When enabled, the backend validates Supabase-issued JWTs using their JWKS endpoint. Social logins (Google/Apple) are delegated to Supabase; backend creates/links `User` record on first login.\n- **Role-Based Guards**: NestJS guards check `USER`, `PROVIDER`, `ADMIN` roles from the JWT payload.\n- **Input Validation**: All DTOs use class-validator decorators. Global `ValidationPipe` rejects invalid input.\n- **HTTPS** enforced in production.\n\n## 6. Mobile App Architecture\n### 6.1 Routing & Screens\nExpo Router provides file-based routing under `app/` directory:\n- `(tabs)/` вҖ“ Tab navigator: Home, Search, Bookings, Favorites, Profile.\n- `(auth)/` вҖ“ Login, Signup, Forgot Password.\n- `business/[id]/` вҖ“ Business detail with nested tabs (Info, Services, Reviews, Gallery).\n- `booking/` вҖ“ Booking flow steps: service select, date/staff, time, confirm.\n- `bookings/` вҖ“ Upcoming/past appointments list.\n- `profile/` вҖ“ User settings, payment methods, etc.\n- `_layout.tsx` files define stack/tab navigation and shared transitions.\n\n### 6.2 State Management\n- **Server State**: TanStack React Query manages all API data. UI state (selected filters, map visible) stored in local component state or a small Zustand store.\n- **Persistence**: Tokens stored in `expo-secure-store`. Each step fetches its own data (staff, slots). Slot selection uses a custom time picker with animated selection feedback.\n- **Favorites**: Heart icon toggles a mutation. Optimistic UI updates the query cache.\n- **Push Notifications**: Expo Notifications with FCM for booking confirmations, reminders.\n- **Offline**: No full offline mode in MVP; stale data is shown with React QueryвҖҷs `staleTime` and `placeholderData` while refetching. Components built with React NativeвҖҷs `StyleSheet` and a small set of primitives (Box, Text, Pressable).\n- Light/dark mode supported via `useColorScheme`.\n\n## 7. Data Flow Examples\n### 7.1 Business Search\n1. User opens search screen, enters query вҖңhair salonвҖқ.\n2. `useInfiniteQuery` fetches `/api/v1/businesses/search?q=hair salon&lat=48.8566&lng=2.3522&radius=10&pageSize=20`.\n3. Backend validates params, constructs Prisma query: full-text search on `Business.name` and `Service.name` (via `@@fulltext` index), `ST_DWithin` for location filter, sorts by distance or rating.\n4. Query result includes eagerly loaded `services` and computed `nextAvailableSlot` (aggregated from availability cache).\n5. Response paginated with `nextCursor`. React Query caches each page.\n6. As user scrolls, next page fetched with cursor. Results displayed in list; map markers from current data.\n\n### 7.2 Availability Computation\n1. Mobile calls `GET /api/v1/availability?serviceId=xyz&date=2025-06-15`.\n2. Backend first checks Redis cache key `availability:{serviceId}:{date}`. If hit, return cached JSON.\n3. On miss:\n   - Fetch service duration, associated business, staff members working that day (from `WorkingHours`).\n   - Fetch all existing bookings for that staff/service on that date.\n   - Generate possible slots (e.g., every 15 min) within working hours, excluding busy periods that overlap with service duration.\n   - Apply business breaks if any.\n   - Result: array of slot objects `{ startTime, endTime, staffId }`.\n4. Cache result in Redis with 60s TTL. Return to client.\n5. When a booking is created/cancelled, invalidate relevant cache keys.\n\n### 7.3 Booking Creation\n1. Mobile calls `POST /api/v1/bookings` with `{ serviceId, staffId, date, startTime, note }`.\n2. Backend middleware validates JWT and role.\n3. BookingService uses a Redis-based distributed lock (`lock:booking:{serviceId}:{staffId}:{date}:{startTime}`) to serialise writes.\n4. Inside transaction (Prisma `$transaction`):\n   - Re-fetch availability for the exact slot (idempotency check) вҖ“ if slot taken, throw `ConflictException`.\n   - Create `Booking` record with status `CONFIRMED`.\n   - Optionally create payment intent if prepayment required.\n5. On success, publish booking event to notification queue (or direct service call) to send user & provider push/email.\n6. Invalidate availability cache for that service/date.\n7. Return created booking.\n\n## 8.
