Service Boundaries\n\n### 4.1 Authentication Service\n- **Responsibility**: User registration, login, OAuth, token management, session handling\n- **Endpoints**: POST /auth/register, POST /auth/login, GET /admin/analytics\n- **Data**: Admin logs table, disputes table\n\n## 5. Data Flow\n\n### 5.1 Authentication Flow\n1. User initiates login (email/password or OAuth)\n2. Mobile app sends credentials to `/auth/login`\n3. NestJS validates credentials, generates JWT access token (15 min) and refresh token (30 days)\n4. Refresh token stored in Redis with device ID\n5. Mobile app stores access token in memory, refresh token in secure storage (expo-secure-store)\n6. On app launch, check for valid access token; if expired, use refresh token to get new access token\n7. Biometric prompt after first successful login (if device supports)\n\n### 5.2 Booking Flow\n1. Customer searches for business (via Search Service)\n2. Views business detail (Business Service + Review Service)\n3. Selects services (Service Catalog Service)\n4. Requests available slots (Booking Service → Redis cache → PostgreSQL)\n5. Selects time and staff (optional)\n6. Confirms booking (Booking Service creates appointment)\n7. Payment processed (Payment Service)\n8. Confirmation sent (Notification Service)\n9. Calendar event added (mobile app)\n\n### 5.3 Search Flow\n1. User types query (debounced 300ms)\n2. Mobile app calls `/search/autocomplete` with query\n3. Backend checks Redis cache for autocomplete suggestions\n4. If cache miss, query PostgreSQL full-text search index\n5. Results cached in Redis with TTL of 5 minutes\n6. User selects filter/sort, calls `/search` with parameters\n7. Backend uses PostGIS for distance calculation, full-text search for relevance\n8. Results paginated and returned\n\n## 6. Caching Strategy\n\n| Cache | Technology | TTL | Invalidation |\n|-------|------------|-----|--------------|\n| Autocomplete suggestions | Redis | 5 min | On business/service update |\n| Business details | Redis | 10 min | On business update |\n| Availability slots | Redis | 30 sec | On booking creation/cancellation |\n| User session | Redis | 30 days | On logout/password change |\n| Search results | Redis | 2 min | On filter change |\n| Category tree | Redis | 1 hour | On admin category update |\n| API rate limits | Redis | Dynamic | Per endpoint configuration |\n\n## 7. Security Considerations\n\n- **Authentication**: JWT with refresh token rotation; refresh tokens invalidated on reuse\n- **Authorization**: Role-based access control (customer, business_owner, admin)\n- **Data Encryption**: HTTPS everywhere; sensitive data encrypted at rest (PostgreSQL pgcrypto)\n- **Input Validation**: Class-validator decorators on all DTOs; sanitize user input\n- **Rate Limiting**: Redis-based rate limiting per user/IP per endpoint\n- **SQL Injection**: Prisma ORM prevents SQL injection\n- **CORS**: Strict CORS policy for API endpoints\n- **Environment Variables**: All secrets stored in environment variables, not in code\n\n## 8. Performance Targets\n\n| Metric | Target |\n|--------|--------|\n| API response time (p95) | < 200ms |\n| Search results | < 500ms |\n| Booking confirmation | < 2s |\n| App cold start | < 3s |\n| Image lazy loading | < 1.5s per image |\n| Map rendering (100 markers) | < 2s |\n| Concurrent users | 10,000 |\n| Database query time (p95) | < 100ms |\n\n## 9. Deployment Strategy\n\n### 9.1 Backend (NestJS)\n- **Containerization**: Docker Compose for local dev; production deployment via Kubernetes (future)\n- **CI/CD**: GitHub Actions → Build Docker image → Push to container registry → Deploy to cloud (AWS ECS/Google Cloud Run)\n- **Database Migrations**: Run Prisma migrations as part of deployment pipeline\n- **Environment**: Staging and production environments\n\n### 9.2 Mobile (Expo)\n- **Development**: Expo Go for quick iteration\n- **Staging**: EAS Build with internal distribution (Ad-hoc)\n- **Production**: EAS Build → App Store Connect / Google Play Console\n- **Updates**: EAS Update for over-the-air JavaScript updates (not native code)\n\n## 10. Testing Strategy\n\n| Layer | Tool | Scope |\n|-------|------|-------|\n| Unit tests | Jest | Services, utilities, hooks |\n| Integration tests | Jest + Supertest | API endpoints, database interactions |\n| Component tests | Jest + React Native Testing Library | UI components |\n| E2E tests | Detox | Critical user flows (booking, search) |\n| Performance tests | k6 | API load testing |\n\n## 11. Monitoring and Observability\n\n- **Logging**: Structured logging with Winston (backend), console.log (mobile development)\n- **Metrics**: Prometheus + Grafana for API metrics (request count, latency, error rate)\n- **Error Tracking**: Sentry for both backend and mobile\n- **Health Checks**: `/health` endpoint for API; Expo updates health check\n- **Alerting**: PagerDuty for critical errors\n\n## 12.
