Service Boundaries & Module Design\n\n### 4.1 Auth Module\n- **Responsibility:** User registration, email verification, login, token refresh, password reset, social login callback.\n- **Dependencies:** Supabase Auth (external), Users module (internal).\n- **Key Decisions:** Supabase handles social OAuth flows and JWT issuance. The backend validates Supabase-issued JWTs using the Supabase JWT secret. On first social login, a webhook or callback creates the local user record. Refresh tokens are managed by Supabase; the mobile app calls `supabase.auth.refreshSession()` on foreground.\n\n### 4.2 Users Module\n- **Responsibility:** CRUD for user profiles, preferences, notification settings, payment methods, account deletion.\n- **Dependencies:** Auth module (for identity), Payments module.\n- **Key Decisions:** User profile is separate from auth identity. Deleting a user cascades to anonymize bookings and reviews (GDPR compliance).\n\n### 4.3 Businesses Module\n- **Responsibility:** Business CRUD (provider-facing), public search with filters, geospatial queries, business detail aggregation.\n- **Dependencies:** Categories module, Reviews module, Services module, PostGIS.\n- **Key Decisions:** Search uses PostgreSQL full-text search (`tsvector`) combined with PostGIS `ST_DWithin` for location-based queries. Results are cached in Redis with a 5-minute TTL, invalidated on business update. The `geo/` subdirectory contains raw SQL query builders for complex spatial queries.\n\n### 4.4 Categories Module\n- **Responsibility:** Hierarchical category tree management, category-business associations.\n- **Dependencies:** None.\n- **Key Decisions:** Categories use a `parentId` self-referencing pattern in Prisma. The tree is cached in Redis and refreshed on admin changes. Mobile app fetches the full tree on launch and caches it locally with React Query.\n\n### 4.5 Services Module\n- **Responsibility:** Service catalog per business (name, duration, price, description, buffer time).\n- **Dependencies:** Businesses module.\n- **Key Decisions:** Services are always queried in the context of a business. Duration is stored in minutes. Buffer time (before/after) is configurable per service and affects slot computation.\n\n### 4.6 Availability Module\n- **Responsibility:** Provider working hours, breaks, time-off (vacations, holidays), staff assignment.\n- **Dependencies:** Businesses module.\n- **Key Decisions:** Working hours are stored as recurring weekly rules with exceptions (overrides for specific dates). Breaks are stored as time ranges within a working day. This module provides the constraints that the Slot Calculator consumes.\n\n### 4.7 Bookings Module\n- **Responsibility:** Booking creation, confirmation, rescheduling, cancellation, concurrent slot protection, booking history.\n- **Dependencies:** Services module, Availability module, Payments module, Notifications module.\n- **Key Decisions:** This is the most critical module for data consistency. Slot computation is handled by `SlotCalculatorService` which:\n  1. Fetches provider working hours and breaks from Availability module.\n  2. Fetches existing bookings for the date range.\n  3. Computes available windows by subtracting booked slots and buffers from working hours.\n  4. Returns available start times as ISO 8601 strings.\n  \n  Concurrent booking protection uses PostgreSQL advisory locks (`pg_advisory_xact_lock`) on the (business_id, staff_id, date) tuple during the booking transaction. If a slot is taken between the availability check and confirmation, the transaction fails and the user is prompted to select a new slot.\n\n### 4.8 Reviews Module\n- **Responsibility:** Review CRUD, rating aggregation, review moderation (admin).\n- **Dependencies:** Businesses module, Bookings module (verify completed booking).\n- **Key Decisions:** Users can only review businesses they have a completed booking with. Rating summary (average, count per star) is denormalized on the Business table and updated via database triggers or application-level hooks on review create/update/delete.\n\n### 4.9 Favorites Module\n- **Responsibility:** Toggle and list user favorites.\n- **Dependencies:** Businesses module.\n- **Key Decisions:** Simple many-to-many relationship. Favorites list endpoint supports pagination and includes the next available slot for each business (computed on-the-fly or cached).\n\n### 4.10 Payments Module\n- **Responsibility:** Store payment methods (tokenized via Stripe), handle pre-authorization holds for no-show protection.\n- **Dependencies:** Users module.\n- **Key Decisions:** The app never handles raw card numbers. Stripe SDK on the mobile app tokenizes cards and sends a payment method ID to the backend. The backend stores only the Stripe payment method ID and last four digits. Actual charges are processed server-side via Stripe API.\n\n### 4.11 Notifications Module\n- **Responsibility:** Trigger push notifications and emails for booking confirmations, reminders, cancellations, and marketing.\n- **Dependencies:** Users module (device tokens, preferences), Bookings module.\n- **Key Decisions:** Notifications are enqueued as BullMQ jobs to avoid blocking API responses. The module exposes an internal API for other modules to request notifications. Push tokens are stored per user per device and managed via Expo Push API.\n\n### 4.12 Admin Module\n- **Responsibility:** Dashboard metrics, user management, business approval, category management, system health.\n- **Dependencies:** All other modules.\n- **Key Decisions:** Admin endpoints are protected by a role guard (`@Roles('admin')`). Admin users are seeded via `prisma/seed.ts` and cannot register publicly.\n\n## 5. Slot Computation Algorithm\n\n```\nInput:\n  - businessId, serviceId, staffId (optional), dateRange (start, end)\n\nAlgorithm:\n  1. Fetch service (duration, bufferBefore, bufferAfter).\n  2. Fetch staff availability for dateRange:\n     - Recurring weekly rules filtered by validFrom/validUntil.\n     - Apply date-specific overrides.\n  3. Fetch time-off entries overlapping dateRange.\n  4. Fetch existing CONFIRMED bookings for the staff/business in dateRange.\n  5. For each day in dateRange:\n     a. Get working intervals from availability.\n     b. Subtract break intervals.\n     c. Subtract time-off intervals.\n     d. Subtract existing booking intervals (including buffers).\n     e. From remaining intervals, generate slots at configurable granularity (default 15 min).\n     f. Filter slots where (slotStart + duration + bufferAfter) fits within interval.\n  6. Return slots grouped by date, each as { startTime: ISO, endTime: ISO }.\n\nCaching:\n  - Computed slots cached in Redis with key: `slots:{businessId}:{serviceId}:{staffId}:{date}`\n  - TTL: 60 seconds (short to ensure freshness).\n  - Invalidated on booking creation/cancellation for that date.\n\nConcurrency:\n  - Booking endpoint acquires advisory lock: `pg_advisory_xact_lock(hashtext(businessId + staffId + date))`.\n  - Re-checks slot availability within the locked transaction.\n```\n\n## 9. This is simpler and less error-prone for the booking flow.\n\n6. **Nx Monorepo:** Shared types and utilities between mobile and API reduce duplication. Nx's dependency graph enables efficient CI (only build/test affected projects).\n\n7. **Expo Managed Workflow:** Faster development cycles, OTA updates via EAS, and no native build toolchain maintenance.
