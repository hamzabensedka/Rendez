  "content": "# Planity Clone Architecture Document\n\n## 1. Overview\n\nPlanity Clone is a booking platform for beauty and wellness businesses. The system includes a customer-facing mobile app, a provider portal (future), and an admin dashboard (future). This document defines the architecture for the initial customer mobile app and backend services, ensuring scalability, maintainability, and clean separation of concerns.\n\n---\n\n## 2. Technology Stack\n\n| Layer | Technology |\n|-------|------------|\n| **Mobile App** | Expo (managed workflow), React Native, TypeScript |\n| **Navigation** | Expo Router (file-based routing) |\n| **UI Animation** | React Native Reanimated 3 |\n| **Server State** | TanStack React Query |\n| **Auth Client** | Supabase Auth (with secure token storage) |\n| **Backend** | NestJS, TypeScript |\n| **ORM** | Prisma with PostgreSQL and PostGIS |\n| **Database** | PostgreSQL 15 (with PostGIS extension) |\n| **Cache** | Redis 7 |\n| **Monorepo** | Nx with pnpm workspaces |\n| **CI/CD** | GitHub Actions, EAS Build (mobile), Docker Compose (backend) |\n| **Testing** | Jest (unit/integration), React Native Testing Library |\n| **Monitoring** | Supabase logs, Sentry (client-side), Prometheus (server) |\n\n---\n\n## 3. Monorepo Structure\n\nThe codebase is organized as an Nx monorepo with pnpm. All applications and shared libraries live under a single repository.\n\n```\nplanity/\n├── apps/\n│   ├── mobile/           # Expo / React Native app\n│   └── api/              # NestJS backend\n├── libs/\n│   ├── shared-types/     # User profiles, favorites, settings\n│   │   ├── location/    # Geospatial queries (PostGIS)\n│   │   ├── notification # Push/email notifications\n│   │   └── payment/     # Payment integration (Stripe)\n│   ├── common/           # Service Boundaries\n\nThe backend follows a modular monolith approach with clearly separated NestJS modules. Each module owns its domain and communicates via in-process calls (service injection). For future scaling, modules can be extracted to microservices without breaking the API contract.\n\n### 4.1 Module Responsibilities\n\n- **Auth Module**: User registration, login (credentials, Google, Apple), email verification, password reset, JWT token issuing and refresh. Integrates with Supabase Auth for identity management but maintains its own user records and roles.\n- **Business Module**: CRUD for businesses, business hours, photos. Supports filtering by category, rating, price, distance.\n- **Service Module**: Services offered by businesses, pricing, duration, buffer times.\n- **Employee Module**: Staff profiles, work schedules, specialties.\n- **Booking Module**: Core appointment booking logic. Computes available slots using business hours, employee schedules, existing bookings, and buffer times. Implements slot locking with Redis TTL (5 minutes) and releases locks on expiry or cancellation. Uses transaction-safe patterns to prevent double-booking.\n- **Location Module**: Geospatial indexing (PostGIS). Provides nearby search, map viewport queries, distance calculation.\n- **User Module**: User profile editing, favorites management, notification preferences.\n- **Notification Module**: Push notification triggers via Firebase Cloud Messaging (FCM) and email (using transactional provider). Processes booking reminders, cancellations, promotional messages.\n- **Payment Module**: Handles payments via Stripe. Supports pre-authorization, capture on booking confirmation, and refunds.\n\n### 4.2 API Design\n\nRESTful API with versioning (`/api/v1/`). Consistent error format using RFC 7807 Problem Details. OpenAPI (Swagger) auto-generated from NestJS decorators.\n\nExample Endpoints:\n- `GET /api/v1/businesses` – search with filters\n- `GET /api/v1/businesses/:id` – business detail\n- `GET /api/v1/businesses/:id/slots?date=...&serviceId=...` – available time slots\n- `POST /api/v1/bookings` – create booking\n- `PATCH /api/v1/bookings/:id/cancel` – cancel booking\n- `GET /api/v1/users/me/favorites` – user favorites\n\nAll endpoints except public ones require JWT Bearer token. Public endpoints: guest browse, business/search, slot availability (for unauthenticated viewing).\n\n---\n\n## 5. Data Model (Prisma Schema)\n\nCore tables (simplified):\n\n- **User** (id, email, phone, name, avatar, role, authProvider, stripeCustomerId)\n- **Business** (id, name, categoryId, description, address, lat, lng, rating, photos[])\n- **BusinessHour** (businessId, dayOfWeek, openTime, closeTime)\n- **Service** (id, businessId, name, duration, price, currency, bufferTime)\n- **Employee** (id, businessId, name, photo, specialties[])\n- **EmployeeSchedule** (employeeId, businessId, dayOfWeek, start, end)\n- **Booking** (id, userId, businessId, serviceId, employeeId, startTime, endTime, status, price, locked)\n- **Favorite** (userId, businessId)\n- **Category** (id, name, icon, parentId)\n- **Review** (id, userId, businessId, rating, comment, createdAt)\n\nPostGIS is used on the `Business` table for location queries: `geometry(Point, 4326)` column. The `lat/lng` fields are maintained for convenience and can be generated from geometry.\n\nAll timestamps (opening hours, bookings) are stored in UTC. Timezones are handled at the application layer.\n\n---\n\n## 6. Caching Strategy\n\n- **Server Side (Redis)**:\n  - Slot lock keys with TTL (5 min).\n  - Frequent queries like categories, trending businesses (cached for 5-10 min).\n  - Session/blacklist for invalidated JWT tokens.\n- **Client Side (React Query)**:\n  - Data fetching hooks with appropriate `staleTime`:\n    - Business details: 5 minutes\n    - Availability slots: 0 (always refetch)\n    - Categories: 30 minutes\n  - Infinite scrolling for search results using React Query's `useInfiniteQuery`.\n  - Optimistic updates for favourites toggling.\n\n---\n\n## 7. Authentication & Security\n\n- Identity management delegated to **Supabase Auth** (supports email/password, Google, Apple).\n- On successful Supabase login, the backend issues a custom JWT (access token 15 min, refresh token 7 days).\n- Refresh tokens are stored in Redis (keyed by userId) for revocation capability.\n- Mobile app stores tokens securely using `expo-secure-store`.\n- All API requests include the access token in `Authorization: Bearer` header.\n- Role-based access control: `CUSTOMER`, `PROVIDER`, `ADMIN` roles enforced via NestJS guards.\n- Input validation via class-validator/class-transformer in DTOs.\n- Rate limiting on auth and booking endpoints using NestJS `@nestjs/throttler` backed by Redis.\n\n---\n\n## 8. Mobile App Architecture\n\n### 8.1 Navigation & Routing\n\nExpo Router provides file-based routing using the `app/` directory. The root layout wraps the app with providers (React Query, Auth, Theme).\n\n- **Auth group**: `(auth)/login`, `(auth)/register`, etc. These screens are accessible when not logged in.\n- **Main tabs**: `(tabs)/_layout.tsx` defines a tab navigator with Home, Search, Bookings, Profile.\n- **Modals**: Booking flow screens are presented modally (Expo Router `_layout` with `presentation: 'modal'`).\n\n### 8.2 State Management\n\n- **Server State**: All data fetched from the API is managed by TanStack React Query. Custom hooks encapsulate query keys, caching, and mutations (e.g., `useBusinesses`, `useSlots`, `useCreateBooking`).\n- **Client State**: Lightweight client state (e.g., selected slot step in booking wizard, map viewport center) is managed with React `useReducer` or Zustand stores within feature modules. No global store is necessary.\n- **Auth State**: Auth context provides current user, login/logout functions, and token refresh logic. React Query's `QueryClient` is cleared on logout.\n\n### 8.3 UI & Animation\n\n- **Design System**: Custom component library in `libs/ui` following atomic design principles (atoms, molecules, organisms). Includes `BookingCard`, `TimeSlotChip`, `BusinessCard`, `StarRating`. Styled using `StyleSheet` and `Restyle` (or similar) for consistent theming.\n- **Animations**: React Native Reanimated for smooth transitions (hero carousel, expanding filters, skeleton loaders, drag-to-refresh).\n\n### 8.4 Offline & Error Handling\n\n- React Query provides retry and background refetching. Cache-first strategy enables basic offline viewing of previously loaded businesses.\n- Network loss feedback via a banner component.\n- Empty states and error components are reused across screens.\n\n---\n\n## 9. Real-time Slot Locking & Booking Integrity\n\nWhen a user selects a time slot:\n1. Client calls `POST /api/v1/slots/lock` with businessId, serviceId, employeeId, startTime.\n2. Server checks availability using the same algorithm but also sets a Redis key `lock:<slot-composite-key>` with TTL 5 min. If key exists, slot is unavailable.\n3. A timer countdown is shown in the UI. If booking is not completed, the lock expires automatically (Redis eviction).\n4. On successful booking, the lock is converted to a permanent booking inside a database transaction, and the Redis key is deleted. The transaction additionally validates that no conflicting booking has been inserted concurrently.\n5. Reschedule and cancellation also release locks appropriately.\n\nThis mechanism prevents double booking without requiring WebSockets.\n\n---\n\n## 10. CI/CD & Deployment\n\n- **GitHub Actions**:\n  - Run lint, type-check, unit tests on PR.\n  - On merge to `main`, build and test backend Docker image, push to container registry.\n  - Trigger EAS Build for mobile app (development build or production).\n\n- **Backend Deployment**:\n  - API container runs on a cloud VM (or Kubernetes).\n  - PostgreSQL & Redis are managed services (Supabase provides PostgreSQL with PostGIS; Redis via cloud provider).\n\n- **Mobile Deployment**:\n  - Preview builds via EAS Build for internal testing.\n  - Production builds submitted to App Store and Google Play via EAS Submit.\n  - OTA updates for JS bundles using EAS Update.\n\n- **Local Development**:\n  - `docker-compose up` starts PostgreSQL, Redis, and Supabase local stack.\n  - pnpm dev in `apps/mobile` and `apps/api` respectively.\n\n---\n\n## 11. Testing Strategy\n\n- **Mobile**:\n  - Unit tests for hooks, utility functions (Jest + React Native Testing Library).\n  - Component tests with mocked API responses.\n  - Snapshot tests for design system components.\n- **Backend**:\n  - Unit tests for services and slot computation logic.\n  - Integration tests using NestJS testing utilities with a real database (Dockerized PostgreSQL).\n  - E2E tests with Supertest for critical booking flow.\n- **CI** ensures tests pass before merge.\n\n---\n\n## 12. Observability\n\n- **Client**: Sentry for crash reporting. Basic analytics via Firebase Analytics.\n- **Server**: Structured logging with `pino`. Prometheus metrics (request duration, booking creation rate). Health check endpoints.\n\n---\n\n## 13. Folder Scaffold Reference\n\nThe complete scaffold is defined in sections 3.1 and 3.2. All paths align with Nx monorepo conventions and mobile/backend separation.\n\n---\n\nThis architecture provides a scalable, maintainable foundation for Planity Clone, balancing rapid development with production-ready patterns.\n",
