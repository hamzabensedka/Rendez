  "content": "# Planity Clone вҖ” System Architecture\n\n## 1. Executive Summary\n\nPlanity Clone is a multi-sided platform connecting customers with beauty and wellness providers. The system is designed as a monorepo managed by Nx with pnpm workspaces, comprising a React Native (Expo) mobile application and a NestJS backend API. The architecture emphasizes clean separation of concerns, horizontal scalability, real-time capabilities, and maintainability through modular design. PostgreSQL with PostGIS powers geospatial queries, Redis handles caching and real-time features, and Prisma provides type-safe database access. The mobile app leverages Expo Router for file-based navigation, TanStack React Query for server state management, and React Native Reanimated for performant animations.\n\n## 2. Design Principles\n\n- **Separation of Concerns** вҖ” Clear boundaries between presentation, application, domain, and infrastructure layers.\n- **Domain-Driven Design** вҖ” Business logic encapsulated in domain modules independent of delivery mechanisms.\n- **API-First** вҖ” All client-server communication via versioned RESTful APIs with OpenAPI documentation.\n- **Offline-Aware** вҖ” Optimistic updates with React Query; graceful degradation when connectivity is lost.\n- **Type Safety** вҖ” End-to-end TypeScript from database schema through API contracts to mobile UI.\n- **Testability** вҖ” Dependency injection, interface-based design, and comprehensive test coverage at all layers.\n- **Observability** вҖ” Structured logging, distributed tracing, and health checks built into every service.\n\n## 3. API Design\n\n### 7.1 RESTful Conventions\n- Base URL: `https://api.planity-clone.com/v1`\n- Versioning via URL path (`/v1/`)\n- Standard HTTP methods: GET, POST, PATCH, DELETE\n- Consistent error response format:\n```json\n{\n  \"error\": {\n    \"code\": \"VALIDATION_ERROR\",\n    \"message\": \"Human-readable message\",\n    \"details\": [{ \"field\": \"email\", \"message\": \"Invalid email format\" }]\n  }\n}\n```\n- Pagination via cursor-based pagination for list endpoints:\n```json\n{\n  \"data\": [...],\n  \"pagination\": {\n    \"nextCursor\": \"eyJpZCI6IjEyMyJ9\",\n    \"hasMore\": true\n  }\n}\n```\n\n### 7.2 Authentication Flow\n- JWT access tokens (15 min expiry) + refresh tokens (7 day expiry, HTTP-only cookie)\n- Access token sent in `Authorization: Bearer <token>` header\n- Refresh endpoint sets new access token and rotates refresh token\n- Social auth: OAuth 2.0 flow with Google/Apple, exchanging provider token for our JWT pair\n\n### 7.3 Real-Time Features\n- WebSocket gateway for:\n  - Real-time availability updates (slot taken/released)\n  - Booking status changes (provider confirms/cancels)\n  - In-app notifications\n- Redis Pub/Sub for cross-instance WebSocket message broadcasting\n\n## 8.
