Overview\nPlanity Clone is a mobile-first marketplace that connects users with local service businesses for discovery, booking, and management of appointments. The architecture follows a clean, layered approach with clear separation of concerns, domain‑driven bounded contexts, and a monorepo strategy powered by Nx to enable code sharing between mobile, API, and shared libraries.\n\n## 2. High‑Level Components\n- **Mobile Client** – Expo (React Native) app using TypeScript, Expo Router, TanStack React Query, React Native Reanimated.\n- **API Server** – NestJS microservice‑style monolith with Prisma ORM, PostgreSQL (extended with PostGIS for geo‑queries), Redis (caching & pub/sub), BullMQ for background jobs.\n- **Infrastructure** – Docker Compose for local dev, GitHub Actions for CI/CD, EAS Build for OTA updates, Supabase (optional auth/storage fallback), Nx for task running and affected computations.\n\n## 3. Bounded Contexts & Service Boundaries\n| Context | Responsibility | Key Entities | Tech / Patterns |
|---------|----------------|--------------|-----------------|
| **Auth** | User registration, login, password reset, JWT issuance, role‑based access (user, provider, admin). | User, Role, Token | NestJS Guard, Passport (local), Prisma, Redis refresh‑token store. |
| **Business** | CRUD of business listings, categories, geo‑search, hours, services. | Business, Category, Service, BusinessHours, Location (PostGIS point). | NestJS Controller/Service, Prisma, PostGIS spatial indexes, Redis cache for frequent queries. |
| **Booking** | Slot computation, appointment creation, modification, cancellation, conflict detection. | Appointment, TimeSlot, ServiceBooking, ProviderSchedule. | Domain service for slot algorithm, BullMQ jobs for reminder notifications, Saga‑like orchestration via NestJS microservices (optional). |
| **Payment** | Payment gateway integration (Stripe/PayPal), webhook handling, refunds. | Payment, Invoice, WebhookLog. | NestJS module, Stripe SDK, idempotency keys, Redis for webhook deduplication. |
| **Notifications** | Push (Expo Push Notifications), email/SMS via third‑party, in‑app alerts. | Notification, DeviceToken. | BullMQ workers, Redis pub/sub, Expo SDK push tool. |
| **Provider Portal** | Business owner dashboard: manage listing, services, schedule, view bookings, respond to reviews. | ProviderProfile, Business (owned), Appointment (filtered). | Separate NestJS module with role‑based guards, shared DTOs with Business context. |
| **Admin** | System oversight: user/business moderation, settings, analytics. | AdminUser, AuditLog, Settings. | NestJS module with RBAC, Prisma, Redis for rate‑limiting. |
| **Reviews & Ratings** | CRUD of reviews, business responses, aggregation of scores. | Review, ReviewResponse. | NestJS service, Redis cache for average scores, periodic recompute job. |
| **Shared** | Common types, constants, UI components, design system, utility hooks, API clients. | – | TypeScript libraries, Expo‑compatible React components, React Native Reanimated animations, shared Zod schemas for validation. |

Communication Contracts\n- **REST/JSON** (NestJS controllers) – versioned under `/api/v1/`.\n- **GraphQL** optional for complex queries (can be added later).\n- **WebSocket** (via NestJS Gateway) for real‑time updates (e.g., new booking, chat).\n- **Expo Push Notifications** – token stored in Redis, sent via Notification worker.\n- **Event‑Driven** – BullMQ jobs publish events (e.g., `appointment.created`) that trigger notification, payment, or review‑request workflows.\n\n## 6. Data Layer\n- **Prisma Schema** defines all entities with relations; PostGIS extension used for `location` field (`Point`).\n- **Indexes**: GIST on location, composite indexes on `(businessId, categoryId)`, `(userId, status)`.\n- **Redis**: \n  - Cache for business list/search results (TTL 5‑15 min).\n  - Token blacklist & refresh‑token store.\n  - Pub/sub channels for real‑time event broadcasting to API instances.\n  - Rate‑limiting sliding window.\n- **Migrations**: Managed via Prisma Migrate; CI runs `prisma migrate deploy` on staging/production.\n\n## 7. State Management & Client‑Side Patterns\n- **React Query** handles server state (fetching, caching, background refetch).\n- **Query keys** are namespaced by context (`['business', id]`, ['bookings', userId]).\n- **Mutations** encapsulate create/update/delete operations with optimistic updates where appropriate (e.g., toggling favorite).\n- **Reanimated 2** used for shared‑element transitions (e.g., business card → detail screen) and gesture‑based slot picker.\n- **Expo Router** provides file‑based navigation; deep linking supported via `expo-linking`.\n- **Authentication** flow: login → store access token in SecureStore, refresh token in Redis (httpOnly cookie alternative). On app load, auth guard validates token via `/auth/me`.\n\n## 8. CI/CD & DevOps\n- **GitHub Actions**:\n  - `ci.yml`: lint (ESLint, Prettier), unit tests (Jest), type check, build mobile (expo export) and API (nest build).\n  - `cd.yml`: on push to `main` → EAS build (iOS/Android) → upload to Expo OTA; Docker build & push API image to registry; trigger Docker Compose deploy on staging.\n- **Docker Compose** (local):\n  - `api`: NestJS container, connects to `db` and `redis`.\n  - `db`: PostgreSQL + PostGIS extension.\n  - `redis`: Standard Redis.\n  - Optional `maildev` for email inspection.\n- **Environment Variables**: managed via `.env` files; Supabase used only as fallback for auth/storage if needed.\n\n## 9. Testing Strategy\n- **Unit Tests**: Jest for service logic, utility functions, and React components (with `@testing-library/react-native`).\n- **Integration Tests**: Supertest for API endpoints; Prisma uses a temporary test database.\n- **E2E**: Detox for critical user flows (login → search → book → cancel).\n- **Test Coverage Goal**: 80%+ for backend, 70%+ for frontend.\n\n## 10. Security Considerations\n- HTTPS everywhere; API behind API gateway (nginx) with rate limiting.\n- JWT access token (15 min) + refresh token (7 days) stored HTTP‑only cookie or SecureStore.\n- Role‑based access control (RBAC) via NestJS guards.\n- Input validation using Zod pipes; output sanitization to prevent XSS.\n- Payment data never touches our servers; only webhook verification.\n- Regular dependency audits (`npm audit`, `nx affected:deps`).\n- Secrets managed via GitHub Actions secrets and Docker secrets.\n\n## 11. Scalability & Performance\n- **Horizontal scaling**: API stateless; Redis session/store enables multiple instances.\n- **Read replicas**: PostgreSQL read replicas for heavy search queries (geo‑search).\n- **Caching**: aggressive caching of business listings, categories, and slot computations.\n- **Background jobs**: BullMQ workers scale independently; failed jobs moved to dead‑letter queue with retry exponential backoff.\n- **CDN**: Expo assets served via Expo’s CDN; static assets (icons) can be offloaded to Supabase Storage.\n\n## 12. Future Extensions\n- Add GraphQL federation for micro‑service split.\n- Introduce WebSocket‑based chat between users and providers.\n- Implement AI‑powered recommendation service (separate Python service).\n- Move to micro‑repo per context if team grows.\n\n---\n*This document provides a concise yet complete blueprint for building Planity Clone with the requested stack, ensuring maintainability, clear boundaries, and a path to production.*\n",
