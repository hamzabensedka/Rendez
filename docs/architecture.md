  "content": "# Planity Clone вЂ” System Architecture Document\n\n## 1. Overview\n\nThis document describes the complete system architecture for the Planity Clone mobile application. The architecture follows clean architecture principles with clear separation of concerns, enabling maintainability, testability, and scalability. The system serves three user types: Customers, Providers (business owners), and Admins.\n\n## 2. Technology Stack\n\n### 2.1 Frontend (Mobile)\n- **Framework:** Expo (React Native) with TypeScript\n- **Routing:** Expo Router (file-based routing)\n- **State Management & Data Fetching:** TanStack React Query\n- **Animations:** React Native Reanimated\n- **Maps:** react-native-maps (Google Maps / Apple Maps)\n- **Authentication:** expo-auth-session, expo-local-authentication\n- **Push Notifications:** Expo Notifications\n- **Testing:** Jest, React Native Testing Library\n\n### 2.2 Backend (Server)\n- **Framework:** NestJS (Node.js) with TypeScript\n- **ORM:** Prisma (with PostgreSQL)\n- **Geospatial:** PostGIS extension\n- **Caching:** Redis\n- **Authentication:** JWT (access + refresh tokens), OAuth 2.0 / OpenID Connect\n- **Real-time:** WebSockets (Socket.IO)\n- **Testing:** Jest, Supertest\n\n### 2.3 Infrastructure & DevOps\n- **Monorepo:** Nx with pnpm\n- **Containerization:** Docker Compose\n- **CI/CD:** GitHub Actions\n- **Mobile Builds:** EAS Build\n- **Backend Hosting:** Supabase (PostgreSQL + PostGIS, Auth, Storage)\n- **Search Index:** Algolia (or Elasticsearch)\n\n## 3. Service Boundaries & Modules\n\n### 4.1 Authentication Module (`auth`)\n- **Responsibilities:** User registration, login, token management, password reset, OAuth social login.\n- **Endpoints:** POST /auth/register, POST /auth/login, POST /auth/refresh, POST /auth/logout, POST /auth/reset-password, POST /auth/oauth/:provider\n- **Data:** Users table, refresh_tokens table.\n- **Dependencies:** Redis (token blacklist), Supabase Auth (optional).\n\n### 4.2 User Module (`users`)\n- **Responsibilities:** Profile CRUD, preferences, payment methods.\n- **Endpoints:** GET /users/me, PATCH /users/me, DELETE /users/me, GET /users/me/payment-methods, etc.\n- **Data:** Users table, payment_methods table.\n\n### 4.3 Business Module (`businesses`)\n- **Responsibilities:** Business CRUD, staff management, hours, photos, amenities.\n- **Endpoints:** GET /businesses, GET /businesses/:id, POST /businesses (provider), PATCH /businesses/:id, etc.\n- **Data:** Businesses table, staff table, business_hours table, business_photos table.\n- **Geospatial:** PostGIS for location queries.\n\n### 4.4 Service Module (`services`)\n- **Responsibilities:** Service CRUD, category assignment, pricing, duration.\n- **Endpoints:** GET /services, GET /services/:id, POST /services, etc.\n- **Data:** Services table, service_categories table.\n\n### 4.5 Booking Module (`bookings`)\n- **Responsibilities:** Booking flow, slot computation, hold management, cancellation, state machine.\n- **Endpoints:** GET /bookings/slots, POST /bookings, GET /bookings/:id, PATCH /bookings/:id/cancel, etc.\n- **Data:** Bookings table, booking_holds table, audit_log table.\n- **Dependencies:** Redis (slot locks, holds), WebSocket (real-time updates).\n\n### 4.6 Search Module (`search`)\n- **Responsibilities:** Full-text search, autocomplete, filters, sorting, geospatial search.\n- **Endpoints:** GET /search, GET /search/autocomplete, GET /search/trending\n- **Data:** Algolia index (synced with PostgreSQL via webhook or cron).\n- **Dependencies:** Algolia, PostGIS.\n\n### 4.7 Reviews Module (`reviews`)\n- **Responsibilities:** Review CRUD, rating aggregation, histogram.\n- **Endpoints:** GET /businesses/:id/reviews, POST /businesses/:id/reviews, etc.\n- **Data:** Reviews table.\n\n### 4.8 Favorites Module (`favorites`)\n- **Responsibilities:** Add/remove favorites, list favorites.\n- **Endpoints:** POST /favorites, DELETE /favorites/:businessId, GET /favorites\n- **Data:** Favorites table (many-to-many).\n\n### 4.9 Notifications Module (`notifications`)\n- **Responsibilities:** Push notifications, email, SMS.\n- **Endpoints:** POST /notifications/send (admin), GET /notifications/preferences\n- **Dependencies:** Expo Push API, SendGrid (email), Twilio (SMS).\n\n### 4.10 Admin Module (`admin`)\n- **Responsibilities:** Platform oversight, user management, analytics, category management.\n- **Endpoints:** Admin-specific CRUD for users, businesses, categories.\n\n## 5. Key Architectural Decisions\n\n### 6.1 Monorepo with Nx\n- **Rationale:** Shared types and utilities between mobile and API. Consistent build and test commands. Code generation.\n- **Tool:** Nx with pnpm workspaces.\n\n### 6.2 Expo Router for Navigation\n- **Rationale:** File-based routing simplifies navigation structure. Supports deep linking and web fallback.\n\n### 6.3 TanStack React Query for Data Fetching\n- **Rationale:** Caching, background refetching, optimistic updates, and pagination out of the box. Reduces boilerplate.\n\n### 6.4 NestJS with Prisma\n- **Rationale:** Modular architecture, dependency injection, and decorators align with clean architecture. Prisma provides type-safe database access and migrations.\n\n### 6.5 PostGIS for Geospatial Queries\n- **Rationale:** Native support for distance calculations, bounding box queries, and spatial indexing.\n\n### 6.6 Redis for Caching and Real-time\n- **Rationale:** Caching popular queries, session store, rate limiting, WebSocket pub/sub, and booking slot locks.\n\n### 6.7 Algolia for Search\n- **Rationale:** Fast typo-tolerant search, faceted filtering, and geospatial search. Offloads complex search from PostgreSQL.\n\n### 6.8 Supabase for Backend Services\n- **Rationale:** Managed PostgreSQL with PostGIS, built-in auth, storage for images, and real-time subscriptions.\n\n### 6.9 EAS Build for Mobile CI/CD\n- **Rationale:** Cloud-based builds for Expo apps. Supports OTA updates via EAS Update.\n\n## 7. API Design\n\n### 7.1 RESTful Conventions\n- Base URL: `/api/v1`\n- Authentication: Bearer JWT token in `Authorization` header.\n- Pagination: Cursor-based with `cursor` and `limit` query params.\n- Responses: Standard envelope `{ data, meta, error }`.\n\n### 7.2 WebSocket Events\n- `booking.slot.update`: Real-time slot availability changes.\n- `booking.created`: Notify provider of new booking.\n- `booking.cancelled`: Notify both parties.\n- `notification`: Push notification trigger.\n\n## 8. Security\n\n- **Authentication:** JWT access tokens (15 min) + refresh tokens (7 days, hashed in DB).\n- **Rate Limiting:** 5 attempts/minute for login, 1000 requests/minute per user.\n- **Input Validation:** Class-validator with NestJS pipes.\n- **SQL Injection:** Prisma parameterized queries.\n- **XSS:** Sanitize user input.\n- **CORS:** Whitelist mobile app origins.\n- **Data Encryption:** HTTPS only. Sensitive data encrypted at rest.\n\n## 9. Performance & Scalability\n\n- **Caching:** Redis for business details, category tree, popular searches (TTL 5 min).\n- **Database Indexing:** Index on `business.location` (GIST), `booking.startTime`, `user.email`.\n- **Search:** Algolia for full-text search, PostGIS for geospatial.\n- **Image Optimization:** Use CDN (Supabase Storage with image transformations).\n- **Lazy Loading:** Infinite scroll for search results and reviews.\n- **Concurrent Booking:** Optimistic locking with 5-minute hold in Redis.\n\n## 10. Testing Strategy\n\n- **Unit Tests:** Jest for services, utilities, and hooks.\n- **Integration Tests:** Supertest for API endpoints with test database.\n- **E2E Tests:** Detox or Maestro for mobile flows.\n- **CI Pipeline:** GitHub Actions runs lint, type check, unit tests, integration tests on PR.\n\n## 11. Deployment\n\n- **Backend:** Docker Compose with NestJS, PostgreSQL, Redis. Deployed to VPS or cloud (AWS/GCP).\n- **Mobile:** EAS Build for iOS/Android binaries. EAS Update for OTA updates.\n- **Database:** Supabase managed PostgreSQL with automated backups.\n- **CI/CD:** GitHub Actions for automated testing and deployment.\n\n## 12. Monitoring & Observability\n\n- **Logging:** NestJS Logger with structured JSON output.\n- **Metrics:** Prometheus + Grafana (optional).\n- **Error Tracking:** Sentry for both backend and mobile.\n- **Performance:** React Native Performance Monitor, Lighthouse for web.\n\n## 13.
