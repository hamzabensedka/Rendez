Overview\n\nThis document describes the complete system architecture for Planity Clone, a two-sided marketplace connecting consumers with local service businesses for online appointment booking. The architecture prioritizes clean separation of concerns, scalability, maintainability, and a delightful mobile experience.\n\n## 2. Technology Stack\n\n| Layer | Technology | Rationale |\n|-------|------------|-----------|\n| Mobile Framework | Expo (React Native) + TypeScript | Cross-platform, fast iteration, OTA updates |\n| Navigation | Expo Router | File-based routing, deep linking, type-safe |\n| State & Data | TanStack React Query | Server state caching, optimistic updates, pagination |\n| Animations | React Native Reanimated | 60fps UI interactions, gesture handling |\n| Backend | NestJS + TypeScript | Modular, opinionated, decorator-based architecture |\n| ORM | Prisma | Type-safe database access, migrations, PostGIS support |\n| Database | PostgreSQL + PostGIS | Relational integrity, spatial queries |\n| Cache & Session | Redis | Session store, rate limiting, job queues |\n| Build & CI | Nx + pnpm + GitHub Actions | Monorepo management, fast builds, CI/CD |\n| Deployment | Docker Compose, Supabase (Auth/Storage), EAS Build | Local dev parity, managed services, mobile builds |\n| Testing | Jest | Unit, integration, E2E |\n\n## 3. High-Level Architecture\n\nThe system follows a **layered microservices** approach with a monorepo structure. The mobile app communicates with a NestJS API Gateway, which routes to domain-specific services. Shared libraries handle cross-cutting concerns.\n\n```\n[Mobile App (Expo)]\n       |\n[API Gateway (NestJS)]\n       |\n[Microservices: Auth, Search, Booking, Business, Notification]\n       |\n[PostgreSQL + PostGIS] [Redis] [Supabase Auth]\n```\n\n## 4. Folder Structure\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   │   ├── (auth)/            # Auth screens (login, register)\n│   │   │   ├── (tabs)/            # Main tab navigation\n│   │   │   ├── business/          # Business detail, booking\n│   │   │   └── _layout.tsx        # Root layout\n│   │   ├── components/            # Shared UI components\n│   │   ├── hooks/                 # Custom hooks (useAuth, useSearch)\n│   │   ├── services/              # API client, TanStack Query hooks\n│   │   ├── utils/                 # Helpers, formatters\n│   │   ├── types/                 # TypeScript types\n│   │   └── app.json\n│   └── api/                       # NestJS backend\n│       ├── src/\n│       │   ├── auth/              # Auth module (JWT, OAuth)\n│       │   ├── search/            # Search module (Elasticsearch/PostGIS)\n│       │   ├── booking/           # Booking module\n│       │   ├── business/          # Business module\n│       │   ├── notification/      # Push/SMS/Email\n│       │   ├── common/            # Shared decorators, guards, filters\n│       │   ├── config/            # Environment config\n│       │   └── main.ts\n│       └── prisma/\n│           ├── schema.prisma\n│           └── migrations/\n├── libs/\n│   ├── shared-types/              # Shared DTOs, enums\n│   ├── api-client/                # Axios/fetch wrapper\n│   └── ui-kit/                    # Reusable React Native components\n├── tools/\n│   ├── scripts/                   # DB seed, codegen\n│   └── generators/                # Nx generators\n├── docker-compose.yml\n├── nx.json\n├── package.json\n├── pnpm-workspace.yaml\n└── .github/workflows/\n    ├── ci.yml\n    └── deploy.yml\n```\n\n## 5. Service Boundaries\n\n### 5.1 Auth Service\n- **Responsibility:** User registration, login, OAuth, token management, session control.\n- **Data:** Users, roles, refresh tokens.\n- **Key APIs:** POST /auth/register, POST /auth/login, POST /auth/refresh, POST /auth/logout.\n- **Dependencies:** Supabase Auth (OAuth providers), Redis (session store).\n\n### 5.2 Search Service\n- **Responsibility:** Business search, autocomplete, map-based discovery, spatial queries.\n- **Data:** Business locations (PostGIS), search index (PostgreSQL full-text + PostGIS).\n- **Key APIs:** GET /search, GET /search/autocomplete, GET /search/map.\n- **Dependencies:** PostgreSQL with PostGIS, Redis (caching).\n\n### 5.3 Booking Service\n- **Responsibility:** Slot management, reservation, payment integration, booking lifecycle.\n- **Data:** Appointments, time slots, payments.\n- **Key APIs:** POST /bookings, GET /bookings, PATCH /bookings/:id, DELETE /bookings/:id.\n- **Dependencies:** Redis (pessimistic locking), PostgreSQL.\n\n### 5.4 Business Service\n- **Responsibility:** Business profiles, services, staff, hours, reviews.\n- **Data:** Businesses, services, staff, reviews, categories.\n- **Key APIs:** GET /businesses/:id, POST /businesses, PATCH /businesses/:id.\n- **Dependencies:** PostgreSQL.\n\n### 5.5 Notification Service\n- **Responsibility:** Push notifications, SMS, email, in-app alerts.\n- **Data:** Notification templates, delivery logs.\n- **Key APIs:** POST /notifications/send, GET /notifications.\n- **Dependencies:** Redis (job queue), Supabase (push).\n\n## 6. Data Flow Examples\n\n### 6.1 Guest Browse & Book\n1. Guest opens app → Expo Router loads home page.\n2. Search Service returns businesses (no auth required).\n3. Guest taps \"Book\" → Auth modal appears with \"Continue as Guest\" option.\n4. Guest enters name, phone, email → Booking Service creates pending booking.\n5. After booking, Auth Service auto-creates account with temporary password.\n6. Notification Service sends confirmation SMS/email.\n\n### 6.2 Authenticated Booking\n1. User logs in → Auth Service returns JWT (stored in SecureStore).\n2. User searches → TanStack React Query caches results, updates on map pan.\n3. User selects service, time → Booking Service acquires Redis lock (5 min).\n4. Payment processed → Booking confirmed, lock released.\n5. Optimistic UI updates immediately; rollback on failure.\n\n## 7. Key Design Decisions\n\n- **Monorepo with Nx:** Shared types, API client, and UI kit reduce duplication and enforce consistency.\n- **Expo Router:** File-based routing simplifies navigation and supports deep linking for business profiles.\n- **TanStack React Query:** Automatic caching, background refetching, and optimistic updates reduce boilerplate.\n- **PostGIS:** Enables efficient spatial queries (find businesses within radius, cluster markers).\n- **Redis:** Used for session storage, rate limiting, and pessimistic locking to prevent double-booking.\n- **Supabase Auth:** Handles OAuth providers, email verification, and biometric unlock out of the box.\n- **Docker Compose:** Ensures local development environment matches production (PostgreSQL, Redis, etc.).\n\n## 8. Security & Performance\n\n- **Authentication:** JWT with short-lived access tokens (15 min) and refresh tokens (7 days). Biometric unlock on mobile after inactivity.\n- **Rate Limiting:** Redis-based, 100 requests/min per user, 1000/min per IP.\n- **Caching:** Redis caches search results (5 min TTL), business details (10 min), and slot availability (1 min).\n- **Database Indexing:** Indexes on business name (gin for full-text), location (gist for PostGIS), appointment time.\n- **Error Handling:** Global NestJS exception filter, structured error responses, Sentry integration.\n\n## 9. Testing Strategy\n\n- **Unit Tests:** Jest for services, guards, and utilities.\n- **Integration Tests:** Supertest for API endpoints with test containers (PostgreSQL, Redis).\n- **E2E Tests:** Detox for mobile flows (booking, search).\n- **CI Pipeline:** GitHub Actions runs lint, type-check, unit tests, integration tests on PR.\n\n## 10. Deployment\n\n- **Mobile:** EAS Build for iOS/Android builds, OTA updates via Expo.\n- **Backend:** Docker Compose to staging/production, managed PostgreSQL (Supabase).\n- **CI/CD:** GitHub Actions builds Docker images, runs migrations, deploys to staging on merge to main.\n\n## 11. Conclusion\n\nThis architecture provides a solid foundation for building a scalable, maintainable Planity Clone. The separation into domain services, use of modern tooling (Expo, NestJS, Prisma), and emphasis on clean code practices will support rapid feature development and future growth.",
