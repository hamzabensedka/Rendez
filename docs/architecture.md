  "content": "# Planity Clone — System Architecture Document\n\n## 1. Overview\n\nThis document describes the complete system architecture for the Planity Clone, a mobile-first appointment booking platform. The architecture follows clean architecture principles, emphasizing separation of concerns, maintainability, and scalability. The system is built using a monorepo structure managed by Nx and pnpm, with a React Native (Expo) frontend and a NestJS backend.\n\n## 2. Technology Stack\n\n| Layer | Technology | Rationale |\n|-------|------------|-----------|\n| **Mobile/Web** | Expo, React Native, TypeScript | Cross-platform, fast iteration, OTA updates |\n| **Routing** | Expo Router | File-based routing, deep linking support |\n| **State & Data** | TanStack React Query | Server state caching, optimistic updates, pagination |\n| **Animations** | React Native Reanimated | 60fps UI animations, gesture handling |\n| **Backend** | NestJS, TypeScript | Modular, testable, decorator-based architecture |\n| **ORM** | Prisma | Type-safe database access, migrations, PostGIS support |\n| **Database** | PostgreSQL + PostGIS | Relational data, geospatial queries for map search |\n| **Cache & Queue** | Redis | Session store, rate limiting, job queues, real-time availability |\n| **Build** | Nx + pnpm | Monorepo management, dependency caching, parallel tasks |\n| **CI/CD** | GitHub Actions | Automated testing, linting, EAS builds |\n| **Deployment** | Docker Compose, Supabase (auth), EAS Build | Local dev parity, managed auth, mobile builds |\n| **Testing** | Jest | Unit, integration, E2E testing |\n\n## 3. Architecture Principles\n\n- **Clean Architecture**: Domain, application, infrastructure, and presentation layers are strictly separated.\n- **Monorepo**: Single source of truth for all code, shared types, and utilities.\n- **API-First**: Backend defines contracts; frontend consumes via generated types.\n- **Offline-First**: Critical flows (browsing, booking) work with stale data and sync when online.\n- **Event-Driven**: Asynchronous communication for non-critical operations (notifications, analytics).\n- **Security**: JWT with refresh tokens, rate limiting, input validation, CORS.\n\n## 4. Folder Structure\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   ├── components/            # Shared UI components\n│   │   ├── features/              # Feature-specific modules\n│   │   ├── hooks/                 # Custom hooks\n│   │   ├── providers/             # Context providers\n│   │   ├── services/              # API client, storage\n│   │   ├── types/                 # Generated types\n│   │   ├── utils/                 # Helpers\n│   │   └── app.json\n│   ├── api/                       # NestJS backend\n│   │   ├── src/\n│   │   │   ├── modules/           # Feature modules (auth, booking, etc.)\n│   │   │   ├── common/            # Shared guards, filters, interceptors\n│   │   │   ├── config/            # Environment config\n│   │   │   ├── database/          # Prisma service, migrations\n│   │   │   └── main.ts\n│   │   └── prisma/\n│   │       ├── schema.prisma\n│   │       └── migrations/\n│   └── web/                       # Optional web app (Next.js)\n├── packages/\n│   ├── shared/                    # Shared types, constants, validation schemas\n│   │   ├── types/\n│   │   ├── validators/\n│   │   └── constants.ts\n│   ├── ui/                        # Design system components (React Native)\n│   ├── api-client/                # Axios instance, interceptors, generated API\n│   └── eslint-config/             # Shared ESLint config\n├── tools/                         # Scripts, generators\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n├── tsconfig.base.json\n└── .github/\n    └── workflows/\n        ├── ci.yml\n        └── deploy.yml\n```\n\n## 5. Service Boundaries\n\nThe system is decomposed into the following modules (NestJS modules in `apps/api/src/modules/`):\n\n### 5.1 Auth Module\n- **Responsibility**: Registration, login, OAuth, token management, password reset.\n- **Dependencies**: Supabase (auth provider), Redis (session store), Prisma (user profiles).\n- **Key Entities**: User, Session, OAuthProvider.\n\n### 5.2 Business Module\n- **Responsibility**: CRUD for businesses, categories, services, staff, hours, images.\n- **Dependencies**: Prisma (PostgreSQL + PostGIS), Redis (cache for search).\n- **Key Entities**: Business, Category, Service, Staff, BusinessHour, BusinessImage.\n\n### 5.3 Search Module\n- **Responsibility**: Full-text search, geospatial queries, autocomplete, filtering.\n- **Dependencies**: PostgreSQL (full-text search + PostGIS), Redis (cached results).\n- **Key Endpoints**: `GET /search`, `GET /search/autocomplete`.\n\n### 5.4 Booking Module\n- **Responsibility**: Slot computation, booking creation, holding slots, cancellation.\n- **Dependencies**: Prisma, Redis (slot locks, temporary holds), queue (email/SMS).\n- **Key Entities**: Booking, BookingSlot, BookingAddon.\n\n### 5.5 Payment Module\n- **Responsibility**: Payment processing, refunds, commission calculation.\n- **Dependencies**: Stripe (or similar), Prisma (transactions), Redis (idempotency keys).\n- **Key Entities**: Payment, Transaction, Commission.\n\n### 5.6 Notification Module\n- **Responsibility**: Push notifications, SMS, email reminders.\n- **Dependencies**: Queue (Bull with Redis), Firebase Cloud Messaging, Twilio/SendGrid.\n- **Key Entities**: Notification, NotificationTemplate.\n\n### 5.7 Review Module\n- **Responsibility**: Reviews, ratings, moderation.\n- **Dependencies**: Prisma, Redis (cached average rating).\n- **Key Entities**: Review, ReviewImage.\n\n### 5.8 Admin Module\n- **Responsibility**: User management, business moderation, analytics, platform settings.\n- **Dependencies**: Prisma, Redis (analytics cache).\n\n## 6. Data Flow & Key Interactions\n\n### 6.1 Booking Flow\n1. **Customer** searches for services → `Search Module` returns results.\n2. **Customer** selects business → `Business Module` provides details.\n3. **Customer** selects date/time → `Booking Module` computes available slots (reads staff schedules, existing bookings, buffer times from DB; caches slots in Redis for 30s).\n4. **Customer** proceeds to checkout → `Booking Module` creates a temporary hold in Redis (10 min TTL).\n5. **Customer** completes payment → `Payment Module` processes transaction; on success, `Booking Module` creates permanent booking, releases hold, enqueues confirmation notification.\n6. **Notification Module** sends push/SMS/email.\n\n### 6.2 Search Flow\n1. **User** types query → debounced 300ms → `Search Module` checks Redis cache.\n2. If miss, query PostgreSQL using `tsvector` for text and PostGIS `ST_DWithin` for distance.\n3. Results are cached in Redis with TTL (5 min for general, 1 min for location-based).\n4. Response includes pagination metadata.\n\n## 7. Database Schema (Prisma)\n\nKey models (simplified):\n\n```prisma\nmodel User {\n  id        String   @id @default(uuid())\n  email     String   @unique\n  name      String?\n  phone     String?\n  role      Role     @default(CUSTOMER)\n  createdAt DateTime @default(now())\n  updatedAt DateTime @updatedAt\n  bookings  Booking[]\n  reviews   Review[]\n}\n\nmodel Business {\n  id          String   @id @default(uuid())\n  name        String\n  description String?\n  categoryId  String\n  location    Geography  // PostGIS point\n  rating      Float    @default(0)\n  createdAt   DateTime @default(now())\n  services    Service[]\n  staff       Staff[]\n  hours       BusinessHour[]\n  bookings    Booking[]\n  reviews     Review[]\n}\n\nmodel Service {\n  id          String   @id @default(uuid())\n  businessId  String\n  name        String\n  duration    Int      // minutes\n  price       Decimal\n  categoryId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  bookings    BookingService[]\n}\n\nmodel Booking {\n  id          String   @id @default(uuid())\n  userId      String\n  businessId  String\n  staffId     String?\n  startTime   DateTime\n  endTime     DateTime\n  status      BookingStatus @default(PENDING)\n  totalPrice  Decimal\n  createdAt   DateTime @default(now())\n  user        User     @relation(fields: [userId], references: [id])\n  business    Business @relation(fields: [businessId], references: [id])\n  services    BookingService[]\n  payment     Payment?\n}\n```\n\n## 8. API Design\n\nRESTful API with versioning (`/api/v1/`). Key endpoints:\n\n| Method | Endpoint | Description |\n|--------|----------|-------------|\n| POST | /auth/register | Register new user |\n| POST | /auth/login | Login |\n| POST | /auth/refresh | Refresh token |\n| GET | /businesses | List businesses (with filters, pagination) |\n| GET | /businesses/:id | Business detail |\n| GET | /search | Full-text search |\n| GET | /search/autocomplete | Autocomplete suggestions |\n| GET | /businesses/:id/slots | Available slots (date, staff) |\n| POST | /bookings | Create booking |\n| GET | /bookings/me | User's bookings |\n| PATCH | /bookings/:id/cancel | Cancel booking |\n| POST | /payments/intent | Create payment intent |\n| POST | /webhooks/stripe | Stripe webhook |\n\n## 9. Security & Performance\n\n- **Authentication**: JWT access tokens (15 min) + refresh tokens (30 days) stored in secure storage (expo-secure-store).\n- **Rate Limiting**: Per endpoint, per user/IP using Redis (e.g., 100 req/min for search, 5 req/min for login).\n- **Input Validation**: Zod schemas shared between frontend and backend via `packages/shared`.\n- **CORS**: Whitelist allowed origins.\n- **SQL Injection**: Prevented by Prisma parameterized queries.\n- **XSS**: Sanitize user-generated content.\n- **Performance**: Redis caching for search results, slot availability, business details. Database indexes on `location`, `tsvector`, `userId`, `businessId`, `startTime`.\n\n## 10. Deployment & CI/CD\n\n### 10.1 Local Development\n```bash\ndocker-compose up -d  # PostgreSQL, Redis, Supabase local\ndocker-compose up     # NestJS backend\ncd apps/mobile && npx expo start\n```\n\n### 10.2 CI/CD Pipeline (GitHub Actions)\n1. **Lint & Type Check**: ESLint, Prettier, TypeScript.\n2. **Test**: Jest unit + integration tests (with test containers).\n3. **Build**: Nx affected:build for changed projects.\n4. **Deploy**:\n   - Backend: Docker image pushed to registry, deployed to cloud (AWS ECS/Google Cloud Run).\n   - Mobile: EAS Build for iOS/Android, submit to stores.\n\n### 10.3 Environment Variables\nManaged via `.env` files (not committed). Backend uses `@nestjs/config` with validation.\n\n## 11. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions, services, hooks.\n- **Integration Tests**: Test NestJS modules with real database (testcontainers).\n- **E2E Tests**: Detox (mobile) + Playwright (web) for critical flows (search, book, cancel).\n- **API Tests**: Supertest for endpoint validation.\n\n## 12. Error Handling & Logging\n\n- **Backend**: Global exception filter returning structured errors (`{ statusCode, message, errors }`). Logging with Winston.\n- **Frontend**: TanStack Query error handling with retry logic; user-friendly error toasts.\n- **Monitoring**: Sentry for error tracking, Datadog for APM.\n\n## 13. Conclusion\n\nThis architecture provides a scalable, maintainable foundation for the Planity Clone. The monorepo structure ensures code reuse and consistency, while the modular backend allows independent scaling of features. The use of modern tooling (Expo, NestJS, Prisma, Redis) ensures high performance and developer productivity.",
