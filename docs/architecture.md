Service Boundaries\n\nThe system is decomposed into logical modules (services) within the NestJS backend, each with its own responsibility:\n\n### 4.1 Auth Service\n- Handles registration, login, OAuth, phone verification\n- JWT token generation & refresh token rotation\n- Rate limiting, CAPTCHA integration\n- Biometric login support (device-side)\n\n### 4.2 User Service\n- User profile CRUD (customers, providers, admins)\n- Account linking (email/social merge)\n- Preferences & settings\n\n### 4.3 Business Service\n- Business profile CRUD (providers only)\n- Categories, amenities, photos\n- Verification badge logic\n\n### 4.4 Service Catalog Service\n- Service definitions per business\n- Categories & subcategories\n- Pricing, duration, staff assignment\n\n### 4.5 Availability Service\n- Weekly recurring schedules\n- Slot generation (time intervals)\n- Real-time slot locking (10-min hold)\n- Conflict detection (optimistic locking via Redis)\n\n### 4.6 Booking Service\n- Booking lifecycle (create, confirm, cancel, modify)\n- Payment integration (stripe, pay-at-venue)\n- Calendar sync (iCal)\n- Notification triggers\n\n### 4.7 Search Service\n- Full-text search with typo tolerance (PostgreSQL tsvector + pg_trgm)\n- Geospatial queries (PostGIS)\n- Auto-complete suggestions (Redis sorted sets)\n- Filtering & sorting\n\n### 4.8 Review Service\n- Review CRUD\n- Rating aggregation\n- Photo reviews\n- Abuse detection\n\n### 4.9 Notification Service\n- Push notifications (Expo Push API)\n- Email (SendGrid/Mailgun)\n- SMS (Twilio)\n- In-app notification center\n\n### 4.10 Payment Service\n- Payment processing (Stripe)\n- Refunds\n- Payouts to providers\n- Payment method management\n\n### 4.11 Admin Service\n- Platform analytics\n- User moderation\n- Business approval\n- System health monitoring\n\n## 5. Data Flow & Communication\n\n### 5.1 Mobile Client to Backend\n- All API calls go through a centralized API client (axios) with interceptors for token refresh.\n- TanStack React Query manages server state caching, pagination, and optimistic updates.\n- WebSocket connection (Socket.IO) for real-time notifications and slot updates.\n\n### 5.2 Backend to Database\n- Prisma ORM handles all database interactions.\n- Read replicas for heavy search queries (optional, future).\n- PostGIS for geospatial queries (distance, bounding box).\n\n### 5.3 Caching Strategy\n- Redis caches:\n  - User sessions (refresh tokens)\n  - Slot availability (per business, per day)\n  - Search auto-complete suggestions\n  - Rate limit counters\n  - Booking holds (10-min TTL)\n- Cache invalidation: on booking creation, schedule update, or explicit admin action.\n\n### 5.4 Event-Driven Communication\n- NestJS event emitter for internal module communication (e.g., booking created → notification sent).\n- Future: message queue (RabbitMQ) for async tasks (email, SMS, analytics).\n\n## 6. Key Architectural Decisions\n\n### 6.1 Monorepo with Nx\n- Single source of truth for shared types (libs/shared).\n- Consistent build and test tooling.\n- Dependency graph optimization.\n\n### 6.2 Expo Router for Navigation\n- File-based routing simplifies navigation structure.\n- Deep linking support out of the box.\n- Type-safe routes.\n\n### 6.3 TanStack React Query for Server State\n- Automatic caching and background refetching.\n- Optimistic updates for booking flow (instant UI feedback).\n- Pagination and infinite queries for search results.\n\n### 6.4 Prisma + PostgreSQL + PostGIS\n- Type-safe database access.\n- Migrations as code.\n- Geospatial queries for map-based search.\n\n### 6.5 Redis for Real-Time Slot Management\n- In-memory speed for slot holds and availability checks.\n- Prevents double-booking with atomic operations.\n\n### 6.6 Supabase for Auth & Realtime\n- Built-in authentication (email, OAuth, phone).\n- Realtime subscriptions for live slot updates.\n- Row-Level Security (RLS) for data access control.\n\n## 7. Security Considerations\n\n- JWT access tokens (15-min expiry) stored in secure device storage (expo-secure-store).\n- Refresh tokens (30-day) rotated on each use, stored in httpOnly cookies (web) or secure storage (mobile).\n- Rate limiting on auth endpoints (5 attempts → 15-min lockout).\n- CAPTCHA after 3 failed attempts.\n- Input validation via Zod schemas (shared between frontend and backend).\n- SQL injection prevention via Prisma parameterized queries.\n- HTTPS enforced.\n- Environment variables managed via .env files (not committed).\n\n## 8. Performance Targets\n\n- API response time: < 200ms for 95th percentile (excluding slow queries).\n- Search auto-complete: < 200ms.\n- Full search results: < 500ms.\n- Slot generation: < 100ms per business per day.\n- Booking confirmation: < 1s end-to-end.\n- App cold start: < 3s on mid-range devices.\n\n## 9. Testing Strategy\n\n- **Unit tests:** Jest for all services, hooks, and utilities.\n- **Integration tests:** NestJS testing utilities with test database (Docker Compose).\n- **E2E tests:** Detox (React Native) for critical flows (booking, search).\n- **CI pipeline:** GitHub Actions runs tests on every PR.\n\n## 10. Deployment & DevOps\n\n- **Local development:** Docker Compose (PostgreSQL, Redis, API).\n- **Mobile builds:** EAS Build (Expo Application Services).\n- **Backend hosting:** AWS ECS or Railway (auto-scaling).\n- **Database:** Managed PostgreSQL (AWS RDS or Supabase).\n- **CI/CD:** GitHub Actions for linting, testing, building, and deploying.\n- **Monitoring:** Sentry (error tracking), Datadog (APM).\n\n## 11. Future Considerations\n\n- Microservices decomposition if monolith grows too large.\n- Message queue (RabbitMQ) for async tasks.\n- GraphQL federation for complex queries.\n- Multi-region deployment for latency reduction.\n- Offline-first capabilities with local database (WatermelonDB).\n\n## 12. Conclusion\n\nThis architecture provides a solid foundation for the Planity Clone, balancing rapid development with production-grade scalability.
