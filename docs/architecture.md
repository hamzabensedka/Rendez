  "content": "# Planity Clone — Architecture Document\n\n## 1. Overview\n\nThis document describes the system architecture for the Planity Clone, a mobile-first platform connecting consumers with beauty/wellness businesses. The architecture follows clean architecture principles, separation of concerns, and maintainability. The system is built using a monorepo structure with Nx, pnpm, and Docker Compose for local development.\n\n## 2. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile App | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated | Cross-platform mobile client |\n| Backend | NestJS, TypeScript | RESTful API server |\n| Database ORM | Prisma | Type-safe database access |\n| Database | PostgreSQL with PostGIS extension | Relational data + geospatial queries |\n| Cache & Session | Redis | Caching, rate limiting, session store |\n| CI/CD | GitHub Actions, EAS Build | Automated testing and deployment |\n| Testing | Jest | Unit and integration tests |\n| Infrastructure | Docker Compose, Supabase | Local development and managed services |\n\n## 3. Monorepo Structure\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   ├── components/            # Shared UI components\n│   │   ├── hooks/                 # Custom hooks\n│   │   ├── services/              # API client, auth service\n│   │   ├── utils/                 # Helper functions\n│   │   ├── types/                 # TypeScript types\n│   │   └── app.json               # Expo config\n│   └── api/                       # NestJS backend\n│       ├── src/\n│       │   ├── modules/           # Feature modules\n│       │   ├── common/            # Shared decorators, guards, filters\n│       │   ├── config/            # Configuration\n│       │   └── main.ts            # Entry point\n│       └── prisma/                # Prisma schema and migrations\n├── packages/\n│   ├── shared-types/              # Shared TypeScript types\n│   ├── ui-kit/                    # Reusable UI components (if needed)\n│   └── eslint-config/             # Shared ESLint config\n├── tools/\n│   ├── generators/                # Nx generators\n│   └── scripts/                   # Build and deploy scripts\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n└── .github/\n    └── workflows/                 # GitHub Actions\n```\n\n## 4. Service Boundaries\n\nThe backend is organized into feature modules, each with its own controller, service, and data access layer.\n\n### 4.1 Auth Module\n- **Responsibility**: User registration, login, OAuth, password reset, phone verification, token management.\n- **Endpoints**: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/forgot-password`, `POST /auth/reset-password`, `POST /auth/verify-phone`, `POST /auth/oauth/{provider}`.\n- **Data**: Users table, refresh tokens table, OTP codes.\n\n### 4.2 User Module\n- **Responsibility**: User profile management, preferences, favorites, search history.\n- **Endpoints**: `GET /users/me`, `PATCH /users/me`, `GET /users/me/favorites`, `POST /users/me/favorites`, `DELETE /users/me/favorites/{id}`, `GET /users/me/search-history`.\n- **Data**: Users table, favorites table, search history table.\n\n### 4.3 Business Module\n- **Responsibility**: Business CRUD, search, geospatial queries, categories, operating hours, photos, amenities.\n- **Endpoints**: `GET /businesses`, `GET /businesses/search`, `GET /businesses/nearby`, `GET /businesses/{id}`, `POST /businesses`, `PATCH /businesses/{id}`, `GET /businesses/{id}/services`, `GET /businesses/{id}/staff`, `GET /businesses/{id}/availability`.\n- **Data**: Businesses table (with PostGIS geometry), categories, services, staff, operating hours, photos, amenities.\n\n### 4.4 Service Module\n- **Responsibility**: Service definitions, categories, pricing, duration.\n- **Endpoints**: `GET /services`, `GET /services/{id}`, `GET /categories`, `GET /categories/{id}/subcategories`.\n- **Data**: Services table, categories table, subcategories table.\n\n### 4.5 Booking Module\n- **Responsibility**: Appointment creation, modification, cancellation, waitlist, availability.\n- **Endpoints**: `POST /bookings`, `GET /bookings`, `GET /bookings/{id}`, `PATCH /bookings/{id}`, `DELETE /bookings/{id}`, `POST /bookings/{id}/reschedule`, `POST /bookings/{id}/cancel`, `POST /bookings/{id}/waitlist`, `GET /bookings/availability`.\n- **Data**: Bookings table, booking services table, waitlist table.\n\n### 4.6 Payment Module\n- **Responsibility**: Payment processing, refunds, promo codes.\n- **Endpoints**: `POST /payments/charge`, `POST /payments/refund`, `POST /payments/promo/validate`.\n- **Data**: Payments table, promo codes table.\n\n### 4.7 Review Module\n- **Responsibility**: Reviews and ratings for businesses and staff.\n- **Endpoints**: `POST /reviews`, `GET /reviews`, `GET /reviews/{id}`, `PATCH /reviews/{id}`, `DELETE /reviews/{id}`.\n- **Data**: Reviews table.\n\n### 4.8 Notification Module\n- **Responsibility**: Push notifications, email notifications, in-app notifications.\n- **Endpoints**: `POST /notifications/send`, `GET /notifications`, `PATCH /notifications/{id}/read`.\n- **Data**: Notifications table, device tokens table.\n\n### 4.9 Admin Module\n- **Responsibility**: Platform governance, user management, business verification, analytics.\n- **Endpoints**: `GET /admin/users`, `PATCH /admin/users/{id}`, `GET /admin/businesses`, `PATCH /admin/businesses/{id}/verify`, `GET /admin/analytics`.\n- **Data**: Admin audit log table.\n\n## 5. Database Schema (Prisma + PostGIS)\n\nKey tables and relationships:\n\n- **User**: id, email, phone, passwordHash, name, avatarUrl, role (CONSUMER, PROVIDER, ADMIN), isVerified, createdAt, updatedAt.\n- **Business**: id, ownerId (FK User), name, description, address, location (PostGIS Geography(Point)), phone, website, email, rating, reviewCount, isVerified, isActive, createdAt, updatedAt.\n- **BusinessCategory**: id, businessId (FK Business), categoryId (FK Category).\n- **Category**: id, name, icon, parentId (self-referential FK for hierarchy).\n- **Service**: id, businessId (FK Business), name, description, durationMinutes, price, depositAmount, isActive, createdAt.\n- **Staff**: id, businessId (FK Business), name, photoUrl, bio, isActive.\n- **OperatingHours**: id, businessId (FK Business), dayOfWeek, openTime, closeTime, isClosed.\n- **Booking**: id, userId (FK User), businessId (FK Business), staffId (FK Staff), startTime, endTime, status (PENDING, CONFIRMED, IN_PROGRESS, COMPLETED, CANCELLED), totalPrice, depositPaid, promoCodeId, createdAt, updatedAt.\n- **BookingService**: id, bookingId (FK Booking), serviceId (FK Service), price.\n- **Review**: id, userId (FK User), businessId (FK Business), staffId (FK Staff), rating, comment, createdAt.\n- **Favorite**: id, userId (FK User), businessId (FK Business), createdAt.\n- **SearchHistory**: id, userId (FK User), query, createdAt.\n- **RefreshToken**: id, userId (FK User), token, expiresAt, createdAt.\n- **Payment**: id, bookingId (FK Booking), amount, currency, status, provider, externalId, createdAt.\n- **PromoCode**: id, code, discountType (PERCENTAGE, FIXED), discountValue, maxUses, currentUses, expiresAt, isActive.\n- **Notification**: id, userId (FK User), title, body, data (JSON), isRead, createdAt.\n- **DeviceToken**: id, userId (FK User), token, platform, createdAt.\n\n## 6. Key Architectural Decisions\n\n### 6.1 Geospatial Queries\n- Use PostGIS `ST_DWithin` for radius-based business search.\n- Store business location as `Geography(Point)` for accurate distance calculations.\n- Index the geography column for performance.\n\n### 6.2 Caching Strategy\n- **Redis** for:\n  - Session store for refresh tokens.\n  - Rate limiting counters.\n  - Business availability cache (TTL: 5 minutes).\n  - Search autocomplete suggestions (TTL: 1 hour).\n  - Trending searches (TTL: 1 hour).\n- **TanStack React Query** for client-side caching with stale-while-revalidate.\n\n### 6.3 Authentication Flow\n- JWT access token (15 min) + refresh token (7 days) stored in Redis.\n- Refresh token rotation: old token invalidated on refresh.\n- Biometric authentication stored locally on device via expo-local-authentication.\n- OAuth tokens stored securely via expo-secure-store.\n\n### 6.4 Booking Availability\n- Pre-compute availability slots for each staff member for the next 30 days.\n- Store in Redis as sorted sets with timestamp as score.\n- Update on booking creation/cancellation.\n- Fallback to database query on cache miss.\n\n### 6.5 Search\n- Full-text search using PostgreSQL `tsvector` and `tsquery`.\n- Autocomplete with trigram similarity (`pg_trgm` extension).\n- Voice search via Expo Speech-to-Text API.\n\n### 6.6 Offline Support\n- Map tiles cached via react-native-maps (offline tiles).\n- Business details cached locally using AsyncStorage.\n- TanStack React Query persistence adapter for offline data.\n\n## 7. API Design\n\n- RESTful API with versioning (`/api/v1/`).\n- Request/Response validation using NestJS pipes and class-validator.\n- Pagination via cursor-based (for lists) and offset-based (for search).\n- Error responses follow RFC 7807 (Problem Details).\n- Rate limiting: 100 requests/min per user, 20 requests/min for unauthenticated.\n\n## 8. Security\n\n- Helmet.js for HTTP headers.\n- CORS configured for mobile app origins.\n- Input sanitization against XSS.\n- SQL injection prevention via Prisma parameterized queries.\n- HTTPS enforced in production.\n- Environment variables managed via NestJS ConfigModule.\n\n## 9. Testing Strategy\n\n- **Unit Tests**: Jest for services, controllers, and utilities.\n- **Integration Tests**: Supertest for API endpoints with test database.\n- **E2E Tests**: Detox for mobile app flows.\n- **Coverage**: Minimum 80% coverage for backend, 70% for mobile.\n\n## 10. CI/CD Pipeline\n\n- **GitHub Actions**:\n  - Lint and type-check on push.\n  - Run unit and integration tests on PR.\n  - Build and deploy backend to staging on merge to `develop`.\n  - Build and deploy backend to production on merge to `main`.\n- **EAS Build**:\n  - Build mobile app for iOS and Android on PR (preview builds).\n  - Submit to App Store and Google Play on release.\n\n## 11. Local Development\n\n- Docker Compose spins up PostgreSQL (with PostGIS), Redis, and a test SMTP server.\n- `pnpm dev` runs both mobile and backend in watch mode.\n- Prisma migrations run automatically via `prisma migrate dev`.\n- Supabase used for managed PostgreSQL and authentication in production.\n\n## 12. Performance Considerations\n\n- Database indexes on foreign keys, location, and search fields.\n- N+1 query prevention via Prisma `include` and `select`.\n- Lazy loading of images in mobile app.\n- Virtualized lists for large data sets (FlatList with windowing).\n- Background tasks for notifications via expo-notifications.\n\n## 13. Monitoring and Logging\n\n- Structured logging with NestJS Logger and Winston.\n- Error tracking with Sentry.\n- Performance monitoring with New Relic or Datadog (optional).\n- Health check endpoint at `/api/v1/health`.\n\n## 14.
