Overview\nPlanity Clone is a monorepo mobile‑first application that connects users with local service businesses for discovery, booking, and management. The architecture follows clean‑layered principles: presentation (Expo/React Native), application (TanStack Query, Reanimated), domain (shared types & design system), infrastructure (NestJS API, Prisma/PostgreSQL/PostGIS, Redis, BullMQ), and DevOps (Docker Compose, Nx, pnpm, GitHub Actions, EAS Build, Supabase).\n\n## 2. Tech Stack Summary\n- **Mobile**: Expo SDK, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Web/Admin**: Same stack (Expo Web) for admin dashboard (optional)\n- **Backend**: NestJS (Node.js), Prisma ORM, PostgreSQL + PostGIS extension, Redis (caching & pub/sub), BullMQ (job queues)\n- **Monorepo**: Nx workspace, pnpm package manager\n- **CI/CD**: GitHub Actions (lint, test, build), EAS Build (expo), Docker Compose (local dev), Supabase (auth & storage fallback)\n- **Testing**: Jest (unit/integration), React Native Testing Library\n\n## 3. High‑Level Architecture\n```\n+-------------------+      +-------------------+      +-------------------+\n|   Mobile App      |<---->|   API Gateway     |<---->|   Business Logic  |\n| (Expo RN)         |      | (NestJS Controllers) |    | (NestJS Services) |\n+-------------------+      +-------------------+      +-------------------+\n        ^                         ^                         |\n        |                         |                         v                         v                         v\n+-------------------+      +-------------------+      +-------------------+\n|   Shared Libs     |      |   Data Layer      |      |   Infra/Services  |\n| (types, ui, ds)   |      | (Prisma, PostGIS) |      | (Redis, BullMQ)   |\n+-------------------+      +-------------------+      +-------------------+\n```\n- **Presentation Layer**: Handles UI, navigation, gestures, animations.\n- **Application Layer**: React Query for data fetching, caching, mutations; Reanimated for UI worklets.\n- **Domain Layer**: Shared TypeScript interfaces/enums, design system tokens, utility functions.\n- **API Layer**: NestJS provides RESTful endpoints (JSON) with JWT auth, role‑based guards, validation (class‑validator), and GraphQL optional.\n- **Data Layer**: Prisma models mapped to PostgreSQL; PostGIS for geo‑queries (nearby businesses).\n- **Infra Layer**: Redis for session cache, rate‑limiting, pub/sub (real‑time notifications); BullMQ for background jobs (email/SMS, payment webhooks, report generation).\n\n## 4. Service Boundaries\n| Boundary          | Responsibility                                                                 | Technology                              |
|-------------------|--------------------------------------------------------------------------------|----------------------------------------|
| **Mobile Client** | UI/UX, navigation, local state, offline optimizations, deep linking, push notifications (via Expo) | Expo, React Native, TS, React Query, Reanimated |
| **Web Admin**     | Admin dashboard (content, users, analytics) – shares same UI libs & API        | Expo Web, React Native Web, TS          |
| **API Gateway**   | Request routing, auth middleware, rate limiting, input validation              | NestJS (Controllers, Guards, Interceptors) |
| **Business Logic**| Core domain services: auth, business/search, booking, slot computation, payments, reviews, notifications | NestJS (Services, Providers)            |
| **Data Access**   | ORM mapping, migrations, geo queries, transactions                               | Prisma + PostgreSQL + PostGIS           |
| **Caching/Queue** | Session cache, rate‑limit counters, pub/sub for real‑time updates, job processing | Redis, BullMQ                           |
| **DevOps/Infra**  | Containerization, local dev, CI/CD, builds, deployments, monitoring            | Docker Compose, Nx, pnpm, GitHub Actions, EAS Build, Supabase (optional auth/storage) |
| **Third‑Party**   | Payment gateway (Stripe/PayPal), map provider (Google/Apple Maps), email/SMS (Twilio/SendGrid) | SDKs via NestJS services                |

## 5. Monorepo Folder Structure (Nx)\n```\nplanity-clone/\n├─ apps/\n│  ├─ mobile/               # Expo RN app (iOS/Android/Web)\n│  │   ├─ src/\n│  │   │   ├─ assets/\n│  │   │   ├─ components/   # shared UI (re‑used via libs)\n│  │   │   ├─ screens/\n│  │   │   ├─ navigation/   # Expo Router config\n│  │   │   ├─ hooks/\n│  │   │   ├─ utils/\n│  │   │   └─ types/        # re‑export from @planity/types\n│  │   ├─ app.json\n│  │   ├─ eas.json\n│  │   └─ tsconfig.json\n│  └─ admin/                # Optional Expo Web admin dashboard\n├─ libs/\n│  ├─ ui/                   # Design system (tokens, primitives, themes)\n│  ├─ types/                # Shared TS interfaces/enums (DTOs, models)\n│  ├─ api/                  # NestJS backend (Nx plugin)\n│  │   ├─ src/\n│  │   │   ├─ auth/\n│  │   │   ├─ business/\n│  │   │   ├─ booking/\n│  │   │   ├─ payment/\n│  │   │   ├─ review/\n│  │   │   ├─ notification/\n│  │   │   ├─ geolocation/\n│  │   │   ├─ common/ (guards, interceptors, pipes)\n│  │   │   └─ main.ts\n│  │   ├─ test/\n│  │   ├─ prisma/\n│  │   │   ├─ schema.prisma\n│  │   │   └─ migrations/\n│  │   ├─ dockerfile\n│  │   └─ tsconfig.json\n│  ├─ data/                 # Prisma client wrapper, reusable repositories\n│  ├─ cache/                # Redis client wrapper\n│  ├─ jobs/                 # BullMQ queue definitions & processors\n│  ├─ utils/                # Logging, config, error handling\n│  └─ eslint/               # Shared ESLint config\n├─ tools/\n│  ├─ generators/           # Nx schematics for libs, components\n│  └─ scripts/\n├─ docker-compose.yml       # Local dev: postgres, redis, redis‑inspector, api\n├─ nx.json\n├─ package.json\n├─ pnpm-workspace.yaml\n└─ .github/\n    └─ workflows/\n        ├─ ci.yml           # lint, test, build\n        └─ release.yml      # EAS build + docker image push\n```\n\n## 6. Data Model Highlights (Prisma)\n```prisma\nmodel User {\n  id            String   @id @default(uuid())\n  email         String   @unique\n  passwordHash  String\n  role          Role     @default(USER)\n  profile       Profile? @relation(fields: [profileId], references: [id])\n  profileId     String?  @unique\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\nmodel Business {\n  id            String   @id @default(uuid())\n  ownerId       String\n  name          String\n  description   String?\n  address       String\n  lat           Float\n  lng           Float\n  location      Point    @db.Point @default(dbgenerated(\"POINT(0 0)\"))\n  categories    Category[] @relation(\"BusinessCategory\")\n  services      Service[] @relation(\"BusinessService\")\n  workingHours  WorkingHour[]\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\nmodel Service {\n  id          String   @id @default(uuid())\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  name        String\n  durationMin Int\n  price       Float\n}\n\nmodel Booking {\n  id            String   @id @default(uuid())\n  userId        String\n  serviceId     String\n  service       Service  @relation(fields: [serviceId], references: [id])\n  startAt       DateTime\n  endAt         DateTime\n  status        BookingStatus\n  paymentId     String?  @unique\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\n// Enums omitted for brevity\n```\n- PostGIS index on `Business.location` for radius queries.\n- Redis stores short‑lived auth tokens and rate‑limit counters.\n\n## 7. API Design (NestJS)\n- **REST** endpoints under `/api/v1/{resource}`\n- **Auth**: JWT access token (15 min) + refresh token (7 days) stored in HttpOnly cookie; refresh endpoint rotates tokens.\n- **Validation**: DTOs with class‑validator, transformation pipes.\n- **Error Handling**: Global exception filter returns `{error, message, statusCode}`.\n- **Rate Limiting**: `@Throttle({ default: { limit: 5, ttl: 60 } })` per‑route.\n- **Geo Search**: `GET /businesses/near?lat=&lng=&radius=&category=` uses PostGIS `ST_DWithin`.\n- **Webhooks**: Payment provider webhook verified via signature, enqueues BullMQ job for order fulfillment.\n\n## 8. State Management & Data Flow (Mobile)\n- **React Query** handles server state: queries (`useBusinesses`, `useBooking`) and mutations (`useCreateBooking`).\n- Query keys are versioned (`['businesses', lat, lng, radius, category]`).\n- Optimistic UI updates for booking creation/cancellation.\n- **Reanimated 2** for gesture‑based map interactions, animated modals, and shared‑element transitions.\n- **Auth Context** wraps app; token refreshed silently via background fetch.\n- **Expo Notifications** (push) receive payloads from BullMQ job processor (via Redis pub/sub or direct HTTP call to Expo push service).\n\n## 9. Authentication & Authorization\n- **User Roles**: `USER`, `BUSINESS_OWNER`, `ADMIN`.\n- **Guards**: `JwtAuthGuard`, `RolesGuard` (`@Roles(USER)`).\n- **Business Owner Portal**: Same API, guarded by `BUSINESS_OWNER`; provides CRUD on own business, services, working hours, booking list.\n- **Admin Dashboard**: Additional endpoints under `/admin` guarded by `ADMIN`.\n\n## 10. Background Jobs (BullMQ)\n| Queue          | Responsibility                                   | Trigger                                   |
|----------------|--------------------------------------------------|-------------------------------------------|
| `notification` | Send push/email/SMS                              | Booking created/updated, payment success, review posted |
| `payment`      | Capture/refund via payment gateway               | Booking confirmation, manual refund       |
| `report`       | Generate CSV/PDF analytics                       | Admin request (scheduled or on‑demand)    |
| `slot‑sync`    | Re‑compute available slots after working‑hour change | Business updates working hours            |
\nEach queue has a dedicated processor (Nx lib `jobs`). Workers run alongside API in Docker (`node src/main.js`).\n\n## 11. Map‑Based Search & Geo Features\n- **Provider**: Expo `MapView` (Google Maps SDK for iOS/Android, Apple Maps via Expo).\n- **Data Fetch**: Query API for businesses within radius; results cached in React Query.\n- **Directions**: Launch native maps app via `Linking` with latitude/longitude.\n- **Clustering**: Use `react-native-maps-super-cluster` (wrapped in Reanimated worklet for performance).\n\n## 12. Payments\n- **Gateway**: Stripe (or PayPal) via NestJS service.\n- **Flow**: \n  1. Frontend creates payment intent (POST `/payments/create-intent`).\n  2. Confirm on client with card details (Stripe Elements via webview or Expo‑Stripe SDK).\n  3. Webhook confirms success → enqueues `payment` job → updates `Booking.paymentId` and status.\n- **Webhook Security**: Verify signature, idempotency via `stripe_event_id` unique column.\n\n## 13. Notifications\n- **Push**: Expo Push Notifications service; token stored on user document.\n- **Email/SMS**: Via SendGrid/Twilio, invoked from `notification` job.\n- **In‑App**: Real‑time list using Redis pub/sub; API publishes events, mobile subscribes via WebSocket (NestJS Gateway) or polling fallback.\n\n## 14. Provider / Business Owner Portal\n- Same mobile app, role‑based UI: after login, if `role === BUSINESS_OWNER` show owner tab.\n- Features: manage business profile, services, working hours, view bookings, respond to reviews.\n- Uses same API endpoints guarded by `BUSINESS_OWNER`.\n\n## 15. Admin Dashboard\n- Optional Expo Web app (`apps/admin`) sharing `ui` and `types` libs.\n- CRUD over users, businesses, settings; analytics via `report` job.\n- Protected by `ADMIN` guard.\n\n## 16. Shared Design System (`libs/ui`)\n- Tokens: colors, spacing, typography (platform‑adaptive).\n- Primitives: Button, Input, Card, Avatar, Badge, Loader.\n- Themes: Light/Dark, accessible contrast.\n- Exported via barrel file; consumed by `apps/*` and `libs/api` (for storybook docs if needed).\n\n## 17. Testing Strategy\n- **Unit**: Jest + React Native Testing Library for components, hooks, utils.\n- **Integration**: API endpoint tests with Supertest; Prisma uses SQLite in‑memory for test runs.\n- **E2E** (optional): Detox for critical flows (login → search → book).\n- **Lint**: ESLint (shared config) + Prettier.\n- **CI**: GitHub Actions runs lint, test, build (Nx affected) on PR; on merge to main triggers EAS build and Docker image push.\n\n## 18. DevOps & Deployment\n- **Local**: `docker compose up` runs PostgreSQL (with PostGIS extension), Redis, API (hot reload via Nx watch). Expo dev client runs separately.\n- **CI/CD**:\n  - `github/workflows/ci.yml`: `pnpm install`, `nx run-many --target=lint --all`, `nx run-many --target=test --all`, `nx run-many --target=build --all`.\n  - `github/workflows/release.yml`: on `push to main` → EAS Build (`eas build --platform all --auto-submit`) → Docker image (`docker build -t planity-api:$(git rev-parse --short HEAD) .`) → push to registry.\n- **Production**: Managed Kubernetes or ECS; API replicas behind ALB; Redis Elasticache; PostgreSQL RDS with PostGIS; Env vars for JWT secret, Stripe keys, Expo push token.\n- **Monitoring**: CloudWatch logs, Sentry (frontend & backend), Prometheus metrics via NestJS `@nestjs/terminus`.\n\n## 19. Security Considerations\n- HTTPS everywhere (Expo uses TLS; API behind TLS terminator).\n- JWT stored HttpOnly, CSRF protection via SameSite cookie.\n- Input validation & sanitization (class-validator, Prisma param queries).\n- Rate limiting & brute‑force protection on auth endpoints.\n- Payment data never touches our servers (Stripe Elements/Webhook).\n- Regular dependency audits (`pnpm audit`, `nx affected:depgraph`).\n- Database backups (AWS RDS snapshots) and encryption at rest.\n\n## 20. Scalability & Performance\n- **Read‑heavy**: Business listings cached in Redis (TTL 5 min) and React Query (stale‑time 2 min).\n- **Write‑heavy**: Booking creation enqueues slot‑calc job; writes go through PostgreSQL with proper indexing.\n- **Geo queries**: PostGIS GIST index on `location` ensures sub‑second radius search.\n- **Horizontal scaling**: API is stateless; can run multiple replicas behind load balancer.\n- **Background jobs**: BullMQ with multiple workers; can be scaled via Kubernetes jobs.\n\n## 21. Future Extensions\n- GraphQL federation for richer client queries.\n- WebSocket subscriptions (NestJS Gateway) for live booking updates.\n- AI‑powered recommendation service (separate microservice).\n- Offline-first mode with Expo SQLite + React Query persister.\n\n---\n*This document provides a complete blueprint for building the Planity Clone with the requested stack, ensuring clean separation of concerns, maintainability, and readiness for production deployment.*\n",
