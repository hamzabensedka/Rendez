Overview\n\nThis document describes the system architecture for the Planity Clone, a mobile-first platform connecting customers with beauty/wellness businesses for appointment booking. The architecture is designed for scalability, maintainability, and clear separation of concerns, using a monorepo structure with Nx, a React Native (Expo) mobile client, and a NestJS backend.\n\n## 2. Architecture Principles\n\n- **Clean Architecture**: Layers are separated (presentation, application, domain, infrastructure) with dependencies pointing inward.\n- **Separation of Concerns**: Each module/service has a single responsibility.\n- **API-First Design**: Backend APIs are designed before frontend implementation.\n- **Type Safety**: TypeScript is used end-to-end, with shared types between client and server.\n- **Testability**: Unit, integration, and e2e tests are first-class citizens.\n- **Offline-First**: The mobile app works offline where possible, with background sync.\n- **Security**: JWT with refresh tokens, rate limiting, input validation, and secure storage.\n\n## 3. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| **Mobile Client** | Expo (React Native), TypeScript, Expo Router, TanStack React Query, React Native Reanimated | Cross-platform mobile app with smooth animations and declarative data fetching |\n| **Backend API** | NestJS, TypeScript | RESTful API server with modular architecture |\n| **Database** | PostgreSQL + PostGIS | Relational data with geospatial queries |\n| **Cache & Session** | Redis | Caching, rate limiting, session store, real-time pub/sub |\n| **ORM** | Prisma | Type-safe database access with migrations |\n| **Monorepo** | Nx, pnpm | Shared code, dependency management, build orchestration |\n| **CI/CD** | GitHub Actions, EAS Build | Automated testing, building, and deployment |\n| **Auth** | Supabase Auth (optional) or custom JWT | Authentication and authorization |\n| **Testing** | Jest, React Native Testing Library | Unit and integration tests |\n| **Containerization** | Docker Compose | Local development environment |\n\n## 4. Data Flow\n\n### 6.1 Authentication Flow\n\n1. User submits credentials (email/password or OAuth token) from mobile app.\n2. Mobile app sends POST /auth/login to backend.\n3. Backend validates credentials, generates JWT access token (15 min) and refresh token (30 days).\n4. Backend stores refresh token hash in database, returns tokens to client.\n5. Mobile app stores access token in memory (secure) and refresh token in secure storage (expo-secure-store).\n6. TanStack React Query uses an Axios interceptor to attach the access token to all requests.\n7. When access token expires, the interceptor automatically calls POST /auth/refresh with the refresh token.\n8. If refresh fails, user is logged out.\n\n### 6.2 Booking Flow\n\n1. Customer searches for businesses (search or map).\n2. Customer selects a business, views services and availability.\n3. Customer initiates booking: selects service(s), professional, date/time.\n4. Mobile app calls POST /bookings/hold to temporarily hold the slot (10 min).\n5. Customer proceeds to payment (Step 5).\n6. On successful payment, mobile app calls POST /bookings/confirm with payment intent ID.\n7. Backend creates the booking, releases the hold, sends confirmation email/SMS.\n8. If payment fails or times out, the hold is released automatically via Redis TTL.\n\n### 6.3 Search & Map Flow\n\n1. User enters search query or pans map.\n2. Mobile app debounces input (300ms) and calls GET /businesses?q=...&lat=...&lng=...&radius=...\n3. Backend uses PostGIS to perform geospatial query: `ST_DWithin(location, ST_MakePoint(lng, lat), radius)`.\n4. Results are paginated (20 per page) and returned with distance, rating, next available slot.\n5. Mobile app renders markers on map or list items.\n6. TanStack React Query caches results; stale time set to 60 seconds.\n7. Offline: cached results shown with stale data indicator.\n\n## 7. API Design (Key Endpoints)\n\n| Method | Endpoint | Description |\n|--------|----------|-------------|\n| POST | /auth/register | Register new user |\n| POST | /auth/login | Login with credentials |\n| POST | /auth/refresh | Refresh access token |\n| POST | /auth/logout | Invalidate refresh token |\n| POST | /auth/forgot-password | Send reset email |\n| POST | /auth/reset-password | Reset password |\n| GET | /businesses | Search businesses (query, lat, lng, radius, filters) |\n| GET | /businesses/:id | Business detail |\n| GET | /businesses/:id/services | Services list |\n| GET | /businesses/:id/availability | Available slots (date, staffId) |\n| POST | /bookings/hold | Hold a slot (10 min) |\n| POST | /bookings/confirm | Confirm booking after payment |\n| GET | /bookings | User's bookings |\n| GET | /bookings/:id | Booking detail |\n| POST | /payments/create-intent | Create payment intent |\n| POST | /payments/webhook | Stripe webhook handler |\n| GET | /reviews?businessId= | Business reviews |\n| POST | /reviews | Create review |\n\n## 9. Caching Strategy (Redis)\n\n- **Session Store**: Refresh tokens and rate limit counters stored in Redis with TTL.\n- **Business Search Results**: Cached for 60 seconds with geohash keys.\n- **Availability Slots**: Cached for 30 seconds; invalidated on booking creation.\n- **Service Categories**: Cached for 1 hour; invalidated on admin update.\n- **Rate Limiting**: Per-IP and per-user counters with sliding window.\n- **Job Queue**: Bull queue for email/SMS notifications.\n\n## 10. Security Considerations\n\n- **Authentication**: JWT with short-lived access tokens (15 min) and long-lived refresh tokens (30 days). Refresh tokens are stored as hashed values in the database.\n- **Authorization**: Role-based access control (RBAC) using NestJS guards. Roles: CUSTOMER, BUSINESS_OWNER, ADMIN.\n- **Input Validation**: All API inputs validated using Zod schemas (shared package).\n- **Rate Limiting**: 5 failed login attempts -> 30 min lockout. General API rate limit: 100 requests/min per user.\n- **Data Protection**: Sensitive data (passwords) hashed with bcrypt. PII encrypted at rest.\n- **HTTPS**: Enforced in production. Mobile app uses certificate pinning.\n- **Webhook Security**: Stripe webhook signature verification.\n\n## 11. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions and components. Coverage > 80%.\n- **Integration Tests**: NestJS e2e tests with test database (PostgreSQL + Redis).\n- **Mobile Tests**: React Native Testing Library for component tests, Detox for e2e.\n- **CI Pipeline**: GitHub Actions runs lint, unit tests, integration tests on every PR.\n- **EAS Build**: For mobile app builds (internal testing, production).\n\n## 12. Deployment & DevOps\n\n- **Local Development**: Docker Compose spins up PostgreSQL, Redis, and the API server.\n- **Backend**: Deployed to a cloud provider (AWS ECS, Railway, or similar) with environment variables.\n- **Mobile**: EAS Build for OTA updates and native builds. Expo Updates for over-the-air JS updates.\n- **CI/CD**: GitHub Actions workflows:\n  - `ci.yml`: Lint, test, build on push/PR.\n  - `deploy.yml`: On merge to main, build and deploy backend, trigger EAS Build for mobile.\n- **Monitoring**: Sentry for error tracking, Datadog or similar for APM.\n\n## 13.
