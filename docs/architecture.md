The architecture separates concerns into a mobile app (Expo + React Native), a NestJS API gateway serving business logic, background workers for scheduled tasks, and a lightweight admin web frontend. Real-time features are handled through WebSockets backed by Redis Pub/Sub. Authentication and file storage leverage Supabase, while the backend uses its own JWT verification. Data is stored in PostgreSQL with PostGIS for location queries, and Redis accelerates heavilyвҖ‘used reads. The entire system is containerized via Docker Compose for local development and deployed through GitHub Actions, EAS Build, and a cloud container orchestrator.\n\n## 2. Service Boundaries (API Modules)\n\n### 5.1 Auth Module\n- **Responsibility**: User registration, login, social auth, password reset, OTP verification, token refresh.\n- **Integration**: Supabase Auth for identity management. API verifies JWTs issued by Supabase using the projectвҖҷs public key. Endpoints: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/verify-otp`.\n- **Details**: On social login, backend creates/links a local `User` record. Refresh tokens are handled by Supabase SDK on mobile; backend accepts only shortвҖ‘lived access tokens.\n\n### 5.2 User Module\n- **Responsibility**: User profiles, preferences, saved businesses, notification settings.\n- **Endpoints**: `GET /users/me`, `PATCH /users/me`, `POST /users/me/phone`, `GET /users/me/favorites`.\n- **Details**: Stores extended profile data in PostgreSQL. Favourites are a manyвҖ‘toвҖ‘many relation between User and Business.\n\n### 5.3 Business Module\n- **Responsibility**: CRUD for business profiles, services, staff, working hours, and amenities. Open status computation.\n- **Endpoints**: `GET /businesses`, `GET /businesses/:id`, `POST /businesses` (provider), `PATCH /businesses/:id`, `GET /businesses/:id/services`, `GET /businesses/:id/staff`, `GET /businesses/:id/reviews`, etc.\n- **Details**: Providers manage their own business. Admin can approve/disable. Business detail returns computed `isOpen` based on `WorkingHours` and `Holiday` records.\n\n### 5.4 Booking Module\n- **Responsibility**: Core booking flow вҖ“ slot calculation, hold slots, confirm booking, cancel, and refund.\n- **Endpoints**: `GET /booking/slots?businessId&serviceId&staffId&date`, `POST /booking/hold`, `POST /booking/confirm`, `POST /booking/cancel`, `GET /bookings` (userвҖҷs bookings).\n- **Details**: Slot availability is computed by intersecting staff working hours, existing appointments, holidays, and service duration. A Redis-based lock (TTL 5 min) is placed when a user holds a slot to prevent double booking. Confirmation finalises the booking, creates a payment intent, and removes the hold. Cancellation within the providerвҖ‘defined window triggers refund processing.\n\n### 5.5 Payment Module\n- **Responsibility**: Integrate with Stripe for payment processing, preвҖ‘authorisations, and refunds.\n- **Endpoints**: `POST /payments/intent`, `POST /payments/webhook`.\n- **Details**: Uses Stripe Connect for direct payments to providers (platform fee deducted). Webhook events (payment succeeded, failed, refund) update booking and payment status. All sensitive operations happen serverвҖ‘side.\n\n### 5.6 Notification Module\n- **Responsibility**: Dispatch transactional emails (welcome, reset, confirmation), SMS (booking reminders via Twilio), and push notifications (Expo Push).\n- **Endpoints**: Internal events from booking/payment modules are published to Redis; worker picks them up and sends via providers.\n- **Details**: Supabase Auth triggers email verification automatically; backend handles password reset via email link.\n\n### 5.7 Search Module\n- **Responsibility**: FullвҖ‘text search, filtering, sorting, and autocomplete.\n- **Endpoints**: `GET /search?q=&category=&radius=&lat=&lng=&filters=&sort=`, `GET /search/autocomplete?q=`.\n- **Details**: Uses PostgreSQL fullвҖ‘text search (`tsvector`) on business name, description, and service names. PostGIS spatial queries (`ST_DWithin`) filter by radius. Results are paginated (20 items). Autocomplete is backed by Redis sorted sets for low latency.\n\n### 5.8 Admin Module\n- **Responsibility**: Platform administration вҖ“ manage businesses, users, categories, reviews, and system health.\n- **Endpoints**: `GET /admin/businesses`, `PATCH /admin/businesses/:id/approve`, `DELETE /admin/users/:id`, etc.\n- **Details**: Role `ADMIN` required. Admin API is restricted to a separate subвҖ‘domain or path prefix with its own authentication guard.\n\n## 6. Mobile App Architecture\n\n### 7.1 Navigation (Expo Router)\n```\napp/\nв”ңв”Җв”Җ (auth)/                  # auth group (stack)\nв”Ӯ   в”ңв”Җв”Җ login.tsx\nв”Ӯ   в”ңв”Җв”Җ register.tsx\nв”Ӯ   в”ңв”Җв”Җ forgot-password.tsx\nв”Ӯ   в””в”Җв”Җ verify-otp.tsx\nв”ңв”Җв”Җ (tabs)/                  # main tab navigator\nв”Ӯ   в”ңв”Җв”Җ index.tsx            # home\nв”Ӯ   в”ңв”Җв”Җ search.tsx           # search & map\nв”Ӯ   в”ңв”Җв”Җ bookings.tsx         # upcoming bookings\nв”Ӯ   в””в”Җв”Җ profile.tsx          # user profile\nв”ңв”Җв”Җ (modals)/                # stack presented as modals\nв”Ӯ   в”ңв”Җв”Җ booking-flow/        # multiвҖ‘step booking wizard\nв”Ӯ   в”Ӯ   в”ңв”Җв”Җ service.tsx\nв”Ӯ   в”Ӯ   в”ңв”Җв”Җ date-time.tsx\nв”Ӯ   в”Ӯ   в”ңв”Җв”Җ review.tsx\nв”Ӯ   в”Ӯ   в”ңв”Җв”Җ payment.tsx\nв”Ӯ   в”Ӯ   в””в”Җв”Җ confirmation.tsx\nв”Ӯ   в”ңв”Җв”Җ business/[id].tsx    # business detail (with tabs)\nв”Ӯ   в””в”Җв”Җ review-write.tsx\nв”ңв”Җв”Җ _layout.tsx              # root layout (providers: React Query, auth)\nв””в”Җв”Җ +not-found.tsx\n```\n- All screens are codeвҖ‘split automatically by Expo Router.\n- Deep links are configured for universal links (e.g., `planity://booking/confirm/...`).\n\n### 7.2 State Management\n- **Server State**: All API data (businesses, services, slots, bookings) is fetched and cached via TanStack React Query. Custom hooks in `libs/api-hooks` provide `useBusiness(id)`, `useSlots(params)`, `useCreateBooking()`, etc. Mutations invalidate related queries (e.g., booking confirmation refreshes the userвҖҷs bookings list and business availability).\n- **Local State**: Zustand stores for:\n  - `authStore` вҖ“ tracks login status, user role, token expiration.\n  - `locationStore` вҖ“ holds userвҖҷs current coordinates and selected location for search.\n  - `bookingStore` вҖ“ manages the multiвҖ‘step booking flow state (selected service, staff, date/time, notes). Cleared on completion or cancellation.\n  - `cartStore` (optional) вҖ“ if guest mode allows adding multiple booking items.\n- **Authentication**: Supabase client is initialised in `libs/auth`. The Expo app uses `@supabase/supabase-js` with React Native AsyncStorage adapter. The access token is automatically attached to every API request via an Axios interceptor. A background refresh job keeps the session alive.\n\n### 7.3 RealвҖ‘Time Updates\n- The mobile app connects to the NestJS WebSocket gateway (`apps/api/src/gateways/booking.gateway.ts`) using Socket.IO client. It listens for these events:\n  - `slotUpdated` (for a business/service) вҖ“ triggers query invalidation for `useSlots`.\n  - `bookingStatusChanged` вҖ“ updates the userвҖҷs booking list.\n- The gateway uses Redis Pub/Sub to propagate events across horizontally scaled API instances.\n- Fallback: Polling via React QueryвҖҷs `refetchInterval` if WebSocket connection drops (e.g., every 30s for slot views).\n\n## 8. RealвҖ‘Time Slot Locking & Availability\n- **Slot Computation**: When the API receives a slot request, it calculates free intervals by merging staff working hours, existing bookings, and holidays. The result is cached in Redis for a short period (2 min) to reduce load during peak browsing.\n- **Slot Hold**: When a user selects a slot and proceeds to review, `POST /booking/hold` creates a temporary lock in Redis with key `hold:{businessId}:{serviceId}:{staffId}:{slot}` and a TTL of 5 minutes. The lock contains the user ID. The lock prevents concurrent holds.\n- **Confirmation**: `POST /booking/confirm` checks the hold still exists and belongs to the user, then creates the booking record in PostgreSQL and deletes the Redis lock. Failure to confirm within TTL expires the hold and the slot becomes available again.\n- **WebSocket Notification**: After any hold/confirm/cancel, a message is published to `slot_updates:{businessId}` and pushed to all clients viewing that businessвҖҷs slots.\n\n## 9. Caching Strategy\n| Data | Cache Store | TTL | Invalidation |\n|------|-------------|-----|--------------|\n| Available slots (computed) | Redis | 2 min | Cleared on booking confirmation/cancellation for that staff |\n| Business search results (first page) | Redis | 5 min | Cleared when business profile or services change (async event) |\n| Autocomplete suggestions | Redis Sorted Set | permanent (updated nightly) | Rebuilt by worker |\n| User session data | Redis | 15 min (JWT expiry) | On logout/token refresh |\n| React Query cache (client) | InвҖ‘memory (AsyncStorage persisted) | Configurable per query (staleTime) | Mutations autoвҖ‘invalidate related queries |\n- Rate limiting (global and perвҖ‘user) is implemented using Redis `INCR` with sliding window.\n\n## 10. Background Workers (apps/worker)\n- **BullMQ queues** (RedisвҖ‘backed):\n  - `email-queue`: sends welcome emails, booking confirmations, password resets (via Nodemailer/SendGrid).\n  - `sms-queue`: sends booking reminders and OTP via Twilio.\n  - `slot-summary-queue`: nightly job that preвҖ‘computes nextвҖ‘day slot summaries for popular businesses, warming the cache.\n  - `cleanup-queue`: removes expired holds, anonymises deleted user data after retention period.\n- Workers run as a separate NestJS process, sharing the same Prisma client and Redis connection.\n\n## 11. Admin Web App (apps/admin)\n- Built with React + Vite, using React Router and React Query.\n- Communicates with the same API but under `/admin` prefix. Admin JWT is obtained via Supabase login with `admin` role; every request is guarded by `RolesGuard('ADMIN')`.\n- Dashboard shows key metrics (total bookings, revenue, active users) aggregated from the database.\n- Simple forms for CRUD on categories, business approval, and user management.\n\n## 12. CI/CD & Deployment\n- **GitHub Actions**: On push to `main` or PR:\n  1. Install pnpm, run `nx affected:lint`, `nx affected:test --ci`, `nx affected:build`.\n  2. Run integration tests against a DockerвҖ‘composed environment (PostgreSQL + Redis).\n  3. Mobile app through App Store/Play Store.\n- **Secrets**: Managed via GitHub Secrets and injected into containers at runtime.\n\n## 13. Security\n- **Authentication & Authorisation**: Supabase issues signed JWTs (HS256). The API validates the token, extracts user ID/Supabase ID, and loads roles from local DB. Guards enforce roleвҖ‘based access (CUSTOMER, PROVIDER, ADMIN).\n- **HTTPS**: Enforced everywhere. Certificate via load balancer.\n- **Input Validation**: All endpoints use Zod schemas (via `@nestjs/zod`) to validate request bodies and query parameters.\n- **Rate Limiting**: Applied globally and per user (100 req/min) using Redis.\n- **CORS**: Configured to allow only the mobile appвҖҷs domain/scheme.\n- **Data Privacy**: Contact info hidden from unauthenticated users; PII encrypted at rest where necessary. Soft delete with retention policies.\n- **Payment**: No card details touch our server; Stripe Elements/SDK on client returns a token; backend confirms with Stripe API.\n\n## 14. Testing Strategy\n- **Unit Tests**: Jest with `@nestjs/testing` for NestJS modules/services; React Native Testing Library for mobile components.\n- **Integration Tests**: API tests using `supertest`, spinning up a fresh PostgreSQL/Redis via Docker.
