The solution leverages a monorepo (Nx) to share code between the Expo/React Native mobile app, NestJS backend, admin/dashboard web apps, and shared libraries. Clean architecture principles guide separation of concerns: presentation, application/domain, and infrastructure layers are distinct, enabling maintainability, testability, and scalability.\n\n## Tech Stack\n- **Mobile**: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Backend**: NestJS, Prisma ORM, PostgreSQL + PostGIS extension, Redis, BullMQ\n- **Web Portals**: Next.js (optional) for Admin Dashboard and Business Owner Portal\n- **Infrastructure**: Docker Compose (local), GitHub Actions (CI/CD), EAS Build (mobile builds), Supabase (file storage/auth fallback), Nx (monorepo tooling), pnpm (package manager)\n- **Testing**: Jest, React Native Testing Library\n- **Observability**: Winston logging, Sentry error tracking\n\n## High-Level Architecture\n```\n+-------------------+      +---------------------+      +-------------------+\n|   Mobile App      |<---->|   API Gateway       |<---->|   Worker Services |\n| (Expo/React Native)|      | (NestJS REST/GraphQL)|      | (NestJS BullMQ)   |\n+-------------------+      +---------------------+      +-------------------+\n        ^                         ^                         ^\n        |                         |                         |\n        |                         |                         |\n+-------------------+      +---------------------+      +-------------------+\n|  Push Notif. The API handles business logic, data access (via Prisma), and orchestrates background jobs via BullMQ. Workers process asynchronous tasks (e.g., email/SMS notifications, payment confirmation, slot recomputation). Redis serves as both a cache layer and the BullMQ broker. PostgreSQL with PostGIS stores geospatial data for map‑based search. Supabase provides object storage for user‑uploaded media and can augment auth if needed. Admin and business owner portals are separate Next.js apps consuming the same API.\n\n## Service Boundaries\n\n### 1. Frontend (Mobile)\n- **Responsibility**: UI presentation, user interactions, local state, offline caching, push notification handling.\n- **Boundaries**:\n  - UI layer: React Native components (shared via `libs/ui` and `libs/design-system`).\n  - Navigation: Expo Router (file‑based routing).\n  - Server state: TanStack React Query (query keys scoped per feature).\n  - Animations/Gestures: React Native Reanimated.\n  - Auth: Access token stored in Expo SecureStore; refresh token rotated via API.\n  - Payments: Stripe SDK via web view or Expo SDK; payment confirmation handled by webhook.\n  - Push Notifications: Expo Push Notification service; token registration on app start.\n\n### 2. Backend (NestJS API)\n- **Responsibility**: Business logic, authentication/authorization, data persistence, API contracts, job orchestration.\n- **Boundaries**:\n  - Controllers: REST endpoints (or GraphQL if preferred) grouped by domain (auth, businesses, services, bookings, users, reviews, payments, admin).\n  - Services: Encapsulate use‑cases; leverage Prisma for repository pattern.\n  - Guards: JWT AuthGuard, RolesGuard (USER, PROVIDER, ADMIN).\n  - Interceptors: Logging, transformation, error handling.\n  - Validation: class‑validator + class‑transformer DTOs.\n  - Events: NestJS EventEmitter for loose coupling (e.g., after booking created -> emit `booking.created`).\n  - Workers: Separate NestJS modules (or standalone processes) that listen to BullMQ queues (`notifications`, `slot-recalc`, `payment-webhook`).\n\n### 3. Admin Dashboard & Business Owner Portal\n- **Responsibility**: CRUD operations for privileged roles, analytics, content management.\n- **Boundaries**: Separate Next.js apps (could be lazily loaded within the monorepo under `apps/admin` and `apps/provider-portal`). They consume the same NestJS API; UI built with shared design system tokens via CSS variables or a shared UI library.\n\n### 4. Infrastructure\n- **Database**: PostgreSQL (primary) + PostGIS extension for location queries. Prisma manages migrations.\n- **Cache**: Redis (TTL‑based caching of frequent queries, session store, BullMQ broker).\n- **Storage**: Supabase Storage (S3‑compatible) for business images, user avatars, service media.\n- **Messaging**: BullMQ backed by Redis; queues: `email-notifications`, `push-notifications`, `slot-recalculation`, `payment-processing`.\n- **Observability**: Winston logs to stdout (captured by Docker); Sentry SDK in NestJS and React Native.\n- **CI/CD**: GitHub Actions runs lint, unit tests, builds Docker images, pushes to registry, triggers EAS Build for mobile, and can deploy to staging/production (ECS/Fargate or similar).\n\n## Folder Structure (Nx Monorepo)\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/               # Expo React Native app\n│   ├── api/ `RolesGuard` checks metadata (`Roles.USER`, `Roles.PROVIDER`, `Roles.ADMIN`).\n- **Social Login**: Expo SDK `Google`, `Apple`; backend verifies token and issues JWT.\n- **Password Reset**: Token‑based flow via email (handled by a background job).\n\n## Background Jobs (BullMQ)\n- **Queues**:\n  1. `email-notifications` – sends confirmation, reminder, cancellation emails via SendGrid/SMTP.\n  2. `push-notifications` – formats payload and calls Expo push API.\n  3. `slot-recalculation` – triggered on booking create/update/delete;
