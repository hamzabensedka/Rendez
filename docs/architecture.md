Architecture Overview\n\nPlanity Clone is a mobile-first appointment booking platform built with a **monorepo** structure managed by **Nx** and **pnpm**. The system follows a **layered architecture** with clear service boundaries, leveraging **Expo** (React Native) for cross-platform mobile and web clients, and a **NestJS** backend with **PostgreSQL** (PostGIS extension) and **Redis**. The architectural design prioritizes scalability, real-time data consistency, and developer productivity through strict separation of concerns and TypeScript across the entire stack.\n\n### 1.1 Service Boundaries & Domain Modules\n\n### 4.1 Auth Service (with Supabase)\n- **Client:** Supabase SDK for mobile handles sign-up, login, social auth, and link identity. JWT stored in secure storage (Expo SecureStore).\n- **Server (NestJS):** Verifies Supabase-issued JWTs via a custom guard that fetches the public key and validates claims. Provides `/profile` endpoint for additional user data (roles, preferences) stored in our PostgreSQL via Prisma. Refresh token rotation is managed by Supabase; logout blacklists token in Redis.\n- **RBAC:** A `RolesGuard` decorator checks `user.role` (customer, provider, admin) from the JWT payload against required roles for each endpoint.\n\n### 4.2 Business & Service Management\n- **Business Module:** CRUD operations for business profiles, including multi-category assignments, working hours (JSONB), amenities, image gallery management (URLs from S3 bucket).\n- **Service Module:** Nested under business, manages services grouped by category, with duration, price, and buffer time.\n- **Staff Module:** Links staff users to businesses, stores specialties and ratings.\n- **Data Integrity:** Provider role allows managing only their own businesses (check ownership via `OwnershipGuard`).\n\n### 4.3 Search & Discovery\n- **Search Module:** Handles text-based search with full Postgres `tsvector` indexes for business name, category, service names. Autocomplete is backed by a `search_history` table and a fast Redis sorted set for popular queries.\n- **Geo Module (PostGIS):** Uses `ST_DWithin` for radius-based business lookup (distance filters) and `ST_MakeEnvelope` for map viewport queries. Business data is enriched with PostGIS geometry points. Results paginated (cursor-based).\n- **Filtering:** Filters are compiled into a Prisma `where` object with dynamic conditions, paired with Redis cache for common filter combinations (TTL 5 min).\n\n### 4.4 Booking & Availability Engine\n- **Availability Module:** Core booking logic. Calculates available time slots based on staff working hours, existing bookings, buffer times, and holidays/blocked periods.\n- **Slot Hold Mechanism:** When a user enters booking flow, a temporary lock is created in Redis (`slot_hold:{businessId}:{staffId}:{date}:{time}`) with TTL of 5 minutes. Subsequent `checkAvailability` reads check both PostgreSQL and Redis holds.\n- **Booking Module:** Handles creation: validates slot still held, creates `Booking` + `BookingService` records in a transaction, sends confirmation via notification module, and removes the Redis hold. Supports reschedule and cancellation with policy enforcement (refund calculation).\n\n### 4.5 Payment Module\n- Integrates with a PSP (Stripe). Stores payment methods securely via PSP tokenization. Handles pre-authorization (if business requires), capture on service completion, and refunds on cancellation. Promo codes validated against `Promotion` table and applied pre-tax.\n\n### 4.6 Notification Module\n- Listens to events (booking.created, booking.cancelled, reminder.24h) via NestJS EventEmitter. Pushes tasks to a job queue (BullMQ backed by Redis). Workers process jobs: send push notifications (FCM/APNs via Expo Push API), email (SendGrid), and in-app messages.\n\n## 5.
