  "content": "# Planity Clone — System Architecture\n\n## 1. Overview\nPlanity Clone is a beauty & wellness booking marketplace with three frontends (consumer mobile, provider portal, admin) and a NestJS backend. The codebase is an Nx monorepo managed by pnpm, promoting clear separation of concerns and shared types.\n\n## 2. Tech Stack\n- Mobile & Web: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- Backend: NestJS, Prisma, PostgreSQL + PostGIS (via Supabase), Redis (cache + BullMQ)\n- Infra: Docker Compose (local dev), GitHub Actions (CI), EAS Build (mobile), Supabase (managed Postgres/Storage), Jest (tests)\n\n## 3. Monorepo Layout (Nx + pnpm)\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/            # Expo consumer app\n│   ├── provider-portal/   # Expo Web for owners\n│   ├── admin/             # Expo Web for admins\n│   └── api/               # NestJS backend\n├── libs/\n│   ├── shared/            # TS types, DTOs, constants\n│   ├── ui/                # Design system components\n│   ├── data-access/       # API client + React Query hooks\n│   ├── feature-auth/\n│   ├── feature-discovery/\n│   ├── feature-business/\n│   ├── feature-booking/\n│   ├── feature-appointments/\n│   ├── feature-profile/\n│   ├── feature-favorites/\n│   ├── feature-reviews/\n│   └── feature-admin/\n├── prisma/\n│   ├── schema.prisma\n│   ├── migrations/\n│   └── seed.ts\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n├── tsconfig.base.json\n├── .github/workflows/ci.yml\n└── docs/\n```\n\n## 4. Backend Service Boundaries\nNestJS modules under apps/api/src/modules:\n- AuthModule: JWT issuance, email/phone auth, password reset, OAuth.\n- UsersModule: profile, addresses, GDPR purge.\n- CategoriesModule: tree CRUD (admin).\n- BusinessesModule: profile, services, staff, hours, gallery.\n- SearchModule: text + filter search; uses PostGIS for distance.\n- GeoModule: bbox queries, pin clustering data.\n- AvailabilityModule: slot engine (15-min granularity, breaks, holidays).\n- BookingModule: transactional booking, conflict detection, Redis lock.\n- AppointmentsModule: state machine, cancel/reschedule.\n- PaymentsModule: Stripe integration, refunds.\n- ReviewsModule: verified reviews, replies, flagging.\n- NotificationsModule: enqueue email/push jobs.\n- JobsModule: BullMQ queues (reminders, emails, slot cache).\n- ProviderModule: portal dashboard, manual blocks.\n- AdminModule: suspend, disputes, metrics.\n\nModules depend only on PrismaService, RedisService, and shared DTOs. No cross-module DB access.\n\n## 5. Data Model\nPrisma schema with PostGIS extension. Key models: User, Business (location Point), Category, Service, Staff, Appointment, Review, Favorite, Payment. Migrations applied via Prisma. Supabase provides hosted Postgres+PostGIS and storage buckets for images.\n\n## 6. Frontend Architecture\n- Expo Router file-based routing: app/(tabs)/home.tsx, app/business/[id].tsx, app/booking/index.tsx.\n- TanStack React Query for server state; hooks in libs/data-access call fetch wrapper.\n- React Native Reanimated for bottom sheets, map preview cards, transitions.\n- libs/ui provides accessible components (contrast, a11y).\n- libs/shared holds types generated from backend DTOs to keep contract.\n\n## 7. Cross-Cutting Concerns\n- Auth: JWT in Expo SecureStore; NestJS Guards protect routes.\n- Caching: Redis caches search results and slot computations (TTL 5m).\n- Background: BullMQ workers consume Redis queues; API non-blocking.\n- Notifications: Expo push + email via Supabase SMTP; opt-out flag.\n- CI: GitHub Actions runs pnpm nx affected test lint build; Jest for unit.\n- Local: docker-compose up starts postgres-postgis and redis.\n- Build: EAS Build for mobile binaries; Expo web build for portals.\n\n## 8. Key Flows\n### Booking\n1. GET /availability?businessId=&serviceId=&date=\n2. AvailabilityModule returns slots (cached or computed).\n3. POST /bookings with slot + payment token.\n4. BookingModule transaction + Redis lock ensures no double-book.\n5. PaymentsModule charges Stripe; on fail rollback.\n6. NotificationsModule enqueues confirmation job.\n7. Appointment persisted; provider portal updates.\n
