Overview\n\nThis document describes the complete system architecture for the Planity Clone, a mobile-first marketplace connecting customers with local service businesses. Service Boundaries & Modules\n\nThe backend is organized into NestJS modules, each with clear responsibilities:\n\n### 4.1 Auth Module (P0)\n- Handles registration, login, OAuth, password reset, session management.\n- Uses JWT (access + refresh tokens) stored in Redis for blacklisting.\n- Rate limiting with Redis-based sliding window.\n\n### 4.2 User Module (P0)\n- Manages user profiles (customers, providers, admins).\n- Role-based access control (RBAC) via guards.\n\n### 4.3 Business Module (P0)\n- CRUD for businesses, staff, services, categories.\n- Search with full-text (PostgreSQL tsvector) and PostGIS for spatial queries.\n- Business hours and holiday management.\n\n### 4.4 Booking Module (P0)\n- Slot computation engine (15/30/60 min increments, buffer times, staff availability).\n- Booking lifecycle: hold, confirm, reschedule, cancel, no-show.\n- Inventory lock with Redis (10-minute hold during payment).\n\n### 4.5 Payment Module (P1)\n- Integration with Stripe (or similar) for processing payments.\n- Refund logic.\n\n### 4.6 Notification Module (P1)\n- Push notifications via Expo Push API or Firebase Cloud Messaging.\n- Email notifications (password reset, booking confirmations).\n\n### 4.7 Review Module (P1)\n- Customer reviews and ratings for businesses and staff.\n\n### 4.8 Analytics Module (P2)\n- Search analytics, booking trends, provider dashboards.\n\n## 5. Data Flow & API Design\n\n### 5.1 API Layer\n- RESTful API with NestJS controllers.\n- All endpoints are prefixed with `/api/v1`.\n- Authentication via `Authorization: Bearer <token>` header.\n- Pagination, filtering, sorting via query parameters (standardized).\n\n### 5.2 Key Endpoints\n\n| Method | Path | Description |\n|--------|------|-------------|\n| POST | /api/v1/auth/register | Register new user |\n| POST | /api/v1/auth/login | Login, returns tokens |\n| POST | /api/v1/auth/refresh | Refresh access token |\n| GET | /api/v1/businesses | List/search businesses (with filters, pagination) |\n| GET | /api/v1/businesses/:id | Business detail with services, staff, hours |\n| GET | /api/v1/businesses/:id/slots | Available slots for a date |\n| POST | /api/v1/bookings | Create booking (with payment if required) |\n| GET | /api/v1/bookings/me | Customer's appointments |\n| GET | /api/v1/providers/me/appointments | Provider's appointments |\n\n### 5.3 Real-time Updates\n- WebSocket gateway (NestJS) for live slot availability updates and booking confirmations.\n- Used for provider dashboard and customer booking flow.\n\n## 6. Database Schema (Prisma + PostgreSQL + PostGIS)\n\nKey models:\n\n- **User**: id, email, passwordHash, role, profile fields, timestamps.\n- **Business**: id, name, description, location (PostGIS geography), categoryIds, hours (JSON), images, verified, timestamps.\n- **Service**: id, businessId, name, duration, price, description, categoryId.\n- **Staff**: id, businessId, userId, specialties, schedule (JSON), timestamps.\n- **Appointment**: id, businessId, customerId, staffId, serviceId, startTime, endTime, status, paymentStatus, timestamps.\n- **Review**: id, appointmentId, rating, comment, timestamps.\n- **Category**: id, name, parentId, icon, coverImage.\n\nPostGIS enables spatial queries like `ST_DWithin` for distance-based search.\n\n## 7. Caching Strategy (Redis)\n\n- **Session store**: Blacklisted tokens, rate limiting counters.\n- **Slot cache**: Computed available slots per business+date, TTL 5 minutes, invalidated on booking changes.\n- **Business detail cache**: Full business profile, TTL 10 minutes.\n- **Search results cache**: For popular queries, TTL 2 minutes.\n\n## 8. Mobile App Architecture\n\n### 8.1 Routing (Expo Router)\n- File-based routing in `apps/mobile/src/app/`.\n- Protected routes via middleware checking authentication state.\n- Deep linking support for booking confirmations and business sharing.\n\n### 8.2 State Management\n- **Server state**: TanStack React Query for all API data (caching, refetching, optimistic updates).\n- **Local state**: Zustand for UI state (modals, filters, guest session).\n- **Persisted state**: AsyncStorage for tokens, guest session, search history.\n\n### 8.3 UI Components\n- Reusable components in `libs/ui-kit` built with React Native Reanimated for smooth animations (e.g., slot selection, map interactions).\n- Design tokens (colors, typography, spacing) defined as constants and consumed via a theme provider.\n\n## 9. Infrastructure & Deployment\n\n### 9.1 Local Development\n- Docker Compose runs PostgreSQL (with PostGIS), Redis, and the NestJS API.\n- Expo development server runs on host machine.\n- Environment variables managed via `.env` files.\n\n### 9.2 CI/CD (GitHub Actions)\n- On push to `main`/`develop`: lint, type-check, run tests (Jest).\n- On release tag: build mobile app with EAS Build (iOS/Android), deploy API to Supabase.\n- Database migrations run as part of API deployment.\n\n### 9.3 Hosting\n- **Backend**: Supabase (PostgreSQL + PostGIS + Redis + serverless functions for NestJS).\n- **Mobile**: EAS Build for OTA updates and app store submissions.\n- **File storage**: Supabase Storage for business images, user avatars.\n\n## 10. Testing Strategy\n\n- **Unit tests**: Jest for backend services and frontend hooks/components.\n- **Integration tests**: Test NestJS modules with test database (Docker).\n- **E2E tests**: Detox or Maestro for mobile flows (booking, auth).\n- **CI pipeline**: Run all tests in parallel, fail fast.\n\n## 11. Security Considerations\n\n- JWT tokens with short expiry (15 min access, 7 day refresh).\n- Refresh token rotation and blacklisting on logout.\n- Rate limiting on auth endpoints (5 attempts per 15 min).\n- Input validation with class-validator (NestJS pipes).\n- SQL injection prevention via Prisma parameterized queries.\n- CORS configured for mobile app origin.\n- HTTPS enforced in production.\n\n## 12.
