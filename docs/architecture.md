The architecture is designed for scalability, maintainability, and a clean separation of concerns across a React Native frontend (Expo) and a NestJS backend, orchestrated within an Nx monorepo. Monorepo Structure\n\nThe project uses an Nx monorepo with pnpm workspaces. Shared code is placed under `libs/`, while deployable units are in `apps/`.\n\n```\nplanity-clone/\n‚îú‚îÄ‚îÄ apps/\n‚îÇ   ‚îú‚îÄ‚îÄ mobile/                      # Expo managed workflow\n‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ app/                     # Expo Router pages (file-based)\n‚îÇ   ‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ (tabs)/              # Tab navigator\n‚îÇ   ‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ (auth)/              # Auth screens\n‚îÇ   ‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ booking/             # Multi-step booking flow\n‚îÇ   ‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ ...\n‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ components/              # Screen-specific components\n‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ hooks/                   # React Query hooks, custom hooks\n‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ utils/                   # Date, formatting, validation\n‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ app.json\n‚îÇ   ‚îî‚îÄ‚îÄ api/                         # NestJS application\n‚îÇ       ‚îú‚îÄ‚îÄ src/\n‚îÇ       ‚îÇ   ‚îú‚îÄ‚îÄ modules/             # Feature modules\n‚îÇ       ‚îÇ   ‚îú‚îÄ‚îÄ common/              # Guards, interceptors, pipes\n‚îÇ       ‚îÇ   ‚îú‚îÄ‚îÄ config/              # Environment configuration\n‚îÇ       ‚îÇ   ‚îî‚îÄ‚îÄ main.ts\n‚îÇ       ‚îú‚îÄ‚îÄ prisma/\n‚îÇ       ‚îÇ   ‚îî‚îÄ‚îÄ schema.prisma\n‚îÇ       ‚îî‚îÄ‚îÄ Dockerfile\n‚îú‚îÄ‚îÄ libs/\n‚îÇ   ‚îú‚îÄ‚îÄ shared-types/                # DTOs, interfaces, enums\n‚îÇ   ‚îú‚îÄ‚îÄ shared-ui/                   # Reusable UI components (BusinessCard, StarRating, ...)\n‚îÇ   ‚îú‚îÄ‚îÄ shared-utils/                # Business logic helpers, distance calcs\n‚îÇ   ‚îú‚îÄ‚îÄ shared-validators/           # Zod schemas for forms & API\n‚îÇ   ‚îî‚îÄ‚îÄ api-client/                  # Generated typed fetch client for mobile\n‚îú‚îÄ‚îÄ tools/                           # Custom Nx executors, scripts\n‚îú‚îÄ‚îÄ docker-compose.yml               # Postgres + Redis\n‚îú‚îÄ‚îÄ nx.json\n‚îú‚îÄ‚îÄ pnpm-workspace.yaml\n‚îú‚îÄ‚îÄ tsconfig.base.json\n‚îî‚îÄ‚îÄ package.json\n```\n\n## 4. Backend Architecture\n\n### 4.1 NestJS Modular Design\n\nThe backend follows a layered architecture with modules, controllers, services, and repositories.\n\n**Core Modules**\n\n- **AuthModule**: Registration, login, token refresh, password reset, social login (Google, Apple). Issues JWT with role claims. Uses `@nestjs/passport`, `passport-jwt`. Stores refresh tokens in Redis (hashed).\n- **UsersModule**: Profile management, onboarding for providers, role assignment.\n- **BusinessModule**: CRUD for businesses, geolocation queries, business details, working hours, amenities.\n- **ServicesModule**: Service catalog management, category-service associations.\n- **CategoriesModule**: Hierarchical categories (tree structure). Admin-managed.\n- **AvailabilityModule**: Provider schedule, time-off, unavailability; slot computation engine.\n- **BookingModule**: Core booking flow, temporary slot reservation (Redis lock), confirmation, rescheduling, cancellation. Handles payment integration and status transitions.\n- **ReviewModule**: User reviews, ratings, moderation.\n- **FavoritesModule**: Customer saved businesses.\n- **PaymentModule**: Stripe integration, pre-payment, pay-at-venue.\n- **NotificationModule**: Push notification dispatch (Firebase via BullMQ), email via templates (BullMQ).\n- **AdminModule**: Dashboard analytics, user management, moderation.\n- **LocationModule**: Geocoding, reverse geocoding, map clustering data.\n- **Common**: Logging, exception filters, validation pipe (Zod), interceptors for caching.\n\n**Service Boundaries**\n\nEach module owns its own data access logic and exposes clear service interfaces. Cross-module communication is minimized; when needed, events are emitted via NestJS `EventEmitter` or queued via BullMQ (e.g., booking status change triggers notifications without direct module coupling).\n\n**Data Flow Example ‚Äì Booking**\n\n1. `BookingController` receives POST `/bookings/reserve` with service, staff, date, time.\n2. `BookingService` validates business hours, staff availability (`AvailabilityService`), and checks for concurrent reservations using Redis key `slot:<business>:<staff>:<date>-<time>` with a 5-minute TTL.\n3. If available, creates a `reserved` booking in PostgreSQL, sets slot lock in Redis.\n4. Upon confirmation (`PUT /bookings/confirm`), lock is renewed into permanent reservation, payment processed, and a BullMQ job is queued to send notifications and update the provider's calendar.\n5. If reservation expires (Redis TTL), a cleanup job removes the reserved record.\n\n### 4.2 Database Schema (Prisma with PostgreSQL+PostGIS)\n\nHigh-level tables:\n\n- `User` (id, email, passwordHash, role, name, phone, avatarUrl, ...)\n- `Business` (id, ownerId, name, description, categoryIds, address, location(geometry), phone, website, avgRating, ...)\n- `BusinessCategory` (id, name, icon, parentId)\n- `Service` (id, businessId, name, duration, price, categoryId, image)\n- `Staff` (id, businessId, userId, name, image)\n- `WorkingHour` (businessId, dayOfWeek, openTime, closeTime)\n- `TimeOff` (staffId, date, startTime, endTime, reason)\n- `Booking` (id, customerId, businessId, staffId, totalPrice, notes, status, ...)\n- `BookingItem` (bookingId, serviceId, price)\n- `Review` (id, userId, businessId, rating, comment, ...)\n- `Favorite` (userId, businessId)\n- `Payment` (id, bookingId, amount, method, status)\n\nGeospatial features leverage PostGIS: the `Business.location` column is of type `geometry(Point, 4326)`. Queries for the map view use `ST_DWithin`, and search results are ordered by distance with `ST_Distance`. Clustering is performed server-side (using `ST_ClusterDBSCAN` or by pre-aggregating clusters in a view).\n\nPrisma schema includes native PostGIS support via `prisma-postgis` or using raw SQL for advanced functions. Migrations are managed with `prisma migrate`.\n\n### 4.3 Authentication & Authorization\n\n- Access token (short-lived, 15 min) and refresh token (7 days) JWT, both signed with RS256. Refresh token stored in Redis with hash. On logout, token is deleted.\n- Social login: `/auth/google/callback` exchanges authorization code for Google tokens, verifies identity, and creates or links user account.\n- Roles: `customer`, `provider`, `admin`. Custom decorators and guards (`@Roles()`, `RolesGuard`) protect endpoints.\n- Profile completion middleware for providers: after registration, if business details missing, requests are redirected to onboarding until completed.\n\n### 4.4 Caching & Background Jobs\n\n**Redis Usage**\n\n- Temporary slot locks (booking reservation). Key: `reserve:<businessId>:<staffId>:<date>:<slot>`, value `userId`, TTL 300s.\n- Refresh token store.\n- Response caching for popular business details (TTL 5 min) via NestJS cache-interceptor.\n- Rate limiting for authentication endpoints.\n\n**BullMQ Job Queues**\n\n- `notifications`: send push (Firebase) and email (via template service) for booking confirmations, reminders, cancellations.\n- `review-moderation`: flag suspicious content.\n- `slot-clearing`: release expired reserved bookings.\n- `analytics`: compute daily metrics for admin dashboard.\n\nThe queue is backed by Redis; workers run as separate processes (using `@nestjs/bullmq`). This keeps the API responsive and decouples heavy processing.\n\n### 4.5 API Design Principles\n\n- RESTful endpoints with JSON. OpenAPI spec generated from NestJS decorators.\n- Versioning via URI prefix `/api/v1/`.\n- Consistent error format: `{ statusCode, message, errors? }`.\n- Pagination uses cursor-based for infinite scroll (e.g., `/businesses?cursor=...&limit=20`).\n- Input validation using Zod schemas shared between frontend and backend (via `libs/shared-validators`).\n- Real-time updates for booking status changes can be optionally delivered via Supabase Realtime (WebSockets) if needed, but push notifications are the primary channel.\n\n## 5. Mobile Architecture\n\n### 5.1 Expo Router & Navigation\n\nThe app uses file-based routing with Expo Router. Layouts define shared tab navigator and auth guards.\n\n- `app/(tabs)/` contains home, search, bookings, favorites, profile.\n- `app/business/[id]/` shows business detail with nested tabs.\n- `app/booking/` holds a stack for the multi-step flow, preserving state via local React context and React Query cache.\n- `app/(auth)/` has sign-in, sign-up, forgot password.\n- `app/map/` displays the map search.\n\nDeep linking to a booking confirmation or business detail is supported using dynamic routes.\n\n### 5.2 Data Flow with TanStack React Query\n\nAll server state (businesses, bookings, reviews, user profile) is managed through React Query. Each API endpoint corresponds to a custom hook in `apps/mobile/hooks/`:\n\n- `useBusinesses(params)`: infinite query for search results.\n- `useBusiness(id)`: detail query.\n- `useAvailableSlots(businessId, staffId, date)`: fetches real-time slots (refetches on mount and periodically).\n- `useReserveSlot()`: mutation with optimistic update and slot lock handling.\n- `useBookings()`: customer's bookings list.\n\n**Caching strategy:**\n\n- Stale time: business listings 2 min, slots 30 sec, user profile 5 min.\n- Cache invalidation: on mutation success (e.g., after booking, invalidate `useBookings` and `useAvailableSlots`).\n- Optimistic updates for favorite toggling and review submission.\n\n### 5.3 State Management\n\nFor complex local UI state (e.g., booking flow selections), a React context + reducer or Zustand (lightweight) is used. The `BookingContext` holds selected services, staff, date, time, and promo code. The state survives navigation via React context, not serialized to URL.\n\nAuth state is managed by a `useAuth` hook wrapping React Query and secure token storage (expo-secure-store). On app launch, it checks for existing refresh token and silently refreshes the session.\n\n### 5.4 UI & Animations\n\n- Shared UI components (`shared-ui`) like `BusinessCard`, `ServiceItem`, `TimeSlotGrid`, `RatingStars` are built using `react-native-reanimated` for fluid interactions (bounce on favorite, slide-in calendar).\n- Skeleton loaders and shimmer effects ensure smooth loading transitions.\n- Map screen uses `react-native-maps` (Expo) with clustering using `supercluster` or `react-native-map-clustering`.\n- Form validation on the mobile uses shared Zod schemas for consistency with backend.\n\n### 5.5 API Client\n\nThe `api-client` library exports a typed fetch wrapper (using `axios` or `fetch`) pre-configured with base URL, auth interceptor (attaches Bearer token, handles 401 refresh logic). Generated from OpenAPI spec to ensure type safety, or manually maintained with TypeScript types from `shared-types`.\n\n## 6. Infrastructure & Deployment\n\n### Local Development Environment\n\n`docker-compose.yml` spins up:\n\n```yaml\nservices:\n  postgres:\n    image: postgis/postgis:15-3.3\n    environment: ...\n    ports: [5432]\n  redis:\n    image: redis:7-alpine\n    ports: [6379]\n```\n\nThe NestJS API and Expo dev server run directly on the host for hot reload. pnpm scripts orchestrate starting both (e.g., `pnpm dev` runs `nx serve api` and `nx start mobile`).\n\n### 6.2 CI/CD Pipeline\n\nGitHub Actions workflow:\n- On pull request: lint, type-check, unit tests (Jest) for `apps/api`, `libs/*`, and `apps/mobile` (with mock backend). Run Prisma migrations check.\n- On merge to `main`:\n  - Build and push API Docker image to container registry.\n  - Deploy API to staging/production (e.g., Fly.io, Railway, or custom VPS).\n  - Run EAS Build for mobile (profile `development` or `preview`) and submit to TestFlight/Play Console for testing.\n  - Run database migrations automatically (with safety checks).\n\n### 6.3 Production Infrastructure\n\n- **API**: Containerized NestJS, stateless, horizontally scalable behind a load balancer. Uses environment variables for DB URL, Redis, and secrets.\n- **Database**: Managed PostgreSQL with PostGIS extension (e.g., Supabase, AWS RDS, or self-hosted). Connection pooling via PgBouncer.\n- **Redis**: Managed Redis instance (ElastiCache, Upstash) for caching and queues.\n- **File Storage**: Supabase Storage buckets (backed by S3) for user uploads; public access through CDN.\n- **Push Notifications**: Firebase Cloud Messaging (FCM).\n- **Monitoring**: Logging via structured JSON, APM with Sentry or OpenTelemetry.\n\n## 7. Mobile: unit tests for hooks with `@testing-library/react-hooks`, component tests with `@testing-library/react-native`.\n\n## 8.
