It defines the folder structure, service boundaries, technology choices, and design principles. The architecture follows clean architecture principles with clear separation of concerns, enabling maintainability, testability, and scalability.\n\n## 2. Technology Stack\n\n### Frontend (Mobile & Web)\n- **Framework:** Expo SDK 50+ with React Native\n- **Language:** TypeScript (strict mode)\n- **Routing:** Expo Router (file-based routing)\n- **State Management:** TanStack React Query (server state), Zustand (client state)\n- **Animations:** React Native Reanimated 3\n- **Maps:** react-native-maps (Mapbox/Google Maps)\n- **Notifications:** Expo Notifications\n- **Auth:** Supabase Auth (social, magic link, OTP)\n- **Storage:** AsyncStorage, expo-secure-store\n- **Build:** EAS Build (iOS, Android), Expo for web\n\n### Backend\n- **Framework:** NestJS (modular monolith)\n- **Language:** TypeScript\n- **ORM:** Prisma (with PostgreSQL)\n- **Spatial:** PostGIS (geolocation queries)\n- **Cache:** Redis (session, rate limiting, real-time slots)\n- **API:** RESTful (primary) + WebSocket (real-time booking holds)\n- **Auth:** JWT (access + refresh tokens), httpOnly cookies\n\n### Infrastructure\n- **Monorepo:** Nx + pnpm (workspaces)\n- **Containerization:** Docker Compose (local dev)\n- **CI/CD:** GitHub Actions\n- **Hosting:** Supabase (auth, database, storage), Vercel (web), EAS (mobile)\n- **Testing:** Jest (unit), React Native Testing Library (component), Cypress (E2E)\n\n## 3. Service Boundaries\n\n### 4.1 Auth Service\n- **Responsibility:** User registration, login, social OAuth, token management, password reset, biometric setup.\n- **Data:** Users table, refresh_tokens table.\n- **Dependencies:** Supabase Auth (external), Redis (rate limiting).\n- **API Endpoints:**\n  - `POST /auth/register`\n  - `POST /auth/login`\n  - `POST /auth/refresh`\n  - `POST /auth/logout`\n  - `POST /auth/forgot-password`\n  - `POST /auth/reset-password`\n  - `POST /auth/verify-email`\n  - `POST /auth/verify-phone`\n\n### 4.2 User Service\n- **Responsibility:** User profile management, preferences, favorites, search history.\n- **Data:** Users table (profile fields), favorites, search_history.\n- **API Endpoints:**\n  - `GET /users/me`\n  - `PATCH /users/me`\n  - `GET /users/me/favorites`\n  - `POST /users/me/favorites`\n  - `DELETE /users/me/favorites/:businessId`\n\n### 4.3 Business Service\n- **Responsibility:** Business CRUD, search (full-text + spatial), autocomplete, categories, staff management.\n- **Data:** Businesses, categories, staff, operating_hours, photos.\n- **Dependencies:** PostGIS (spatial queries), Redis (caching popular searches).\n- **API Endpoints:**\n  - `GET /businesses` (search with filters, pagination)\n  - `GET /businesses/:id`\n  - `POST /businesses` (owner only)\n  - `PATCH /businesses/:id`\n  - `GET /businesses/:id/staff`\n  - `GET /businesses/:id/availability` (with date range)\n  - `GET /categories`\n\n### 4.4 Booking Service\n- **Responsibility:** Appointment lifecycle: create, hold, confirm, reschedule, cancel, check-in.\n- **Data:** Appointments, appointment_services, holds.\n- **Dependencies:** Redis (slot holds with TTL), WebSocket (real-time slot updates).\n- **API Endpoints:**\n  - `POST /bookings` (initiate with hold)\n  - `POST /bookings/:id/confirm` (after payment)\n  - `POST /bookings/:id/reschedule`\n  - `POST /bookings/:id/cancel`\n  - `GET /bookings` (user's bookings)\n  - `GET /bookings/:id`\n\n### 4.5 Review Service\n- **Responsibility:** Review creation, moderation, aggregation.\n- **Data:** Reviews table.\n- **API Endpoints:**\n  - `POST /businesses/:id/reviews`\n  - `GET /businesses/:id/reviews`\n  - `PATCH /reviews/:id` (owner only)\n  - `DELETE /reviews/:id`\n\n### 4.6 Notification Service\n- **Responsibility:** Push notifications, email, SMS for booking confirmations, reminders, cancellations.\n- **Data:** Notification_logs.\n- **Dependencies:** Expo Push API, SendGrid (email), Twilio (SMS).\n- **Triggers:** Event-driven (via NestJS EventEmitter or Redis pub/sub).\n\n### 4.7 Admin Service\n- **Responsibility:** Platform analytics, user management, business verification, content moderation.\n- **Data:** Admin-specific views (aggregated).\n- **API Endpoints:** Protected by admin role guard.\n\n## 5. Key Architectural Decisions\n\n### 6.1 Monorepo with Nx\n- **Why:** Shared types, validation schemas, and UI components across mobile and web. Nx provides dependency graph, caching, and generators.\n- **Impact:** Faster builds, consistent code quality, easy refactoring.\n\n### 6.2 Modular Monolith (NestJS)\n- **Why:** MVP phase benefits from simplicity of a single deployable unit. Modules are organized by domain (auth, bookings, etc.) with clear boundaries. Future extraction to microservices is possible via module isolation.\n- **Impact:** Faster development, simpler deployment, but requires discipline to avoid tight coupling.\n\n### 6.3 TanStack React Query for Server State\n- **Why:** Automatic caching, background refetching, optimistic updates, and pagination support. Reduces boilerplate for API calls.\n- **Impact:** Smooth UX with loading/error states, reduced network requests.\n\n### 6.4 PostGIS for Spatial Queries\n- **Why:** Efficient geolocation queries (nearby businesses, bounding box). Native PostgreSQL extension avoids additional service.\n- **Impact:** Fast map-based search, no external geocoding service dependency.\n\n### 6.5 Redis for Real-Time Slot Holds\n- **Why:** In-memory key-value store with TTL for booking holds. Prevents double-booking without database locks.\n- **Impact:** Low latency, scalable, simple implementation.\n\n### 6.6 Expo Router for Navigation\n- **Why:** File-based routing simplifies navigation structure, supports deep linking, and is well-suited for mobile/web.\n- **Impact:** Cleaner code, easier to add new screens.\n\n## 7. API Design Principles\n\n- **RESTful:** Resources mapped to endpoints, HTTP verbs for actions.\n- **Pagination:** Cursor-based for lists (e.g., bookings, reviews).\n- **Error Handling:** Consistent error response format: `{ error: { code, message, details } }`.\n- **Versioning:** URL prefix `/v1/` for future iterations.\n- **Rate Limiting:** 100 requests/min per user, 20 requests/min for auth endpoints.\n\n## 8. Security Considerations\n\n- **Authentication:** JWT access tokens (15 min) stored in memory; refresh tokens (7 days) in httpOnly cookies.\n- **Authorization:** Role-based guards (CUSTOMER, BUSINESS_OWNER, ADMIN).\n- **Data Validation:** Zod schemas shared between frontend and backend.\n- **SQL Injection:** Prevented by Prisma parameterized queries.\n- **XSS:** Input sanitization, Content Security Policy headers.\n- **CORS:** Whitelist allowed origins.\n\n## 9. Testing Strategy\n\n- **Unit Tests:** Jest for services, utilities, hooks.\n- **Integration Tests:** NestJS testing module with test database (PostgreSQL).\n- **Component Tests:** React Native Testing Library for UI components.\n- **E2E Tests:** Detox for mobile, Cypress for web.\n- **CI/CD:** GitHub Actions runs tests on every PR; EAS Build for mobile deployments.\n\n## 10. Deployment\n\n- **Local Development:** Docker Compose (PostgreSQL, Redis, API).\n- **Staging:** Supabase (DB, Auth), Vercel (API), EAS (mobile builds).\n- **Production:** Same as staging with horizontal scaling (API behind load balancer).\n\n## 11.
