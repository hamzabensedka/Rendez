System Overview\nPlanity Clone is a mobile-first beauty & wellness booking platform. Backend Architecture (NestJS)\n### 6.1 Module Boundaries\nEach module encapsulates a bounded context and exposes a well-defined API (service interface).\n\n- **User Module:** Handles registration, auth (local + social), profile CRUD, payment method management, notification preferences. Contains `AuthService` (JWT issuance, refresh token rotation), `UserService`.\n- **Catalog Module:** Business & Service discovery. Manages business profiles, operating hours, staff, services, categories, and reviews. Heavily relies on PostGIS for `findNearby` queries.\n- **Booking Module:** The core transaction module. Orchestrates the booking flow: validates availability, reserves a slot atomically in Redis, creates appointments, integrates with Payment Module. Emits events (`booking.created`, `booking.cancelled`).\n- **Availability Module (Sub-module of Booking):** A high-performance service responsible for computing available time slots. Considers staff schedules, existing appointments, service durations, buffer times, and business holidays. Uses Redis for caching pre-computed slot grids and atomic slot reservations.\n- **Payment Module:** Abstraction over Stripe. Handles payment intent creation, webhook processing, and linking payment methods to customers. Never stores raw card details.\n- **Notification Module:** Abstracts push notifications (via Firebase Cloud Messaging) and email (SendGrid). Listens to application events (e.g., `booking.confirmed`) to trigger delivery.\n- **Admin Module:** Dashboard with aggregated metrics, category management, and moderation tools.\n\n### 6.2 Data Flow Example: Booking a Slot\n1.  **Client Request:** `POST /bookings/reserve` with `{ serviceId, staffId, date, startTime }`.\n2.  **BookingController:** Validates input, calls `BookingService.reserveSlot()`.\n3.  **BookingService:**\n    - Calls `AvailabilityService.isSlotAvailable()` (checks Redis cache, falls back to DB query).\n    - If available, atomically sets a Redis key `slot:reservation:{businessId}:{staffId}:{date}:{startTime}` with a TTL of 10 minutes.\n    - Returns a reservation token.\n4.  **Client:** Proceeds to confirmation step, calls `POST /bookings/confirm` with payment details.\n5.  **BookingService:**\n    - Calls `PaymentService.charge()`.\n    - On success: creates `Appointment` in DB, removes Redis reservation TTL, extends key to match appointment, emits `booking.confirmed`.\n    - On failure: releases Redis reservation, returns error.\n\n### 6.3 Database Schema Highlights (Prisma + PostGIS)\n- **`Business` model:** Has `coordinates` field of PostGIS `Geometry(Point, 4326)`. Indexed using GIST.\n- **`Appointment` model:** Contains `status` enum (`PENDING`, `CONFIRMED`, `CANCELLED`, `COMPLETED`). Relations to `Customer`, `Staff`, `Service`.\n- **`Staff` model:** Has `businessId` and a `services` many-to-many relation.\n- **`Review` model:** Includes rating (1-5) and text, ensuring one review per customer per appointment.\n- Geospatial queries leverage Prisma's raw SQL support or a dedicated `PostgisService` using `$queryRaw` for complex operations like `ST_DWithin` and clustering.\n\n## 7. Custom hooks (`useBusinesses`, `useAppointments`) encapsulate query keys and caching policies.\n  - **Stale Time:** Stale times vary by resource: business details (2 min), search results (30 sec), user profile (5 min).\n  - **Optimistic Updates:** Used for favorites toggling and appointment cancellation for instant UI feedback.\n  - **Infinite Queries:** `useInfiniteQuery` for search results and reviews pagination.\n- **Client State:** **Zustand** stores handle purely local UI state (e.g., `useAuthStore` for token presence, `useBookingFlowStore` for wizard step data). This avoids prop-drilling in complex flows.\n\n### 7.3 API Layer\nA centralized API client (`lib/api.ts`) wraps `fetch` or `axios`, automatically attaching JWT tokens from the Zustand auth store and handling refresh token logic in a queue to prevent race conditions.\n\n## 8. Geospatial & Search Strategy\nTo support “nearby salons,” we use PostgreSQL with PostGIS.\n- **Indexing:** GIST index on `Business.coordinates`.\n- **Query Pattern:** When a user searches, the API receives `lat`, `lng`, and `radius`. A raw SQL query via Prisma calculates `ST_Distance` and filters `WHERE ST_DWithin(...)`.\n- **Redis Caching:** Frequently searched areas (grid cells) have their results cached for 5 minutes to reduce DB load.\n- **Autocomplete:** A Redis Sorted Set or PostgreSQL trigram index (`pg_trgm`) on `Business.name` provides fast autocomplete suggestions.\n\n## 9. Slot Availability Algorithm\nThis is the most computationally intensive part. The `AvailabilityEngine`:\n1.  Fetches the **staff’s working hours** for the requested date.\n2.  Fetches all **existing appointments** and **blocked time** (breaks, holidays) for that staff member.\n3.  Calculates available windows by subtracting blocked times from working hours.\n4.  Generates time slots based on the **service duration** and any **buffer time**.\n5.  Caches the generated grid in Redis (`availability:{businessId}:{staffId}:{date}`) with a TTL that expires when any appointment for that staff/date changes.\n6.  **Slot Reservation:** When a user clicks a slot, the server atomically calls `SET NX` on the specific slot key in Redis. This guarantees no two users can book the same slot.\n\n## 10. Security Considerations\n- **Authentication:** Short-lived JWT access tokens (15 min), long-lived refresh tokens (7 days) stored in HTTP-only cookie or secure storage (Expo SecureStore). Refresh token rotation with reuse detection.\n- **Authorization:** NestJS Guards (`RolesGuard`, `OwnershipGuard`) ensure customers can only modify their own bookings/profile, and providers manage only their businesses.\n- **Input Validation:** Global `ValidationPipe` using class-validator & class-transformer DTOs. Rate limiting via `@nestjs/throttler` on auth and sensitive endpoints.\n- **Data Protection:** PII (email, phone) encrypted at rest. Passwords hashed with bcrypt. Stripe handles all PCI-compliant payment data; we store only references (`paymentMethodId`).\n\n## 11.
