Key Design Decisions & Patterns\n\n### 6.1 Availability Slot Computation\n\nThe booking service computes available slots dynamically based on:\n- Business regular hours (`BusinessHour`)\n- Existing bookings for the date\n- Service duration\n- Staff assignments (if applicable)\n- Active availability locks (5-minute TTL)\n\n**Algorithm**:\n1. Fetch business hours for the requested day of week.\n2. Generate all possible start times at 15-minute intervals within operating hours.\n3. Subtract slots that overlap with existing `Booking` records (status != CANCELLED).\n4. Subtract slots currently locked in `AvailabilityLock` (not expired).\n5. Return available slots with a flag for limited availability (< 3 slots remaining in next 2 hours).\n\n**Caching**: Computed slots for a given (businessId, serviceId, date) are cached in Redis with a 60-second TTL. Cache is invalidated when a booking is confirmed or a lock is acquired.\n\n### 6.2 Concurrency Control (Slot Locking)\n\nTo prevent double-booking:\n1. When a user selects a slot, the client calls `POST /booking/lock`.\n2. Backend checks if the slot is available (no booking, no active lock).\n3. If available, creates an `AvailabilityLock` record with a 5-minute TTL.\n4. Lock is released if:\n - User confirms booking (lock converted to booking).\n - User abandons checkout (lock expires naturally).\n - User explicitly releases lock (navigates back).\n5. Redis is used for fast lock checks: key = `lock:{businessId}:{date}:{startTime}:{staffId?}`.\n\n### 6.3 Geospatial Search\n\nPostGIS enables efficient location-based queries:\n```sql\nSELECT *, ST_Distance(location, ST_SetSRID(ST_MakePoint(:lng, :lat), 4326)) AS distance\nFROM \"Business\"\nWHERE ST_DWithin(location, ST_SetSRID(ST_MakePoint(:lng, :lat), 4326), :radius)\nORDER BY distance;\n```\nCombined with full-text search on business name and service names using PostgreSQL `tsvector`.\n\n### 6.4 Image Handling\n\n- Uploads go directly to Supabase Storage from the mobile client using pre-signed URLs.\n- Backend receives the Supabase object key and stores it in `BusinessPhoto.url`.\n- Images are served via Supabase CDN with on-the-fly transformations (resize, format).\n\n### 6.5 Push Notification Flow\n\n1. Booking confirmed → `BookingService` publishes event to Redis channel `booking.confirmed`.\n2. `NotificationService` subscribes to the channel, creates a `Notification` record, and dispatches via Expo Push API.\n3. Mobile client receives push, updates TanStack Query cache for bookings.\n\n### 6.6 Payment Flow\n\n1. Client collects card details via Stripe React Native SDK (PCI-compliant).\n2. Stripe returns a `paymentMethodId`.\n3. Client sends `paymentMethodId` to `POST /booking/confirm`.\n4. Backend creates a Stripe PaymentIntent and confirms it.\n5. Webhook `payment_intent.succeeded` updates `Payment.status` and triggers booking confirmation.\n\n---\n\n## 7.
