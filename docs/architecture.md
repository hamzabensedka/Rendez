Service Boundaries\n\n### 4.1 Auth Service\n- **Responsibility:** User registration, login, OAuth, token management, role-based access control.\n- **Data:** Users table, roles, refresh tokens.\n- **Endpoints:** `/auth/register`, `/auth/login`, `/auth/refresh`, `/auth/logout`, `/auth/oauth/:provider`.\n- **Dependencies:** Supabase Auth (optional), Redis (session store).\n\n### 4.2 Business Service\n- **Responsibility:** CRUD for businesses, staff profiles, services, categories, opening hours.\n- **Data:** Businesses, Staff, Services, Categories, OpeningHours tables.\n- **Endpoints:** `/businesses`, `/businesses/:id`, `/businesses/:id/staff`, `/businesses/:id/services`.\n- **Dependencies:** PostgreSQL, PostGIS (for location queries).\n\n### 4.3 Search Service\n- **Responsibility:** Full-text search, geo-spatial search, filtering, sorting, autocomplete.\n- **Data:** Search index (PostgreSQL full-text + PostGIS), Redis cache.\n- **Endpoints:** `/search`, `/search/autocomplete`, `/search/map`.\n- **Dependencies:** PostgreSQL, PostGIS, Redis.\n\n### 4.4 Booking Service\n- **Responsibility:** Appointment creation, availability management, booking flow, cancellations.\n- **Data:** Appointments, AvailabilitySlots, AddOns, Payments (via external).\n- **Endpoints:** `/bookings`, `/bookings/:id`, `/bookings/availability`, `/bookings/confirm`.\n- **Dependencies:** PostgreSQL, Redis (for real-time slot locking).\n\n### 4.5 Payment Service\n- **Responsibility:** Payment processing, refunds, commission calculation, payout to businesses.\n- **Data:** Transactions, PaymentMethods, Payouts.\n- **Endpoints:** `/payments/charge`, `/payments/refund`, `/payments/payout`.\n- **Dependencies:** Stripe API (or similar), PostgreSQL.\n\n### 4.6 Notification Service\n- **Responsibility:** Push notifications, emails, SMS for booking confirmations, reminders, promotions.\n- **Data:** NotificationLogs, Templates.\n- **Endpoints:** `/notifications/send`, `/notifications/preferences`.\n- **Dependencies:** Redis (queue), Firebase Cloud Messaging, SendGrid.\n\n### 4.7 Admin Service\n- **Responsibility:** Admin dashboard, analytics, user management, content moderation.\n- **Data:** AdminUsers, AuditLogs, AnalyticsEvents.\n- **Endpoints:** `/admin/*`.\n- **Dependencies:** PostgreSQL, Redis.\n\n## 5. Key Design Decisions\n\n### 7.1 Monorepo with Nx\n- **Why:** Shared code between mobile, web, and backend; consistent tooling; incremental builds.\n- **Impact:** Faster CI, easier dependency management, code reuse.\n\n### 7.2 Microservices vs. Modular Monolith\n- **Decision:** Start with modular monolith (NestJS modules) for faster development; extract to microservices when scaling.\n- **Rationale:** Avoid premature complexity; clear module boundaries allow future extraction.\n\n### 7.3 TanStack React Query for Server State\n- **Why:** Automatic caching, background refetching, optimistic updates, pagination.\n- **Impact:** Reduced boilerplate, better UX with offline support.\n\n### 7.4 Expo Router for Navigation\n- **Why:** File-based routing, deep linking, web support, TypeScript integration.\n- **Impact:** Simplified navigation structure, automatic deep link handling.\n\n### 7.5 PostGIS for Spatial Queries\n- **Why:** Efficient geo-spatial queries (radius search, clustering).\n- **Impact:** Fast map-based search, location-aware recommendations.\n\n### 7.6 Redis for Session and Caching\n- **Why:** Fast session store, rate limiting, real-time slot locking.\n- **Impact:** Improved performance, reduced database load.\n\n### 7.7 Supabase for Auth and Storage\n- **Why:** Managed auth (OAuth, magic links), file storage for images.\n- **Impact:** Reduced backend complexity, built-in security.\n\n## 8. API Design\n\n### 8.1 RESTful Conventions\n- Base URL: `/api/v1`\n- Authentication: Bearer JWT token in `Authorization` header\n- Pagination: Cursor-based with `cursor` and `limit` query params\n- Errors: Standard JSON error response with `code`, `message`, `details`\n- Versioning: URL-based (`/api/v1/...`)\n\n### 8.2 Key Endpoints\n\n| Method | Path | Description |\n|--------|------|-------------|\n| POST | /auth/register | Register new user |\n| POST | /auth/login | Login |\n| GET | /businesses | List businesses (with filters) |\n| GET | /businesses/:id | Business detail |\n| GET | /search | Full-text search |\n| GET | /search/map | Map-based search |\n| GET | /bookings/availability | Check availability |\n| POST | /bookings | Create booking |\n| POST | /payments/charge | Process payment |\n| GET | /notifications | List user notifications |\n\n## 9. Security Considerations\n\n- **Authentication:** JWT with short-lived access tokens (15 min) and long-lived refresh tokens (30 days).\n- **Authorization:** Role-based middleware in NestJS; guard decorators.\n- **Data Protection:** HTTPS only; password hashing with bcrypt; sensitive data encrypted at rest.\n- **Rate Limiting:** Redis-based rate limiting per user/IP for auth endpoints.\n- **Input Validation:** Class-validator decorators on all DTOs.\n- **CORS:** Whitelist allowed origins.\n\n## 10. Performance & Scalability\n\n- **Database Indexing:** Index on `Business.geom` (GIST), `Appointment.startTime`, `User.email`.\n- **Caching:** Redis cache for search results (TTL 5 min), business details (TTL 10 min).\n- **CDN:** Images served via Supabase Storage CDN.\n- **Horizontal Scaling:** Stateless NestJS services behind load balancer.\n- **Database Connection Pooling:** Prisma with PgBouncer.\n\n## 11. Testing Strategy\n\n- **Unit Tests:** Jest for services, utilities, and components.\n- **Integration Tests:** Test NestJS modules with test database.\n- **E2E Tests:** Detox (mobile), Playwright (web).\n- **CI Pipeline:** GitHub Actions runs lint, type-check, unit tests, integration tests on every PR.\n\n## 12. Deployment\n\n- **Mobile:** EAS Build for iOS/Android; EAS Submit for store deployment.\n- **Backend:** Docker containers deployed to cloud (AWS ECS or Google Cloud Run).\n- **Database:** Managed PostgreSQL (Supabase or AWS RDS).\n- **CI/CD:** GitHub Actions for automated build, test, and deploy.\n\n## 13. Monitoring & Observability\n\n- **Logging:** Structured JSON logs (Pino) shipped to CloudWatch.\n- **Metrics:** Prometheus metrics for request latency, error rates, database queries.\n- **Tracing:** OpenTelemetry for distributed tracing across services.\n- **Alerting:** PagerDuty integration for critical errors.\n\n## 14. Conclusion\n\nThis architecture provides a solid foundation for building the Planity Clone with clean separation of concerns, scalability, and maintainability. The monorepo structure with Nx enables code sharing and efficient development, while the modular backend allows future extraction into microservices as needed.",
