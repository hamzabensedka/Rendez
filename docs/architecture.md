The architecture follows clean architecture principles, separates concerns into well‑defined boundaries, and leverages an Nx monorepo with pnpm for efficient dependency management.\n\n---\n\n## High‑Level Components\n\n| Component | Technology | Responsibility |\n|-----------|------------|----------------|\n| **Mobile Client** | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated | UI, navigation, state management, gestures, offline‑optimized queries |\n| **API Gateway** | NestJS (Node.js) | Entry point for all HTTP/WebSocket requests, auth validation, request routing, rate limiting |\n| **Core Services** (NestJS modules) | NestJS, Prisma, PostgreSQL + PostGIS, Redis | Business logic split into bounded contexts: Auth, Users, Businesses, Services, Bookings, Payments, Reviews, Favorites, Notifications, Admin, Provider Portal |\n| **Data Store** | PostgreSQL (with PostGIS extension) | Relational data, geospatial indexes for map search |\n| **ORM** | Prisma | Type‑safe data access, migrations |\n| **Cache** | Redis | Session store, query caching, rate‑limit counters, BullMQ job queue backend |\n| **Background Jobs** | BullMQ (Redis‑based) | Asynchronous processing: email/SMS notifications, payment webhooks, reminder generation, data sync |\n| **DevOps & CI/CD** | Docker Compose (local), GitHub Actions, GitHub Actions, EAS Build (Expo), Nx, pnpm | Containerized services, automated testing, builds, deployments |\n| **Third‑Party** | Supabase (optional Auth & Storage), Payment Gateway (Stripe/PayPal) | Auth fallback, file storage, secure payments |\n\n---\n\n## Architecture Layers\n\n1. **Presentation Layer** (Mobile)\n   - **Expo Router** provides file‑based navigation (tabs, stack, modals).\n   - **TanStack React Query** manages server state, caching, background refetch, and optimistic updates.\n   - **React Native Reanimated** powers smooth animations and gesture‑based interactions (e.g., map drag, swipe‑to‑cancel).\n   - **Shared UI Library** (Nx lib) contains design system components (buttons, inputs, cards) and theme.\n   - **TypeScript** ensures end‑to‑end type safety via generated API clients (or manual DTOs).\n\n2. **API Layer** (NestJS)\n   - **Modular Structure**: each domain is a NestJS module with its own controllers, services, DTOs, and guards.\n   - **Guards & Interceptors**: JWT auth guard, role‑based access control (RBAC), logging, transformation, exception handling.\n   - **WebSocket Gateway** (NestJS) for real‑time notifications (booking updates, chat).\n   - **Validation** via class‑validator and class‑transformer.\n   - **OpenAPI (Swagger)** docs generated automatically.\n\n3. **Application Layer** (Service Classes)\n   - Encapsulates use‑cases: e.g., `CreateBookingUseCase`, `SearchBusinessesUseCase`.\n   - Depends on abstractions (repositories) defined in the domain layer.\n   - Utilizes **React Query**‑like patterns on the server for caching frequently accessed data (e.g., business listings).\n\n4. **Domain Layer**\n   - **Entities**: TypeScript classes/interfaces representing core concepts (User, Business, Service, Booking, Review, Payment).\n   - **Repositories**: Interfaces defining CRUD + custom query methods (e.g., `findNearby`, `findByCategory`).\n   - **Domain Services**: Business rules that span multiple entities (availability computation, slot generation, price calculation).\n\n5. **Infrastructure Layer**\n   - **Prisma Repositories**: concrete implementations of repository interfaces using Prisma Client.\n   - **PostGIS**: spatial indexes (`gist`) on `location` column for efficient radius/box queries.\n   - **Redis**: stores cached query results, session tokens, and BullMQ job queues.\n   - **BullMQ Workers**: separate NestJS microservices (or processes) that listen to queues: `notification-worker`, `payment-worker`, `reminder-worker`.\n\n---\n\n## Service Boundaries (Nested Nx Libraries)\n\n```
libs/
├─ api-gateway          # NestJS entry point (main.ts)
├─ auth                 # JWT strategies, refresh token handling
├─ user                 # User CRUD, profile, settings
├─ business             # Business CRUD, geospatial search, categories
├─ service              # Service catalog tied to businesses
├─ booking              # Booking lifecycle, slot generation, conflict detection
├─ payment              # Payment intent creation, webhook handling, receipts
├─ review               # Ratings & reviews CRUD, aggregation
├─ favorite             # User‑business many‑to‑many
├─ notification         # Template service, dispatch to BullMQ
├─ admin                # Admin‑only endpoints (content moderation, user mgmt)
├─ provider-portal      # Business owner dashboard APIs
├─ shared               # DTOs, interfaces, enums, constants, utils
└─ infra                 # Prisma client # PrismaClient wrapper, repository base classes
```
Each library is **lint‑able, test‑able, and versioned** within the Nx workspace. Dependencies flow inward: UI → API Gateway → Domain → Infra. No circular dependencies are allowed (enforced by Nx).\n\n---\n\n## Data Model Highlights\n\n- **User**: id, email, passwordHash, role (user|provider|admin), createdAt, updatedAt.\n- **Business**: id, name, description, address, latitude, longitude (PostGIS Point), categoryId, ownerId, images (URLs), isActive.\n- **Category**: id, name, icon.\n- **Service**: id, businessId, name, duration, price, isActive.\n- **Booking**: id, userId, businessId, serviceId, startTime, endTime, status, paymentId, createdAt.\n- **Review**: id, bookingId, userId, rating, comment, createdAt.\n- **Favorite**: id, userId, businessId, unique constraint.\n- **Notification**: id, recipientId, type, payload, sentAt, readAt.\n\nIndexes: composite indexes on (`businessId`, `categoryId`), (`latitude`, `longitude`) using GIST, (`userId`, `status`) for booking lookups.\n\n---\n\n## Communication Protocols\n\n- **REST/JSON**: Standard CRUD endpoints under `/api/*`.\n- **GraphQL (optional)**: Could be added later via NestJS GraphQL module for flexible client queries.\n- **WebSocket (Socket.io or NestJS WS gateway)**: Real‑time updates for booking status, new messages, notification alerts.\n- **Async Messaging**: BullMQ publishes jobs; workers process and may emit events via WebSocket or email/SMS providers.\n\n---\n\n## DevOps & CI/CD\n\n- **Local Development**: `docker-compose.yml` spins up PostgreSQL, Redis, and the NestJS API. Expo dev client runs locally.\n- **Testing**: Jest unit tests for libs, end‑to‑end tests using Detox (or Expo’s built‑in test runner) for critical flows.\n- **CI (GitHub Actions)**: \n    1. Checkout & cache pnpm store.\n    2. Run `nx lint`, `nx test`, `nx build` for affected projects.\n    3. Build Docker images for each NestJS service (multi‑stage).\n    4. Push images to registry (GHCR).\n    5. Run `eas build` for iOS/Android (expo).\n    6. Deploy API stack to staging/production via Docker Swarm or Kubernetes (helm charts).\n- **Environment Variables**: Managed via `.env` files; secrets injected via GitHub Secrets.\n- **Monitoring**: Prometheus + Grafana (optional) for API metrics; logs shipped to ELK or Loki.\n\n---\n\n## Shared Kernel & Design System\n\n- **`libs/shared`** contains:\n    - TypeScript interfaces that mirror DTOs used across client and server (generated via `swagger-typescript-api` or manual sync).\n    - Theme tokens (colors, spacing, typography) consumed by React Native components.\n    - Utility functions (date formatting, currency, geohash helpers).\n    - Constants (role enum, booking status, payment provider).\n\n---\n\n## Security Considerations\n\n- **Authentication**: JWT (access token 15 min, refresh token 7 days) stored securely in Expo’s `SecureStore`.\n- **Authorization**: Role‑based guards; providers can only manage their own businesses.\n- **Data Protection**: Passwords hashed with bcrypt; TLS enforced for all external traffic.\n- **Input Validation**: DTO validation + Prisma schema constraints.\n- **Rate Limiting**: NestJS `ThrottlerGuard` per‑IP and per‑user.\n- **Secrets**: API keys (payment gateway, Supabase) stored in Docker secrets or CI secrets, never in repo.\n\n---\n\n## Scalability & Performance\n\n- **Read‑Heavy Scenarios** (business listings, map search): Redis caches query results for 30‑60 s; fallback to DB.\n- **Geospatial Queries**: PostGIS GIST index enables fast radius searches; results paginated with cursor.\n- **Horizontal Scaling**: NestJS services are stateless; can run multiple replicas behind a load balancer.\n- **Background Jobs**: BullMQ allows dynamic worker scaling based on queue depth.\n- **CDN**: Static assets (images, icons) served via Supabase storage or Cloudflare.\n\n---\n\n## Diagram (Conceptual)\n\n```
