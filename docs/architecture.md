  "content": "# Planity Clone System Architecture\n\n## Overview\nThis document outlines the system architecture for the Planity Clone mobile application. The solution follows a clean, modular approach using a monorepo managed by Nx and pnpm, separating concerns into distinct apps and libraries. The mobile client is built with Expo and React Native, while the backend leverages NestJS with Prisma ORM, PostgreSQL (extended with PostGIS for geospatial queries), Redis for caching and BullMQ‑based background jobs. CI/CD is handled via GitHub Actions, Docker Compose for local development, EAS Build for mobile distribution, and Supabase for authentication and file storage.\n\n## Tech Stack\n- **Mobile**: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **API**: NestJS, Prisma, PostgreSQL, PostGIS, Redis, BullMQ\n- **Dev Tooling**: Nx, pnpm, Docker Compose, GitHub Actions, EAS Build, Supabase, Jest\n- **Testing**: Jest (unit/integration), React Native Testing Library, E2E with Detox (optional)\n\n## High-Level Components\n1. **Mobile Client** (`apps/mobile`) – Expo app handling UI, navigation, state management, and offline optimizations.\n2. **API Gateway** (`apps/api`) – NestJS monolith exposing REST/GraphQL endpoints; acts as entry point for all client requests.\n3. **Auth Service** (`libs/auth`) – Handles user registration, login, password reset, JWT issuance; integrates with Supabase Auth.\n4. **Business Service** (`libs/business`) – CRUD for businesses, categories, location (PostGIS), hours, and services.\n5. **Booking Service** (`libs/booking`) – Manages appointment slots, availability computation, booking creation, cancellation, rescheduling.\n6. **Payment Service** (`libs/payment`) – Encapsulates payment gateway integration (e.g., Stripe), secure token storage, webhook handling.\n7. **Notification Service** (`libs/notification`) – Sends push/email/SMS via Supabase Functions or external providers; backed by BullMQ queues.\n8. **Admin Portal** (`apps/admin`) – Optional Next.js web app for platform admins to view analytics, manage users/businesses.\n9. **Provider Portal** (`apps/provider`) – Optional Next.js web app for business owners to manage listings, services, and bookings.\n10. **Shared UI & Design System** (`libs/ui`) – Reusable React Native components, theming, icons built with Reanimated for animations.\n11. **Shared Types & Utilities** (`libs/shared`) – TypeScript interfaces, constants, helpers, API clients, and React Query wrappers used across mobile and web apps.\n\n## Service Boundaries\nEach bounded context owns its data model and business logic, communicating via well‑defined DTOs and events:\n- **Auth** – Owns `users` table; emits `UserCreated`, `PasswordResetRequested`.\n- **Business** – Owns `businesses`, `categories`, `business_services`, `business_hours`; emits `BusinessUpdated`.\n- **Booking** – Owns `appointments`, `time_slots`; listens to `BusinessUpdated` to recompute availability.\n- **Payment** – Owns `payments`, `payment_intents`; listens to `BookingConfirmed` to charge.\n- **Notification** – Owns `notifications` table; listens to events from Booking, Payment, Auth to dispatch messages.\n- **Admin/Provider** – Mostly read‑only aggregates; may issue commands via API gateway.\n\nCommunication is synchronous via NestJS controllers/services for request‑response flows and asynchronous via BullMQ (Redis) for background jobs (e.g., sending reminders, recalculating slot availability after a booking).\n\n## Folder Structure (Nx Monorepo)\n```\nplanity-clone/\n├─ apps/\n│  ├─ mobile/               # Expo React Native app\n│  ├─ api/                  # NestJS backend (gateway)\n│  ├─ admin/                # Optional Next.js admin dashboard\n│  └─ provider/             # Optional Next.js provider portal\n├─ libs/\n│  ├─ auth/                 # Auth service (NestJS module)\n│  ├─ business/             # Business service\n│  ├─ booking/              # Booking service\n│  ├─ payment/              # Payment service\n│  ├─ notification/         # Notification service + BullMQ queues\n│  ├─ ui/                   # Shared React Native components, theming, Reanimated animations\n│  ├─ shared/               # Shared TS types, utils, API client, React Query helpers\n│  └─ data-access/          # Prisma schema, repositories, PostGIS extensions (shared across libs)\n├─ tools/\n│  ├─ eslint/               # Shared ESLint config\n│  ├─ jest/                 # Shared Jest config\n│  └─ typescript/           # Shared TS base config\n├─ docker-compose.yml       # Local dev: API, PostgreSQL, Redis, Supabase (studio)\n├─ nx.json\n├─ package.json             # pnpm workspace root\n└─ README.md\n```\n\n## Data Model (Prisma)\n- **User**: id, email, passwordHash, role, createdAt, updatedAt\n- **Business**: id, name, description, address, lat, lng (PostGIS Point), categoryId, ownerId, createdAt\n- **Category**: id, name, slug\n- **BusinessService**: id, businessId, name, duration, price\n- **BusinessHour**: id, businessId, dayOfWeek, openTime, closeTime, isClosed\n- **Appointment**: id, userId, businessId, serviceId, startTime, endTime, status, paymentId, createdAt\n- **Payment**: id, appointmentId, amount, currency, provider, providerId, status, createdAt\n- **Notification**: id, recipientId, type, payload, channel, status, sentAt, createdAt\n\nIndexes: spatial index on `businesses.(lat,lng)`, composite indexes for availability queries (`businessId`, `dayOfWeek`, `startTime`).\n\n## Caching & Queue\n- **Redis**: Stores short‑lived caches (business listings, slot availability) and BullMQ job queues.\n- **Queues**: `booking-availability`, `notification-send`, `payment-webhook`, `reminder-scheduler`.\n- Workers are NestJS services annotated with `@Processor` that consume jobs and update relevant entities.\n\n## DevOps & CI/CD\n- **Local**: `docker compose up` spins up API, PostgreSQL, Redis, Supabase Studio.\n- **CI**: GitHub Actions runs lint, unit tests (Jest), build (Nx), and pushes Docker images to a registry.\n- **CD**: \n  - API: Deploy Docker image to a managed Kubernetes service or VM; auto‑migrate Prisma schema on release.\n  - Mobile: EAS Build creates IPA/APK; distributed via internal TestFlight/Play Store track.\n  - Web (Admin/Provider): Vercel/Netlify auto‑deploy on main branch.\n- **Env**: Separate `.env` files for dev, staging, prod; secrets managed via GitHub Secrets and injected at runtime.\n\n## Security Considerations\n- Authentication: JWT (access token 15 min, refresh token 7 days) stored securely; refresh token rotation.\n- Passwords: bcrypt hash.\n- Data in transit: TLS enforced.\n- Data at rest: PostgreSQL encryption at cloud provider level; sensitive payment data never stored (only gateway tokens).\n- Rate limiting & input validation via NestJS guards and class‑validator.\n- Webhook signatures verified (Stripe, Supabase).\n- CORS restricted to trusted domains.\n\n## Testing Strategy\n- **Unit**: Jest tests for services, utilities, and React Native components (with React Native Testing Library).\n- **Integration**: API endpoint tests using Supertest; database rolled back per test.\n- **E2E** (optional): Detox for mobile flows; Cypress for web portals.\n- **Test Coverage**: Target >80% for critical paths (auth, booking, payment).\n\n## Deployment Summary\n1. Developer pushes code → GitHub Actions CI runs.\n2. On success, Docker image for API is built and pushed.\n3. CD pipeline (ArgoCD/Flux or simple SSH) pulls image, runs db migrations, restarts service.\n4. Mobile: EAS Build triggered on tag or branch; artifact uploaded to store.\n5. Web apps: Vercel/Netlify auto‑deploy on merge to main.\n\n## Future Extensions\n- Microservice split: extract Booking and Payment into separate NestJS apps if scale demands.\n- GraphQL subscription for real‑time slot updates.\n- Advanced analytics via ClickHouse or BigQuery.\n- Offline-first mobile state using MMKV + React Query persistence.\n- AI‑powered recommendation engine for business discovery.\n\n---\n*This architecture provides a clear separation of concerns, scalability paths, and a foundation for rapid feature delivery while maintaining code quality and testability.*\n",
