Overview\n\nThis document describes the complete system architecture for the Planity Clone, a mobile-first platform connecting customers with local service businesses. The architecture follows clean architecture principles with clear separation of concerns, modular monolith backend, and a well-structured mobile app.\n\n## 2. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile Framework | Expo (React Native) | Cross-platform mobile app |\n| Language | TypeScript | Type safety across stack |\n| Navigation | Expo Router | File-based routing, deep linking |\n| State/Server | TanStack React Query | Server state management, caching |\n| Animations | React Native Reanimated | High-performance UI animations |\n| Backend Framework | NestJS | Modular, scalable Node.js backend |\n| ORM | Prisma | Type-safe database access |\n| Database | PostgreSQL + PostGIS | Relational data + geospatial queries |\n| Cache | Redis | Session store, rate limiting, job queue |\n| Monorepo | Nx + pnpm | Build orchestration, dependency management |\n| Containerization | Docker Compose | Local development environment |\n| CI/CD | GitHub Actions | Automated testing and deployment |\n| Auth/Backend-as-a-Service | Supabase | Authentication, storage, realtime |\n| Build/Deploy | EAS Build | Expo app builds and submissions |\n| Testing | Jest | Unit and integration tests |\n\n## 3. Architecture Principles\n\n- **Separation of Concerns**: Each service has a single responsibility.\n- **Modular Monolith**: Backend is a single deployable unit but organized into modules.\n- **API-First Design**: All features are accessible via RESTful APIs.\n- **Offline-First**: Mobile app works with degraded functionality when offline.\n- **Security by Design**: JWT authentication, input validation, rate limiting.\n- **Observability**: Structured logging, metrics, and health checks.\n\n## 4. Service Boundaries\n\n### 5.1 Mobile App (Expo/React Native)\n\n**Responsibilities**:\n- User interface for all three personas (customer, business owner, admin)\n- Local state management with Zustand\n- Server state management with TanStack React Query\n- Offline caching of business data and user appointments\n- Push notification handling\n- Deep linking support via Expo Router\n\n**Key Modules**:\n- `services/api.ts`: Axios instance with JWT interceptor\n- `services/supabase.ts`: Supabase client for auth and realtime\n- `stores/authStore.ts`: Authentication state\n- `stores/locationStore.ts`: User location state\n- `stores/cartStore.ts`: Booking cart state (supports guest conversion)\n- `hooks/useBookings.ts`: TanStack Query hooks for bookings\n- `hooks/useBusiness.ts`: TanStack Query hooks for business data\n\n### 5.2 Backend API (NestJS)\n\n**Responsibilities**:\n- RESTful API for all business logic\n- Authentication and authorization (JWT, role-based)\n- Geospatial search with PostGIS\n- Real-time availability updates via Supabase Realtime\n- Payment processing integration\n- Notification dispatch (push, email, SMS)\n\n**Module Breakdown**:\n\n| Module | Endpoints | Key Logic |\n|--------|-----------|-----------|\n| Auth | POST /auth/register, /auth/login, /auth/refresh | JWT generation, OAuth callbacks, password reset |\n| comment, photos TEXT[], created_at)\n- `favorites` (id, user_id, business_id, created_at)\n- `payment_transactions` (id, booking_id, amount, currency, status, stripe_payment_intent_id)\n\n**PostGIS Usage**:\n- Spatial index on `businesses.location` for fast radius queries\n- `ST_DWithin` for distance-based filtering\n- `ST_AsGeoJSON` for map rendering\n\n### 5.4 Redis\n\n**Use Cases**:\n- Session store for refresh tokens\n- Rate limiting (per user, per IP)\n- Job queue for async tasks (email, notifications)\n- Cache for business search results (TTL 5 minutes)\n- Slot locking (pessimistic lock with 5-minute expiry)\n\n## 6. Data Flow\n\n### 6.1 Booking Flow\n\n1. User selects service(s) and staff on mobile\n2. Mobile app calls `GET /staff/:id/availability?date=2024-01-15`\n3. Backend queries PostgreSQL for available slots (excluding existing bookings and staff schedules)\n4. Backend checks Redis for any temporary locks on slots\n5. Returns available time slots\n6. User selects time slot\n7. Mobile app calls `POST /bookings` with service, staff, time\n8. Backend creates a pending booking in PostgreSQL\n9. Backend sets a Redis key `lock:slot:{staff_id}:{start_time}` with 5-minute TTL\n10. Payment is processed (or held)\n11. Booking status changes to confirmed\n12. Redis lock is removed\n13. Notifications dispatched via queue\n\n### 6.2 Search Flow\n\n1. User enters query or moves map\n2. Mobile app calls `GET /search?q=haircut&lat=48.8566&lng=2.3522&radius=10`\n3. Backend checks Redis cache for identical query\n4. If cache miss, executes PostgreSQL full-text search with PostGIS filter\n5. Results are paginated (cursor-based)\n6. Response cached in Redis for 5 minutes\n7. Mobile app displays results with skeleton loading\n\n## 7. Authentication & Authorization\n\n- **JWT Strategy**: Access token (15 min) + Refresh token (7 days)\n- **Refresh Token Rotation**: Each refresh returns a new pair, invalidates old\n- **Supabase Auth**: Handles OAuth (Google, Apple) and email/password\n- **Role-Based Access Control (RBAC)**:\n  - `customer`: Can book, manage own appointments\n  - `business_owner`: Can manage own business, staff, appointments\n  - `admin`: Full access\n- **Guest Users**: Temporary JWT with limited scope, converted on registration\n\n## 8. API Design\n\n- **Base URL**: `/api/v1`\n- **Authentication**: Bearer token in `Authorization` header\n- **Error Format**: `{ \"statusCode\": 400, \"message\": \"Validation failed\", \"errors\": [...] }`\n- **Pagination**: Cursor-based with `cursor` and `limit` query params\n- **Idempotency**: `Idempotency-Key` header for booking creation\n\n## 9. Offline Strategy\n\n- **TanStack React Query**: Persists cache to AsyncStorage\n- **Business Data**: Cached for 24 hours; stale-while-revalidate\n- **User Appointments**: Cached with background sync on app open\n- **Map Tiles**: Cached via react-native-maps (default behavior)\n- **Booking Flow**: Requires network; validation at step 3 (date/time)\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for services, hooks, utilities\n- **Integration Tests**: NestJS e2e tests with testcontainers (PostgreSQL, Redis)\n- **Component Tests**: React Native Testing Library for UI components\n- **API Tests**: Supertest for endpoint validation\n- **CI Pipeline**: GitHub Actions runs tests on every PR\n\n## 11. Deployment\n\n- **Mobile**: EAS Build for iOS/Android; OTA updates via EAS Update\n- **Backend**: Docker container deployed to cloud (AWS ECS / GCP Cloud Run)\n- **Database**: Managed PostgreSQL (Supabase / AWS RDS)\n- **Redis**: Managed Redis (Upstash / AWS ElastiCache)\n- **CI/CD**: GitHub Actions → build, test, deploy\n\n## 12. Monitoring & Observability\n\n- **Logging**: Structured JSON logs (Pino)\n- **Metrics**: Prometheus metrics for request rate, error rate, latency\n- **Health Checks**: `/health` endpoint for liveness and readiness\n- **Error Tracking**: Sentry for mobile and backend\n\n## 13. Security Considerations\n\n- **Rate Limiting**: 100 requests/min per user, 1000 requests/min per IP\n- **Input Validation**: Zod schemas on all endpoints\n- **SQL Injection**: Prevented by Prisma parameterized queries\n- **CORS**: Restricted to mobile app origins\n- **HTTPS**: Enforced in production\n- **Data Privacy**: GDPR compliance with user data export and deletion\n\n## 14. Scalability\n\n- **Horizontal Scaling**: Backend is stateless; scale via container orchestration\n- **Database**: Read replicas for search queries; connection pooling with PgBouncer\n- **Caching**: Redis for hot data; CDN for static assets\n- **Async Processing**: Job queue for notifications, email, analytics\n\n## 15. Conclusion\n\nThis architecture provides a solid foundation for the Planity Clone, emphasizing clean separation, testability, and scalability. The modular monolith allows for easy extraction into microservices if needed in the future. The mobile app is designed for offline resilience and smooth user experience.",
