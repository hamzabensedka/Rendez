High-Level Overview\n\nThe Planity Clone is a full‑stack application split into three main concerns:\n\n- Mobile Client – Expo + React Native app (Expo Router, TanStack React Query, Reanimated, etc.)\n- API Server – NestJS monolith (or micro‑services) powered by Prisma, PostgreSQL/PostGIS, Redis, BullMQ.\n- Infrastructure – Docker Compose for local dev, GitHub Actions CI/CD, EAS Build for mobile, Supabase for auth & storage, Nx monorepo with pnpm.\n\n## 2. Monorepo Structure (Nx + pnpm)\n\nworkspace/\n├─ apps/\n│  ├─ mobile/          # Expo RN app\n│  ├─ api/             # NestJS backend\n│  └─ web/             # Optional admin dashboard (Next.js or Expo Web)\n├─ libs/\n│  ├─ shared-types/    # TypeScript interfaces/DTOs used by mobile & api\n│  ├─ ui/              # Reusable native components (React Native)\n│  ├─ design-system/   # Tokens, themes, styled components\n│  ├─ data-access/     # Prisma client wrapper, repository helpers\n│  ├─ utils/           # Logging, validation, date helpers\n│  ├─ auth/            # Auth utilities (JWT, Supabase helpers)\n│  └─ geo/             # PostGIS helpers, distance calculations\n├─ tools/\n│  ├─ eslint/\n│  ├─ jest/\n│  └─ typescript/\n├─ docker-compose.yml\n└─ nx.json\n\nAll projects use pnpm workspaces; Nx handles task caching and affected commands.\n\n## 3. Service Boundaries (Backend)\n\n| Service | Responsibility | Tech |\n|---------|----------------|------|\n| Auth Service | User registration, login, social auth, JWT issuance, Supabase integration | NestJS module (`apps/api/src/auth`) |\n| Business Service | CRUD for businesses, categories, location (PostGIS), search indexing | NestJS module (`apps/api/src/business`) |\n| Booking Service | Slot generation, appointment creation, rescheduling, cancellation, payment integration | NestJS module (`apps/api/src/booking`) |\n| Provider Portal | Business owner dashboard for managing listings, availability, bookings | NestJS module (`apps/api/src/provider`) (could be separate micro‑service) |\n| Admin Service | User & business moderation, analytics, reports | NestJS module (`apps/api/src/admin`) |\n| Notification Service | Sending push/email/SMS via BullMQ workers, handling preferences | NestJS module (`apps/api/src/notification`) |\n| Payment Service | Wrapper around payment gateway (Stripe/PayPal) | NestJS module (`apps/api/src/payment`) |\n| Geo Service (optional) | Proximity queries, heatmaps, caching of geocode results | NestJS module (`apps/api/src/geo`) |\n\nAll services share a Prisma schema (`apps/api/prisma/schema.prisma`) with PostGIS extension via `prisma-postgis` provider or raw SQL migrations.\n\n## 4. Database & Caching\n\n- PostgreSQL (with PostGIS) – primary data store for users, businesses, appointments, reviews, etc.\n- Redis – caching of frequent queries (search results, business details), pub/sub for real‑time updates, BullMQ job queue backend.\n- BullMQ – background jobs for sending notifications, slot recomputation, email verification, etc.\n- Supabase – optional auth & storage (avatars, business images) – used via Supabase client in both mobile and API.\n\n## 5. Mobile Client Architecture\n\napps/mobile/\n├─ src/\n│  ├─ app/                 # Expo Router file‑based routes\n│  │  ├─ (tabs)/           # Bottom tab navigator (Home, Search, Map, Profile, etc.)\n│  │  ├─ auth/             # Login, Register screens\n│  │  ├─ businesses/       # Business list, search, map\n│  │  ├─ business-detail/  # Business detail view\n│  │  ├─ booking/          # Booking flow screens\n│  │  ├─ appointments/     # Upcoming/past appointments\n│  │  ├─ favorites/        # Favorites tab\n│  │  ├─ profile/          # Edit profile\n│  │  └─ provider/         # Provider portal (if embedded) or deep link to web\n│  ├─ components/          # Reusable UI (buttons, cards, modals)\n│  ├─ hooks/               # Custom React Query hooks, Reanimated gestures\n│  ├─ lib/                 # API client (axios/query-client), Supabase, utils\n│  ├─ theme/               # Design tokens, dark/light mode\n│  └─ assets/              # Images, fonts, icons\n\n- State Management: TanStack React Query for server state; React Context or Zustand for UI state if needed.\n- Navigation: Expo Router provides type‑safe, file‑based navigation with deep linking.\n- Animations: React Native Reanimated 2 for gesture‑based UI (e.g., swipe to cancel appointment, map interactions).\n- Type Safety: Shared Types lib (`libs/shared-types`) exported to both mobile and API via Nx build.\n- Testing: Jest + React Native Testing Library for unit/component tests; E2E with Detox (optional) configured in GitHub Actions.\n\n## 6. API Server Architecture\n\napps/api/\n├─ src/\n│  ├─ main.ts\n│  ├─ app.module.ts\n│  ├─ common/               # Guards, interceptors, pipes, exceptions\n│  ├─ auth/                 # JWT strategy, Supabase wrapper\n│  ├─ business/             # Controllers, services, DTOs\n│  ├─ booking/\n│  ├─ provider/\n│  ├─ admin/\n│  ├─ notification/\n│  ├─ payment/\n│  ├─ geo/\n│  ├─ prisma/               # Prisma client extension\n│  └─ utils/                # Logging, config, helpers\n\n- Validation: class-validator + class-transformer.\n- Authentication: JWT access token; refresh token stored in HttpOnly cookie (or Supabase session).\n- Authorization: Role‑based guards (USER, PROVIDER, ADMIN).\n- CORS: Configured for Expo dev client and web.\n- Health Checks: `/health` endpoint.\n- Documentation: Swagger/OpenAPI generated via NestJS Swagger module.\n\n## 7. DevOps & CI/CD\n\n- Docker Compose (local):\n  ```yaml\n  services:\n    postgres:\n      image: postgres:15\n      environment: POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB\n      volumes: pg_data:/var/lib/postgresql/data\n      ports: [\"5432:5432\"]\n    redis:\n      image: redis:7\n      ports: [\"6379:6379\"]\n    api:\n      build: ./apps/api\n      env_file: .env\n      depends_on: [postgres, redis]\n      ports: [\"3000:3000\"]\n  ```\n- GitHub Actions workflow:\n  - `lint`: run `nx lint` across monorepo.\n  - `test`: `nx test` (Jest) for mobile and api.\n  - `build`: `nx build api` and `expo prebuild` + `eas build` for mobile.\n  - `docker`: build and push API image to registry.\n  - `deploy`: trigger on `main` to deploy API to a VPS/K8s and upload mobile build to Expo/EAS.\n- EAS Build: Configured for iOS and Android with preview and production profiles.\n- Supabase: Used for auth (email/social) and storage; secrets stored in GitHub repo secrets.\n- Monitoring: Basic health checks; can be extended with Prometheus/Grafana (out of scope).\n\n## 8. Shared Libraries\n\n- libs/shared-types: Contains DTOs, enums, interfaces (e.g., User, Business, Appointment, Slot). Built as a local Nx lib and consumed via `tsconfig` path mapping.\n- libs/ui: Stateless native components (Button, Input, Card, MapMarker) styled with the design system.\n- libs/design-system: Tokens (colors, spacing, typography) using `react-native-reanimated`‑compatible styling (e.g., `react-native-mmkv` for theme persistence).\n- libs/data-access: Thin wrapper around Prisma client (`prismaClient`) with transaction helpers; exported to API only.\n- libs/geo: Haversine formula, PostGIS query builders, distance caching.\n\n## 9. Data Flow Example (Booking)\n\n1. User taps \"Book\" on business detail → navigation to `BookingScreen`.\n2. Screen uses React Query hook `useAvailableSlots(businessId, date)` → calls GET `/bookings/slots?businessId=…&date=…` (API).\n3. API `BookingService` queries PostgreSQL for business opening hours (PostGIS not needed) and existing appointments, computes free slots, returns array.\n4. User selects slot, confirms → POST `/bookings` with JWT.\n5. API validates, creates appointment record, enqueues a BullMQ job `send-confirmation` (email/push) and optionally a payment intent via Payment Service.\n6. On success, React Query invalidates `appointments` list; UI shows confirmation screen.\n7. Background job sends notification via Supabase/FCM; updates Redis cache for upcoming appointments.\n\n## 10.
