Service Boundaries & API Design\nAll backend logic is exposed via a single NestJS application, internally modularised. Each module encapsulates its own controller, service, and Prisma repository.\n\n### 6.1 Modules\n- **Auth Module**: Registration, login, token refresh, password reset, social login (Google/Apple). Uses Supabase Auth under the hood; synchronises user profiles to local DB.\n- **Business Module**: CRUD for businesses, search with filters, geospatial queries (PostGIS), business hours, staff management.\n- **Booking Module**: Appointment creation, rescheduling, cancellation, slot computation, payment integration (Stripe).\n- **User Module**: Profile management, favourites, notification preferences, payment methods.\n- **Review Module**: Create and list reviews, aggregate ratings.\n- **Category Module**: Hierarchical category tree.\n- **Slot Module**: Internal service for computing available time slots; used by Booking module.\n\n### 6.2 API Endpoints (Representative)\n- `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`\n- `GET /businesses?lat=&lng=&radius=&category=&...`\n- `GET /businesses/:id`\n- `GET /businesses/:id/services`\n- `GET /businesses/:id/slots?serviceId=&staffId=&date=`\n- `POST /bookings`\n- `PATCH /bookings/:id/cancel`\n- `GET /users/me/favorites`\n- `POST /reviews`\n\nAll endpoints are versioned under `/api/v1/`. Request/response validation uses class-validator and DTOs. Pagination follows cursor-based pattern for infinite scroll.\n\n## 7. Database Design\nPostgreSQL with PostGIS for location-based queries. Prisma manages schema and migrations.\n\n### 7.1 Core Tables\n- **User**: id, email, passwordHash, name, phone, avatarUrl, role (CUSTOMER, PROVIDER, ADMIN), createdAt\n- **Business**: id, name, description, address, location (PostGIS geography point), phone, email, ownerId (FK User), categoryId, ratingAvg, photos[]\n- **BusinessHour**: id, businessId, dayOfWeek, openTime, closeTime\n- **Service**: id, businessId, name, durationMinutes, price, currency, description\n- **Staff**: id, businessId, userId (optional), name, photo\n- **StaffService**: staffId, serviceId (many-to-many)\n- **Booking**: id, customerId, businessId, serviceId, staffId, startTime, endTime, status (PENDING, CONFIRMED, CANCELLED, COMPLETED), notes, cancellationReason\n- **Review**: id, bookingId, customerId, businessId, rating, comment, createdAt\n- **Favorite**: id, userId, businessId\n- **Category**: id, name, slug, parentId (self-referential), icon\n- **BlockedTime**: id, staffId, businessId, startTime, endTime, reason (BREAK, HOLIDAY)\n\n### 7.2 Spatial Indexing\nA GiST index on `Business.location` enables efficient `ST_DWithin` queries for radius search. The API uses raw SQL via Prisma for geospatial filtering.\n\n### 7.3 Slot Computation Data Model\nSlots are not stored; they are computed on-the-fly from:\n- Business hours (BusinessHour)\n- Staff working hours (if staff-specific, otherwise business hours)\n- Service duration + buffer time\n- Existing bookings (Booking) for that staff/date\n- BlockedTime entries\n\n## 8. Slot Computation Service\nThis is the most critical backend component. It must be fast (<200ms) and prevent double-booking.\n\n### 8.1 Algorithm\n1. Receive request: businessId, serviceId, staffId (optional), date, timezone.\n2. Fetch business hours for the day, service duration + buffer.\n3. Generate all possible start times (e.g., every 15 minutes) within working hours.\n4. For each candidate slot, check if the interval `[start, start + duration + buffer]` overlaps any existing booking or blocked time for the staff (or any staff if not specified).\n5. Return list of available start times.\n\n### 8.2 Concurrency Control\nWhen a booking is created, the service uses a database transaction with `SELECT ... FOR UPDATE` on the staffвҖҷs bookings for that date to serialise writes. This ensures no two bookings can claim the same slot. Alternatively, a Redis lock (`SETNX`) on `staff:{id}:date:{date}` can be used for distributed locking.\n\n### 8.3 Performance Optimisations\n- Cache business hours and service durations in Redis (TTL 1 hour).\n- Pre-compute and cache slot availability for popular businesses/dates with short TTL (30 seconds) to reduce DB load.\n- Use PostgreSQL range types and exclusion constraints (`EXCLUDE USING gist (staff_id WITH =, tsrange(start_time, end_time) WITH &&)`) as an additional safety net.\n\n## 9. Caching Strategy\nRedis is used for:\n- **Session & token blacklist**: JWT refresh tokens stored with TTL.\n- **Rate limiting**: Sliding window counters per IP/user.\n- **Frequently accessed data**: Business details, categories, top-rated businesses (TTL 5-10 minutes).\n- **Slot computation cache**: As described above.\n- **Real-time features**: Pub/sub for booking confirmations (optional).\n\nCache invalidation is handled via explicit deletion on data mutation (e.g., when a business updates its hours, clear related slot caches).\n\n## 10. Authentication & Authorization\n- **Authentication**: Supabase Auth handles email/password, Google, Apple sign-in. The mobile app uses Supabase client SDK to obtain JWT. The NestJS API validates the JWT on each request using a Passport strategy and SupabaseвҖҷs JWKS endpoint.\n- **User Synchronisation**: On first login, a user record is created/updated in the local DB via a webhook or API call.\n- **Authorization**: Role-based guards (`@Roles('CUSTOMER')`) protect endpoints. Business owners can manage their own business; admins have full access.\n- **Guest Access**: Public endpoints (business search, details) are unauthenticated. Favorites stored locally in Zustand; merged on login via API.\n\n## 11. Mobile App Architecture\n### 11.1 Navigation & Routing\nExpo Router provides file-based routing with typed routes. Layout routes define stacks, tabs, and modals.\n- `app/` (root layout with auth state)\n  - `(auth)/` вҖ“ protected screens (bookings, profile, favorites)\n  - `(public)/` вҖ“ home, search, business detail (accessible to guests)\n  - `_layout.tsx` вҖ“ root layout wrapping providers\n\n### 11.2 State Management\n- **Server State**: TanStack React Query manages all API data fetching, caching, and mutations. Custom hooks like `useBusinesses`, `useSlots` encapsulate query keys and functions.\n- **Client State**: Zustand stores for UI state (selected filters, map/list toggle), guest favorites, and auth token.\n- **Optimistic Updates**: Booking creation and cancellation use `onMutate` to update UI instantly, with rollback on error.\n\n### 11.3 API Layer\nA dedicated `api` service module uses `axios` with interceptors for token attachment and refresh. React QueryвҖҷs `queryFn` calls these functions.\n\n### 11.4 Animations & UX\nReact Native Reanimated powers smooth transitions: map/list toggle, card expansions, booking step transitions, and skeleton loaders. Shared element transitions (e.g., business image to detail) are implemented where possible.\n\n### 11.5 Offline & Guest Support\n- Guest browsing works offline by caching search results and business details with React QueryвҖҷs `persist` plugin (AsyncStorage).\n- Favorites stored locally; on login, a merge API combines local and server favorites.\n\n## 12. Deployment & CI/CD\n### 12.1 Local Development\nDocker Compose spins up PostgreSQL (with PostGIS), Redis, and the NestJS API. The mobile app runs via Expo Go or development build.\n\n### 12.2 CI/CD Pipeline (GitHub Actions)\n- **CI**: On PR, run linting, type-check, unit tests (Jest) for both apps and libs.\n- **API Deployment**: Build Docker image, push to registry, deploy to cloud (e.g., AWS ECS, Fly.io).\n- **Mobile Deployment**: EAS Build triggered on main branch pushes; creates development, preview, and production builds. Over-the-air updates via EAS Update.\n\n### 12.3 Infrastructure\n- **Database**: Managed PostgreSQL with PostGIS (Supabase or AWS RDS).\n- **Redis**: Managed instance (Upstash, ElastiCache).\n- **File Storage**: Supabase Storage for user avatars, business photos.\n- **Push Notifications**: Expo Push Notification service.\n\n## 13. Testing Strategy\n- **Unit Tests**: Jest for utility functions, hooks, NestJS services, and Prisma repositories (mocked).\n- **Integration Tests**: Supertest for API endpoints with a test database; React Native Testing Library for components.\n- **E2E Tests**: Detox or Maestro for critical flows (booking, auth).\n- **Slot Computation**: Dedicated test suite with various schedule configurations and concurrency scenarios.\n\n## 14. Security Considerations\n- All communications over HTTPS.\n- JWTs short-lived (15 min), refresh tokens rotated and stored securely.\n- Input validation and sanitisation (class-validator).\n- Rate limiting on auth and booking endpoints.\n- SQL injection prevented by Prisma parameterised queries.\n- Sensitive data (keys, secrets) stored in environment variables, never in code.\n\n## 15. Performance & Scalability\n- Database indexing on foreign keys, location, and date fields.\n- Connection pooling (PgBouncer) for PostgreSQL.\n- Horizontal scaling of API behind a load balancer.\n- Redis caching reduces primary DB load.\n- Mobile app uses FlatList with pagination, image lazy loading, and Reanimated for 60fps interactions.\n\n## 16. Conclusion\nThis architecture delivers a robust, maintainable, and scalable Planity Clone. The monorepo structure with Nx ensures code sharing and consistency. NestJS provides a solid backend foundation, while Expo and React Native enable a high-quality mobile experience.
