Service Boundaries & Communication\n\n- **API Gateway (`apps/api`)** exposes a unified REST/GraphQL surface. Internally it delegates to NestJS modules representing each bounded context (auth, business, booking, payment, notification). Communication between modules is via direct NestJS service injection (in‑process) – no extra network hop.\n- **Background Workers** run as separate NestJS processes (`libs/workers/bullmq`) that connect to the same Redis instance used for BullMQ queues. They consume jobs emitted by API services (e.g., `payment.process`, `notification.send`, `booking.recomputeSlots`).\n- **Mobile Client** communicates exclusively with the API Gateway using TanStack React Query for data fetching, caching, and optimistic updates. Reanimated is used for gesture‑based UI (e.g., map drag, swipe actions).\n- **Admin & Provider Portals** are separate web apps that also call the same API Gateway. They share the same auth JWT verification logic.\n- **Data Layer**: All services use a single Prisma client wrapper (`libs/data-access/prisma`) pointing to the PostgreSQL+PostGIS database. This ensures schema consistency and type‑safe queries across services.\n- **Caching**: Redis stores session tokens, short‑lived OTPs, and cached geo‑search results. The API uses Redis as a BullMQ broker and as a generic cache layer.\n- **File Storage**: Supabase Storage (or S3‑compatible) holds user avatars, business photos, and receipts. Upload URLs are generated by the API.\n\n## 5. Authentication & Authorization\n- **Auth Strategy**: JWT (access token 15 min, refresh token 7 days). Refresh token stored httpOnly cookie on web, AsyncStorage on mobile.\n- **Roles**: `USER`, `PROVIDER`, `ADMIN`. Middleware guards routes based on role and entity ownership (e.g., a provider can only modify their own businesses).\n- **Social Login**: Optional OAuth2 via providers (Google, Apple) handled by NestJS `passport` strategies; tokens exchanged for internal JWT.\n- **Password Security**: bcrypt with salt factor 12.\n\n## 7. Real‑Time Features\n- **Notifications**: When a booking is created/updated/cancelled, the API publishes a job to the `notifications` queue. Workers send push notifications via Expo Push Service (mobile) and email/SMS via third‑party providers (Twilio, SendGrid).\n- **Slot Recomputation**: On booking creation/cancellation or business schedule change, a job is pushed to `slot-recompute` queue. Worker recalculates available slots for affected services and updates the `Slot` table; React Query invalidates related queries on the client.\n\n## 8. Payment Flow\n1. User confirms booking → API creates a `PaymentIntent` with Stripe (or similar).\n2. Client confirms payment via Stripe SDK (or redirects to web view).\n3. Webhook endpoint verifies event, updates `Booking.paymentId` and sets status to `CONFIRMED`.\n4. On success, a `notification.send` job is queued for user and provider.\n5. Refunds follow similar webhook handling.\n\n## 9. Search & Discovery\n- **Text Search**: PostgreSQL full‑text search on `Business.name` and `Business.description`.\n- **Geo Search**: PostGIS radius query as described; results sorted by distance.\n- **Filters**: Category ID, price range, rating (aggregated from `Review`).\n- **Pagination**: Cursor‑based using `createdAt` or distance.\n\n## 10.
