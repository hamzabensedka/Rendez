Overview\n\nThis document describes the system architecture for the Planity Clone, a two-sided marketplace connecting customers with beauty/wellness businesses. The architecture emphasizes clean separation of concerns, maintainability, and scalability using a monorepo structure with Nx, a mobile app built with Expo/React Native, and a backend powered by NestJS, Prisma, PostgreSQL, PostGIS, and Redis.\n\n## 2. Technology Stack\n\n- **Mobile App**: Expo (managed workflow), React Native, TypeScript, Expo Router (file-based routing), TanStack React Query (server state), React Native Reanimated (animations)\n- **Backend**: NestJS (Node.js framework), TypeScript, Prisma (ORM), PostgreSQL + PostGIS (geospatial), Redis (caching, session, queue)\n- **Infrastructure**: Docker Compose (local dev), GitHub Actions (CI/CD), Supabase (auth, real-time), EAS Build (app builds), Jest (testing)\n- **Monorepo**: Nx, pnpm\n\n## 3. Folder Structure\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   │   ├── (auth)/            # Auth group (login, register)\n│   │   │   ├── (tabs)/            # Main tabs (home, search, bookings, profile)\n│   │   │   ├── business/[id]/     # Business detail page\n│   │   │   ├── booking/           # Booking flow pages\n│   │   │   └── _layout.tsx        # Root layout\n│   │   ├── src/\n│   │   │   ├── components/        # Shared UI components\n│   │   │   ├── hooks/             # Custom hooks\n│   │   │   ├── services/          # API client (TanStack Query)\n│   │   │   ├── stores/            # Zustand stores (client state)\n│   │   │   ├── utils/             # Helpers\n│   │   │   └── types/             # Shared types\n│   │   ├── app.json\n│   │   └── package.json\n│   └── web/                       # (Future) Business owner web portal\n├── libs/\n│   ├── shared/\n│   │   ├── types/                 # Shared DTOs and interfaces\n│   │   ├── utils/                 # Common utilities\n│   │   └── constants/             # Design tokens, API endpoints\n│   └── api/\n│       ├── core/                  # NestJS core module (guards, interceptors)\n│       ├── auth/                  # Auth module\n│       ├── users/                 # User module\n│       ├── businesses/            # Business module\n│       ├── services/              # Service module\n│       ├── bookings/              # Booking module\n│       ├── availability/          # Slot computation module\n│       ├── reviews/               # Review module\n│       ├── notifications/         # Notification module\n│       └── search/                # Search module\n├── tools/                         # Build scripts, generators\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n├── tsconfig.base.json\n└── .github/\n    └── workflows/\n        ├── ci.yml\n        └── deploy.yml\n```\n\n## 4. Service Boundaries\n\n### 4.1 Mobile App (Consumer)\n- **Responsibility**: Customer-facing mobile application for browsing, searching, booking, and managing appointments.\n- **Key Libraries**: Expo Router (navigation), TanStack React Query (server state caching, optimistic updates), React Native Reanimated (smooth transitions), Zustand (local state).\n- **Data Flow**: All server data fetched via React Query hooks. Mutations use optimistic updates with rollback on error. Guest data stored in AsyncStorage and transferred on auth.\n\n### 4.2 Backend API (NestJS)\n- **Responsibility**: RESTful API for all business logic, authentication, and data persistence.\n- **Modules**:\n  - **Auth Module**: JWT + refresh token rotation, OAuth (Google, Apple), phone OTP, rate limiting.\n  - **Users Module**: Profile management, preferences, GDPR export/delete.\n  - **Businesses Module**: CRUD for businesses, categories, staff, hours, images.\n  - **Services Module**: Service definitions, categories, attributes.\n  - **Bookings Module**: Booking lifecycle (create, reschedule, cancel, waitlist), slot holds, conflict detection.\n  - **Availability Module**: Core slot computation algorithm using business hours, staff schedules, buffer times, and existing bookings. Cached in Redis (TTL 5 minutes).\n  - **Reviews Module**: Review creation, aggregation, moderation.\n  - **Notifications Module**: Push notifications (Expo Push), SMS (Twilio), email (SendGrid).\n  - **Search Module**: Full-text search with PostgreSQL tsvector, geospatial queries with PostGIS, autocomplete.\n\n### 4.3 Database (PostgreSQL + PostGIS)\n- **Schema**: Managed via Prisma migrations. Key tables: users, businesses, business_hours, staff, services, service_categories, bookings, reviews, favorites, notifications.\n- **Geospatial**: PostGIS extension for location-based queries (distance, bounding box, clustering).\n- **Full-Text Search**: tsvector columns on business name, service name, description.\n\n### 4.4 Cache (Redis)\n- **Uses**: Slot availability cache (TTL 5 min), session store for refresh tokens, rate limiting counters, job queue (Bull) for background tasks (email, SMS, push).\n\n### 4.5 External Services\n- **Supabase**: Auth (email/password, OAuth), real-time subscriptions for booking updates.\n- **Google Maps / Mapbox**: Map rendering, geocoding.\n- **Twilio**: SMS OTP.\n- **SendGrid / Resend**: Transactional emails.\n- **Expo Push**: Push notifications.\n\n## 5. Key Design Decisions\n\n### 5.1 Slot Computation\n- Algorithm runs server-side in the Availability module. Inputs: business hours, staff schedules, service duration, buffer, existing bookings. Output: list of available time slots.\n- Results cached in Redis with business+staff+service+date as key. Cache invalidated on new booking or schedule change.\n- Edge cases: multi-staff services, overlapping appointments, DST transitions handled via UTC storage and timezone conversion.\n\n### 5.2 Booking Flow & Slot Hold\n- When user selects a slot, a 10-minute hold is created in Redis (key: `hold:{businessId}:{slot}`) with TTL. Prevents double-booking during checkout.\n- On payment success, hold is removed and booking persisted. On timeout or abandonment, hold expires automatically.\n\n### 5.3 Authentication\n- JWT access token (15 min) + refresh token (7 days) stored in secure storage (expo-secure-store). Refresh token rotation: each use returns a new refresh token; old one revoked.\n- Rate limiting: 5 login attempts per 15 min per IP (Redis-based).\n\n### 5.4 Search & Discovery\n- Full-text search via PostgreSQL tsvector with GIN index. Geospatial queries use PostGIS `ST_DWithin` for radius search.\n- Autocomplete: prefix search on business name and service name, cached in Redis.\n- Filter stack persisted in URL params (Expo Router) for shareability.\n\n### 5.5 Real-time Updates\n- Supabase Realtime for live booking status changes (e.g., business confirms, no-show).\n- Push notifications for reminders (24h, 2h, 15min) scheduled via Bull queue.\n\n## 6. Data Flow Example: Booking a Service\n\n1. User searches for \"haircut\" with location → `GET /api/search?q=haircut&lat=...&lng=...` → PostgreSQL full-text + PostGIS → returns businesses with next available slot.\n2. User taps business → `GET /api/businesses/:id` → returns detail + services.\n3. User selects service → `GET /api/availability?businessId=...&serviceId=...&date=...` → Availability module computes slots → cached in Redis → returns slots.\n4. User picks slot → `POST /api/bookings/hold` → creates hold in Redis (10 min TTL).\n5. User confirms → `POST /api/bookings` → validates hold, creates booking in DB, removes hold, schedules reminders, sends confirmation.\n6. If payment required → `POST /api/payments/intent` → Stripe integration.\n\n## 7. Performance Targets\n\n- Search results: <500ms for 100k businesses.\n- Slot computation: <100ms (cached).\n- Business detail page load: <2s.\n- Booking confirmation: <5s (including push/email).\n\n## 8. Testing Strategy\n\n- **Unit Tests**: Jest for all modules (services, controllers, utilities).\n- **Integration Tests**: Supertest for API endpoints with test database (Docker Compose).\n- **E2E Tests**: Detox for mobile app critical flows (search, booking, auth).\n- **CI**: GitHub Actions runs lint, type-check, unit tests, integration tests on every PR.\n\n## 9. Deployment\n\n- **Mobile**: EAS Build for iOS/Android builds. Over-the-air updates via EAS Update for JS changes.\n- **Backend**: Docker container deployed to cloud (AWS ECS / GCP Cloud Run). PostgreSQL managed (RDS / Cloud SQL). Redis managed (ElastiCache / Memorystore).\n- **CI/CD**: GitHub Actions → build Docker image → push to registry → deploy to staging/prod.\n\n## 10. Security & Compliance\n\n- All API endpoints behind JWT auth (except public browse).\n- Rate limiting on auth endpoints.\n- GDPR: user data export, account deletion with 30-day grace period.\n- Data encryption at rest and in transit.\n- Input validation with class-validator (NestJS pipes).\n\n## 11. Monitoring & Observability\n\n- Logging: Winston (structured JSON logs).\n- Metrics: Prometheus + Grafana (request rate, latency, error rate).\n- Tracing: OpenTelemetry for distributed tracing.\n- Alerts: PagerDuty for critical errors (booking failures, high latency).\n\n## 12. Future Considerations\n\n- Business owner web portal (React admin).\n- Real-time chat between customer and business.\n- AI-powered recommendations.\n- Multi-language support (i18n).\n\n---\n\n*This architecture document is a living artifact and should be updated as the system evolves.*",
