Service Boundaries\n\n### 4.1 Mobile Client (apps/mobile)\n\n- **Responsibility**: User interface, navigation, local state, offline support, push notifications.\n- **Key Libraries**: Expo Router (routing), TanStack React Query (server state), React Native Reanimated (animations), Supabase Auth (client-side auth).\n- **Data Flow**: Components use React Query hooks to fetch data from the API. Mutations invalidate cache. Local state managed via Zustand for UI-specific concerns.\n\n### 4.2 Backend API (apps/api)\n\n- **Responsibility**: Business logic, data persistence, authentication, authorization, external integrations.\n- **Modules**: Each feature module is a NestJS module with its own controllers, services, DTOs, and Prisma queries.\n- **API Style**: RESTful with versioned endpoints (e.g., /api/v1/bookings).\n\n### 4.3 Database Layer\n\n- **PostgreSQL**: Primary data store for users, businesses, services, bookings, reviews.\n- **PostGIS**: Geospatial extension for location-based queries (search by distance, map clustering).\n- **Prisma**: ORM with migrations, type-safe queries, and connection pooling.\n\n### 4.4 Cache Layer\n\n- **Redis**: Used for session storage, rate limiting, caching frequently accessed data (business details, availability slots), and pub/sub for real-time notifications.\n\n### 4.5 Authentication Service\n\n- **Supabase Auth**: Handles user registration, OAuth providers, JWT issuance, and token refresh. The backend validates tokens via Supabase Admin SDK.\n\n## 5. Data Flow\n\n### 5.1 Booking Flow\n\n1. User selects service, professional, date/time on mobile.\n2. Mobile sends POST /api/v1/bookings/check-availability with slot details.\n3. Backend checks availability in PostgreSQL (via Prisma) and holds the slot in Redis (TTL 10 minutes).\n4. User confirms booking; mobile sends POST /api/v1/bookings.\n5. Backend creates booking in PostgreSQL, releases Redis hold, sends confirmation notification via push/SMS.\n6. React Query invalidates relevant queries (user's bookings, business availability).\n\n### 5.2 Search Flow\n\n1. User types query in search bar.\n2. After 300ms debounce, mobile sends GET /api/v1/search?q=...&lat=...&lng=...&radius=...\n3. Backend uses PostgreSQL full-text search (tsvector) and PostGIS ST_DWithin for distance filtering.\n4. Results are cached in Redis for 5 minutes.\n5. Mobile renders results with infinite scroll (React Query pagination).\n\n## 6. Key Design Decisions\n\n### 6.1 Monorepo with Nx\n\n- **Why**: Shared types and utilities between mobile and backend reduce duplication. Nx provides dependency graph, affected commands, and build caching.\n- **Impact**: Faster CI/CD, consistent code style, easier refactoring.\n\n### 6.2 TanStack React Query\n\n- **Why**: Declarative server state management with automatic caching, background refetching, and optimistic updates. Reduces boilerplate for loading/error states.\n- **Impact**: Better UX with instant feedback, reduced API calls.\n\n### 6.3 Expo Router\n\n- **Why**: File-based routing simplifies navigation structure, supports deep linking, and integrates with React Navigation.\n- **Impact**: Easier to maintain and scale routing logic.\n\n### 6.4 Prisma with PostgreSQL + PostGIS\n\n- **Why**: Type-safe queries, auto-generated migrations, and native geospatial support.\n- **Impact**: Reduced runtime errors, faster development, accurate location-based features.\n\n### 6.5 Redis for Slot Holding\n\n- **Why**: Prevents double-booking during checkout without locking database rows. TTL ensures slots are released if user abandons.\n- **Impact**: High concurrency support, improved user experience.\n\n## 7. Security Considerations\n\n- **Authentication**: JWT with short-lived access tokens (15 min) and long-lived refresh tokens (30 days). Refresh token rotation invalidates old tokens.\n- **Rate Limiting**: Implemented via Redis-based rate limiter on auth endpoints (5 attempts/15 min) and general API (1000 requests/hour per user).\n- **Input Validation**: Zod schemas on both client and server to prevent injection attacks.\n- **Data Encryption**: Sensitive data (phone numbers, addresses) encrypted at rest using PostgreSQL pgcrypto.\n- **CORS**: Strict origin policy for API access.\n\n## 8. Scalability & Performance\n\n- **Horizontal Scaling**: Backend API can be scaled via Docker Compose or Kubernetes. Stateless design allows multiple instances behind a load balancer.\n- **Database**: Read replicas for search queries, connection pooling via PgBouncer.\n- **Caching**: Redis for hot data (business profiles, availability slots). CDN for static assets (images).\n- **Image Optimization**: Images served via CDN with WebP format and responsive sizes.\n\n## 9. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions and services. Mock Prisma and external services.\n- **Integration Tests**: Test API endpoints with a test database (Docker Compose).\n- **E2E Tests**: Detox or Maestro for mobile app flows.\n- **CI/CD**: GitHub Actions run tests on every PR. EAS Build for mobile app builds.\n\n## 10. Deployment\n\n- **Mobile**: EAS Build generates APK/IPA. Distributed via TestFlight (iOS) and internal testing (Android). Production via App Store and Google Play.\n- **Backend**: Docker image pushed to container registry. Deployed to cloud provider (AWS ECS, GCP Cloud Run, or Railway).\n- **Database**: Managed PostgreSQL with automated backups and point-in-time recovery.\n\n## 11. Monitoring & Observability\n\n- **Logging**: Structured JSON logs via NestJS Logger, shipped to ELK stack.\n- **Metrics**: Prometheus metrics for API latency, error rates, database queries.\n- **Alerting**: PagerDuty or Slack alerts for high error rates or service degradation.\n\n## 12. Conclusion\n\nThis architecture provides a solid foundation for the Planity Clone, balancing developer productivity with production-grade reliability. The modular design allows independent scaling of components, while the monorepo ensures code consistency.
