  "content": "# Planity Clone Architecture\n\n## Overview\nPlanity Clone is a mobile-first platform connecting customers with beauty and wellness businesses. The system is built as a monorepo using Nx, with a React Native (Expo) mobile app and a NestJS backend. PostgreSQL with PostGIS handles spatial queries, Redis provides caching and real-time slot holds, and BullMQ manages background jobs for notifications and slot computation. The architecture emphasizes clean separation of concerns, scalability, and maintainability.\n\n## System Architecture Diagram\n```\n┌─────────────────────────────────────────────────────────────┐\n│ Query (server state), React Native Reanimated (animations), Zustand (client state).\n- **Backend**: NestJS (modular monolith), TypeScript, Prisma ORM, PostgreSQL 15+ with PostGIS extension, Redis 7+, BullMQ (job queues).\n- **Infrastructure**: Docker Compose (local dev), GitHub Actions (CI/CD), Supabase (auth, storage, real-time optional), EAS Build (mobile builds).\n- **Testing**: Jest (unit/integration), React Native Testing Library, Supertest (e2e).\n- **Monorepo**: Nx with pnpm workspaces.\n\n## Monorepo Structure (Nx)\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                 # Expo app\n│   └── api/                    # NestJS backend\n├── libs/\n│   ├── shared-types/           # TypeScript interfaces, DTOs\n│   ├── ui/                     # Shared UI components (if web admin later)\n│   ├── utils/                  # Common utilities\n│   └── config/                 # Environment configs\n├── tools/                      # Nx generators, scripts\n├── docker-compose.yml\n├── nx.json\n├── package.json\n└── pnpm-workspace.yaml\n```\n\n## Backend Architecture (NestJS)\nThe backend follows a modular monolith pattern, organized by domain. Each module encapsulates its own controllers, services, repositories (via Prisma), and DTOs. Cross-cutting concerns (auth, logging, validation) are handled via NestJS guards, interceptors, and pipes.\n\n### Modules\n- **AuthModule**: JWT authentication, social login (Google, Apple), password reset, email verification. Uses `@nestjs/passport`, `passport-jwt`, `passport-google-oauth20`, `passport-apple`. Rate limiting via `@nestjs/throttler`.\n- **UserModule**: Customer and business owner profiles, preferences, payment methods, account deletion.\n- **BusinessModule**: Business CRUD, services, staff, working hours, holidays, categories, reviews, gallery. Staff schedules override business hours.\n- **BookingModule**: Booking flow, slot holds (Redis), appointment management, rescheduling, cancellation, status transitions. Real-time slot hold with 5-minute TTL.\n- **SearchModule**: Full-text search (PostgreSQL `tsvector`), geospatial queries (PostGIS), autocomplete, filters, sorting. Fuzzy search via `pg_trgm`.\n- **NotificationModule**: Push notifications (Expo Push), email (Nodemailer), SMS (Twilio). Templates managed via database.\n- **PaymentModule**: Integration with Stripe for payments, refunds, saved payment methods.\n- **AdminModule**: Dashboard analytics, user management, category management, approval workflows.\n- **AvailabilityModule**: Slot computation engine using BullMQ jobs. Computes available slots based on business hours, staff schedules, service duration, buffers, and existing bookings. Results cached in Redis.\n\n### Data Layer\n- **Prisma ORM**: Schema defines all models. Migrations handled via Prisma Migrate.\n- **PostgreSQL + PostGIS**: Primary database. PostGIS enables `geography` columns for business locations, distance queries (`ST_DWithin`), and spatial indexes.\n- **Redis**: Caching (frequently accessed data like categories, business details), slot holds (temporary locks with TTL), session blacklisting, rate limiting.\n- **BullMQ**: Job queues for slot computation, notification dispatch, email sending, data export. Redis-backed.\n\n### API Design\nRESTful APIs with versioning (`/api/v1/`). Request/response validation using `class-validator` and `class-transformer`. Pagination follows cursor-based pattern for lists. WebSocket (Socket.IO) for real-time slot updates and booking status changes (optional).\n\n### Authentication & Authorization\n- JWT access tokens (15 min) and refresh tokens (7 days) stored securely in mobile (SecureStore).\n- Role-based guards (`@Roles('customer','business_owner','admin')`).\n- Business owners scoped to their own business; admins have full access.\n- Social login tokens verified with provider, then mapped to local user.\n\n## Frontend Architecture (Expo)\nThe mobile app uses Expo Router for file-based navigation, TanStack React Query for server state management, and Zustand for local UI state. Reanimated powers smooth animations for transitions, skeleton loaders, and interactive maps.\n\n### Directory Structure (apps/mobile)\n```\nmobile/\n├── app/                        # Expo Router routes (file-based)\n│   ├── (auth)/                 # Auth screens (login, signup, forgot password)\n│   ├── (tabs)/                 # Main tab navigator\n│   │   ├── home.tsx\n│   │   ├── search.tsx\n│   │   ├── bookings.tsx\n│   │   ├── favorites.tsx\n│   │   └── profile.tsx\n│   ├── business/[id].tsx       # Business detail\n│   ├── booking/                # Booking flow steps\n│   ├── map.tsx                 # Map view\n│   └── _layout.tsx\n├── components/                 # Reusable UI components\n│   ├── ui/                     # Button, Card, Input, etc.\n│   ├── business/               # BusinessCard, ServiceList, StaffCard\n│   ├── booking/                # SlotPicker, DatePicker, ServiceSelector\n│   └── map/                    # MapView, ClusteredMarkers\n├── hooks/                      # Custom hooks (useAuth, useBooking, useSearch)\n├── services/                   # API client functions (axios/fetch wrappers)\n├── stores/                     # Zustand stores (auth, booking flow, filters)\n├── utils/                      # Helpers, formatters\n├── constants/                  # Theme, config\n└── app.json                    # Expo config\n```\n\n### State Management\n- **Server State**: TanStack React Query handles all API data fetching, caching, pagination, and optimistic updates. Query keys follow `[domain, ...params]` convention.\n- **Client State**: Zustand for global UI state (e.g., selected filters, map viewport, booking flow step).\n- **Auth State**: Managed via React Query with persistent storage (SecureStore) for tokens.\n\n### Navigation & Deep Linking\nExpo Router provides deep linking out of the box. Routes like `business/[id]`, `booking/[businessId]` enable direct navigation from notifications or external links.\n\n### Real-time Updates\n- Slot availability: Polling with React Query `refetchInterval` (every 30s) or WebSocket subscription for critical booking flow.\n- Booking status changes: Push notifications trigger query invalidation.\n\n## Data Flow\n1. **Booking Flow**:\n   - User selects service → app fetches available slots from `GET /availability?businessId=&serviceId=&date=`.\n   - Backend checks Redis cache; if miss, triggers slot computation job and returns cached/pending.\n   - User selects slot → app calls `POST /bookings/hold` which sets a Redis key with TTL 300s.\n   - User confirms → `POST /bookings` validates hold, creates booking in DB, publishes event to BullMQ for notifications.\n   - On success, invalidate queries for availability and bookings.\n\n2. **Search**:\n   - Query sent to `GET /search?q=&lat=&lng=&radius=&filters=`. Backend uses PostgreSQL full-text search and PostGIS `ST_DWithin` with spatial index. Results paginated.\n\n3. **Notifications**:\n   - Events (booking created, cancelled, reminder) are pushed to BullMQ queue `notifications`. Worker processes job, determines channels (push, email, SMS), and dispatches via respective providers.\n\n## Background Jobs (BullMQ)\n- **Slot Computation**: Triggered when business hours, staff schedules, or bookings change. Recomputes slots for affected date range and stores in Redis (hash per business:date).\n- **Notification Dispatch**: Handles push, email, SMS with retry logic and rate limiting.\n- **Data Cleanup**: Expired slot holds, soft-deleted accounts, old logs.\n- **Analytics Aggregation**: Daily rollups for admin dashboard.\n\n## Deployment & CI/CD\n- **Local Development**: Docker Compose spins up PostgreSQL, Redis, and API. Mobile app runs via Expo Go or development build.\n- **CI/CD (GitHub Actions)**:\n  - Lint, type-check, unit tests on PR.\n  - Build and push Docker image for API.\n  - EAS Build for mobile (development, preview, production).\n  - Deploy API to cloud (e.g., AWS ECS, Railway) and run migrations.\n- **Supabase**: Used for PostgreSQL hosting (with PostGIS), auth (optional, can be replaced by custom), and storage for business images.\n- **EAS Build**: Handles native builds, code signing, and OTA updates via EAS Update.\n\n## Testing Strategy\n- **Unit Tests**: Jest for pure logic (utils, hooks, services). Backend services tested with mocked Prisma/Redis.\n- **Integration Tests**: Supertest for API endpoints with test database. React Native Testing Library for component interactions.\n- **E2E**: Detox or Maestro for critical flows (booking, auth).\n- **Performance**: Slot computation benchmarked; Redis caching monitored.\n\n## Security Considerations\n- All endpoints (except public) require JWT. Refresh token rotation.\n- Input sanitization via `class-validator`. SQL injection prevented by Prisma.\n- Rate limiting on auth endpoints and booking holds.\n- Sensitive data (tokens, payment info) never stored in plain text; payment processing delegated to Stripe.\n- HTTPS enforced. API keys managed via environment variables.\n\n## Monitoring & Logging\n- Structured logging with Pino (backend) and Sentry for error tracking.\n- Health checks for database, Redis, and BullMQ.\n- Prometheus metrics for API latency, booking rates, job queue sizes.\n\n## Folder Scaffold (Complete)\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/\n│   │   ├── app/\n│   │   │   ├── (auth)/\n│   │
