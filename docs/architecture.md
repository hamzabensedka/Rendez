Overview\n\nThis document describes the system architecture for the Planity Clone, a multi-platform application connecting customers with local service businesses. The architecture is designed for scalability, maintainability, and clear separation of concerns. It uses a monorepo structure with Nx, a NestJS backend, and an Expo React Native frontend.\n\n## 2. Architecture Principles\n\n- **Clean Architecture**: Each layer has distinct responsibilities (presentation, application, domain, infrastructure).\n- **Separation of Concerns**: Services are modular and independently deployable where feasible.\n- **API-First Design**: The backend exposes a RESTful API consumed by the mobile app and future web clients.\n- **Type Safety**: TypeScript is used throughout the stack.\n- **Testability**: Unit and integration tests are first-class citizens.\n- **Observability**: Logging, metrics, and distributed tracing are built-in.\n\n## 3. Technology Stack\n\n| Layer | Technology |\n|-------|------------|\n| Mobile Framework | Expo (React Native) |\n| Mobile Language | TypeScript |\n| Navigation | Expo Router |\n| State Management (Server) | TanStack React Query |\n| Animations | React Native Reanimated |\n| Backend Framework | NestJS |\n| ORM | Prisma |\n| Database | PostgreSQL + PostGIS |\n| Cache / Session | Redis |\n| Monorepo Tool | Nx |\n| Package Manager | pnpm |\n| Containerization | Docker Compose |\n| CI/CD | GitHub Actions |\n| Mobile Builds | EAS Build |\n| Backend Hosting | Supabase (optional) |\n| Testing | Jest |\n\n## 4. Service Boundaries (Backend Modules)\n\n### 6.1 Auth Module\n- **Responsibility**: User registration, login, token management, password reset, social login.\n- **Key Entities**: User, RefreshToken\n- **Endpoints**:\n  - `POST /auth/register`\n  - `POST /auth/login`\n  - `POST /auth/refresh`\n  - `POST /auth/forgot-password`\n  - `POST /auth/reset-password`\n- **Dependencies**: Users module, JWT service, Redis (for refresh tokens).\n\n### 6.2 Users Module\n- **Responsibility**: User profile management, roles, preferences.\n- **Key Entities**: User, UserRole (enum)\n- **Endpoints**:\n  - `GET /users/me`\n  - `PATCH /users/me`\n  - `GET /users/:id` (admin only)\n- **Dependencies**: Auth module.\n\n### 6.3 Businesses Module\n- **Responsibility**: Business CRUD, operating hours, photos, staff management.\n- **Key Entities**: Business, Staff, BusinessHour, BusinessPhoto\n- **Endpoints**:\n  - `GET /businesses` (with filters, pagination)\n  - `GET /businesses/:id`\n  - `POST /businesses` (owner)\n  - `PATCH /businesses/:id` (owner)\n  - `GET /businesses/:id/staff`\n- **Dependencies**: Users module, Search module.\n\n### 6.4 Services Module\n- **Responsibility**: Service catalog per business, categories, pricing, duration.\n- **Key Entities**: Service, Category, ServiceCategory\n- **Endpoints**:\n  - `GET /businesses/:businessId/services`\n  - `POST /businesses/:businessId/services` (owner)\n  - `PATCH /services/:id` (owner)\n  - `GET /categories`\n- **Dependencies**: Businesses module, Categories module.\n\n### 6.5 Bookings Module\n- **Responsibility**: Appointment creation, slot management, rescheduling, cancellation.\n- **Key Entities**: Booking, BookingStatus (enum), Slot\n- **Endpoints**:\n  - `GET /businesses/:id/slots?date=...&staffId=...`\n  - `POST /bookings`\n  - `GET /bookings/me`\n  - `PATCH /bookings/:id/reschedule`\n  - `PATCH /bookings/:id/cancel`\n  - `PATCH /bookings/:id/status` (business)\n- **Dependencies**: Businesses module, Services module, Users module, Notifications module.\n\n### 6.6 Reviews Module\n- **Responsibility**: Customer reviews and ratings.\n- **Key Entities**: Review\n- **Endpoints**:\n  - `GET /businesses/:id/reviews`\n  - `POST /businesses/:id/reviews` (authenticated customer)\n  - `PATCH /reviews/:id` (owner)\n- **Dependencies**: Businesses module, Bookings module (only reviewed after completed booking).\n\n### 6.7 Search Module\n- **Responsibility**: Full-text search, geospatial queries, autocomplete, trending.\n- **Endpoints**:\n  - `GET /search?q=...&lat=...&lng=...&radius=...&category=...&sort=...&page=...`\n  - `GET /search/autocomplete?q=...`\n  - `GET /search/trending`\n- **Dependencies**: Businesses module, Redis (caching, trending), PostgreSQL with PostGIS.\n\n### 6.8 Notifications Module\n- **Responsibility**: Push notifications, email, SMS.\n- **Endpoints**:\n  - `POST /notifications/send` (internal)\n- **Dependencies**: Users module, external services (Firebase, SendGrid, Twilio).\n\n### 6.9 Categories Module\n- **Responsibility**: Category taxonomy management.\n- **Key Entities**: Category\n- **Endpoints**:\n  - `GET /categories`\n  - `POST /categories` (admin)\n- **Dependencies**: None.\n\n## 7. Key Design Decisions\n\n### 8.1 Geospatial Queries\n- Use PostgreSQL with PostGIS extension for location-based queries.\n- Store business locations as `geography(Point, 4326)` for accurate distance calculations.\n- Use ST_DWithin for radius searches and ST_Distance for sorting.\n\n### 8.2 Caching Strategy\n- **Redis** for:\n  - Refresh token storage (with TTL)\n  - Rate limiting\n  - Session data\n  - Caching search results (short TTL: 60s)\n  - Caching business details (medium TTL: 300s)\n  - Trending searches (longer TTL: 3600s)\n- Use TanStack React Query's built-in caching on the mobile side.\n\n### 8.3 Authentication Flow\n- JWT access tokens (short-lived, 15 minutes) + refresh tokens (long-lived, 7 days).\n- Refresh tokens stored in Redis with user identifier for revocation.\n- On token refresh, issue new access token and rotate refresh token.\n- Mobile app stores tokens securely using `expo-secure-store`.\n\n### 8.4 Booking Concurrency\n- Use optimistic locking with a version field on the Slot entity (or use database row-level locks).\n- Slots are generated on-the-fly based on business hours, staff availability, and existing bookings.\n- A booking holds a slot for 10 minutes (via Redis lock) during payment processing.\n\n### 8.5 Search Indexing\n- Use PostgreSQL full-text search with GIN indexes on business name, description, and service names.\n- For autocomplete, use trigram indexes (pg_trgm extension).\n- Consider Elasticsearch if search complexity grows (P2).\n\n## 9. API Design\n\n- RESTful API with consistent error responses.\n- Pagination: cursor-based for lists (e.g., bookings, search results).\n- Versioning: URL prefix `/v1/`.\n- Authentication: Bearer token in `Authorization` header.\n- Rate limiting: 100 requests per minute per user (stricter for anonymous).\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions and services.\n- **Integration Tests**: NestJS testing utilities with a test database.\n- **E2E Tests**: For critical flows (registration, booking).\n- **Mobile Tests**: Jest + React Native Testing Library for components.\n\n## 11. CI/CD Pipeline\n\n- **GitHub Actions**:\n  - On PR: lint, type-check, unit tests, integration tests.\n  - On merge to main: build Docker images, run E2E tests, deploy to staging.\n  - On tag: deploy to production.\n- **EAS Build**: For mobile app builds (development, preview, production).\n\n## 12. Deployment\n\n- Backend: Docker Compose for local development; production on cloud (AWS ECS or similar).\n- Database: Managed PostgreSQL (e.g., Supabase, AWS RDS).\n- Redis: Managed Redis (e.g., Upstash, AWS ElastiCache).\n- Mobile: Over-the-air updates via EAS Update.\n\n## 13. Security Considerations\n\n- Passwords hashed with bcrypt.\n- JWT secrets stored in environment variables.\n- HTTPS enforced.\n- Input validation and sanitization (class-validator).\n- CORS configured for mobile app origin.\n- Rate limiting on auth endpoints.\n- SQL injection prevention via Prisma.\n\n## 14. Monitoring and Observability\n\n- Structured logging (JSON format).\n- Health check endpoint (`GET /health`).\n- Integration with Sentry for error tracking.\n- Metrics via Prometheus (optional, P2).\n\n## 15. Future Considerations\n\n- **Multi-language support**: i18n in mobile app.\n- **Offline mode**: Local storage of business data and bookings.\n- **Web version**: Next.js frontend sharing API.\n- **Real-time updates**: WebSockets for booking status changes.\n\n---\n\n*This document is a living artifact and should be updated as the system evolves.*",
