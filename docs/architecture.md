  "content": "# Architecture Document for Planity Clone\n\n## 1. Overview\nPlanity Clone is a mobile-first appointment booking platform connecting users with local service businesses. The system follows a clean, layered architecture with clear separation of concerns: presentation (Expo/React Native), application services (NestJS API), data layer (Prisma + PostgreSQL/PostGIS), caching (Redis), and background processing (BullMQ). The monorepo is managed with Nx and pnpm, enabling code sharing between mobile and backend (shared types, design system tokens, utilities). CI/CD is handled via GitHub Actions, with EAS Build for mobile artifacts and Docker Compose for local/dev environments.\n\n## 2. Router.\n- **API Gateway (NestJS)** – Entry point for all mobile requests; responsible for authentication, request validation, rate limiting, and routing to domain services.\n- **Domain Services (NestJS modules)** – Each bounded context lives in its own NestJS module: `auth`, `business`, `booking`, `payment`, `notification`, `review`, `admin`, `provider`. Services contain application logic, use Prisma repositories, and publish domain events to BullMQ.\n- **Data Layer** – PostgreSQL with PostGIS extension for geo‑queries; Prisma provides type‑safe ORM and migration system.\n- **Caching Layer** – Redis stores JWT blacklist, session data, rate‑limit counters, and frequently accessed read‑only data (e.g., categories, business listings).\n- **Background Workers** – BullMQ processes jobs: email/SMS notifications, payment webhook handling, appointment reminders, analytics aggregation.\n- **DevOps** – Docker Compose defines local services (api, db, redis, worker). GitHub Actions runs lint, test, build, and pushes Docker images; EAS Build creates iOS/Android binaries.\n\n## 3. Technology Choices Rationale\n- **Expo + React Native** – Rapid development, over‑the‑air updates, unified codebase for iOS/Android.\n- **TypeScript** – End‑to‑end type safety, shared interfaces between mobile and backend.\n- **Expo Router** – File‑based routing, supports web and native, simplifies navigation.\n- **TanStack React Query** – Data fetching, caching, synchronization, background updates.\n- **React Native Reanimated** – Performant, declarative animations for UI polish.\n- **NestJS** – Modular, scalable server‑side framework with built‑in DI, validation, and microservice support.\n- **Prisma** – Type‑safe ORM, migration management, works well with PostgreSQL.\n- **PostgreSQL + PostGIS** – Reliable relational DB; PostGIS enables location‑based search (nearby businesses).\n- **Redis** – Fast in‑memory store for caching, rate limiting, session handling.\n- **BullMQ** – Reliable job queue with retry, prioritization, and monitoring.\n- **Nx** – Monorepo tooling for code sharing, affected commands, consistent linting/formatting.\n- **pnpm** – Efficient disk usage, deterministic installs.\n- **Docker Compose** – Local development environment parity.\n- **GitHub Actions** – CI/CD pipeline automation.\n- **Supabase** (optional) – Provides managed Postgres + Auth; can be swapped for self‑hosted.\n- **EAS Build** – Expo’s managed build service for iOS/Android binaries.\n\n## 4. API Contracts (REST/JSON)\n- **Auth**: `POST /auth/register`, `POST /auth/login`, `POST /auth/logout`\n- **Business**: `GET /businesses?lat=&lng=&radius=&category=&search=`, `GET /businesses/:id`\n- **Booking**: `POST /bookings`, `GET /bookings/me`, `PATCH /bookings/:id/cancel`, `PATCH /bookings/:id/reschedule`\n- **Payment**: `POST /payments/webhook` (Stripe/PayPal), `GET /payments/:id`\n- **Notification**: `GET /notifications/me`, `POST /notifications/:id/read`\n- **Provider**: `GET /provider/dashboard`, `PATCH /provider/business/:id`\n- **Admin**: `GET /admin/stats`, `GET /admin/users`, `GET /admin/businesses`\n\nAll endpoints are protected by JWT (access token) with refresh token rotation; role‑based guards enforce provider/admin scopes.\n\n## 7. Caching Strategy\n- **Redis** stores:\n  - JWT blacklist (logout)\n  - Rate limit counters (IP + user)\n  - Frequently read business lists (TTL 5 min)\n  - Category hierarchies (TTL 1 h)\n- React Query provides client‑side cache with stale‑while‑revalidate; background refetch on focus/reconnect.\n\n## 8. Background Jobs (BullMQ)\n| Job Queue   | Trigger                         | Actions                                      |\n|-------------|---------------------------------|----------------------------------------------|\n| notification| Booking created/updated/cancelled| Send email/SMS via provider (Twilio/SendGrid) |\n| payment     | Payment webhook received        | Verify, update booking status, emit event    |\n| reminder    | Scheduled (cron)                | Fetch upcoming bookings, push notifications  |\n| analytics   | Daily aggregation               | Compute KPIs, store in reporting table       |\n\nWorkers are horizontally scalable; failed jobs retry with exponential backoff and are logged to a centralized logging service (e.g., Loki).\n\n## 9. DevOps & CI/CD\n- **Local**: `docker compose up` starts API, DB, Redis, worker.\n- **CI**: GitHub Actions runs `nx lint`, `nx test`, `nx build` on PRs; on merge to `main` builds Docker images and pushes to registry.\n- **CD**: \n  - API/Worker: Kubernetes (or ECS) pulls latest image, runs migrations via Prisma.\n  - Mobile: EAS Build creates IPA/APK; release channel configured for OTA updates.\n- **Environments**: `dev`, `staging`, `prod` with separate DB instances; secrets managed via GitHub Secrets / AWS Parameter Store.\n\n## 10. Shared Code & Design System\n- **`libs/shared`** contains TypeScript interfaces (User, Business, Booking, etc.) used by both mobile (via barrel export) and NestJS (via `paths` alias). Guarantees contract consistency.\n- **Design System tokens** (colors, spacing, typography) live in `libs/design-system` and are consumed by exporting a theme object; mobile components use `styled-components/native` or `React Native Paper` with the theme.\n- **Utilities** (geo calculations, date formatting, validation) are in `libs/shared/utils`.\n\n## 11. Testing Strategy\n- **Unit**: Jest for services, repositories, utilities; mocking Prisma with `@nestjs/testing`.\n- **Integration**: Supertest for API endpoints; testcontainers for PostgreSQL/Redis.\n- **E2E (Mobile)**: Detox (or Expo’s built‑in test) on critical flows: auth → search → booking → confirmation.\n- **CI** runs all test suites; coverage threshold 80%.\n\n## 12. Security Considerations\n- Authentication via JWT with HttpOnly cookies (web) or AsyncStorage (mobile) + refresh token rotation.\n- Passwords hashed with bcrypt (salt 12).\n- Role‑based access control (RBAC) guards.\n- Input validation using class‑validator & DTOs.\n- Rate limiting & brute‑force protection via Redis.\n- SQL injection prevented by Prisma parameterization.\n- Data in transit encrypted with TLS.\n- Sensitive data (payment details) never stored; only tokenized references from payment gateway.\n- Audit logs for admin/provider actions.\n\n## 13. Scalability & Performance\n- **Read scaling**: Read replicas for PostgreSQL; cache‑aside pattern with Redis.\n- **Write scaling**: Partition bookings by business ID if needed; async processing via BullMQ.\n- **Geo queries**: PostGIS `GIST` index on `location` enables fast nearest‑neighbor searches.\n- **Horizontal**: Stateless API workers; scale behind load balancer.\n- **Mobile**: React Query’s pagination and infinite queries limit payload size.\n\n## 14. Future Extensions\n- GraphQL federation for richer client queries.\n- WebSocket subscriptions for real‑time updates (e.g., admin dashboard).\n- Multi‑tenant SaaS model with separate schemas per provider.\n- AI‑powered recommendation engine for business discovery.\n\n---\n*This document captures the complete system architecture, folder structure, and service boundaries for the Planity Clone using the specified stack.
