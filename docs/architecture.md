Service Boundaries & Modules\n\n### 5.1 Mobile App (Expo)\n- **Routing**: Expo Router with file-based navigation. Stacks for auth, main (tabs: Home, Search, Map, Appointments, Profile), booking flow.\n- **State Management**: TanStack React Query for server state, React Context for auth state, Zustand for local UI state (filters, map viewport).\n- **Data Fetching**: All API calls go through the shared `api-client` library, which provides typed hooks (e.g., `useBusinesses`, `useAppointments`). Optimistic updates for favorites and booking.\n- **Animations**: React Native Reanimated for smooth transitions, map interactions, and micro-interactions.\n- **Offline Support**: React Query's `persistQueryClient` with AsyncStorage for caching recent searches and business details.\n- **Maps**: `react-native-maps` with custom markers and clustering.\n\n### 5.2 Provider Web Portal\n- **Framework**: Next.js with server-side rendering for initial load, client-side for interactivity.\n- **Features**: Dashboard (appointments, revenue), business profile editing, staff & service management, availability calendar.\n- **Auth**: Supabase Auth with role-based access (provider).\n- **Shared**: Uses `design-system` and `api-client` libraries.\n\n### 5.3 Admin Web Dashboard\n- **Framework**: Next.js, similar to provider portal but with admin-specific views.\n- **Features**: User management, business approval, category configuration, analytics, platform settings.\n- **Auth**: Supabase Auth with admin role.\n\n### 5.4 Backend API (NestJS)\n- **Architecture**: Modular monolith with clear domain boundaries. Each module encapsulates its own controller, service, and repository (via Prisma).\n- **Modules**:\n  - **AuthModule**: Registration, login, social auth, JWT issuance, refresh tokens, email verification, password reset. Integrates with Supabase Auth for social providers; custom JWT strategy for session management.\n  - **UsersModule**: Profile CRUD, favorites, preferences.\n  - **BusinessesModule**: Business CRUD, services, staff, working hours, images. Geospatial queries using PostGIS.\n  - **AppointmentsModule**: Booking, rescheduling, cancellation, status management. Concurrency control to prevent double-booking.\n  - **SearchModule**: Full-text search (PostgreSQL `tsvector`), geospatial search, filtering, sorting. Uses Redis for caching popular queries.\n  - **NotificationsModule**: Push notifications (Expo push), email (via BullMQ), in-app notifications.\n  - **PaymentsModule**: Integration with Stripe for payment processing (future scope).\n  - **ReviewsModule**: Review submission, moderation, rating aggregation.\n- **Database**: PostgreSQL with PostGIS extension for location-based queries (`ST_DWithin`, `ST_Distance`). Prisma schema defines all models with relations.\n- **Caching**: Redis for session blacklist, rate limiting, search cache, and real-time data (e.g., available slots).\n- **Background Jobs**: BullMQ for sending emails, push notifications, generating reports, and cleaning up expired tokens.\n- **API Style**: RESTful with OpenAPI (Swagger) documentation. JSON request/response. Versioned (`/api/v1/...`).\n\n## 6. Database Design (Key Entities)\n\n- **User**: id, email, passwordHash, name, phone, role (CLIENT, PROVIDER, ADMIN), avatarUrl, emailVerified, createdAt, updatedAt.\n- **Business**: id, ownerId (FK User), name, description, categoryId, address, coordinates (PostGIS geography), phone, website, images[], averageRating, isActive.\n- **Category**: id, name, slug, parentId (self-referencing for hierarchy), icon, isActive.\n- **Service**: id, businessId, name, duration, price, description, category (optional sub-category).\n- **Staff**: id, businessId, name, title, avatarUrl.\n- **WorkingHours**: id, businessId, dayOfWeek, openTime, closeTime.\n- **Appointment**: id, userId, businessId, serviceId, staffId, startTime, endTime, status (PENDING, CONFIRMED, CANCELLED, COMPLETED), notes, createdAt.\n- **Review**: id, userId, businessId, rating, comment, createdAt.\n- **Favorite**: userId, businessId (composite key).\n- **Notification**: id, userId, type, title, body, data (JSON), read, createdAt.\n\nAll tables use UUID primary keys. PostGIS geography column on Business for efficient radius and distance queries. Indexes on foreign keys, search fields, and geospatial column.\n\n## 7. API Design & Data Flow\n\n### 7.1 Authentication Flow\n1. Client sends credentials to `/auth/register` or `/auth/login`.\n2. Backend validates, creates/verifies user, returns `accessToken` (short-lived, 15 min) and `refreshToken` (long-lived, 7 days) in secure HTTP-only cookies (web) or secure storage (mobile).\n3.
