The system is a mobile-first marketplace connecting consumers with beauty, wellness, and service businesses for appointment booking. The architecture follows clean architecture principles with clear separation of concerns, modular monolith backend, and a well-structured React Native frontend.\n\n## 2. Service Boundaries\n\nThe system is organized into the following bounded contexts (modules):\n\n### 4.1 Auth Service\n- **Responsibility:** User registration, login, OAuth, password reset, session management, biometric authentication.\n- **Data:** Users table, sessions table, refresh tokens.\n- **Key Technologies:** Supabase Auth (or Auth0), JWT, Redis (session store).\n- **Exposed APIs:** `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/forgot-password`, `POST /auth/reset-password`, `POST /auth/oauth/:provider`.\n\n### 4.2 User Service\n- **Responsibility:** User profile management, preferences, account deletion, guest session handling.\n- **Data:** Users table, user_preferences table, guest_sessions table.\n- **Exposed APIs:** `GET /users/me`, `PATCH /users/me`, `DELETE /users/me`, `GET /users/me/sessions`, `DELETE /users/me/sessions/:id`.\n\n### 4.3 Business Service\n- **Responsibility:** Business CRUD, photo gallery, staff management, business hours, amenities, policies.\n- **Data:** Businesses table, business_photos table, staff table, business_hours table, amenities table.\n- **Geospatial:** PostGIS for location queries.\n- **Exposed APIs:** `GET /businesses`, `GET /businesses/:id`, `POST /businesses`, `PATCH /businesses/:id`, `DELETE /businesses/:id`, `GET /businesses/:id/staff`, `GET /businesses/:id/photos`.\n\n### 4.4 Service Catalog Service\n- **Responsibility:** Service definitions, categories, pricing, duration, provider assignment.\n- **Data:** Services table, categories table, service_categories table, service_providers table.\n- **Exposed APIs:** `GET /categories`, `GET /categories/:id/services`, `GET /services`, `GET /services/:id`, `POST /services`, `PATCH /services/:id`.\n\n### 4.5 Search Service\n- **Responsibility:** Full-text search, autocomplete, filtering, sorting, map-based search.\n- **Data:** Search index (PostgreSQL full-text search + PostGIS).\n- **Caching:** Redis for hot queries.\n- **Exposed APIs:** `GET /search?q=...&lat=...&lng=...&radius=...&filters=...`.\n\n### 4.6 Booking Service\n- **Responsibility:** Appointment booking, availability management, rescheduling, cancellation, waitlist.\n- **Data:** Bookings table, availability_slots table, waitlist table.\n- **Key Logic:** Timezone handling, conflict detection, business policy enforcement.\n- **Exposed APIs:** `GET /bookings`, `POST /bookings`, `GET /bookings/:id`, `PATCH /bookings/:id`, `DELETE /bookings/:id`, `POST /bookings/:id/waitlist`.\n\n### 4.7 Notification Service\n- **Responsibility:** Push notifications (Expo Push API), email notifications, in-app notifications.\n- **Data:** Notifications table, push_tokens table.\n- **Exposed APIs:** `POST /notifications/send`, `GET /notifications`, `PATCH /notifications/:id/read`.\n\n### 4.8 Admin Service\n- **Responsibility:** Platform monitoring, user management, business verification, dispute resolution, analytics.\n- **Data:** Admin users, audit logs.\n- **Exposed APIs:** `GET /admin/dashboard`, `GET /admin/users`, `PATCH /admin/businesses/:id/verify`, `GET /admin/analytics`.\n\n## 5. Data Flow & Communication\n\n### 5.1 API Communication\n- **Protocol:** REST over HTTPS.\n- **Authentication:** JWT access token in `Authorization: Bearer <token>` header.\n- **Rate Limiting:** Per user/IP using Redis (e.g., 100 requests/min for authenticated users, 20/min for guests).\n- **Error Handling:** Consistent error response format: `{ \"statusCode\": 400, \"message\": \"...\", \"error\": \"Bad Request\" }`.\n\n### 5.2 Real-time Updates\n- **WebSockets:** For booking status changes, notifications (via Supabase Realtime or Socket.IO).\n- **Polling fallback:** TanStack React Query with `refetchInterval` for non-critical updates.\n\n### 5.3 Caching Strategy\n- **Redis:** Session data, rate limit counters, hot search results, business detail cache (TTL: 5 minutes).\n- **TanStack React Query:** Client-side caching with stale-while-revalidate pattern. Cache invalidation on mutations.\n\n### 5.4 Data Flow Example: Booking\n\n1. User selects service and time slot on mobile app.\n2. Mobile app sends `POST /bookings` with JWT token.\n3. API Gateway (NestJS) validates token, rate limits.\n4. Booking Service checks availability in PostgreSQL (with row-level locking).\n5. If available, creates booking record, updates availability slot.\n6. Returns booking confirmation to client.\n7. Triggers notification to business owner via Notification Service.\n8. Client updates local cache via TanStack Query invalidation.\n\n## 6. Security Considerations\n\n- **Authentication:** JWT with short-lived access tokens (15 min) and refresh tokens (7 days). Refresh token rotation.\n- **Authorization:** Role-based access control (RBAC) with NestJS guards. Providers can only manage their own businesses.\n- **Input Validation:** Zod schemas on both frontend and backend.\n- **SQL Injection:** Prevented by Prisma's parameterized queries.\n- **Rate Limiting:** Redis-based throttling.\n- **Data Encryption:** HTTPS everywhere. Sensitive data encrypted at rest.\n- **Device Tracking:** `X-Device-ID` header for session management and security alerts.\n\n## 8. Performance & Scalability\n\n- **Horizontal Scaling:** NestJS is stateless (sessions in Redis). Multiple API instances behind a load balancer.\n- **Database:** PostgreSQL with connection pooling (PgBouncer). Read replicas for heavy read operations (search, business listing).\n- **Caching:** Redis for session data, hot data. TanStack React Query for client-side caching.\n- **Search:** PostgreSQL full-text search with GIN indexes. PostGIS spatial indexes for location queries.\n- **Image Optimization:** Use Supabase Storage with image transformation (resize, compress) for fast loading.\n- **Lazy Loading:** React Native screens and components lazy-loaded via Expo Router.\n\n## 9. Testing Strategy\n\n- **Unit Tests:** Jest for individual functions and components.\n- **Integration Tests:** NestJS e2e tests with test database (Docker Compose).\n- **API Tests:** Supertest for endpoint testing.\n- **Mobile Tests:** Jest + React Native Testing Library for component tests.\n- **CI/CD:** GitHub Actions runs lint, type-check, unit tests, integration tests on every PR.\n\n## 10. Deployment & DevOps\n\n- **Local Development:** Docker Compose spins up PostgreSQL, Redis, and NestJS API.\n- **Mobile Builds:** EAS Build for iOS and Android. Expo Go for development.\n- **Backend Deployment:** Docker containers to cloud (AWS ECS, Google Cloud Run, or Vercel).\n- **Database Migrations:** Prisma Migrate run as part of CI/CD pipeline.\n- **Environment Variables:** Managed via `.env` files (local) and GitHub Secrets (CI/CD).\n\n## 11. Monitoring & Observability\n\n- **Logging:** NestJS built-in logger with structured JSON logs.\n- **Error Tracking:** Sentry for both backend and mobile.\n- **Performance Monitoring:** New Relic or Datadog for API response times.\n- **Analytics:** PostHog or Mixpanel for user behavior.\n\n## 12. Conclusion\n\nThis architecture provides a solid foundation for building the Planity Clone with clean separation of concerns, scalability, and maintainability. The use of Nx monorepo ensures code sharing and consistent tooling across the stack. The modular NestJS backend allows independent development and scaling of services. The Expo + React Native frontend delivers a native experience with fast development cycles.",
