Overview & Design Principles\\n\\nPlanity Clone is a mobile-first appointment booking platform built on a monorepo architecture managed by Nx and pnpm. Service Boundaries & Domain Design\\n\\nEach NestJS module is a self-contained domain with clear responsibility, defined API contract, and dedicated Prisma service. No module directly accesses another module's database tables; communication happens via service methods or events (for cross-cutting concerns).\\n\\n### 3.1 Auth Module\\n- **Responsibility:** User registration, login, token management, password reset, email verification.\\n- **Key Entities:** `User`, `RefreshToken`, `VerificationToken`.\\n- **Technologies:** JWT (access + refresh tokens), bcrypt for hashing, secure storage on mobile via `expo-secure-store`.\\n- **API Endpoints:** `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/forgot-password`, `POST /auth/verify-email`, `GET /auth/session`.\\n- **Dependencies:** Email service (for verification and password reset emails).\\n- **Security:** Rate limiting on login and password reset endpoints, refresh token rotation, token blacklisting (Redis).\\n\\n### 3.2 Users Module\\n- **Responsibility:** User profile management, preferences, favourites.\\n- **Key Entities:** `UserProfile`, `Favourite`.\\n- **API Endpoints:** `GET /users/me`, `PATCH /users/me`, `GET /users/me/favourites`, `POST /users/me/favourites`, `DELETE /users/me/favourites/:businessId`.\\n- **Dependencies:** Auth module (for user identity), Businesses module (for favouriting).\\n\\n### 3.3 Businesses Module\\n- **Responsibility:** Business profiles, CRUD for providers, search indexing.\\n- **Key Entities:** `Business`, `BusinessPhoto`, `OperatingHours`, `Amenity`.\\n- **API Endpoints:** `GET /businesses` (search, filter, paginate), `GET /businesses/:id`, `POST /businesses` (provider), `PATCH /businesses/:id`, `GET /businesses/map` (geospatial query).\\n- **Search Strategy:** PostgreSQL full-text search with `tsvector` on name, description, and service names; trigram (`pg_trgm`) for fuzzy matching; PostGIS `ST_DWithin` for radius queries.\\n- **Indexes:** GIN index on `tsvector`, GiST index on location, B-tree on category and rating.\\n\\n### 3.4 Services Module (Catalog)\\n- **Responsibility:** Service catalog per business (name, duration, price, category).\\n- **Key Entities:** `Service` (belongs to `Business` and `Category`).\\n- **API Endpoints:** `GET /businesses/:businessId/services`, `POST /businesses/:businessId/services`, `PATCH /services/:id`, `DELETE /services/:id`.\\n\\n### 3.5 Categories Module\\n- **Responsibility:** Hierarchical service category management.\\n- **Key Entities:** `Category` (self-referencing for parent/child, flat for MVP).\\n- **API Endpoints:** `GET /categories`, `POST /categories` (admin), `PATCH /categories/:id`, `DELETE /categories/:id`.\\n\\n### 3.6 Availability Module (Critical)\\n- **Responsibility:** Compute and cache real-time available slots for any business/service over a given date range; ensure transactional consistency during booking.\\n- **Key Entities:** `Staff`, `StaffSchedule`, `TimeOff`, `Appointment` (shared via Booking module).\\n- **API Endpoints:** `GET /businesses/:businessId/availability?serviceId=&date=`, `GET /availability/slots/:businessId?serviceId=&staffId=&start=&end=`.\\n- **Computation Logic:**\\n  1. Fetch business operating hours and staff schedules.\\n  2. Generate all possible slots based on service duration and buffer time.\\n  3. Subtract existing appointments (confirmed statuses) and staff time-offs.\\n  4. Apply slot restrictions (max concurrent bookings per staff).\\n  5. Cache computed slots in Redis with TTL (e.g., 60 seconds) for fast reads; invalidate on booking creation/cancellation.\\n- **Concurrency:** Use PostgreSQL advisory locks (`pg_advisory_xact_lock`) within the booking creation transaction to prevent double booking of the same slot.\\n- **Background Job:** A BullMQ job recalculates availability cache when appointments are created, cancelled, or rescheduled.\\n\\n### 3.7 Bookings Module\\n- **Responsibility:** Appointment lifecycle: create, confirm, cancel, reschedule.\\n- **Key Entities:** `Appointment`, `AppointmentStatus` (PENDING, CONFIRMED, CANCELLED, COMPLETED, NO_SHOW).\\n- **API Endpoints:** `POST /bookings` (customer), `GET /bookings` (user's bookings), `GET /bookings/:id`, `PATCH /bookings/:id/cancel`, `PATCH /bookings/:id/reschedule`, `GET /businesses/:id/bookings` (provider).\\n- **Booking Flow:**\\n  1. Client requests available slots (Availability module).\\n  2. Client calls `POST /bookings` with `businessId`, `serviceId`, `staffId`, `startTime`.\\n  3. Server acquires advisory lock, re-checks availability, creates appointment atomically, emits `appointment.created` event.\\n  4. Queue worker sends confirmation notification, triggers availability cache invalidation.\\n\\n### 3.8 Reviews Module\\n- **Responsibility:** Ratings and written reviews for businesses.\\n- **Key Entities:** `Review` (belongs to `User` and `Business`).\\n- **API Endpoints:** `GET /businesses/:businessId/reviews`, `POST /businesses/:businessId/reviews`, `PATCH /reviews/:id`, `DELETE /reviews/:id`.\\n- **Rules:** Only confirmed appointments can be reviewed; one review per user per business (enforced by unique constraint).\\n\\n### 3.9 Notifications Module\\n- **Responsibility:** Push notifications, email, and SMS dispatch (abstracted behind a common interface).\\n- **Key Entities:** `NotificationPreference`, `NotificationLog`.\\n- **Technologies:** Firebase Cloud Messaging (push), Nodemailer/SendGrid (email), Twilio (SMS, future).\\n- **Events Driven:** Listens to `appointment.created`, `appointment.reminder`, `auth.password_reset`, `review.published` via BullMQ queues.\\n\\n### 3.10 Payments Module (P1)\\n- **Responsibility:** Payment processing (pre-payment, no-show fees), provider payouts.\\n- **Key Entities:** `PaymentMethod`, `Transaction`, `Payout`.\\n- **Technologies:** Stripe Connect (handles platform fees and provider payouts).\\n\\n### 3.11 Admin Module\\n- **Responsibility:** Platform oversight, business approval, issue resolution, analytics.\\n- **API Endpoints:** `GET /admin/businesses`, `PATCH /admin/businesses/:id/approve`, `GET /admin/analytics`.\\n- **Authorization:** Role-based access control (RBAC) using `Admin` role.\\n\\n## 4.
