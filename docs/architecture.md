  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nThis document outlines the architecture for the Planity Clone, a mobile app that connects users with local businesses for discovering, booking, and managing appointments. The design follows clean architecture principles, separates concerns into distinct layers, and uses a monorepo managed by Nx and pnpm to share code between mobile, backend, and shared libraries.\n\n## 2. High-Level Architecture\n- **Frontend**: Expo (React Native) application handling UI, navigation, animations, and client‑side state.\n- **Backend**: NestJS server exposing REST/GraphQL APIs, using Prisma ORM to interact with PostgreSQL (PostGIS extension) and Redis for caching; BullMQ processes background jobs.\n- **Infrastructure**: Docker Compose for local development, GitHub Actions for CI/CD, EAS Build for Expo builds, optional Supabase for hosted database/storage, and Nx for task running and code sharing.\n- **Shared**: TypeScript libraries containing DTOs, validation schemas, UI component library, design system tokens, and reusable utilities.\n\n## 3. Service Boundaries\n| Boundary | Responsibility | Key Technologies |\n|----------|----------------|------------------|\n| Mobile App | UI rendering, navigation, local state, offline cache, animations, authentication flow | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated |\n| API Gateway | Entry point, request validation, authentication, rate limiting, routing to domain services | NestJS, Express adapter, class-validator, JWT |\n| Auth Service | User registration, login, password reset, token issuance | NestJS, Prisma, bcrypt, JWT |\n| Business Service | CRUD for businesses, categories, geospatial queries, favorites | NestJS, Prisma, PostGIS, Redis |\n| Booking Service | Slot computation, appointment creation, rescheduling, cancellation | NestJS, Prisma, BullMQ (for confirmation emails/payments) |\n| Payment Service | Secure payment processing, webhook handling | NestJS, Stripe SDK (or similar), Prisma |\n| Notification Service | Sending push/email notifications, preference management | NestJS, BullMQ, Expo Push Notification service |\n| Admin Service | Managing users, businesses, system settings | NestJS, Prisma, RBAC middleware |\n| Shared Libraries | DTOs, validation schemas, UI component library, design system tokens, utility functions | TypeScript, React Native, Storybook (optional) |\n\n## 4. Data Model (Simplified)\n- **User**: id, email, passwordHash, role (user/provider/admin), createdAt, updatedAt\n- **Business**: id, name, description, address, location (Point), phone, categoryId, ownerId, createdAt, updatedAt\n- **Category**: id, name, icon\n- **Service**: id, businessId, name, duration, price, createdAt, updatedAt\n- **Appointment**: id, userId, businessId, serviceId, startTime, endTime, status, createdAt, updatedAt\n- **Favorite**: id, userId, businessId, createdAt\n- **Review**: id, businessId, userId, rating, comment, createdAt, updatedAt\n- **Availability**: id, businessId, weekday, startTime, endTime, isActive\n\nPostGIS stores the `location` field as a geography type to enable radius and proximity queries.\n\n## 5. API Design\n- RESTful endpoints with JSON payloads; versioned under `/api/v1`.\n- Authentication via JWT in Authorization header.\n- Use of DTOs (data transfer objects) for request/response validation (class-validator + class-transformer).\n- Pagination via `limit` and `offset` query parameters.\n- Geospatial search: `GET /businesses?near=[lng,lat]&distance=1000` returns businesses within given meters.\n- Webhooks for payment provider and external services.\n\n## 6. State Management & Data Fetching\n- **TanStack React Query** handles server state: caching, background updates, optimistic updates, and query invalidation.\n- Local UI state (e.g., form inputs, navigation) managed with React's `useState`/`useReducer` or Reanimated worklets for animations.\n- Authentication context provides current user and token across the app.\n\n## 7. Offline & Caching Strategy\n- React Query caches API responses with configurable staleTime.\n- Critical data (business list, user profile) persisted via `expo-secure-store` or `AsyncStorage` for offline browsing.\n- Redis on the backend caches frequent queries (e.g., business listings, categories) to reduce DB load.\n\n## 8. Background Jobs (BullMQ)\n- Job queues: `email-notifications`, `payment-processing`, `appointment-reminders`.\n- Workers defined in NestJS modules; processed via separate Docker service or same container with concurrency limits.\n- Retry policies and dead‑letter queues for failure handling.\n\n## 9. Real‑Time Features (Optional)\n- For live updates (e.g., new booking for a provider), consider integrating Expo Push Notifications or WebSocket via NestJS Gateway.\n- Not covered in MVP but can be added later.\n\n## 10. Design System & Shared UI\n- A dedicated Nx library (`ui`) exports styled components using `react-native-reanimated` for gestures and animations.\n- Design tokens (spacing, typography, colors) stored in `shared/design-tokens.ts`.\n- Storybook can be used for visual testing of components.\n\n## 11. Project Structure (Nx Monorepo)\n```\nplanity-clone/\n├─ apps/\n│  ├─ mobile/            # Expo React Native app\n│  └─ api/               # NestJS server\n├─ libs/\n│  ├─ shared/            # TypeScript utilities, DTOs, validation schemas\n│  ├─ ui/                # UI component library (Reanimated)\n│  ├─ design-tokens/     # Token definitions (colors, spacing, etc.)\n│  ├─ auth/              # Auth-related DTOs and guards\n│  ├─ business/          # Business service DTOs\n│  ├─ booking/           # Booking service DTOs\n│  ├─ payment/           # Payment service DTOs\n│  ├─ notification/      # Notification service DTOs\n│  └─ admin/             # Admin service DTOs\n├─ docker/\n│  ├─ docker-compose.yml # Local dev stack (api, db, redis, worker)\n│  └─ Dockerfile.api\n│  └─ Dockerfile.worker\n├─ .github/\n│  └─ workflows/\n│     ├─ ci.yml          # Lint, test, build\n│     └─ eas-build.yml   # Expo build trigger\n├─ prisma/\n│  └─ schema.prisma      # DB schema with PostGIS extension\n├─ nx.json\n├─ package.json          # Root pnpm workspace\n└─ tsconfig.base.json\n```\n\n## 12. Development Workflow\n- **Local**: `pnpm dev` runs Nx targets: `nx run mobile:start` (Expo dev client) and `nx run api:start` (NestJS in watch mode). Docker Compose provides Postgres, Redis, and BullMQ UI.\n- **Testing**: Jest for unit tests (mobile components, backend services). React Native Testing Library for UI unit tests. End‑to‑end tests with Detox (optional).\n- **Linting/Formatting**: ESLint + Prettier via Nx.\n- **Version Control**: GitHub; feature branches, pull requests with required CI checks.\n- **CI**: GitHub Actions installs dependencies, runs lint, test, builds Docker images, and pushes to registry.\n- **CD**: EAS Build creates iOS/Android binaries from the `main` tag; Docker images deployed to a staging/k8s environment via separate pipeline (not detailed here).\n\n## 13. Deployment & Infrastructure\n- **Development**: Docker Compose (api, postgres, redis, worker). Expo Dev Client for rapid iteration.\n- **Staging/Production**: \n  - API deployed as Docker containers (ECS, Kubernetes, or similar).\n  - PostgreSQL managed service (e.g., Amazon RDS) with PostGIS extension enabled.\n  - Redis managed service (Elasticache).\n  - BullMQ workers as separate worker services.\n  - Expo builds distributed via App Store/Google Play; OTA updates via Expo EAS.\n  - Optional Supabase for hosted DB/storage if avoiding self‑managed PostgreSQL.\n\n## 14. Security Considerations\n- HTTPS enforced via reverse proxy (NGINX) or managed TLS.\n- JWTs signed with strong secret; short-lived access tokens + refresh token rotation.\n- Passwords hashed with bcrypt.\n- Rate limiting on auth endpoints.\n- Input validation and sanitization via class-validator and Prisma parameterized queries.\n- Payment data never touches our servers; processed directly by PCI‑compliant provider (Stripe).\n- Environment secrets managed via GitHub Secrets and injected into Docker/K8s.\n\n## 15. Monitoring & Observability\n- Backend: NestJS built‑in logger + integration with Winston or Pino; logs shipped to ELK or CloudWatch.\n- Metrics: Prometheus endpoint via `@nestjs/terminus`; Grafana dashboards for DB latency, queue lengths, request rates.\n- Health checks: `/health` endpoint verifying DB, Redis, and queue connectivity.\n- Mobile: Expo logs; Sentry for crash reporting and performance monitoring.\n\n## 16. Scalability & Performance\n-Performance\n- Read‑heavy operations (business listing, search) served from Redis cache with TTL.\n- Write‑heavy operations (booking, payments) go directly to PostgreSQL; connection pooling via Prisma.\n- Geospatial queries leverage PostGIS indexes for fast proximity search.\n- Background jobs decouple long‑running tasks (email, payment confirmation) from request cycle.\n- Horizontal scaling of API workers behind a load balancer.\n\n## 17. Future Enhancements\n- WebSocket/GraphQL subscriptions for real‑time updates.\n- AI‑powered search and recommendation.\n- Multi‑language support (i18n).\n- Advanced analytics dashboard for providers.\n- Offline‑first sync using Expo SQLite + React Query persister.\n\n--- End of Architecture Document ---\n",
