  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nThe Planity Clone is a mobile-first marketplace connecting users with local service businesses. The architecture follows a clean, layered approach: a cross‑platform Expo/React Native frontend consumes a NestJS API layer, which interacts with PostgreSQL (extended with PostGIS for geo‑queries) and Redis for caching and background job queuing. The monorepo is managed by Nx and pnpm, enabling shared types, design system, and reusable libraries across mobile, web (if added later), and backend services. CI/CD is powered by GitHub Actions, with Docker Compose for local development, EAS Build for Expo binaries, and optional Supabase for auth/storage fallback.\n\n## 2. Technology Stack\n- **Mobile**: Expo SDK, React Native, TypeScript, Expo Router (file‑based navigation), TanStack React Query (data fetching & caching), React Native Reanimated (gestures & animations)\n- **Backend**: NestJS (Node.js framework), TypeScript, Prisma ORM, PostgreSQL + PostGIS extension, Redis, BullMQ (job queues)\n- **Monorepo & Tooling**: Nx (workspace & task runner), pnpm (package manager), Docker Compose, GitHub Actions, EAS Build, Jest (testing)\n- **Optional Services**: Supabase (auth, storage, real‑time), Sentry (error monitoring), Firebase Cloud Messaging (push notifications)\n\n## 3. High‑Level Architecture\n```\n[Mobile App] <--HTTPS/WS--> [API Gateway (NestJS)] <---> [PostgreSQL/PostGIS]\n                               |                |\n                               |                --> [Redis (cache & pub/sub)]\n                               |\n                               --> [BullMQ Workers] --> [External Services (payment gateway, email/SMS, map tiles)]\n```\nThe API gateway exposes REST endpoints (and optional GraphQL) consumed by the mobile app via React Query. Background jobs (e.g., slot recomputation, notification dispatch) are processed by BullMQ workers. Geo‑search leverages PostGIS for efficient radius queries.\n\n## 4. Service Boundaries\n| Boundary | Responsibility | Technologies |\n|----------|----------------|--------------|\n| **Auth Service** | User registration, login, social OAuth, JWT issuance | NestJS (Passport), Prisma, Redis (session store) |\n| **Business Service** | CRUD for businesses, categories, geo‑indexing, hours | NestJS, Prisma, PostGIS |\n| **Service & Availability Service** | Service definitions, slot generation, availability rules | NestJS, Prisma, BullMQ (recompute jobs) |\n| **Booking Service** | Appointment creation, validation, conflict detection, rescheduling/cancel | NestJS, Prisma, BullMQ (confirmation emails) |\n| **Payment Service** | Payment tokenization, webhook handling, invoicing | NestJS (Stripe SDK), Prisma |\n| **Review & Rating Service** | Reviews, ratings, responses | NestJS, Prisma |\n| **Notification Service** | Push, email, SMS dispatch, preference management | NestJS, BullMQ, FCM/APNS, SendGrid/Twilio |\n| **User Profile Service** | Profile CRUD, preferences | NestJS, Prisma |\n| **Admin & Provider Portal** | Dashboard UI (could be separate web app) – shares same API | NestJS, Prisma |\n| **Shared Library** | TypeScript types, Zod schemas, design system components, utils | Nx library, Expo, React Native |\n\n## 5. Monorepo Structure (Nx + pnpm)\n```\nplanity-clone/\n├─ apps/\n│   ├─ mobile/            # Expo React Native app\n│   │   ├─ src/\n│   │   │   ├─ assets/\n│   │   │   ├─ components/   # shared UI (design system)\n│   │   │   ├─ screens/      # Expo Router file‑based routes\n│   │   │   ├─ hooks/\n│   │   │   ├─ lib/          # React Query clients, Reanimated helpers\n│   │   │   ├─ navigation/   # custom navigation guards (if needed)\n│   │   │   ├─ services/     # API service layer (thin wrapper around React Query)\n│   │   │   ├─ store/        # optional Zustand/Jotai for UI state\n│   │   │   ├─ types/        # re‑exported from @planity/shared-types\n│   │   │   └─ App.tsx\n│   │   ├─ expo/\n│   │   │   └─ ... (app.json, eas.json)\n│   │   └─ ...\n│   └─ admin/             # optional Next.js or NestJS‑served portal (future)\n├─ libs/\n│   ├─ shared-types/      # DTOs, Zod schemas, enum definitions\n│   ├─ design-system/     # reusable native components (Button, Input, Card)\n│   ├─ ui-kit/            # platform‑agnostic UI (if using React Native Web later)\n│   ├─ utils/             # logging, analytics, date helpers\n│   └─ test-helpers/      # Jest custom matchers, mocks\n├─ packages/\n│   └─ prisma/            # Prisma schema, migrations, client wrapper\n│       ├─ prisma/\n│       │   ├─ schema.prisma\n│       │   └─ migrations/\n│       └─ src/\n│           └─ prisma.client.ts\n├─ services/\n│   └─ api/               # NestJS monolith (could be split into microservices later)\n│       ├─ src/\n│       │   ├─ auth/\n│       │   ├─ business/\n│       │   ├─ service/\n│       │   ├─ booking/\n│       │   ├─ payment/\n│       │   ├─ review/\n│       │   ├─ notification/\n│       │   ├─ user/\n│       │   ├─ admin/\n│       │   ├─ common/    # guards, interceptors, DTO base classes\n│       │   ├─ app.module.ts\n│       │   └─ main.ts\n│       ├─ test/\n│       │   └─ ...\n│       └─ tsconfig.json\n├─ docker/\n│   └─ compose.yml        # defines api, db, redis, worker services\n├─ .github/\n│   └─ workflows/\n│       ├─ ci.yml         # lint, test, build\n│       └─ cd.yml         # publish to EAS, Docker images\n├─ nx.json\n├─ package.json           # pnpm workspace root\n└─ README.md\n```\n\n## 6. Frontend Details\n- **Routing**: Expo Router provides file‑based navigation under `apps/mobile/src/screens`. Each folder corresponds to a route (e.g., `src/screens/business/[id]/index.tsx`).\n- **State Management**: React Query handles server state; UI‑only state (modals, form inputs) lives in React state or optional Zustand.\n- **Data Fetching**: Custom hooks in `src/lib/query.ts` wrap React Query with automatic retries, stale‑time based on endpoint.\n- **Animations**: Reanimated 2 used for gesture‑based filters, modal transitions, and map interactions.\n- **Design System**: `libs/design-system` exports themed components using `react-native-paper` or custom styled components with `styled-components/native`.\n- **Shared Types**: Imported from `@planity/shared-types` ensuring DTO parity between mobile and NestJS.\n- **Environment**: Expo's `app.config.js` and `eas.json` define dev/stage/prod builds; secrets managed via EAS secrets and GitHub Actions.\n- **Assets**: Images, icons, fonts under `apps/mobile/src/assets`.\n- **Testing**: Jest with React Native Testing Library; unit tests for hooks, components, and integration tests using Expo's dev client.\n\n## 7. Backend Details (NestJS)\n- **Module Architecture**: Each bounded context is a NestJS module (AuthModule, BusinessModule, etc.) with its own controllers, services, DTOs, and guards.\n- **Validation**: DTOs use `class-validator` + `class-transformer`; shared Zod schemas in `libs/shared-types` for runtime validation.\n- **ORM**: Prisma client generated from `packages/prisma/prisma/schema.prisma`. Transactions handled via Prisma's `$transaction`.\n- **Geo Queries**: Prisma extensions allow raw SQL for PostGIS functions (`ST_DWithin`, `ST_MakePoint`).\n- **Caching**: Redis layer via `ioredis`; cached responses for business listings, slot calculations.\n- **Background Jobs**: BullMQ queues defined in `src/queues/`; workers in `src/workers/` (e.g., `SlotRecomputeWorker`, `NotificationWorker`).\n- **Authentication**: JWT strategy (access token) + refresh token stored in HttpOnly cookies; social OAuth via `passport-google-oauth20` and `passport-facebook`.\n- **Payment**: Stripe integration; webhook endpoint verifies signatures and updates booking status.\n- **Error Handling**: Global exception filter transforms errors to consistent JSON `{ error, message, statusCode }`.\n- **Testing**: Jest with Supertest for endpoint tests; Prisma uses SQLite in-memory for unit tests; integration tests use Docker Compose test profile.\n- **API Documentation**: Swagger UI enabled in dev (`@nestjs/swagger`).\n\n## 8. Database & Caching\n- **PostgreSQL** (version 15) with PostGIS extension for geographic indexing.\n- **Prisma Schema** highlights:\n```prisma\nmodel Business {\n  id            String   @id @default(uuid())\n  name          String\n  description   String?\n  location      Point    @db.Point\n  address       String\n  phone         String?\n  website       String?\n  hours         Json\n  services      Service[] @relation(\"BusinessServices\")\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\nmodel Service {\n  id          String   @id @default(uuid())\n  name        String\n  businessId  String\n  business    Business @relation(\"BusinessServices\", fields: [businessId], references: [id])\n  durationMin Int\n  price       Float\n}\nmodel Booking {\n  id            String   @id @default(uuid())\n  userId        String\n  serviceId     String\n  service       Service  @relation(fields: [serviceId], references: [id])\n  startTime     DateTime\n  endTime       DateTime\n  status        BookingStatus\n  paymentId     String?\n  createdAt     DateTime @default(now())\n}\n```\n- **Redis** stores:\n  - Session refresh tokens (optional)\n  - Cached business lists (TTL 5 min)\n  - Computed slot maps per business/day (TTL 1 min)\n  - BullMQ job queue backend.\n\n## 9. Infrastructure\n- **Local Development**: `docker-compose.yml` spins up `postgres`, `redis`, and `api` (NestJS) containers; Expo dev client runs on host.\n- **CI/CD**:\n  - **GitHub Actions CI** (`ci.yml`): `pnpm install`, `pnpm lint`, `pnpm test`, `pnpm build` (NestJS build, Expo web build for lint).\n  - **GitHub Actions CD** (`cd.yml`): on push to `main`, trigger `eas build --platform all --auto-submit` for iOS/Android; push Docker images to registry; optionally deploy API to a managed Kubernetes or VM.\n- **EAS Build**: Configured for preview and production builds; secrets injected via EAS Secrets.\n- **Supabase** (optional fallback): Provides auth and storage if we want to offload auth; can be swapped via environment variable.\n- **Monitoring**: Sentry SDK in both frontend and backend; custom metrics (e.g., job latency) exposed via Prometheus endpoint in NestJS (`@nestjs/microservices`).\n\n## 10. Development Workflow\n1. **Feature Branch**: Create branch from `main`.\n2. **Add/Update Types**: Modify `libs/shared-types`; run `pnpm nx run shared-types:build`.\n3. **Backend**: Implement NestJS service/module; write unit tests; run `pnpm nx run api:test`.\n4. **Frontend**: Consume updated types via generated API hooks; write component/tests; run `pnpm nx run mobile:test`.\n5. **Lint & Format**: `pnpm lint` (ESLint + Prettier).\n6. **Commit**: Use conventional commits; CI validates.\n7. **PR**: Requires approvals, passes CI, then merges to `main`.\n8. **Release**: `main` triggers CD pipeline; publishes new Expo builds and Docker images.\n\n## 11. Testing Strategy\n- **Unit Tests**: Jest for pure functions, utilities, Prisma model methods, NestJS services (mocked repositories).\n- **Component Tests**: React Native Testing Library + Jest for UI snapshots and interactions.\n- **Integration Tests**: \n  - Backend: Supertest against real PostgreSQL (Docker) and Redis.\n  - Frontend: Expo's dev client with Jest + React Native Testing Library for navigation flows.\n- **E2E (Future)**: Detox for critical user journeys (login → search → book → confirmation).\n- **Test Coverage Goal**: ≥80% for backend, ≥70% for frontend.\n\n## 12. Security & Privacy\n- **Authentication**: JWT with short‑lived access token (15 min) + refresh token stored HTTP‑Only, SameSite.\n- **Authorization**: Role‑based guards (`AuthGuard`, `RolesGuard`) for admin/provider endpoints.\n- **Data Protection**: TLS everywhere; sensitive fields (payment tokens) never stored; only Stripe IDs.\n- **Input Validation**: Zod schemas at DTO level; Prisma prevents SQL injection.\n- **Rate Limiting**: NestJS `ThrottlerGuard` on auth and public endpoints.\n- **Secrets Management**: EAS secrets for mobile; GitHub Secrets for CI; Docker secrets or env files for local.\n- **GDPR**: Provide endpoints to export/delete user data; data retention policy defined.\n\n## 13. Deployment & Release\n- **Mobile**: EAS Build produces `.ipa` and `.apk`; distributed via TestFlight/Google Play Internal Testing; promotion to stores via manual release.\n- **Backend**: Docker image built in CI; pushed to registry (GHCR/ECR); deployed via rolling update on Kubernetes or VM with health checks.\n- **Database**: Managed PostgreSQL (e.g., Supabase, AWS RDS) with automated backups; migrations run via `prisma migrate deploy` in release pipeline.\n- **Redis**: Elasticache or managed Redis; configured for persistence (AOF) and replication.\n- **Feature Flags**: LaunchDarkly or simple env‑based toggles for gradual rollout.\n\n## 14. Monitoring & Observability\n- **Logging**: Structured JSON logs (pino) in NestJS; console in Expo with `expo-dev-client`.\n- **Tracing**: OpenTelemetry instrumentation on NestJS (HTTP, Prisma, Redis) exported to Jaeger/Tempo.\n- **Metrics**: Prometheus endpoint (`/metrics`) exposing request latency, job queue depth, error rates.\n- **Alerting**: Sentry for error alerts; Prometheus alerts on latency > 2s or job failure rate > 5%.\n- **Analytics**: Segment or Amplitude integration in mobile for funnel tracking (search → view → book).\n\n## 15. Future Extensions\n- **Web Portal**: Share `libs/design-system` and `shared-types` with a Next.js or React web app using Expo Web.\n- **Micro‑service Split**: As load grows, split NestJS monolith into separate services (Auth, Booking, Payment) communicating via NATS or gRPC.\n- **Real‑time Updates**: Use Supabase Realtime or WebSocket gateway (NestJS `@nestjs/websockets`) for live slot changes.\n- **AI‑powered Recommendations**: Integrate a recommendation service leveraging user history and ML models.\n- **Offline Support**: Persist query results with MMKV or SQLite and sync when connectivity returns.\n\n---\n*This document captures the core architectural decisions for the Planity Clone. It should be revisited as the product evolves and new requirements emerge.*\n",
