  "content": "# Planity Clone – System Architecture\n\n## 1. Overview\nPlanity Clone is a mobile-first platform connecting customers with local beauty and wellness businesses. The system enables service discovery, real-time appointment booking, payment processing, and provider management. This document defines the technical architecture using a monorepo managed by Nx, with an Expo/React Native mobile app, a NestJS API backend, PostgreSQL/PostGIS, Redis, and Supabase for authentication and storage. The design prioritises clean separation of concerns, scalability, and maintainability.\n\n## 2. Technology Stack\n- **Mobile App:** Expo (SDK 50+), React Native, TypeScript, Expo Router (file-based navigation), TanStack React Query, React Native Reanimated, Stripe React Native, Expo Notifications\n- **Backend API:** NestJS, Prisma ORM, PostgreSQL with PostGIS, Redis, Supabase (Auth + Storage), ioredis, class-validator/class-transformer, @nestjs/throttler\n- **Monorepo & Package Management:** Nx, pnpm workspaces, Turbo (optional)\n- **Infrastructure:** Docker Compose (local dev), GitHub Actions (CI/CD), EAS Build (mobile deployment), Supabase Cloud (database, auth, storage), Redis (managed or container)\n- **Testing:** Jest, React Native Testing Library, Supertest (API e2e)\n\n## 3. High-Level Architecture\nThe system follows a client-server architecture where the mobile app communicates exclusively with the NestJS API via REST over HTTPS. The API acts as a single backend service, consuming internal domain modules, reading/writing to PostgreSQL and Redis, and relying on Supabase for authentication and file storage. A brief component diagram:\n\n```mermaid\ngraph TD\n    A[Expo Mobile App] -->|REST/HTTPS| B[NestJS API]\n    B --> C[(PostgreSQL / PostGIS)]\n    B --> D[(Redis)]\n    B --> E[Supabase Auth]\n    B --> F[Supabase Storage]\n    B --> G[Stripe API]\n    B --> H[External Calendar APIs]\n```\n\nAll business logic resides in the API. The mobile app focuses on presentation, state management, and user interactions. Real-time features (like slot availability) are handled through polling or webhooks; for instant updates Supabase Realtime can be added later.\n\n## 4. E2e tests\n├── Dockerfile\n└── tsconfig.build.json\n```\n\n## 5. Backend Architecture (NestJS)\n\nThe NestJS backend is organized as a modular monolith, each module encapsulating a bounded context. The modules are independent in terms of domain logic, sharing only infrastructure concerns through a CommonModule.\n\n### Module Responsibilities\n- **AuthModule:** Handles registration (via Supabase), login, token verification, role assignment. Uses `SupabaseGuard` that validates JWT tokens with Supabase’s JWKS endpoint, sets `request.user` with claims.\n- **UserModule:** CRUD for user profiles, payment methods, preferences, GDPR account deletion.\n- **BusinessModule:** Manages business entities, working hours, images, amenities. Images uploaded to Supabase Storage; URLs stored in DB.\n- **CategoryModule & ServiceModule:** Taxonomy management and service definitions per business.\n- **StaffModule:** Staff profiles, working schedules (availability windows, breaks).\n- **AvailabilityModule:** Core engine that computes available time slots given a business, service, date, and optional staff. Details in Section 7.\n- **BookingModule:** Manages booking lifecycle (create, confirm, cancel, reschedule). Integrates with PaymentModule and locks slots via Redis.\n- **PaymentModule:** Creates Stripe PaymentIntents, handles webhooks for async payment status updates.\n- **NotificationModule:** Dispatches push notifications (via Expo Push API) and email (via Supabase or email service). Listens to booking events.\n- **ReviewModule:** Handles user reviews, aggregations.\n- **FavoriteModule:** Toggles and lists favorites.\n- **SearchModule:** Full-text search and geospatial queries using PostgreSQL’s `tsvector` and PostGIS functions.\n- **AdminModule:** Dashboard analytics, category CRUD, user management.\n\nAll modules expose REST endpoints following consistent patterns: versioned (`/api/v1/...`), DTO validation, and serialisation via interceptors. Cross-module communication occurs through NestJS’s event emitter (for notification triggers) or direct service injection only when strictly necessary (e.g., Booking calls Availability to check final slot availability).\n\n### Middleware & Guards\n- **Authentication Guard:** Validates JWT, attaches user. Applied globally, with `@Public()` decorator for guest routes.\n- **Roles Guard:** Checks `user.role` against required roles (`customer`, `provider`, `admin`).\n- **Validation Pipe:** Global pipe using `class-validator`.\n- **Throttler Guard:** Rate limiting backed by Redis, configured per endpoint (e.g., booking creation: 5 requests per minute per user).\n- **Logging & Exception Filters:** Standardized error responses and request logging.\n\n## 6. Data Layer\n\n### PostgreSQL + PostGIS\nPrimary database. Prisma ORM manages schema migrations and type-safe queries. For geospatial features (distance calculation, radius search, location-aware queries), PostGIS extension is enabled. Since Prisma does not directly support geometry types, we store latitude and longitude as floats for simple read queries, and use `$queryRaw` with PostGIS functions (`ST_DistanceSphere`, `ST_MakePoint`, `ST_DWithin`) for advanced search and ordering.\n\n#### Core Prisma Schema (simplified)\n```prisma\nmodel User {\n  id            String   @id @default(uuid()) @db.Uuid\n  supabaseId    String   @unique\n  email         String   @unique\n  role          Role\n  profile       Profile?\n  favorites     Favorite[]\n  bookings      Booking[]\n  reviews       Review[]\n}\n\nmodel Business {\n  id            String    @id @default(uuid()) @db.Uuid\n  name          String\n  description   String?\n  latitude      Float\n  longitude     Float\n  address       String\n  phone         String?\n  workingHours  WorkingHour[]\n  categories    BusinessCategory[]\n  services      Service[]\n  staff         Staff[]\n  images        Image[]\n  reviews       Review[]\n  rating        Float     @default(0)\n  reviewCount   Int       @default(0)\n}\n\nmodel Service {\n  id            String    @id @default(uuid()) @db.Uuid\n  businessId    String    @db.Uuid\n  business      Business  @relation(fields: [businessId], references: [id])\n  name          String\n  durationMin   Int\n  price         Decimal\n  bufferMin     Int       @default(0)\n  categoryId    String?   @db.Uuid\n}\n\nmodel Staff {\n  id            String    @id @default(uuid()) @db.Uuid\n  businessId    String    @db.Uuid\n  business      Business  @relation(...)\n  user          User?     @relation(fields: [userId])\n  workingSchedules StaffSchedule[]\n}\n\nmodel Booking {\n  id            String    @id @default(uuid()) @db.Uuid\n  userId        String    @db.Uuid\n  businessId    String    @db.Uuid\n  serviceId     String    @db.Uuid\n  staffId       String?   @db.Uuid\n  startTime     DateTime\n  endTime       DateTime\n  status        BookingStatus\n  lockExpiresAt DateTime?\n}\n```\n\n### Redis\nUsed for:\n- **Temporary slot locks:** Key pattern `lock:slot:{businessId}:{serviceId}:{staffId}:{startTime}` with TTL 10 minutes.\n- **Caching:** API responses for business details, search results, precomputed availability (short TTL) to reduce DB load.\n- **Rate limiting:** Redis-backed `ThrottlerModule` storage.\n- **Session management:** Not needed (JWT), but can store refresh tokens if Supabase is not used for token refresh.\n\n### Supabase\n- **Authentication:** Supabase Auth issues JWT tokens. The mobile app uses `@supabase/supabase-js` for login/register, directly interacting with Supabase. The backend receives the Supabase JWT, verifies it using public key, and extracts user info (sub, email, app_metadata.role). No password storage in our DB.\n- **Storage:** All user/business images (profile, hero) are uploaded to Supabase Storage buckets (e.g., `business-images`, `profile-pics`). Public URLs are saved in the database.\n\n## 7. Real-Time Availability & Slot Computation\n\nThe `AvailabilityEngine` service in the `AvailabilityModule` is the heart of the booking system. It generates available time slots on a given date for a specific service and optionally a staff member, considering:\n1. **Business working hours** (per day of week).\n2. **Staff working schedules** and defined **breaks**.\n3. **Service duration** plus configurable **buffer time** (clean-up).\n4. **Existing bookings** (confirmed) and **temporary locked slots** (Redis).\n5. **Lead time** (e.g., bookings must be made at least 2 hours in advance).\n6. **Slot interval** defined by the business (e.g., 15, 30 minutes).\n\nThe algorithm:\n- Fetch all relevant constraints from DB.\n- Generate candidate start times from opening hour to closing hour minus service duration, stepping by slot interval.\n- Filter out times that fall during breaks, outside working hours.\n- Remove slots that overlap with any existing booking or active lock for any staff (if staff not specified, check all staff – any staff with available slot counts as available).\n- The output is a list of UTC times that are bookable.\n- Results are cached in Redis with a short TTL (1 minute) to speed up repeated requests (e.g., when user flips between dates).\n- When user selects a slot and enters booking flow, a **lock** is placed in Redis (TTL 10 min). If the booking is confirmed within that window, the lock is converted to a confirmed booking; if cancelled or expired, the lock is removed automatically.\n- The lock is designed to be lightweight: key includes business, service, staff, and timestamp, and value is userId. A simple `SET NX` with TTL ensures atomicity.\n\n## 8. Mobile App Architecture\n\nThe mobile app uses **Expo Router** for navigation (file-based routing), dividing screens into the following sections:\n- `(auth)/` – Login, Register, Forgot Password\n- `(tabs)/` – Home, Search, Appointments, Profile tabs\n- `business/[id]/` – Business detail screen\n- `booking/` – Multi-step booking flow (service selection, date/time, confirmation, payment)\n\n**State management:**\n- **Server state:** Entirely handled by **TanStack React Query**. Custom hooks like `useBusinesses`, `useAvailableSlots(date, serviceId, staffId)`, `useUserBookings()` are defined under `queries/`. Queries are cached and automatically refetched on stale or window focus.\n- **Auth state:** Managed via React Context (`AuthProvider`) that initialises supabase client and listens to session changes. Provides `user`, `session`, and helper functions.\n- **UI state:** Local component state or short-lived context for multi-step booking flow.\n\n**Key libraries:**\n- **React Native Reanimated:** Smooth transitions, gesture handling, map animations.\n- **Reanimated Map (react-native-maps):** Custom clustering using supercluster library, responsive markers.\n- **Stripe React Native:** Presents payment sheet for card details, token generation.\n- **Expo Notifications:** Registers for push notifications, handles receiving and routing.\n- **Expo Calendar:** Adds appointments to device calendar.\n\n**API Communication:**\n- A central API client (axios or fetch) is configured with interceptors to attach Supabase JWT from secure storage. On 401 responses, attempts to refresh session using Supabase SDK; if still fails, redirects to login.\n\n## 9. Caching & Performance Strategy\n- **Client-side:** TanStack Query default `staleTime` of 1 minute for dynamic data (business lists, slots), 5 minutes for static (categories). `gcTime` longer to keep previously viewed data. Query keys are structured to allow manual invalidation after mutations (e.g., after booking, invalidate appointments and availability).\n- **Server-side:** Redis cache for heavy queries:\n  - Search results: 5 minutes\n  - Business details: 1 minute (invalidate on business update)\n  - Availability: 1 minute (but real-time locks are always checked against Redis first before confirming booking)\n- **Database:** Indexes on foreign keys and search fields, plus GIN indexes for full-text search (`tsvector`) and GiST indexes for geometry columns in PostGIS.\n\n## 10. Authentication & Authorization Flow\n1. User registers/logs in via Supabase Auth (email/password or OAuth) directly from the mobile app using the Supabase client.\n2. On success, Supabase returns a JWT and refresh token. The mobile app stores the session in secure storage.\n3. Every API call to the NestJS backend includes `Authorization: Bearer <jwt>`.\n4. Backend’s `SupabaseGuard` decodes the token using Supabase’s public key (JWKS endpoint), verifies expiry, and extracts user id, email, role from `app_metadata` (e.g., `{ role: 'customer' }`).\n5. Roles (`customer`, `provider`, `admin`) are enforced by a custom roles decorator/guard.\n6. The guest browsing (unauthenticated) routes are marked with `@Public()` and return limited data.\n\n## 11. API Design (RESTful)\n- Base URL: `/api/v1/`\n- Endpoints follow resource-oriented conventions:\n  - `GET /businesses` – search with query params (keyword, lat, lng, radius, category, page, sort)\n  - `GET /businesses/:id` – detail\n  - `GET /businesses/:id/services`\n  - `GET /businesses/:id/availability?serviceId=&staffId=&date=` – precomputed slots\n  - `POST /bookings` – create a booking (requires lock token)\n  - `POST /bookings/:id/confirm` – confirm after payment\n  - `GET /appointments` – user’s appointments, etc.\n- Responses are wrapped in a standard envelope: `{ success: boolean, data: ... }`.\n- Pagination follows cursor-based or offset-based (configurable).\n- Error format: `{ success: false, error: { code, message } }`.\n\n## 12. Deployment & DevOps\n\n### Local Development\n```yaml\ndocker-compose.yml:\n  services:\n    postgres:\n      image: supabase/postgres:15.1.0.88 (with PostGIS)\n    redis:\n      image: redis:7-alpine\n    api:\n      build: ./apps/api\n      ports: '3000:3000'\n      depends_on: [postgres, redis]\n```\nSupabase local stack can be used if needed, but typically developers connect to a shared dev Supabase project.\n\n### CI/CD Pipeline (GitHub Actions)\n- **Checkout & Setup:** pnpm install, Nx cache.\n- **Lint & Test:** Run linting (eslint), unit tests (Jest), integration tests against test DB.\n- **API Build:** Build NestJS app, create Docker image, push to registry.\n- **Mobile Build:** Using EAS Build, triggered on commits to `main` for QA builds, tags for production. EAS Update can deploy over-the-air updates.\n- **Deployment:** Docker container deployed to cloud provider (AWS ECS/Fargate, Fly.io, etc.). Environment variables managed via secrets.\n\n### Production Architecture\n- API runs as a stateless container behind a load balancer.\n- Supabase cloud provides PostgreSQL (with PostGIS) and Auth service.\n- Redis cache (ElastiCache or Upstash).\n- Stripe for payment processing.\n- Push notifications via Expo Push API.\n\n## 13. Testing Strategy\n- **Unit Tests:** Jest with NestJS testing utilities for services and controllers (mocked dependencies). Mobile components tested with React Native Testing Library.\n- **Integration/E2E Tests:** Supertest for API endpoints against a test database. For critical flows (booking) test optimistic concurrency and lock releasing.\n- **Manual Testing:** Use EAS Build for internal distribution.\n- **Performance Tests:** Load test availability endpoint to ensure slot computation is performant (cached).\n\n## 14. Conclusion\nThis architecture delivers a scalable, maintainable, and modern platform for Planity Clone. The clear separation into Nx monorepo, domain-driven NestJS modules, real-time availability via Redis, and a native-quality Expo mobile app ensures a fast development cycle and high performance.
