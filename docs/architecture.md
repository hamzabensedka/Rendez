  "content": "# Planity Clone Architecture\n\n## 1. Overview\nPlanity Clone is a mobile-first platform connecting customers with beauty and wellness businesses. The system enables service discovery, appointment booking, schedule management, and payments. It serves three primary user roles: Customer, Provider (Business Owner), and Admin. The architecture is designed for scalability, maintainability, and a clear separation of concerns, leveraging a modern TypeScript stack across mobile, backend, and infrastructure.\n\n## 2. High-Level Architecture\n```\n┌──────────────────────┐\n│   Mobile App (Expo)  │\n│ React Native + TS    │\n│ Expo Router          │\n│ TanStack React Query │\n│ Reanimated           │\n└─────────┬────────────┘\n          │ HTTPS\n          ▼\n┌──────────────────────┐      ┌──────────────────────┐\n│   API Server (NestJS)│◄────►│ Background Worker    │\n│   Business Logic     │      │ (BullMQ + Redis)     │\n│   Prisma ORM         │      │ Email, Reminders,    │\n│   Stripe Integration │      │ Slot Computation     │\n└─────────┬────────────┘      └──────────┬───────────┘\n          │                              │\n          ▼                              ▼\n┌─────────────────────────────────────────────────┐\n│              Supabase Platform                  │\n│  ┌──────────────┐ ┌────────────┐ ┌───────────┐ │\n│  │ PostgreSQL   │ │ Auth       │ │ Storage   │ │\n│  │ + PostGIS    │ │ (JWT)      │ │ (images)  │ │\n│  └──────────────┘ └────────────┘ └───────────┘ │\n│  ┌──────────────────────────────┐               │\n│  │       Realtime (WebSockets)  │               │\n│  └──────────────────────────────┘               │\n└─────────────────────────────────────────────────┘\n          ▲\n          │ (optional)\n┌──────────────────────┐\n│  Admin Web App       │\n│  (React + Vite)      │\n└──────────────────────┘\n```\n\n- **Mobile App**: The primary client built with Expo and React Native. It uses Expo Router for navigation, TanStack React Query for server state, and Reanimated for animations. Authentication is handled via Supabase Auth SDK, while all business data flows through the NestJS API.\n- **API Server**: A NestJS application that encapsulates all business logic (booking, payments, search, etc.). It uses Prisma to interact with the Supabase-managed PostgreSQL database and integrates Stripe for payments.\n- **Background Worker**: A separate process using BullMQ and Redis to handle asynchronous tasks such as sending emails, computing time slots, processing reminders, and cleaning up expired tokens.\n- **Supabase**: Provides managed PostgreSQL with PostGIS, authentication (with JWT issuance and verification), file storage for business photos, and real-time capabilities via WebSockets. The API server connects to the same database, ensuring a single source of truth.\n- **Admin Web App** (optional): A lightweight React application for platform administrators, consuming the same API.\n\n## 3. Monorepo Structure (Nx + pnpm)\nThe project is organized as an Nx monorepo with pnpm workspaces. Shared code lives in `libs/`, while deployable units reside in `apps/`.\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo app\n│   │   ├── app/                   # Expo Router screens\n│   │   ├── components/            # UI components\n│   │   ├── hooks/                 # Custom hooks (React Query)\n│   │   ├── services/              # API client, Supabase client\n│   │   ├── utils/\n│   │   ├── app.json\n│   │   └── eas.json\n│   ├── api/                       # NestJS backend\n│   │   ├── src/\n│   │   │   ├── modules/\n│   │   │   │   ├── auth/\n│   │   │   │   ├── businesses/\n│   │   │   │   ├── bookings/\n│   │   │   │   ├── categories/\n│   │   │   │   ├── payments/\n│   │   │   │   ├── reviews/\n│   │   │   │   ├── search/\n│   │   │   │   └── users/\n│   │   │   ├── common/            # Guards, decorators, filters\n│   │   │   ├── prisma/            # Prisma service, schema\n│   │   │   ├── config/            # Environment configuration\n│   │   │   └── main.ts\n│   │   ├── prisma/\n│   │   │   ├── schema.prisma\n│   │   │   └── migrations/\n│   │   ├── test/\n│   │   └── Dockerfile\n│   ├── worker/                    # Background job processor\n│   │   ├── src/\n│   │   │   ├── processors/        # Job handlers\n│   │   │   ├── queues/            # Queue definitions\n│   │   │   └── main.ts\n│   │   └── Dockerfile\n│   └── admin-web/                 # Optional admin dashboard (React)\n│       ├── src/\n│       └── vite.config.ts\n├── libs/\n│   ├── shared-types/              # TypeScript interfaces, DTOs\n│   ├── ui/                        # Shared UI components (if any)\n│   └── utils/                     # Common utilities\n├── tools/                         # Nx custom generators, scripts\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n├── package.json\n└── .github/\n    └── workflows/\n        ├── ci.yml\n        └── mobile-build.yml\n```\n\n## 5. Service Boundaries & Responsibilities\n\n### 5.1 Mobile App (`apps/mobile`)\n- **Responsibilities**: UI rendering, navigation, local state, optimistic updates, push notification handling, deep linking.\n- **Data Access**: Uses Supabase client for authentication and real-time subscriptions. All business data is fetched/mutated via the NestJS API using a typed API client generated from shared types.\n- **Key Libraries**: Expo Router for routing, TanStack React Query for server state (caching, pagination, background refetch), Reanimated for animations.\n\n### 5.2 API Server (`apps/api`)\n- **Responsibilities**: Core business logic, data validation, authorization, payment orchestration, search, slot computation coordination.\n- **Data Ownership**: Owns the Prisma schema and all database mutations. It is the only service allowed to write to the database (except Supabase Auth tables managed by Supabase).\n- **Integration**: Validates Supabase JWTs, processes Stripe payments, publishes events to Redis for the worker.\n- **API Design**: RESTful endpoints versioned under `/api/v1/`. Uses class-validator and class-transformer for DTOs.\n\n### 5.3 Background Worker (`apps/worker`)\n- **Responsibilities**: Asynchronous, non-critical tasks that can be delayed.\n  - Send transactional emails (verification, password reset, booking confirmation, reminders).\n  - Compute and cache available time slots after any booking change.\n  - Process refunds for cancellations.\n  - Clean up expired tokens and stale data.\n- **Communication**: Listens to BullMQ queues populated by the API server. Uses the same Prisma service to read/write data when necessary.\n\n### 5.4 Admin Web App (`apps/admin-web`)\n- **Responsibilities**: Dashboard for platform administrators to manage categories, moderate reviews, view analytics, and handle support.\n- **Data Access**: Consumes the same API with admin-scoped endpoints.\n\n## 6. Data Flow Examples\n\n### 6.1 Booking Flow\n1. Customer selects services, staff, date/time in the mobile app.\n2. Mobile calls `POST /api/v1/bookings` with selected details.\n3. API server validates the request, checks real-time slot availability (via a cached Redis set or a direct query), and creates a pending booking with a Stripe PaymentIntent.\n4. API returns the client secret for the PaymentIntent.\n5. Mobile confirms the payment using Stripe SDK.\n6. On payment success, mobile calls `POST /api/v1/bookings/:id/confirm`.\n7. API updates booking status to `confirmed`, publishes a job to the `slot-recompute` queue, and returns success.\n8. Worker recomputes available slots for the affected staff/business and updates a `slot_availability` table (which has Supabase Realtime enabled).\n9. All clients subscribed to that business receive the updated slots via WebSocket.\n\n### 6.2 Search Flow\n1. User enters a search query or applies filters.\n2. Mobile calls `GET /api/v1/search?q=hair&lat=...&lng=...&radius=10`.\n3. API uses PostgreSQL full-text search (with `tsvector` and `tsquery`) combined with PostGIS `ST_DWithin` for location filtering.\n4. Results are cached in Redis with a TTL of 5 minutes to reduce database load.\n5. Mobile displays results using React Query’s `useInfiniteQuery` for pagination.\n\n## 7. Database Design\n\n### 7.1 Core Tables (Prisma Schema)\n```prisma\n// Essential models (simplified)\nmodel User {\n  id        String   @id @default(uuid())\n  supabaseId String  @unique // Supabase Auth UID\n  email     String   @unique\n  role      Role     // CUSTOMER, PROVIDER, ADMIN\n  profile   Profile?\n  businesses Business[] // if PROVIDER\n  appointments Appointment[]\n  reviews   Review[]\n  favorites Favorite[]\n}\n\nmodel Business {\n  id          String   @id @default(uuid())\n  ownerId     String\n  owner       User     @relation(fields: [ownerId], references: [id])\n  name        String\n  description String?\n  address     String\n  location    Unsupported(\"geometry(Point, 4326)\")? // PostGIS\n  categoryId  String?\n  category    Category? @relation(fields: [categoryId], references: [id])\n  services    Service[]\n  staff       Staff[]\n  reviews     Review[]\n  favorites   Favorite[]\n  openingHours OpeningHours[]\n  photos      Photo[]\n}\n\nmodel Service {\n  id          String   @id @default(uuid())\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  name        String\n  duration    Int      // minutes\n  price       Decimal\n  categoryId  String?\n  category    Category? @relation(fields: [categoryId], references: [id])\n}\n\nmodel Staff {\n  id          String   @id @default(uuid())\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  name        String\n  services    Service[] @relation(\"StaffServices\")\n  schedule    Schedule[]\n  appointments Appointment[]\n}\n\nmodel Appointment {\n  id          String   @id @default(uuid())\n  customerId  String\n  customer    User     @relation(fields: [customerId], references: [id])\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  staffId     String?\n  staff       Staff?   @relation(fields: [staffId], references: [id])\n  date        DateTime\n  startTime   DateTime\n  endTime     DateTime\n  status      AppointmentStatus // PENDING, CONFIRMED, CANCELLED, COMPLETED\n  services    Service[] @relation(\"AppointmentServices\")\n  totalPrice  Decimal\n  notes       String?\n  review      Review?\n}\n\nmodel SlotAvailability {\n  id          String   @id @default(uuid())\n  businessId  String\n  staffId     String?\n  date        DateTime\n  slots       Json     // array of { start, end }\n  updatedAt   DateTime @updatedAt\n}\n\n// Additional models: Category, Review, Favorite, OpeningHours, Photo, Schedule, etc.\n```\n- **PostGIS**: The `location` column on `Business` uses `geometry(Point, 4326)` for geospatial queries.\n- **Full-Text Search**: A generated `tsvector` column on `Business` (combining name, description, services) with a GIN index for fast search.\n- **Indexes**: B-tree indexes on foreign keys, `Appointment.date`, `Business.categoryId`, and GIST index on `location`.\n\n### 7.2 Supabase Managed Tables\nSupabase Auth manages its own schemas (`auth.users`, `auth.refresh_tokens`, etc.). Our `User` table links to Supabase via `supabaseId`.\n\n## 8. Caching Strategy (Redis)\n- **Slot Availability**: Precomputed slots for the next 30 days are stored as JSON per business/staff/date. Updated by the worker after any booking change. API reads from Redis first, falling back to computation if missing.\n- **Search Results**: Cache search queries (keyed by query string + filters + location) for 5 minutes.\n- **Rate Limiting**: Use Redis to track request counts per user/IP for sensitive endpoints (login, booking).\n- **Session Blacklist**: On logout or token revocation, add JWT `jti` to a Redis set with TTL equal to token expiry.\n- **Job Queue**: BullMQ uses Redis as its broker.\n\n## 9. Authentication & Authorization\n- **Supabase Auth**: Handles sign-up, email verification, password reset, and OAuth (Google, Apple). Issues JWTs signed with a project-specific secret.\n- **Mobile**: Uses `@supabase/supabase-js` for authentication flows. On login, the session (access token + refresh token) is stored securely via Expo SecureStore.\n- **API Server**: Validates the JWT on every request using a NestJS guard that fetches Supabase’s JWKS and verifies the token. Extracts `sub` (Supabase user ID) and role from the token. The `supabaseId` is used to look up our internal `User` record.\n- **Role-Based Access**: Custom decorators (`@Roles(Role.CUSTOMER)`) and guards restrict endpoints. Admin endpoints require `ADMIN` role.\n- **Refresh Token Rotation**: Managed automatically by Supabase. The mobile client refreshes tokens silently.\n\n## 10. API Design\n- **Base URL**: `/api/v1/`\n- **Resource Endpoints**:\n  - `POST /auth/register` – (optional, if not using Supabase directly; otherwise mobile uses Supabase)\n  - `GET /businesses` – list with filters, pagination\n  - `GET /businesses/:id` – detail\n  - `GET /businesses/:id/services`\n  - `GET /businesses/:id/reviews`\n  - `POST /businesses/:id/favorite` – toggle favorite\n  - `GET /search` – full-text + geospatial search\n  - `POST /bookings` – create booking (returns PaymentIntent client secret)\n  - `POST /bookings/:id/confirm` – confirm after payment\n  - `GET /bookings` – user’s appointments\n  - `PATCH /bookings/:id/cancel`\n  - `POST /bookings/:id/reschedule`\n  - `GET /slots?businessId=&staffId=&date=` – get available slots\n  - `POST /reviews` – submit review\n  - Admin endpoints under `/admin/...`\n- **Request/Response**: JSON. Use DTOs with class-validator. Pagination follows cursor-based or offset pattern.\n- **Versioning**: URI versioning (`/api/v1/`). Future versions can coexist.\n\n## 11. Real-Time Slot Updates\n- When a booking is confirmed or cancelled, the worker recomputes slots and upserts a row in the `SlotAvailability` table.\n- Supabase Realtime is enabled on `SlotAvailability` with row-level security allowing read for authenticated users.\n- Mobile subscribes to changes on `SlotAvailability` filtered by `businessId`. On update, React Query’s cache is invalidated for the relevant slot queries, triggering a UI refresh.\n- This ensures that multiple users see up-to-date availability without polling.\n\n## 12. They share the same Prisma client and Redis connection.\n\n## 13. Deployment & CI/CD\n\n### 13.1 Local Development\n- `docker-compose.yml` defines:\n  - `postgres` (or use Supabase CLI for local Supabase stack)\n  - `redis`\n  - `api` (NestJS in dev mode with hot reload)\n  - `worker`\n- Mobile app runs via `npx expo start`.\n\n### 13.2 CI/CD Pipeline (GitHub Actions)\n- **Pull Request**: Lint, type-check, unit tests (Jest) across all projects, API integration tests with a test database.\n- **Main Branch**: Build and push Docker images for API and worker to a container registry. Testing Strategy\n- **Unit Tests**: Jest with mocks for external dependencies. Each lib/app has its own `jest.config.ts`.\n- **API Integration Tests**: Use `supertest` with a dedicated test database (Dockerized PostgreSQL). Prisma migrations run before tests. Seed data for consistent scenarios.\n- **Mobile Tests**: Component tests with React Native Testing Library. Hook tests for React Query configurations.\n- **E2E** (future): Detox for critical user flows (booking, login).\n- **CI**: All tests run on every PR. Coverage thresholds enforced for core business logic.\n\n## 15. Key Design Decisions\n- **Supabase as the Foundation**: Using Supabase for auth, database, and real-time reduces infrastructure overhead and provides a unified developer experience. The NestJS API acts as a backend-for-frontend (BFF) that adds business logic without duplicating auth concerns.\n- **Prisma over raw SQL**: Type-safe queries and migrations accelerate development. PostGIS and full-text search are supported via raw SQL functions where needed.\n- **Separate Worker Process**: Keeps the API responsive by offloading heavy or time-sensitive tasks. BullMQ on Redis provides reliability with retries and monitoring.\n- **TanStack React Query**: Handles caching, background updates, and optimistic mutations, reducing boilerplate and improving perceived performance.\n- **Monorepo with Nx**: Enforces module boundaries, enables code sharing (types, utilities), and provides efficient build/test caching.\n\nThis architecture balances rapid development with production readiness, ensuring the Planity Clone can scale from MVP to a full-featured platform.",
