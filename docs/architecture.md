  "content": "# Planity Clone - System Architecture Document\n\n## 1. Overview\n\nThis document describes the complete system architecture for the Planity Clone mobile application. The platform connects customers with beauty/wellness service providers for appointment booking. It serves three user types: customers, business owners, and administrators.\n\nThe architecture follows clean architecture principles, with clear separation of concerns, domain-driven design, and a monorepo structure using Nx for code organization and build optimization.\n\n## 2. Technology Stack\n\n### Frontend (Mobile)\n- **Framework:** Expo (managed workflow) with React Native\n- **Language:** TypeScript (strict mode)\n- **Routing:** Expo Router (file-based routing)\n- **State Management:** TanStack React Query (server state), Zustand (client state)\n- **Animations:** React Native Reanimated 3\n- **Maps:** react-native-maps with Mapbox\n- **Push Notifications:** Expo Notifications + Firebase Cloud Messaging\n- **Biometrics:** expo-local-authentication\n- **Calendar:** expo-calendar\n- **Testing:** Jest, React Native Testing Library\n\n### Backend (API)\n- **Framework:** NestJS (modular monolith with eventual microservice extraction)\n- **Language:** TypeScript\n- **ORM:** Prisma (with PostgreSQL and PostGIS extensions)\n- **Cache:** Redis (session store, rate limiting, query caching)\n- **Authentication:** JWT (access + refresh tokens), OAuth 2.0 with PKCE\n- **Real-time:** WebSockets (Socket.IO) for booking updates\n- **File Storage:** Supabase Storage (S3-compatible)\n- **Background Jobs:** Bull (Redis-backed queue)\n- **Testing:** Jest, Supertest\n\n### Infrastructure\n- **Monorepo:** Nx (with pnpm workspaces)\n- **Containerization:** Docker Compose (local development)\n- **CI/CD:** GitHub Actions\n- **Hosting:** EAS Build (mobile), Vercel/Railway (API)\n- **Database:** PostgreSQL 15 with PostGIS\n- **Search:** PostgreSQL full-text search + Redis search (for autocomplete)\n- **Monitoring:** Sentry (error tracking), Datadog (APM)\n\n## 3. Service Boundaries & Modules\n\n### 4.1 Authentication Module\n- **Responsibility:** User registration, login, token management, OAuth, biometrics\n- **Key Entities:** User, RefreshToken, Session\n- **Endpoints:** POST /auth/register, POST /auth/login, POST /auth/refresh, POST /auth/logout, POST /auth/forgot-password, POST /auth/reset-password\n- **Rate Limiting:** 5 req/min per IP on auth endpoints\n- **Security:** JWT with RS256, refresh token rotation, account lockout after 5 failed attempts\n\n### 4.2 User Module\n- **Responsibility:** Profile management, notification preferences, account deletion\n- **Key Entities:** User, UserProfile, NotificationPreference\n- **Endpoints:** GET /users/me, PATCH /users/me, DELETE /users/me, POST /users/me/export-data\n\n### 4.3 Business Module\n- **Responsibility:** Business CRUD, search, geolocation, map clustering\n- **Key Entities:** Business, BusinessHour, BusinessPhoto, Address (with PostGIS geometry)\n- **Endpoints:** GET /businesses (search with filters), GET /businesses/:id, POST /businesses (owner), PATCH /businesses/:id, GET /businesses/:id/availability\n- **Search:** Full-text search on name, description, service names; PostGIS for distance queries\n\n### 4.4 Service Module\n- **Responsibility:** Service catalog, pricing, durations, categories\n- **Key Entities:** Service, Category, Subcategory, ServiceStaff\n- **Endpoints:** GET /services, GET /services/:id, GET /categories, GET /categories/:id/services\n\n### 4.5 Staff Module\n- **Responsibility:** Staff profiles, schedules, ratings\n- **Key Entities:** Staff, StaffSchedule, StaffService\n- **Endpoints:** GET /staff, GET /staff/:id, GET /staff/:id/availability\n\n### 4.6 Booking Module\n- **Responsibility:** Appointment creation, modification, cancellation, check-in, no-show\n- **Key Entities:** Booking, BookingService, BookingStatus, CancellationPolicy\n- **Endpoints:** POST /bookings, GET /bookings, GET /bookings/:id, PATCH /bookings/:id, DELETE /bookings/:id, POST /bookings/:id/check-in\n- **Real-time:** WebSocket events for booking confirmation, updates\n- **Background Jobs:** Send reminders (Bull queue), process cancellations\n\n### 4.7 Reviews Module\n- **Responsibility:** Customer reviews, ratings, moderation\n- **Key Entities:** Review, ReviewImage, ReviewReport\n- **Endpoints:** GET /businesses/:id/reviews, POST /reviews, PATCH /reviews/:id, POST /reviews/:id/report\n\n### 4.8 Favorites Module\n- **Responsibility:** Save and manage favorite businesses\n- **Key Entities:** Favorite\n- **Endpoints:** GET /favorites, POST /favorites, DELETE /favorites/:id\n\n### 4.9 Notifications Module\n- **Responsibility:** Push, SMS, email notifications\n- **Key Entities:** Notification, NotificationTemplate, DeviceToken\n- **Endpoints:** POST /notifications/register-device, PATCH /notifications/preferences\n- **Integration:** Firebase Cloud Messaging (push), Twilio (SMS), SendGrid (email)\n\n### 4.10 Payments Module\n- **Responsibility:** Payment processing, refunds, pay-at-venue option\n- **Key Entities:** Payment, PaymentMethod, Transaction, Refund\n- **Endpoints:** POST /payments/charge, POST /payments/refund, GET /payments/history\n- **Integration:** Stripe (primary), Apple Pay, Google Pay\n\n### 4.11 Admin Module\n- **Responsibility:** User management, business verification, content moderation, analytics\n- **Key Entities:** Admin, AuditLog, Report\n- **Endpoints:** GET /admin/users, PATCH /admin/businesses/:id/verify, GET /admin/reports, GET /admin/analytics\n\n## 5. Data Flow & Communication\n\n### 5.1 API Communication\n- **Mobile to API:** REST over HTTPS (JSON), WebSocket for real-time updates\n- **Authentication:** Bearer token in Authorization header\n- **Error Handling:** Consistent error response format with code, message, and details\n- **Pagination:** Cursor-based pagination for lists, offset-based for admin\n\n### 5.2 Caching Strategy\n- **Redis Cache:**\n  - Session store (refresh tokens, OAuth state)\n  - Rate limiting counters\n  - Query results (business search, categories) with TTL of 5 minutes\n  - Autocomplete suggestions (TTL 1 hour)\n- **Client Cache (TanStack React Query):**\n  - Stale time: 5 minutes for list data, 30 minutes for static data (categories)\n  - Cache invalidation on mutation (e.g., after booking, refetch bookings list)\n  - Optimistic updates for favorites, reviews\n\n### 5.3 Real-time Updates\n- **WebSocket Channels:**\n  - Booking status changes (confirmed, cancelled, modified)\n  - Availability updates (time slot taken by another user)\n  - Admin notifications (new business registration, reports)\n- **Socket.IO with Redis adapter** for horizontal scaling\n\n### 5.4 Background Jobs (Bull)\n- **Queues:**\n  - booking-reminders: Send push/SMS/email reminders at 24h, 2h, 15min before appointment\n  - booking-cleanup: Mark no-shows after 15min grace period\n  - email-verification: Send verification emails\n  - data-export: Generate user data export (async)\n  - analytics: Aggregate daily metrics\n\n## 6. Database Design (Prisma Schema Highlights)\n\n### 6.1 Core Entities\n- **User:** id, email, phone, passwordHash, role (CUSTOMER, OWNER, ADMIN), verified, lockedUntil, createdAt, deletedAt (soft delete)\n- **Business:** id, ownerId, name, description, address (PostGIS geography point), phone, website, verified, rating, totalReviews, createdAt\n- **Service:** id, businessId, categoryId, name, description, price, duration, active, createdAt\n- **Staff:** id, businessId, name, bio, photo, rating, active, createdAt\n- **Booking:** id, userId, businessId, staffId, startTime, endTime, status (PENDING, CONFIRMED, CANCELLED, COMPLETED, NO_SHOW), totalPrice, cancellationReason, createdAt\n- **Review:** id, userId, businessId, bookingId, rating, comment, createdAt\n- **Favorite:** id, userId, businessId, createdAt (unique constraint on userId + businessId)\n\n### 6.2 PostGIS Usage\n- **Business location:** geography(Point, 4326) for accurate distance calculations\n- **Spatial queries:** ST_DWithin for radius search, ST_Distance for sorting\n- **Index:** GIST index on location column\n\n### 6.3 Indexes\n- Business name, service name (GIN for full-text search)\n- Booking startTime + businessId (for availability queries)\n- User email (unique)\n- User phone (unique)\n- Review businessId + rating\n- Favorite userId + businessId (unique composite)\n\n## 7. Security Architecture\n\n### 7.1 Authentication & Authorization\n- **JWT Access Token:** 15-minute expiry, signed with RS256\n- **Refresh Token:** 30-day expiry, stored in Redis with rotation\n- **OAuth 2.0 with PKCE:** For Google and Apple sign-in\n- **Role-based access control (RBAC):** Guards on NestJS routes (CUSTOMER, OWNER, ADMIN)\n- **API Key:** For internal service-to-service communication\n\n### 7.2 Data Protection\n- **Encryption at rest:** Database encryption (TDE), file storage encryption (AES-256)\n- **Encryption in transit:** TLS 1.3 for all API and WebSocket connections\n- **Password hashing:** bcrypt with cost factor 12\n- **PII masking:** Logs never contain raw passwords, full credit card numbers\n\n### 7.3 Rate Limiting\n- **Auth endpoints:** 5 req/min per IP\n- **Search endpoints:** 30 req/min per user\n- **Booking creation:** 10 req/min per user\n- **General API:** 100 req/min per user\n\n## 8. Performance & Scalability\n\n### 8.1 Mobile Performance\n- **Image optimization:** Lazy loading with progressive JPEG, WebP format\n- **List virtualization:** FlashList for long lists (search results, bookings)\n- **Animations:** Reanimated worklets on UI thread for smooth 60fps\n- **Offline support:** AsyncStorage for cache, NetInfo for connectivity detection\n\n### 8.2 Backend Performance\n- **Connection pooling:** Prisma with pgBouncer for PostgreSQL\n- **Caching:** Redis for frequent queries, session data\n- **Database indexing:** Proper indexes on frequently queried columns\n- **Query optimization:** N+1 prevention with Prisma include/select, batch loading\n\n### 8.3 Horizontal Scaling\n- **API servers:** Stateless NestJS instances behind load balancer\n- **WebSocket:** Redis adapter for sticky sessions\n- **Database:** Read replicas for search queries, write master for mutations\n- **Background jobs:** Bull queue with Redis, worker processes can scale independently\n\n## 9. Testing Strategy\n\n### 9.1 Unit Tests (Jest)\n- **Mobile:** Test hooks, services, stores, utility functions\n- **API:** Test services, guards, pipes, interceptors\n- **Coverage target:** 80%+ for business logic\n\n### 9.2 Integration Tests\n- **API:** Supertest for endpoint testing with test database (PostgreSQL + Redis)\n- **Mobile:** React Native Testing Library for component integration\n\n### 9.3 End-to-End Tests\n- **Mobile:** Detox for critical flows (login, search, booking)\n- **API:** Postman/Newman collections for API contract testing\n\n### 9.4 CI/CD Pipeline (GitHub Actions)\n- **On PR:** Lint, type-check, unit tests, integration tests\n- **On merge to main:** Build, e2e tests, deploy to staging\n- **On release tag:** Deploy to production with blue/green deployment\n\n## 10. Deployment & DevOps\n\n### 10.1 Local Development\n- **Docker Compose:** PostgreSQL, Redis, API server with hot reload\n- **Expo Go:** Mobile app development with tunnel\n- **Environment variables:** .env files with examples\n\n### 10.2 Production Deployment\n- **Mobile:** EAS Build for iOS and Android, distributed via App Store Connect and Google Play Console\n- **API:** Containerized NestJS app deployed on Railway/Vercel with auto-scaling\n- **Database:** Managed PostgreSQL (Supabase or AWS RDS) with automated backups\n- **Cache:** Managed Redis (Upstash or AWS ElastiCache)\n- **File Storage:** Supabase Storage (S3-compatible) with CDN\n\n### 10.3 Monitoring & Alerting\n- **Error tracking:** Sentry for mobile and backend\n- **APM:** Datadog for API performance monitoring\n- **Logging:** Structured JSON logs, aggregated in Datadog\n- **Uptime monitoring:** Pingdom or Better Uptime\n\n## 11. Conclusion\n\nThis architecture provides a scalable, maintainable foundation for the Planity Clone platform. The modular design allows for independent development and testing of features, while the monorepo structure with Nx ensures code sharing and build efficiency.
