Service Boundaries\n\n### 6.1 Mobile Client (apps/mobile)\n\n- **Responsibility**: User interface, navigation, local state, offline caching, API calls.\n- **Key Libraries**: Expo, React Navigation (Expo Router), TanStack React Query, React Native Reanimated.\n- **Data Flow**: Components use TanStack Query hooks to fetch data from API. Mutations update cache optimistically. Offline data is stored via AsyncStorage or SQLite.\n\n### 6.2 API Gateway (apps/api)\n\n- **Responsibility**: Authentication, routing, rate limiting, request validation.\n- **Key Libraries**: NestJS, Passport, JWT, Redis.\n- **Modules**:\n  - `AuthModule`: Handles login, registration, token refresh, OAuth.\n  - `UsersModule`: User profile management.\n  - `BusinessesModule`: Business CRUD, operating hours, staff management.\n  - `ServicesModule`: Service categories and offerings.\n  - `BookingsModule`: Booking creation, cancellation, availability computation.\n  - `SearchModule`: Full-text search, geospatial queries, filtering.\n  - `ReviewsModule`: Review creation, moderation, aggregation.\n  - `NotificationsModule`: Push notification scheduling (via Firebase Cloud Messaging).\n  - `AdminModule`: Admin dashboard endpoints, user management.\n\n### 6.3 Database Layer\n\n- **PostgreSQL + PostGIS**: Stores all relational data including users, businesses, services, bookings, reviews.\n- **Redis**: Caches session tokens, rate limit counters, popular search results, temporary booking locks.\n- **Prisma**: ORM for type-safe database access and migrations.\n\n## 7. Data Flow\n\n### 7.1 Booking Flow\n\n1. Customer opens app, browses businesses (search results cached in Redis).\n2. Customer selects a business, views details (fetched from API, cached client-side).\n3. Customer selects service and time slot (API computes availability using PostGIS for location and business hours).\n4. Customer confirms booking (API creates booking record, sends notification to provider via Redis pub/sub or push).\n5. Booking confirmation stored in PostgreSQL; TanStack Query cache updated.\n\n### 7.2 Authentication Flow\n\n1. User enters phone/email.\n2. API sends OTP (SMS via Twilio) or magic link (email via SendGrid).\n3. User verifies OTP/clicks link.\n4. API issues JWT access token (15 min) and refresh token (7 days, stored in Redis).\n5. Mobile client stores tokens securely (expo-secure-store).\n6. TanStack Query interceptors attach token to requests; refresh logic on 401.\n\n## 8. Key Design Decisions\n\n### 8.1 Monorepo with Nx\n\n- **Why**: Single source of truth for shared types, utilities, and configurations. Nx provides efficient build caching and dependency graph management.\n- **Workspace**: pnpm workspaces for package management.\n\n### 8.2 TanStack React Query\n\n- **Why**: Declarative data fetching, caching, and synchronization. Automatic background refetching, optimistic updates for booking mutations, and offline support.\n- **Usage**: All API calls go through query hooks. Mutations invalidate related queries.\n\n### 8.3 Expo Router\n\n- **Why**: File-based routing simplifies navigation structure. Supports deep linking and authentication guards.\n- **Layouts**: Group routes for auth, tabs, and modal flows.\n\n### 8.4 Prisma with PostgreSQL + PostGIS\n\n- **Why**: Type-safe queries, auto-generated migrations, and native geospatial support for distance-based search.\n- **Schema**: Includes spatial indexes on business locations.\n\n### 8.5 Redis for Session and Cache\n\n- **Why**: Fast in-memory storage for session tokens, rate limiting, and temporary booking locks (prevents double booking).\n- **TTL**: Refresh tokens (7 days), search cache (1 hour), rate limit counters (sliding window).\n\n## 9. API Endpoints (High-Level)\n\n| Method | Endpoint | Description |\n|--------|----------|-------------|\n| POST | /auth/register | Register new user |\n| POST | /auth/login | Login with email/phone |\n| POST | /auth/refresh | Refresh access token |\n| GET | /businesses | List businesses (search) |\n| GET | /businesses/:id | Business detail |\n| GET | /businesses/:id/availability | Available slots |\n| POST | /bookings | Create booking |\n| GET | /bookings | User's bookings |\n| DELETE | /bookings/:id | Cancel booking |\n| POST | /reviews | Create review |\n| GET | /categories | List service categories |\n\n## 10. Security Considerations\n\n- **Authentication**: JWT with short-lived access tokens, refresh tokens stored in Redis.\n- **Authorization**: Role-based guards (customer, provider, staff, admin) at controller level.\n- **Rate Limiting**: Per-user and per-IP limits using Redis (e.g., 100 requests/min for search).\n- **Input Validation**: Class-validator DTOs for all endpoints.\n- **SQL Injection**: Prevented by Prisma parameterized queries.\n- **HTTPS**: Enforced in production.\n- **Secrets Management**: Environment variables via .env files (not committed) or secret manager.\n\n## 11. Testing Strategy\n\n- **Unit Tests**: Jest for individual services and utilities.\n- **Integration Tests**: Test API endpoints with test database (Docker Compose).\n- **E2E Tests**: Detox for mobile app flows.\n- **CI Pipeline**: GitHub Actions runs lint, type check, unit tests, integration tests on every PR.\n\n## 12. Deployment\n\n- **Mobile**: EAS Build for iOS and Android builds. OTA updates via EAS Update.\n- **Backend**: Docker Compose for local development; production deployment via Kubernetes or cloud provider (AWS ECS/GCP Cloud Run).\n- **Database**: Managed PostgreSQL (Supabase or AWS RDS) with automated backups.\n- **Redis**: Managed Redis (Upstash or AWS ElastiCache).\n\n## 13. Monitoring & Observability\n\n- **Logging**: Structured logging via NestJS Logger and Winston.\n- **Metrics**: Prometheus metrics for API performance.\n- **Error Tracking**: Sentry for both mobile and backend.\n- **Analytics**: Mixpanel or Amplitude for user behavior.\n\n## 14. Conclusion\n\nThis architecture provides a scalable, maintainable foundation for the Planity Clone. The separation of concerns between mobile client, API gateway, and database layers allows independent development and deployment.
