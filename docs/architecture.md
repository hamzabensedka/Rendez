Overview & Design Principles\n\nPlanity Clone is a mobile-first beauty & wellness booking platform. The system consists of a cross-platform client (iOS, Android, Web) built with Expo, a NestJS backend API, a PostgreSQL (+ PostGIS) database, Redis caching, and Supabase for authentication and real-time data. A background job engine (BullMQ) handles asynchronous tasks like slot computation, reminders, and data cleanup. The architecture prioritises **separation of concerns**, **high cohesion within bounded contexts**, **offline-first where feasible**, and **scalable real-time updates**.\n\n**Key Design Principles:**\n- **Clean Architecture** across all layers: the mobile app uses a feature-based structure with shared UI components; the backend is structured as a modular monolith with clearly defined service boundaries.\n- **Type Safety** end-to-end: shared TypeScript types and Zod validation schemas between backend and mobile.\n- **Resilience**: TanStack React Query for server state, aggressive caching, and optimistic updates on the client; rate limiting, circuit breakers, and slot locking on the server.\n- **Real-time First**: Appointment status changes and availability updates use Supabase Realtime, reducing unnecessary polling.\n- **Developer Experience**: Nx monorepo with pnpm workspaces, strict linting, shared libs, and comprehensive testing with Jest.\n\n## 2. System Diagram\n\n```\n┌────────────────────┐        ┌────────────────────┐\n│  Mobile / Web App  │───────▶│  NestJS API        │\n│  (Expo, React      │        │  (REST + WebSocket)│\n│   Native, Reanimated)       └─────────┬──────────┘\n└───┬────────────────┘                  │\n    │ (Supabase Client)                │ (Prisma)\n    │                                   │\n    ▼                                   ▼\n┌────────────────────┐        ┌────────────────────┐\n│   Supabase         │        │  PostgreSQL +      │\n│   (Auth, Realtime) │        │  PostGIS           │\n└────────────────────┘        └─────────┬──────────┘\n                                        │\n                              ┌─────────▼──────────┐\n                              │    Redis            │\n                              │  (Cache, Queue,     │\n                              │   Lock)             │\n                              └────────┬───────────┘\n                                       │\n                              ┌────────▼───────────┐\n                              │  BullMQ Workers     │\n                              │  (Jobs: slots,      │\n                              │   reminders, etc.)  │\n                              └────────────────────┘\n```\n\n- **Mobile App** communicates with NestJS API for all business logic (booking, search, profile management).\n- **Authentication** is handled by Supabase; the mobile app obtains a JWT from Supabase and attaches it to API requests.\n- **Real-time Updates**: The mobile app subscribes directly to Supabase Realtime channels for appointment status changes and favourite notifications, respecting RLS policies.\n- **Background Jobs**: The NestJS API enqueues jobs (slot recomputation, booking expiry, reminders) to BullMQ. Workers process them, accessing the same database and Redis.\n\n## 3. Mobile App Architecture\n\nThe mobile app follows a feature-driven approach. Expo Router provides file-based navigation; each route maps to a screen under `app/`. TanStack React Query handles all server-state fetching, caching, and mutations with automatic refetch on window focus. React Native Reanimated powers smooth transitions and animated booking flows. Zustand stores transient UI state (e.g., map viewport, guest session).\n\n### 5.1 Folder Structure\n\n```\nmobile/\n├── app/\n│   ├── (auth)/                  # Auth screens (sign-in, sign-up, forgot-password)\n│   ├── (tabs)/                  # Main tabs (Home, Search, Bookings, Profile)\n│   │   ├── index.tsx            # Home feed\n│   │   ├── search.tsx           # Search screen\n│   │   ├── bookings.tsx         # Appointments list\n│   │   └── profile.tsx          # User profile\n│   ├── (modals)/                # Modal screens (booking review, confirm)\n│   ├── business/[id].tsx        # Business detail page\n│   └── _layout.tsx              # Root layout\n├── components/\n│   ├── ui/                      # Primitive components (Button, TextField, Card, …)\n│   ├── booking/                 # Booking steps, slot picker, service selector\n│   ├── business/               # Business card, carousel, review list\n│   ├── search/                 # Search bar, filters, results list, map toggle\n│   └── animations/             # Reanimated animations (confetti, tick)\n├── features/                    # Feature-specific logic containers (hooks, queries)\n│   ├── auth/\n│   ├── booking/\n│   ├── search/\n│   └── profile/\n├── hooks/                       # Reusable hooks (useDebounce, useLocation, …)\n├── lib/\n│   ├── api/                     # Axios instance with interceptors (attach Supabase token)\n│   ├── query/                   # React Query client, query keys factory\n│   ├── supabase.ts              # Supabase client init (auth + realtime)\n│   └── geolocation.ts\n├── stores/                      # Zustand stores (guestSession, filterStore, mapStore)\n└── app.config.ts\n```\n\n### 5.2 Key Patterns\n\n- **Data Fetching**: Every API call goes through a React Query `queryFn`. Query keys are generated by a shared factory (`lib/query/keys.ts`) to ensure cache invalidation matches.\n- **Offline Support**: React Query’s `networkMode: 'offlineFirst'` is used. Search results and business details are cached for 5 minutes; stale data is shown with an indicator. Favourites are persisted in MMKV (via Zustand middleware) for offline access.\n- **Authentication Flow**: The app checks for an existing Supabase session on launch. If valid, it sets the JWT in the Axios instance. On token refresh, the new token is automatically updated. Login screens redirect to OAuth or email/password via Supabase’s built-in UI.\n- **Guest Mode**: Guest sessions use a temporary identifier stored in Zustand (cleared after 30 minutes). API calls that do not require authentication pass an `x-guest-id` header for personalisation (search history).\n- **Map Search**: React Native Maps displays clustered markers; map interactions are debounced and filter state is synced with the list view via Zustand.\n- **Real-time Appointments**: The mobile app subscribes to a Supabase Realtime channel filtered by user ID. On status change, React Query’s `queryClient.invalidateQueries` refreshes the appointments list instantly.\n\n## 6. Backend Architecture (NestJS)\n\nThe backend is a modular monolith. Each domain has its own NestJS module with clear boundaries. Modules communicate through the Prisma service (shared data) and via an event emitter (for cross-cutting concerns like notifications). External integrations (Stripe, Supabase Admin) are encapsulated in dedicated services.\n\n### 6.1 Module Boundaries\n\n| Module            | Responsibility                                                                                     | Key Entities / Services                                                                     |\n|-------------------|-----------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------|\n| AuthModule        | Validates Supabase JWTs, provides user context, handles account deletion webhooks from Supabase.    | JwtAuthGuard, SupabaseService (admin API for user lookup), AccountDeletionHandler           |\n| BusinessModule    | CRUD for businesses, services, staff, working hours, and categories. Geospatial search.             | BusinessService, ServiceService, CategoryService, ReviewService, GeoSearchService           |\n| BookingModule     | Manages booking lifecycle: create, confirm, reschedule, cancel. Handles slot locking and payments.  | BookingService, SlotLockService, PaymentService (Stripe integration)                        |\n| AvailabilityModule| Generates available slots from provider schedules, services, and existing bookings. Uses Redis cache.| AvailabilityService, SlotGenerator, BullMQ job for recomputation                           |\n| UserModule        | User profile management (name, phone, avatar), saved payment methods, notification preferences.     | ProfileService, PaymentMethodService (Stripe Customer)                                      |\n| NotificationModule| Sends push notifications (Expo Push) and emails. Listens to booking events.                         | NotificationService, UserDeviceService, BullMQ jobs for batch sending                       |\n| AdminModule       | Dashboard for administrators: manage categories, monitor bookings, view analytics.                  | AdminGuard, AdminController, aggregated queries                                             |\n\n### 6.2 API Design\n\n- **RESTful endpoints** for all CRUD operations with versioning (`/api/v1/`).\n- **JSON:API** conventions (consistent error format, pagination links).\n- **DTO Validation**: All inputs are validated using Zod schemas shared from `libs/shared`. NestJS uses `ZodValidationPipe`.\n- **Authentication**: Global `JwtAuthGuard` verifies the Supabase JWT and extracts user details. Roles (user, business owner, admin) are enforced via custom decorators.\n- **Rate Limiting**: Throttle by user ID (applied globally, with higher limits for authenticated users).\n\n### 6.3 Data Layer\n\n- **Prisma ORM** with a single `PrismaService` per request (via NestJS provider). Migrations live in `apps/api/prisma/migrations`.\n- **PostGIS** enables geospatial queries: `Business` model includes a `location` column of type `Geometry(Point, 4326)`. Search uses `ST_DWithin` and `ST_Distance` with spatial indexes.\n- **Redis** is used for:\n  - Slot availability cache (key: `slots:{businessId}:{serviceId}:{date}`) with TTL 5 minutes, invalidated by booking mutations.\n  - Slot locks during booking: Redis key `lock:slot:{businessId}:{serviceId}:{date}:{startTime}` with TTL 10 minutes prevents double bookings.\n  - Rate limiting token buckets.\n  - BullMQ job queues.\n\n### 6.4 Background Jobs (BullMQ)\n\nAll asynchronous and heavy computation is offloaded to BullMQ queues processed by separate worker processes (included in the same Docker container but can be scaled independently).\n\n| Queue                 | Jobs                                                                 | Trigger                                                                 |\n|-----------------------|----------------------------------------------------------------------|-------------------------------------------------------------------------|\n| availability          | `recompute-slots`: for a given business/service/date                | After any booking create/cancel/reschedule, staff schedule change       |\n| notification          | `send-push`, `send-email`                                            | Event emitted from BookingModule (status change, reminder)              |\n| booking-cleanup       | `release-expired-locks`, `cancel-unpaid-bookings-after-timeout`      | Recurring jobs (cron) every 5 minutes                                   |\n\n### 6.5 Supabase Integration\n\n- **Authentication**: Users register/log in via Supabase Auth (email/password, Google, Apple). The mobile client uses Supabase SDK. The NestJS API relies on the JWT issued by Supabase; the JWT secret is configured in the API’s environment, so `JwtAuthGuard` can verify the token without calling Supabase on every request.\n- **User Management**: The API uses Supabase Admin APIs to look up user emails, delete accounts, etc., in the rare cases where direct DB access isn’t desirable.\n- **Realtime**: The mobile app uses Supabase Realtime to subscribe to changes in the `Booking` table (RPC or direct table subscription with RLS). This allows instant status updates without WebSocket infrastructure on the API. Write operations (booking creation, cancellation) go through NestJS, which then updates the database. Row Level Security ensures users can only see their own bookings.\n\n### 6.6 Security Considerations\n\n- JWTs validated with Supabase’s signing secret (HS256).\n- API enforces Row-Level Security through Prisma access patterns (e.g., `where: { userId: currentUser.id }`).\n- Stripe tokens are never stored; only customer IDs and payment method references.\n- Rate limiting and input sanitisation prevent abuse.\n- Sensitive environment variables injected at runtime via Docker.\n\n## 7. Key Flows\n\n### 8.1 Booking Flow (with Slot Lock)\n\n1. Client requests available slots via `GET /availability?businessId=...&serviceId=...&date=...`.\n   - API checks Redis cache; if miss, calls AvailabilityService to compute from working hours, existing bookings, buffer times, etc.\n   - Returned slots are timezone-adjusted to the user’s device.\n2. User selects a slot and initiates booking by `POST /bookings` (holds a lock).\n   - API acquires a slot lock in Redis (atomic, with TTL). If lock exists, return 409 Conflict.\n   - Creates a Stripe PaymentIntent with `capture_method: manual` (card authorization only).\n   - Returns client secret and booking ID with status `pending_payment`.\n3. Client confirms payment via Stripe Elements; upon success, sends `PATCH /bookings/:id/confirm`.\n   - API validates payment intent, changes booking status to `confirmed`, releases lock (but persists a definitive slot), enqueues availability recomputation and notification jobs.\n4. If lock expires (10 minutes) or user cancels payment, a cleanup job removes the pending booking and releases the lock.\n\n### 8.2 Search & Discovery (Geospatial)\n\n- `GET /businesses/search?lat=...&lng=...&radius=...&query=...&category=...&price=...`\n- Uses raw SQL with Prisma’s `$queryRaw` for geospatial functions: `ST_DWithin`, `ST_Distance`.\n- Results paginated (cursor-based), ordered by distance, rating, relevance (full-text search on name and description).\n- Response includes each business’s next available slot for the current day (precomputed and cached).\n- Map view returns GeoJSON for markers; clustered on the client.\n\n## 9.
