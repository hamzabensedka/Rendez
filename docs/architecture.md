Data Flow\n\n### 5.1 Authentication Flow\n\n1. User submits credentials → Mobile app calls POST /auth/login\n2. NestJS validates credentials, returns JWT access token (7d) + refresh token (30d)\n3. Mobile stores tokens securely in Expo SecureStore\n4. TanStack Query client attaches Authorization header via Axios interceptor\n5. On 401 response, interceptor attempts refresh via POST /auth/refresh\n6. If refresh fails, user is redirected to login\n\n### 5.2 Booking Flow\n\n1. User selects service → Mobile fetches available slots via GET /bookings/slots?businessId=X&serviceId=Y&date=Z\n2. Slots are computed server-side using business schedule, staff availability, existing bookings\n3. User selects time → Mobile creates booking via POST /bookings\n4. Server validates slot availability (optimistic locking with Prisma transaction)\n5. If payment required, Stripe payment intent is created\n6. On payment success, booking status set to `confirmed`\n7. Server sends confirmation push notification via Expo Push API\n8. Mobile navigates to booking confirmation screen\n\n### 5.3 Search Flow\n\n1. User types query → Mobile debounces 300ms, calls GET /search?q=...&lat=...&lng=...&radius=...\n2. NestJS Search module uses PostgreSQL full-text search with `tsvector` and trigram indexes\n3. Results include business name, rating, price range, distance, next available slot\n4. Mobile displays results with TanStack Query caching (staleTime: 5 minutes)\n5. Infinite scroll triggers cursor-based pagination\n\n## 6. Key Architectural Decisions\n\n### 7.1 Monorepo with Nx\n- Shared types and validation schemas in `libs/shared`\n- Generated API client in `libs/api-client` using OpenAPI spec\n- Unified linting, formatting, and testing configuration\n- Dependency graph visualization for CI optimization\n\n### 7.2 PostgreSQL + PostGIS for Geospatial\n- All location queries use PostGIS geography types for accurate distance calculations\n- GiST index on location columns for fast radius searches\n- Full-text search with `tsvector` and `pg_trgm` for fuzzy matching\n- Avoids additional infrastructure (Elasticsearch) for MVP\n\n### 7.3 Redis for Caching and Queues\n- Session store for refresh tokens\n- Rate limiting per user/IP\n- Job queue for async tasks (email, push notifications, analytics)\n- Cache for frequently accessed data (business hours, service categories)\n\n### 7.4 TanStack React Query for Server State\n- Automatic caching with configurable stale times\n- Optimistic updates for booking creation\n- Infinite queries for paginated search results\n- Mutation hooks with rollback on error\n\n### 7.5 Supabase for Auth and Storage\n- Handles OAuth providers (Google, Apple)\n- Email verification and password reset flows\n- File storage for business images and user avatars\n- Row Level Security (RLS) for direct database access (future)\n\n## 8. Deployment Architecture\n\n### 8.1 Local Development\n- Docker Compose runs PostgreSQL, Redis, and NestJS backend\n- Expo Go for mobile development on physical devices\n- Nx task runner for parallel builds and tests\n\n### 8.2 CI/CD (GitHub Actions)\n- **CI Pipeline**: Lint → Type check → Unit tests → Build → Integration tests\n- **Mobile Deploy**: EAS Build → Submit to App Store/Google Play\n- **Backend Deploy**: Docker image → Deploy to cloud (AWS ECS/Google Cloud Run)\n\n### 8.3 Production\n- Backend: Containerized NestJS behind load balancer\n- Database: Managed PostgreSQL (AWS RDS / Supabase)\n- Cache: Managed Redis (AWS ElastiCache / Upstash)\n- File Storage: Supabase Storage / AWS S3\n- Push Notifications: Expo Push API\n- Monitoring: Sentry for error tracking, DataDog for APM\n\n## 9.
