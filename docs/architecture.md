  "content": "# Planity Clone System Architecture\n\n## 1. Overview\n\nPlanity Clone is a mobile-first platform connecting customers with beauty and wellness businesses. The system follows a modular monolith approach with clear service boundaries, using Nx monorepo for shared code and independent deployability. The backend is a NestJS application exposing RESTful APIs, backed by PostgreSQL (with PostGIS for geospatial queries) and Redis for caching and job queues. The mobile client is built with Expo (React Native), Expo Router for navigation, TanStack React Query for server state, and Reanimated for smooth animations. Authentication leverages Supabase Auth, while asynchronous tasks (slot computation, notifications) run on BullMQ. The entire stack is containerized with Docker Compose, built and deployed via GitHub Actions and EAS Build.\n\n---\n\n## 2. Design Principles\n\n- **Separation of Concerns**: Clean boundaries between presentation, application, domain, and infrastructure layers.\n- **API-First**: Backend defines OpenAPI specs; mobile consumes typed APIs with generated service clients.\n- **Shared Types**: Central TypeScript library for DTOs, enums, and validation schemas, used by both frontend and backend.\n- **Performance**: Predictive caching, optimistic UI updates, pagination, and lazy loading.\n- **Observability**: Structured logging, request tracing, and error monitoring.\n- **Security**: Rate limiting, JWT with refresh token rotation, HTTPS, input validation, and GDPR-compliant data handling.\n\n---\n\n## 3. High-Level Architecture\n\nThe system consists of two main deployable units:\n- **Mobile App**: Expo/React Native application (iOS, Android).\n- **Backend API**: NestJS server (containerised) handling business logic, persistence, and communication.\n\nSupporting infrastructure:\n- PostgreSQL + PostGIS for relational data and location queries.\n- Redis for caching (slot results, session data) and as a message broker for BullMQ.\n- Supabase Auth for authentication (email/password, SSO, JWT issuance).\n- Supabase Storage for user and business image uploads.\n- Background workers (BullMQ) for slot computation, notifications, and reminders.\n\n```\nв”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ\nв”Ӯ   Mobile Client      в”Ӯ\nв”Ӯ (Expo / React Native)в”Ӯ\nв””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ\n         в”Ӯ HTTPS\n         в–ј\nв”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ        в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ\nв”Ӯ    NestJS API        в”Ӯв—„в”Җв”Җв”Җв”Җв”Җв”Җв–әв”Ӯ   Supabase Auth в”Ӯ\nв”Ӯ (REST / Swagger)     в”Ӯ        в”Ӯ   (JWT Issuer)  в”Ӯ\nв””в”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ        в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ\n   в”Ӯ         в”Ӯ\n   в–ј         в–ј\nв”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ\nв”ӮPostgreSQLв”Ӯ в”Ӯ  Redis   в”Ӯ\nв”Ӯ (PostGIS)в”Ӯ в”Ӯ (cache,  в”Ӯ\nв””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ queue)   в”Ӯ\n            в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ\n```\n\n---\n\n## 4. Monorepo Structure (Nx)\n\nWe use Nx with pnpm workspaces to manage multiple projects. The folder scaffold:\n\n```\n/planity/\nв”ңв”Җ apps/\nв”Ӯ  в”ңв”Җ mobile/               # Expo React Native app\nв”Ӯ  в”Ӯ  в”ңв”Җ app/               # Expo Router file-based routing\nв”Ӯ  в”Ӯ  в”ңв”Җ components/        # UI components (screens, shared)\nв”Ӯ  в”Ӯ  в”ңв”Җ hooks/             # Custom hooks\nв”Ӯ  в”Ӯ  в”ңв”Җ services/          # API service abstractions (React Query)\nв”Ӯ  в”Ӯ  в”ңв”Җ utils/             # Helpers, constants\nв”Ӯ  в”Ӯ  в”ңв”Җ app.json\nв”Ӯ  в”Ӯ  в””в”Җ eas.json\nв”Ӯ  в””в”Җ api/                  # NestJS backend\nв”Ӯ     в”ңв”Җ src/\nв”Ӯ     в”Ӯ  в”ңв”Җ modules/        # Domain modules (auth, business, booking, etc.)\nв”Ӯ     в”Ӯ  в”ңв”Җ common/         # Guards, decorators, filters, interceptors\nв”Ӯ     в”Ӯ  в”ңв”Җ config/         # Environment configuration\nв”Ӯ     в”Ӯ  в””в”Җ main.ts\nв”Ӯ     в”ңв”Җ prisma/            # Schema, migrations\nв”Ӯ     в”ңв”Җ test/\nв”Ӯ     в””в”Җ Dockerfile\nв”Ӯ\nв”ңв”Җ libs/\nв”Ӯ  в”ңв”Җ shared-types/         # TypeScript types, DTOs, enums, validation schemas (Zod)\nв”Ӯ  в”ңв”Җ shared-ui/            # Cross-platform UI kit (components, theme)\nв”Ӯ  в”ңв”Җ shared-utils/         # Formatting, date, location utilities\nв”Ӯ  в””в”Җ shared-hooks/         # Common React hooks (useDebounce, etc.)\nв”Ӯ\nв”ңв”Җ tools/                   # Custom Nx generators, scripts\nв”ңв”Җ docker-compose.yml\nв”ңв”Җ nx.json\nв”ңв”Җ package.json\nв”ңв”Җ pnpm-workspace.yaml\nв””в”Җ .github/workflows/       # CI/CD pipelines\n```\n\nAll code sharing is done through TypeScript path aliases configured in `tsconfig.base.json`. The `shared-types` lib serves as the single source of truth for API contracts.\n\n---\n\n## 5. Frontend Architecture (Mobile)\n\n### 5.1 Expo & Expo Router\n- File-based routing inside `app/` directory, supporting nested layouts, tabs (Appointments, Explore, Profile), and dynamic routes (business/[id]).\n- Deep linking configured for business sharing.\n- Authentication guard implemented via route groups (e.g., `(auth)` and `(guest)`).\n\n### 5.2 State Management & Data Fetching\n- **Server State**: TanStack React Query handles all API data with automatic caching, background refetching, and optimistic updates (e.g., toggling favorites).\n- **Client State**: React Context for global UI state (e.g., selected location, map/list toggle).\n- **Persistence**: AsyncStorage for storing refresh token and user preferences (combined with React QueryвҖҷs persister).\n\n### 5.3 UI & Animations\n- **Design System**: `libs/shared-ui` provides a theme (colors, typography, spacing) and reusable components (Button, Card, SearchBar, StarRating, SlotPicker).\n- **Animations**: React Native Reanimated and Gesture Handler for map interactions, smooth transitions, drag-to-refresh, and skeleton loaders.\n\n### 5.4 Component Structure\nEach screen follows a container/presentational pattern:\n- `screens/HomeScreen.tsx` (container) вҖ“ wires React Query and navigation logic.\n- `components/home/FeaturedList.tsx` вҖ“ purely presentational.\n\nMap search uses `react-native-maps` with clustering. Business cards are reusable across list and map views.\n\n### 5.5 API Integration\nA `services/api-client.ts` sets up Axios with interceptors for JWT refresh. Dedicated hooks like `useBusinesses(query)`, `useAvailability(params)` encapsulate caching keys and query functions, all using typed responses from `shared-types`.\n\n---\n\n## 6. Backend Architecture (NestJS)\n\nThe backend is organized as a modular monolith with clear domain boundaries. Each module can be extracted into a microservice later if needed.\n\n### 6.1 Module Structure\n- **AuthModule**: Validates Supabase JWT, manages user roles, refresh token rotation (stored in Redis).\n- **UserModule**: Customer profiles, preferences, saved payment methods.\n- **BusinessModule**: Business profiles, services, staff, schedules, galleries, reviews.\n- **BookingModule**: Booking creation, reschedule, cancellation, conflict detection.\n- **AvailabilityModule**: Slot computation engine (triggers via BullMQ), caching, and query endpoints.\n- **CategoryModule**: CRUD for service categories (admin).\n- **NotificationModule**: Push/email/SMS notifications via jobs.\n- **AdminModule**: Dashboard stats, user management, platform config.\n- **PaymentModule**: Integration with Stripe, holds charges, captures on completion.\n- **Common**: Guards (JWT, Roles), rate limiter, exception filters, Swagger decorators.\n\nEach module contains:\n```\nmodule/\n  controller.ts  # Route handlers (input validation with Zod/DTOs)\n  service.ts     # Business logic\n  repository.ts  # Database access (Prisma)\n  *.spec.ts\n```\n\n### 6.2 Authentication & Authorization\n- **Supabase Auth**: Handles sign-up, login, SSO, password reset. Issues JWT access token (short-lived) and refresh token. Backend validates tokens using SupabaseвҖҷs JWKS endpoint via a `SupabaseAuthGuard`.\n- **Refresh tokens** are stored in Redis with rotation capability.\n- **Roles** (customer, provider, admin) are embedded in the JWT claims and enforced using NestJS `RolesGuard`.\n\n### 6.3 Slot Computation Engine\n- **Asynchronous Processing**: Booking creation/update/cancellation event triggers a BullMQ job to recalculate availability for affected date range.\n- **Algorithm**: Determines free slots for a service/staff/date by:\n  1. Fetch business & staff working hours, breaks, service duration + buffer.\n  2. Retrieve existing bookings for that date and staff.\n  3. Subtract booked intervals from working intervals.\n  4. Slide a window of service duration to generate start times.\n  5. Store result array in Redis with key `slots:{businessId}:{serviceId}:{staffId?}:{date}` and TTL of 15 minutes (invalidated on mutation).\n- **API endpoint** `GET /availability` returns cached slots; on cache miss, it triggers a synchronous quick compute for just that query (with fallback).\n\n### 6.4 Background Jobs (BullMQ)\n- **Slot Recalculation**: On booking/schedule change, enqueue job to recompute slots.\n- **Reminders**: Schedules push notifications 24h and 1h before appointment.\n- **Expiry**: Clean up old slots, canceled bookings, etc.\n\nBullMQ uses Redis and runs in the same NestJS process or as a separate worker (configurable).\n\n### 6.5 API Design\n- RESTful with JSON, documented via Swagger (OpenAPI).\n- Consistent error format: `{ statusCode, message, errors }`.\n- Versioning via URL prefix (e.g., `/v1/`).\n- Rate limiting on auth and public search endpoints.\n\n---\n\n## 7. Database Design (PostgreSQL + PostGIS)\n\nPrisma ORM used with a relational schema. Key tables:\n\n```\nusers          | id, email, name, phone, avatar_url, supabase_id, role, ...\nbusinesses     | id, name, description, address, location (geometry point), cover_image, ...\nservices       | id, business_id, category_id, name, duration, price, ...\nstaff          | id, business_id, name, avatar, ...\nstaff_schedules| id, staff_id, day_of_week, start_time, end_time, breaks (jsonb)\nbookings       | id, customer_id, business_id, service_id, staff_id, start_time, end_time, status\ncategories     | id, name, icon, parent_id (self-referential)\nfavorites      | user_id, business_id (unique constraint)\nreviews        | id, booking_id, rating, text, photos (array), created_at\npayments       | id, booking_id, amount, status, stripe_payment_intent_id\n```\n\nPostGIS extension enables:\n- `location geometry(Point, 4326)` for business coordinates.\n- Distance-based filtering using `ST_DWithin` and ordering by distance.\n\nPrisma schema uses `Unsupported` types for PostGIS, with raw SQL where needed; we write a thin abstraction in repositories for geospatial queries.\n\n---\n\n## 8. Caching Strategy (Redis)\n\n- **Slot Availability**: Cached per business/service/staff/date, TTL 15 min, immediate invalidation on write.\n- **Business Search Results**: Only cache top search terms for 5 minutes; otherwise rely on database full-text search (pg_trgm) and indexes.\n- **User Sessions**: Refresh token blacklist/whitelist for logout/rotation.\n- **Rate Limiting**: Redis-backed sliding window rate limiter for auth endpoints.\n\n---\n\n## 9. Deployment & CI/CD\n\n### 9.1 Local Development\n- Docker Compose spins up PostgreSQL (with PostGIS), Redis, and the NestJS API.\n- Mobile app can connect to local API via Expo dev server.\n\n### 9.2 CI/CD (GitHub Actions)\n- Lint, type-check, unit/integration tests (Jest) on pull requests.\n- Build Docker image for API and push to container registry on merge to main.\n- EAS Build triggered for mobile app via `eas build` command (production & preview).\n- Automated deployment to staging (container orchestration like Railway/Fly) and production.\n\n### 9.3 EAS Build\n- Configured via `eas.json` for Android (aab) and iOS (ipa).\n- Secrets (API URLs, Supabase keys) injected at build time using EAS secrets.\n\n### 9.4 Supabase\n- Used as authentication provider and for file storage (user avatars, business images).\n- RLS policies managed for bucket access.\n\n---\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for shared libs, utility functions, and NestJS services.\n- **Integration Tests**: NestJS supertest for API endpoints with a test database; Prisma migrations run before tests.\n- **E2E**: (Future) Detox for mobile critical paths.\n- **Contract Testing**: Optionally, generate API client from OpenAPI to verify mobile compliance.\n- **Performance Tests**: Load testing on slot computation endpoints.\n\n---\n\n## 11. Non-Functional Requirements\n\n- **Scalability**: Stateless API allows horizontal scaling; background workers scale independently.\n- **Resilience**: Retry logic for external services (Stripe, push notifications). Circuit breaker for slot recalculation queue.\n- **Security**: All traffic HTTPS; input sanitization; Helmet headers; CORS; GDPR compliance (account deletion, data export).\n- **Observability**: Winston logger with request ID; Prometheus metrics endpoint; Sentry error tracking on both mobile and backend.\n\n---\n\n## 12. Shared Types & Contracts\n\n`libs/shared-types` exports:\n- `DTOs`: `CreateBookingDto`, `SearchBusinessQueryDto`, `UserProfileDto`, etc.\n- `Responses`: `BusinessCardDto`, `BookingSlotDto`.\n- `Enums`: `BookingStatus`, `UserRole`.\n- `Validation`: Zod schemas for each DTO (used by both NestJS `ZodValidationPipe` and React Native forms).\n\nThis ensures end-to-end type safety without code generation, as both apps import the same package.\n\n---\n\n## 13.
