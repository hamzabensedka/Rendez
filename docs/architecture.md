Service Boundaries\n\n### 5.1 Backend Services (Modular Monolith)\n\nAll backend services are part of a single NestJS application but organized into independent modules with clear boundaries. They share the same database but access only their own tables via Prisma.\n\n- **Auth Service**  \n  Handles registration, login, social auth, phone OTP, JWT issuance, refresh tokens, password reset. Integrates with Supabase Auth for identity management; stores user profiles in PostgreSQL.\n\n- **Business Service**  \n  Manages businesses, services, staff, operating hours, categories. Provides search endpoints with full-text search (PostgreSQL tsvector) and geospatial filtering (PostGIS). Caches category trees and popular searches in Redis.\n\n- **Booking Service**  \n  Core booking flow: slot computation, appointment creation, status management, cancellation, and payment status tracking. Uses pessimistic locking on staff calendars to prevent double-booking. Emits events to recompute slots asynchronously.\n\n- **Notification Service**  \n  Sends push notifications (via Expo Push API) and emails (via SendGrid) for booking confirmations, reminders, cancellations. Listens to events from Booking and Auth services.\n\n- **Admin Service**  \n  Provides dashboard APIs for platform admins: user management, business approval, category management, analytics. Protected by role-based guards.\n\n- **Worker Service** (separate process)  \n  Runs BullMQ workers for slot recomputation, notification dispatch, and scheduled tasks (e.g., appointment reminders). Shares the same codebase but runs as a standalone NestJS application.\n\n### 5.2 Frontend Feature Modules\n\nEach feature is a self-contained folder with screens, components, hooks, and API integration.\n\n- **Auth**: login, signup, OTP, password reset screens.\n- **Explore**: home feed, categories, search, map view.\n- **Business**: detail view, staff list, services, reviews.\n- **Booking**: multi-step wizard, slot selection, confirmation.\n- **Profile**: user settings, appointment history, favorites.\n- **Admin** (web only): dashboard, business approval, analytics.\n\n## 6. Data Flow\n\n### 6.1 Authentication Flow\n\n1. User signs up/in via Supabase Auth (email/password or social).\n2. Supabase returns JWT access token and refresh token.\n3. Mobile app stores tokens securely (expo-secure-store).\n4. All API requests include `Authorization: Bearer <token>`.\n5. API Gateway validates token using Supabase's JWKS endpoint.\n6. On token expiry, the app uses the refresh token to obtain a new access token silently.\n\n### 6.2 Search & Discovery\n\n1. User enters search query or applies filters.\n2. Mobile app calls `GET /businesses?search=...&lat=...&lng=...&category=...`.\n3. API Gateway forwards to Business Service.\n4. Business Service queries PostgreSQL with full-text search and `ST_DWithin` for distance.\n5. Results are cached in Redis for 5 minutes with a key based on query parameters.\n6. Paginated response (20 items) returned; React Query manages infinite scroll.\n\n### 6.3 Booking Flow\n\n1. User selects service, staff, and date.\n2. App requests available slots: `GET /booking/slots?serviceId=...&staffId=...&date=...`.\n3. Booking Service checks Redis cache; if miss, computes slots considering staff schedule, existing appointments, buffer times.\n4. Slots returned as array of start times.\n5. User selects a slot and confirms booking: `POST /booking/appointments`.\n6. Booking Service acquires a pessimistic lock on the staff's calendar row, validates slot availability again, creates appointment, releases lock.\n7. A BullMQ job is enqueued to recompute affected slots for that staff/date.\n8. Notification Service is notified to send confirmation.\n\n### 6.4 Slot Recomputation (Async)\n\n- Triggered by appointment creation, cancellation, or staff schedule change.\n- Worker picks up job, recomputes slots for the affected staff and date, updates Redis cache.\n- Cache key pattern: `slots:{businessId}:{staffId}:{date}`.\n- TTL set to 24 hours; invalidated on change.\n\n## 7. Database Schema (Prisma)\n\nKey models (simplified):\n\n```prisma\nmodel User {\n  id        String   @id @default(uuid())\n  email     String   @unique\n  phone     String?\n  role      Role     @default(CUSTOMER)\n  profile   Profile?\n  businesses Business[]\n  appointments Appointment[]\n  reviews   Review[]\n  favorites Favorite[]\n}\n\nmodel Business {\n  id          String   @id @default(uuid())\n  name        String\n  description String?\n  address     String\n  location    Unsupported(\"geometry(Point, 4326)\")? // PostGIS\n  ownerId     String\n  owner       User     @relation(fields: [ownerId], references: [id])\n  services    Service[]\n  staffs      Staff[]\n  hours       OperatingHours[]\n  photos      String[]\n  categoryId  String?\n  category    Category? @relation(fields: [categoryId], references: [id])\n}\n\nmodel Service {\n  id         String   @id @default(uuid())\n  businessId String\n  business   Business @relation(fields: [businessId], references: [id])\n  name       String\n  duration   Int      // minutes\n  price      Decimal\n  category   String?\n  active     Boolean  @default(true)\n}\n\nmodel Staff {\n  id         String   @id @default(uuid())\n  businessId String\n  business   Business @relation(fields: [businessId], references: [id])\n  name       String\n  title      String?\n  serviceIds String[] // array of Service IDs they perform\n  schedule   StaffSchedule[]\n}\n\nmodel Appointment {\n  id            String   @id @default(uuid())\n  customerId    String\n  customer      User     @relation(fields: [customerId], references: [id])\n  businessId    String\n  business      Business @relation(fields: [businessId], references: [id])\n  staffId       String\n  staff         Staff    @relation(fields: [staffId], references: [id])\n  serviceId     String\n  service       Service  @relation(fields: [serviceId], references: [id])\n  startTime     DateTime\n  endTime       DateTime\n  status        AppointmentStatus\n  price         Decimal\n  paymentStatus PaymentStatus\n}\n\n// Additional models: Review, Favorite, Category, OperatingHours, StaffSchedule, etc.\n```\n\nPostGIS extension is enabled for geospatial queries. Indexes on `Business.location` (GIST), `Appointment.startTime`, `Appointment.staffId`, and full-text search vectors on `Business.name` and `Service.name`.\n\n## 8. Caching Strategy\n\n- **Redis** used for:\n  - Slot availability cache (key: `slots:{businessId}:{staffId}:{date}`, TTL 24h)\n  - Category tree (key: `categories:tree`, TTL 1h)\n  - Search results (key: `search:{hash(query, filters)}`, TTL 5min)\n  - Session store for rate limiting and refresh token blacklist\n- Cache invalidation: explicit deletion on data change (e.g., new appointment, updated business hours).\n- Cache-aside pattern: check Redis first, on miss query DB and populate cache.\n\n## 9. Background Jobs (BullMQ)\n\n- **SlotRecomputation**: triggered after booking/cancellation, processes one staff/date at a time.\n- **AppointmentReminder**: scheduled 24h and 1h before appointment, sends push notification.\n- **NotificationDispatch**: sends emails/push for booking confirmations, cancellations, etc.\n- **AnalyticsAggregation**: daily rollups for admin dashboard (optional).\n\nAll jobs are idempotent and retryable. Failed jobs go to a dead-letter queue for inspection.\n\n## 10. API Design\n\nRESTful endpoints grouped by resource:\n\n- `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/otp/send`, `POST /auth/otp/verify`\n- `GET /businesses`, `GET /businesses/:id`, `GET /businesses/:id/services`, `GET /businesses/:id/staff`\n- `GET /categories`, `GET /categories/:id`\n- `GET /booking/slots?serviceId=&staffId=&date=`, `POST /booking/appointments`, `PATCH /booking/appointments/:id/cancel`\n- `GET /users/me`, `PATCH /users/me`, `GET /users/me/appointments`, `GET /users/me/favorites`\n- `POST /reviews`, `GET /businesses/:id/reviews`\n- Admin endpoints under `/admin/*` with role guard.\n\nAll responses follow a standard envelope: `{ data, meta, error }`. Pagination uses cursor-based or offset-based as appropriate.\n\n## 11. Frontend Architecture\n\n### 11.1 Navigation (Expo Router)\n\nFile-based routing with groups:\n- `(auth)` вҖ“ unprotected screens (login, signup, guest explore)\n- `(tabs)` вҖ“ main app tabs: Home, Search, Bookings, Profile\n- `business/[id]` вҖ“ business detail\n- `booking/[businessId]` вҖ“ booking wizard\n\nDeep linking supported for business profiles and booking flows.\n\n### 11.2 State Management\n\n- **Server state**: TanStack React Query for all API data fetching, caching, and mutations.
