Overview\n\nThis document describes the complete system architecture for the Planity Clone mobile application, a platform connecting consumers with beauty, wellness, and health service providers. The architecture is designed for scalability, maintainability, and a clear separation of concerns. It uses a monorepo structure with a mobile client (Expo/React Native) and a backend API (NestJS), sharing TypeScript types and utilities.\n\n## 2. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| **Mobile Client** | Expo SDK 50+, React Native, TypeScript | Cross-platform mobile app |\n| **Routing** | Expo Router (file-based) | Declarative navigation |\n| **State Management** | TanStack React Query | Server state caching and synchronization |\n| **Animations** | React Native Reanimated | High-performance UI animations |\n| **Backend API** | NestJS, TypeScript | RESTful API server |\n| **ORM** | Prisma | Type-safe database access |\n| **Database** | PostgreSQL + PostGIS | Relational data + geospatial queries |\n| **Cache & Session** | Redis | Session store, rate limiting, caching |\n| **Build Tool** | Nx + pnpm | Monorepo management, dependency caching |\n| **Containerization** | Docker Compose | Local development environment |\n| **CI/CD** | GitHub Actions | Automated testing and deployment |\n| **Hosting** | Supabase (PostgreSQL, Auth, Storage) | Managed database, auth, file storage |\n| **Build & Deploy** | EAS Build | Mobile app builds and OTA updates |\n| **Testing** | Jest | Unit and integration testing |\n\n## 3. High-Level Architecture\n\nThe system follows a client-server architecture with a clear separation between the mobile client and the backend API. The backend is decomposed into microservices (or modules) that communicate via HTTP and Redis pub/sub for real-time events.\n\n```\n+-------------------+       +-------------------+       +-------------------+\n|                   |       |                   |       |                   |\n|  Mobile Client    |------>|  API Gateway      |------>|  Auth Service     |\n|  (Expo/RN)        |       |  (NestJS)         |       |  (NestJS)         |\n|                   |       |                   |       |                   |\n+-------------------+       +-------------------+       +-------------------+\n        |                           |                           |\n        |                           |                           |\n        v                           v                           v\n+-------------------+       +-------------------+       +-------------------+\n|                   |       |                   |       |                   |\n|  Redis Cache      |       |  Business Service |       |  Booking Service  |\n|                   |       |  (NestJS)         |       |  (NestJS)         |\n|                   |       |                   |       |                   |\n+-------------------+       +-------------------+       +-------------------+\n                                    |                           |\n                                    |                           |\n                                    v                           v\n                            +-------------------+       +-------------------+\n                            |                   |       |                   |\n                            |  PostgreSQL       |       |  PostgreSQL       |\n                            |  + PostGIS        |       |  + PostGIS        |\n                            |                   |       |                   |\n                            +-------------------+       +-------------------+\n```\n\n## 4. Folder Structure\n\nThe monorepo is managed by Nx with pnpm workspaces. The root structure is:\n\n```\nplanity-clone/\n├── .github/\n│   └── workflows/\n│       ├── ci.yml\n│       └── deploy.yml\n├── apps/\n│   ├── mobile/                 # Expo React Native app\n│   │   ├── app/                # Expo Router pages\n│   │   ├── components/         # Shared UI components\n│   │   ├── hooks/              # Custom hooks\n│   │   ├── services/           # API client, auth, etc.\n│   │   ├── utils/              # Helpers\n│   │   ├── types/              # TypeScript types\n│   │   ├── app.json\n│   │   ├── package.json\n│   │   └── tsconfig.json\n│   └── api/                    # NestJS backend\n│       ├── src/\n│       │   ├── modules/\n│       │   │   ├── auth/\n│       │   │   ├── users/\n│       │   │   ├── businesses/\n│       │   │   ├── services/\n│       │   │   ├── bookings/\n│       │   │   ├── reviews/\n│       │   │   ├── search/\n│       │   │   └── notifications/\n│       │   ├── common/\n│       │   ├── prisma/\n│       │   ├── main.ts\n│       │   └── app.module.ts\n│       ├── test/\n│       ├── prisma/\n│       │   └── schema.prisma\n│       ├── package.json\n│       └── tsconfig.json\n├── libs/\n│   ├── shared-types/           # Shared TypeScript interfaces\n│   ├── shared-utils/           # Shared utility functions\n│   └── shared-ui/              # Shared React Native components (if any)\n├── docker-compose.yml\n├── nx.json\n├── package.json\n├── pnpm-workspace.yaml\n├── tsconfig.base.json\n└── README.md\n```\n\n## 5. Service Boundaries\n\n### 5.1 Mobile Client (apps/mobile)\n\nThe mobile client is an Expo application using Expo Router for file-based routing. It communicates with the backend via a RESTful API client (Axios) with TanStack React Query for caching and synchronization.\n\n**Key modules:**\n- **app/**: Pages and layouts (Expo Router)\n- **components/**: Reusable UI components (buttons, cards, modals, etc.)\n- **hooks/**: Custom hooks for business logic (useAuth, useBookings, etc.)\n- **services/**: API client, authentication service, push notifications\n- **utils/**: Formatting, validation, geolocation helpers\n\n### 5.2 Backend API (apps/api)\n\nThe backend is a NestJS application organized into modules, each responsible for a specific domain.\n\n**Modules:**\n- **auth**: Registration, login, OAuth, JWT management, password recovery\n- **users**: User profiles, roles, preferences\n- **businesses**: Business CRUD, categories, staff management, availability\n- **services**: Service definitions, pricing, duration\n- **bookings**: Appointment creation, cancellation, rescheduling, conflict detection\n- **reviews**: Review submission, moderation, rating aggregation\n- **search**: Full-text search, geospatial search, autocomplete\n- **notifications**: Push notifications (via Firebase Cloud Messaging), email (via SendGrid)\n\n### 5.3 Shared Libraries (libs/)\n\n- **shared-types**: TypeScript interfaces for API request/response payloads, database models\n- **shared-utils**: Validation schemas (Zod), date helpers, constants\n- **shared-ui**: (Optional) Cross-app UI components if needed\n\n## 6. Data Flow\n\n### 6.1 Authentication Flow\n\n1. User enters credentials on mobile client.\n2. Mobile client sends POST /auth/login to API Gateway.\n3. Auth service validates credentials, returns JWT access token (15 min) and refresh token (7 days).\n4. Mobile client stores tokens securely (expo-secure-store).\n5. TanStack React Query interceptor adds access token to all requests.\n6. On 401 response, interceptor attempts token refresh via POST /auth/refresh.\n7. If refresh fails, user is redirected to login.\n\n### 6.2 Booking Flow\n\n1. Consumer searches for services, selects a business.\n2. Mobile client fetches available slots via GET /businesses/:id/availability?date=...\n3. User selects a slot, confirms booking.\n4. Mobile client sends POST /bookings with service ID, staff ID, start time.\n5. Booking service validates slot availability (using database lock or Redis atomic operation).\n6. On success, booking is created, provider receives push notification.\n7. Mobile client invalidates related queries (availability, upcoming bookings).\n\n### 6.3 Search Flow\n\n1. User types query in search bar.\n2. Mobile client debounces input (300ms), sends GET /search?q=...&lat=...&lng=...&radius=...\n3. Search service uses PostgreSQL full-text search (tsvector) and PostGIS for distance.\n4. Results are cached in Redis for 5 minutes (keyed by query + location).\n5. Mobile client displays results, with infinite scroll pagination.\n\n## 7.   @relation(fields: [staffId], references: [id])\n  startTime   DateTime\n  endTime     DateTime\n  status      BookingStatus @default(PENDING)\n  createdAt   DateTime @default(now())\n  updatedAt   DateTime @updatedAt\n}\n\nmodel Review {\n  id          String   @id @default(uuid())\n  userId      String\n  user        User     @relation(fields: [userId], references: [id])\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  rating      Int      // 1-5\n  comment     String?\n  createdAt   DateTime @default(now())\n}\n```\n\n## 8. API Endpoints (RESTful)\n\n### 8.1 Authentication\n- POST /auth/register\n- POST /auth/login\n- POST /auth/refresh\n- POST /auth/logout\n- POST /auth/password-reset-request\n- POST /auth/password-reset\n- POST /auth/oauth/{provider}\n\n### 8.2 Users\n- GET /users/me\n- PATCH /users/me\n- GET /users/:id/bookings\n\n### 8.3 Businesses\n- GET /businesses (with search, filter, pagination)\n- GET /businesses/:id\n- POST /businesses (provider only)\n- PATCH /businesses/:id (owner only)\n- DELETE /businesses/:id (owner only)\n- GET /businesses/:id/availability?date=...&staffId=...\n\n### 8.4 Services\n- GET /businesses/:id/services\n- POST /businesses/:id/services (owner only)\n- PATCH /services/:id (owner only)\n- DELETE /services/:id (owner only)\n\n### 8.5 Bookings\n- GET /bookings (user's bookings)\n- POST /bookings\n- PATCH /bookings/:id/cancel\n- PATCH /bookings/:id/reschedule\n\n### 8.6 Reviews\n- GET /businesses/:id/reviews\n- POST /businesses/:id/reviews\n- DELETE /reviews/:id\n\n### 8.7 Search\n- GET /search?q=...&lat=...&lng=...&radius=...&category=...&date=...\n- GET /search/autocomplete?q=...\n\n## 9. Security Considerations\n\n- **Authentication**: JWT with short-lived access tokens (15 min) and refresh tokens (7 days). Tokens stored in secure device storage (expo-secure-store).\n- **Authorization**: Role-based access control (RBAC) enforced at API gateway and service level using NestJS guards.\n- **Data Validation**: Input validation using Zod schemas on both client and server.\n- **Rate Limiting**: Redis-based rate limiting per IP and per user (e.g., 100 requests/minute).\n- **SQL Injection**: Prevented by Prisma ORM parameterized queries.\n- **HTTPS**: All communication over TLS.\n- **CORS**: Restricted to allowed origins.\n- **Environment Variables**: Secrets stored in GitHub Actions secrets and .env files (not committed).\n\n## 10. Performance & Scalability\n\n- **Caching**: Redis caches search results (5 min TTL), business details (10 min), and availability (1 min).\n- **Database Indexing**: Indexes on foreign keys, search columns (GIN for full-text), and location (GiST for PostGIS).\n- **Pagination**: Cursor-based pagination for lists to avoid offset performance issues.\n- **Background Jobs**: Use Bull (Redis-backed queue) for email notifications, push notifications, and data aggregation.\n- **CDN**: Business images served via Supabase Storage CDN.\n- **Lazy Loading**: Mobile client lazy loads screens and components using React.lazy and Suspense.\n\n## 11. Testing Strategy\n\n- **Unit Tests**: Jest for services, utilities, and hooks. Coverage target: 80%.\n- **Integration Tests**: NestJS testing module with test database (Docker Compose).\n- **E2E Tests**: Detox for mobile app (future).\n- **CI/CD**: GitHub Actions runs tests on every push. Pull requests require passing tests and linting.\n\n## 12. Deployment\n\n- **Mobile**: EAS Build for iOS and Android builds. OTA updates via EAS Update.\n- **Backend**: Dockerized NestJS app deployed to a cloud provider (e.g., AWS ECS, Google Cloud Run).\n- **Database**: Supabase managed PostgreSQL with automated backups.\n- **CI/CD Pipeline**: GitHub Actions builds Docker image, runs tests, deploys to staging/production.\n\n## 13. Monitoring & Logging\n\n- **Logging**: Structured logging with Winston (NestJS) and console.log (mobile, filtered in production).\n- **Error Tracking**: Sentry for both mobile and backend.\n- **Performance Monitoring**: New Relic or Datadog for backend.\n- **Analytics**: Mixpanel or Amplitude for user behavior.\n\n## 14. Conclusion\n\nThis architecture provides a solid foundation for building the Planity Clone platform. It emphasizes clean separation of concerns, type safety, and scalability.
