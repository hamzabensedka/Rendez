  "content": "# Planity Clone Architecture Document\n\n## 1. Overview\nPlanity Clone is a mobile-first beauty and wellness booking platform with three interfaces: Customer App (mobile + PWA), Provider Portal, and Admin Dashboard. The system leverages a monorepo architecture with Nx and pnpm workspaces to share code between the mobile/web client applications and the NestJS backend. The entire stack is TypeScript-first, with PostgreSQL + PostGIS for spatial data, Redis for caching and real-time slot locks, and Supabase for authentication and file storage.\n\n## 2. Technology Stack\n- **Frontend Customer App**: Expo (React Native) + Expo Router (file-based routing), TanStack React Query (server state), React Native Reanimated (animations). Targets iOS, Android, and Web.\n- **Web Portals** (Provider & Admin): React (Vite) with React Router, TanStack React Query, shared UI components.\n- **Shared**: TypeScript strict mode, shared types (generated from Prisma/Zod), shared hooks, and UI kit.\n- **Backend**: NestJS (Express) microservices-lite modular monolith with REST API (OpenAPI/Swagger).\n- **ORM & Database**: Prisma ORM, PostgreSQL 15+ with PostGIS extension for geographic queries.\n- **Cache & Queues**: Redis (for slot availability cache, session store, rate limiting, temporary locks), BullMQ (for background jobs like slot recalculation, notifications).\n- **Auth & Storage**: Supabase (Auth, Row Level Security optional, Storage for user/business images).\n- **Monorepo Tooling**: Nx (task orchestration, caching, dependency graph), pnpm workspaces.\n- **Local Dev**: Docker Compose (PostgreSQL, Redis) + Supabase CLI local stack.\n- **CI/CD**: GitHub Actions for backend and web; EAS Build for Expo mobile apps.\n- **Testing**: Jest (unit/integration), Supertest (API tests), Detox (E2E mobile – plan).\n\n## 3. High-Level Architecture\nThe system follows a **clean architecture** with clear separation of concerns:\n- **Presentation Layer** (mobile app, web portals): handles UI, routing, and user interaction. Uses React Query for optimistic updates and cache management.\n- **API Gateway / Backend for Frontend (BFF)**: NestJS serves as a single backend for all clients, with dedicated modules per domain. It validates Supabase JWTs, enforces RBAC, and orchestrates business logic.\n- **Domain Layer** (inside NestJS): services encapsulating business rules (booking rules, slot calculation, cancellation policies).\n- **Infrastructure Layer**: Prisma for data access, Redis for caching, BullMQ for async jobs, Supabase SDK for auth/storage.\n\nAll services run as a monolithic NestJS application for simplicity, with clear module boundaries. The backend does **not** expose GraphQL – REST API with versioning ensures mobile and web clients consume the same endpoints. Real-time features (slot locks) are implemented via short polling with Redis-backed atomic operations.\n\n## 4. Service Boundaries\nEach NestJS module is a self-contained domain with its own service, controller, and DTOs. Dependencies between modules are explicit and injected via interfaces:\n- **auth**: validates JWT, provides `AuthGuard` and `RolesGuard`. No business logic.\n- **businesses**: manages business profiles, services, staff, working hours. Emits events when business hours or services change (to trigger slot cache invalidation).\n- **bookings**: core booking engine. Exposes endpoints for slot queries (`GET /businesses/:id/slots`), booking creation, confirmation, cancellation, rescheduling. Depends on `businesses` and `redis` modules.\n- **slot-calculator** (internal service within bookings): algorithmic computation of available time windows based on business hours, staff schedules, existing bookings, and buffer times. Outputs slots as ISO datetime strings.\n- **search**: uses PostgreSQL full-text search (`tsvector`) and PostGIS for radius-based filtering. Combines with business/service metadata.\n- **categories**: tree structure with parent–child relationships for service categorization.\n- **reviews** & **favorites**: simple CRUD with aggregation.\n- **notifications**: listens for booking events (created, cancelled, reminder) and pushes jobs to BullMQ for email (SendGrid), push (FCM/APNs), SMS (Twilio).\n- **media**: generates presigned upload URLs via Supabase Storage, stores object keys in DB.\n\nAll modules expose their domain types through `shared-types` package (generated from Zod schemas and Prisma models). The API client library (`api-client`) is auto-generated from Swagger/OpenAPI spec and shared across clients.\n\n## 6. Data Flow and API Design\n### 6.1 REST Endpoints (versioned `/api/v1`)\n- **Public**: `GET /businesses`, `GET /businesses/:id`, `GET /search?q=...`, `GET /categories`, `GET /businesses/:id/reviews`.\n- **Authenticated (Customer)**: `GET /me`, `PATCH /me`, `POST /bookings`, `GET /bookings`, `PATCH /bookings/:id/cancel`, `PATCH /bookings/:id/reschedule`, `POST /favorites`, `GET /favorites`, etc.\n- **Provider/Admin**: role-based endpoints for business management, staff CRUD, analytics. (scoped under `/provider/` and `/admin/`).\n\nAll responses follow a consistent envelope: `{ data, meta, status }`. Pagination uses cursor-based or offset (cursor for infinite scroll, offset for simpler lists). Errors return Problem Details (RFC 7807).\n\n### 6.2 Request Flow Example (Booking Creation)\n1. Mobile app: user selects service, staff (optional), date. App calls `GET /businesses/:id/slots?date=...&serviceId=...&staffId=...`.\n2. Backend `bookings` service queries Redis cache for precomputed slots; if miss, calculates from DB (business hours, staff, existing bookings) and caches for 60 seconds.\n3. Mobile displays slots. User picks one and submits booking request `POST /bookings` with selected slot and any intake answers.\n4. Booking service validates slot availability again by checking Redis lock key `slot:lock:<business>:<staff>:<start>`. If free, sets a 5-minute lock (atomic `SET NX EX`). If lock already held, returns conflict.\n5. After successful lock, creates a provisional booking record with status `PENDING`. The payment intent is created if payment is required (Stripe integration) and returned to client.\n6. Upon successful payment (or non-payment confirmation), client calls `POST /bookings/:id/confirm`. Backend changes status to `CONFIRMED` and removes lock early. A BullMQ job is queued to send confirmation email/push, recalculate slot cache, and schedule reminders.\n7. If lock expires before confirmation, the provisional booking is voided by a scheduling job.\n\n## 7. Database Design (PostgreSQL + PostGIS)\nPrisma schema (excerpt):\n```prisma\nmodel User {\n  id            String   @id @default(uuid())\n  supabaseId    String   @unique\n  email         String   @unique\n  firstName     String\n  lastName      String?\n  phone         String?\n  role          Role     @default(CUSTOMER)\n  // relations\n  favorites     Favorite[]\n  bookings      Booking[]\n  reviews       Review[]\n  // profile image from Supabase\n  avatarUrl     String?\n}\n\nmodel Business {\n  id            String   @id @default(uuid())\n  name          String\n  slug          String   @unique\n  description   String?\n  address       String\n  coordinates   Unsupported(\"geometry(Point, 4326)\")? // PostGIS\n  radiusKm      Float?   // service radius\n  phone         String?\n  // relations\n  ownerId       String?\n  owner         User?    @relation(fields: [ownerId])\n  hours         WorkingHour[]\n  services      Service[]\n  staff         Staff[]\n  images        BusinessImage[]\n  categories    BusinessCategory[]\n  reviews       Review[]\n}\n\nmodel WorkingHour {\n  // dayOfWeek (0–6), open/close times, breaks JSON\n}\n\nmodel Service {\n  id            String   @id @default(uuid())\n  businessId    String\n  name          String\n  durationMin   Int\n  price         Decimal\n  categoryId    String?\n  // ...\n}\n\nmodel Staff {\n  id            String   @id @default(uuid())\n  businessId    String\n  name          String\n  specialty     String?\n  // schedule override: staff-specific hours/breaks\n  schedule      StaffSchedule?\n  // ...\n}\n\nmodel Booking {\n  id            String   @id @default(uuid())\n  customerId    String\n  businessId    String\n  serviceId     String\n  staffId       String?\n  startTime     DateTime @db.Timestamptz\n  endTime       DateTime @db.Timestamptz\n  status        BookingStatus @default(PENDING)\n  // ...\n}\n```\nPostGIS usage: `coordinates` field stores `POINT(lng lat)` with SRID 4326. Search endpoints use `ST_DWithin` and `ST_DistanceSphere` for radius filtering and ordering by distance. Indexes: GIST on coordinates, GIN on full-text search column (generated from business name + service names).\n\n## 8. Caching & Real-Time Slot Locking (Redis)\nRedis is crucial for performance and consistency:\n- **Slot cache**: precomputed available slots for each (business, date, service, staff optional) combination. Cached as JSON arrays with TTL of 60 seconds. Invalidated by any booking creation/cancellation or business hours change.\n- **Slot locks**: key pattern `slot:lock:{businessId}:{staffId}:{startTime}` with value of a unique request ID and TTL 300 seconds. Used to prevent double booking during checkout. Released upon confirmation or expiry.\n- **Session store**: (optional) for rate limiting.\n- **Rate limiting**: Redis-backed sliding window for API endpoints.\n- **Leaderboard**: not applicable, but could store trending businesses.\n\n## 9. Authentication & Authorization Flow\n1. Customers/providers sign up/in via Supabase Auth (email/password, Google, Apple). The Supabase client returns a JWT.\n2. The mobile/web app attaches the JWT as `Authorization: Bearer <token>` on every request.\n3. NestJS `SupabaseJwtAuthGuard` validates the token using Supabase's public key. It extracts the `sub` (Supabase user ID) and metadata (role) and injects a `UserPrincipal` into the request context.\n4. Backend resolves the internal database user via the `sub` field (ensures sync). An interceptor auto-creates the `User` record on first API call if it doesn't exist (after signup).\n5. `RolesGuard` checks the user's role for provider/admin endpoints.\n\n## 10. Mobile App Architecture (Customer)\n- **Expo Router**: file-based routing in `app/` directory. Uses deep linking for booking flows (e.g., `business/[id]/book`).\n- **State Management**: TanStack React Query for all server state. Query keys modeled per domain (`['business', businessId]`, `['slots', {...}]`). Optimistic updates for favorite toggles and cancellation status.\n- **Local state**: minimal, handled by React Context (auth context from Supabase).\n- **UI**: Shared UI kit from `libs/shared-ui` using React Native Reanimated for smooth transitions (tab switches, calendar animations).\n- **Maps**: Use `react-native-maps` with custom markers; map view toggled on search screen.\n- **Permissions**: Location, push notifications. Expo modules manage permissions.\n- **Offline**: limited offline support; React Query's `networkMode: 'offlineFirst'` shows cached data when disconnected.\n\n## 11. Web Portals (Provider & Admin)\nBuilt as separate Vite React SPAs, reusing:\n- `api-client` for typed HTTP calls.\n- `shared-types` for DTO validation.\n- `shared-ui` adapted for React DOM (with react-native-web for some components if reused).\n\nProvider portal includes full business management (services, staff, schedules, booking calendar, etc.). Admin dashboard provides platform analytics, moderation, and category management.\n\n## 12. Background Jobs & Asynchronous Processing (BullMQ)\nAll time-consuming or side-effect tasks are offloaded to BullMQ queues processed by the same NestJS process (or separate workers):\n- **slot-recalculation**: triggered when business hours, staff schedules, or bookings change. It bulk-computes slot caches for affected time windows.\n- **send-email**, **send-push**, **send-sms**: notification dispatchers.\n- **reminder-scheduler**: scans upcoming bookings and enqueues reminders.\n- **cleanup-expired-locks**: periodic job to release stale slot locks.\n\n## 13. File Storage\nUser avatars, business images are stored in Supabase Storage. The backend media module generates presigned upload URLs (or clients use the Supabase SDK directly, then inform backend). Object paths stored in DB for easy retrieval. Public images served via Supabase CDN.\n\n## 14. Local Development & Testing\n- **Docker Compose** (docker-compose.yml) spins up PostgreSQL (with PostGIS), Redis, and a Supabase local stack (via `supabase start`).\n- `pnpm dev` starts all apps in parallel using Nx.\n- **Testing**:\n  - Jest for unit tests (each module, lib).\n  - Integration tests for API using Supertest with a test database.\n  - Mobile: unit tests with Jest, component tests with React Native Testing Library, E2E with Detox (planned).\n\n## 15. CI/CD Pipeline (GitHub Actions + EAS Build)\n- **Backend & Web**: GitHub Actions on push to `main`/\n  - Lint, type-check, test\n  - Build Docker image, push to container registry\n  - Deploy to staging/production (cloud provider via IaC)\n- **Mobile (Customer)**:\n  - EAS Build triggers on version tags.\n  - Expo OTA updates for non-native changes.\n  - Staging builds distributed via TestFlight/Internal Testing.\n\n## 16. Security Considerations\n- All endpoints enforce authentication/authorization.\n- CSRF protection for web clients (if same-domain, cookies).\n- Rate limiting on booking endpoints to prevent abuse.\n- Input validation using Zod DTOs (integrated with NestJS pipes).\n- SQL injection prevented by Prisma.\n- Sensitive data (tokens) never stored in mobile AsyncStorage; use SecureStore.\n- Payment data handled via Stripe SDK, never touches our server.\n- GDPR compliance: account deletion APIs, data export.\n\n## 17. Monitoring & Observability\n- Structured JSON logging (pino) with correlation IDs.\n- Error tracking via Sentry (integrated in mobile and backend).\n- Health check endpoint for orchestration.\n- API performance metrics with Prometheus / Grafana (future).\n\nThis architecture provides a scalable, maintainable foundation for Planity Clone, balancing rapid mobile development with robust backend services.",
