Monolith**: Single deployable API with clear module boundaries; can be split into microservices later.\n- **Supabase for Auth & Storage**: Offloads authentication (social login, JWT) and file storage, reducing backend complexity.\n- **PostGIS for Geospatial**: Enables location-based search, distance calculation, and map clustering.\n- **Redis for Caching & Queues**: Caches category tree, business details, and availability slots; powers Bull job queues for notifications and slot pre-computation.\n- **TanStack Query**: Manages server state, caching, and optimistic updates on the mobile side.\n- **Expo Router**: File-based routing with deep linking support.\n\n## 3. Technology Stack\n\n### 3.1 Mobile App\n- **Framework**: Expo SDK 50+ with React Native\n- **Language**: TypeScript (strict mode)\n- **Navigation**: Expo Router (file-based routing)\n- **State Management**: TanStack React Query for server state, React Context for UI state\n- **Animations**: React Native Reanimated 3\n- **Maps**: react-native-maps with custom clustering\n- **Auth**: Supabase JS client (handles social login, token refresh)\n- **Payments**: Stripe React Native SDK (if prepayment required)\n- **Testing**: Jest, React Native Testing Library\n\n### 3.2 Backend\n- **Runtime**: Node.js 20 LTS\n- **Framework**: NestJS 10\n- **Language**: TypeScript\n- **ORM**: Prisma 5\n- **Database**: PostgreSQL 15 + PostGIS extension\n- **Cache & Queue**: Redis 7 (ioredis, BullMQ)\n- **Auth Integration**: Supabase Auth (verify JWT, manage user metadata)\n- **File Storage**: Supabase Storage (avatars, business photos)\n- **Validation**: class-validator + class-transformer\n- **API Documentation**: Swagger (NestJS OpenAPI)\n- **Testing**: Jest, Supertest\n\n### 3.3 Infrastructure & DevOps\n- **Monorepo**: Nx 18, pnpm workspaces\n- **Containerization**: Docker Compose for local dev (PostgreSQL, Redis, Supabase local stack)\n- **CI/CD**: GitHub Actions (lint, test, build, deploy)\n- **Mobile Builds**: EAS Build (Expo Application Services)\n- **Hosting**: Backend on AWS ECS/Fargate or Railway; Supabase Cloud for auth/storage/database\n- **Monitoring**: Sentry for errors, Axiom/Datadog for logs\n\n## 4. Service Boundaries & Modules\n\n### 5.1 Mobile App (Expo)\n- **Pages (Expo Router)**:\n  - `(auth)/login`, `(auth)/register`, `(auth)/forgot-password`\n  - `(tabs)/home` – feed, search\n  - `(tabs)/explore` – map view\n  - `(tabs)/bookings` – upcoming/past\n  - `(tabs)/profile` – user settings\n  - `business/[id]` – detail view\n  - `booking/` – multi-step flow\n- **State Management**: TanStack Query for all API data; optimistic updates for favorites and booking.\n- **Auth Flow**: Supabase client handles sign-up/in, token refresh; stores session in secure storage (expo-secure-store).\n- **Maps**: Custom hook `useMapBusinesses` fetches businesses within viewport bounds using PostGIS query.\n\n### 5.2 Backend Modules (NestJS)\nEach module follows NestJS best practices: controller, service, repository (Prisma), DTOs, and guards.\n\n#### Auth Module\n- Integrates with Supabase Auth: verifies JWT on every request using `SupabaseAuthGuard`.\n- Endpoints: `POST /auth/register` (creates user in Supabase + our DB), `POST /auth/login` (delegates to Supabase), `POST /auth/refresh`, `POST /auth/logout`.\n- Manages user roles (customer, provider, admin) stored in our `User` table.\n\n#### Business Module\n- CRUD for businesses (provider/admin).\n- Search: `GET /businesses?query=&category=&lat=&lng=&radius=&date=&sort=`\n  - Uses PostgreSQL full-text search (`tsvector`) and PostGIS `ST_DWithin` for location filtering.\n  - Caches results in Redis (TTL 5 min).\n- Detail: `GET /businesses/:id` with services, staff, reviews.\n- Favorites: `POST /businesses/:id/favorite`, `DELETE /businesses/:id/favorite`.\n\n#### Booking Module\n- Slot computation engine: `GET /businesses/:id/slots?serviceId=&staffId=&date=`\n  - Real-time calculation considering business hours, staff schedules, existing bookings, buffers.\n  - Uses Redis to cache pre-computed slots for 7 days; invalidates on new booking.\n- Booking creation: `POST /bookings` with optimistic concurrency control (version field on `StaffAvailability`).\n- Manage: `GET /bookings`, `PATCH /bookings/:id` (cancel, reschedule), `POST /bookings/:id/review`.\n\n#### User Module\n- Profile management: `GET/PATCH /users/me`.\n- Payment methods: stored via Stripe customer ID; no raw card data.\n- Notification preferences.\n\n#### Notification Module\n- Sends push notifications via Expo Push API or Firebase Cloud Messaging.\n- Email via Supabase or SendGrid.\n- Triggered by booking events (confirmation, reminder, cancellation).\n\n#### Admin Module\n- Category management (hierarchical).\n- Business verification, user management.\n- Analytics dashboards (future).\n\n### 5.3 Background Jobs (BullMQ)\n- **Slot Pre-computation**: Runs nightly to cache next 30 days of slots for all active businesses.\n- **Reminder Notifications**: Checks upcoming appointments and sends 24h/1h reminders.\n- **Data Cleanup**: Anonymizes deleted user data after grace period.\n- **Category Tree Cache Refresh**: Updates Redis cache when admin modifies categories.\n\n## 6.  // PostGIS\n  phone         String?\n  email         String?\n  website       String?\n  photos        String[]  // Supabase Storage URLs\n  hours         Json      // weekly schedule\n  services      Service[]\n  staffs        Staff[]\n  reviews       Review[]\n  favorites     Favorite[]\n  createdAt     DateTime  @default(now())\n}\n\nmodel Category {\n  id        String     @id @default(uuid())\n  name      String\n  icon      String?\n  parentId  String?\n  parent    Category?     // date-specific overrides\n  bookings    Booking[]\n}\n\nmodel Booking {\n  id          String    @id @default(uuid())\n  userId      String\n  user        User      @relation(fields: [userId], references: [id])\n  businessId  String\n  business    Business  @relation(fields: [businessId], references: [id])\n  serviceId   String\n  service     Service   @relation(fields: [serviceId], references: [id])\n  staffId     String?\n  staff       Staff? API Design\n\n### 7.1 RESTful Conventions\n- Base URL: `https://api.planity-clone.com/v1`\n- Authentication: Bearer token (Supabase JWT) in `Authorization` header.\n- Versioning via URL path (`/v1/`).\n- Consistent error format:\n```json\n{\n  \"statusCode\": 400,\n  \"message\": \"Validation failed\",\n  \"errors\": [...]\n}\n```\n\n### 7.2 Key Endpoints\n| Method | Path | Description |\n|--------|------|-------------|\n| POST | /auth/register | Create account (calls Supabase + DB) |\n| POST | /auth/login | Login (delegates to Supabase) |\n| GET | /businesses | Search & filter businesses |\n| GET | /businesses/:id | Business detail |\n| GET | /businesses/:id/slots | Available time slots |\n| POST | /bookings | Create booking |\n| GET | /bookings | User's bookings |\n| PATCH | /bookings/:id | Cancel/reschedule |\n| POST | /businesses/:id/favorite | Add favorite |\n| DELETE | /businesses/:id/favorite | Remove favorite |\n| GET | /users/me | Profile |\n| PATCH | /users/me | Update profile |\n| GET | /categories | Category tree |\n\n### 7.3 Slot Computation Algorithm\n1. Fetch business hours, staff schedule, service duration + buffers.\n2. Generate all possible start times within requested date range.\n3. Remove slots conflicting with existing bookings (using SQL overlap query).\n4. Apply staff-specific overrides (time off, custom hours).\n5. If multiple staff, merge and sort by earliest.\n6. Cache result in Redis with TTL until next booking or midnight.\n\n## 8. Authentication & Authorization\n\n### 8.1 Flow\n- **Sign Up**: Mobile app calls Supabase Auth `signUp()`. On success, backend webhook (or direct call) creates a `User` record in our DB with the `supabaseId`.\n- **Login**: Supabase Auth returns JWT (access + refresh tokens). Mobile stores session securely.\n- **API Requests**: NestJS `SupabaseAuthGuard` extracts token, verifies with Supabase public key, and attaches user context (role, id).\n- **Role-Based Access**: Custom decorators `@Roles(Role.PROVIDER)` and guard check user role from DB.\n\n### 8.2 Token Refresh\n- Supabase client automatically refreshes tokens. Mobile interceptor retries failed requests with new token.\n\n## 9. Real-time & Caching Strategy\n\n### 9.1 Real-time Features (Future)\n- Booking status updates: Could use Supabase Realtime (websockets) to notify users of changes.\n- Provider calendar sync: Polling or websockets.\n\n### 9.2 Caching\n- **Mobile**: TanStack Query caches API responses with stale-while-revalidate. Optimistic updates for favorites and booking cancellation.\n- **Backend**: Redis caches heavy queries (search, slots, categories). Cache invalidation via events (new booking, business update).\n\n## 10. Background Jobs\n\n### 10.1 Job Queue (BullMQ)\n- **slot-precompute**: Nightly job for all active businesses; stores slots in Redis.\n- **send-reminder**: Checks appointments starting in 24h/1h, sends push/email.\n- **cleanup-expired**: Deletes unverified accounts after 7 days.\n- **category-cache-refresh**: On category change event.\n\n### 10.2 Scheduling\n- Use `@nestjs/bull` to register queues and processors.\n- Cron-like scheduling via `BullMQ` repeatable jobs.\n\n## 11. Deployment & CI/CD\n\n### 11.1 Local Development\n- `docker-compose up` starts PostgreSQL (with PostGIS), Redis, and Supabase local stack (using `supabase start`).\n- `pnpm dev:api` runs NestJS with hot reload.\n- `pnpm dev:mobile` starts Expo dev client.\n\n### 11.2 CI/CD Pipeline (GitHub Actions)\n- **Pull Request**: Lint, type-check, unit tests (Jest), build mobile (EAS Build for preview).\n- **Main Branch**: Run integration tests, build API Docker image, push to container registry, deploy to staging.\n- **Release**: Tagged release triggers production deployment (API to AWS ECS, mobile OTA update via EAS Update).\n\n### 11.3 Mobile Builds\n- EAS Build creates development, preview, and production builds.\n- Over-the-air updates via EAS Update for non-native changes.\n\n### 11.4 Backend Deployment\n- NestJS app containerized with Docker.\n- Deployed to AWS ECS Fargate or Railway.app.\n- Environment variables for Supabase URL, keys, database URL, Redis URL.\n- Database migrations run as part of deployment (Prisma Migrate).\n\n## 12. Testing Strategy\n\n### 12.1 Unit Tests\n- **Mobile**: Jest + React Native Testing Library for components, hooks, utils.\n- **Backend**: Jest for services, guards, pipes; mock Prisma and Redis.\n\n### 12.2 Integration Tests\n- **API**: Supertest with test database (Dockerized PostgreSQL). Test booking flow, slot computation, auth.\n- **Mobile**: Mock Service Worker (MSW) to mock API; test user flows.\n\n### 12.3 E2E Tests (Future)\n- Detox or Maestro for critical paths (booking, login).\n\n## 13. Security Considerations\n- All traffic over HTTPS.\n- JWT validation on every request.\n- Input validation and sanitization (class-validator).\n- Rate limiting on auth and booking endpoints.\n- SQL injection prevention via Prisma parameterized queries.\n- File uploads scanned and restricted to images.\n- Sensitive data (API keys) stored in environment variables, never in code.\n\n## 14.
