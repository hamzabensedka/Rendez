Architecture Principles\\n\\n- **Type Safety End-to-End:** Shared TypeScript types from a central `@planity/types` package.\\n- **Separation of Concerns:** Frontend (UI/UX) and backend (business logic, data) are loosely coupled via a RESTful API.\\n- **Domain-Driven Design (Light):** Backend modules align with business domains (Auth, Business, Booking, etc.).\\n- **Offline-Unfriendly:** The mobile app requires network connectivity; no offline-first mandate, but React Query provides stale-while-revalidate caching.\\n- **Scalability:** Stateless NestJS services can be horizontally scaled; PostgreSQL can be read-replicated; Redis handles session/cache; BullMQ processes background jobs asynchronously.\\n- **Monorepo & Code Sharing:** Nx workspace with pnpm. Shared libraries for types, design tokens, UI components, and utilities are consumed by all apps.\\n\\n## 3. Service Boundaries (Backend Modules)\\n\\nThe NestJS application is structured as a **modular monolith** with clear domain boundaries, ready to be split into microservices if load increases. Each module exposes its own service and optionally a controller.\\n\\n| Module         | Responsibility                                                                 | Key Prisma Models / Tables                 |\\n|----------------|--------------------------------------------------------------------------------|--------------------------------------------|\\n| Auth           | User registration, login, email verification, OTP, token refresh, logout.      | User, RefreshToken, VerificationToken      |\\n| User           | Profile management, favourites, search history, guest -> authenticated merge.  | User, UserFavourite, SearchHistory         |\\n| Business       | Business CRUD, service catalog, staff management, opening hours, photos.       | Business, Service, StaffMember, BusinessPhoto |\\n| Booking        | Slot availability, appointment creation/cancellation, status updates, booking validation (no overlaps). | Appointment, AppointmentServiceM2M, Slot (computed) |\\n| Payment        | Payment intent creation, capture, refund, webhook handling from Stripe.        | Payment                                   |\\n| Notification   | Push notification dispatch, email sending, in-app notification storage.        | Notification, NotificationQueue            |\\n| Admin          | Provider approval, platform analytics, user reports, flagging.                 | AdminAuditLog, ProviderVerification        |\\n\\nEach module has its own\\n- `*.module.ts`\\n- `*.service.ts`\\n- `*.controller.ts` (if API endpoints)\\n- DTOs and validators (using `class-validator`)\\n- Prisma service injection (via shared `PrismaModule`)\\n\\n**Cross-Module Communication:** Modules communicate only through the public interfaces of other services (injected via NestJS DI). No direct database access across modules. Events (e.g., `booking.confirmed`) are emitted with NestJS EventEmitter and consumed by other modules (e.g., Notification listens for booking confirmed to send push).\\n\\n## 7. Data Flow & API Design\\n\\n**REST API** with JSON payloads. Endpoints are versioned (`/api/v1/`).\\n\\n- **Authentication:** `/api/v1/auth/*` (Supabase Auth directly used for login/register; the NestJS API validates Supabase JWT on subsequent requests).\\n- **Business Search:** `GET /api/v1/businesses?lat=...&lng=...&radius=...&category=...&sort=...`\\n  - Uses PostGIS `ST_DWithin` and `ST_Distance` for location filtering and distance calculation.\\n  - Full-text search with PostgreSQL `tsvector` on business name, service name, description.\\n  - Paginated with cursor-based pagination (keyset).\\n- **Booking:** `POST /api/v1/appointments` – accepts selected services, staff, datetime; validates availability using a transactional lock (advisory lock) to prevent double-booking.\\n- **Favourites:** `POST /api/v1/users/me/favourites` – upserts a favourite; merges local guest favourites from request body on first login.\\n\\n**Mobile Data Flow:**\\n1. React Query hook triggers a fetch to the API.\\n2. API client (axios/fetch) adds Supabase JWT via interceptor.\\n3. NestJS extracts user from JWT, validates permissions via guard, processes request.\\n4. Response cached in React Query with stale time and revalidation strategies.\\n5. Optimistic updates used for mutation (like favouriting) to provide instant feedback.\\n\\n## 8. Authentication & Security\\n\\n- **Provider:** Supabase Auth (GoTrue). Supports email/password, Google, Apple, phone OTP.\\n- **Session Management:**\\n  - Access token (JWT) with 15-minute expiry.\\n  - Refresh token (7 days) stored securely in Supabase; on mobile, it sits in the secure storage (Expo SecureStore).\\n  - Supabase client SDK automatically handles token refresh.\\n- **NestJS Integration:**\\n  - `SupabaseGuard` validates the JWT from the `Authorization` header.\\n  - `RolesGuard` checks user role (`CUSTOMER`, `PROVIDER`, `ADMIN`) from the decoded JWT.\\n  - `ProviderApprovalGuard` ensures a provider’s business is verified before accessing provider-only endpoints.\\n- **Rate Limiting:** Implemented with `@nestjs/throttler` backed by Redis. Different limits per endpoint (e.g., 5 login attempts/minute).\\n- **Input Validation:** Global validation pipe with `class-validator` and `class-transformer`.\\n\\n## 9. Caching Strategy\\n\\n**Redis as cache layer:**\\n- **API Response Caching:** Frequently requested business lists, category trees, and search suggestions are cached with TTL (e.g., 5 minutes). Cache invalidation on business update via NestJS `@CacheInterceptor` or manual invalidation in service methods.\\n- **Session Store:** (Handled by Supabase, but if custom, Redis).\\n- **Rate Limiting:** Redis-backed Throttler.\\n- **Job Queues:** BullMQ uses Redis.\\n- **Local Frontend Cache:** React Query provides memory cache; we configure `staleTime` and `gcTime` appropriately for each query type.\\n\\n## 10. Database Schema & PostGIS\\n\\nPrisma schema defined inside `apps/api/prisma/schema.prisma`. Key aspects:\\n\\n- **User & Profile:** `User` table with `role` enum, `providerProfile` one-to-one relation.\\n- **Business & Location:**\\n  - `Business` model includes `coordinates Geography(Point, 4326)` using PostGIS.\\n  - Indexes: GiST on `coordinates` for spatial queries.\\n- **Service & Staff:** Many-to-many between `StaffMember` and `Service` (via `StaffService`).\\n- **Appointment:** Links `customerUser`, `business`, `staffMember`, plus a JSON array of service IDs (or a join table). Includes `startDateTime` and `endDateTime` timestamps.\\n- **Booking Constraints:** A unique composite index on `(staffMemberId, startDateTime)` to prevent overlapping appointments. Additionally, the booking service uses database transactions with `SELECT ... FOR UPDATE` or advisory locks when creating an appointment.\\n- **Search Optimisation:**\\n  - Generated `tsvector` column on business `name`, `description`, and `services.name`.\\n  - Gin index on `tsvector` for full-text search.\\n  - Materialised view for `search_index` refreshed periodically or by triggers.\\n\\n## 11. Background Jobs & Queue Management\\n\\n**BullMQ** queues backed by Redis:\\n\\n| Queue                | Job Types                                                       | Processor Module       |\\n|----------------------|-----------------------------------------------------------------|------------------------|\\n| notifications        | Send push, send email, send SMS                                 | NotificationService    |\\n| reminders            | Appointment reminder (24h, 1h before)                           | NotificationService    |\\n| payment-capture      | Capture authorised payment (e.g., after service completion)     | PaymentService         |\\n| search-reindex       | Rebuild search materialised view                                | BusinessService        |\\n| admin-tasks          | Provider verification follow-ups, data cleanup                  | AdminService           |\\n\\nJobs are added via NestJS Queue services. Processors are defined as separate classes with `@Processor` decorator. Failure handling with retries and dead-letter queue.\\n\\n## 12. Deployment & CI/CD\\n\\n- **GitHub Actions:**\\n  - **Pull Request:** Lint (ESLint), type check, unit tests (Jest), integration tests (Supertest for API).\\n  - **Main Branch:** Build Docker images for API, push to container registry. Trigger EAS Build for mobile. Testing Strategy\\n\\n- **Unit Tests:**\\n  - Backend: Jest for services, guards, utilities. Prisma mocked or in-memory SQLite.\\n  - Frontend: Jest + React Native Testing Library for components and hooks. Mock API calls.\\n- **Integration Tests:**\\n  - API endpoint tests using Supertest with a real test database (PostgreSQL in CI via Docker service). Key Design Decisions\\n\\n1. **Monorepo over separate repos:** Facilitates code sharing (types, UI) and unified tooling, reducing drift.\\n2. **Supabase Auth over custom implementation:** Faster time-to-market, robust security, built-in social login. Custom auth would be redundant.\\n3. **PostGIS over separate search engine (Elasticsearch):** Sufficient for MVP-scale geospatial search; PostgreSQL full-text search is adequate, and PostGIS offers rich spatial queries without operational overhead.\\n4. **Modular monolith over immediate microservices:** Simplifies development, testing, and deployment; boundaries are clear and can be extracted later if traffic demands.\\n5. **BullMQ + Redis over RabbitMQ:** Redis is already in the stack; BullMQ provides a mature, performant job queue with nice NestJS integration.\\n6. **Expo Router over React Navigation standalone:** File-based routing aligns with Expo ecosystem and simplifies deep linking; typed routes reduce errors.\\n7. **NativeWind for mobile styling:** Allows reuse of Tailwind utility classes and design tokens, maintaining consistency with the admin web.\\n\\n## 16.
