The architecture follows clean architecture principles, with clear separation of concerns, modular design, and scalability in mind. The system is built using a monorepo managed by Nx, with a React Native (Expo) frontend and a NestJS backend, communicating via REST and WebSockets.\n\n## 2. Service Boundaries\n\n### 4.1 Frontend (Mobile App)\n\nThe mobile app is built with Expo and uses Expo Router for navigation. It is divided into:\n\n- **Pages**: Each screen is a file in `app/` directory. Expo Router handles routing and deep links.\n- **Components**: Reusable UI components (buttons, cards, modals).\n- **Hooks**: Custom hooks for business logic (e.g., `useBooking`, `useSearch`).\n- **Services**: API client using TanStack React Query for data fetching and caching.\n\n**Data Flow**:\n- User actions → React Query mutation → API call → Cache invalidation → UI update.\n- Real-time updates (e.g., booking confirmation) via WebSocket or polling.\n\n### 4.2 Backend (API)\n\nThe NestJS backend is organized into feature modules. Each module encapsulates its own controllers, services, DTOs, and Prisma queries.\n\n**Modules**:\n- **Auth Module**: Handles registration, login, OAuth, JWT tokens, refresh tokens, biometric authentication.\n- **Users Module**: User profiles, preferences, account management.\n- **Businesses Module**: Business CRUD, staff management, hours, photos.\n- **Services Module**: Service catalog, categories, dynamic pricing.\n- **Bookings Module**: Booking flow, availability, slot management, calendar.\n- **Reviews Module**: Review creation, moderation, rating aggregation.\n- **Search Module**: Full-text search, geospatial queries (PostGIS), autocomplete, filters.\n- **Payments Module**: Payment processing (Stripe), deposits, refunds.\n- **Admin Module**: Platform analytics, user management, content moderation.\n\n**Communication**:\n- REST API for CRUD operations.\n- WebSocket (Socket.IO) for real-time notifications (booking confirmations, reminders).\n- Redis for caching frequently accessed data (business details, search results).\n\n### 4.3 Database\n\nPostgreSQL with PostGIS extension for geospatial queries. Prisma ORM provides type-safe database access.\n\n**Key Tables**:\n- `users` (id, email, password_hash, role, profile)\n- `businesses` (id, name, description, location (PostGIS geometry), hours)\n- `services` (id, business_id, name, duration, price, category)\n- `staff` (id, business_id, name, specialties, schedule)\n- `bookings` (id, user_id, business_id, service_id, staff_id, start_time, end_time, status)\n- `reviews` (id, user_id, business_id, rating, text)\n- `categories` (id, name, parent_id)\n\n**Indexes**:\n- GiST index on `businesses.location` for spatial queries.\n- B-tree indexes on foreign keys and frequently filtered columns (status, date).\n- Full-text search index on business name and service name.\n\n### 4.4 Caching (Redis)\n\nRedis is used for:\n- Session store (JWT refresh tokens).\n- Rate limiting (per user/IP).\n- Caching search results and business details (TTL-based).\n- Queue for background jobs (email notifications, reminders).\n\n## 5. Key Architectural Decisions\n\n### 5.1 Monorepo with Nx\n\n- **Why**: Shared types between frontend and backend, unified build and test pipelines, code generation.\n- **How**: Nx manages dependencies, caching, and task orchestration. pnpm for package management.\n\n### 5.2 TanStack React Query for Server State\n\n- **Why**: Automatic caching, background refetching, optimistic updates for booking flow.\n- **How**: Each API endpoint has a corresponding query hook. Mutations invalidate related queries.\n\n### 5.3 Expo Router for Navigation\n\n- **Why**: File-based routing simplifies navigation structure, supports deep linking for sharing businesses.\n- **How**: Pages are organized in folders. Dynamic routes for business detail (`/business/[id]`).\n\n### 5.4 PostGIS for Geospatial Search\n\n- **Why**: Efficient location-based queries (nearby businesses, map clustering).\n- **How**: Use ST_DWithin for radius search, ST_ClusterDBSCAN for clustering.\n\n### 5.5 Prisma with PostgreSQL\n\n- **Why**: Type-safe queries, migrations, and schema management.\n- **How**: Prisma schema defines models. Migrations are version-controlled.\n\n## 6. Security Considerations\n\n- JWT tokens with short expiry (15 min) and refresh tokens (7 days) stored securely.\n- Biometric authentication on device (Face ID/Touch ID) for quick re-auth.\n- Rate limiting on auth endpoints (5 attempts per minute).\n- Input validation using Zod on both frontend and backend.\n- HTTPS enforced in production.\n- Environment variables for secrets (DB URL, JWT secret, API keys).\n\n## 7. Performance Targets\n\n- Search results < 500ms (95th percentile).\n- Business detail page load < 1.5s (above the fold).\n- Map viewport load < 2s on 4G.\n- Booking confirmation < 2s.\n- App cold start < 3s.\n\n## 8. CI/CD Pipeline\n\nGitHub Actions workflows:\n- **CI**: Run lint, type check, unit tests on pull request.\n- **CD**: On merge to main, build and deploy backend to staging, then production.\n- **EAS Build**: Triggered manually or on tag for mobile app builds (iOS/Android).\n\n## 9. Local Development\n\nDocker Compose provides:\n- PostgreSQL with PostGIS\n- Redis\n- NestJS API (with hot reload)\n\nCommands:\n```bash\npnpm install\npnpm nx serve api\npnpm nx start mobile\n```\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions and services.\n- **Integration Tests**: Test API endpoints with supertest.\n- **E2E Tests**: Detox or Maestro for mobile (future).\n- **Coverage**: Minimum 80% for critical modules (auth, bookings, payments).\n\n## 11. Future Considerations\n\n- Microservices migration when needed (e.g., separate search service).\n- GraphQL federation for complex queries.\n- Offline-first support with WatermelonDB or similar.\n- Push notifications via Firebase Cloud Messaging.\n\n---\n\nThis architecture provides a solid foundation for building a scalable, maintainable mobile application that connects customers with local service businesses.",
