  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nPlanity Clone is a mobile-first appointment booking platform connecting users with local businesses. The system follows a clean, layered architecture with clear separation of concerns: presentation (Expo/React Native), application/services (NestJS), data (Prisma/PostgreSQL/PostGIS/Redis), and infrastructure (Docker, GitHub Actions, EAS Build). The monorepo is managed by Nx and uses pnpm for workspace package management.\n\n## 2. Tech Stack Summary\n- **Mobile**: Expo (SDK 49), React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Backend**: NestJS (Node.js), TypeORM via Prisma, PostgreSQL + PostGIS for geospatial queries, Redis for caching & pub/sub\n- **DevOps**: Docker Compose (local), GitHub Actions (CI/CD), EAS Build (app stores), Supabase (auth & storage fallback), BullMQ (background jobs)\n- **Monorepo**: Nx, pnpm\n- **Testing**: Jest (unit/integration), React Native Testing Library\n\n## 3. High-Level Architecture\n```\n+-------------------+      +---------------------+      +-------------------+\n|   Mobile App      | <---> |   API Gateway (Nest) | <---> |   Services (Nest) |\n|   |               |   Background Jobs   |<-------+   - Provider      |\n        |               |   (BullMQ + Redis)  |          +-------------------+\n        |               +---------------------+\n        |                         |\n        v                         v\n+-------------------+      +---------------------+\n|   Data Layer      |      |   Infra & CI/CD Data Flow Examples\n\n### 8.1 Business Search (Map)\n1. Mobile sends GET `/businesses/search?lat=&lng=&radius=&category=` (React Query).\n2. API validates params, uses PostGIS `ST_DistanceSphere` to filter within radius.\n3. Results cached in Redis (key: `search:{lat}:{lng}:{radius}:{category}`).\n4. Returns paginated list with distance.\n5. Mobile displays markers via `react-native-maps`; tapping opens detail screen.\n\n### 8.2 Booking Flow\n1. User selects service → mobile fetches availability via GET `/businesses/:id/availability?date=`.\n2. Backend computes slots: iterates over `Availability` rows, generates time slots, excludes existing `Booking` overlaps (range query).\n3. Mobile presents slot picker (Reanimated bottom sheet).\n4. On confirm, mobile POST `/bookings` with `serviceId`, `startTime`.\n5. Backend creates Booking with status `PENDING`, triggers payment intent creation (Stripe).\n6. Returns `clientSecret`; mobile completes payment via Stripe SDK.\n7. Webhook `payment.succeeded` → backend updates Booking to `CONFIRMED`, enqueues `reminder` job (24h before) and sends confirmation push/email.\n8. Mobile updates UI optimistically; React Query invalidates `bookings` query.\n\n### 8.3 Provider Availability Management\n1. Provider portal (protected route) GET `/provider/business/:id/availability`.\n2. Returns weekly recurrence rules; provider edits via drag‑&‑drop UI.\n3. On save, POST `/provider/business/:id/availability` upserts `Availability` rows.\n4. Backend invalidates cached availability for that business (Redis DEL).\n5. Slot generation jobs run nightly to refresh future slots.\n\n## 9. Security & Privacy\n- Data at rest: PostgreSQL encryption at volume level (cloud provider).\n- Data in transit: TLS 1.3 enforced via API gateway and Expo’s `https`.\n- PII: minimal storage (email, name, phone). GDPR‑ready endpoints for data export/deletion.\n- Passwords: bcrypt salt rounds 12.\n- JWT: short-lived (15 min) access token, refresh token rotated and stored HttpOnly cookie (if using cookies) or SecureStore.\n- Rate limiting: 100 req/IP/min for auth endpoints, 500 for others.\n- Input sanitization: class-validator whitelist, Prisma parameterized queries prevent SQLi.\n\n## 10. Internationalization & Localization\n- Mobile: `i18next` with react-native; JSON locale files.\n- Backend: error messages and enum values kept language‑agnostic; UI translates.\n- Date/Time: stored in UTC, displayed in user’s locale via `date-fns-tz`.\n\n## 12.
