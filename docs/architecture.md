  "content": "# Architecture Document: Planity Clone\n\n## 1. Introduction\n\nThis document defines the complete system architecture for Planity Clone, a mobile-first booking platform for beauty and wellness services. The architecture is designed to support high scalability, maintainability, and a clean separation of concerns across services. The stack leverages Expo (React Native) with TypeScript for cross-platform mobile, a NestJS backend, PostgreSQL with PostGIS, Redis, and Supabase for authentication and real-time features. Nx monorepo management, pnpm, Docker Compose, and GitHub Actions constitute the development and CI/CD infrastructure.\n\n## 2. System Architecture Overview\n\nThe system is built around a modular monolith backend and a singular mobile frontend, with separate background workers for asynchronous tasks.\n\n**High-Level Components:**\n- **Mobile App (Expo / React Native)** – user-facing client.\n- **API Gateway / BFF (NestJS)** – backend for frontend, handles all business logic, authentication, and integrations.\n- **Database (PostgreSQL + PostGIS)** – primary data store, managed via Supabase and accessed with Prisma ORM.\n- **Redis** – caching, session stores, bull queues, real-time slot reservation locks.\n- **Supabase** – provides managed PostgreSQL, authentication (GoTrue), storage, and real-time subscriptions.\n- **Background Workers** – NestJS standalone processes using Bull queues for sending notifications, releasing expired slot reservations, and triggering scheduled jobs.\n- **External Services** – Stripe for payments, email/SMS providers.\n\n**Architecture Diagram (Textual):**\n```\n[Mobile App (Expo)]\n    |\n    | Supabase Realtime (WebSocket) + REST API (NestJS)\n    |\n    v\n[NestJS API Gateway] ---> [PostgreSQL (Supabase)]\n    |                         |\n    |---> [Redis]             |\n    |---> [Bull Queue]        |\n    |---> [Stripe API]        |\n    |---> [Email/SMS]         |\n    |\n[Background Workers (NestJS)]\n```\n\n## 3. Technology Stack and Rationale\n\n- **Frontend:** Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated. Expo provides a managed workflow; Expo Router enables file-based routing; React Query handles server state, caching, and optimistic updates; Reanimated delivers performant animations.\n- **API:** NestJS (TypeScript), Prisma ORM, PostgreSQL, PostGIS, Redis, Bull. NestJS provides a modular, enterprise-ready backend; Prisma gives type-safe database access; PostGIS enables geospatial queries; Redis serves as a cache and queue broker; Bull manages background jobs.\n- **Authentication & Real-time:** Supabase Auth (GoTrue) for social/email login, secure refresh tokens, and real-time subscriptions for slot availability updates. Supabase is directly integrated with the PostgreSQL database, ensuring consistency.\n- **Monorepo & Build:** Nx with pnpm for efficient dependency management and task orchestration; Docker Compose for local development environments; GitHub Actions for CI/CD; EAS Build for mobile app distribution.\n- **Testing:** Jest for both frontend and backend unit/integration tests.\n\n## 4. Monorepo Structure (Nx + pnpm)\n\nThe repository is organised as an Nx integrated monorepo using pnpm workspaces. All applications and libraries live under the `apps/` and `libs/` directories.\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                 # Expo React Native app\n│   │   ├── app/                # Expo Router routes\n│   │   ├── src/                # App source code\n│   │   │   ├── components/     # Shared UI components\n│   │   │   ├── hooks/          # Custom hooks\n│   │   │   ├── providers/      # React Query, Auth contexts\n│   │   │   ├── services/       # API client, Supabase client\n│   │   │   ├── utils/          # Helpers\n│   │   │   └── types/          # TypeScript types\n│   │   ├── assets/             # Static assets\n│   │   ├── app.json\n│   │   ├── eas.json\n│   │   └── package.json\n│   ├── api/                    # NestJS backend\n│   │   ├── src/\n│   │   │   ├── modules/        # Feature modules\n│   │   │   ├── common/         # Shared guards, decorators, filters\n│   │   │   ├── config/         # Configuration\n│   │   │   ├── prisma/         # Prisma schema, migrations\n│   │   │   ├── queue/          # Bull queue definitions\n│   │   │   └── main.ts\n│   │   ├── test/\n│   │   ├── Dockerfile\n│   │   └── package.json\n│   └── worker/                 # Background job processor\n│       ├── src/\n│       │   ├── processors/     # Job handlers\n│       │   └── main.ts\n│       └── package.json\n├── libs/\n│   ├── shared-types/           # TypeScript interfaces shared between frontend and backend\n│   ├── ui/                     # Shared UI library (React Native components)\n│   ├── utils/                  # Shared utilities\n│   └── api-client/             # Frontend API client library\n├── docker-compose.yml          # Local development services (PostgreSQL, Redis, Supabase local)\n├── nx.json\n├── pnpm-workspace.yaml\n└── package.json\n```\n\n## 5. Frontend Architecture (Mobile App)\n\n### 5.1 Routing & Navigation\nExpo Router provides file-system based routing with support for nested layouts, tabs, and deep linking. The `app/` directory structure mirrors the user-facing screens: `(tabs)`, `(auth)`, `booking`, `business/[id]`, `profile`, etc. Deep linking is configured for business sharing and password reset.\n\n### 5.2 State Management & Data Fetching\n- **Server State:** TanStack React Query is used for all API data fetching, caching, and mutation. It manages optimistic updates for favorites and booking cancellations/reschedules.\n- **Global Auth State:** A React Context (`AuthProvider`) wraps the app, providing the current user, session, and authentication methods (signIn, signUp, signOut). This context uses Supabase Auth’s `onAuthStateChange` listener.\n- **Local UI State:** Use React’s `useState`/`useReducer` for form inputs, filter selections, and UI toggles.\n\n### 5.3 API Communication\nA dedicated `api-client` library (inside `libs/`) encapsulates all REST calls to the NestJS backend using Axios. It automatically attaches the Supabase access token and handles token refresh. React Query hooks are built on top of this client.\n\n### 5.4 Real-time Features\nSupabase Realtime (WebSockets) is used for:\n- Live slot availability updates during booking flow.\n- Instant booking status changes (e.g., “confirmed” after provider accepts).\nThe frontend subscribes to PostgreSQL changes via Supabase channels and updates React Query cache accordingly.\n\n### 5.5 Authentication Flow\nSupabase Auth manages the full authentication lifecycle:\n- Email/password, Google, Apple sign-in.\n- Email verification (enforced before booking).\n- Password reset with token expiry.\n- Session persistence via secure storage (Expo SecureStore) and automatic token refresh.\n- Logout clears session both client-side and server-side (invalidates refresh token).\nThe `AuthProvider` exposes `isAuthenticated` and `user` for conditional rendering and booking flow gating.\n\n### 5.6 Performance & Animations\n- FlatList with windowing for search results and reviews.\n- React Native Reanimated for smooth transitions, map interactions, and booking timer animations.\n- Image caching and lazy loading.\n\n## 6. Backend Architecture (NestJS)\n\n### 6.1 Modular Design\nNestJS’s module system isolates business domains:\n\n- **AuthModule** – handles token validation, Supabase integration, user registration hooks.\n- **UserModule** – profile management, notification preferences, payment methods.\n- **BusinessModule** – CRUD for businesses, categories, services, staff, business hours.\n- **BookingModule** – booking flow, slot reservation, appointment management, cancellation/rescheduling.\n- **SearchModule** – full-text search, geospatial queries, filtering, sorting, pagination.\n- **AvailabilityModule** – computes and caches real-time slot availability.\n- **PaymentModule** – Stripe integration for payment processing and refunds.\n- **FavoritesModule** – user favorites management.\n- **ReviewModule** – review creation, moderation, rating aggregation.\n- **NotificationModule** – push notifications, email, SMS via queue.\n- **AdminModule** – admin dashboard endpoints.\n\nEach module follows a layered architecture: Controller -> Service -> Repository (Prisma).\n\n### 6.2 Database & ORM\nPostgreSQL is the primary database, managed by Supabase. Prisma ORM provides type-safe querying and schema migrations. The schema is defined in `api/prisma/schema.prisma`. PostGIS extension is enabled for geospatial indexing on business locations, enabling distance-based search and map clustering.\n\n**Key Entities:**\n- `User`, `Business`, `Service`, `Staff`, `Booking`, `Review`, `Favorite`, `Category`, `PaymentMethod`, `Notification`, `SlotReservation`.\n- `Business` has a `location` column of type `geometry(Point, 4326)` with a GiST index for spatial queries.\n- `SlotReservation` is a temporary table or Redis-backed for holding slots during the booking timer.\n\n### 6.3 Caching & Performance\nRedis is used for:\n- Caching frequently accessed reference data (categories, popular businesses).\n- Storing temporary slot reservations with TTL to enforce the 10-minute timer.\n- Caching user session data for quick validation.\n- Rate limiting.\n\n### 6.4 Background Jobs\nBull queues (backed by Redis) process asynchronous tasks:\n- **notification-queue** – sends push notifications, emails, and SMS.\n- **booking-queue** – handles post-booking actions (e.g., sending confirmation emails).\n- **slot-cleanup-queue** – scheduled job to release expired slot reservations.\n- **analytics-queue** – aggregates booking data for admin dashboards.\n\nA separate `worker` app (NestJS standalone) consumes these queues, allowing independent scaling.\n\n### 6.5 Authentication & Authorization\nSupabase Auth handles user authentication. The NestJS backend validates incoming JWTs using Supabase’s public key and attaches the user to the request context via a custom `AuthGuard`. Role-based guards protect admin endpoints.\n\n### 6.6 API Design\nRESTful APIs with versioning (`/api/v1/`). Key endpoints follow resource-oriented patterns:\n- `GET /api/v1/businesses/search?query=...&lat=...&lng=...&radius=...&filters=...`\n- `POST /api/v1/bookings/reserve-slot`\n- `POST /api/v1/bookings/confirm`\n- `GET /api/v1/users/me/appointments`\n\nPagination uses cursor-based method for infinite scroll.\n\n## 7. Database Schema Highlights\n\n- **User**: id, email, phone, full_name, avatar_url, auth_provider, created_at, etc.\n- **Business**: id, owner_id, name, description, location (geometry), address, phone, rating_avg, cover_image, status.\n- **Category**: id, name, parent_id (self-referencing), icon.\n- **Service**: id, business_id, name, description, duration_minutes, price, category_id, is_active.\n- **Staff**: id, business_id, name, photo, bio, working_hours (JSONB).\n- **StaffService**: many-to-many relation between Staff and Service.\n- **Booking**: id, user_id, business_id, service_id, staff_id, start_time, end_time, status (PENDING, CONFIRMED, COMPLETED, CANCELLED, NO_SHOW), special_requests, total_price, payment_intent_id.\n- **SlotReservation**: (Redis key) reservation_id, slot_start, user_id, expires_at, service_id, staff_id.\n- **Review**: id, booking_id, user_id, business_id, rating, comment, created_at.\n- **Favorite**: composite key (user_id, business_id).\n\nPostGIS spatial queries use `ST_DWithin` and `ST_Distance` to filter businesses by distance and order by proximity.\n\n## 8. Payment Integration\n\nStripe is used for payment processing. The mobile app collects card details via Stripe’s React Native SDK, which tokenizes the card directly. The NestJS backend creates a PaymentIntent, confirms it, and handles post-payment webhooks to update booking status. Saved payment methods are stored as Stripe customer payment methods (not raw card data).\n\n## 9. Deployment & CI/CD\n\n- **Local Development:** Docker Compose spins up PostgreSQL (with PostGIS), Redis, and a local Supabase instance. The API and mobile app run on the host machine.\n- **CI/CD:** GitHub Actions run linting, type checking, and tests on every push. On merge to main, it builds and pushes Docker images for API and Worker to a container registry, and triggers EAS Build for the mobile app.\n- **Staging/Production:** The API and Worker are deployed as containerized services on a cloud provider (e.g., AWS ECS, GCP Cloud Run). Supabase is used in its managed cloud form. Redis is provisioned via a managed service. Environment variables are injected via secure secrets.\n- **Mobile App Distribution:** EAS Build creates development, preview, and production builds. Production builds are submitted to app stores via EAS Submit.\n\n## 10. Security Considerations\n\n- All communications over HTTPS/WSS.\n- JWTs are short-lived, refresh tokens are rotated and revoked on logout.\n- Input validation via NestJS Pipes and class-validator.\n- Rate limiting on sensitive endpoints (login, booking).\n- Row-Level Security (RLS) policies in Supabase for direct database access from real-time subscriptions, but most mutations go through the API.\n- Payment data never touches our servers; tokenization is done via Stripe SDK.\n- Secrets stored in GitHub Secrets and injected at build time.\n\n## 11. Future Enhancements\n\n- GraphQL API for more flexible data fetching.\n- Admin dashboard SPA.\n- Multi-language support.\n- Offline support with local data sync.\n\n## 12. Conclusion\n\nThis architecture provides a scalable, maintainable, and modern foundation for the Planity Clone. The clear separation of mobile, API, and background workers, combined with a monorepo tooling and robust cloud services, ensures rapid development and operational excellence.",
