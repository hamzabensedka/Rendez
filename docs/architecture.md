Overview\n\nThis document describes the complete system architecture for the Planity Clone, a mobile-first marketplace connecting customers with beauty, wellness, and service professionals. The architecture follows clean architecture principles, separation of concerns, and is designed for scalability, maintainability, and testability.\n\n## 2. Technology Stack\n\n### Mobile Client\n- **Expo** (managed workflow) with **React Native** and **TypeScript**\n- **Expo Router** for file-based routing and deep linking\n- **TanStack React Query** for server state management and caching\n- **React Native Reanimated** for smooth animations (map markers, bottom sheets, transitions)\n- **Jest** for unit and integration tests\n\n### Backend Services\n- **NestJS** (monorepo with multiple modules) for API and business logic\n- **Prisma** as ORM with PostgreSQL (primary database)\n- **PostGIS** for geospatial queries (search by location, radius)\n- **Redis** for caching, session store, rate limiting, and job queues\n\n### Infrastructure & DevOps\n- **Nx** for monorepo management (shared libraries, code generation)\n- **pnpm** for package management\n- **Docker Compose** for local development\n- **GitHub Actions** for CI/CD\n- **Supabase** for authentication (optional, can be self-hosted Auth0/Keycloak)\n- **EAS Build** for Expo builds and OTA updates\n\n## 3. Service Boundaries\n\n### 4.1 Authentication Service\n- **Responsibilities**: User registration, login (email, OAuth), token management, password recovery, session handling.\n- **Data**: Users table, refresh tokens (Redis), OAuth profiles.\n- **Endpoints**: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/forgot-password`, `POST /auth/reset-password`.\n\n### 4.2 User Service\n- **Responsibilities**: Profile management, preferences, roles, guest-to-account merge.\n- **Data**: Users, profiles, settings.\n- **Endpoints**: `GET /users/me`, `PATCH /users/me`, `POST /users/merge-guest`.\n\n### 4.3 Business Service\n- **Responsibilities**: Business profiles, services, staff, availability, categories.\n- **Data**: Businesses, services, staff, schedules, categories.\n- **Endpoints**: `GET /businesses`, `GET /businesses/:id`, `POST /businesses`, `PUT /businesses/:id`, `GET /businesses/:id/services`, `GET /businesses/:id/availability`.\n\n### 4.4 Search Service\n- **Responsibilities**: Full-text search, geospatial queries, filtering, autocomplete, caching.\n- **Data**: Search indexes (Redis + PostgreSQL full-text), geospatial data (PostGIS).\n- **Endpoints**: `GET /search?q=&lat=&lng=&radius=&category=&sort=`.\n\n### 4.5 Booking Service\n- **Responsibilities**: Slot management, booking creation, modification, cancellation, waitlist, hold management.\n- **Data**: Appointments, holds (Redis), waitlist.\n- **Endpoints**: `POST /bookings`, `GET /bookings`, `PATCH /bookings/:id`, `DELETE /bookings/:id`, `POST /bookings/:id/hold`.\n\n### 4.6 Payment Service\n- **Responsibilities**: Payment processing, refunds, pay-at-store option, invoice generation.\n- **Data**: Payments, transactions, refunds.\n- **Endpoints**: `POST /payments/charge`, `POST /payments/refund`, `GET /payments/history`.\n\n### 4.7 Notification Service\n- **Responsibilities**: Push notifications (Expo Push), SMS (Twilio), email (SendGrid), in-app notifications.\n- **Data**: Notification templates, logs, device tokens.\n- **Endpoints**: `POST /notifications/send`, `POST /notifications/register-device`.\n\n### 4.8 Review Service\n- **Responsibilities**: Reviews, ratings, moderation, photo uploads.\n- **Data**: Reviews, ratings, review photos.\n- **Endpoints**: `GET /reviews`, `POST /reviews`, `PATCH /reviews/:id`, `DELETE /reviews/:id`.\n\n### 4.9 Admin Service\n- **Responsibilities**: Platform oversight, user management, dispute resolution, category management, analytics.\n- **Data**: Admin logs, reports.\n- **Endpoints**: `GET /admin/dashboard`, `GET /admin/users`, `POST /admin/disputes`.\n\n## 5. Data Flow & Communication\n\n### 5.1 API Gateway\nAll client requests go through a single NestJS gateway that handles authentication, rate limiting, and routing to appropriate microservices (or modules within the monorepo).\n\n### 5.2 Caching Strategy\n- **Redis** caches:\n  - Search results (TTL: 5 minutes, invalidated on business update)\n  - Business detail (TTL: 10 minutes)\n  - Availability slots (TTL: 1 minute, real-time)\n  - User sessions (TTL: 30 days)\n  - Rate limit counters\n\n### 5.3 Real-time Updates\n- **WebSockets** (Socket.io) for:\n  - Booking confirmation updates\n  - Slot availability changes\n  - Notification delivery\n\n### 5.4 Background Jobs\n- **Bull** (Redis-backed queue) for:\n  - Sending reminders (push, SMS, email)\n  - Processing payment webhooks\n  - Generating analytics reports\n  - Cleaning up expired holds\n\n## 6. Database Schema (Prisma)\n\nKey models (abbreviated):\n\n```prisma\nmodel User {\n  id            String   @id @default(uuid())\n  email         String   @unique\n  passwordHash  String?\n  role          Role     @default(CUSTOMER)\n  profile       Profile?\n  bookings      Booking[]\n  reviews       Review[]\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n}\n\nmodel Business {\n  id          String   @id @default(uuid())\n  name        String\n  description String?\n  location    Geography?  // PostGIS point\n  address     String\n  phone       String?\n  email       String?\n  verified    Boolean  @default(false)\n  services    Service[]\n  staff       Staff[]\n  reviews     Review[]\n  bookings    Booking[]\n  hours       BusinessHour[]\n  createdAt   DateTime @default(now())\n}\n\nmodel Service {\n  id          String   @id @default(uuid())\n  businessId  String\n  name        String\n  description String?\n  duration    Int      // minutes\n  price       Float\n  category    Category\n  business    Business @relation(fields: [businessId], references: [id])\n}\n\nmodel Appointment {\n  id          String   @id @default(uuid())\n  userId      String\n  businessId  String\n  serviceId   String\n  staffId     String?\n  startTime   DateTime\n  endTime     DateTime\n  status      AppointmentStatus @default(CONFIRMED)\n  user        User     @relation(fields: [userId], references: [id])\n  business    Business @relation(fields: [businessId], references: [id])\n  service     Service  @relation(fields: [serviceId], references: [id])\n  payment     Payment?\n  createdAt   DateTime @default(now())\n}\n\nenum Role { CUSTOMER PROVIDER ADMIN }\nenum AppointmentStatus { CONFIRMED PENDING CHECKED_IN IN_PROGRESS COMPLETED CANCELLED NO_SHOW }\n```\n\n## 7. Security & Performance\n\n### 7.1 Authentication\n- JWT access tokens (15 min expiry) + refresh tokens (30 days, stored in Redis)\n- OAuth 2.0 with Google and Apple\n- Biometric authentication stored securely in device keychain\n- Rate limiting: 5 login attempts per 15 minutes per IP\n\n### 7.2 Data Protection\n- All traffic over HTTPS\n- Passwords hashed with bcrypt (salt rounds: 12)\n- PCI compliance handled by Stripe (no raw card data stored)\n- GDPR compliance: user data export, deletion, consent management\n\n### 7.3 Performance Targets\n- API response time: <200ms for 95th percentile (cached), <500ms for cold queries\n- Search results: <500ms for cached, <2s for full-text + geospatial\n- Map load: <3s with clustering\n- Infinite scroll: 20 items per page, prefetch next page\n\n## 8. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions, hooks, and utilities.\n- **Integration Tests**: NestJS testing module with in-memory database (SQLite) for API endpoints.\n- **E2E Tests**: Detox for mobile, Supertest for API.\n- **CI Pipeline**: GitHub Actions runs lint, unit tests, integration tests on every PR.\n\n## 9. Deployment\n\n- **Docker Compose** for local development (PostgreSQL, Redis, API)\n- **EAS Build** for mobile app builds (development, staging, production)\n- **GitHub Actions** for CI/CD:\n  - `ci.yml`: lint, test, build on PR\n  - `deploy.yml`: deploy API to staging/production (e.g., AWS ECS, Railway)\n- **Supabase** for managed PostgreSQL and authentication (optional)\n\n## 10. Conclusion\n\nThis architecture provides a scalable, maintainable foundation for the Planity Clone. It separates concerns into modular services, leverages caching for performance, and follows best practices for security and testing.
