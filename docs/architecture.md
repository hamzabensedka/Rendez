Overview\nPlanity Clone is a monorepo‑based full‑stack application that serves consumers (guest/browse, booking, profile) and providers (business owner portal) with an admin dashboard. The frontend is built with Expo + React Native (TypeScript) using Expo Router for navigation, TanStack React Query for data fetching, and React Native Reanimated for UI animations. The backend consists of NestJS microservices communicating via REST/GraphQL (or tRPC) backed by Prisma ORM, PostgreSQL with PostGIS extension for geospatial queries, and Redis for caching and pub/sub. Background jobs are handled by BullMQ. The repository is managed with Nx, pnpm, Docker Compose for local development, GitHub Actions for CI/CD, and EAS Build for Expo app distribution. Supabase is optionally used for authentication and file storage.\n\n## 2. High‑Level Architecture\n```\n+-------------------+        +-------------------+        +-------------------+\n|   Expo (Mobile)   | <--->  |   API Gateway     | <--->  |   NestJS Services  |\n| (Expo Router)     |        | (NestJS Gateway)  |        |  - BullMQ + Redis  |\n                                                     +-------------------+\n```\n\n### 2.1 Frontend (apps/mobile)\n- **Framework**: Expo SDK (managed workflow) with TypeScript.\n- **Routing**: Expo Router (file‑based) for stack, tab, and modal navigation.\n- **State & Data**: TanStack React Query (queryClient) for server state; React Context or Zustand for UI state.\n- **Animations**: React Native Reanimated 2 + Reanimated‑Gesture‑Handler.\n- **Design System**: Shared UI library (`ui`) built with React Native Paper / custom components, styled with StyleSheet and theme provider.\n- **Authentication**: Supabase Auth (email/password, social) wrapped in a custom auth hook; fallback to NestJS Auth Service if needed.\n- **Maps**: `expo-location` + `react-native-maps` (or Mapbox) leveraging PostGIS for nearby queries.\n- **Payments**: Stripe SDK via a payment service endpoint.\n- **Notifications**: Foreground push via Expo Push Notifications; background handling via a lightweight service listening to Redis pub/sub.\n\n### 2.2 Backend (apps/api)\n- **Monorepo NestJS Application** with multiple microservices (each as an NestJS module) for clear boundaries.\n- **API Gateway**: NestJS Gateway (using `@nestjs/microservices` with Redis transport) that routes requests, handles authentication (JWT validation), rate limiting, and request logging.\n- **Services**:\n  - **Auth Service**: JWT issuance, refresh, social login (via Supabase or custom OAuth), password reset.\n  - **Business Service**: CRUD for businesses, categories, geospatial indexing (PostGIS), search aggregation.\n  - **Booking Service**: Slot computation, appointment creation, rescheduling, cancellation, conflict detection.\n  - **Payment Service**: Stripe integration, webhook handling, invoicing.\n  - **Notification Service**: Template rendering, push notification dispatch (via Expo push API), email/SMS via external providers.\n  - **Admin Service**: Role‑based access, user/business moderation, analytics.\n  - **Provider Portal Service**: Business owner dashboard (listings, appointments, reviews).\n- **ORM**: Prisma with PostgreSQL provider; PostGIS extension enabled via Prisma raw queries for location‑based searches.\n- **Caching**: Redis (TTL‑based) for hot data (business listings, user sessions, slot availability).\n- **Background Jobs**: BullMQ queues (e.g., `send-notification`, `process-payment`, `cleanup-expired-bookings`) processed by a dedicated NestJS worker microservice.\n- **Validation**: DTOs with class‑validator & class‑transformer; global validation pipe.\n- **Error Handling**: Centralized exception filter returning consistent JSON error format.\n- **Documentation**: Auto‑generated OpenAPI (Swagger) UI per service.\n\n### 2.3 Infrastructure & DevOps\n- **Containerization**: Dockerfile for each NestJS service; docker‑compose.yml defines postgres, redis, redis‑inspector, mailhog (for email dev), and optionally supabase‑studio.\n- **CI/CD**: GitHub Actions workflow runs lint, unit tests (Jest), e2e (Detox for mobile), builds Docker images, pushes to registry, and triggers EAS Build for Expo (iOS/Android).\n- **Environment**: Separate `.env` files for dev, staging, prod; secrets managed via GitHub Secrets.\n- **Monitoring**: Basic health endpoints; integration with Prometheus/Grafana optional.\n- **Testing**: Jest for unit/integration tests; Detox for mobile E2E; Pact for contract testing between mobile and API.\n\n## 3. Service Boundaries & Responsibilities\n| Service | Core Responsibilities | Key Tables (Prisma) | External Dependencies |
|---------|-----------------------|---------------------|-----------------------|

| **Worker (BullMQ)** | Process queued jobs: send notifications, payment reconciliation, cleanup expired bookings | None | Redis (queue store) |

### 4.1 Data Flow Example (Booking)\n1. Mobile app queries `/businesses/nearby` → Gateway → Business Service (uses PostGIS, returns cached list).\n2. User selects a business → `/business/:id` → Business Service.\n3. User picks a service & date → `/booking/slots` → Booking Service computes slots (checks `BusinessHours`, existing `Appointment`s, staff availability) → returns slots (cached in Redis for short TTL).\n4. User confirms → `/booking` → Booking Service creates `Appointment` record, publishes `appointment.created` event to Redis.\n5. Worker picks up event → sends confirmation notification via Notification Service (push/email).\n6. Payment flow: frontend calls `/payment/create-intent` → Payment Service creates Stripe intent → returns client secret → frontend confirms with Stripe → webhook hits Payment Service → updates `Payment` status → triggers `payment.success` event → Worker may send receipt.\n\n## 5. Communication Protocols\n- **Internal**: NestJS microservices communicate via Redis Pub/Sub (for events) and direct TCP/RPC (for synchronous calls) using `@nestjs/microservices`.\n- **External**: Mobile ↔ Gateway uses HTTPS (REST) with JSON payloads; optional GraphQL/tRPC can be added later.\n- **Webhooks**: Stripe, Supabase, external SMS/email providers send HTTP POST to Gateway → validated → forwarded to appropriate service.\n\n## 6. State Management & Caching Strategy\n- **Client‑Side**: TanStack React Query handles server‑state caching, background refetching, and optimistic updates (e.g., toggling favorite).\n- **Redis**: \n  - Session store (JWT blacklist/whitelist).\n  - Hot business listings (TTL 5 min).\n  - Slot availability (TTL 1 min).\n  - Rate limiting counters (window 1 min).\n  - BullMQ job queues.\n\n## 7. Security Considerations\n- **Authentication**: JWT (HS256) with short-lived access token (15 min) + refresh token (7 d) stored securely (SecureStore).\n- **Authorization**: Role‑based guards (`AdminGuard`, `ProviderGuard`, `UserGuard`).\n- **Data Validation**: DTOs + class‑validator; Prisma ensures DB‑level constraints.\n- **SQL Injection**: Prisma ORM eliminates raw string queries; PostGIS uses parameterized inputs.\n- **Secrets**: Managed via GitHub Secrets & Docker env files; never committed.\n- **Transport**: TLS enforced in production (NGINX reverse proxy or cloud load balancer).\n- **Input Sanitization**: Sanitize geo‑coordinates, text fields (XSS prevention).\n\n## 8. Testing Strategy\n- **Unit Tests**: Jest for services, utilities, DTO validation.\n- **Integration Tests**: Supertest for NestJS endpoints; Prisma uses a temporary test database (Docker).\n- **E2E Mobile**: Detox for critical flows (login, search, booking, payment).\n- **Contract Tests**: Pact between mobile app and API gateway to ensure backward compatibility.\n- **Load Testing**: k6 scripts for peak booking scenarios.\n\n## 9. Deployment & Release Process\n1. **Local Dev**: `docker compose up` spins up postgres, redis, mailhog, and API; Expo dev client runs via `expo start`.\n2. **CI**: On PR → lint, unit tests, build Docker images (no push).\n3. **CD (main branch)**: \n   - Build & push Docker images to registry (GHCR/ECR).\n   - Deploy API to Kubernetes/ECS via Helm/CloudFormation (outside scope but defined).\n   - Run DB migrations via Prisma (`prisma migrate deploy`).\n   - Trigger EAS Build for iOS/Android; distribute to internal test tracks via Expo.\n   - Run smoke tests against staging environment.\n4. **Feature Flags**: Optional launchdarkly‑style flags stored in Redis for gradual rollout.\n\n## 10. Future Extensibility\n- **Micro‑service Split**: As traffic grows, each NestJS module can be extracted to its own deployable service.\n- **GraphQL / tRPC**: Replace REST with typed endpoints for better mobile‑backend contract.\n- **WebSocket / Socket.io**: Real‑time updates for appointment status, chat between user and provider.\n- **AI‑Powered Recommendations**: Leverage business tags & user history; store embeddings in PostgreSQL (pgvector).\n- **Multi‑tenant SaaS**: Introduce tenant ID in Prisma schema for white‑labeling.\n\n--- End of Document ---\n",
