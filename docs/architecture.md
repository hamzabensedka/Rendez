Each module encapsulates its own domain logic, Prisma repositories, and controllers. Modules communicate via NestJSвҖҷs dependency injection and, for crossвҖ‘cutting concerns, through a lightweight event bus (NestJS EventEmitter) to keep them decoupled. This design allows future extraction into microservices if needed.\n\n## 4. Service Boundaries\n- **Auth Module**: Handles user registration, login (email/password, social), token validation, password reset. Relies on Supabase Auth for identity and JWT issuance; the module only manages user profiles and roles in the local database.\n- **Business Module**: CRUD for businesses, services, staff, working hours, amenities. Provides search endpoints with fullвҖ‘text search and geospatial filtering.\n- **Availability Module**: Computes realвҖ‘time time slots based on provider availability rules, existing bookings, buffer times, and date overrides. Caches slot results in Redis.\n- **Booking Module**: Manages the booking flow, reservation locks (Redis), payment integration (stub), appointment lifecycle (confirm, cancel, reschedule). Ensures idempotency and atomicity.\n- **Review Module**: Handles review submission (linked to completed appointments), moderation, and average rating updates (async via event).\n- **Favorites Module**: Toggle and list user favorites, synced across devices.\n- **Notification Module**: Sends push notifications (via Expo Push API / FCM), email (via Supabase or external service), and SMS. Listens to booking events.\n- **Search Module**: Aggregates search across businesses, services, and locations using PostgreSQL fullвҖ‘text search and PostGIS.\n\n## 5. Data Flow Examples\n### 5.1 Booking Flow\n1. Customer selects service вҶ’ staff вҶ’ date.\n2. Mobile calls `GET /availability/slots?businessId=&serviceId=&staffId=&date=`.\n3. Availability Module checks Redis cache; if miss, computes slots from rules and existing bookings, caches for 30s.\n4. User picks a slot and submits booking вҶ’ `POST /bookings` with idempotency key.\n5. Booking Module acquires a Redis lock (`SET lock:slot:<id> NX EX 600`). If lock fails, return conflict.\n6. Validates slot availability again, creates a `PENDING` appointment, initiates payment (stub).\n7. On payment success, updates appointment to `CONFIRMED`, releases lock, emits `booking.confirmed` event.\n8. Notification Module sends confirmation email/push.\n9. Mobile receives realвҖ‘time update via Supabase Realtime channel (provider dashboard).\n\n### 5.2 Search with Location\n1. Mobile sends `GET /businesses/search?query=hair&lat=48.8566&lng=2.3522&radius=5000`.\n2. Search Module uses PostgreSQL `tsvector` for text search and `ST_DWithin` for geospatial filtering.\n3. Results ordered by distance (using `ST_Distance`) or rating.\n4. Response includes next available slot (preвҖ‘computed or fetched from Availability Module).\n\n## 6. The mobile app subscribes to a channel filtered by business ID.
