Overview\n\nThis document describes the system architecture for the Planity Clone, a multi-platform application for discovering and booking beauty and wellness services. The architecture is designed for scalability, maintainability, and clear separation of concerns, using a monorepo structure with Nx and pnpm.\n\n## 2. Architecture Principles\n\n- **Clean Architecture**: Business logic is isolated from frameworks and infrastructure.\n- **Separation of Concerns**: Each module has a single responsibility.\n- **API-First Design**: The mobile app consumes a well-defined REST API.\n- **Type Safety**: TypeScript is used throughout the stack.\n- **Testability**: Unit, integration, and e2e tests are first-class citizens.\n- **Scalability**: Services are designed to scale horizontally.\n\n## 3. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile Framework | Expo (React Native) | Cross-platform mobile app |\n| Routing | Expo Router | File-based navigation |\n| State Management | TanStack React Query | Server state caching and sync |\n| Animations | React Native Reanimated | High-performance UI animations |\n| Backend Framework | NestJS | Modular, scalable Node.js server |\n| ORM | Prisma | Type-safe database access |\n| Database | PostgreSQL + PostGIS | Relational data + geospatial queries |\n| Cache | Redis | Session store, rate limiting, caching |\n| Monorepo | Nx + pnpm | Code sharing and build orchestration |\n| Containerization | Docker Compose | Local development environment |\n| CI/CD | GitHub Actions | Automated testing and deployment |\n| Hosting | Supabase (PostgreSQL) | Managed database |\n| Build | EAS Build | Expo cloud builds |\n| Testing | Jest | Unit and integration tests |\n\n## 4. Service Boundaries\n\n### 5.1 Auth Service (NestJS Module)\n- Handles registration, login, password reset, OAuth.\n- Issues JWT access tokens (15 min) and refresh tokens (7 days).\n- Rate limiting on login (5 attempts → 30 min lockout).\n- Uses Redis for token blacklisting and rate limit counters.\n\n### 5.2 User Service\n- Manages user profiles (Client, Provider, Admin).\n- Handles role selection and verification.\n- Stores preferences and favorites.\n\n### 5.3 Business Service\n- CRUD for business profiles.\n- Manages business hours, amenities, images.\n- Geospatial indexing via PostGIS for location queries.\n\n### 5.4 Service Catalog Service\n- Manages categories, subcategories, and services.\n- Supports up to 3 levels of hierarchy.\n- Caches category data in Redis for fast retrieval.\n\n### 5.5 Booking Service\n- Core booking engine with pessimistic locking (10-min holds).\n- Handles multi-service bookings, provider assignment.\n- Integrates with payment service (future).\n- Manages cancellation policies and refunds.\n\n### 5.6 Search Service\n- Full-text search with fuzzy matching (trigram indexes).\n- Autocomplete with <200ms response.\n- Cursor-based pagination (20 results per page).\n- Real-time availability computation.\n\n### 5.7 Notification Service\n- Sends push notifications (Expo Push API).\n- Sends SMS via Twilio.\n- Sends emails via SendGrid.\n- Manages reminder scheduling (24h and 1h before).\n\n### 5.8 Review Service\n- CRUD for reviews and ratings.\n- Photo upload support.\n- Aggregates average rating (half-star precision).\n\n### 5.9 Admin Service\n- Platform health monitoring.\n- User management and dispute resolution.\n- Category and content management.\n\n## 6. Data Flow\n\n### 6.1 Booking Flow\n1. Client selects service(s) in mobile app.\n2. App sends booking request to Booking Service.\n3. Booking Service checks availability (pessimistic lock on slot).\n4. If available, creates a hold (10 min) in Redis.\n5. Client proceeds to payment (or pay-at-venue).\n6. On confirmation, Booking Service creates appointment in PostgreSQL.\n7. Notification Service sends confirmation (push, email, in-app).\n8. Hold is released if payment fails or client cancels.\n\n### 6.2 Search Flow\n1. Client enters query in search bar.\n2. Mobile app sends debounced request to Search Service.\n3. Search Service queries PostgreSQL with full-text search and PostGIS for distance.\n4. Results are cached in Redis for 5 minutes.\n5. Response includes next available slot (computed from Booking Service).\n6. Mobile app displays results with TanStack React Query caching.\n\n## 7. Database Schema (Prisma)\n\nKey models:\n\n- **User**: id, email, passwordHash, role, phone, createdAt\n- **Business**: id, name, description, categoryId, location (PostGIS point), hours, images\n- **Service**: id, businessId, name, duration, price, description\n- **Appointment**: id, userId, businessId, serviceId, providerId, startTime, endTime, status, paymentStatus\n- **Review**: id, userId, businessId, rating, comment, createdAt\n- **Category**: id, name, parentId, icon\n\nPostGIS extension enables spatial queries for \"nearby\" searches.\n\n## 8. Caching Strategy\n\n- **Redis**: Session store, rate limit counters, booking holds, search result cache (5 min TTL), category cache (1 hour TTL).\n- **TanStack React Query**: Client-side cache for API responses (stale time 5 min, cache time 30 min).\n- **Supabase**: PostgreSQL connection pooling and caching.\n\n## 9. Security\n\n- JWT with refresh token rotation.\n- Rate limiting on auth endpoints.\n- Input validation with class-validator (NestJS) and Zod (shared libs).\n- HTTPS enforced.\n- Environment variables for secrets (no hardcoded values).\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions and services.\n- **Integration Tests**: Test NestJS modules with real PostgreSQL (via Docker Compose).\n- **E2E Tests**: Detox for mobile app (future).\n- **CI Pipeline**: GitHub Actions runs tests on every PR.\n\n## 11. Deployment\n\n- **Mobile**: EAS Build for iOS and Android builds.\n- **Backend**: Docker container deployed to cloud (AWS ECS or similar).\n- **Database**: Supabase managed PostgreSQL.\n- **CI/CD**: GitHub Actions for automated testing and deployment.\n\n## 12. Development Setup\n\n1. Clone the repository.\n2. Run `pnpm install`.\n3. Start Docker Compose (`docker-compose up -d`) for PostgreSQL and Redis.\n4. Run `nx run api:dev` for backend.\n5. Run `nx run mobile:start` for Expo dev server.\n\n## 13. Conclusion\n\nThis architecture provides a solid foundation for the Planity Clone, ensuring scalability, maintainability, and a clear separation of concerns.
