# Planity Clone System Architecture\n\n## 1. High-Level Overview\nPlanity Clone is a full‑stack mobile‑first application consisting of:\n- **Client**: Expo managed workflow React Native app (TypeScript) using Expo Router, TanStack React Query, React Native Reanimated.\n- **Server**: NestJS microservice (monolith) with Prisma ORM, PostgreSQL + PostGIS for spatial queries, Redis for caching/pub‑sub, BullMQ for background jobs.\n- **Infrastructure**: Docker Compose for local dev, GitHub Actions for CI, EAS Build for OTA updates, Supabase for optional auth/storage fallback, Nx monorepo for code sharing.\n\n## 2. Technology Stack Mapping\n| Layer | Tech | Purpose |\n|-------|------|---------|\n| Mobile UI | Expo (SDK 49), React Native, TypeScript | Cross‑platform UI |\n| Navigation | Expo Router (file‑based) | Declarative routing |\n| State / Data | TanStack React Query | Server state, caching, mutations |\n| Animations | React Native Reanimated 2 | Smooth gestures, transitions |\n| Backend API | NestJS (TypeScript) | REST/GraphQL endpoints, modular services |\n| ORM | Prisma | Type‑safe DB access, migrations |\n| Database | PostgreSQL + PostGIS extension | Relational data, geo‑queries |\n| Cache / PubSub | Redis | Session store, caching, BullMQ queues |\n| Background Jobs | BullMQ (Redis‑based) | Async tasks: notifications, slot computation |\n| Auth | Custom JWT + Supabase (optional) | Email/password, social login, password reset |\n| Payments | Stripe SDK (via backend) | Secure payment processing |\n| Design System | Shared TS components in `libs/ui` | Consistent look & feel |\n| Monorepo | Nx | Code sharing, linting, testing |\n| Package Manager | pnpm | Efficient installs |\n| CI/CD | GitHub Actions (lint, test, build) + EAS Build (OTA) + Docker Compose (env) | Automated pipelines |\n| Testing | Jest (unit/integration) + React Native Testing Library | Quality assurance |\n| Analytics | (optional) Firebase / Sentry | Monitoring |\n\n## 3. Service Boundaries\n### 3.1 Client (apps/mobile)\n- **Presentation Layer**: Screens (Expo Router), components, hooks.\n- **Application Layer**: React Query hooks encapsulating API calls.\n- **Domain Layer**: Shared TypeScript interfaces (`libs/types`) and business logic (e.g., slot computation helpers) kept in `libs/domain`.\n- **Infrastructure Layer**: Reanimated gestures, Expo modules (Location, Notifications), AsyncStorage wrapper.\n\n### 3.2 Server (apps/api)\n- **API Layer**: NestJS controllers (REST) and optional GraphQL resolver.\n- **Application Layer**: Service classes (`@Injectable`) orchestrating use‑cases (auth, booking, availability).\n- **Domain Layer**: Entities & DTOs in `libs/domain` (shared with client via Nx).\n- **Infrastructure Layer**: Prisma models, Redis clients, BullMQ queues, external service adapters (Stripe, email provider, Supabase).\n- **Cross‑cutting**: Guards (JWT auth), interceptors (logging, transformation), exception filters.\n\n### 3.3 Background Workers (apps/worker)\n- Lightweight NestJS context that shares same modules as API but only registers BullMQ processors.\n- Handles: notification dispatch, slot recomputation, payment webhook handling, cleanup jobs.\n\n### 3.4 Shared Libraries (libs/)\n- `libs/types`: Platform‑agnostic TypeScript interfaces (User, Business, Service, Slot, Booking, etc.).\n- `libs/ui`: Design system components (buttons, cards, theming) built with React Native Reanimated where needed.\n- `libs/util`: Helper functions (date/time, geo‑haversine, validation).\n- `libs/config`: Environment schema (zod) used by both client and server.\n\n## 4. Data Flow Examples\n### 5.1 Guest Browse (no auth)\n1. User opens `apps/mobile` → Expo Router renders `HomeScreen`.\n2. `useQuery('businesses', fetchBusinesses)` calls `/api/businesses` (NestJS).\n3. NestJS controller delegates to `BusinessesService` → Prisma reads `business` table (with PostGIS for location filtering if query includes lat/lng).\n4. Response cached via React Query; UI renders list.\n5. Tapping a business navigates to `BusinessDetailScreen` via router; similar query for `/api/businesses/:id`.\n\n### 5.2 Booking Flow (authenticated)\n1. User logs in → `/api/auth/login` (NestJS) validates credentials, issues JWT (access + refresh) stored in SecureStore (via expo‑secure‑store).\n2. Subsequent requests include `Authorization: Bearer <token>`; NestJS `AuthGuard` verifies.\n3. User selects service → `useMutation` calls `/api/bookings` (POST).\n4. `BookingsService`:\n   - Checks business schedule + existing bookings (Prisma).\n   - Computes available slots (algorithm in `libs/util/date.ts`).\n   - Creates booking record (status PENDING).\n   - Calls Stripe SDK to create PaymentIntent; returns client secret.\n5. Mobile collects payment via Stripe SDK, confirms → `/api/bookings/:id/pay`.\n6. On success, booking status set to CONFIRMED; BullMQ job `send-confirmation-email` enqueued.\n7. Worker processes job → sends email/push via Expo push notification service.\n8. Slot computation job runs nightly (BullMQ cron) to refresh availability cache in Redis.\n\n### 5.3 Map‑Based Search\n- Mobile uses `expo-location` to get user coordinates.\n- Sends GET `/api/businesses?near=[lat,lng]&radius=5km`.\n- NestJS uses PostGIS `ST_DWithin` to find businesses within radius, returns GeoJSON.\n- React Query caches; UI renders MapView with markers (using `react-native-maps`).\n\n### 5.4 Notifications & Background Jobs\n- Booking confirmation triggers BullMQ `email` processor.\n- Processor uses `nodemailer` or SendGrid; also pushes via Expo push token stored on user record.\n- Cancellation/reschedule triggers similar jobs.\n- Admin dashboard reads aggregated metrics from materialized views or cached Redis counters.\n\n## 6. Deployment & CI/CD\n- **Local**: `docker compose up` spins PostgreSQL (with PostGIS), Redis, NestJS API, and worker.\n- **CI (GitHub Actions)**:\n  1. Checkout, setup pnpm, install.\n  2. Run `nx lint`, `nx test`, `nx build api` and `nx build mobile` (expo export).\n  3. On push to `main`: Docker images built and pushed to registry (optional).\n  4. Tag triggers EAS Build for iOS/Android OTA updates.\n- **Production**:\n  - API & worker run as Docker containers behind NGINX or cloud load balancer (AWS ECS/Fargate, GCP Cloud Run, etc.).\n  - PostgreSQL managed (AWS RDS with PostGIS extension) + read replica.\n  - Redis Elasticache.\n  - BullMQ workers scale based on queue depth.\n  - Expo OTA updates via EAS; fallback to store builds quarterly.\n  - Secrets managed via GitHub Actions secrets / Doppler / AWS Parameter Store.\n\n## 7. Quality Assurance\n- **Unit Tests**: Jest for services, utilities, reducers.\n- **Integration Tests**: Supertest for API endpoints; React Native Testing Library for screens with mocked React Query.\n- **E2E**: Detox (optional) for critical flows (login → booking).\n- **Linting**: ESLint (Nx preset) + Prettier.\n- **Type Checking**: tsc under Nx.\n- **Performance**: React Query stale‑time caching; Redis caching for frequent geo queries; bundle analysis with `expo doctor`.\n- **Security**: JWT short‑lived (15m) + refresh token rotation; helmet, rate‑limiting (NestJS throttler); input validation via class‑validator/zod; OWASP ASVS compliance.\n\n## 8. Shared Types & Design System\n- `libs/types` exported as a local npm package via Nx, consumed by both `mobile` and `api`.\n- UI components built with React Native Reanimated for gestures (e.g., swipe‑to‑delete, animated modal) and styled with `styled-components/native` or `NativeBase`.\n- Theme file defines light/dark palettes, spacing scale, typography; consumed via React Context.\n\n## 9. Future Extensibility\n- **Micro‑service split**: Separate NestJS modules into independent services (e.g., payments, notifications) communicating via Redis Pub/Sub or gRPC.\n- **Web Admin Portal**: Next.js app sharing same `libs/types` and `api` via Nx.\n- **Analytics**: Integrate Segment or Amplitude via middleware.\n- **Offline First**: Persist React Query cache to MMKV; sync via background fetch.\n\n---\n*This document captures the architectural decisions, boundaries, and folder layout for the Planity Clone using the specified stack. It serves as a single source of truth for developers, DevOps, and product stakeholders.*\n
