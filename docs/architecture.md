Document Information\n\n| Property | Value |\n|----------|-------|\n| Version | 1.0 |\n| Status | Draft |\n| Author | Jordan, Staff Software Engineer |\n| Date | 2024 |\n\n## 2. Architectural Overview\n\nPlanity Clone is a mobile-first platform built with a **monorepo** structure using Nx and pnpm. The frontend is an Expo React Native application with TypeScript, Expo Router for navigation, TanStack React Query for server state management, and React Native Reanimated for animations. The backend is a NestJS monolith with clear module boundaries, using Prisma ORM with PostgreSQL (PostGIS for geospatial queries) and Redis for caching and real-time features. The system is containerized with Docker Compose and deployed via EAS Build (mobile) and GitHub Actions (backend). Supabase provides authentication and real-time subscriptions.\n\n## 3. Service Boundaries and Module Design\n\n### 5.1 Frontend (Mobile App)\n\nThe mobile app is organized by feature, with clear separation between UI components, hooks, services, and types.\n\n- **Expo Router** handles navigation with file-based routing, supporting deep links for business pages and booking flows.\n- **TanStack React Query** manages all server state. Each API endpoint has a corresponding hook in `services/` that defines query keys, caching strategies, and mutation handlers.\n- **React Native Reanimated** is used for smooth transitions, shared element animations (e.g., business card to detail), and gesture-based interactions.\n- **Supabase client** is used for real-time subscriptions (e.g., booking status updates) and authentication flows.\n\n### 5.2 Backend (NestJS)\n\nThe backend follows a modular architecture. Each module encapsulates its own controllers, services, DTOs, and database access.\n\n| Module | Responsibility | Key Endpoints |\n|--------|----------------|---------------|\n| **Auth** | User registration, login, token management, OAuth | POST /auth/register, POST /auth/login, POST /auth/refresh, POST /auth/logout |\n| **Users** | User profile CRUD, device management | GET /users/me, PATCH /users/me, GET /users/me/devices |\n| **Businesses** | Business CRUD, photos, hours | GET /businesses, GET /businesses/:id, POST /businesses, PATCH /businesses/:id |\n| **Services** | Service CRUD per business | GET /businesses/:id/services, POST /businesses/:id/services |\n| **Categories** | Category CRUD (admin) | GET /categories, POST /categories (admin) |\n| **Staff** | Staff management per business | GET /businesses/:id/staff, POST /businesses/:id/staff |\n| **Availability** | Real-time slot calculation | GET /businesses/:id/availability?date=...&staffId=... |\n| **Bookings** | Booking CRUD, status management | POST /bookings, GET /bookings, PATCH /bookings/:id/cancel |\n| **Reviews** | Review CRUD | POST /businesses/:id/reviews, GET /businesses/:id/reviews |\n| **Payments** | Payment intent creation, webhooks | POST /payments/create-intent, POST /payments/webhook |\n| **Notifications** | Push notification sending | POST /notifications/send (internal) |\n| **Search** | Full-text and geospatial search | GET /search?q=...&lat=...&lng=...&radius=... |\n| **Admin** | Platform management | GET /admin/dashboard, PATCH /admin/businesses/:id/status |\n\n### 5.3 Database Schema (Prisma)\n\nKey models:\n\n- **User**: id, email, phone, passwordHash, role (CONSUMER, BUSINESS_OWNER, ADMIN), verifiedAt, createdAt\n- **Business**: id, name, description, address, location (PostGIS Point), photos[], hours (JSON), primaryCategoryId, ownerId\n- **Service**: id, businessId, name, description, duration, price, categoryId\n- **Staff**: id, businessId, userId, name, bio, photo, specialties[]\n- **Booking**: id, businessId, userId, staffId, serviceIds[], startTime, endTime, status (PENDING, CONFIRMED, CANCELLED, COMPLETED), totalPrice\n- **Review**: id, businessId, userId, rating, comment, createdAt\n- **Category**: id, name, parentId, icon, sortOrder\n- **AvailabilitySlot**: id, businessId, staffId, date, startTime, endTime, isBooked\n\n### 5.4 Geospatial Search\n\nPostGIS extension enables efficient location-based queries. The `Business` table has a `location` column of type `GEOGRAPHY(Point)`. Search queries use `ST_DWithin` for radius filtering and `ST_Distance` for sorting by distance. An index on `location` ensures performance.\n\n## 6. Data Flow\n\n### 6.1 Booking Flow\n\n1. Consumer searches for businesses → `GET /search` with query parameters.\n2. Consumer selects business → `GET /businesses/:id` returns full details.\n3. Consumer selects service(s) and staff → `GET /businesses/:id/availability?date=...&staffId=...` returns available time slots.\n4. Consumer selects time slot and confirms booking → `POST /bookings` with serviceIds, staffId, startTime.\n5. Backend creates booking, marks slot as booked in Redis cache, triggers payment flow.\n6. Payment success → Booking status updated to CONFIRMED, notification sent to consumer and business.\n7. Real-time update pushed via Supabase Realtime to both parties.\n\n### 6.2 Search Flow\n\n1. User types in search bar → debounced 300ms → `GET /search?q=...&lat=...&lng=...&radius=...&filters=...`.\n2. Backend performs full-text search on business name, service names, and descriptions using PostgreSQL `tsvector`.\n3. Results are filtered by geospatial radius and other filters, sorted by relevance or distance.\n4. Results cached in Redis with TTL of 5 minutes for identical queries.\n5. Response includes paginated list of businesses with distance and rating.\n\n## 7. Caching Strategy\n\n| Cache | Technology | Purpose | TTL |\n|-------|------------|---------|-----|\n| Search results | Redis | Reduce database load for frequent queries | 5 minutes |\n| Business details | Redis | Quick access to business profiles | 10 minutes |\n| Availability slots | Redis | Real-time slot management, invalidated on booking | Until booking confirmation or 30 min |\n| Session tokens | Redis | JWT blacklist and refresh token store | 7 days |\n| Rate limiting | Redis | Sliding window rate limiter | Per request window |\n\n## 8. Authentication & Authorization\n\n- **Supabase Auth** handles user registration, login, OAuth, and email verification.\n- **JWT tokens** are issued by Supabase with a 15-minute access token and 7-day refresh token.\n- **NestJS guards** validate JWT and extract user info. Role-based guards restrict admin endpoints.\n- **Device fingerprinting** is stored in Redis to detect suspicious activity.\n- **Rate limiting** is applied per IP and per user using Redis-based sliding window.\n\n## 9. Deployment & CI/CD\n\n### 9.1 Local Development\n\n```bash\ndocker-compose up -d  # Starts PostgreSQL, Redis, and API\ncd apps/mobile && npx expo start  # Starts Expo dev server\n```\n\n### 9.2 CI Pipeline (GitHub Actions)\n\n- On push to `main` or PR:\n  1. Install dependencies with pnpm\n  2. Lint and type-check all packages\n  3. Run unit tests with Jest\n  4. Build mobile app (EAS Build for PR preview)\n  5. Build and push Docker image for API\n\n### 9.3 Deployment\n\n- **Mobile**: EAS Build generates OTA updates via Expo Updates and native builds for App Store/Google Play.\n- **API**: Docker image deployed to a container orchestration platform (e.g., AWS ECS, Google Cloud Run).\n- **Database**: Managed PostgreSQL with automated backups.\n- **Redis**: Managed Redis instance.\n\n## 10. Testing Strategy\n\n| Layer | Tool | Scope |\n|-------|------|-------|\n| Unit | Jest | Services, utilities, hooks |\n| Integration | Jest + Supertest | API endpoints with test database |\n| Component | Jest + React Native Testing Library | UI components |\n| E2E | Detox (future) | Critical user flows |\n\n## 11. Security Considerations\n\n- All API endpoints use HTTPS.\n- Passwords hashed with bcrypt (cost 12).\n- JWT stored in httpOnly cookies to prevent XSS.\n- Input validation with class-validator and class-transformer.\n- SQL injection prevention via Prisma parameterized queries.\n- Rate limiting on auth endpoints.\n- CORS configured to allow only mobile app origin.\n- Environment variables managed via NestJS ConfigModule.\n\n## 12.
