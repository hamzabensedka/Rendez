  "content": "# Planity Clone System Architecture\n\n## 1. Introduction\nPlanity Clone is a mobile-first platform connecting customers with beauty, wellness, and grooming businesses. This document defines the complete system architecture, including service boundaries, technology choices, folder structure, data models, API design, caching, authentication, deployment, and testing strategies. The architecture prioritizes clean separation of concerns, scalability, maintainability, and a smooth developer experience.\n\n## 2. System Architecture Overview\nThe system follows a **modular monorepo** managed by Nx and pnpm, with a **mobile client** (Expo/React Native) and a **backend API** (NestJS). PostgreSQL with PostGIS handles persistent storage, Redis provides caching and real-time features, and Supabase is used for authentication and file storage. The architecture is split into three main domains:\n- **Mobile App**: Customer-facing and provider-facing features via Expo Router.\n- **Backend API**: RESTful services for business logic, booking engine, payments, notifications.\n- **Infrastructure**: Docker Compose for local development, GitHub Actions for CI/CD, EAS Build for mobile deployment.\n\nService boundaries are defined by NestJS modules, each encapsulating its own domain logic, Prisma models, and API endpoints. The mobile app uses TanStack React Query for server state management, React Native Reanimated for animations, and Expo Router for file-based navigation.\n\n## 3. Backend Architecture\n### 5.1 NestJS Module Design\nEach domain is a NestJS module with its own controller, service, and Prisma repository. Modules are loosely coupled and communicate via dependency injection. Shared concerns (auth guards, logging, error handling) reside in `common/`.\n\n**Key Modules:**\n- **AuthModule**: Handles Supabase JWT verification, role-based guards, refresh token rotation.\n- **UsersModule**: Customer and provider profile management, avatar upload.\n- **BusinessesModule**: Business CRUD, geospatial search (PostGIS), gallery management.\n- **ServicesModule**: Service definitions, pricing, duration, buffer times.\n- **StaffModule**: Staff profiles, working hours, breaks, holidays.\n- **BookingsModule**: Core booking engine: slot computation, concurrency control (optimistic locking), rescheduling, cancellation policies.\n- **ReviewsModule**: Post-booking review prompts, moderation, average rating calculation.\n- **PaymentsModule**: Stripe integration, payment intent creation, webhook handling, refunds.\n- **NotificationsModule**: Event-driven notification dispatch (push via FCM, email via Nodemailer).\n- **CategoriesModule**: Hierarchical category management.\n- **AdminModule**: Dashboard analytics, user/business moderation.\n\n### 5.2 Prisma Schema & PostGIS\nPrisma schema defines all entities with relations. PostGIS extension enables geospatial queries for location-based search. Key models:\n\n```prisma\nmodel User {\n  id        String   @id @default(uuid())\n  email     String   @unique\n  role      Role     // CUSTOMER, PROVIDER, ADMIN\n  profile   Profile?\n  businesses Business[]\n  bookings  Booking[]\n  reviews   Review[]\n  favorites Favorite[]\n}\n\nmodel Business {\n  id          String   @id @default(uuid())\n  ownerId     String\n  owner       User     @relation(fields: [ownerId], references: [id])\n  name        String\n  description String?\n  address     String\n  location    Unsupported(\"geometry(Point, 4326)\")?  // PostGIS\n  categoryId  String\n  category    Category @relation(fields: [categoryId], references: [id])\n  services    Service[]\n  staff       Staff[]\n  reviews     Review[]\n  photos      Photo[]\n  avgRating   Float    @default(0)\n}\n\nmodel Service {\n  id          String   @id @default(uuid())\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  name        String\n  duration    Int      // minutes\n  price       Decimal\n  bufferBefore Int?    // buffer time before appointment\n  bufferAfter  Int?\n}\n\nmodel Staff {\n  id          String   @id @default(uuid())\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  name        String\n  workingHours WorkingHour[]\n  breaks      Break[]\n  holidays    Holiday[]\n}\n\nmodel Booking {\n  id          String   @id @default(uuid())\n  customerId  String\n  customer    User     @relation(fields: [customerId], references: [id])\n  businessId  String\n  business    Business @relation(fields: [businessId], references: [id])\n  serviceId   String\n  service     Service  @relation(fields: [serviceId], references: [id])\n  staffId     String?\n  staff       Staff?   @relation(fields: [staffId], references: [id])\n  startTime   DateTime\n  endTime     DateTime\n  status      BookingStatus\n  notes       String?\n  payment     Payment?\n  review      Review?\n}\n```\n\nGeospatial queries use raw SQL via Prisma's `$queryRaw` for `ST_DWithin`, `ST_Distance`, etc. Redis caches frequently accessed business locations and availability slots.\n\n### 5.3 Redis Caching & Real-Time\nRedis is used for:\n- **Slot Availability Cache**: Precomputed available time slots for the next 7 days per staff/service, invalidated on booking/cancellation.\n- **Session Store**: JWT refresh token blacklist/whitelist.\n- **Rate Limiting**: API rate limiting per user/IP.\n- **Job Queues**: (Optional) BullMQ for email/push notification dispatch.\n\n## 6. Frontend Architecture\n### 6.1 Expo Router & Navigation\nFile-based routing with Expo Router provides a clear navigation structure. The `app/` directory defines:\n- `(auth)/` – unauthenticated flows (login, signup, forgot password).\n- `(tabs)/` – main bottom tabs: Home, Search, Bookings, Favorites, Profile.\n- `business/[id]/` – dynamic business detail, service selection, booking flow.\n- `profile/` – user settings, payment methods, notifications.\n\nDeep linking is configured for booking confirmations and password resets.\n\n### 6.2 State Management & Data Fetching\nTanStack React Query manages all server state (businesses, bookings, user data). Each API call is wrapped in a custom hook (e.g., `useBusinesses`, `useAvailableSlots`). Query keys follow a consistent pattern: `['businesses', filters]`, `['slots', businessId, serviceId, date]`. Mutations (booking, cancellation) use `useMutation` with optimistic updates for instant UI feedback.\n\nLocal UI state (e.g., selected date, map viewport) uses React's `useState` or `useReducer`. Zustand is reserved for complex cross-component state (e.g., booking wizard multi-step form).\n\n### 6.3 UI & Animations\nReact Native Reanimated powers smooth transitions: map-to-list toggle, booking step progress, card expansions. The shared `ui` library provides design system components built with accessibility in mind (proper roles, labels, contrast).\n\n## 7. Shared Types & Design System\nThe `libs/shared-types` package contains all TypeScript interfaces and enums used by both frontend and backend. This ensures type consistency across the stack. Examples:\n```typescript\n// libs/shared-types/src/booking.ts\nexport interface Booking {\n  id: string;\n  customerId: string;\n  businessId: string;\n  serviceId: string;\n  staffId?: string;\n  startTime: string; // ISO 8601\n  endTime: string;\n  status: 'PENDING' | 'CONFIRMED' | 'CANCELLED' | 'COMPLETED';\n  notes?: string;\n}\n```\n\nThe `libs/ui` package exports reusable components (`Button`, `Card`, `StarRating`, `TimeSlotPicker`, `MapView`) and a theme object defining colors, typography, spacing, and shadows. All components are tested with Jest and React Native Testing Library.\n\n## 8. API Design & Communication\n### 8.1 RESTful Endpoints\nThe backend exposes a REST API versioned under `/api/v1`. Key endpoints:\n- `GET /businesses?lat=&lng=&radius=&category=&query=` – search with geospatial filters.\n- `GET /businesses/:id` – business detail with services, staff, reviews.\n- `GET /businesses/:id/slots?serviceId=&staffId=&date=` – available time slots.\n- `POST /bookings` – create booking (requires auth).\n- `PATCH /bookings/:id/cancel` – cancel booking.\n- `POST /reviews` – submit review.\n- `GET /users/me` – current user profile.\n\nAll requests use JSON, authenticated via `Authorization: Bearer <token>`. Responses follow a standard envelope: `{ data, meta, error }`.\n\n### 8.2 Real-Time Updates\nBooking slot availability changes are pushed to clients via Supabase Realtime (WebSockets) or polling with React Query's `refetchInterval`. Critical operations (booking creation) use optimistic concurrency control with version numbers to prevent double-booking.\n\n## 9. Authentication & Authorization\nSupabase Auth handles user sign-up, login, and JWT issuance. The NestJS backend verifies the JWT on each request using a custom guard. Roles (`CUSTOMER`, `PROVIDER`, `ADMIN`) are embedded in the JWT claims and enforced via role decorators. Refresh tokens are stored securely in Supabase and rotated on use. Social logins (Google, Apple) are configured in Supabase and the Expo app.\n\n## 10. Data Flow & Caching Strategy\n- **Read-heavy data** (business listings, categories) is cached in Redis with a TTL of 5 minutes, invalidated on updates.\n- **Slot availability** is precomputed and cached per staff/service/date. When a booking is made or cancelled, the affected slots are recalculated and cache is updated atomically.\n- **User-specific data** (bookings, favorites) relies on React Query's stale-while-revalidate pattern, with background refetch on focus.\n- **Geospatial queries** use PostGIS indexes and are cached only for identical bounding boxes to avoid stale results.\n\n## 11. Deployment & CI/CD\n### 11.1 Local Development\nDocker Compose spins up PostgreSQL (with PostGIS), Redis, and a local Supabase instance. The API and mobile app run natively via `pnpm dev`.\n\n### 11.2 CI/CD Pipeline\nGitHub Actions workflows:\n- **CI**: On every PR, runs linting, type checking, unit tests (Jest), and integration tests for both apps. Nx affected commands optimize execution.\n- **Deploy API**: On merge to `main`, builds NestJS Docker image, pushes to container registry, and deploys to a cloud provider (e.g., Railway, Fly.io).\n- **Mobile**: EAS Build is triggered via `eas build --platform all --profile production` on version tags. OTA updates are published for minor fixes.\n\n### 11.3 Environment Management\nSecrets (Stripe keys, Supabase URL, FCM keys) are stored in GitHub Secrets and injected at build time. The mobile app uses `expo-constants` to access environment variables.\n\n## 12. Testing Strategy\n- **Unit Tests**: Jest tests for utility functions, hooks, and NestJS services. Prisma is mocked using `jest-mock-extended`.\n- **Integration Tests**: API endpoints tested with supertest, using a test database seeded with fixtures.\n- **E2E Tests**: Detox or Maestro for critical user flows (booking, login).\n- **Component Tests**: React Native Testing Library for UI components.\n\nNx orchestrates test execution, running only affected projects.\n\n## 13. Security & Compliance\n- **PCI Compliance**: No card data touches our servers; Stripe Elements/React Native SDK handles payment collection.\n- **Data Encryption**: All data in transit uses TLS; sensitive fields (email) are encrypted at rest if needed.\n- **GDPR**: Users can export or delete their data via API endpoints. Consent management for notifications.\n- **Rate Limiting**: Redis-based rate limiter on auth and booking endpoints.\n- **Input Validation**: Class-validator decorators in NestJS DTOs, Zod on the frontend.\n\n## 14. Conclusion\nThis architecture provides a scalable, maintainable foundation for Planity Clone. The monorepo structure with Nx enables code sharing and efficient CI/CD, while the chosen technologies (Expo, NestJS, Prisma, PostGIS, Redis) address the core requirements of real-time booking, geospatial search, and cross-platform mobile delivery.
