Data Flow\n\n### 5.1 Booking Flow\n\n1. **Customer** opens app вҶ’ Expo Router loads home page\n2. **TanStack React Query** fetches nearby businesses (cached 5 min)\n3. Customer searches вҶ’ API Gateway вҶ’ Search module вҶ’ PostgreSQL (full-text + PostGIS)\n4. Customer selects business вҶ’ Business detail page loads (React Query cache)\n5. Customer starts booking вҶ’ Booking module checks availability (Redis cache for slots)\n6. Customer selects time вҶ’ Slot held in Redis with TTL (10 min)\n7. Customer confirms вҶ’ Booking module creates appointment in PostgreSQL (transaction)\n8. Payment module processes payment (if required)\n9. Notification module sends confirmation (push + email)\n10. React Query invalidates relevant caches\n\n### 5.2 Authentication Flow\n\n1. User submits credentials вҶ’ Auth module validates\n2. NestJS generates JWT access token (15 min) + refresh token (30 days)\n3. Refresh token stored in Redis (hashed) for revocation\n4. Mobile app stores tokens in Expo SecureStore\n5. TanStack React Query interceptor attaches access token to requests\n6. On 401, interceptor attempts refresh; if fails, redirects to login\n\n## 6. Key Design Decisions\n\n### 6.1 Monorepo with Nx\n- **Why**: Shared types, code reuse, consistent tooling, parallel builds\n- **Impact**: Faster CI, easier refactoring, single version of truth for types\n\n### 6.2 Prisma with PostgreSQL + PostGIS\n- **Why**: Type-safe queries, auto-generated migrations, geospatial support for location-based search\n- **Impact**: Reduced runtime errors, faster development, accurate distance calculations\n\n### 6.3 TanStack React Query\n- **Why**: Declarative data fetching, caching, background refetching, optimistic updates\n- **Impact**: Better UX (instant UI updates), reduced API calls, simplified state management\n\n### 6.4 Redis for Session and Caching\n- **Why**: Fast in-memory operations, TTL support, pub/sub for real-time updates\n- **Impact**: Low-latency slot holding, rate limiting, session revocation\n\n### 6.5 Expo Router\n- **Why**: File-based routing, deep linking support, type-safe navigation\n- **Impact**: Easier navigation management, SEO-friendly URLs, shareable links\n\n## 7. Security Considerations\n\n- **JWT tokens**: Signed with RS256, short-lived access tokens\n- **Refresh tokens**: Stored hashed in Redis, revocable\n- **Rate limiting**: Per-IP and per-user, implemented via Redis\n- **Input validation**: DTOs with class-validator in NestJS\n- **SQL injection**: Prevented by Prisma's parameterized queries\n- **XSS**: React Native's default escaping\n- **Sensitive data**: Encrypted at rest (PostgreSQL pgcrypto) and in transit (HTTPS)\n\n## 8. Performance Targets\n\n| Operation | Target | Caching Strategy |\n|-----------|--------|------------------|\n| Search results | < 500ms | Redis cache (5 min), PostgreSQL full-text index |\n| Business detail | < 2s | React Query cache (10 min), CDN for images |\n| Booking confirmation | < 5s | Optimistic UI, background sync |\n| Map load | < 3s | Tile caching, viewport-based queries |\n| Login | < 2s | Redis session store |\n\n## 9. Testing Strategy\n\n| Layer | Tools | Coverage Target |\n|-------|-------|-----------------|\n| **Unit** | Jest | 80%+ for services, utilities |\n| **Integration** | Jest + Supertest (NestJS) | 70%+ for API endpoints |\n| **Component** | React Native Testing Library | 60%+ for UI components |\n| **E2E** | Detox (mobile), Playwright (web) | Critical user flows |\n\n## 10. CI/CD Pipeline (GitHub Actions)\n\n1. **Push to main/PR**: Lint, type-check, unit tests, integration tests\n2. **Merge to main**: Build backend Docker image, push to registry\n3. **Deploy backend**: Deploy to Supabase or cloud provider\n4. **Mobile build**: EAS Build for iOS/Android (manual trigger or tag)\n5. **Deploy mobile**: EAS Submit to App Store/Google Play\n\n## 11. Monitoring & Observability\n\n- **Logging**: Winston (NestJS), console.log (mobile, filtered in production)\n- **Error tracking**: Sentry for both backend and mobile\n- **Performance**: React Native Performance Monitor, NestJS metrics endpoint\n- **Uptime**: Health check endpoint (`/health`)\n\n## 12. Future Considerations\n\n- **Microservices**: If scale requires, split NestJS modules into separate services\n- **WebSockets**: For real-time slot updates and chat\n- **Offline support**: WatermelonDB or similar for offline-first data\n- **Feature flags**: LaunchDarkly or similar for gradual rollouts\n- **Internationalization**: i18n support for multiple languages\n\n## 13. Conclusion\n\nThis architecture provides a solid foundation for the Planity Clone, balancing rapid development with production-grade reliability.
