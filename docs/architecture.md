  "content": "# Planity Clone ‚Äì System Architecture\n\n## 1. Introduction\nPlanity Clone is a mobile-first platform for booking appointments with local service businesses. The architecture is designed for scalability, maintainability, and rapid development using a monorepo approach with clear service boundaries. The system follows a modern client-server model: a React Native (Expo) mobile app communicates with a NestJS API, which uses PostgreSQL (with PostGIS) for persistence and Redis for caching. Authentication is delegated to Supabase Auth, while the backend owns all business logic and data.\n\n## 2. High-Level Architecture Overview\nThe system consists of two main deployable units:\n- **Mobile Application** (Expo) ‚Äì cross-platform iOS/Android app with file-based routing (Expo Router). Data fetching is handled by TanStack Query, animations by Reanimated.\n- **Backend API** (NestJS monolith, optionally modular monolith) ‚Äì RESTful JSON API serving business discovery, booking, user settings, and administrative functions.\n\nSupporting infrastructure:\n- **PostgreSQL** with PostGIS extension for geospatial queries (distance-based business search).\n- **Redis** for short-lived caching (available time slots, category lists, rate limiting) and session management.\n- **Supabase** provides authentication as a service; the mobile app uses the Supabase client SDK for sign-up/login, and the backend validates incoming JWTs against Supabase‚Äôs JWKS endpoint.\n\nAll components are developed inside an Nx monorepo, enabling shared TypeScript types, reusable UI components, and unified tooling.\n\n## 4. Mobile App Architecture\n### 5.1 Routing\nExpo Router provides file-based routing with nested layouts. Protected routes (booking, profile) are wrapped in an auth guard layout that checks for a valid session token (from Supabase) and redirects to login if missing. Unauthenticated users can browse the app in guest mode with limited functionality; the guard layout allows public routes.\n\n### 5.2 State Management & Data Fetching\n- **Server state**: All API data is fetched and cached using TanStack Query. Each domain entity (businesses, services, slots) has a custom query hook (`useBusinesses`, `useAvailableSlots`) with proper stale times and cache invalidation after mutations.\n- **Client state**: Global UI state (e.g., selected filters, map/list toggle) is managed with lightweight Zustand stores or React context.\n- **Auth state**: The Supabase client‚Äôs session listener updates a context/global store; access token is attached to API requests automatically via an Axios interceptor.\n\n### 5.3 API Layer\nA centralized API service (`apps/mobile/src/services/api.ts`) configures an Axios instance with base URL, token injection, and response error handling (refresh token rotation). All query hooks consume this service.\n\n### 5.4 Animations\nReact Native Reanimated powers smooth transitions, bottom sheets (for booking confirmation), and gesture-driven interactions (e.g., map cards). Shared element transitions (business image to detail) are implemented where feasible.\n\n## 6. Backend Architecture (NestJS)\n### 6.1 Module Structure\nEach business domain is encapsulated as a NestJS module with clear layers:\n- **Controller** ‚Äì handles HTTP requests (REST endpoints).\n- **Service** ‚Äì business logic and orchestration.\n- **Repository** ‚Äì data access via Prisma (custom repository classes).\n- **DTOs** ‚Äì validation and serialization schemes.\n\nThe modules are largely independent and communicate only through loosely-coupled events (using NestJS EventEmitter) or direct service calls when necessary.\n\n### 6.2 Database & ORM\nPrisma is used as the ORM with migrations managed in `apps/api/src/prisma/`. The schema models all core entities and uses PostGIS geometry types for business locations (`Point`). Geospatial queries (e.g., \"find businesses within 20 km of user's location\") leverage raw SQL through Prisma's `$queryRaw` to utilise PostGIS functions like `ST_DWithin` and `ST_Distance`.\n\n### 6.3 Authentication & Authorization\n- The API does not manage credentials directly. It trusts Supabase Auth JWTs.\n- A global NestJS guard (`SupabaseAuthGuard`) verifies every incoming `Authorization: Bearer <token>` against Supabase‚Äôs JWKS endpoint and extracts user ID, role, and other claims.\n- Role-based access control is implemented via custom decorators (`@Roles(Role.CUSTOMER)`) and a roles guard.\n- Public endpoints (business search, guest browsing) skip authentication.\n\n### 6.4 Caching\nRedis is used for:\n- **Available time slots**: Slots are precomputed and cached for the next 7 days per staff; invalidated on booking or schedule change.\n- **Search autocomplete**: A Redis set/sorted set holds business names and service keywords for fast prefix search.\n- **Category tree**: Static category hierarchy cached for quick retrieval.\n- **Rate limiting**: Per-endpoint rate limiting using Redis-backed counters.\n\n### 6.5 Booking Flow Transaction\nTo handle concurrency and prevent double-booking, the booking creation uses optimistic locking (checking slot availability in a database transaction) and a short-lived Redis lock on the specific slot to serialise concurrent attempts. On successful booking, the slot status is updated in PostgreSQL, cache is invalidated, and a confirmation notification is sent (via a background job queue).\n\n### 6.6 File Storage\nBusiness photos are uploaded directly to Supabase Storage (using the mobile Supabase client), and the resulting URLs are stored in the database. The API validates that uploads belong to the authenticated provider before linking.\n\n## 7. Data Models (Prisma Schema Excerpt)\nKey models:\n- **User**: `id`, `supabaseUid`, `role`, `fullName`, `phone`, `avatarUrl`\n- **Business**: `id`, `ownerId`, `name`, `description`, `location` (PostGIS point), `address`, `phone`, `photos[]`, `categoryId`\n- **Service**: `id`, `businessId`, `name`, `durationMinutes`, `price`, `categoryId`, `isActive`\n- **Staff**: `id`, `userId`, `businessId`, `servicesOffered[]`, `workingHours` (JSON)\n- **Slot**: composite id (`staffId`, `date`, `startTime`), `status` enum (AVAILABLE, BOOKED, BLOCKED)\n- **Appointment**: `id`, `customerId`, `businessId`, `serviceId`, `staffId`, `slotDate`, `slotStart`, `status`, `paymentMethod`, `createdAt`\n- **Review**: `id`, `appointmentId`, `rating`, `text`\n\nFull schema located at `apps/api/src/prisma/schema.prisma`.\n\n## 8. Authentication Flow\n1. User signs up/logs in via Supabase client in the mobile app using email/password or OAuth (Google, Apple).\n2. Supabase returns JWT access and refresh tokens stored securely on the device.\n3. For each API call, the Axios interceptor attaches the access token. If the token is expired, the interceptor transparently refreshes it using the Supabase refresh token before retrying.\n4. Backend NestJS guard decodes the token without network calls using a cached JWKS keyset, validating expiration, signature, and audience. It then fetches/creates the local User record (by `supabaseUid`) if necessary, and injects the user object into the request.\n\n## 10. Geospatial Search Implementation\n- Business coordinates are stored as `geometry(Point, 4326)`.\n- Search endpoint accepts optional `latitude`, `longitude`, `radiusMeters` parameters.\n- Raw query:\n  ```sql\n  SELECT *, ST_Distance(location, ST_MakePoint(:lng, :lat)::geography) AS distance\n  FROM \"Business\"\n  WHERE ST_DWithin(location, ST_MakePoint(:lng, :lat)::geography, :radius)\n  ORDER BY distance\n  LIMIT 20 OFFSET :skip\n  ```\n- For map clustering, results are sent to the mobile app, which uses Supercluster on the client to cluster markers.\n\n## 11. Real-Time Notifications\n- Backend emits an event on appointment creation/status change.\n- Notification module dispatches push notifications via Firebase Cloud Messaging (FCM) or APNs through a background job queue (Bull + Redis).\n- For real-time UI updates (e.g., new review added), the mobile app can optionally subscribe to Supabase Realtime channels for specific tables (reviews, appointments) to invalidate TanStack Query cache, keeping the UI fresh.\n\n## 12.
