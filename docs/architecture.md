  "content": "# Planity Clone — Architecture Document\n\n## 1. System Overview\n\nPlanity Clone is a mobile-first platform connecting customers with local service businesses. The architecture follows a **modular monolith** pattern with clear service boundaries, using a **NestJS** backend, **Expo/React Native** frontend, and **PostgreSQL** with **PostGIS** for geospatial data. The system is designed for scalability, maintainability, and rapid iteration.\n\n## 2. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile & Web | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated | Cross-platform UI, routing, data fetching, animations |\n| Backend | NestJS, TypeScript, Prisma, PostgreSQL, PostGIS, Redis | API server, ORM, database, geospatial, caching |\n| Infrastructure | Docker Compose, GitHub Actions, Supabase, EAS Build, Jest | Containerization, CI/CD, backend services, build, testing |\n| Monorepo | Nx, pnpm | Code organization, dependency management, build orchestration |\n\n## 3. Architecture Principles\n\n- **Clean Architecture**: Separation of concerns into layers (presentation, application, domain, infrastructure).\n- **Domain-Driven Design**: Each bounded context has its own service, database schema, and API.\n- **API-First**: Backend exposes RESTful APIs; frontend consumes via TanStack React Query.\n- **Offline-First**: Guest data, favorites, and cached search results persist locally.\n- **Security**: JWT with refresh tokens, rate limiting, input validation, and secure storage.\n\n## 4. Service Boundaries (Bounded Contexts)\n\n| Service | Responsibility | Database Tables | API Endpoints |\n|---------|---------------|----------------|---------------|\n| **Auth** | User registration, login, OAuth, token management, guest linking | `users`, `refresh_tokens`, `oauth_providers` | `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/forgot-password`, `POST /auth/reset-password`, `POST /auth/link-guest` |\n| **Business** | Business profiles, services, staff, categories, availability | `businesses`, `services`, `staff`, `categories`, `business_hours`, `availability_slots` | `GET /businesses`, `GET /businesses/:id`, `POST /businesses`, `PUT /businesses/:id`, `GET /businesses/:id/services`, `GET /businesses/:id/staff`, `GET /businesses/:id/availability` |\n| **Search** | Full-text search, geospatial queries, autocomplete, filtering | `search_index` (materialized view), `business_locations` (PostGIS) | `GET /search`, `GET /search/autocomplete`, `GET /search/map` |\n| **Booking** | Appointment creation, management, cancellation, reminders | `bookings`, `booking_services`, `booking_history` | `POST /bookings`, `GET /bookings`, `GET /bookings/:id`, `PUT /bookings/:id/cancel`, `GET /bookings/available-slots` |\n| **Review** | Customer reviews, ratings, moderation | `reviews`, `review_reactions` | `POST /reviews`, `GET /reviews`, `PUT /reviews/:id`, `DELETE /reviews/:id` |\n| **Notification** | Push notifications, email, SMS, in-app alerts | `notifications`, `notification_templates` | `POST /notifications/send`, `GET /notifications`, `PUT /notifications/:id/read` |\n| **Admin** | Platform management, analytics, user/business moderation | `admin_users`, `audit_logs`, `platform_metrics` | `GET /admin/dashboard`, `GET /admin/users`, `PUT /admin/businesses/:id/verify`, `GET /admin/analytics` |\n\n## 6. Data Flow\n\n### 6.1 Authentication Flow\n```\nMobile App -> Expo Router -> Auth Screen -> API Gateway (/auth/login) -> Auth Service -> Prisma -> PostgreSQL\n                ^                                                                                    |\n                |                                                                                    v\n                +--- JWT stored in SecureStore <--------- Auth Service returns tokens <------------+\n```\n\n### 6.2 Search Flow\n```\nMobile App -> TanStack Query (useSearch) -> API Gateway (/search?q=...&lat=...&lng=...) -> Search Service\n                ^                                                                                    |\n                |                                                                                    v\n                +--- Cache in Redis <--- Search Service queries PostGIS (ST_DWithin) <--- PostgreSQL\n```\n\n### 6.3 Booking Flow\n```\nMobile App -> Booking Screen -> TanStack Query (useCreateBooking) -> API Gateway (/bookings) -> Booking Service\n                ^                                                                                    |\n                |                                                                                    v\n                +--- Confirmation screen <--- Booking Service creates booking, triggers notification <--- PostgreSQL\n```\n\n## 7. Key Design Decisions\n\n### 7.1 Monorepo with Nx\n- **Why**: Single source of truth for shared types, libraries, and configurations. Nx provides dependency graph, affected commands, and caching for fast CI.\n- **Trade-off**: Slightly more complex initial setup but pays off in maintainability.\n\n### 7.2 NestJS with Modular Architecture\n- **Why**: Each bounded context is a NestJS module with its own controllers, services, and DTOs. This allows independent development and testing.\n- **Trade-off**: More boilerplate than a flat structure, but clear separation.\n\n### 7.3 Prisma + PostgreSQL + PostGIS\n- **Why**: Prisma provides type-safe ORM with migrations. PostGIS enables efficient geospatial queries (e.g., \"find businesses within 5km\").\n- **Trade-off**: Prisma can be slower for complex raw queries, but we use raw SQL for geospatial operations.\n\n### 7.4 Redis for Caching & Sessions\n- **Why**: Cache search results, availability slots, and business details. Also used for rate limiting and session storage.\n- **Trade-off**: Additional infrastructure, but critical for performance.\n\n### 7.5 TanStack React Query\n- **Why**: Handles caching, refetching, pagination, and optimistic updates. Reduces boilerplate for data fetching.\n- **Trade-off**: Learning curve for advanced patterns.\n\n### 7.6 Expo Router\n- **Why**: File-based routing simplifies navigation. Supports deep linking and web compatibility.\n- **Trade-off**: Less flexible than custom navigation, but sufficient for this app.\n\n## 8. Database Schema (Key Tables)\n\n### 8.1 Users\n```prisma\nmodel User {\n  id            String   @id @default(cuid())\n  email         String?  @unique\n  phone         String?  @unique\n  passwordHash  String?\n  role          Role     @default(CUSTOMER) // CUSTOMER, BUSINESS_OWNER, ADMIN\n  guestToken    String?  @unique\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n  bookings      Booking[]\n  reviews       Review[]\n}\n```\n\n### 8.2 Businesses\n```prisma\nmodel Business {\n  id          String   @id @default(cuid())\n  name        String\n  description String?\n  categoryId  String\n  location    Location @relation(fields: [locationId], references: [id])\n  locationId  String   @unique\n  verified    Boolean  @default(false)\n  createdAt   DateTime @default(now())\n  services    Service[]\n  staff       Staff[]\n  bookings    Booking[]\n}\n\nmodel Location {\n  id      String   @id @default(cuid())\n  address String\n  lat     Float\n  lng     Float\n  geom    Unsupported(\"geometry(Point, 4326)\")? // PostGIS\n  business Business?\n}\n```\n\n### 8.3 Bookings\n```prisma\nmodel Booking {\n  id          String   @id @default(cuid())\n  userId      String\n  businessId  String\n  staffId     String?\n  startTime   DateTime\n  endTime     DateTime\n  status      BookingStatus @default(PENDING) // PENDING, CONFIRMED, CANCELLED, COMPLETED\n  totalPrice  Decimal\n  createdAt   DateTime @default(now())\n  user        User     @relation(fields: [userId], references: [id])\n  business    Business @relation(fields: [businessId], references: [id])\n  staff       Staff?   @relation(fields: [staffId], references: [id])\n  services    BookingService[]\n}\n```\n\n## 9. API Design\n\n### 9.1 Authentication\n- `POST /auth/register` — Register with email/phone + password\n- `POST /auth/login` — Login with email/phone + password or OTP\n- `POST /auth/refresh` — Refresh access token\n- `POST /auth/forgot-password` — Send reset code\n- `POST /auth/reset-password` — Reset password with code\n- `POST /auth/link-guest` — Link guest data to registered user\n\n### 9.2 Search\n- `GET /search?q=&lat=&lng=&radius=&category=&priceMin=&priceMax=&rating=&sort=&page=&limit=` — Full search\n- `GET /search/autocomplete?q=` — Autocomplete suggestions\n- `GET /search/map?swLat=&swLng=&neLat=&neLng=&filters=` — Map-based search (bounding box)\n\n### 9.3 Business\n- `GET /businesses/:id` — Business detail with services, staff, reviews\n- `GET /businesses/:id/availability?date=&serviceId=&staffId=` — Available time slots\n- `GET /businesses/:id/reviews?sort=&page=&limit=` — Paginated reviews\n\n### 9.4 Booking\n- `POST /bookings` — Create booking (requires auth)\n- `GET /bookings` — List user's bookings\n- `PUT /bookings/:id/cancel` — Cancel booking\n- `GET /bookings/:id` — Booking detail\n\n## 10. Security\n\n- **Authentication**: JWT access tokens (15 min) + refresh tokens (7 days) stored in SecureStore (mobile) or httpOnly cookies (web).\n- **Rate Limiting**: 5 failed login attempts triggers 30-minute lockout. General API rate limit: 100 requests/min per IP.\n- **Input Validation**: All endpoints validate using class-validator + NestJS pipes.\n- **CORS**: Restricted to known origins.\n- **Database**: Prepared statements via Prisma prevent SQL injection.\n- **Secrets**: Managed via environment variables, never committed.\n\n## 11. Performance & Caching\n\n- **Redis Caching**:\n  - Search results: TTL 5 minutes\n  - Business details: TTL 10 minutes\n  - Availability slots: TTL 1 minute (invalidated on booking)\n- **Database Indexes**: Index on `businesses.location.geom` (GIST), `bookings.startTime`, `users.email`.\n- **Pagination**: All list endpoints use cursor-based pagination for consistency.\n- **Debouncing**: Search autocomplete debounced at 150ms on client.\n\n## 12. Testing Strategy\n\n- **Unit Tests**: Jest for backend services and frontend hooks.\n- **Integration Tests**: Test NestJS modules with real database (testcontainers).\n- **E2E Tests**: Detox for mobile, Playwright for web.\n- **CI/CD**: GitHub Actions runs tests on every PR; EAS Build for mobile builds.\n\n## 13. Deployment\n\n- **Backend**: Docker Compose for local dev; production uses Supabase (managed PostgreSQL) + Redis Cloud.\n- **Mobile**: EAS Build generates iOS/Android builds; distributed via TestFlight/Google Play.\n- **CI/CD Pipeline**:\n  1. Lint & type-check\n  2. Run unit & integration tests\n  3. Build backend Docker image\n  4. Deploy to staging\n  5. Run E2E tests\n  6. Deploy to production\n\n## 14. Conclusion\n\nThis architecture provides a solid foundation for the Planity Clone, balancing performance, maintainability, and developer experience.
