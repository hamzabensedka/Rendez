Overview\n\nThis document describes the complete system architecture for the Planity Clone, a mobile-first appointment booking platform. The architecture follows clean architecture principles, ensuring separation of concerns, testability, and scalability. The system is built using a monorepo structure managed by Nx and pnpm, with a React Native (Expo) frontend and a NestJS backend.\n\n## 2. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile Framework | Expo (React Native) | Cross-platform mobile app |\n| Routing | Expo Router | File-based routing for navigation |\n| State & Data Fetching | TanStack React Query | Server state management, caching, pagination |\n| Animations | React Native Reanimated | Smooth, performant UI animations |\n| Backend Framework | NestJS | Modular, scalable server-side application |\n| ORM | Prisma | Type-safe database access and migrations |\n| Database | PostgreSQL + PostGIS | Relational data with geospatial queries |\n| Cache & Session | Redis | Caching, rate limiting, session store |\n| Build System | Nx | Monorepo management, task orchestration |\n| Package Manager | pnpm | Fast, disk-efficient package management |\n| Containerization | Docker Compose | Local development environment |\n| CI/CD | GitHub Actions | Automated testing, building, deployment |\n| Auth & Backend Services | Supabase | Authentication, storage, real-time subscriptions |\n| Mobile Builds | EAS Build | Cloud-based Expo builds |\n| Testing | Jest | Unit and integration testing |\n\n## 3. Architecture Principles\n\n- **Separation of Concerns**: Each module has a single responsibility.\n- **Layered Architecture**: Presentation, Application, Domain, and Infrastructure layers.\n- **Dependency Inversion**: High-level modules do not depend on low-level modules; both depend on abstractions.\n- **Testability**: All components are designed to be easily testable with mocks and stubs.\n- **Scalability**: Stateless backend services, horizontal scaling via Redis and PostgreSQL.\n- **Offline First**: Local storage and optimistic updates for a seamless user experience.\n\n## 4. Service Boundaries\n\n### 5.1 Frontend (Mobile App)\n\n**Responsibilities:**\n- User interface rendering and navigation\n- Client-side state management (TanStack React Query for server state, Zustand for local state)\n- Offline data caching and optimistic updates\n- Biometric authentication integration\n- Map rendering and geolocation\n- Push notification handling\n\n**Key Modules:**\n- **Auth Module**: Handles login, registration, token storage, biometrics.\n- **Explore Module**: Search, map, categories, business list.\n- **Business Module**: Business detail, service menu, staff list.\n- availability, search\n- Integration with Supabase for auth and storage\n- Background jobs (notifications, reminders)\n\n**Key Modules:**\n- **Auth Module**: Register, login, OAuth, password reset, pricing, add-ons.\n- **Bookings Module**: Slot computation, booking creation, cancellation, waitlist.\n- **Categories Module**: Hierarchical taxonomy management.\n- **Reviews Module**: Review creation, moderation, rating aggregation.\n- **Favorites Module**: Favorite toggle, sync across devices.\n- **Notifications Module**: Push notification scheduling and delivery.\n- **Payments Module**: Payment processing, refunds, invoice generation.\n- **Admin Module**: Dashboard, user management, analytics.\n\n### 5.3 Shared Libraries\n\n- **shared-types**: DTOs, enums, interfaces used by both frontend and backend.\n- **ui-kit**: Reusable React Native components (buttons, inputs, cards, modals).\n- **utils**: Common functions (date formatting, validation, geolocation helpers).\n\n## 6. Data Flow\n\n1. **User Action** → Mobile app dispatches a query/mutation via TanStack React Query.\n2. **API Request** → HTTP request to NestJS backend (REST endpoints).\n3. **Authentication** → JWT token validation via AuthGuard.\n4. **Business Logic** → NestJS service layer processes the request.\n5. **Data Access** → Prisma ORM interacts with PostgreSQL (with PostGIS for spatial queries).\n6. **Caching** → Redis caches frequent queries (business details, categories).\n7. **Response** → JSON response returned to mobile app.\n8. **State Update** → React Query updates cache and triggers UI re-render.\n\n## 7. Key Design Decisions\n\n### 7.1 Authentication\n- JWT access tokens (15 min) + refresh tokens (7 days) stored securely in Keychain/Keystore.\n- Supabase Auth for OAuth (Google, Apple) and phone verification.\n- Biometric login uses local authentication only.\n\n### 7.2 Booking Slot Computation\n- Business defines weekly recurring slots (e.g., Mon-Fri 9:00-17:00).\n- Slots are computed on-the-fly based on existing bookings and staff availability.\n- Optimistic locking with a 5-minute hold on selected slots (stored in Redis).\n- PostGIS used for spatial queries (find businesses within radius).\n\n### 7.3 Search\n- Full-text search via PostgreSQL tsvector with GIN indexes.\n- Autocomplete uses Redis sorted sets for fast prefix matching.\n- Geospatial queries use PostGIS `ST_DWithin` with geohash pre-filtering.\n\n### 7.4 Offline Support\n- TanStack React Query persists cache to AsyncStorage.\n- Critical data (categories, favorite business IDs) cached locally.\n- Booking mutations use optimistic updates with rollback on failure.\n\n## 8. Deployment Architecture\n\n```\n[User Device] ↔ [CDN (CloudFront)] ↔ [Load Balancer (ALB)] ↔ [NestJS API (ECS Fargate)]\n                                                              ↕\n                                                    [Redis (ElastiCache)]\n                                                              ↕\n                                                    [PostgreSQL + PostGIS (RDS)]\n                                                              ↕\n                                                    [Supabase (Auth, Storage)]\n```\n\n- **Mobile App**: Built with EAS Build, distributed via App Store and Google Play.\n- **API**: Dockerized NestJS app deployed on AWS ECS Fargate (auto-scaling).\n- **Database**: Amazon RDS for PostgreSQL with PostGIS extension.\n- **Cache**: Amazon ElastiCache for Redis.\n- **CDN**: CloudFront for static assets (images, fonts).\n- **CI/CD**: GitHub Actions runs tests, builds, and deploys to staging/production.\n\n## 9. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions and services.\n- **Integration Tests**: NestJS testing utilities with test database (PostgreSQL + Redis).\n- **E2E Tests**: Detox for mobile, Supertest for API.\n- **CI Pipeline**: Tests run on every pull request; coverage threshold enforced.\n\n## 10. Security Considerations\n\n- All API endpoints require HTTPS.\n- JWT tokens stored securely (not in AsyncStorage).\n- Rate limiting on auth endpoints (Redis-based).\n- Input validation and sanitization at the API layer.\n- PCI compliance for payment data (tokenization via Stripe).\n- GDPR compliance: data export, account deletion with 30-day grace period.\n\n## 11. Monitoring and Observability\n\n- **Logging**: Structured JSON logs via NestJS Logger (shipped to CloudWatch).\n- **Metrics**: Custom metrics (booking rate, error rate, latency) via Prometheus.\n- **Tracing**: Distributed tracing with OpenTelemetry.\n- **Alerts**: CloudWatch alarms for critical errors and performance degradation.\n\n## 12. Conclusion\n\nThis architecture provides a solid foundation for building a scalable, maintainable, and feature-rich appointment booking platform. The monorepo structure with Nx ensures code sharing and consistency, while the separation into frontend, backend, and shared libraries promotes clean boundaries and independent development.",
