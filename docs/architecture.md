  "content": "# Planity Clone — Architecture Document\n\n## 1. System Overview\n\nPlanity Clone is a mobile-first appointment booking platform connecting customers with local service businesses. The architecture follows a **clean architecture** approach with clear separation of concerns, enabling independent development, testing, and deployment of each bounded context.\n\n### 1.1 Architecture Principles\n\n- **Separation of Concerns**: Each module has a single responsibility.\n- **Domain-Driven Design**: Core business logic isolated from infrastructure.\n- **API-First**: All client-server communication via well-defined REST/GraphQL APIs.\n- **Offline-First**: Mobile app works with degraded functionality when offline.\n- **Security by Design**: JWT with refresh rotation, biometric auth, encrypted data at rest.\n- **Scalability**: Stateless services, horizontal scaling, caching layers.\n\n### 1.2 Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile | Expo, React Native, TypeScript | Cross-platform mobile app |\n| Routing | Expo Router | File-based routing with deep linking |\n| State & Data | TanStack React Query | Server state management, caching, pagination |\n| Animations | React Native Reanimated | Smooth UI transitions, gesture handling |\n| Backend | NestJS, TypeScript | Modular server-side framework |\n| ORM | Prisma | Type-safe database access |\n| Database | PostgreSQL + PostGIS | Relational data + geospatial queries |\n| Cache | Redis | Session store, rate limiting, job queues |\n| Monorepo | Nx, pnpm | Code sharing, build orchestration |\n| Containerization | Docker Compose | Local development environment |\n| CI/CD | GitHub Actions | Automated testing, builds, deployments |\n| Auth & Backend | Supabase (optional) | Auth, real-time subscriptions, storage |\n| Build & Deploy | EAS Build | Expo app builds and OTA updates |\n| Testing | Jest | Unit and integration tests |\n\n## 2. Service Boundaries (Bounded Contexts)\n\n### 3.1 Auth Service\n- **Responsibility**: User registration, login, token management, OAuth integration.\n- **Endpoints**: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/forgot-password`, `POST /auth/reset-password`.\n- **Data**: Users table, refresh tokens (Redis).\n- **Dependencies**: Redis (session store), Supabase Auth (optional).\n\n### 3.2 User Service\n- **Responsibility**: User profile management, preferences, favorites.\n- **Endpoints**: `GET /users/me`, `PATCH /users/me`, `GET /users/me/favorites`, `POST /users/me/favorites`, `DELETE /users/me/favorites/:id`.\n- **Data**: Users, favorites tables.\n\n### 3.3 Business Service\n- **Responsibility**: CRUD for businesses, services, staff, categories, availability.\n- **Endpoints**: `GET /businesses`, `GET /businesses/:id`, `POST /businesses`, `PATCH /businesses/:id`, `GET /businesses/:id/services`, `GET /businesses/:id/staff`, `GET /businesses/:id/availability`.\n- **Data**: Businesses, services, staff, categories, availability slots.\n- **Geospatial**: PostGIS for location queries.\n\n### 3.4 Booking Service\n- **Responsibility**: Appointment creation, rescheduling, cancellation, slot management.\n- **Endpoints**: `POST /bookings`, `GET /bookings`, `GET /bookings/:id`, `PATCH /bookings/:id`, `DELETE /bookings/:id`, `GET /bookings/slots?businessId=&date=`.\n- **Data**: Bookings, booking_services, booking_addons.\n- **Concurrency**: Pessimistic locking via Redis for slot hold (5 min TTL).\n\n### 3.5 Search Service\n- **Responsibility**: Full-text search, filtering, geospatial queries, autocomplete.\n- **Endpoints**: `GET /search?q=&lat=&lng=&radius=&category=&priceMin=&priceMax=&rating=&openNow=`.\n- **Data**: Elasticsearch index (optional) or PostgreSQL full-text search with PostGIS.\n- **Caching**: Redis for trending searches, autocomplete suggestions.\n\n### 3.6 Review Service\n- **Responsibility**: Reviews, ratings, moderation.\n- **Endpoints**: `GET /businesses/:id/reviews`, `POST /reviews`, `PATCH /reviews/:id`, `DELETE /reviews/:id`.\n- **Data**: Reviews table, review_photos.\n\n### 3.7 Notification Service\n- **Responsibility**: Push notifications (Expo Push API), email (SendGrid), in-app notifications.\n- **Endpoints**: `POST /notifications/send`, `GET /notifications`.\n- **Queue**: Bull with Redis for async delivery.\n\n### 3.8 Admin Service\n- **Responsibility**: Platform management, analytics, dispute resolution.\n- **Endpoints**: `GET /admin/dashboard`, `GET /admin/users`, `PATCH /admin/businesses/:id/verify`, etc.\n- **Authorization**: Role-based access (Admin role).\n\n## 4. Data Flow & Communication\n\n### 4.1 Mobile App to Backend\n- **Protocol**: HTTPS REST (JSON).\n- **Authentication**: Bearer token in `Authorization` header.\n- **State Management**: TanStack React Query handles caching, refetching, optimistic updates.\n- **Offline Support**: AsyncStorage for persisted queries; mutations queued and retried.\n\n### 4.2 Backend Services\n- **Inter-service Communication**: Synchronous via HTTP (NestJS microservices or direct module calls). For high-volume async tasks (notifications, email), use Redis-backed message queues.\n- **Database**: Each service has its own schema in PostgreSQL; shared via Prisma relations where necessary.\n\n### 4.3 Authentication Flow\n```\n1. User logs in -> POST /auth/login\n2. Server validates credentials -> returns { accessToken, refreshToken }\n3. Client stores tokens securely (expo-secure-store)\n4. Access token sent with every request\n5. On 401 -> interceptor calls POST /auth/refresh -> new access token\n6. Refresh token rotation: old refresh token invalidated on use\n```\n\n## 5. phone\n- website\n- email\n- hours (JSON)\n- amenities (JSON)\n- cancellationPolicy\n- isVerified\n- rating (computed)\n- createdAt, updatedAt\n\n**Service**\n- id (UUID, PK)\n- businessId (FK -> Business)\n- categoryId (FK -> Category)\n- name\n- description\n- duration (minutes)\n- price (decimal)\n- isActive\n- createdAt, updatedAt\n\n**Staff**\n- id (UUID, PK)\n- businessId (FK -> Business)\n- userId (FK -> User, optional)\n- name\n- bio\n- photoUrl\n- specialties (JSON)\n- averageRating\n- isActive\n\n**Availability**\n- id (UUID, PK)\n- staffId (FK -> Staff, nullable)\n- businessId (FK -> Business)\n- dayOfWeek (int 0-6)\n- startTime (time)\n- endTime (time)\n- isRecurring (boolean)\n- date (date, nullable for recurring)\n\n**Booking**\n- id (UUID, PK)\n- customerId (FK -> User)\n- businessId (FK -> Business)\n- staffId (FK -> Staff, nullable)\n- startTime (timestamptz)\n- endTime (timestamptz)\n- status (PENDING, CONFIRMED, CANCELLED, COMPLETED)\n- totalPrice Caching Strategy (Redis)\n\n| Cache Key | TTL | Purpose |\n|-----------|-----|---------|\n| `session:{userId}` | 7 days | Refresh token storage |\n| `slot_hold:{slotId}` | 5 min | Pessimistic lock during booking |\n| `trending_searches` | 1 hour | Autocomplete suggestions |\n| `business:{id}` | 5 min | Business detail cache |\n| `user:{id}:favorites` | 10 min | Favorites list |\n| `rate_limit:{ip}` | 1 min | API rate limiting |\n\n## 7. Mobile Architecture (Expo + React Native)\n\n### 7.1 Navigation\n- **Expo Router** with file-based routing.\n- **Deep linking**: `planity://business/123`, `planity://booking/456`.\n- **Auth guard**: Root layout checks auth state; redirects to login if unauthenticated.\n\n### 7.2 State Management\n- **Server State**: TanStack React Query for all API data.\n- **Client State**: React Context for auth, theme, and UI state (modals, toasts).\n- **Persisted State**: AsyncStorage for guest data (favorites, cart).\n\n### 7.3 Offline Support\n- React Query `networkMode: 'offlineFirst'`.\n- Mutations queued with `onlineManager`.\n- Guest data stored locally; merged on registration.\n\n### 7.4 Animations\n- **React Native Reanimated** for:\n  - Bottom sheet transitions\n  - Map marker clustering animations\n  - Booking flow step transitions\n  - Pull-to-refresh\n  - Image carousel swipe\n\n### 7.5 Testing\n- **Jest** for unit tests (hooks, utils).\n- **React Native Testing Library** for component tests.\n- **Detox** for end-to-end tests (optional).\n\n## 8. Backend Architecture (NestJS)\n\n### 8.1 Module Structure\nEach module follows NestJS conventions:\n- `module.ts` - Module definition\n- `controller.ts` - REST endpoints\n- `service.ts` - Business logic\n- `dto/` - Data transfer objects (validated with class-validator)\n- `entities/` - Prisma models or TypeORM entities\n- `guards/` - Auth guards\n- `interceptors/` - Logging, transformation\n\n### 8.2 Authentication & Authorization\n- **Passport.js** with JWT strategy.\n- **Guards**: `@UseGuards(JwtAuthGuard)` for protected routes.\n- **Roles**: Custom `@Roles('ADMIN')` decorator.\n- **Rate Limiting**: `@nestjs/throttler` with Redis store.\n\n### 8.3 Error Handling\n- Global exception filter returning consistent error format.\n- HTTP status codes: 200 (success), 201 (created), 400 (validation), 401 (unauth), 403 (forbidden), 404 (not found), 409 (conflict), 429 (rate limit), 500 (server error).\n\n## 9. Deployment & CI/CD\n\n### 9.1 Local Development\n```bash\ndocker-compose up -d          # PostgreSQL, Redis, API\npnpm install                  # Install dependencies\npnpm nx run api:dev           # Start NestJS in watch mode\npnpm nx run mobile:start      # Start Expo dev server\n```\n\n### 9.2 CI/CD Pipeline (GitHub Actions)\n\n**ci.yml**\n- Trigger: Push to main, PR to main.\n- Steps:\n  1. Checkout code\n  2. Setup Node.js 18, pnpm\n  3. Install dependencies\n  4. Lint (ESLint, Prettier)\n  5. Type check (tsc --noEmit)\n  6. Run unit tests (Jest)\n  7. Build mobile app (EAS Build for preview)\n  8. Build API (Docker image)\n\n**deploy.yml**\n- Trigger: Merge to main.\n- Steps:\n  1. Build and push Docker image to container registry (Docker Hub / ECR)\n  2. Deploy API to Kubernetes or VPS (via SSH)\n  3. Run Prisma migrations\n  4. Build and submit mobile app to stores (EAS Submit)\n\n### 9.3 Environment Variables\n- `.env` files for local development.\n- GitHub Secrets for CI/CD.\n- Secrets manager (AWS Secrets Manager / Vault) for production.\n\n## 10. Security Considerations\n\n- **Data Encryption**: All PII encrypted at rest (PostgreSQL pgcrypto).\n- **HTTPS**: Enforced in production; HSTS headers.\n- **JWT**: Short-lived access tokens (15 min); refresh tokens rotated.\n- **Input Validation**: Zod schemas on both client and server.\n- **SQL Injection**: Prevented by Prisma parameterized queries.\n- **Rate Limiting**: 100 requests/min per IP; 10 auth attempts/min per IP.\n- **CORS**: Whitelist mobile app origins.\n\n## 11.
