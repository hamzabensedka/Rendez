The architecture follows a clean, layered approach separating concerns into mobile client, API gateway, backend services, and data stores. Expo managed workflow powers the React Native client, while NestJS provides a modular server‑side framework. Communication is REST/GraphQL over HTTPS, with real‑time updates via WebSocket (or Redis pub/sub) where needed. Background jobs are handled by BullMQ backed by Redis. All services are containerized with Docker and orchestrated via Docker Compose for local development; CI/CD uses GitHub Actions, EAS Build for Expo, and Supabase for optional hosted Postgres/PostGIS.\n\n## 2. Technology Stack\n- Mobile: Expo (SDK 49), React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated, React Native SVG, Expo Constants, Expo SecureStore.\n- API Gateway / BFF: NestJS (Node.js) with TypeScript, using Express adapter.\n- Backend Services: NestJS modules for Auth, Business, Booking, Provider Portal, Admin, Payments, Notifications, Reviews.\n- ORM: Prisma with PostgreSQL + PostGIS extension.\n- Cache / PubSub: Redis (used for session store, rate limiting, BullMQ job queue, and real‑time notifications).\n- Background Jobs: BullMQ (Redis-backed).\n- Monorepo: Nx workspace managing apps/libs.\n- Package Manager: pnpm.\n- Containerization: Docker, Docker Compose.\n- CI/CD: GitHub Actions (lint, test, build), EAS Build (Expo), Supabase (optional DB hosting).\n- Testing: Jest (unit/integration), React Native Testing Library, @nestjs/testing.\n- Design System: Custom UI library built with React Native Reanimated for animations, shared via Nx lib.\n\n## 3. Data Flow\n### 5.1 User Authentication Flow\n1. Mobile app collects credentials → calls /auth/login (NestJS).\n2. Auth service validates, creates JWT (access + refresh) stored in SecureStore.\n3. Subsequent API calls include Authorization: Bearer <access>; Auth guard verifies.\n4. Refresh token endpoint rotates tokens.\n\n### 5.2 Business Search & Discovery (Guest)\n1. Mobile sends GET /businesses?query=&category=&lat=&lng=&radius=.\n2. Business Service uses PostGIS ST_DWithin for geo‑filtering, full‑text search on name/category.\n3. Results cached in Redis (TTL 5 min) for frequent queries.\n4. Mobile uses React Query to fetch, paginate, and invalidate on map move.\n\n### 5.3 Map‑Based Search\nSame endpoint as above; map component receives GeoJSON features; Reanimated handles smooth pan/zoom.\n\n### 5.4 Booking Flow\n1. User selects service → mobile calls /booking/slots?businessId=&date= (Booking Service).\n2. Booking Service computes available slots: reads business schedule, existing bookings, staff availability; returns slots.\n3. User picks slot → POST /booking/appointments with payload.\n4. Service validates, creates appointment record, publishes appointment.created event to Redis pub/sub.\n5. Notification Service worker receives event, enqueues BullMQ job to send push/email.\n6. Payment Service invoked if payment required; after successful payment, appointment status set to confirmed.\n\n### 5.5 Real‑Time Availability Updates\n- When an appointment is created/cancelled, Booking Service publishes to Redis channel availability:{businessId}.\n- Mobile subscribes via WebSocket (or polling with React Query refetch interval) to update slot list instantly.\n\n### 5.6 Provider Portal & Admin\n- Role‑based guards (RolesGuard) restrict endpoints.\n- Provider can CRUD own business, staff, services; changes trigger cache invalidation.\n- Admin can view all entities, run batch jobs via BullMQ.\n\n### 5.7 Reviews & Ratings\n- POST /reviews creates review; triggers event to update business average rating (materialized view or computed on read).\n\n### 5.8 Background Jobs\n- BullMQ defines queues: notifications, slot-recalc, cleanup.\n- Workers reside in NestJS modules; they process jobs, retry with exponential backoff.\n\n## 6.
