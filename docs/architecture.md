# Planity Clone Architecture\n\n## Overview\nThe Planity Clone is a mobile-first appointment booking platform connecting users with local businesses. The architecture follows a clean, modular approach using a monorepo managed by Nx and pnpm, separating concerns into frontend, backend, shared libraries, and infrastructure.\n\n## Tech Stack\n- **Frontend**: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Backend**: NestJS, Prisma ORM, PostgreSQL with PostGIS extension, Redis, BullMQ (for background jobs)\n- **Tooling**: Nx, pnpm, Docker Compose, GitHub Actions, EAS Build, Jest, Supabase (for auth & storage optional)\n- **DevOps**: Docker Compose for local dev, CI/CD pipelines, automated testing\n\n## High-Level Components\n1. **Mobile App (apps/mobile)** – Expo managed workflow, handles UI, navigation, state management, offline caching.\n2. **API Server (apps/api)** – NestJS application exposing REST/GraphQL endpoints, business logic, validation, integration with Prisma.\n3. **Shared Libraries (libs)** – Type‑safe DTOs, utility functions, design system components, API client, constants.\n4. **Database** – PostgreSQL (with PostGIS for geo‑queries), Redis for caching and job queue.\n5. **Infrastructure** – Docker Compose defines services for api, db, redis, worker; GitHub Actions runs tests, builds, deploys; EAS Build creates iOS/Android binaries.\n\n## Folder Structure (Nx Monorepo)\nplanity-clone/\n├─ apps/\n│  ├─ mobile/          # Expo React Native app\n│  │   ├─ src/\n│  │   │   ├─ assets/          # images, fonts, icons\n│  │   │   ├─ components/      # reusable UI components (design system)\n│  │   │   ├─ screens/         # Expo Router Service Boundaries & Responsibilities\n- **Mobile App**\n  - UI presentation, gesture handling, Reanimated animations.\n  - State management via React Query (server state) and optional local store for UI state.\n  - Navigation via Expo Router (file‑based routing).\n  - Offline caching of business listings and user data.\n  - Calls API client (libs/api-client) which wraps React Query.\n- **API Server**\n  - Auth: JWT issuance, refresh, integration with Supabase (optional) for email/password.\n  - Business Module: CRUD, geo‑search using PostGIS (`ST_DWithin`), pagination.\n  - Service Module: service catalog, pricing, duration.\n  - Booking Module: slot computation based on business operating hours, existing bookings, timezone handling.\n  - User Module: profile, favorites, password change.\n  - Review Module: create, read, respond.\n  - Payment Module: stub/gateway integration (e.g., Stripe) with webhook handling.\n  - Notification Module: enqueue jobs to BullMQ (email, SMS, push); workers process queues.\n  - Admin & Provider Portals: protected roles (RBAC) using NestJS guards.\n  - Validation: class‑validator + DTOs.\n  - Logging: Pino logger.\n  - Health checks and metrics.\n- **Shared Libraries**\n  - **design-system**: reusable UI components with consistent theming, dark mode support, animations via Reanimated.\n  - **api-client**: typed React Query hooks (`useBusinesses`, `useCreateBooking`, etc.) generated from OpenAPI or manual.\n  - **types**: ensures TypeScript safety across mobile and API (DTOs, enums).\n  - **utils**: shared functions (date formatting, geo calculations, validation) to avoid duplication.\n  - **config**: zod schemas for env variables, used by both apps.\n- **Database**\n  - PostgreSQL with PostGIS extension for location‑based queries.\n  - Tables: users, businesses, services, bookings, reviews, payments, favorites, notifications.\n  - Indexes: GIST on location columns, composite indexes for frequent queries.\n  - Redis: caching of frequent queries (nearby businesses), session store, BullMQ job queue.\n- **Infrastructure**\n  - Docker Compose defines services: `api`, `db`, `redis`, `worker` (BullMQ).\n  - Each service has its own Dockerfile; volumes for Prisma migrations.\n  - GitHub Actions runs on push/pull request: install dependencies, lint, test (Jest), build Docker images, push to registry, trigger EAS Build for mobile.\n  - EAS Build configured for iOS and Android, with incremental builds and distribution to internal/test tracks.\n  - Monitoring: optional integration with Sentry/logging services (not required for MVP).\n\n## Data Flow Examples\n1. **Guest Browse**\n   - Mobile sends GET `/businesses?lat=&lng=&radius=` via React Query.\n   - API uses Prisma with PostGIS `ST_DWithin` to fetch nearby businesses, returns DTO list.\n   - Mobile caches response, displays list via FlatList using design‑system components.\n2. **Booking Flow**\n   - User selects service → mobile calls `/bookings/check-availability` with businessId, serviceId, date.\n   - API computes slots: generates possible start times based on business opening hours, service duration, subtracts existing bookings from DB.\n   - Returns array of free slots.\n   - User picks slot → mobile calls `/bookings` with payload (businessId, serviceId, slot, userId).\n   - API creates booking record, enqueues payment job (if prepaid) and confirmation notification job.\n   - Returns booking confirmation; mobile shows success screen.\n3. **Notification**\n   - After booking confirmation, API pushes job to `notifications` queue in Redis.\n   - Worker processes job: sends email via SendGrid, push via Expo push notification service, SMS via Twilio.\n   - Updates notification status in DB.\n\n## Development Workflow\n- **Local Dev**: `docker compose up` starts api, db, redis, worker. Run `nx serve mobile` for Expo dev client.\n- **Testing**: Jest unit tests for libs and api; e2e tests using Detox for mobile (optional).\n- **CI**: On PR, run `nx affected:test`, `nx affected:lint`, build Docker images, push to registry.\n- **CD**: On merge to main, trigger EAS Build for preview builds; Docker images deployed to staging/k8s (if applicable).\n- **Environment**: Use `.env` files; CI injects secrets via GitHub Secrets.\n\n## Quality Attributes\n- **Scalability**: Stateless API workers, horizontal scaling via container orchestrator; Redis for caching; PostGIS for efficient geo queries.\n- **Maintainability**: Clear module boundaries, shared types, linting (ESLint, Prettier), unit test coverage >80%.\n- **Performance**: React Query caching, pagination, indexed DB queries, Reanimated 65fps animations.\n- **Security**: JWT authentication, role‑based access, input validation, HTTPS, secrets management, OWASP guidelines.\n- **Reliability**: BullMQ with retries, dead‑letter queues, health checks, automated backups (managed PostgreSQL).\n\n## Future Extensions\n- Web admin dashboard (Next.js) sharing same NestJS API.\n- GraphQL endpoint alongside REST for flexible client queries.\n- Push notification segmentation via topics.\n- AI‑driven recommendation engine for businesses.\n- Multi‑currency payment support.\n\n---\n*End of Architecture Document.*
