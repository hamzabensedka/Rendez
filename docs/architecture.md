  "content": "# Planity Clone - System Architecture\n\n## 1. Overview\nPlanity Clone is a mobile-first platform for beauty/wellness bookings. The system is built as an Nx monorepo managed with pnpm, containing an Expo React Native mobile app, a NestJS backend API, BullMQ workers, and shared libraries. Data is stored in PostgreSQL with PostGIS (via Supabase) and cached in Redis.\n\n## 2. Monorepo Structure (Nx + pnpm)\nThe root layout:\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/            # Expo + React Native (Expo Router)\n│   ├── backend/           # NestJS API\n│   ├── worker/            # BullMQ workers (NestJS)\n│   └── web/               # Provider/Admin portal (React via Expo web or Vite)\n├── libs/\n│   ├── types/             # Shared TS types & API contracts\n│   ├── ui/                # Design system (Button, Card, Input)\n│   ├── api-client/        # Typed fetch wrapper for React Query\n│   ├── prisma/            # Prisma schema & migrations\n│   └── config/            # Shared eslint, jest, tsconfig\n├── tools/\n├── docker-compose.yml     # Postgres, Redis for local dev\n├── nx.json\n├── package.json\n└── pnpm-workspace.yaml\n```\n\n## 3. Applications\n### 3.1 Mobile App (Expo)\n- **Stack**: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated.\n- **Features**: Guest browse, auth, search, map (P1), business detail, booking flow, appointments, favorites, profile, payments, notifications.\n- **State**: React Query for server state; secure storage for JWT; Reanimated for transitions.\n- **Build**: EAS Build for iOS/Android.\n\n### 3.2 Backend API (NestJS)\n- **Stack**: NestJS, Prisma, PostgreSQL+PostGIS, Redis, Supabase Auth.\n- **Responsibilities**: REST/JSON API consumed by mobile and web. Implements business logic, validation, authorization.\n\n### 3.3 Worker App (BullMQ)\n- Processes jobs: send push/email notifications, invalidate slot cache, analytics. Uses Redis.\n\n### 3.4 Web Portals (P1)\n- Provider and Admin dashboards share libs/ui and libs/api-client. Can be built with Expo web or a lightweight React app.\n\n## 4. Shared Libraries\n- **@planity/types**: DTOs, entities, enum definitions used by frontend and backend.\n- **@planity/ui**: Reusable components, design tokens, typography.\n- **@planity/api-client**: Configured fetch/axios instance with React Query hooks.\n- **@planity/prisma**: Single Prisma schema, seed scripts, migration runner.\n- **@planity/config**: Jest, ESLint, TS base configs.\n\n## 5. Backend Service Boundaries (NestJS Modules)\nEach module owns a domain:\n- **AuthModule**: Integrates Supabase Auth (email, phone OTP, social). Issues JWT, handles password reset.\n- **UsersModule**: Profile, addresses, notification preferences.\n- **BusinessesModule**: CRUD business profiles, staff, images (Supabase Storage).\n- **CategoriesModule**: Taxonomy management (admin).\n- **ServicesModule**: Service offerings per business.\n- **AvailabilityModule**: Slot computation using business hours, staff shifts, service duration, buffers; uses PostGIS for radius queries; caches in Redis.\n- **BookingModule**: Multi-step booking, appointment creation, cancellation, reschedule.\n- **PaymentsModule**: Stripe integration, deposits, refunds.\n- **ReviewsModule**: Verified reviews, ratings aggregation.\n- **FavoritesModule**: User favorites sync.\n- **NotificationsModule**: Push (Firebase/Supabase) and email templates; enqueues BullMQ jobs.\n- **AdminModule**: Platform governance, metrics.\n- **ProviderModule**: Business owner portal APIs.\n- **JobsModule**: BullMQ queue definitions, dead-letter handling.\n\n## 6. Data Layer\n- **PostgreSQL + PostGIS** (Supabase): Core relational data and geographic queries (map search, radius).\n- **Prisma**: ORM with schema in libs/prisma. Migrations versioned.\n- **Redis**: Caches computed availability slots; serves as BullMQ broker.\n- **Supabase Storage**: Business images, avatars.\n\nKey entities: User, Business, Category, Service, Staff, Appointment, Review, Favorite, Payment, Notification, Job.\n\n## 7. Cross-Cutting Concerns\n- **Authentication**: Supabase Auth handles identity; NestJS guards validate JWT. Mobile stores token in Expo SecureStore.\n- **Caching**: Availability results cached in Redis with keys like `slots:{businessId}:{date}`. Invalidation via worker on appointment change.\n- **Async Processing**: BullMQ ensures reliable notifications, reminders, cache invalidation. Retry with backoff; dead-letter queue.\n- **Testing**: Jest for unit/integration; React Testing Library for components; Prisma test DB.\n- **CI/CD**: GitHub Actions run lint, typecheck, Jest, Nx affected; EAS Build for mobile; Docker images for backend; Supabase project for cloud DB.\n- **Local Dev**: docker-compose.yml spins up Postgres+PostGIS and Redis; Supabase CLI for auth emulation.\n\n## 8. Example Booking Flow\n1. Mobile calls `GET /businesses/:id/slots?serviceId=&date=` (AvailabilityModule, Redis-cached).\n2. User confirms -> `POST /bookings` (BookingModule) with payment intent from PaymentsModule.\n3. On success, NotificationsModule enqueues confirmation job; worker sends push/email.\n4. JobsModule invalidates slot cache for affected staff.\n\n## 9. Summary\nThe architecture enforces separation of concerns via Nx libraries and NestJS modules, uses Supabase for managed Postgres/PostGIS and auth, Redis+BullMQ for scale, and Expo tooling for mobile delivery.",
