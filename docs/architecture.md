  "content": "# Planity Clone — System Architecture Document\n\n## 1. Overview\n\nThis document describes the complete system architecture for the Planity Clone, a multi-platform appointment booking platform. The architecture is designed for scalability, maintainability, and clear separation of concerns, leveraging a monorepo structure with Nx, a mobile-first frontend (Expo/React Native), and a modular backend (NestJS).\n\n## 2. High-Level Architecture\n\nThe system follows a client-server architecture with a clear boundary between the mobile/web frontends and the backend API. The backend is decomposed into microservices (or service modules) that communicate via REST and internal events. A shared library layer provides types, utilities, and validation schemas.\n\n### 2.1 Key Principles\n- **Clean Architecture**: Dependency inversion, domain-driven design, separation of concerns.\n- **Monorepo**: Single source of truth for code, types, and configuration using Nx and pnpm.\n- **Type Safety**: TypeScript throughout, shared types between frontend and backend.\n- **Testability**: Unit, integration, and E2E tests using Jest.\n- **DevOps Ready**: Docker Compose for local development, GitHub Actions for CI/CD, EAS Build for mobile deployment.\n\n### 2.2 Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile Frontend | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated | Cross-platform mobile apps (iOS/Android) |\n| Web Frontend | React, Next.js (or separate React app) | Customer portal, business owner portal, admin dashboard |\n| Backend | NestJS, TypeScript | REST API, business logic, event handling |\n| Database | PostgreSQL with PostGIS | Relational data, geospatial queries |\n| Cache & Session | Redis | Caching, rate limiting, session store, pub/sub |\n| ORM | Prisma | Type-safe database access, migrations |\n| CI/CD | GitHub Actions, EAS Build | Automated testing, building, deployment |\n| Containerization | Docker Compose | Local development environment |\n| Auth & Storage | Supabase | Authentication (optional), file storage |\n\n## 3. Folder Structure\n\nThe monorepo is organized using Nx with the following top-level structure:\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   ├── components/            # Shared UI components\n│   │   ├── hooks/                 # Custom hooks\n│   │   ├── services/              # API client, auth service\n│   │   ├── utils/                 # Utilities\n│   │   └── app.json\n│   ├── web/                       # Web frontend (Next.js or CRA)\n│   ├── admin/                     # Admin dashboard (separate React app)\n│   └── api/                       # NestJS backend\n│       ├── src/\n│       │   ├── modules/           # Feature modules (auth, booking, etc.)\n│       │   ├── common/            # Shared decorators, guards, filters\n│       │   ├── config/            # Configuration files\n│       │   ├── database/          # Prisma service, migrations\n│       │   └── main.ts\n│       └── test/\n├── libs/\n│   ├── shared/                    # Shared types, interfaces, enums\n│   │   ├── src/\n│   │   │   ├── types/\n│   │   │   ├── dto/\n│   │   │   └── constants/\n│   │   └── index.ts\n│   ├── ui/                        # Shared React Native UI components\n│   └── utils/                     # Shared utilities (date, validation)\n├── tools/                         # Nx workspace scripts\n├── docker-compose.yml\n├── nx.json\n├── package.json\n├── pnpm-workspace.yaml\n└── tsconfig.base.json\n```\n\n## 4. Service Boundaries (Backend Modules)\n\nThe NestJS backend is organized into feature modules, each responsible for a specific domain. Modules communicate via REST endpoints and internal event bus (NestJS EventEmitter or Redis pub/sub).\n\n### 4.1 Module List\n\n| Module | Responsibility | Key Entities |\n|--------|----------------|--------------|\n| **Auth** | User registration, login, JWT management, OAuth, biometric | User, Session, RefreshToken |\n| **User** | Profile management, preferences, GDPR | User, UserProfile |\n| **Business** | Business CRUD, verification, operating hours, staff | Business, Staff, BusinessHours, Holiday |\n| **Category** | Service category hierarchy, assignment | Category, BusinessCategory |\n| **Service** | Service menu management, pricing, duration | Service, ServiceCategory |\n| **Search** | Full-text search, autocomplete, geospatial search | (uses Business, Service, Category) |\n| **Booking** | Appointment lifecycle, slot computation, waitlist | Appointment, BookingSlot, WaitlistEntry |\n| **Review** | Ratings, reviews, moderation | Review, ReviewVote |\n| **Payment** | Payment processing, deposits, refunds | Payment, Transaction, Invoice |\n| **Notification** | Push notifications, emails, SMS | Notification, NotificationTemplate |\n| **Analytics** | Search analytics, booking metrics, dashboards | (aggregated data) |\n\n### 4.2 Module Communication\n- **Synchronous**: REST endpoints exposed by each module. API Gateway (NestJS) routes requests.\n- **Asynchronous**: Internal events for cross-cutting concerns (e.g., booking created → send notification).\n- **Caching**: Redis for frequently accessed data (business details, available slots).\n\n## 5. Data Flow\n\n### 5.1 Booking Flow Example\n\n1. **Mobile App** (Expo) sends GET /businesses/:id with search params.\n2. **API Gateway** routes to Business module → queries PostgreSQL (with PostGIS for location).\n3. **Business module** returns data, cached in Redis for subsequent requests.\n4. **User selects service and time** → POST /bookings to Booking module.\n5. **Booking module** validates slot availability (optimistic lock using database row version).\n6. **If available**, creates Appointment record, emits `booking.created` event.\n7. **Notification module** listens, sends push notification to business owner and confirmation to customer.\n8. **Payment module** handles deposit if required.\n\n### 5.2 Authentication Flow\n\n1. User submits credentials → POST /auth/login.\n2. Auth module validates against database (bcrypt), generates JWT access token (15 min) and refresh token (7 days).\n3. Refresh token stored in Redis with device info.\n4. Mobile app stores tokens securely (expo-secure-store).\n5. TanStack React Query uses axios interceptor to attach JWT and handle refresh.\n\n## 6. Key Architectural Decisions\n\n### 6.1 Why NestJS?\n- Provides a modular structure out of the box.\n- Built-in support for guards, interceptors, pipes (clean separation).\n- Easy integration with Prisma and Redis.\n- TypeScript native.\n\n### 6.2 Why Prisma?\n- Type-safe database queries, reducing runtime errors.\n- Auto-generated migrations and client.\n- Excellent support for PostgreSQL and PostGIS (raw queries for geospatial).\n\n### 6.3 Why TanStack React Query?\n- Manages server state, caching, and background refetching.\n- Reduces boilerplate for loading/error states.\n- Works well with React Native and Expo.\n\n### 6.4 Why Expo Router?\n- File-based routing simplifies navigation structure.\n- Supports deep linking and universal links.\n- Works with React Native and web.\n\n### 6.5 Why Redis?\n- Caching for high-read endpoints (business listings, available slots).\n- Session store for refresh tokens.\n- Rate limiting (sliding window).\n- Pub/sub for real-time notifications.\n\n### 6.6 Why PostGIS?\n- Enables efficient geospatial queries (radius search, area search).\n- Built on PostgreSQL, so relational data stays consistent.\n\n## 7. Database Schema (Key Tables)\n\n```sql\n-- Users\nCREATE TABLE \"User\" (\n  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),\n  email VARCHAR(255) UNIQUE NOT NULL,\n  password_hash VARCHAR(255),\n  first_name VARCHAR(100),\n  last_name VARCHAR(100),\n  phone VARCHAR(20),\n  role VARCHAR(20) NOT NULL DEFAULT 'customer', -- customer, business_owner, admin\n  email_verified BOOLEAN DEFAULT false,\n  created_at TIMESTAMPTZ DEFAULT NOW(),\n  updated_at TIMESTAMPTZ DEFAULT NOW()\n);\n\n-- Businesses\nCREATE TABLE \"Business\" (\n  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),\n  owner_id UUID REFERENCES \"User\"(id),\n  name VARCHAR(255) NOT NULL,\n  description TEXT,\n  location GEOGRAPHY(Point, 4326), -- PostGIS\n  address TEXT,\n  phone VARCHAR(20),\n  email VARCHAR(255),\n  website VARCHAR(255),\n  verified BOOLEAN DEFAULT false,\n  rating_avg DECIMAL(2,1) DEFAULT 0,\n  review_count INTEGER DEFAULT 0,\n  created_at TIMESTAMPTZ DEFAULT NOW()\n);\n\n-- Services\nCREATE TABLE \"Service\" (\n  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),\n  business_id UUID REFERENCES \"Business\"(id),\n  name VARCHAR(255) NOT NULL,\n  description TEXT,\n  duration_minutes INTEGER NOT NULL,\n  price DECIMAL(10,2) NOT NULL,\n  category_id UUID REFERENCES \"Category\"(id),\n  created_at TIMESTAMPTZ DEFAULT NOW()\n);\n\n-- Appointments\nCREATE TABLE \"Appointment\" (\n  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),\n  customer_id UUID REFERENCES \"User\"(id),\n  business_id UUID REFERENCES \"Business\"(id),\n  staff_id UUID REFERENCES \"Staff\"(id),\n  start_time TIMESTAMPTZ NOT NULL,\n  end_time TIMESTAMPTZ NOT NULL,\n  status VARCHAR(20) DEFAULT 'confirmed', -- confirmed, cancelled, completed\n  total_price DECIMAL(10,2),\n  special_requests TEXT,\n  version INTEGER DEFAULT 1, -- optimistic locking\n  created_at TIMESTAMPTZ DEFAULT NOW()\n);\n\n-- Add PostGIS index\nCREATE INDEX idx_business_location ON \"Business\" USING GIST (location);\n```\n\n## 8. Caching Strategy\n\n- **Business details**: Cache in Redis with TTL 5 minutes. Invalidate on update.\n- **Available slots**: Computed on the fly, cached for 30 seconds. Use Redis sorted sets for time ranges.\n- **Search results**: Cache for 2 minutes. Invalidate on business/service changes.\n- **User sessions**: Refresh tokens stored in Redis with TTL 7 days.\n- **Rate limiting**: Sliding window counters in Redis.\n\n## 9. Security Considerations\n\n- **JWT**: Short-lived access tokens (15 min), refresh token rotation.\n- **Rate Limiting**: 5 login attempts per 15 min per IP. General API rate limit: 1000 req/min per user.\n- **Input Validation**: DTO validation using class-validator and class-transformer.\n- **SQL Injection**: Prevented by Prisma parameterized queries.\n- **CORS**: Restricted to known origins.\n- **HTTPS**: Enforced in production.\n- **GDPR**: User data export and deletion endpoints.\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions, services, and components.\n- **Integration Tests**: NestJS testing utilities with test database (PostgreSQL in Docker).\n- **E2E Tests**: Detox for mobile, Cypress for web.\n- **CI/CD**: GitHub Actions runs tests on every push. EAS Build for mobile builds.\n\n## 11. Deployment & DevOps\n\n- **Local Development**: Docker Compose with PostgreSQL, Redis, and NestJS in watch mode.\n- **CI/CD Pipeline**: GitHub Actions → lint, test, build → deploy to staging/production.\n- **Mobile Builds**: EAS Build for iOS/Android. Distribution via TestFlight/Google Play.\n- **Environment Variables**: Managed via .env files (local) and GitHub Secrets (CI).\n\n## 12. Monitoring & Observability\n\n- **Logging**: NestJS Logger with structured JSON logs.\n- **Metrics**: Prometheus (optional) for request rates, error rates, latency.\n- **Error Tracking**: Sentry integration for frontend and backend.\n- **Health Checks**: /health endpoint for Docker health checks.\n\n## 13. Conclusion\n\nThis architecture provides a solid foundation for building the Planity Clone with scalability, maintainability, and testability in mind. The monorepo structure ensures code reuse and consistency, while the modular backend allows independent development and scaling of features.",
