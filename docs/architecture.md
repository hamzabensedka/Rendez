  "content": "# Architecture Document\n\n## Overview\n\nThe Planity Clone is a mobile first application that connects users with local businesses for booking appointments. The architecture follows a clean modular approach using a monorepo managed by Nx, separating concerns into mobile app, backend API, shared libraries and infrastructure.\n\n## Tech Stack\n\n- Mobile: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- Backend: NestJS, TypeScript, Prisma ORM, PostgreSQL, PostGIS extension, Redis\n- DevOps: Docker Compose, Nx, pnpm, GitHub Actions, EAS Build, Supabase (for auth/storage optional), BullMQ (background jobs)\n- Testing: Jest\n\n## High-Level Architecture\n\n```\n+-------------------+        +-------------------+        +-------------------+\n|   Mobile App      | <--->  |   API Gateway     | <--->  |   Business Logic  |\n| (Expo RN)         |        | (NestJS)          |        | (NestJS Services) |\n+-------------------+        +-------------------+        +-------------------+\n        ^                           ^                           ^\n        |                           |                           |\n        |                           v                           v\n        |                 +-------------------+        +-------------------+\n        |                 |   Prisma ORM      |        |   Redis Cache     |\n        |                 +-------------------+        +-------------------+\n        |                           |                           |\n        |                           v                           v\n        |                 +-------------------+        +-------------------+\n        |                 | PostgreSQL/PostGIS|        |   BullMQ Workers  |\n        |                 +-------------------+        +-------------------+\n        +-----------------------------------------------------------+\n```\n\n- The mobile app communicates with the backend via REST over HTTPS.\n- The backend is a NestJS monolith split into feature modules (Auth, Business, Booking, Payment, Notification, Admin, ProviderPortal).\n- Prisma handles DB migrations and queries; PostGIS enables location based queries.\n- Redis stores session tokens, rate limiting and caches frequent queries (e.g., business listings).\n- BullMQ runs background jobs for notifications, slot computation, email/SMS.\n- Docker Compose defines local dev services (db, redis, worker).\n- Nx manages monorepo: apps/mobile, apps/api, libs/shared-types, libs/ui-kit, libs/utils.\n- pnpm workspace for package management.\n- GitHub Actions runs lint, test, build and pushes Docker images; EAS Build creates iOS/Android binaries.\n- Supabase can be used Dockerfile\ndocker-compose.yml\nnx.json\npackage.json\npnpm-workspace.yaml\n\n## Service Boundaries\n\n1. Auth Service (NestJS)\n   - Responsible for user registration, login, token issuance (JWT), password reset.\n   - Exposes /auth/register, /auth/login, /auth/refresh.\n   - Uses Prisma User model; stores hashed passwords with bcrypt.\n   - Mobile app stores access token in SecureStore and refreshes silently.\n\n2. Business Service\n   - CRUD for businesses, categories, services.\n   - Geo search using PostGIS (ST_DWithin) for location based queries.\n   - Provides pagination, filtering, sorting.\n   - Caches list queries in Redis for 5 minutes.\n\n3. Booking Service\n   - Validates provider availability, computes available slots (considering business hours, existing bookings, buffer times).\n   - Creates booking records, triggers payment intent.\n   - Emits events to BullMQ for confirmation notifications and reminders.\n   - Uses transactional Prisma operations.\n\n4. Payment Service\n   - Wrapper around Stripe API.\n   - Creates PaymentIntent on booking confirmation.\n   - Handles webhook events (payment_intent.succeeded, failed) to update booking status.\n   - Idempotent webhook processing.\n\n5. Notification Service\n   - BullMQ queues: email-notifications, sms-notifications, push-notifications.\n   - Workers render templates (using handlebars) and send via external providers (SendGrid, Twilio, Expo Push).\n   - Triggered by Booking Service (booking created, reminder 24h/1h before, cancellation) and Review Service (new review).\n\n6. Review Service\n   - CRUD for reviews and ratings.\n   - Allows business owners to respond.\n   - Aggregates average rating stored denormalized on Business table (updated via trigger or job).\n\n7. Admin Service\n   - APIs for managing users, businesses, content moderation.\n   - Role based access control (RBAC) using JWT roles claim.\n\n8. Provider Portal Service\n   - Similar to Admin but scoped to a specific business owner.\n   - Allows managing own business profile, services, viewing bookings, accepting/rejecting, setting availability.\n\n9. Shared Types Library\n   - Defines DTOs, API response types, and UI model interfaces.\n   - Published as a local Nx library; both mobile and api consume it to guarantee contract consistency.\n\n10. UI Kit Library\n    - Reusable React Native components (Button, Input, Card, MapMarker) styled with Reanimated for smooth animations.\n    - Themed via context (light/dark).\n\n## Data Flow Examples\n\n### User Registration\n1. Mobile -> POST /auth/register with email, password.\n2. Auth Service validates, hashes password, creates User record.\n3. Returns access & refresh tokens.\n4. Mobile stores tokens.\n\n### Business Search (Map)\n1. Mobile reads user location, sends GET /businesses?lat=&lng=&radius=&category=.\n2. Business Service uses PostGIS to find businesses within radius.\n3. Optionally joins with cached list in Redis.\n4. Returns array with distance.\n5. Mobile displays on map using Reanimated Map component.\n\n### Booking Flow\n1. Mobile fetches provider availability via GET /providers/:id/availability?date=.\n2. Booking Service computes slots: loads business hours, existing bookings for provider/date, subtracts, returns free slots.\n3. User selects slot, submits POST /bookings with serviceId, providerId, startTime.\n4. Booking Service validates slot still free (optimistic lock), creates booking pending payment, creates Stripe PaymentIntent.\n5. Returns clientSecret to mobile.\n6. Mobile confirms payment with Stripe SDK, then POST /bookings/:id/confirm with paymentId.\n7. Booking Service verifies payment via webhook or direct check, marks booking confirmed, enqueues notification jobs.\n8. Mobile shows confirmation screen.\n\n### Notification (Reminder)\n- BullMQ worker runs scheduled job (cron) that queries bookings where startTime - now < 1h and reminder not sent.\n- Sends push notification via Expo Push API, email via SendGrid, SMS via Twilio.\n- Updates reminder flag.\n\n## DevOps & CI/CD\n\n- Local Development: docker compose up -d starts PostgreSQL (with PostGIS extension), Redis, and optionally a mock worker.\n- Nx Commands: nx run mobile:build, nx run api:test, nx run api:lint.\n- GitHub Actions:\n  - On push to main: install dependencies, run lint, test (Jest), build Docker images for api and worker, push to registry.\n  - On tag: trigger EAS Build for iOS/Android, distribute to internal testers.\n- Environment Variables: Managed via .env files; secrets stored in GitHub Secrets.\n- Supabase: Optional for file uploads (business photos) and auth fallback; bucket policies restrict to authenticated users.\n\n## Scalability & Reliability\n\n- Stateless API: NestJS instances can be scaled behind a load balancer; session state stored in Redis or JWT.\n- Database Read Replicas: For heavy search traffic, route read queries to replicas.\n- Caching: Redis caches business lists, slot computations (TTL 2-5 min).\n- Background Jobs: BullMQ with multiple workers; retries with exponential backoff.\n- Monitoring: Integrate with Prometheus/Grafana via NestJS Prometheus module; log aggregation via Loki or similar.\n- Testing: Unit tests with Jest for services; e2e tests for mobile using Detox (optional); contract tests using Pact (shared-types).\n\n## Security Considerations\n\n- HTTPS everywhere; enforce HSTS.\n- JWT access token short-lived (15 min), refresh token rotated and stored HTTP-only cookie (if using cookies) or SecureStore.\n- Rate limiting via NestJS Throttler backed by Redis.\n- Input validation using class-validator and DTOs.\n- Prisma prevents SQL injection.\n- Stripe PCI compliance: we never handle raw card data.\n- File uploads validated for mime type and size; stored in Supabase bucket with signed URLs.\n\n## Future Extensions\n\n- GraphQL API for flexible data fetching.\n- WebSocket (NestJS Gateway) for real-time updates (e.g., provider availability changes).\n- Multi-language support via i18next.\n- Push notification preferences.\n- AI-based recommendation for businesses.\n\n---\n",
