Service Boundaries & Domain Design\n\nEach **module** is an independent domain with its own controller, service, repository (via Prisma), and DTOs. Communication between modules is allowed only through public interfaces (services) and event-driven patterns (Redis Pub/Sub).\n\n### 4.1 Module Definitions\n\n| Module         | Responsibility                                                                 | Key Dependencies |\n|----------------|---------------------------------------------------------------------------------|------------------|\n| **Auth**       | Supabase JWT verification, role extraction, refresh token rotation, RBAC guard. | Supabase, Redis  |\n| **Users**      | Customer/Provider/Admin profile CRUD, device tokens, notification prefs.        | Supabase Auth UID|\n| **Businesses** | Business profile, location (PostGIS), working hours, categories, search.        | PostGIS, Redis   |\n| **Services**   | Service catalog per business, staff assignments, duration, pricing.             | Businesses       |\n| **Bookings**   | Slot engine, appointment creation, reschedule, cancel, availability cache.      | Services, Users, Redis, PostGIS |\n| **Reviews**    | Customer reviews, ratings distribution, provider responses, moderation.         | Bookings (completed appointments) |\n| **Payments**   | Stripe customer management, payment intents, saved cards, refunds.              | Bookings (pre-auth) |\n| **Notifications** | Push (Expo, FCM), email (Resend/SendGrid), SMS (Twilio), real-time events.  | Redis Pub/Sub    |\n| **Admin**      | Dashboard metrics, content moderation, configuration management.                | All modules      |\n\n### 4.2 Cross-Module Communication\n- **Synchronous**: Direct service injection only for read models (e.g., `BusinessService.getById`) to avoid tight coupling.\n- **Asynchronous**: Redis Pub/Sub channels: `appointment.created`, `appointment.cancelled`, `review.created`. Notification module subscribes to these channels and dispatches the appropriate messages without blocking the originating request.\n\n---\n\n## 5. Data Layer\n\n### 5.1 PostgreSQL + PostGIS\nPrisma schema includes PostGIS extension for geo-indexes. Key tables:\n- `User` (id, supabase_uid, role, profile fields)\n- `Business` (location geometry(Point, 4326), name, address, hours JSON)\n- `Staff` (businessId, workingHours)\n- `Service` (businessId, categoryId, duration, price)\n- `Appointment` (customerId, serviceId, staffId, start, end, status)\n- `Review` (rating, comment, photos)\n- `PaymentMethod` (stripe_payment_method_id, user_id)\n\nGeospatial query example (using raw SQL via Prisma):\n```sql\nSELECT *, ST_Distance(location, ST_MakePoint($1, $2)::geography) AS distance\nFROM \"Business\"\nWHERE ST_DWithin(location, ST_MakePoint($1, $2)::geography, $3)\nORDER BY distance;\n```\n\n### 5.2 Redis\n- **Session & refresh tokens** (TTL: 30 days).\n- **Slot availability cache**: Precomputed available slots per business/staff per date, updated on booking write-through.\n- **Rate limiting**: `rate-limiter-flexible` using Redis.\n- **Pub/Sub channels** for async events.\n\n### 5.3 Supabase\n- **Auth**: Managed user identities with Row Level Security disabled (we manage RBAC server-side).\n- **Storage**: Buckets for `business-photos`, `user-avatars`, `review-photos` with signed URLs.\n\n---\n\n## 6. Key Flows\n\n### 6.1 Authentication\n1. Mobile initiates Supabase sign-in (email/password, Google, Apple).\n2. Supabase returns access (short-lived) + refresh tokens.\n3. Mobile stores refresh token securely (via expo-secure-store).\n4. Every API call passes `Authorization: Bearer <access_token>`.\n5. Backend `AuthGuard` verifies token using Supabase JWKS, extracts `sub` and role.\n6. Refresh handled transparently by TanStack Query interceptor on 401.\n\n### 6.2 Booking Flow\n1. Customer selects service, staff, date.\n2. Mobile requests `GET /api/bookings/slots?businessId=...&staffId=...&date=...`.\n3. Booking module computes available slots:\n   - Fetch business & staff working hours (Prisma).\n   - Fetch existing appointments for that date.\n   - Subtract service duration + buffer.\n   - Return list of `start` times.\n4. Slots cached in Redis with a short TTL (30 seconds) to avoid recomputation under load.\n5. On booking confirmation, `POST /api/bookings` creates appointment in a transaction:\n   - Check slot availability again (optimistic lock).\n   - Create Appointment.\n   - Clear slot cache for that date/staff.\n   - Publish `appointment.created` event.\n6. Payment module may pre-authorize via Stripe if prepayment required.\n7. Notification module listens to event and schedules push reminder.\n\n### 6.3 Proximity Search\n- `GET /api/businesses/search?lat=...&lng=...&radius=5000&category=hair`\n- Uses PostGIS `ST_DWithin` and `ST_Distance`.\n- Results paginated, sorted by distance (default) or rating.\n- Populate with next available slot via a Redis cache lookup.\n\n---\n\n## 7. Frontend Architecture (Mobile)\n\n### 7.1 State Management Strategy\n- **Server State**: All API data managed by TanStack React Query. Stale times: searches (30s), business detail (2 min), bookings (0 вҖ“ always refetch).\n- **Client State**: UI state (filter open/closed, map visible) in Zustand stores.\n- **Auth State**: Tracked via custom `useAuth` hook wrapping Supabase session listener; stored in React Query cache for global access.\n\n### 7.2 Responsive UI & Designs System\n- **Theme** defined in `shared/ui` with tokens: colors, spacing, typography, shadows.\n- **Reusable components**: `Button`, `Card`, `RatingStars`, `Avatar`, `Modal`, `Input` all implement accessibility (44px touch targets, labels).\n- **Animations**: Use Reanimated for:\n  - Map marker clustering.\n  - Bottom sheet sliding for booking summary.\n  - Fade transitions between booking steps.\n- **Expo Router** provides deep linking for sharing business profiles, appointment confirmations.\n\n### 7.3 Map View\n- `react-native-maps` with provider Google Maps (or Mapbox for OSS).\n- Marker clustering via `supercluster` library.\n- List/Map toggle component, reusing the same React Query data.\n\n---\n\n## 8. API Design & Versioning\n\n- RESTful APIs with OpenAPI spec auto-generated from NestJS (`@nestjs/swagger`).\n- Base path: `/api/v1`.\n- Versioning handled via URI prefix; future versions `/api/v2`.\n- `api-client` library in the monorepo is generated from the spec and used by the mobile app for type-safe requests.\n\nExample endpoints:\n```\nGET    /api/v1/businesses?lat=...&lng=...&radius=...&page=...\nGET    /api/v1/businesses/:id\nGET    /api/v1/businesses/:id/services\nGET    /api/v1/bookings/slots?serviceId=&staffId=&date=\nPOST   /api/v1/bookings\nPATCH  /api/v1/bookings/:id/reschedule\nDELETE /api/v1/bookings/:id\nGET    /api/v1/users/me/appointments\nPOST   /api/v1/reviews\n```\n\n---\n\n## 9. Security\n\n- **Transport**: TLS 1.3 enforced.\n- **Auth**: Supabase JWT; backend validates using public key.\n- **RBAC**: Custom NestJS guard checking roles from JWT claims.\n- **Data Validation**: `class-validator` DTOs, sanitization against XSS.\n- **Database**: Prisma parameterized queries; row-level ownership checks.\n- **Sensitive Data**: Refresh tokens stored in OS keychain (expo-secure-store).
