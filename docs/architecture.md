Overview\nPlanity Clone is a cross‑platform mobile app (Expo/React Native) that lets users discover, book, and manage services from local businesses. The backend is a NestJS monolith powered by Prisma, PostgreSQL (with PostGIS for geo‑queries), Redis, and BullMQ for background jobs. The repository uses Nx for monorepo management, pnpm as the package manager, Docker Compose for local development, GitHub Actions for CI/CD, and EAS Build for Expo releases. Supabase provides optional authentication and file storage, while the core auth logic lives in the NestJS API.\n\n## 2. Tech Stack Summary\n- **Mobile**: Expo SDK, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Web/Admin**: Same React Native web build (Expo Web) or a separate Next.js admin portal (out of scope)\n- **API**: NestJS (Node.js), TypeScript, Prisma ORM, PostgreSQL + PostGIS extension, Redis, BullMQ\n- **Infra**: Docker, Docker Compose, Nx monorepo, pnpm, GitHub Actions, EAS Build, Supabase (auth/storage optional)\n- **Testing**: Jest (unit/integration), React Native Testing Library\n\n## 3. High‑Level Architecture\n```\n+-------------------+        +---------------------+        +-------------------+\n|   Mobile App      | <----> |   API Gateway (Nest) | <----> |   PostgreSQL      |\n| (Expo/React)      |        |   - Auth Service    |        |   + PostGIS       |\n|   - UI (Reanimated) |      |   - Business Service|        |   - Redis Cache   |\n|   - State (React Query) |   |   - Booking Service |        |   - BullMQ Workers|\n|   - Offline (AsyncStorage) | |   - Payment Service |        +-------------------+\n+-------------------+        |   - Notification Svc|        |   Supabase (optional)\n                             |   - Admin Service   |        |   - Auth & Storage|\n                             +---------------------+        +-------------------+\n```\n\n### 3.1 Service Boundaries (Backend)\n| Service          | Responsibility                                                                 | Key Tables / Collections                     |\n|------------------|-----------------------------------------------------------------------------|----------------------------------------------|\n| Auth             | User registration, login, JWT `businesses`, `categories`, `services`, `business_hours` |\n| Booking          | Appointment creation, validation, slot computation, cancellation/reschedule | `bookings`, `time_slots`, `appointments`             |\n| Shared           | DTOs, validation pipes, exception filters, constants, utils used across services| N/A                                          |\n\n### 3.2 Data Model Highlights\n- **Users**: `id`, `email`, `passwordHash`, `role` (user/provider/admin), `createdAt`, `updatedAt`\n- **Businesses**: `id`, `ownerId`, `name`, `description`, `address`, `lat`, `lng` (PostGIS point), `categoryId`, `isActive`\n- **Services**: `id`, `businessId`, `name`, `durationMinutes`, `price`, `categoryId`\n- **BusinessHours**: `id`, `businessId`, `dayOfWeek`, `openTime`, `closeTime`, `isClosed`\n- **TimeSlots**: Generated on‑demand via a stored procedure or service that intersects business hours with existing bookings; stored as `id`, `serviceId`, `startAt`, `endAt`, `isBooked`\n- **Bookings**: `id`, `userId`, `serviceId`, `slotId`, `status` (pending/confirmed/cancelled/completed), `createdAt`\n- **Payments**: `id`, `bookingId`, `amount`, `currency`, `provider` (stripe), `status`, `transactionId`\n- **Notifications**: `id`, `userId`, `type`, `payload`, `sentAt`, `readAt`\n\n### 3.3 Caching & Real‑time\n- Redis caches: business list (geo‑query results), user sessions, frequently accessed service metadata.\n- TTL: 5‑15 minutes for search results; infinite for session tokens with refresh rotation.\n- Real‑time updates (e.g., new booking) use React Query’s invalidation; optional WebSocket via NestJS Gateway for provider portal.\n\n### 3.4 Background Jobs (BullMQ)\n- Queues: `email-notifications`, `push-notifications`, `payment-reconciliation`, `slot-regeneration` (nightly).\n- Workers are separate NestJS processes launched via `nx run worker:start`.\n- Failed jobs retry with exponential backoff; dead‑letter queue for manual inspection.\n\n### 3.5 Mobile Architecture\n- **Routing**: Expo Router (file‑based) provides stack, tabs, and deep linking.\n- **State**: TanStack React Query handles server state; React Native Reanimated 2 drives gestures and animations (e.g., map drag, swipe‑to‑cancel).\n- **Offline**: AsyncStorage caches user profile and recent searches; queries are refetched on reconnect.\n- **Design System**: Shared `ui` library (Nx) with styled components, theme, icons.\n- **Shared Types**: Nx library `@planity/shared-types` contains TypeScript interfaces (User, Business, Booking, etc.) generated from Prisma schema via `prisma-typescript-plugin`.\n- **Authentication Flow**: Supabase optional for magic‑link/email; fallback to custom JWT endpoint (`/auth/login`). Tokens stored securely in SecureStore.\n- **Payment**: Stripe SDK via `@stripe/stripe-react-native`; payment sheet opened from Booking confirmation screen.\n- **Map**: `react-native-maps` with custom markers; geo‑queries performed via backend (`/businesses/near?lat=&lng=&radius=`).\n\n## 4. Service Communication\n- **Internal**: NestJS modules communicate via direct injection (in‑process) or via Redis Pub/Sub for decoupled events (e.g., `booking.created` → notification worker).\n- **External**: Mobile apps call REST/GraphQL endpoints (REST chosen for simplicity). All endpoints are versioned (`/api/v1/`).\n- **Webhooks**: Stripe webhook hits `/api/v1/payment/webhook`; verified with signature.\n\n## 6. API Contracts (Example)\n```\nGET   /api/v1/businesses/near?lat=48.85&lng=2.35&radius=5000\n      → { businesses: [{ id, name, lat, lng, rating, distance }] }\n\nPOST  /api/v1/bookings\n      Body: { serviceId, slotId }\n      → { bookingId, status }\n\nPOST  /api/v1/payments/intents\n      Body: { bookingId, amount }\n      → { clientSecret }\n\nPOST  /api/v1/notifications/push\n      Body: { userId, title, body, data }\n      → { jobId }\n```\nAll DTOs are validated with `class-validator` and typed via `shared-types`.\n\n## 7. Dev & CI/CD Workflow\n1. **Local dev**: `docker compose up` starts API, PostgreSQL, Redis, and workers.\n2. **Feature branch**: GitHub Actions runs `nx affected:test --base=main --head=${{ github.sha }}` and `nx lint`.\n3. **PR merge**: Triggers `cd.yml` which:\n   - Builds Docker images (`api`, `worker`) and pushes to registry.\n   - Runs `nx run mobile:build` and `eas build --platform all --auto-submit`.\n   - Deploys API to a managed Kubernetes/ECS cluster (or simple VM) via Helm/ECS task definition.\n   - Runs migration (`prisma migrate deploy`).\n4. **Release**: EAS provides OTA updates for JS bundle; native updates via store releases.\n\n## 8. Observability\n- **Logging**: Winston with JSON format; correlated via request‑id.\n- **Metrics**: Prometheus endpoint (`/metrics`) exposed by NestJS (via `@nestjs/microservices`).\n- **Tracing**: OpenTelemetry instrumentation for HTTP, Redis, Prisma queries.\n- **Mobile**: Sentry for crash reporting; Firebase Performance for JS bundle timing.\n\n## 9. Security Considerations\n- JWTs signed with RS256; private key stored in Vault/secret manager.\n- Passwords hashed with bcrypt (cost 12).\n- Rate limiting via `@nestjs/throttler` (IP + user).\n- CORS restricted to approved domains (expo dev, web).\n- Input sanitization and SQL injection mitigated by Prisma.\n- Stripe PCI compliance: no card data touches our servers.\n- Supabase Row Level Security (if used) enforces per‑user data access.\n\n## 10. Scalability & Future Enhancements\n- Horizontal scaling of API workers behind a load balancer.\n- Read replicas for PostgreSQL for geo‑search heavy traffic.\n- Redis Cluster for higher throughput.\n- Replace BullMQ with Bull Arena UI for monitoring.\n- Introduce GraphQL or tRPC for tighter type safety.\n- Add push notification service via Expo Push or Firebase Cloud Messaging.\n- Implement offline-first booking queue using `expo-sqlite`.\n\n---\n*This document captures the complete system architecture, folder layout, and service boundaries for the Planity Clone using the specified technologies.*\n",
