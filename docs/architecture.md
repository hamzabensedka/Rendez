  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nPlanity Clone is a monorepo‑based full‑stack application built with Expo (React Native) for the mobile client and NestJS for the API server. The architecture follows Clean Architecture principles: separation of concerns, dependency inversion, and testability. Core concerns are split into distinct domains (Auth, Business, Booking, Payments, Reviews, Notifications, Admin, Provider Portal) each owning its use‑cases, entities, and infrastructure adapters.\n\n## 2. Technology Stack\n- **Mobile**: Expo SDK, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **API**: NestJS (Node.js), TypeScript, Prisma ORM, PostgreSQL + PostGIS, Redis, BullMQ\n- **DevOps**: Nx monorepo, pnpm, Docker Compose, GitHub Actions, EAS Build (Expo), Supabase (auth & storage fallback), Jest\n\n## 3. High‑Level Architecture\n```\n+-------------------+        +-------------------+        +-------------------+\n|   Mobile App      | <--->  |   API Gateway     | <--->  |   Data Stores     |\n| (Expo/React)      |        | (NestJS)          |        | (Postgres, Redis) |\n+-------------------+        +-------------------+        +-------------------+\n        ^                           ^                           ^\n        |                           |                           |\n+-------------------+        +-------------------+        +-------------------+\n|   Shared Libs     |        |   Background Jobs |        |   External Svcs   |\n| (types, ui, utils)|        | (BullMQ + Workers)|        | (Payment, Email)  |\n+-------------------+        +-------------------+        +-------------------+\n```\n- **Mobile App** consumes REST/GraphQL‑like endpoints via a thin API client generated from NestJS DTOs (using `@nestjs/swagger` + `openapi-typescript`). State is managed with React Query; UI animations use Reanimated 2.\n- **API Gateway** is a NestJS monolith split into feature modules (see Service Boundaries). Each module encapsulates its own controllers, services, repositories, and Prisma models.\n- **Data Stores**: Primary relational store is PostgreSQL with PostGIS extension for geo‑queries. Redis is used for caching (search results, session tokens) and as BullMQ broker.\n- **Background Jobs**: BullMQ workers run as separate NestJS microservices (or within the same process in dev) handling email/SMS notifications, payment webhook processing, and periodic slot recomputation.\n- **External Services**: Payment gateway (Stripe/PayPal), email provider (SendGrid/Supabase Functions), map tiles (Mapbox/Google Maps).\n\n## 4. Service Boundaries (Domain Modules)\n| Module            | Responsibilities                                                                 | Key Entities (Prisma)                |
|-------------------|---------------------------------------------------------------------------------|--------------------------------------|
| **Auth**          | User registration, login, password reset, social OAuth, JWT issuance/validation | `User`, `Session`                    |
| **Business**      | CRUD of business profiles, service categories, media, geo‑location indexing   | `Business`, `Category`, `BusinessImage` |
| **Booking**       | Availability slot generation, appointment creation, rescheduling, cancellation | `Appointment`, `TimeSlot`, `BusinessHours` |
| **Payments**      | Payment intent creation, webhook handling, invoice storage                     | `Payment`, `Invoice`                 |
| **Reviews**       | Create, read, moderate reviews & ratings                                       | `Review`, `Rating`                   |
| **Notifications** | Template management, delivery via BullMQ workers (email/push/SMS)               | `NotificationTemplate`, `SentNotif`  |
| **Admin**         | User & business moderation, analytics, role‑based access control               | `Role`, `Permission`, `AuditLog`     |
| **ProviderPortal**| Business owner dashboard: manage listing, availability, view bookings          | (reuses Business & Booking entities) |
| **Shared**        | UI components, design system, TypeScript types, utility functions, constants   | –                                    |
| **Core**          | Cross‑cutting concerns: logging, error handling, validation pipes, config      | –                                    |

Each module follows the Clean Architecture layers:\n- **Presentation**: NestJS Controllers (or Expo screens).\n- **Application**: Use‑case services (DTOs, command/query handlers).\n- **Domain**: Entities and business rules (TypeScript classes/interfaces).\n- **Infrastructure**: Prisma repositories, external service adapters, Redis cache.\n\n## 5. Caching & Background Jobs\n- **Redis**:\n  - Token blacklist/store for JWT refresh.\n  - Search result cache (key: `search:${hash(params)}` TTL 5 min).\n  - Rate‑limiting counters (API gateway).\n- **BullMQ**:\n  - Queues: `notifications`, `payments`, `slot-recompute`.\n  - Workers defined in `apps/api/src/jobs/`.\n  - Retry policy: exponential backoff, max 3 attempts.\n\n## 8. DevOps & CI/CD\n- **Nx** enables affected‑based testing and caching.\n- **pnpm** workspace for hoisted dependencies.\n- **Docker Compose** (local dev):\n  ```yaml\n  services:\n    postgres:\n      image: postgres:15-alpine\n      environment: POSTGRES_PASSWORD: postgres\n      volumes: pg_data:/var/lib/postgresql/data\n      ports: [\"5432:5432\"]\n    redis:\n      image: redis:7-alpine\n      ports: [\"6379:6379\"]\n    api:\n      build: ./apps/api\n      env_file: .env\n      depends_on: [postgres, redis]\n      ports: [\"3000:3000\"]\n    worker:\n      build: ./apps/api\n      command: npx ts-node src/jobs/notification.worker.ts\n      env_file: .env\n      depends_on: [postgres, redis]\n  volumes:\n    pg_data:\n  ```\n- **GitHub Actions**:\n  - `ci.yml`: lint (`nx lint`), type‑check (`nx type-check`), unit tests (`nx test`), build (`nx build`).\n  - `cd-mobile.yml`: on tag push, run `eas build --platform all --auto-submit`.\n  - `cd-api.yml`: on main push, build Docker image, push to GHCR, trigger rolling update on staging/prod.\n- **EAS Build** handles OTA updates and store submissions.\n- **Supabase** (optional): used for social login providers and file storage (avatars, business images) via its S3‑compatible API; fallback to local uploads in dev.\n\n## 9. Cross‑Cutting Concerns\n- **Authentication**: JWT (access token 15 min, refresh token 7 days) stored in SecureStore (expo); refresh route rotates tokens.\n- **Validation**: Zod schemas for DTOs; NestJS ValidationPipe with `whitelist:true`.\n- **Error Handling**: Global exception filter maps known errors to HTTP responses; logs via Winston to stdout (Docker captures).\n- **Logging**: Structured JSON logs (`timestamp`, `level`, `msg`, `traceId`).\n- **Configuration**: `@nestjs/config` with environment variables; Nx provides workspace‑level `env.json` for shared constants.\n- **Testing**: Jest unit tests for services/controllers; React Native Testing Library + Jest for mobile components; E2E with Detox (mobile) and Supertest (API).\n\n## 10. Summary of Architecture Decisions\n- **Monorepo with Nx** enables code sharing (types, UI, utils) and atomic commits across mobile and API.\n- **Clean Architecture layers** keep business logic independent of frameworks, simplifying testing and future migration (e.g., to GraphQL or microservices).\n- **Expo Router** provides file‑based navigation, reducing boilerplate and enabling deep linking.\n- **TanStack React Query** handles server state, caching, and background refetching, aligning with the API’s caching strategy.\n- **Prisma + PostGIS** gives type‑safe data access and powerful geo‑queries for map‑based search.\n- **Redis + BullMQ** decouple long‑running tasks (notifications, payments) from request‑response cycle.\n- **Docker Compose** mirrors production services locally, ensuring parity.\n- **GitHub Actions + EAS Build** automate testing, builds, and store releases.\n\nThis architecture satisfies all functional requirements while maintaining scalability, maintainability, and clear separation of concerns.\n",
