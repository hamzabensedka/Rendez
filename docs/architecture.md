  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nThe Planity Clone is a mobile-first appointment booking platform that connects users with local service providers. The system consists of a React Native/Expo frontend, a NestJS/Prisma backend powered by PostgreSQL with PostGIS for geospatial queries, Redis for caching and pub/sub, BullMQ for background jobs, and an Nx monorepo managed with pnpm. CI/CD is handled by GitHub Actions, with builds via EAS and deployment via Docker Compose (local) or cloud services.\n\n## 2. Tech Stack\n- **Frontend**: Expo (SDK 49), React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated, Nx, pnpm\n- **Backend**: NestJS, TypeScript, Prisma ORM, PostgreSQL + PostGIS extension, Redis, BullMQ\n- **DevOps**: Docker Compose, GitHub Actions, EAS Build, Supabase (optional for auth/storage), Nx\n- **Testing**: Jest, React Native Testing Library\n- **Design System**: Shared UI library built with React Native Reanimated and custom tokens\n\n## 3. High-Level Architecture\n```\n[Mobile App] <--HTTPS/WS--> [API Gateway (NestJS)] <---> [PostgreSQL/PostGIS]\n                              |                     |\n                              v                     v\n                           [Redis Cache]      [BullMQ Workers]\n```\nThe frontend communicates exclusively with the NestJS API via REST (JSON) and occasional WebSocket connections for real‑time updates (e.g., slot changes, notifications). The API layer handles authentication, business logic, validation, and delegates heavy computation (slot generation, geospatial search) to services backed by PostGIS. Redis stores short‑lived caches (search results, session tokens) and publishes events that BullMQ workers consume for jobs such as email/SMS reminders, payment reconciliation, and data synchronization.\n\n## 4. Service Boundaries\n| Boundary | Responsibility | Technologies |\n|----------|----------------|--------------|\n| **Auth Service** | User registration, login, JWT issuance, password reset, social login (optional) | NestJS Guard, Prisma User model, Redis for refresh token store |\n| **Business Service** | CRUD for businesses, categories, photos, hours, location (PostGIS) | NestJS Controller/Service, Prisma, PostGIS |\n| **Service & Provider Service** | Manage services, providers, pricing, availability rules | NestJS, Prisma |\n| **Booking Service** | Slot computation, appointment creation, cancellation, rescheduling, conflict detection | NestJS, BullMQ for async confirmation, PostGIS for location‑based slot lookup |\n| **Payment Service** | Integration with payment gateway (Stripe/PayPal), receipt generation, webhook handling | NestJS, gateway SDK |\n| **Review & Rating Service** | CRUD for reviews, business responses, aggregation of ratings | NestJS, Prisma |\n| **Notification Service** | Send push, email, SMS via Supabase/FCM or third‑party, preference management | NestJS, BullMQ, Redis pub/sub |\n| **Admin & Portal Services** | Admin dashboard, business owner portal, role‑based access control | NestJS, Prisma, RBAC guards |\n| **Shared Library** | TypeScript interfaces, DTOs, validation schemas, utility functions, design tokens | Nx lib, used by both frontend and backend |\n\n## 5. API Contracts\nAll endpoints are versioned under `/api/v1`. Example:\n\n```\nGET   /api/v1/businesses?lat=...&lon=...&radius=1000&category=beauty\nPOST  /api/v1/auth/register\nPOST  /api/v1/auth/login\nGET   /api/v1/users/me\nPOST  /api/v1/appointments\nPUT   /api/v1/appointments/:id/cancel\nGET   /api/v1/appointments/upcoming\nPOST  /api/v1/payments/webhook\n...\n```\n- Request/response bodies use DTOs from `libs/api-shared`.\n- Validation via NestJS `class-validator` and `class-transformer`.\n- Errors follow RFC‑7807 Problem JSON.\n- Real‑time updates: WebSocket gateway (`/ws/slots`) pushes slot changes; clients subscribe via React Query’s `useSubscription` wrapper.\n\n## 8. State Management & Client‑Side\n- **React Query** handles server state (fetching, caching, optimistic updates). Query keys are namespaced: `['businesses', lat, lon, radius]`, `['appointments', userId]`.\n- **Auth state** managed via a custom `useAuth` hook that reads JWT from SecureStore and exposes login/logout.\n- **UI state** (modal visibility, form values) kept in React `useState` or lightweight Zustand store if needed.\n- **Reanimated** used for gesture‑based interactions (e.g., swipe to cancel appointment, animated map markers).\n\n## 9. Real‑time & Notifications\n- **WebSocket** (NestJS `@nestjs/websockets`) pushes:\n  - Slot availability changes when an appointment is booked/cancelled.\n  - New messages in chat (future extension).\n- **Push notifications** via Expo’s push notification service; backend enqueues a job in BullMQ (`send-push`) which calls Expo’s API.\n- **Email/SMS** handled similarly via BullMQ workers using SendGrid/Twilio.\n\n## 10. Caching & Background Jobs\n- **Redis**:\n  - LRU cache for geospatial search results (TTL 5 min).\n  - Token blacklist for logout.\n  - Pub/sub channel for broadcasting slot updates to API instances.\n- **BullMQ** queues:\n  - `email`: send confirmation, reminders.\n  - `sms`: OTP, appointment alerts.\n  - `payment-reconcile`: verify webhook events.\n  - `data-sync`: nightly aggregation of business stats.\n- Workers are separate Node processes (can be same container with multiple workers or dedicated containers).\n\n## 11. CI/CD & DevOps\n- **GitHub Actions** workflow:\n  - `ci.yml`: lint (`nx lint`), type‑check (`nx type-check`), unit tests (`nx test mobile api`), build Docker images.\n  - `cd.yml`: on push to `main`, push images to registry, trigger EAS build for mobile, run database migrations, deploy via Docker Compose (staging) or Kubernetes (prod).\n- **EAS Build**: configured for iOS and Android; uses `eas.json` with profiles for development, preview, production.\n- **Docker Compose** (local/dev):\n```yaml\nservices:\n  postgres:\n    image: postgres:15\n    environment: POSTGRES_USER=planity, POSTGRES_PASSWORD=secret, POSTGRES_DB=planity\n    volumes: pg_data:/var/lib/postgresql/data\n  redis:\n    image: redis:7\n  api:\n    build: ./apps/api\n    env_file: .env.api\n    depends_on: [postgres, redis]\n  worker:\n    build: ./apps/api\n    command: [\"npx\", \"bullmq-worker\"]\n    env_file: .env.api\n      depends_on: [postgres, redis]\n```\n- **Supabase** (optional) used for:\n  - Auth (if preferring magic link) – can replace custom JWT.\n  - Storage for business photos and user avatars.\n\n## 12. Testing Strategy\n- **Unit tests**: Jest for both frontend and backend; aim for >80% coverage on critical paths (auth, booking, payment).\n- **Integration tests**: API contract tests using Supertest; end‑to‑end mobile flows with Detox or Expo’s testing library.\n- **E2E**: GitHub Actions runs Detox on Android emulator and iOS simulator.\n- **Performance**: Load test booking flow with k6; monitor Redis hit‑rate and DB query times.\n\n## 13. Security Considerations\n- Authentication: JWT signed with RS256; short‑lived access token (15 min) + refresh token stored in Redis with rotation.\n- Data validation: DTOs with `class-validator`; sanitize inputs to prevent SQL injection (Prisma parameterized).\n- Rate limiting: NestJS `ThrottlerGuard` per IP and per user.\n- CORS: restrict to allowed origins (expo dev, production domains).\n- Secrets: managed via GitHub Secrets and injected into Docker/K8s env; never committed.\n- Data at rest: PostgreSQL SSL; S3‑compatible storage (Supabase) with signed URLs for uploads.\n- OWASP Top 10 mitigations applied throughout.\n\n## 14.
