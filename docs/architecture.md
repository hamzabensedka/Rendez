Overview\n\nThis document describes the complete system architecture for the Planity Clone, a mobile-first platform connecting customers with beauty, wellness, and service professionals. The architecture is designed for scalability, maintainability, and clear separation of concerns.\n\n## 2. Architecture Principles\n\n- **Clean Architecture**: Separation into layers (presentation, application, domain, infrastructure).\n- **Domain-Driven Design**: Each bounded context has its own domain model and service.\n- **Monorepo with Nx**: All code in a single repository for consistency and shared tooling.\n- **TypeScript Everywhere**: End-to-end type safety from mobile to database.\n- **API-first**: NestJS backend with RESTful APIs, GraphQL optional for complex queries.\n- **Offline-first**: Mobile app caches data locally using TanStack React Query and AsyncStorage.\n- **Event-driven**: Redis pub/sub for real-time updates and async processing.\n\n## 3. Technology Stack\n\n| Layer | Technology | Purpose |\n|---|---|---|\n| Mobile | Expo (React Native), TypeScript, Expo Router, TanStack React Query, React Native Reanimated | Cross-platform mobile app |\n| Backend | NestJS, TypeScript, Prisma, PostgreSQL, PostGIS, Redis | API server, business logic, data persistence |\n| Infrastructure | Docker Compose, GitHub Actions, Supabase (Auth + Storage), EAS Build | CI/CD, hosting, build |\n| Testing | Jest (unit), React Native Testing Library (component), Supertest (e2e) | Quality assurance |\n\n## 4. System Context Diagram\n\n```\n[Mobile App] <--> [NestJS API] <--> [PostgreSQL + PostGIS]\n                    |\n                    v\n               [Redis Cache]\n                    |\n                    v\n               [Supabase Auth]\n                    |\n                    v\n               [Supabase Storage]\n```\n\n## 5. Service Boundaries (Bounded Contexts)\n\n### 6.1 Auth Context\n- **Responsibilities**: User registration, login, token management, OAuth, password reset.\n- **Entities**: User, RefreshToken, VerificationToken.\n- **Events**: user.registered, user.logged_in, user.password_reset.\n- **Dependencies**: Supabase Auth (for OAuth), Redis (token blacklist).\n\n### 6.2 User Context\n- **Responsibilities**: Profile management, preferences, favorites.\n- **Entities**: UserProfile, UserPreference, FavoriteBusiness.\n- **Events**: user.profile_updated.\n\n### 6.3 Business Context\n- **Responsibilities**: Business CRUD, service catalog, team management, availability.\n- **Entities**: Business, Service, ServiceVariant, ServiceAddon, Professional, BusinessHours, AvailabilitySlot.\n- **Events**: business.created, business.updated, service.added.\n- **Dependencies**: PostGIS for location queries.\n\n### 6.4 Booking Context\n- **Responsibilities**: Appointment creation, slot management, booking lifecycle.\n- **Entities**: Booking, BookingSlot, BookingHistory.\n- **Events**: booking.created, booking.confirmed, booking.cancelled.\n- **Dependencies**: Redis for slot caching (30-second TTL).\n\n### 6.5 Search Context\n- **Responsibilities**: Full-text search, geo-search, filtering, sorting.\n- **Entities**: SearchQuery, SearchResult (view).\n- **Dependencies**: PostgreSQL full-text search, PostGIS for distance queries, Redis for popular searches cache.\n\n### 6.6 Review Context\n- **Responsibilities**: Review creation, rating aggregation, moderation.\n- **Entities**: Review, ReviewReport.\n- **Events**: review.created, review.reported.\n\n### 6.7 Notification Context\n- **Responsibilities**: Push notifications, email notifications, in-app alerts.\n- **Entities**: Notification, NotificationTemplate.\n- **Dependencies**: Firebase Cloud Messaging (push), SendGrid (email).\n\n### 6.8 Admin Context\n- **Responsibilities**: Platform oversight, user management, analytics.\n- **Entities**: AdminUser, AuditLog.\n\n## 7. Data Flow Diagrams\n\n### 7.1 Booking Flow\n```\nUser selects service -> API fetches available slots (cached in Redis for 30s)\n  -> User picks slot -> API creates booking (transactional)\n  -> API sends confirmation notification (async via Redis pub/sub)\n  -> API updates slot cache (invalidate)\n```\n\n### 7.2 Search Flow\n```\nUser enters query -> API performs full-text search on PostgreSQL\n  -> If geo-filter: PostGIS distance query\n  -> Results cached in Redis for 60 seconds (keyed by query+filters)\n  -> Return paginated results\n```\n\n## 8. API Design\n\n### 8.1 RESTful Endpoints\n\n| Method | Endpoint | Description |\n|---|---|---|\n| POST | /api/auth/register | Register new user |\n| POST | /api/auth/login | Login |\n| POST | /api/auth/refresh | Refresh token |\n| POST | /api/auth/logout | Logout |\n| GET | /api/users/me | Get current user profile |\n| PATCH | /api/users/me | Update profile |\n| GET | /api/businesses | List businesses (with filters) |\n| GET | /api/businesses/:id | Business detail |\n| GET | /api/businesses/:id/services | Services for a business |\n| GET | /api/businesses/:id/availability | Available slots (query params: date, professionalId) |\n| POST | /api/bookings | Create booking |\n| GET | /api/bookings | List user bookings |\n| PATCH | /api/bookings/:id/cancel | Cancel booking |\n| GET | /api/search | Full-text search |\n| GET | /api/categories | List categories |\n| GET | /api/reviews | List reviews for business |\n| POST | /api/reviews | Create review |\n\n### 8.2 Response Format\n\nAll responses follow a consistent envelope:\n```json\n{\n  \"status\": \"success\",\n  \"data\": { ... },\n  \"meta\": {\n    \"page\": 1,\n    \"limit\": 20,\n    \"total\": 100\n  },\n  \"error\": null\n}\n```\n\n## 9. Database Schema (Prisma)\n\nKey entities:\n\n- **User**: id, email, passwordHash, phone, role (CUSTOMER, PROVIDER, ADMIN), verified, createdAt\n- **Business**: id, name, description, address, location (PostGIS Point), phone, website, rating, reviewCount, ownerId\n- **Service**: id, businessId, name, description, duration, basePrice, categoryId\n- **ServiceVariant**: id, serviceId, name, price, duration\n- **ServiceAddon**: id, serviceId, name, price, duration\n- **Professional**: id, businessId, name, bio, photo\n- **BusinessHours**: id, businessId, dayOfWeek, openTime, closeTime\n- **AvailabilitySlot**: id, businessId, professionalId, startTime, endTime, isBooked\n- **Booking**: id, userId, businessId, professionalId, serviceId, variantId, addonIds, startTime, endTime, status (PENDING, CONFIRMED, CANCELLED), notes, createdAt\n- **Review**: id, userId, businessId, bookingId, rating, comment, createdAt\n- **Category**: id, name, parentId, icon\n- **Notification**: id, userId, type, title, body, read, createdAt\n\n## 10. Caching Strategy\n\n- **Availability Slots**: Redis cache with 30-second TTL. Key: `availability:{businessId}:{date}`. Invalidated on booking creation.\n- **Search Results**: Redis cache with 60-second TTL. Key: `search:{query}:{filters}:{page}`.\n- **Business Details**: Redis cache with 5-minute TTL. Key: `business:{id}`.\n- **User Sessions**: Refresh tokens stored in Redis with 7-day TTL.\n- **Rate Limiting**: Redis-based sliding window counter.\n\n## 11. Security Considerations\n\n- **Authentication**: JWT access tokens (15 min) + refresh tokens (7 days). Refresh tokens stored in Redis with user ID mapping.\n- **Password Hashing**: bcrypt with salt rounds = 12.\n- **Rate Limiting**: 5 login attempts per minute per IP; 100 API requests per minute per user.\n- **Input Validation**: Zod schemas on both client and server.\n- **SQL Injection Prevention**: Prisma ORM parameterized queries.\n- **CORS**: Restricted to mobile app origin.\n- **HTTPS**: Enforced in production.\n\n## 12. Testing Strategy\n\n- **Unit Tests**: Jest for services, controllers, utilities. Coverage > 80%.\n- **Component Tests**: React Native Testing Library for UI components.\n- **Integration Tests**: Supertest for API endpoints with test database.\n- **E2E Tests**: Detox for mobile app flows.\n- **CI/CD**: GitHub Actions runs tests on every PR; deploys to staging on merge to main.\n\n## 13. Deployment\n\n- **Mobile**: EAS Build for iOS and Android; OTA updates via EAS Update.\n- **Backend**: Docker Compose for local development; production deployment on cloud (AWS ECS or similar).\n- **Database**: Supabase PostgreSQL with PostGIS extension.\n- **Cache**: Redis (managed service or Docker).\n- **CI/CD**: GitHub Actions with separate workflows for mobile and API.\n\n## 14. Monitoring and Observability\n\n- **Logging**: NestJS Logger with structured JSON output.\n- **Metrics**: Prometheus metrics endpoint for API.\n- **Error Tracking**: Sentry for both mobile and backend.\n- **Performance**: React Native Performance Monitor for mobile; New Relic for backend.\n\n## 15. Conclusion\n\nThis architecture provides a scalable, maintainable foundation for the Planity Clone. The monorepo structure with Nx ensures code reuse and consistency, while the bounded contexts allow independent development and scaling. The use of modern technologies (Expo, NestJS, Prisma, Redis) ensures performance and developer productivity.",
