Architecture Principles\n\n- **Separation of Concerns**: Each layer (presentation, application, domain, infrastructure) has distinct responsibilities.\n- **Dependency Inversion**: High-level modules do not depend on low-level modules; both depend on abstractions.\n- **Domain-Driven Design**: Core business logic is encapsulated in domain entities and services.\n- **API-First**: The mobile client consumes a well-defined REST API; internal services communicate via events or gRPC.\n- **Stateless Backend**: Horizontal scaling is achieved by keeping NestJS stateless; session data lives in Redis.\n- **Offline Resilience**: The mobile app uses local caching and optimistic updates to handle intermittent connectivity.\n\n## 4. Service Boundaries\n\n### 5.1 Auth Service\n- **Responsibility**: User registration, login, token management, OAuth, password reset.\n- **Data**: Users table, refresh tokens in Redis.\n- **APIs**: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `POST /auth/reset-password`.\n\n### 5.2 User Service\n- **Responsibility**: Profile management, preferences, device tracking.\n- **Data**: Users table, profiles table.\n- **APIs**: `GET /users/me`, `PATCH /users/me`, `GET /users/me/sessions`.\n\n### 5.3 Business Service\n- **Responsibility**: Business CRUD, search (full-text + geospatial), categories, operating hours, staff management.\n- **Data**: Businesses, categories, staff, operating_hours tables. Data Flow (Booking Example)\n\n1. **Mobile App** sends `GET /businesses/:id/availability?date=2025-06-15&serviceId=123`.\n2. **NestJS API** receives request, validates with Zod, checks authentication via JWT.\n3. **Booking Service** queries Redis for cached availability; if miss, queries PostgreSQL via Prisma (generates slots based on business schedule and existing bookings).\n4. **Booking Service** returns available time slots; response is cached in Redis with TTL 5 minutes.\n5. **Mobile App** displays slots; user selects a slot.\n6. **Mobile App** sends `POST /bookings` with service, professional, time.\n7. **Booking Service** locks the slot in Redis (atomic operation) to prevent double booking.\n8. **Booking Service** creates appointment in PostgreSQL within a transaction.\n9. **Booking Service** publishes event `booking.created` to Redis pub/sub.\n10. **Notification Service** listens to event, sends push notification to business owner and confirmation to customer.\n11. **Mobile App** receives 201 response, updates local cache via React Query invalidation.\n\n## 7. Key Design Decisions\n\n### 7.1 Why Expo over bare React Native?\nExpo provides managed workflow, OTA updates, and simplifies build pipeline with EAS. For this project, we don't need native modules beyond what Expo supports (maps, camera, etc.). If needed, we can eject to bare workflow later.\n\n### 7.2 Why NestJS?\nNestJS offers modular architecture (similar to Angular), built-in dependency injection, and excellent support for TypeScript. It aligns with clean architecture principles and scales well.\n\n### 7.3 Why Prisma + PostGIS?\nPrisma provides type-safe database access and auto-generated migrations. PostGIS enables efficient geospatial queries (radius search, distance sorting) natively in PostgreSQL.\n\n### 7.4 Why Redis for Availability?\nBooking availability requires low-latency reads and atomic slot locking. Redis provides sub-millisecond response times and supports atomic operations (SETNX) to prevent race conditions.\n\n### 7.5 Why Nx + pnpm?\nNx manages the monorepo with dependency graph, caching, and distributed task execution. pnpm offers fast, disk-efficient package management with strict dependency isolation.\n\n## 8. Security Considerations\n\n- **JWT**: Access tokens expire in 15 minutes; refresh tokens in 7 days. Tokens are stored in HTTP-only secure cookies on mobile (via Supabase Auth).\n- **Rate Limiting**: Implemented at API Gateway level (NestJS middleware) using Redis. 100 requests/min per user, 5 failed login attempts per 15 min.\n- **Input Validation**: All API inputs validated with Zod schemas.\n- **SQL Injection**: Prevented by Prisma parameterized queries.\n- **CORS**: Strict origin policy for API.\n- **Data Encryption**: Sensitive data (PII, payment info) encrypted at rest using AES-256.\n\n## 9. Performance & Scalability\n\n- **Horizontal Scaling**: NestJS API is stateless; multiple instances behind a load balancer.\n- **Database Scaling**: PostgreSQL read replicas for read-heavy queries; write master handles writes.\n- **Caching**: Redis caches availability, business search results (with TTL), and session data.\n- **Image Optimization**: Supabase Storage with CDN; images are resized on upload (thumbnail, medium, full).\n- **Lazy Loading**: Mobile app uses React.lazy and Expo Router's code splitting for page-level chunks.\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for individual functions and services (NestJS modules, React hooks).\n- **Integration Tests**: Test NestJS endpoints with test database (Docker Compose).\n- **E2E Tests**: Detox for mobile app flows (booking, search, auth).\n- **CI Pipeline**: GitHub Actions runs lint, unit tests, integration tests on every PR. E2E runs on merge to main.\n\n## 11. Deployment\n\n- **Mobile**: EAS Build generates iOS and Android builds; distributed via TestFlight and Google Play Console.\n- **Backend**: Docker containers deployed to AWS ECS (or similar). GitHub Actions builds Docker image, pushes to ECR, and updates ECS service.\n- **Database**: Managed PostgreSQL (RDS) with automated backups and point-in-time recovery.\n- **Redis**: ElastiCache for Redis cluster.\n\n## 12. Monitoring & Observability\n\n- **Logging**: Winston (NestJS) + Logtail for centralized logging.\n- **Metrics**: Prometheus + Grafana for API latency, error rates, database performance.\n- **APM**: Sentry for error tracking and performance monitoring.\n- **Alerts**: PagerDuty for critical errors (payment failures, high error rate).\n\n## 13. Conclusion\n\nThis architecture provides a solid foundation for building a scalable, maintainable appointment booking platform.
