  "content": "# Planity Clone — System Architecture\n\n## 1. Introduction\nPlanity Clone is a two‑sided marketplace for beauty & wellness services. This document describes the end‑to‑end architecture – monorepo structure, service boundaries, data flow, and infrastructure – using a stack composed of Expo (React Native), NestJS, PostgreSQL with PostGIS, Redis, and Supabase for managed auth, all orchestrated with Nx and pnpm.\n\n## 2. High‑Level Architecture\n```\n┌────────────────────────────┐\n│   Mobile App (Expo/RN)     │\n│   Expo Router, TanStack    │\n│   Query, Reanimated        │\n└────────────┬───────────────┘\n             │ HTTPS/REST/WS\n┌────────────▼───────────────┐\n│   API Gateway (NestJS)     │\n│   - Auth middleware        │\n│   - Rate limiting          │\n│   - Validation             │\n└───┬───────────┬───────────┘\n    │           │\n┌───▼───┐  ┌───▼────────────────┐\n│ Redis │  │  PostgreSQL+PostGIS│\n│ Cache │  │  (primary DB)      │\n│ Queue │  └────────────────────┘\n└───────┘\n```\n- **Mobile client**: Expo app with file‑based routing (Expo Router), state management via TanStack React Query and animated UI with Reanimated 2/3.\n- **Backend**: NestJS monolith initially, internal modularisation by domain. Exposes REST API; may add WebSocket for real‑time features.\n- **Database**: PostgreSQL with PostGIS for geospatial queries. Prisma ORM used for schema management and type‑safe queries.\n- **Cache & Session Store**: Redis holds session blacklist, rate‑limit counters, cached availability slots, and background job queues.\n- **Authentication**: Supabase Auth (email/password, Google, Apple) – NestJS validates Supabase‑issued JWTs, maps Supabase user IDs to internal User table.\n- **File Storage**: Supabase Storage for user/business images.\n\n## 3. Frontend Architecture (Expo / React Native)\n\n### 4.1 Routing & Navigation\nExpo Router provides file‑based routing deep‑linked with universal links. Layouts:\n- `(auth)` – unauthenticated stack (login, register, password reset).\n- `(tabs)` – main bottom tabs after login (Explore, Map, Bookings, Profile).\n- `business/[id]` – business detail.\n- `booking/[serviceId]` – checkout flow (modal presentation).\n\nDeep linking: `planity://business/abc` opens the business page; `planity://booking/def` resumes booking.\n\n### 4.2 State Management & Data Fetching\nTanStack React Query handles all server state:\n- Custom hooks (e.g., `useBusinesses(params)`) encapsulate query keys, API calls, optimistic updates.\n- Infinite queries for search/explore with pagination.\n- Mutations (book, review, favorite) use `onMutate` for optimistic UI, rollback on error.\n- Query client configured with stale time 30s, gc time 5min; retry 1 on network errors.\n\n### 4.3 UI & Design System\n- All components from `@planity/ui` use design tokens (colors via `useTheme()` context, spacing via `spacing` function, typography components).\n- Reanimated 2/3 used for map interaction (clustering animations), skeleton shimmer, card transitions.\n- Map screen uses `react-native-maps` with custom marker clustering. Queries are driven by viewport bounds; debounced (500ms) call to `/search/geo`.\n- Optimistic updates: favorite toggle, booking cancellation, rating submission.\n\n### 4.4 Authentication Flow on Mobile\n- `expo-auth-session` + `@supabase/supabase-js` manages social logins and email/password.\n- Supabase session persisted using `expo-secure-store`; access token attached to API requests via Axios interceptor.\n- When token expires, interceptor uses refresh token from Supabase to obtain new session before retrying request.\n- Login required middleware (Expo Router) redirects to `(auth)`. After login, deep link is restored.\n\n## 5. Backend Architecture (NestJS)\n\n### 5.1 Modular Design\nEach domain is a NestJS module with controller, service, and repository layer (Prisma).\n- **AuthModule**: Validates Supabase JWTs using passport strategy (`supabase` strategy). Provides `RolesGuard`, `CurrentUser` decorator. On signup, it creates/updates local User record linked to Supabase ID.\n- **UserModule**: Profile management, account deletion (soft delete).\n- **BusinessModule**: CRUD for businesses, services, staff. Owner/provider role restricted.\n- **BookingModule**: Availability engine, booking creation, status transitions. Uses Redis for slot calculations (cache) and distributed locking for double‑booking prevention.\n- **ReviewModule**: Rating & reviews, aggregate rating calculation (trigger on review insert).\n- **NotificationModule**: Push notifications via Expo Push API. Email via external service.\n- **SearchModule**: Full‑text search using `to_tsvector` / `plainto_tsquery` with trigram extension, PostGIS spatial queries. Supports filters, sorting, pagination.\n\n### 5.2 API Design\n- Base URL: `/api/v1`.\n- Envelope: `{ success: boolean, data?: T, error?: { code, message }, meta?: PaginationMeta }`.\n- Input validation with Zod schemas (shared with frontend) via custom `ValidationPipe`.\n- Rate limiting: `@nestjs/throttler` backed by Redis; gradients for login (5/15min), API (100/min).\n\n### 5.3 Database – PostgreSQL + PostGIS + Prisma\n- Prisma schema in `prisma/schema.prisma` defines all models (User, Business, Service, Staff, Booking, Review, etc.) with relations.\n- PostGIS extension enabled; `Business.address` uses `Geography` column for geolocation queries.\n- Full‑text search index on `Business.name`, `description`, `Service.name` via `@@fulltext`.\n- Soft delete via `deletedAt` timestamp field on User; cascade soft delete on related data.\n- Migrations handled by `prisma migrate`; seeding for development.\n- Redis cache for frequently accessed data: featured businesses, categories, availability windows.\n\n### 5.4 Caching & Real‑time\n- Redis module (`@nestjs/cache-manager` with `cache-manager-redis-store`) caches computed slot availability with TTL (1min).\n- When a booking is created or cancelled, related slot cache entries are invalidated.\n- Real‑time updates (provider dashboard, booking status) can be implemented via Supabase Realtime (subscriptions on booking table changes) or a simple WebSocket gateway in NestJS using Redis pub/sub for horizontal scale.\n\n### 5.5 Background Jobs (optional)\n- Redis‑based queue (BullMQ) for sending notification emails, generating daily reports, cleaning expired tokens.\n\n## 6. Data Models & Validation\nAll types and Zod schemas reside in `packages/shared`. Prisma generates database types; we map to/from API responses using DTO classes with `class-transformer`. The shared package exports TypeScript interfaces used in both frontend hooks and backend services.\n\nKey entities:\n- `User` with `role` enum (CUSTOMER, PROVIDER, ADMIN). `supabaseId` links to Supabase.\n- `Business` with `coordinates` (lat/lng) stored as PostGIS geometry.\n- `Service` linked to `Business` and `Staff`.\n- `Booking` with status machine; constraints: `UNIQUE(businessId, staffId, startTime)` to prevent double booking.\n- `Review` linked to `Booking` (one per booking).\n- `Favorite` composite key (customerId, businessId).\n- `Notification` with type discriminator.\n\n## 7. Authentication & Authorization\n**External Identity Provider**: Supabase Auth.\n- Registration/Login with email+password or OAuth (Google, Apple).\n- Supabase issues JWT (access token) and refresh token.\n\n**NestJS Integration**:\n- Passport strategy `SupabaseStrategy` extracts JWT, verifies signature using Supabase public key, fetches user from local DB (or creates if not exists).\n- Role‑based guards check `user.role` for `CUSTOMER`, `PROVIDER`, `ADMIN` endpoints.\n- Refresh token rotation handled by Supabase SDK on client; server does not issue its own refresh tokens.\n- Session blacklisting (on logout) stored in Redis with TTL equal to token expiry.\n\n## 8. Key Non‑Functional Requirements\n- **Performance**: API response times <200ms p95 for cached queries, <500ms for uncached. Mobile app uses optimistic UI to hide latency.\n- **Scalability**: Stateless NestJS containers behind a load balancer; PostgreSQL can be vertically scaled initially, PostGIS queries indexed. Redis for fast read‑through caching.\n- **Resilience**: Circuit breakers on external services (email). Graceful degradation – map can show cached businesses even if search down.\n- **Security**: HTTPS only, input sanitisation, rate limiting, CORS, JWT validation, role checks on every protected endpoint.\n- **Testing**: Jest for unit/integration tests (frontend hooks, backend services). E2E with Detox (mobile) and Supertest (API).\n\n## 9.
