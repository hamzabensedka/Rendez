  "content": "# Planity Clone - System Architecture\n\n## 1. Overview\nPlanity Clone is a mobile-first platform for discovering and booking beauty/wellness services. The system is implemented as an Nx monorepo with pnpm workspaces. It comprises a consumer mobile app (Expo React Native), two web portals (provider and admin, built with Expo Web), a NestJS API, and a BullMQ worker. PostgreSQL with PostGIS is accessed via Prisma; Redis provides caching and queues; Supabase provides auth and storage. CI uses GitHub Actions, local dev uses Docker Compose, mobile builds use EAS Build, and Jest is the test runner.\n\n## 2. Technology Mapping\n- Monorepo: Nx, pnpm, Docker Compose\n- Mobile: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- Web: Expo Web (same RN components)\n- API: NestJS, Prisma, PostgreSQL, PostGIS, Redis, BullMQ\n- Auth/Storage: Supabase Auth, Supabase Storage\n- Payments: Stripe\n- Push: Firebase Cloud Messaging\n- CI/CD: GitHub Actions, EAS Build\n- Tests: Jest\n\n## 3. Service Boundaries\n- **Client Apps (mobile, provider, admin)**: Responsible for UI, navigation, local form state, and calling the API via typed clients. They never connect to Postgres directly. Auth tokens obtained from Supabase are attached to API requests.\n- **API Service (apps/api)**: Single NestJS HTTP server exposing REST endpoints. Validates JWT, enforces RBAC (consumer, business_owner, admin), executes business logic via modules, persists via Prisma, caches in Redis, publishes BullMQ jobs.\n- **Worker Service (apps/worker)**: Separate process consuming BullMQ queues. Shares module services with API via Nx library imports. Handles notifications, reminders, payment reconciliation, and availability warm-up.\n- **Supabase**: Provides Authentication (email/phone/social), Storage for business photos/avatars, and the PostgreSQL database with PostGIS extension. The API uses the Supabase DB URL and Prisma.\n- **Redis**: Used for (1) caching availability slots and search results, (2) rate limiting, (3) BullMQ job/queue storage.\n- **External**: Stripe (payments), Firebase (push), SMTP/email (via Supabase or provider).\n\n## 5. Backend Architecture (NestJS)\nThe API is modular. Each domain has a folder with controller, service, repository (Prisma), and DTOs.\n\nModules:\n- `auth`: Verifies Supabase JWT, exposes user profile sync.\n- `users`: Profile, preferences, payment methods (tokenized via Stripe).\n- `categories`: Service taxonomy CRUD (admin).\n- `businesses`: Profiles, photos, locations (PostGIS point), hours.\n- `services`: Service definitions, pricing, duration, category mapping.\n- `staff`: Staff members, working hours, time-off.\n- `availability`: Engine computing 15-min slots from business hours, staff schedules, service duration, and existing appointments. Caches results in Redis.\n- `booking`: Appointment creation, reschedule, cancel with policy checks.\n- `payments`: Stripe intents, webhooks, refunds.\n- `reviews`: Verified reviews, rating aggregation.\n- `favorites`: User-business saves.\n- `notifications`: Push/email dispatch (publishes jobs).\n- `admin`: Platform metrics, suspend users/businesses.\n- `provider`: Business owner calendar, manual bookings.\n\nAll DB access goes through PrismaService. Geo queries use `@prisma/client` with raw `ST_DWithin` or PostGIS extension functions for map search.\n\n## 6. Availability Engine\nInput: businessId, serviceId, date. Algorithm:\n1. Load business hours and staff shifts for date.\n2. Subtract breaks/time-off.\n3. For each staff, generate 15-min increments covering remaining windows.\n4. Remove slots already booked (query appointments).\n5. Return earliest available across staff.\nResult cached in Redis with TTL 5 min key `avail:{businessId}:{serviceId}:{date}`.\n\n## 7. Mobile App Architecture\n- **Expo Router**: File-based routing under `app/`. Routes: `(tabs)/explore`, `search`, `map`, `business/[id]`, `booking`, `profile`, `favorites`.\n- **TanStack React Query**: All server state fetched via hooks in `packages/query-client`. Queries use `useQuery` for businesses, `useMutation` for bookings. Cache invalidation on mutations.\n- **React Native Reanimated**: Used for skeleton loaders, sheet transitions in booking flow, map marker animations.\n- **Supabase Auth**: Session stored securely; token injected into API client.\n- **Offline**: Minimal guest browse cached via React Query persist.\n\n## 8. Web Portals\nProvider and Admin are Expo Web apps importing the same `packages/ui` and `packages/query-client`. They use role-guarded routes. Provider manages business data, staff, and calendar. Admin manages categories and oversight. Both call the same NestJS API with elevated roles.\n\n## 9. Shared Packages\n- `types`: Interfaces for `User`, `Business`, `Service`, `Appointment`, API request/response DTOs. Consumed by all apps and API.\n- `ui`: Themed `Button`, `Card`, `Input`, `MapView`, `SlotPicker` built with RN + Reanimated, compatible with Web.\n- `query-client`: Configured `QueryClient`, `apiFetch` wrapper, generated hooks.\n- `config`: Jest, ESLint, Tsconfig base.\n\n## 10. Data Model & PostGIS\nCore entities: `User`, `Business` (location `geometry(Point,4326)`), `Category`, `Service`, `Staff`, `Appointment`, `Review`, `Favorite`, `Payment`.\nPostGIS enables:\n- Map viewport search: `ST_Within` with bounding box.\n- Distance filter: `ST_Distance` / `ST_DWithin`.\nIndexes
