Service Boundaries & Responsibilities\n\n### 3.1 Mobile App (apps/mobile)\n**Framework:** Expo SDK 52+, React Native, Expo Router v3\n**State Management:** TanStack React Query for server state, Zustand for client state\n**Animations:** React Native Reanimated 3 for gesture-driven transitions\n\n**Key Responsibilities:**\n- File-based routing with Expo Router (auth groups, tab navigation, dynamic routes)\n- Optimistic UI updates with React Query mutations\n- Offline-first booking flow with conflict detection\n- Map integration (react-native-maps) with marker clustering\n- Secure token storage (expo-secure-store)\n- Push notification handling (expo-notifications)\n- Deep linking for booking confirmations and calendar integration\n\n**Component Architecture:**\n- Atomic design: atoms (Button, Input) → molecules (SearchBar, BusinessCard) → organisms (BookingFlow, MapView) → templates → screens\n- Shared UI kit in `libs/ui-kit` for design consistency\n- Feature-based code splitting with lazy loading\n\n### 3.2 API Gateway (apps/api)\n**Framework:** NestJS 10+ with Fastify adapter\n**Database ORM:** Prisma 5+ with PostgreSQL 15\n**Geospatial:** PostGIS extension for location queries\n**Cache:** Redis 7 for session store, rate limiting, and slot cache\n\n**Module Breakdown:**\n\n#### Auth Module\n- JWT-based authentication with access/refresh token rotation\n- Social login (Google, Apple) via Supabase Auth\n- Email verification and password reset flows\n- Rate limiting on auth endpoints (Redis-backed)\n- Session management with token blacklisting\n\n#### Business Module\n- CRUD operations for business profiles\n- Photo gallery management (Supabase Storage)\n- Working hours configuration\n- Staff assignment to services\n- Business verification workflow\n\n#### Availability Engine\n- **Core Algorithm:** Computes available slots considering:\n  - Data Flow Patterns\n\n### 6.1 Booking Flow (Critical Path)\n```\n1. User selects service → GET /businesses/:id/services\n2. User selects staff (optional) → GET /businesses/:id/staff\n3. User picks date → GET /businesses/:id/availability?serviceId=X&staffId=Y&date=2025-04-15\n   - Backend computes slots: working hours - existing bookings - breaks\n   - Returns array of { startTime, endTime } slots\n4. User selects slot → Client holds slot with optimistic UI\n5. User confirms → POST /bookings\n   - Backend validates slot still available (version check)\n   - Creates booking in transaction\n   - If paid: creates Stripe PaymentIntent, returns client_secret\n   - Returns booking with status PENDING\n6. Client confirms payment (if applicable) → Stripe SDK\n7. Webhook updates booking to CONFIRMED\n8. Push notification sent to user and business owner\n```\n\n### 6.2 Search Flow\n```\n1. User enters query → GET /businesses?q=hair+salon&lat=40.7128&lng=-74.0060&radius=10\n2. Backend:\n   - Full-text search on business name, service names, description\n   - Geospatial filter: ST_DWithin(location, ST_MakePoint(lng, lat), radius)\n   - Apply filters (category, rating, price)\n   - Sort by relevance/distance/rating\n   - Paginate (cursor-based for infinite scroll)\n3. Results cached in Redis (key: search:{query_hash})\n4. Client renders BusinessCard components with distance, rating, next slot\n```\n\n### 6.3 Real-time Availability\n```\n- Slot computation is CPU-intensive; pre-computed and cached\n- Cache key: availability:{businessId}:{serviceId}:{staffId}:{date}\n- TTL: 5 minutes (short to ensure freshness)\n- Invalidation triggers: new booking, cancellation, staff schedule change\n- On cache miss: compute slots, store in Redis, return\n- Booking creation uses optimistic locking (version field) to prevent double-booking\n```\n\n## 7. Security Considerations\n\n- **Authentication:** JWT with short-lived access tokens (1h) and refresh tokens (30d). Refresh token rotation on use.\n- **Authorization:** Role-based guards (CLIENT, PROVIDER, ADMIN). Resource ownership validation.\n- **Rate Limiting:** Redis-backed rate limiter on auth endpoints (5 attempts/minute), search (60/minute), booking creation (10/minute).\n- **Input Validation:** Class-validator DTOs with strict whitelisting. SQL injection prevented by Prisma parameterized queries.\n- **Payment Security:** PCI compliance via Stripe Elements/SDK. No raw card data touches our servers.\n- **Data Encryption:** TLS 1.3 for all communications. Sensitive data encrypted at rest (Supabase).\n- **API Security:** Helmet.js for HTTP headers, CORS whitelist, CSRF protection for web.\n\n## 9. Performance Optimizations\n\n- **Mobile:** Image optimization (expo-image), lazy loading, skeleton screens, paginated lists (FlashList), Reanimated for 60fps animations.\n- **API:** Response compression (gzip), connection pooling (Prisma), Redis caching for hot data (categories, popular businesses, availability slots).\n- **Database:** Strategic indexes (GIST for geospatial, B-tree for foreign keys and query patterns), read replicas for search queries, connection pooling (PgBouncer).\n- **Network:** GraphQL not used; REST with selective field projection (sparse fieldsets) to minimize payload. ETags for cache validation.\n\n## 10. Testing Strategy\n\n- **Unit Tests:** Jest for business logic (availability computation, booking state machine, validation). Target: 80% coverage on services.\n- **Integration Tests:** Supertest for API endpoints with test database. Monitoring & Observability\n\n- **Logging:** Structured JSON logs (Pino) with correlation IDs.\n- **Metrics:** Prometheus metrics for request duration, error rates, booking volume.\n- **Tracing:** OpenTelemetry for distributed tracing across API and database calls.\n- **Error Tracking:** Sentry for both mobile and backend.\n- **Alerts:** PagerDuty integration for booking failures, payment errors, API downtime.\n\n## 12. Key Architectural Decisions (ADRs)\n\n1. **Monorepo with Nx:** Shared types and utilities reduce duplication. Nx provides intelligent build caching and dependency graph.\n2. **REST over GraphQL:** Simpler caching, better tooling for mobile (React Query), lower complexity for a focused domain.\n3. **PostGIS over Elasticsearch:** Single database for transactional and geospatial data reduces operational complexity at this scale.\n4. **Redis for Availability Cache:** Pre-computed slots with short TTL balance freshness and performance. Avoids real-time computation on every request.\n5. **Supabase for Auth:** Offloads secure authentication, social login, and file storage.
