Service Boundaries\n\n### 5.1 Auth Service\n- **Responsibilities:** User registration, login, OTP, password reset, session management, role-based access control, biometric authentication setup.\n- **Data owned:** Users (credentials, roles, sessions), OTP codes, refresh tokens.\n- **Endpoints:** `POST /auth/register`, `POST /auth/login`, `POST /auth/otp/send`, `POST /auth/otp/verify`, `POST /auth/password/reset`, `GET /auth/sessions`, `DELETE /auth/sessions/:id`.\n\n### 5.2 Business Service\n- **Responsibilities:** Business CRUD, search (full-text, geospatial), category management, business hours, staff profiles, photos.\n- **Data owned:** Businesses, categories, staff, business hours, photos.\n- **Endpoints:** `GET /businesses`, `GET /businesses/:id`, `POST /businesses`, `PUT /businesses/:id`, `GET /businesses/search`, `GET /categories`.\n\n### 5.3 Service Catalog Service\n- **Responsibilities:** Service CRUD per business, pricing, duration, add-ons.\n- **Data owned:** Services, add-ons, service categories.\n- **Endpoints:** `GET /businesses/:id/services`, `POST /businesses/:id/services`, `PUT /services/:id`.\n\n### 5.4 Booking Service\n- **Responsibilities:** Slot availability computation (real-time), booking creation, slot holds, cancellation, reminders.\n- **Data owned:** Appointments, slot holds, booking history.\n- **Endpoints:** `GET /businesses/:id/availability`, `POST /bookings`, `GET /bookings/:id`, `PUT /bookings/:id/cancel`, `POST /bookings/:id/hold`.\n- **Redis usage:** Slot holds (TTL 10 minutes), availability cache (TTL 5 minutes), rate limiting.\n\n### 5.5 Review Service\n- **Responsibilities:** Review CRUD, rating aggregation, owner responses.\n- **Data owned:** Reviews, ratings, responses.\n- **Endpoints:** `GET /businesses/:id/reviews`, `POST /businesses/:id/reviews`, `PUT /reviews/:id`, `POST /reviews/:id/response`.\n\n### 5.6 Notification Service\n- **Responsibilities:** Push notifications (via Expo Push API), email (via SendGrid/Mailgun), SMS (via Twilio), in-app notifications.\n- **Data owned:** Notification templates, delivery logs, user notification preferences.\n- **Endpoints:** `POST /notifications/send`, `GET /notifications`, `PUT /notifications/:id/read`.\n\n### 5.7 Admin Service\n- **Responsibilities:** Admin dashboard, user management, business verification, category management, analytics.\n- **Data owned:** Admin users, audit logs, system settings.\n- **Endpoints:** `GET /admin/users`, `PUT /admin/businesses/:id/verify`, `POST /admin/categories`.\n\n## 6. Data Model (Prisma Schema Highlights)\n\n```prisma\nmodel User {\n  id            String   @id @default(cuid())\n  email         String?  @unique\n  phone         String?  @unique\n  passwordHash  String?\n  role          Role     @default(CONSUMER)\n  createdAt     DateTime @default(now())\n  updatedAt     DateTime @updatedAt\n  bookings      Booking[]\n  reviews       Review[]\n  sessions      Session[]\n}\n\nmodel Business {\n  id          String   @id @default(cuid())\n  name        String\n  description String?\n  categoryId  String\n  location    Geography?  // PostGIS point\n  address     String\n  phone       String?\n  website     String?\n  rating      Float    @default(0)\n  verified    Boolean  @default(false)\n  ownerId     String\n  owner       User     @relation(fields: [ownerId], references: [id])\n  services    Service[]\n  staff       Staff[]\n  reviews     Review[]\n  hours       BusinessHour[]\n}\n\nmodel Service {\n  id          String   @id @default(cuid())\n  businessId  String\n  name        String\n  description String?\n  duration    Int      // minutes\n  price       Decimal\n  categoryId  String\n  business    Business @relation(fields: [businessId], references: [id])\n}\n\nmodel Booking {\n  id          String   @id @default(cuid())\n  userId      String\n  businessId  String\n  serviceId   String\n  staffId     String?\n  startTime   DateTime\n  endTime     DateTime\n  status      BookingStatus @default(PENDING)\n  notes       String?\n  user        User     @relation(fields: [userId], references: [id])\n  business    Business @relation(fields: [businessId], references: [id])\n  service     Service  @relation(fields: [serviceId], references: [id])\n}\n\nenum Role { CONSUMER, PROVIDER, ADMIN }\nenum BookingStatus { PENDING, CONFIRMED, CANCELLED, COMPLETED }\n```\n\n## 7. Key Design Decisions\n\n1. **Monorepo with Nx and pnpm:** Ensures shared types, linting, and build consistency across mobile and backend.\n2. **NestJS modules:** Each domain (auth, business, booking) is a self-contained module with its own controller, service, and DTOs, promoting separation of concerns.\n3. **TanStack React Query:** Handles server state caching, background refetching, and optimistic updates on the mobile client.\n4. **Expo Router:** File-based routing for the mobile app, simplifying navigation and deep linking.\n5. **PostGIS for geospatial:** Enables efficient radius-based search and map clustering.\n6. **Redis for slot holds:** Temporary holds with TTL prevent double-booking during checkout.\n7. **Supabase for auth and real-time:** Optional; can be used as an external auth provider or for real-time subscriptions.\n8. **Docker Compose for local development:** Spins up PostgreSQL, Redis, and the API for consistent local environment.\n9. **GitHub Actions for CI/CD:** Runs linting, tests, and builds on every push; deploys to staging/production.\n10. **EAS Build for mobile builds:** Simplifies building and submitting iOS and Android apps.\n\n## 8. API Design Principles\n\n- RESTful endpoints with consistent naming (`/api/v1/...`).\n- Pagination via cursor-based or offset-based (default 20 per page).\n- Error responses follow RFC 7807 (Problem Details).\n- Authentication via JWT Bearer token in `Authorization` header.\n- Rate limiting per endpoint (e.g., 100 requests/minute for search, 10 for booking creation).\n- WebSocket for real-time availability updates (optional, can use polling with React Query).\n\n## 9. Security Considerations\n\n- Passwords hashed with bcrypt (cost factor 12).\n- JWT access tokens short-lived (15 minutes), refresh tokens rotated.\n- OTP expiry and cooldown enforced server-side.\n- Account lockout after 5 failed attempts.\n- Input validation using class-validator and DTOs.\n- CORS configured for allowed origins.\n- Helmet middleware for HTTP security headers.\n- GDPR consent stored in database.\n\n## 10. Testing Strategy\n\n- **Unit tests:** Jest for individual functions and services.\n- **Integration tests:** Supertest for API endpoints, with test database.\n- **E2E tests:** Detox or Maestro for mobile flows.\n- **CI pipeline:** Runs lint, unit tests, integration tests on every PR.\n\n## 11. Deployment\n\n- **Mobile:** EAS Build -> App Store / Google Play.\n- **API:** Docker container deployed to cloud (AWS ECS, Google Cloud Run, or similar).\n- **Database:** Managed PostgreSQL (Supabase, AWS RDS, or similar).\n- **Redis:** Managed Redis (Upstash, AWS ElastiCache).\n- **CI/CD:** GitHub Actions builds Docker image, runs tests, deploys to staging, then production after approval.\n\n## 12. Future Considerations\n\n- Microservices extraction (e.g., separate notification service) if load increases.\n- Event-driven architecture with message queue (RabbitMQ / Kafka) for async tasks like sending emails.\n- GraphQL layer for complex queries (e.g., dashboard analytics).\n- Offline-first support with WatermelonDB or similar.\n\n---\n\n*This architecture document is a living artifact and should be updated as the system evolves.*",
