  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nThe Planity Clone connects users with local businesses for discovering, booking, and managing appointments. The system follows a clean architecture with clear separation between the mobile client, API layer, and data stores. It leverages an Nx monorepo to share code and enforce linting/testing standards.\n\n## 2. Tech Stack\n- **Mobile**: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Backend**: NestJS, Prisma ORM, PostgreSQL (with PostGIS extension), Redis, BullMQ\n- **Tooling**: Nx monorepo, pnpm, Docker Compose, GitHub Actions, EAS Build, Supabase (optional for auth/storage), Jest\n\n## 3. High-Level Architecture\nThe application consists of three main horizontal layers:\n1. **Presentation Layer** – Expo mobile app handling UI, navigation, state management, and local caching.\n2. **Application Layer** – NestJS API providing REST/GraphQL endpoints, business logic, validation, and integration with external services (payment gateway, push notifications).\n3. **Data Layer** – PostgreSQL for relational data, PostGIS for geo‑queries, Redis for caching and pub/sub, and BullMQ for background job processing.\n\nCross‑cutting concerns (authentication, logging, error handling, monitoring) are implemented as NestJS middleware and React Native hooks.\n\n## 4. Monorepo Structure (Nx + pnpm)\n```\nplanity-clone/\n├── apps/\n│   ├── api/               # NestJS server\n│   └── mobile/            # Expo React Native app\n├── libs/\n│   ├── shared/            # Shared TypeScript interfaces, enums, constants\n│   ├── ui/                # Reusable React Native components (buttons, cards, modals)\n│   ├── api-types/         # Generated API DTOs (via swagger or manual)\n│   ├── design-system/     # Theme, typography, spacing tokens\n│   ├── utils/             # Helper functions (date, geo, validation)\n│   ├── eslint-config/     # Shared ESLint rules\n│   └── jest-config/       # Jest presets\n├── tools/\n│   └── scripts/           # DB migrations, seed scripts\n├── docker-compose.yml\n├── nx.json\n├── package.json\n└── pnpm-workspace.yaml\n```\n\n## 5. Mobile Application (apps/mobile)\n- **Routing**: Expo Router (file‑based) provides stack and tab navigation.\n- **State Management**: TanStack React Query for server state; React Native Reanimated for gestures and animations.\n- **Architecture**: Feature‑sliced structure.\n```\nsrc/\n  app/\n    (tabs)/\n      _layout.tsx          # Tab navigator\n      index.tsx            # Home (guest browse)\n      explore/\n        index.tsx          # Search & list\n        [businessId]/      # Business detail\n      map/\n        index.tsx          # Map view\n      bookings/\n        index.tsx          # Upcoming appointments\n        [bookingId]/       # Booking detail\n      profile/\n        index.tsx          # User profile\n        edit.tsx\n      favorites/\n        index.tsx\n      auth/\n        sign-in.tsx\n        sign-up.tsx\n        reset-password.tsx\n  components/\n    # Shared UI from libs/ui\n  hooks/\n    # Custom React Query hooks, geo hooks\n  utils/\n    # Formatters, constants\n  assets/\n    # Images, fonts\n  types/\n    # Local TypeScript types (re-export from libs/shared)\n```\n\n## 6. Backend API (apps/api)\nNestJS modular architecture.\n```\nsrc/\n  main.ts\n  app.module.ts\n  config/\n    # Environment validation\n  common/\n    # Guards, interceptors, pipes, exceptions\n  auth/\n    # JWT strategy, local guard\n  users/\n    # User service, controller, DTOs\n  businesses/\n    # Business CRUD, geo search, services\n  services/\n    # Service definitions tied to businesses\n  bookings/\n    # Booking creation, validation, slot computation\n  appointments/\n    # Appointment entity, status transitions\n  reviews/\n    # Review creation, moderation\n  payments/\n    # Payment gateway integration (stub)\n  notifications/\n    # Notification service (email, push)\n  jobs/\n    # BullMQ processors (send‑email, push‑notification)\n  geo/\n    # PostGIS helpers for distance queries\n  prisma/\n    # Prisma client extension, custom types\n```\n\n## 7. Shared Libraries (libs)\n- **shared**: Contains `User`, `Business`, `Service`, `Booking`, `Appointment`, `Review` interfaces, enums (UserRole, BookingStatus), and utility types.\n- **ui**: Stateless React Native components built with Reanimated 2 gestures where needed.\n- **design-system**: Theme object (colors, spacing, typography) consumed via React Context.\n- **utils**: Date formatting, haversine distance, slug generation, validation helpers.\n- **api-types**: Generated from NestJS DTOs using `nestjs-swagger` or manual keep‑in‑sync script.\n\n## 8. Database Design\n### PostgreSQL Schema (simplified)\n- **users**: id, email, password_hash, role, created_at, updated_at\n- **businesses**: id, owner_id, name, description, address, latitude, longitude, phone, website, created_at, updated_at\n- **business_hours**: id, business_id, day_of_week, open_time, close_time, is_closed\n- **services**: id, business_id, name, description, duration, price\n- **service_categories**: id, name\n- **business_service_categories**: business_id, service_category_id (many‑to‑many)\n- **bookings**: id, user_id, service_id, start_time, end_time, status, payment_status, created_at, updated_at\n- **appointments**: id, booking_id, business_confirmation, notes, created_at, updated_at\n- **reviews**: id, user_id, business_id, rating, comment, created_at, updated_at\n- **favorites**: id, user_id, business_id, created_at\n- **refresh_tokens**: id, user_id, token, expires_at, created_at\n\nPostGIS extension adds geography column `location` on businesses for radius queries.\n\n## 9. Caching & Pub/Sub (Redis)\n- **Cache Layer**: Frequently read‑only data (business list, service categories, user profile) cached with TTL.\n- **Pub/Sub**: Channels for real‑time events (new booking, review posted) used by NestJS to push updates via WebSocket or trigger push notifications.\n\n## 10. Background Jobs (BullMQ)\nQueues:\n- `email-notifications`: sends confirmation, reminder, cancellation emails.\n- `push-notifications`: triggers Expo push notifications.\n- `slot-recalculation`: recomputes available slots when business hours or bookings change.\nProcessors are defined in `apps/api/src/jobs`.\n\n## 11. Authentication & Authorization\n- **Auth Strategy**: JWT (access token 15 min, refresh token 7 days) stored in SecureStore on mobile.\n- **Guards**: `AuthGuard` (JWT), `RolesGuard` (admin, business_owner, user).\n- **Password Reset**: Token‑based flow with expiry.\n- **Social Login** (future): can be added via Supabase Auth or OAuth2.\n\n## 12. Real‑time Notifications\n- **Expo Push Notifications** for mobile; triggered by BullMQ jobs.\n- **WebSocket (optional)**: NestJS `@nestjs/websockets` for live updates (e.g., admin dashboard seeing new bookings).\n\n## 13. CI/CD (GitHub Actions, EAS Build, Docker Compose)\n- **Backend**: Docker image built and pushed to registry; deployed via Docker Compose on staging/production.\n- **Mobile**: EAS Build creates IPA/APK; GitHub Actions runs Expo prebuild, runs tests, and uploads to EAS.\n- **Workflow**: \n  1. Push to `main` triggers CI.\n  2. Lint (ESLint), TypeCheck, Unit Tests (Jest) for both mobile and api.\n  3. On success, backend image built and deployed via Docker Compose (or Kubernetes).\n  4. Mobile build triggered via EAS; artifact uploaded to TestFlight/Google Play internal track.\n\n## 14. Deployment & Environments\n- **Local**: `docker-compose up` runs API, PostgreSQL, Redis, and optionally Expo dev client.\n- **Staging**: Separate Docker Compose stack with environment variables pointing to staging DB.\n- **Production**: Managed PostgreSQL (e.g., Supabase or AWS RDS), Elasticache Redis, EC2/ECS for API, Expo EAS for OTA updates.\n\n## 15. Service Boundaries & Responsibilities\n| Boundary | Responsibility |\n|----------|----------------|\n| Mobile Client | UI/UX, navigation, local state, offline caching, gesture animations, Expo push handling. |\n| API Gateway (NestJS) | Request validation, auth, business logic, transaction management, third‑party integrations. |\n| Data Store (PostgreSQL + PostGIS) | Persistent relational data, geo‑indexed queries. |\n| Cache/PubSub (Redis) | Read‑through caching, event broadcasting. |\n| Background Workers (BullMQ) | Async jobs: emails, push notifications, slot recomputation. |\n| Design System / Shared Libs | Consistent UI components, TypeScript types, utility functions. |\n\n## 16. Data Flow Examples\n### Guest Browse Businesses\n1. Mobile sends GET `/businesses?lat=...&lon=...&radius=...` (cached).\n2. API queries PostGIS for nearby businesses, returns DTOs.\n3. React Query caches response; UI renders list.\n\n### User Booking Flow\n1. User selects service → mobile opens time‑slot picker (computed locally from cached business hours + existing bookings).\n2. On submit, POST `/bookings` with JWT.\n3. API validates slot availability (double‑check via transaction), creates booking, publishes `booking.created` event.\n4. BullMQ `email-notifications` and `push-notifications` jobs send confirmation.\n5. Mobile updates local query cache via React Query invalidation.\n\n### Review Submission\n1. POST `/reviews` with JWT.\n2. API stores review, increments business rating aggregate.\n3. Publishes `review.created` event.\n4. BullMQ pushes notification to business owner.\n\n## 17. Security Considerations\n- HTTPS everywhere; API behind TLS terminator.\n- JWT stored in SecureStore; refresh token rotation.\n- Rate limiting via NestJS `ThrottlerGuard`.\n- Input validation with class‑validator and DTOs.\n- SQL injection prevented by Prisma ORM.\n- Geo‑queries use parameterized PostGIS functions.\n- File uploads (if any) scanned and stored in S3 with private ACL.\n- Audit logs for admin actions.\n\n## 18. Scalability & Performance\n- Read replicas for PostgreSQL to scale business listing queries.\n- Redis clustering for cache and pub/sub.\n- Horizontal scaling of NestJS API behind load balancer.\n- BullMQ workers can be scaled per queue.\n- Mobile uses pagination and infinite scroll; React Query keeps stale‑while‑revalidate.\n\n## 19. Testing Strategy\n- **Unit Tests**: Jest for services, utilities, React Native hooks.\n- **Integration Tests**: Supertest for API endpoints; React Native Testing Library for screens.\n- **E2E**: Cypress for web‑like flows (optional) and Detox for mobile critical paths.\n- **Test Coverage Goal**: >80% for backend, >70% for mobile.\n\n## 20. Open Questions / Future Work\n- Adoption of GraphQL vs REST for flexible data fetching.\n- WebSocket implementation for live admin dashboard.\n- Multi‑tenant SaaS scaling (separate schemas or row‑level security).\n- Advanced analytics with data warehouse (e.g., ClickHouse).\n- Offline-first capabilities using Expo SQLite.\n\n---\n*Architecture documented by Jordan, Staff Software Engineer.*\n",
