{"content": "# Planity Clone — System Architecture\n\n## 1. Overview\n\nThis document defines the complete system architecture for the Planity Clone, a mobile-first marketplace connecting service providers with clients. The architecture supports three client applications (Consumer Mobile, Provider Portal, Admin Dashboard) backed by a unified API layer, with clear service boundaries and horizontal scaling capabilities.\n\n## 2. Architectural Principles\n\n| Principle | Application |\n|-----------|-------------|\n| **Mobile-First** | Consumer experience optimized for native mobile; web views for provider/admin |\n| **API-First** | All clients consume the same GraphQL/REST API surface |\n| **Domain-Driven Design** | Bounded contexts align with business capabilities |\n| **Eventual Consistency** | Async processing for non-critical paths via message queue |\n| **Infrastructure as Code** | Docker Compose local; Terraform for cloud staging/prod |\n| **Observability by Design** | Structured logging, distributed tracing, metrics everywhere |\n\n## 3. Service Boundaries\n\n### 5.1 Identity Service\n**Responsibility:** Authentication, authorization, user management\n\n| Component | Implementation |\n|-----------|----------------|\n| Auth Provider | Supabase Auth (primary), fallback self-hosted JWT |\n| Token Strategy | Access: JWT (15min), Refresh: opaque token (7d) in httpOnly cookie |\n| OAuth | Google, Apple Sign-In via Supabase |\n| Phone Verify | Twilio Verify integration |\n| Rate Limiting | Redis-based, 5 attempts/15min/IP |\n\n**Key Flows:**\n- Guest → Registered: Merge anonymous ID with authenticated account\n- Account Deletion: 30-day grace period, scheduled purge job\n- Token Rotation: Refresh token family pattern for theft detection\n\n### 5.2 Discovery Service\n**Responsibility:** Search, geospatial queries, category hierarchy, business profiles\n\n| Component | Implementation |\n|-----------|----------------|\n| Full-text Search | Meilisearch with typo tolerance (`hair saln` → `hair salon`) |\n| Geospatial | PostGIS `ST_DWithin`, `ST_Distance`, spatial index on `business.location` |\n| Category Tree | Closure table pattern for 3-level hierarchy |\n| Results Caching | Redis, 5min TTL for popular queries |\n\n**Performance Targets:**\n- Search query: < 500ms p95\n- Map clustering: Client-side for < 500 markers, server-side aggregation for dense regions\n- Image delivery: CDN with WebP, 3 responsive sizes\n\n### 5.3 Booking Service\n**Responsibility:** Availability engine, appointment lifecycle, conflict resolution\n\n| Component | Implementation |\n|-----------|----------------|\n| Slot Generation | Materialized availability view, regenerated on schedule change |\n| Conflict Prevention | `SELECT FOR UPDATE` + unique constraint on `(business_id, staff_id, start_time)` |\n| Pessimistic Lock | Redis SETNX 10min hold during checkout |\n| State Machine | Pending → Confirmed → Checked-in → Completed / Cancelled / No-show |\n\n**Critical Path:**\n```\n1. Client requests slots → Query materialized availability\n2. User selects slot → Redis lock acquired\n3. Payment initiated → Lock extended\n4. Payment confirmed → Transactional booking insert\n5. Failure at any step → Lock released, slot returns to pool\n```\n\n### 5.4 Notification Service\n**Responsibility:** Multi-channel delivery, preference management, scheduling\n\n| Channel | Provider | Use Case |\n|---------|----------|----------|\n| Push | Firebase Cloud Messaging | Real-time updates, reminders |\n| SMS | Twilio | Booking confirmations, 2FA |\n| Email | SendGrid / Resend | Receipts, marketing, password reset |\n| In-app | WebSocket | Live slot availability |\n\n## 6.
