  "content": "# Planity Clone Architecture Document\n\n## 1. Overview\nPlanity Clone is a mobile-first platform connecting customers with local service businesses. The system is built as a monorepo using Nx and pnpm, with a React Native (Expo) mobile app, a NestJS backend, and shared TypeScript packages. The architecture emphasizes separation of concerns, scalability, and maintainability. Real-time features like slot holding and notifications are handled via BullMQ and Redis. Supabase provides authentication, file storage, and optional real-time capabilities. PostgreSQL with PostGIS powers geospatial search, while Prisma serves as the ORM. The entire stack is containerized with Docker Compose for local development and deployed via GitHub Actions and EAS Build.\n\n## 2. Technology Stack\n- **Mobile**: Expo SDK 50+, React Native, TypeScript, Expo Router (file-based routing), TanStack React Query (server state), React Native Reanimated (animations), Supabase JS client.\n- **Backend**: NestJS (Node.js), TypeScript, Prisma ORM, PostgreSQL 15, PostGIS extension, Redis 7, BullMQ (job queue).\n- **Shared**: TypeScript interfaces, DTOs, validation schemas (Zod), design system tokens.\n- **Monorepo**: Nx workspace, pnpm workspaces.\n- **Infrastructure**: Docker Compose (local dev), GitHub Actions (CI/CD), EAS Build (mobile builds), Supabase (managed auth/storage).\n- **Testing**: Jest (unit/integration), React Native Testing Library, Supertest (e2e for API).\n\n## 4. Service Boundaries\n\n### 5.1 Mobile App (apps/mobile)\n- **Responsibilities**: UI rendering, navigation, local state, optimistic updates, offline support (basic), push notification handling, deep linking.\n- **Key Libraries**: Expo Router (file-based routing with `app/` directory), TanStack React Query for data fetching/caching, React Native Reanimated for smooth transitions, Supabase client for auth and real-time subscriptions.\n- **State Management**: React Query for server state, React Context for auth/user state, Zustand for complex local UI state (e.g., booking flow wizard).\n- **Folder Structure**:\n  ```\n  apps/mobile/\n  в”ңв”Җв”Җ app/                      # Expo Router routes (screens)\n  в”Ӯ   в”ңв”Җв”Җ (auth)/               # Auth group (login, signup)\n  в”Ӯ   в”ңв”Җв”Җ (tabs)/               # Main tabs (home, search, bookings, profile)\n  в”Ӯ   в”ңв”Җв”Җ business/[id]/        # Business detail\n  в”Ӯ   в”ңв”Җв”Җ booking/              # Booking flow\n  в”Ӯ   в””в”Җв”Җ _layout.tsx           # Root layout\n  в”ңв”Җв”Җ components/               # Screen-specific components\n  в”ңв”Җв”Җ hooks/                    # Custom hooks (useAuth, useBooking)\n  в”ңв”Җв”Җ services/                 # API calls, Supabase queries\n  в”ңв”Җв”Җ utils/                    # Helpers\n  в”ңв”Җв”Җ constants/                # Theme, config\n  в””в”Җв”Җ assets/                   # Images, fonts\n  ```\n\n### 5.2 API Gateway & Core Services (apps/api)\nThe NestJS backend is organized into modules that map to business domains. A thin API Gateway layer handles cross-cutting concerns (auth, rate limiting, logging) and routes requests to the appropriate service module.\n\n- **Modules**:\n  - `AuthModule`: User registration, login, social auth, token refresh, password reset. Integrates with Supabase Auth for identity management; validates Supabase JWT and syncs user profile in local DB.\n  - `BusinessModule`: CRUD for businesses, services, staff, working hours, categories. Geospatial queries using PostGIS.\n  - `BookingModule`: Slot computation, booking creation, rescheduling, cancellation, payment integration (Stripe). Real-time slot hold via Redis.\n  - `ReviewModule`: Review submission, moderation, responses.\n  - `UserModule`: Profile management, favorites, payment methods, notification preferences.\n  - `NotificationModule`: Push/email/SMS dispatch via BullMQ jobs.\n  - `AdminModule`: Dashboard analytics, moderation, category management (used by admin app).\n  - `CommonModule`: Guards, interceptors, pipes, decorators.\n\n- **Folder Structure**:\n  ```\n  apps/api/\n  в”ңв”Җв”Җ src/\n  в”Ӯ   в”ңв”Җв”Җ main.ts               # Bootstrap\n  в”Ӯ   в”ңв”Җв”Җ app.module.ts\n  в”Ӯ   в”ңв”Җв”Җ common/                # Shared guards, filters, pipes\n  в”Ӯ   в”ңв”Җв”Җ config/                # Environment configuration\n  в”Ӯ   в”ңв”Җв”Җ auth/\n  в”Ӯ   в”ңв”Җв”Җ business/\n  в”Ӯ   в”ңв”Җв”Җ booking/\n  в”Ӯ   в”ңв”Җв”Җ review/\n  в”Ӯ   в”ңв”Җв”Җ user/\n  в”Ӯ   в”ңв”Җв”Җ notification/\n  в”Ӯ   в”ңв”Җв”Җ admin/\n  в”Ӯ   в””в”Җв”Җ prisma/                # Prisma service, migrations\n  в”ңв”Җв”Җ prisma/\n  в”Ӯ   в”ңв”Җв”Җ schema.prisma\n  в”Ӯ   в””в”Җв”Җ migrations/\n  в”ңв”Җв”Җ test/                      # E2E tests\n  в””в”Җв”Җ Dockerfile\n  ```\n\n### 5.3 Background Jobs (integrated in apps/api)\nBullMQ workers run in the same NestJS process (or separate process for scalability) and handle:\n- Appointment reminders (24h, 1h before)\n- Review request after appointment completion\n- Slot hold expiry cleanup\n- Data export/import\n- Notification dispatch (push, email, SMS)\n\n### 5.4 Admin Dashboard (apps/admin)\nA separate web application (Next.js or Vite + React) for platform administrators and business owners. It consumes the same API but with admin-specific endpoints. Not detailed here but follows similar patterns.\n\n## 6. Data Layer\n\n### 6.1 PostgreSQL + PostGIS\nPrimary database. PostGIS enables geospatial queries for map-based search and distance calculations. Key tables:\n- `users`, `businesses`, `services`, `staff`, `bookings`, `reviews`, `categories`, `favorites`, `payments`, `notifications`.\n- `businesses` has a `location` column of type `geometry(Point, 4326)`.\n- Indexes on foreign keys, search fields, and spatial indexes.\n\n### 6.2 Prisma ORM\n- Schema defined in `prisma/schema.prisma`.\n- Use Prisma Client for type-safe database access.\n- Migrations managed via Prisma Migrate.\n- Raw SQL queries for complex geospatial operations (e.g., `ST_DWithin`).\n\n### 6.3 Redis\nUsed for:\n- **Slot hold cache**: Temporary reservation of a time slot (key: `slot_hold:{businessId}:{staffId}:{date}:{time}`, TTL 5 min).\n- **Session/token blacklist** (optional).\n- **Rate limiting**.\n- **Caching**: Business details, category lists, frequently accessed data (invalidate on updates).\n- **BullMQ queue backend**.\n\n## 7. Authentication & Authorization\n- **Identity Provider**: Supabase Auth handles sign-up, login, social OAuth, and JWT issuance. The mobile app uses Supabase SDK for direct auth flows.\n- **Backend Validation**: NestJS `AuthGuard` extracts the JWT from the `Authorization` header, verifies it using Supabase's JWKS endpoint, and attaches the user context. A local `users` table syncs essential profile data (id, email, role) on first login.\n- **Roles**: `customer`, `provider`, `admin`. Role-based access control (RBAC) enforced via guards.\n- **Token Refresh**: Supabase SDK automatically refreshes tokens; the mobile app attaches the latest access token to API requests.\n\n## 8. API Design\n- **RESTful** endpoints with JSON.\n- **Versioning**: URL prefix `/api/v1/`.\n- **Pagination**: Cursor-based for lists (bookings, reviews), offset for search results.\n- **Error Handling**: Standardized error response format `{ statusCode, message, errors }`.\n- **Validation**: Zod schemas shared between frontend and backend via `libs/shared-types`; NestJS `ValidationPipe` uses them.\n- **Documentation**: Swagger/OpenAPI auto-generated from NestJS decorators.\n\nKey endpoints:\n- `POST /auth/signup`, `POST /auth/login`, `POST /auth/refresh`\n- `GET /businesses?lat=&lng=&radius=&category=&query=`\n- `GET /businesses/:id`\n- `GET /businesses/:id/services`\n- `GET /businesses/:id/staff`\n- `GET /businesses/:id/slots?date=&serviceId=&staffId=`\n- `POST /bookings` (body: serviceIds, staffId, startTime, notes)\n- `PATCH /bookings/:id/cancel`\n- `GET /users/me/favorites`\n- `POST /users/me/favorites/:businessId`\n- `DELETE /users/me/favorites/:businessId`\n- `POST /reviews`\n\n## 9. Real-time Slot Hold Mechanism\n1. When a user enters the date/time selection step, the mobile app requests available slots via `GET /slots`. The backend computes slots based on business hours, staff schedules, existing bookings, and service duration, excluding any slots currently held (checked in Redis).\n2. When the user selects a specific slot, the app calls `POST /bookings/hold` with the slot details. The backend:\n   - Checks Redis for an existing hold (atomic `SETNX` with TTL 300s).\n   - If successful, returns a hold token.\n   - If conflict, returns an error prompting the user to pick another slot.\n3. The hold token must be submitted with the final booking creation (`POST /bookings`). The backend verifies the hold is still valid and deletes it upon successful booking.\n4. A BullMQ delayed job is scheduled to release the hold after 5 minutes if not consumed.\n\n## 10. Caching Strategy\n- **Redis** caches:\n  - Business details (key: `business:{id}`, TTL 10 min). Invalidated on update.\n  - Category tree (key: `categories`, TTL 1 hour).\n  - Staff and service lists per business (TTL 5 min).\n  - Available slots (short TTL, 30 seconds) to reduce DB load during peak booking times.\n- **TanStack Query** on the mobile side provides client-side caching with stale-while-revalidate, reducing network requests and improving perceived performance.\n\n## 11. File Storage\n- User avatars, business photos, review images stored in **Supabase Storage**.\n- Public bucket for business photos, private bucket for user uploads.\n- Image URLs stored in the database; mobile app uses Supabase SDK to upload directly from the device, bypassing the backend for large files.\n\n## 12. Deployment & CI/CD\n- **Mobile**: EAS Build creates development, preview, and production builds. GitHub Actions triggers EAS builds on pushes to `main` and `develop`. Over-the-air updates via EAS Update.\n- **Backend**: NestJS app containerized with Docker. GitHub Actions builds the Docker image, runs tests, and deploys to a cloud provider (e.g., AWS ECS, Fly.io, or Railway). Docker Compose used for local development.\n- **Database**: Managed PostgreSQL with PostGIS (Supabase or AWS RDS). Redis managed service (Upstash or ElastiCache).\n- **Environment Variables**: Managed via GitHub Secrets and injected at build/deploy time.\n\n## 13. Testing Strategy\n- **Unit Tests**: Jest for all services, utilities, and hooks. Shared logic in `libs/` tested independently.\n- **Integration Tests**: NestJS supertest for API endpoints with a test database (Dockerized PostgreSQL). Prisma migrations run before tests.\n- **Mobile Tests**: React Native Testing Library for component tests, Jest for hooks and utilities. E2E with Detox (optional).\n- **CI**: All tests run on pull requests; coverage thresholds enforced.\n\n## 14. Security Considerations\n- All communications over HTTPS.\n- JWT validation on every request; short-lived access tokens (15 min), longer refresh tokens.\n- Input sanitization and strict validation (Zod).\n- Rate limiting on auth and booking endpoints.\n- SQL injection prevented by Prisma parameterized queries.\n- CORS configured for mobile app and admin dashboard origins.\n- Sensitive data (API keys, secrets) stored in environment variables, never in code.\n\n## 15. Monitoring & Observability (Future)\n- Structured logging (Pino) with correlation IDs.\n- Health check endpoints for load balancers.\n- Metrics collection (Prometheus) for API performance, booking rates, slot hold conflicts.\n- Error tracking (Sentry) for both mobile and backend.\n\n## 16.
