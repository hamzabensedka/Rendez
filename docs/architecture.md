Architecture Overview\n\nThis document describes the system architecture for the Planity Clone, a mobile-first platform connecting customers with beauty and wellness businesses. Technology Stack\n\n| Layer | Technology | Rationale |\n|-------|------------|-----------|\n| Mobile Framework | Expo (React Native) | Cross-platform, OTA updates, managed workflow |\n| Language | TypeScript | Type safety across frontend and backend |\n| Routing | Expo Router | File-based routing, deep linking support |\n| State Management | TanStack React Query | Server state caching, optimistic updates |\n| Animations | React Native Reanimated | 60fps UI animations, gesture handling |\n| Backend Framework | NestJS | Modular architecture, dependency injection |\n| ORM | Prisma | Type-safe database access, migrations |\n| Database | PostgreSQL + PostGIS | Relational data, geospatial queries |\n| Cache & Session | Redis | Session store, rate limiting, caching |\n| Monorepo | Nx + pnpm | Scalable monorepo, dependency management |\n| Containerization | Docker Compose | Local development consistency |\n| CI/CD | GitHub Actions | Automated testing, builds, deployments |\n| Backend Hosting | Supabase | Auth, storage, real-time subscriptions |\n| Mobile Builds | EAS Build | Cloud-based Expo builds |\n| Testing | Jest | Unit and integration testing |\n\n## 3. Service Boundaries\n\n### 4.1 Authentication Service\n- **Responsibility**: User registration, login, OAuth, token management, session handling\n- **Key Features**: JWT access/refresh tokens, OAuth 2.0 (Google, Apple), biometric authentication, account lockout, role-based access control\n- **Database Tables**: `users`, `sessions`, `oauth_providers`\n- **Redis**: Session store, rate limiting counters\n\n### 4.2 Business Service\n- **Responsibility**: Business profiles, locations, hours, images, amenities\n- **Key Features**: CRUD operations, geolocation queries (PostGIS), image management, verification badge\n- **Database Tables**: `businesses`, `business_hours`, `business_images`, `business_amenities`, `business_locations`\n\n### 4.3 Service Catalog Service\n- **Responsibility**: Service definitions, categories, pricing, duration\n- **Key Features**: Hierarchical categories, service grouping, pricing rules\n- **Database Tables**: `categories`, `services`, `service_categories`, `service_variants`\n\n### 4.4 Staff Service\n- **Responsibility**: Staff profiles, availability, skills\n- **Key Features**: Staff scheduling, skill mapping, availability slots\n- **Database Tables**: `staff`, `staff_availability`, `staff_skills`, `staff_schedules`\n\n### 4.5 Booking Service\n- **Responsibility**: Appointment creation, modification, cancellation, waitlist\n- **Key Features**: Slot management, conflict detection, booking rules, waitlist auto-notify\n- **Database Tables**: `bookings`, `booking_services`, `booking_history`, `waitlist`\n- **Redis**: Lock slots during booking flow, cache availability\n\n### 4.6 Search Service\n- **Responsibility**: Full-text search, autocomplete, filters, trending\n- **Key Features**: Full-text search with fuzzy matching, autocomplete, recent searches, trending queries\n- **Database**: PostgreSQL full-text search indexes, Redis for trending cache\n\n### 4.7 Review Service\n- **Responsibility**: Reviews, ratings, moderation\n- **Key Features**: Rating distribution, photo reviews, owner responses, moderation queue\n- **Database Tables**: `reviews`, `review_photos`, `review_reports`\n\n### 4.8 Payment Service\n- **Responsibility**: Payment processing, refunds, payout to businesses\n- **Key Features**: Stripe integration, payment methods, invoicing, payout scheduling\n- **Database Tables**: `payments`, `payment_methods`, `transactions`, `payouts`\n\n### 4.9 Notification Service\n- **Responsibility**: Push notifications, email, SMS\n- **Key Features**: Booking reminders, promotional notifications, transactional emails\n- **Database Tables**: `notifications`, `notification_templates`, `device_tokens`\n\n### 4.10 Admin Service\n- **Responsibility**: Platform administration, analytics, moderation\n- **Key Features**: User management, business approval, content moderation, analytics dashboard\n- **Database Tables**: `admin_actions`, `audit_logs`\n\n## 5. Data Flow\n\n### 5.1 Booking Flow\n1. User selects business → Mobile app fetches business details via TanStack Query (cached)\n2. User selects services → API validates availability\n3. User selects staff → API returns available slots (Redis cache for hot slots)\n4. User selects time → API locks slot temporarily (Redis lock, 10 min TTL)\n5. User confirms → Booking service creates booking in transaction\n6. Payment processed (if required) → Payment service\n7. Confirmation sent → Notification service\n8. Slot lock released, booking persisted\n\n### 5.2 Search Flow\n1. User types query → Debounced (300ms) API call\n2. API searches PostgreSQL full-text index\n3. Results enriched with distance (PostGIS), ratings, availability\n4. Response cached in Redis (5 min TTL for popular queries)\n5. Mobile app displays results with loading shimmer\n6. User filters → New API call with filter parameters\n\n## 6. Caching Strategy\n\n### 8.1 Redis Usage\n- **Session Store**: User sessions with TTL (7 days for refresh tokens)\n- **Rate Limiting**: Per-IP and per-user rate limit counters\n- **Slot Locking**: Temporary locks during booking flow (10 min TTL)\n- **Search Cache**: Popular search results (5 min TTL)\n- **Business Cache**: Business details (10 min TTL)\n- **Category Cache**: Category tree (1 hour TTL)\n\n### 8.2 TanStack React Query Caching\n- **Stale Time**: 5 minutes for business listings, 1 minute for availability\n- **Cache Time**: 30 minutes for static data (categories, business details)\n- **Optimistic Updates**: Booking creation, favorite toggling\n\n## 9. Security\n\n### 9.1 Authentication\n- JWT access tokens (15 min) + refresh tokens (7 days)\n- Refresh token rotation\n- OAuth 2.0 with PKCE for mobile\n- Biometric authentication stored in SecureStore\n\n### 9.2 Authorization\n- Role-based access control (customer, business_owner, admin)\n- Resource ownership checks\n- API rate limiting (100 req/min per user, 1000 req/min per IP)\n\n### 9.3 Data Protection\n- HTTPS everywhere\n- Input validation (class-validator in NestJS)\n- SQL injection prevention via Prisma parameterized queries\n- XSS protection via output encoding\n\n## 10.
