Overview\n\nThis document defines the complete system architecture for the Planity Clone, a cross-platform appointment booking marketplace. The architecture follows clean architecture principles, emphasizing separation of concerns, maintainability, and scalability. The system is built using a monorepo managed by Nx and pnpm, with a React Native (Expo) mobile client, a NestJS backend, and PostgreSQL with PostGIS for geospatial queries. Redis is used for caching and session management, and Supabase provides authentication and real-time capabilities.\n\n## 2. Technology Stack\n\n| Layer | Technology | Purpose |\n|---|---|---|\n| Mobile Client | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated | Cross-platform mobile app for consumers |\n| Backend API | NestJS, TypeScript, Prisma, PostgreSQL, PostGIS, Redis | RESTful API with geospatial support, caching, and background jobs |\n| Database | PostgreSQL + PostGIS | Relational data with geospatial queries |\n| Cache & Session | Redis | Session store, rate limiting, job queue |\n| Authentication | Supabase Auth (JWT) | OAuth, email/password, phone OTP |\n| Real-time | Supabase Realtime | Live availability updates, notifications |\n| CI/CD | GitHub Actions, EAS Build | Automated testing, building, and deployment |\n| Testing | Jest | Unit and integration tests |\n| Containerization | Docker Compose | Local development environment |\n\n## 3. Folder Structure\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   ├── components/            # Reusable UI components\n│   │   ├── hooks/                 # Custom hooks\n│   │   ├── services/              # API client, auth service\n│   │   ├── utils/                 # Helpers, constants\n│   │   ├── app.json\n│   │   └── package.json\n│   └── api/                       # NestJS backend\n│       ├── src/\n│       │   ├── modules/           # Feature modules (auth, business, booking, etc.)\n│       │   ├── common/            # Shared guards, interceptors, filters\n│       │   ├── config/            # Environment configuration\n│       │   ├── prisma/            # Prisma schema, migrations\n│       │   └── main.ts\n│       ├── test/\n│       └── package.json\n├── libs/\n│   ├── shared/                    # Shared types, DTOs, interfaces\n│   │   ├── types/                 # TypeScript interfaces\n│   │   ├── dto/                   # Data transfer objects\n│   │   └── constants/             # Shared constants\n│   └── utils/                     # Shared utility functions\n├── tools/                         # Nx generators, scripts\n├── docker-compose.yml\n├── nx.json\n├── pnpm-workspace.yaml\n├── package.json\n└── .github/\n    └── workflows/\n        ├── ci.yml\n        └── deploy.yml\n```\n\n## 4. Service Boundaries\n\nThe system is decomposed into the following bounded contexts (modules) within the NestJS backend:\n\n### 4.1 Authentication Module (`AuthModule`)\n- **Responsibility**: User registration, login, social auth, token management, password reset, MFA.\n- **Dependencies**: Supabase Auth client, Redis (blacklisted tokens).\n- **Exposed API**: `/auth/*` endpoints.\n\n### 4.2 User Module (`UserModule`)\n- **Responsibility**: User profile management, preferences, account deletion.\n- **Dependencies**: Prisma (PostgreSQL).\n- **Exposed API**: `/users/*` endpoints.\n\n### 4.3 Business Module (`BusinessModule`)\n- **Responsibility**: Business CRUD, staff management, service menu, hours, policies, image gallery.\n- **Dependencies**: Prisma (PostgreSQL), PostGIS (location queries), Redis (caching).\n- **Exposed API**: `/businesses/*` endpoints.\n\n### 4.4 Search Module (`SearchModule`)\n- **Responsibility**: Full-text search, filters, auto-complete, trending recommendations.\n- **Dependencies**: PostgreSQL full-text search, PostGIS (distance), Redis (cached results).\n- **Exposed API**: `/search/*` endpoints.\n\n### 4.5 Booking Module (`BookingModule`)\n- **Responsibility**: Appointment creation, cancellation, rescheduling, availability management.\n- **Dependencies**: Prisma, Redis (locking slots), Supabase Realtime (live updates).\n- **Exposed API**: `/bookings/*` endpoints.\n\n### 4.6 Review Module (`ReviewModule`)\n- **Responsibility**: Customer reviews, ratings, photo uploads.\n- **Dependencies**: Prisma, file storage (Supabase Storage).\n- **Exposed API**: `/reviews/*` endpoints.\n\n### 4.7 Notification Module (`NotificationModule`)\n- **Responsibility**: Push notifications, email, SMS reminders.\n- **Dependencies**: Redis (job queue), external providers (FCM, APNs, SendGrid, Twilio).\n- **Exposed API**: Internal event-driven.\n\n### 4.8 Admin Module (`AdminModule`)\n- **Responsibility**: Platform administration, analytics, user management, moderation.\n- **Dependencies**: Prisma, Redis (analytics cache).\n- **Exposed API**: `/admin/*` endpoints (web dashboard).\n\n## 5. Data Flow\n\n### 5.1 Authentication Flow\n1. User initiates login via email/password or social provider.\n2. Mobile app calls `POST /auth/login` with credentials.\n3. Backend validates credentials against Supabase Auth.\n4. On success, backend issues JWT access token (15 min) and refresh token (7 days).\n5. Refresh token is stored in Redis with user ID for revocation.\n6. Mobile app stores tokens securely (expo-secure-store).\n7. TanStack React Query interceptor attaches token to all requests.\n8. On 401, interceptor attempts token refresh via `POST /auth/refresh`.\n\n### 5.2 Search & Discovery Flow\n1. User enters search query or moves map.\n2. Mobile app sends request to `GET /search?q=...&lat=...&lng=...&radius=...`.\n3. Backend checks Redis cache for identical query; if hit, return cached results.\n4. If miss, execute PostgreSQL full-text search with PostGIS distance filter.\n5. Results are paginated, sorted, and returned with business summaries.\n6. Backend caches results in Redis with TTL of 5 minutes.\n7. Mobile app renders list/map using TanStack React Query with stale time of 2 minutes.\n\n### 5.3 Booking Flow\n1. User selects service and staff on Business Detail screen.\n2. Mobile app fetches available slots via `GET /businesses/:id/availability?date=...&staffId=...`.\n3. Backend queries PostgreSQL for existing bookings and calculates free slots.\n4. User selects a slot and confirms booking.\n5. Mobile app sends `POST /bookings` with service, staff, time, and notes.\n6. Backend uses Redis distributed lock on the slot to prevent double booking.\n7. Booking is created in PostgreSQL, lock released.\n8. Backend publishes event to Notification module (email/SMS/push confirmation).\n9. Supabase Realtime broadcasts update to business owner dashboard.\n10. Mobile app invalidates relevant TanStack React Query caches.\n\n## 6. Database Schema (Prisma)\n\nKey entities:\n\n- **User**: id, email, phone, passwordHash, authProvider, profile, createdAt, deletedAt (soft delete)\n- **Business**: id, name, description, address, location (PostGIS point), phone, website, hours (JSON), policies (JSON), ownerId, status, createdAt\n- **Service**: id, businessId, name, description, duration, price, categoryId, staffIds[]\n- **Staff**: id, businessId, name, bio, photoUrl, services[]\n- **Booking**: id, userId, businessId, staffId, serviceId, startTime, endTime, status, notes, createdAt\n- **Review**: id, userId, businessId, bookingId, rating, text, photos[], createdAt\n- **Category**: id, name, parentId (self-referencing for hierarchy)\n\nPostGIS: `location` field uses `geography(Point, 4326)` for accurate distance queries.\n\n## 7. Caching Strategy\n\n- **Redis**:\n  - Session tokens (refresh tokens) with TTL.\n  - Rate limiting counters (per IP, per user).\n  - Search results (keyed by query hash) with 5-minute TTL.\n  - Business details (keyed by business ID) with 1-minute TTL.\n  - Available slots (keyed by business+date+staff) with 30-second TTL.\n  - Distributed locks for booking slots.\n- **TanStack React Query**:\n  - Client-side caching with stale times (search: 2 min, business details: 5 min, availability: 30 sec).\n  - Optimistic updates for booking creation.\n\n## 8. Real-time Architecture\n\n- **Supabase Realtime** is used for:\n  - Broadcasting booking confirmations to business dashboard.\n  - Updating availability in real-time when a booking is made.\n- **WebSocket fallback**: If Supabase Realtime is unavailable, the system falls back to polling via TanStack React Query's refetchInterval.\n\n## 9. Security Considerations\n\n- **Authentication**: JWT with short-lived access tokens, refresh token rotation, blacklist on logout.\n- **Rate Limiting**: Redis-based rate limiting per endpoint (e.g., 5 login attempts per 15 minutes).\n- **Data Privacy**: GDPR compliance with soft delete and 30-day grace period for account deletion.\n- **Input Validation**: NestJS validation pipes with class-validator.\n- **SQL Injection**: Prevented by Prisma ORM parameterized queries.\n- **CORS**: Strict origin whitelist for API.\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for individual modules (services, controllers, guards).\n- **Integration Tests**: Test modules with real database (Docker Compose PostgreSQL).\n- **E2E Tests**: Supertest for API endpoints.\n- **Mobile Tests**: Jest with React Native Testing Library for components and hooks.\n\n## 11. CI/CD Pipeline (GitHub Actions)\n\n1. **CI**: On push to main/develop, run lint, unit tests, integration tests, build check.\n2. **CD**: On merge to main, build mobile app via EAS Build (iOS/Android), deploy backend to cloud (e.g., Railway or AWS ECS).\n\n## 12. Local Development Setup\n\n1. Clone repo, run `pnpm install`.\n2. Start Docker Compose: `docker-compose up -d` (PostgreSQL, Redis).\n3. Run Prisma migrations: `nx run api:prisma:migrate`.\n4. Start backend: `nx serve api`.\n5. Start mobile app: `nx run mobile:start`.\n\n## 13. Conclusion\n\nThis architecture provides a scalable, maintainable foundation for the Planity Clone. The clear separation into bounded contexts, combined with modern tooling (Nx, pnpm, TanStack Query, Prisma), ensures that the system can evolve with new features while remaining performant and secure.",
