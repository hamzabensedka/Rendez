Overview\nPlanity Clone is a monorepo‑based full‑stack application built with Expo (React Native) for the mobile client, NestJS for the backend API, and supporting services for background jobs, real‑time updates, and admin tooling. The architecture follows clean‑architecture principles: separation of concerns, domain‑driven boundaries, and dependency inversion. All code is typed with TypeScript and managed via Nx and pnpm.\n\n## 2. High‑Level Components\n\n| Component | Responsibility | Tech Stack |\n|-----------|----------------|------------|\n| **Mobile App** | UI, navigation, state management, offline caching, animations | Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated, React Native Vector Icons, Expo Secure Store |\n| **API Gateway** | HTTP entry point, authentication, request validation, routing to domain services | NestJS (Express adapter), TypeScript, Joi/Zod validation, Passport (JWT), Redis (rate‑limit/cache) |\n| **Domain Services** (Business, Booking, User, Review, Payment, Notification) | Encapsulate business logic, interact with Prisma ORM, publish domain events | NestJS modules, TypeScript, Prisma, PostgreSQL + PostGIS, Redis (caching/pub‑sub) |\n| **Worker Service** | Background job processing (emails, SMS, push notifications, slot recomputation) | NestJS BullMQ module, Redis, TypeScript |\n| **Admin Dashboard** | Internal UI for admins & business owners (CRUD, analytics) | Next.js (optional) or same NestJS serving static React build, TypeScript, TanStack Query |\n| **Infrastructure** | Container orchestration, CI/CD, observability | Docker Compose (local), GitHub Actions, EAS Build (mobile), Supabase (optional fallback), Prometheus/Grafana (metrics), Loki (logs) |\n\n## 3. Service Boundaries & Communication\n\n- **Mobile ↔ API**: REST/JSON over HTTPS; real‑time updates via Redis Pub/Sub transformed to WebSocket endpoints (NestJS `@WebSocketGateway`).\n- **API ↔ Domain Services**: In‑process NestJS modules; loose coupling via DTOs and interfaces; domain events emitted through NestJS `EventEmitter` (or Redis Pub/Sub for cross‑instance).\n- **Domain Services ↔ Data Layer**: Prisma Client (typed) interacting with PostgreSQL; PostGIS extension used for geo‑queries (nearby businesses).\n- **Worker ↔ Redis**: BullMQ queues store jobs; workers process and may publish results back to API via internal HTTP or Redis Pub/Sub.\n- **Admin Dashboard ↔ API**: Same API endpoints; protected by role‑based JWT claims (`admin`, `provider`).\n\n## 4. Communication Protocols\n\n- **REST**: JSON over HTTP; versioned via URL (`/api/v1/`).\n- **WebSocket**: NestJS `@WebSocketGateway` with JWT auth; used for live booking updates and chat‑like notifications.\n- **Redis Pub/Sub**: Decouples API workers (e.g., after a booking is created, publish `booking.created`; workers send push/email).\n- **Internal Calls**: NestJS modules call each other via dependency injection; no network overhead.\n\n## 7. Security\n\n- Authentication: JWT (access token 15 min, refresh token 7 days) stored in Expo Secure Store.\n- Passwords: bcrypt hash.\n- Role‑based access control (RBAC) middleware in NestJS (`@Roles('admin')`).\n- Input validation: DTOs with class‑validator/Zod.\n- Rate limiting: Redis‑based store in NestJS.\n- Data protection: TLS everywhere; sensitive fields encrypted at rest (pgcrypto).\n- CORS: restricted to allowed origins (expo dev, web domains).\n\n## 8. External Integrations\n\n- **Payment Gateway**: Stripe (or similar) via NestJS service; webhook endpoint verifies signatures.\n- **Push Notifications**: Expo Push API (worker service sends requests).\n- **Email/SMS**: Provider agnostic (SendGrid, Twilio) invoked by BullMQ workers.\n- **Maps**: Mapbox / Google Maps SDK in mobile for rendering; geocoding via same provider.\n- **Analytics**: Firebase Analytics / Segment (optional).\n\n## 9. DevOps & CI/CD\n\n- **Local Development**: `docker compose up` spins up PostgreSQL (with PostGIS), Redis, API, Worker.\n- **Testing**: Jest for unit/integration tests; React Native Testing Library for mobile UI.\n- **CI**: GitHub Actions runs lint (`eslint`), unit tests (`jest`), build (Nx `npm run build`), and Docker image lint.\n- **CD**: \n  - Mobile: EAS Build creates IPA/APK; auto‑submit to TestFlight/Play Store on tag push.\n  - API/Worker: Docker images pushed to registry; Kubernetes or ECS service rolls out via ArgoCD or similar.\n  - Admin: Static build deployed to Vercel/Netlify or served by API.\n- **Observability**: Prometheus metrics exposed by NestJS (`@nestjs/microservices`); Grafana dashboards; Loki for log aggregation.\n\n## 10. Scalability & Performance Considerations\n\n- **Read‑heavy search**: Redis cache for frequent queries (business list, categories). Cache invalidation via `business.updated` event.\n- **Geo‑search**: PostGIS `ST_DWithin` with index on `location`; fallback to Redis GEO for simple radius.\n- **Booking slot computation**: Pre‑compute slots nightly via worker; real‑time adjustments on booking/create/cancel.\n- **Horizontal scaling**: API and worker are stateless; scale via replica count.\n- **Database**: Use read replicas for heavy reporting; connection pooling via Prisma.\n\n## 11. Shared Types & Design System\n\n- A `libs/types` package contains generated TypeScript interfaces from Prisma schema (via `prisma-typescript` generator) and OpenAPI spec (from NestJS `@nestjs/swagger`).\n- `libs/design-system` provides reusable RN components (Button, Card, Input, Avatar) styled with Tailwind/NativeWind, ensuring UI consistency.\n\n## 12. Testing Strategy\n\n- **Unit**: Jest tests for services, utilities, reducers.\n- **Integration**: API endpoint tests with supertest; mobile navigation and query mocking with React Native Testing Library.\n- **E2E**: Detox for critical flows (login → search → book → cancel).\n- **Contract**: Pact or schema‑driven testing to ensure mobile ↔ API compatibility.\n\n## 13. Summary\nThe architecture cleanly separates mobile UI, NestJS API (domain‑driven), background workers, and admin tooling, leveraging a monorepo with Nx and pnpm for code sharing, Prisma + PostGIS for data, Redis for caching and job queuing, and Docker/GitHub Actions/EAS for reliable CI/CD.\n\n",
