Communication between modules occurs via well‑defined interfaces (DTOs) or events published to BullMQ.\\n\\n- **Auth** – handles user identity, issues JWT tokens, integrates with Supabase for social logins.\\n- **Business** – manages business profiles, categories, location (PostGIS point), operating hours.\\n- **Booking** – computes available slots using business hours + existing bookings, creates appointments, enforces business rules.\\n- **Payment** – creates payment intents, confirms transactions, emits `payment.completed` event.\\n- **Notification** – listens to events (booking.created, payment.completed, appointment.reminder) and sends emails/SMS/push via third‑party providers; uses BullMQ for retry.\\n- **Admin** – privileged APIs for managing users, businesses, system settings.\\n- **Provider** – business‑owner facing APIs: manage own business details, view/bookings, set schedule, see analytics.\\n\\nData flow example (booking flow):\\n1. Payment module listens to `appointment.created` (or separate step) → creates payment intent, returns client secret to mobile.\\n5.
