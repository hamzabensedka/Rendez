`BusinessesController`, `BusinessesService`.\n  - `StaffController`, `StaffService`.\n- **Data**: Businesses, staff, business_hours, business_photos, amenities tables.\n- **Spatial**: PostGIS for location queries (point geometry).\n\n#### 3.1.4 Services Module\n- **Responsibilities**: Service catalog, categories, add-ons, pricing.\n- **Key Components**:\n  - `ServicesController`, `ServicesService`.\n  - `CategoriesController`, `CategoriesService`.\n- **Data**: Services, service_categories, service_addons, service_prices tables.\n\n#### 3.1.5 Bookings Module\n- **Responsibilities**: Booking creation, slot management, calendar, waitlist, reminders.\n- **Key Components**:\n  - `SearchController`, `SearchService`.\n  - `ElasticsearchService` (optional, can use PostgreSQL full-text search with PostGIS).\n- **Data**: Uses PostgreSQL full-text search indexes and PostGIS spatial queries.\n- **Redis**: Cache search results and autocomplete suggestions.\n\n#### 3.1.7 Reviews Module\n- **Responsibilities**: Review CRUD, rating aggregation, moderation.\n- **Key Components**:\n  - Data Flow & Communication\n\n### 4.1 Client-Server Communication\n\n```\nMobile App (Expo) <--HTTPS/WebSocket--> NestJS API <--Prisma--> PostgreSQL\n                                              |\n                                              v\n                                            Redis\n```\n\n- **REST API**: Primary communication for CRUD operations.\n- **WebSocket**: Real-time updates for booking confirmations, notifications.\n- **GraphQL**: Optional for complex queries (search, business details).\n\n### 4.2 Authentication Flow\n\n1. User logs in via email/password or OAuth.\n2. Server validates credentials, returns JWT access token (15 min) and refresh token (30 days).\n3. Mobile app stores tokens securely (expo-secure-store).\n4. Axios interceptor refreshes token automatically when 401 is received.\n5. Refresh token rotation: old refresh token is invalidated on each refresh.\n\n### 4.3 Booking Flow\n\n1. Customer selects service, staff, date, time.\n2. Client sends `POST /bookings/hold` to lock the slot (10-minute hold in Redis).\n3. Customer proceeds to payment (or selects \"pay at venue\").\n4. On successful payment, `POST /bookings/confirm` creates the booking in PostgreSQL.\n5. Slot hold is released and marked as booked.\n6. If payment fails or timeout, slot is released automatically.\n7. Confirmation is sent via push notification and email.\n\n### 4.4 Search Flow\n\n1. User types query in search bar (debounced 300ms).\n2. Client calls `GET /search/autocomplete?q=...`.\n3. Server returns suggestions from Redis cache (if available) or PostgreSQL.\n4. User selects a suggestion or submits full query.\n5. Client calls `GET /search?q=...&lat=...&lng=...&radius=...&filters=...`.\n6. Server performs full-text search with PostGIS spatial filter.\n7. Results are paginated (20 per page) and cached in Redis for 5 minutes.\n8. Client renders list or map view.\n\n## 5.
