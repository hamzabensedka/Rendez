  "content": "# System Architecture\n\n## Overview\nThis document describes the complete system architecture for the Planity Clone, a mobile app that connects users with local businesses for discovery, booking, and management of appointments. The architecture follows clean‑architecture principles, separates concerns into distinct services, and leverages a monorepo managed by Nx and pnpm for scalable development.\n\n## Tech Stack\n- **Mobile**: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **API**: NestJS (Node.js), Prisma ORM, PostgreSQL + PostGIS, Redis, BullMQ (background jobs)\n- **Infrastructure**: Docker Compose, GitHub Actions, EAS Build, Supabase (optional storage/realtime), Nx monorepo, pnpm\n- **Testing**: Jest\n\n## High‑Level Components\n1. **Mobile Client** – Expo app handling UI, navigation, state, and animations.\n2. **API Gateway** – NestJS server exposing RESTful endpoints; acts as entry point for all client requests.\n3. **Core Services** (modules within the NestJS app):\n   - Auth Service\n   - Business Service\n   - Booking Service\n   - Payment Service\n   - Notification Service\n   - Provider Portal (business‑owner module)\n   - Admin Dashboard (admin module)\n4. **Data Layer** – PostgreSQL with PostGIS for geospatial queries; Prisma as ORM.\n5. **Caching & Job Queue** – Redis for caching (session, search results) and BullMQ-backed background jobs.\n6. **DevOps** – Docker Compose for local dev; GitHub Actions for CI; EAS Build for Expo binaries.\n\n## Monorepo Folder Structure\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/            # Expo React Native app\n│   └── api/               # NestJS server\n├── libs/\n│   ├── shared-types/      # TypeScript interfaces, enums, DTOs shared between mobile and api\n│   ├── ui-kit/            # Design system components (buttons, inputs, themes)\n│   ├── utils/             # Logging, validation helpers, constants\n│   └── test-utils/        # Custom Jest matchers, test factories\n├── tools/\n│   └── scripts/           # DB migration, seed, dev utilities\n├── docker-compose.yml\n├── nx.json\n├── package.json (pnpm workspace root)\n└── ...\n```\n\n## Service Boundaries\n\n### Auth Service\n- Responsibilities: user registration, login, JWT issuance, password reset, token refresh.\n- Exposes: `/auth/register`, `/auth/login`, `/auth/refresh`, `/auth/reset-password`.\n- Uses Prisma `User` model; stores hashed passwords (bcrypt).\n\n### Business Service\n- Responsibilities: CRUD for businesses, service categories, geospatial search, map markers.\n- Exposes: `/businesses` (list, create), `/businesses/:id`, `/businesses/search`, `/businesses/nearby`.\n- Leverages PostGIS `ST_DWithin` for location‑based queries.\n- Caches frequent search results in Redis (TTL 5 min).\n\n### Booking Service\n- Responsibilities: slot computation, appointment creation, modification, cancellation, availability updates.\n- Exposes: `/appointments` (create, list, update, cancel), `/slots/:businessId/:date`.\n- Slot computation algorithm runs as a BullMQ job when a business updates its schedule or after each booking to keep availability fresh.\n- Uses PostgreSQL transactions to avoid double‑booking.\n\n### Payment Service\n- Responsibilities: integration with a payment gateway (e.g., Stripe), payment confirmation, receipt generation.\n- Exposes: `/payments/create-intent`, `/payments/webhook` (gateway callback).\n- Stores minimal payment reference; sensitive data handled by gateway.\n\n### Notification Service\n- Responsibilities: sending push notifications (via Expo push service) and emails (via Supabase/SMTP) for booking confirmations, reminders, cancellations.\n- Triggered by BullMQ jobs scheduled based on appointment time.\n- Exposes internal API for other services to enqueue notification jobs.\n\n### Provider Portal\n- Responsibilities: business‑owner authentication, managing business profile, services, working hours, viewing incoming bookings.\n- Reuses Auth Service; additional endpoints under `/provider/*`.\n- Provides a separate UI (could be a web dashboard built with NestJS + templating or a separate Expo app; for simplicity we expose API only).\n\n### Admin Dashboard\n- Responsibilities: admin authentication, overseeing users, businesses, payments, system settings.\n- Endpoints under `/admin/*` with role‑based access control (RBAC).\n\n### Shared Library\n- Contains TypeScript interfaces (User, Business, Appointment, etc.) used by both mobile and API to guarantee contract consistency.\n- UI kit provides styled components adhering to the design system.\n- Utils include logging, error formatting, and validation helpers.\n\n## Data Models (Prisma Schema Highlights)\n```prisma\nmodel User Errors follow a uniform `{ error: string, details?: any }` format.\n- Authentication via JWT in `Authorization: Bearer <token>` header; role‑based guards protect routes.\n- Geospatial queries use raw Prisma `$queryRaw` with PostGIS functions.\n\n## Mobile Architecture\n- **Routing**: Expo Router provides file‑based navigation (tabs, stacks).\n- **State Management**: TanStack React Query handles server state (caching, background updates, optimistic updates).\n- **Animations**: React Native Reanimated 2 for smooth transitions, gesture‑based interactions.\n- **Design System**: Shared `ui-kit` library ensures consistent typography, spacing, theming (light/dark).\n- **Feature‑Folders**: Each feature (auth, home, search, booking, profile) lives under `src/features/<feature>` containing screens, components, hooks, and types.\n- **Dependency Injection**: React Context provides lightweight services (auth, query client, navigation).\n- **Offline**: Query client enables stale‑while‑revalidate caching; critical actions are optimistic updates via WebSocket (optional via Supabase Realtime).\n\n## DevOps & CI/CD\n- **Local Development**: `docker compose up` spins up API, PostgreSQL, Redis, and BullMQ worker.\n- **Testing**: Jest runs unit and integration tests; GitHub Actions workflow runs on every PR (lint, test, build).\n- **Build**: EAS Build creates iOS/Android binaries; version controlled via `eas.json`.\n- **Deployment**: API deployed as Docker container to a cloud provider (e.g., AWS ECS, Fly.io) with rolling updates; DB migrations run via Prisma Migrate in release pipeline.\n- **Monitoring**: Basic health checks (`/health`), logging to stdout; production logging via external service (e.g., Datadog) can be added.\n\n## Security Considerations\n- Passwords hashed with bcrypt (salt rounds 12).\n- JWTs signed with strong secret; short-lived access tokens (15 min) + refresh tokens stored HTTP‑only via secure cookie (if using web) or secure storage on mobile.\n- Rate limiting on auth endpoints (NestJS `@Throttle`).\n- Input validation and sanitization to prevent injection.\n- HTTPS enforced in production.\n- CORS restricted to allowed origins (Expo dev, production domains).\n\n## Testing Strategy\n- **Unit Tests**: Jest tests for service methods, utility functions, DTO validation.\n- **Integration Tests**: API endpoints tested with supertest against a temporary PostgreSQL instance (using Docker in CI).\n- **Mobile Tests**: Jest with React Native Testing Library for component logic; Detox could be added later for E2E.\n- **Test Coverage**: Aim >80% for core business logic.\n\n## Summary\nThe architecture separates concerns into focused services within a NestJS API, leverages a shared TypeScript library and design system for the Expo mobile client, uses PostgreSQL with PostGIS for geospatial business search, Redis for caching and background jobs, and is orchestrated via Nx monorepo with pnpm, Docker Compose, GitHub Actions, and EAS Build for reliable, scalable development and delivery.",
