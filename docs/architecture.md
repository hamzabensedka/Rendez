  "content": "# Planity Clone — System Architecture Document\n\n## 1. Overview\n\nThis document describes the complete system architecture for the Planity Clone, a mobile-first marketplace connecting customers with local service businesses. The architecture is designed for scalability, maintainability, and clear separation of concerns, using a monorepo structure with Nx and pnpm.\n\n## 2. Technology Stack\n\n| Layer | Technology | Purpose |\n|-------|------------|---------|\n| Mobile App | Expo, React Native, TypeScript, Expo Router | Cross-platform mobile and web client |\n| State & Data | TanStack React Query, React Native Reanimated | Server state management, animations |\n| Backend | NestJS, TypeScript | RESTful API server |\n| Database | PostgreSQL, PostGIS | Relational data, geospatial queries |\n| ORM | Prisma | Type-safe database access |\n| Cache | Redis | Session store, rate limiting, job queues |\n| CI/CD | GitHub Actions, EAS Build | Automated testing and deployment |\n| Testing | Jest | Unit and integration tests |\n| Containerization | Docker Compose | Local development environment |\n| Auth | Supabase | Authentication, JWT management |\n\n## 3. Folder Structure\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/                    # Expo React Native app\n│   │   ├── app/                   # Expo Router pages\n│   │   │   ├── (auth)/            # Auth screens (login, register, forgot-password)\n│   │   │   ├── (tabs)/            # Main tab navigator (home, search, bookings, profile)\n│   │   │   ├── business/          # Business detail, staff, services\n│   │   │   ├── booking/           # Booking flow screens\n│   │   │   └── _layout.tsx        # Root layout\n│   │   ├── components/            # Shared UI components\n│   │   ├── hooks/                 # Custom React hooks\n│   │   ├── services/              # API client, auth service\n│   │   ├── utils/                 # Utility functions\n│   │   └── types/                 # TypeScript types\n│   └── api/                       # NestJS backend\n│       ├── src/\n│       │   ├── modules/           # Feature modules\n│       │   │   ├── auth/\n│       │   │   ├── users/\n│       │   │   ├── businesses/\n│       │   │   ├── services/\n│       │   │   ├── bookings/\n│       │   │   ├── reviews/\n│       │   │   ├── favorites/\n│       │   │   ├── search/\n│       │   │   ├── notifications/\n│       │   │   └── payments/\n│       │   ├── common/            # Shared utilities, guards, interceptors\n│       │   ├── config/            # Environment configuration\n│       │   └── main.ts\n│       └── prisma/\n│           ├── schema.prisma\n│           └── migrations/\n├── libs/\n│   ├── shared/                    # Shared types, constants, utilities\n│   └── api-client/                # Generated API client for mobile\n├── tools/                         # Nx generators, scripts\n├── docker-compose.yml\n├── nx.json\n├── package.json\n├── pnpm-workspace.yaml\n└── tsconfig.base.json\n```\n\n## 4. Service Boundaries\n\n### 4.1 Auth Service\n- **Responsibility**: User registration, login, token management, password recovery, biometric authentication.\n- **Endpoints**:\n  - `POST /auth/register`\n  - `POST /auth/login`\n  - `POST /auth/refresh`\n  - `POST /auth/forgot-password`\n  - `POST /auth/reset-password`\n- **Data**: Users table, refresh tokens in Redis.\n- **Dependencies**: Supabase for OAuth, Redis for rate limiting.\n\n### 4.2 Business Service\n- **Responsibility**: CRUD for businesses, categories, staff profiles, services, hours.\n- **Endpoints**:\n  - `GET /businesses` (search with filters)\n  - `GET /businesses/:id`\n  - `POST /businesses` (owner only)\n  - `PUT /businesses/:id`\n  - `GET /businesses/:id/staff`\n  - `GET /businesses/:id/services`\n- **Data**: Businesses, Staff, Services, Categories tables.\n- **Geospatial**: PostGIS for distance-based queries.\n\n### 4.3 Search Service\n- **Responsibility**: Full-text search, autocomplete, filtering, sorting.\n- **Endpoints**:\n  - `GET /search` (text, filters, pagination)\n  - `GET /search/suggestions`\n  - `GET /search/map` (bounding box)\n- **Data**: Search index in PostgreSQL (tsvector), Redis for caching.\n\n### 4.4 Booking Service\n- **Responsibility**: Appointment creation, modification, cancellation, availability management.\n- **Endpoints**:\n  - `GET /bookings/availability` (slots for business/staff/date)\n  - `POST /bookings` (create with idempotency key)\n  - `PUT /bookings/:id` (reschedule)\n  - `DELETE /bookings/:id` (cancel)\n  - `GET /bookings` (user's bookings)\n- **Data**: Bookings table, Redis for slot locks (30s TTL).\n- **Real-time**: WebSocket for slot updates.\n\n### 4.5 Payment Service\n- **Responsibility**: Payment processing, refunds, commission calculation.\n- **Endpoints**:\n  - `POST /payments/intent` (create payment intent)\n  - `POST /payments/confirm` (confirm payment)\n  - `POST /payments/refund`\n- **Data**: Payments, Transactions tables.\n- **Integration**: Stripe API.\n\n### 4.6 Notification Service\n- **Responsibility**: Push notifications, email, in-app notifications.\n- **Endpoints**:\n  - `POST /notifications/send` (admin)\n  - `GET /notifications` (user's notifications)\n- **Data**: Notifications table.\n- **Queue**: Redis Bull queue for async delivery.\n\n### 4.7 Review Service\n- **Responsibility**: Review CRUD, star distribution, moderation.\n- **Endpoints**:\n  - `GET /businesses/:id/reviews`\n  - `POST /reviews` (authenticated)\n  - `PUT /reviews/:id`\n- **Data**: Reviews table.\n\n### 4.8 Favorite Service\n- **Responsibility**: Manage user favorites, sync across devices.\n- **Endpoints**:\n  - `GET /favorites`\n  - `POST /favorites` (add)\n  - `DELETE /favorites/:id` (remove)\n- **Data**: Favorites table.\n\n## 5.   @relation(fields: [staffId], references: [id])\n}\n\nmodel Review {\n  id         String   @id @default(cuid())\n  userId     String\n  businessId String\n  rating     Int      // 1-5\n  comment    String?\n  photos     String[]\n  createdAt  DateTime @default(now())\n  user       User     @relation(fields: [userId], references: [id])\n  business   Business @relation(fields: [businessId], references: [id])\n}\n\nmodel Favorite {\n  id         String   @id @default(cuid())\n  userId     String\n  businessId String\n  createdAt  DateTime @default(now())\n  user       User     @relation(fields: [userId], references: [id])\n  business   Business @relation(fields: [businessId], references: [id])\n  @@unique([userId, businessId])\n}\n\nenum UserRole { CUSTOMER BUSINESS_OWNER ADMIN }\n\nenum UserStatus { ACTIVE SUSPENDED PENDING_VERIFICATION }\n\nenum BookingStatus { PENDING CONFIRMED CANCELLED COMPLETED NO_SHOW }\n```\n\n## 6. API Design\n\n### 6.1 General Principles\n- RESTful with JSON responses.\n- Versioning via URL prefix: `/api/v1/...`.\n- Pagination: cursor-based for lists (20 items per page).\n- Error format: `{ \"error\": { \"code\": \"VALIDATION_ERROR\", \"message\": \"...\", \"details\": [] } }`.\n- Authentication: Bearer JWT token in `Authorization` header.\n- Idempotency: `Idempotency-Key` header for booking creation.\n\n### 6.2 Key Endpoints\n\n#### Auth\n- `POST /api/v1/auth/register` — Register new user.\n- `POST /api/v1/auth/login` — Login, returns access + refresh tokens.\n- `POST /api/v1/auth/refresh` — Refresh access token.\n- `POST /api/v1/auth/forgot-password` — Send OTP.\n- `POST /api/v1/auth/reset-password` — Reset password with OTP.\n\n#### Businesses\n- `GET /api/v1/businesses?q=&category=&lat=&lng=&radius=&page=&sort=` — Search businesses.\n- `GET /api/v1/businesses/:id` — Business detail.\n- `GET /api/v1/businesses/:id/staff` — Staff list.\n- `GET /api/v1/businesses/:id/services` — Service list.\n- `GET /api/v1/businesses/:id/reviews` — Reviews.\n\n#### Bookings\n- `GET /api/v1/bookings/availability?businessId=&staffId=&date=` — Get available slots.\n- `POST /api/v1/bookings` — Create booking (requires idempotency key).\n- `GET /api/v1/bookings` — List user's bookings.\n- `PUT /api/v1/bookings/:id` — Reschedule.\n- `DELETE /api/v1/bookings/:id` — Cancel.\n\n#### Favorites\n- `GET /api/v1/favorites` — List favorites.\n- `POST /api/v1/favorites` — Add favorite.\n- `DELETE /api/v1/favorites/:businessId` — Remove.\n\n## 7. Authentication & Authorization\n\n### 7.1 Flow\n1. User registers/logs in via Supabase Auth (email/password or OAuth).\n2. Backend issues JWT access token (15 min expiry) and refresh token (30 days).\n3. Mobile app stores tokens securely (expo-secure-store).\n4. Access token sent in every API request.\n5. On 401, app automatically refreshes token using refresh token.\n6. Biometric authentication: after first successful password login, app prompts to enable biometrics. On subsequent launches, biometric check unlocks the app and refreshes tokens if needed.\n\n### 7.2 Authorization\n- Role-based access: `CUSTOMER`, `BUSINESS_OWNER`, `ADMIN`.\n- Guards in NestJS: `@UseGuards(AuthGuard)` and `@Roles('BUSINESS_OWNER')`.\n- Business owners can only modify their own businesses.\n\n## 8. Caching Strategy\n\n| Data | Cache | TTL | Invalidation |\n|------|-------|-----|--------------|\n| Business search results | Redis | 5 min | On business update |\n| Business detail | Redis | 10 min | On business update |\n| Available slots | Redis | 30s | On booking creation |\n| User session | Redis | 30 days | On logout |\n| Rate limiting counters | Redis | 1 min | Automatic expiry |\n\n## 9. Real-time Updates\n\n- **WebSocket**: Used for slot availability updates when another user books.\n- **Push Notifications**: Via Expo Push API for booking confirmations, reminders, cancellations.\n- **In-app Notifications**: Polled every 60 seconds or via WebSocket for immediate updates.\n\n## 10. Search Implementation\n\n- **Full-text search**: PostgreSQL `tsvector` on business name, service name, staff name.\n- **Geospatial**: PostGIS `ST_DWithin` for distance queries, `ST_AsGeoJSON` for map markers.\n- **Autocomplete**: Redis sorted sets for trending searches, PostgreSQL `ILIKE` for recent.\n- **Caching**: Redis cache for frequent queries with TTL.\n\n## 11. Booking Concurrency\n\n1. When user selects a slot, backend creates a temporary lock in Redis (30s TTL).\n2. Lock contains businessId, staffId, startTime, endTime.\n3. On payment confirmation, lock is released and booking is created.\n4. If lock expires, slot becomes available again.\n5. Double-booking prevention: unique constraint on (businessId, staffId, startTime) in PostgreSQL.\n\n## 12. Error Handling\n\n- **Client errors (4xx)**: Return specific error codes (e.g., `SLOT_UNAVAILABLE`, `INVALID_TOKEN`).\n- **Server errors (5xx)**: Log full error, return generic message.\n- **Network errors**: Mobile app retries with exponential backoff (TanStack React Query).\n- **Idempotency**: Duplicate booking requests with same idempotency key return existing booking.\n\n## 13. Testing Strategy\n\n- **Unit tests**: Jest for individual functions and services.\n- **Integration tests**: NestJS testing module with test database.\n- **E2E tests**: Detox for mobile, Playwright for web.\n- **CI/CD**: GitHub Actions runs tests on every PR, deploys to staging on merge to main.\n\n## 14. Deployment\n\n- **Backend**: Docker container deployed to cloud (AWS ECS or similar).\n- **Database**: Managed PostgreSQL with PostGIS extension.\n- **Mobile**: EAS Build for iOS and Android, OTA updates via EAS Update.\n- **Web**: Expo web build deployed to Vercel.\n- **CI/CD Pipeline**:\n  1. Lint and type-check\n  2. Run unit and integration tests\n  3. Build Docker images\n  4. Push to container registry\n  5. Deploy to staging\n  6. Run E2E tests\n  7. Deploy to production\n\n## 15. Monitoring & Logging\n\n- **Logging**: Structured JSON logs via NestJS Logger.\n- **Metrics**: Request rate, latency, error rate (Prometheus + Grafana).\n- **Alerts**: On error rate > 1% or p95 latency > 2s.\n- **Mobile**: Crash reporting via Sentry.\n\n## 16. Security Considerations\n\n- **Rate limiting**: 5 requests per minute for auth endpoints per IP.\n- **Password hashing**: bcrypt with 12 rounds.\n- **SQL injection**: Prevented by Prisma parameterized queries.\n- **CORS**: Whitelist mobile app origins.\n- **HTTPS**: Enforced at load balancer.\n- **Data privacy**: GDPR compliance, user data export and deletion endpoints.\n\n## 17. Performance Targets\n\n- **API response time**: < 200ms for 95th percentile (excluding search).\n- **Search response**: < 500ms for cached queries.\n- **Mobile app cold start**: < 3s.\n- **Booking flow**: < 5 taps after service selection.\n\n## 18. Development Setup\n\n1. Clone repo and run `pnpm install`.\n2. Start Docker Compose: `docker-compose up -d` (PostgreSQL, Redis).\n3. Run migrations: `pnpm nx run api:prisma:migrate`.\n4. Seed data: `pnpm nx run api:prisma:seed`.\n5. Start backend: `pnpm nx run api:serve`.\n6. Start mobile app: `pnpm nx run mobile:start`.\n\n## 19. Conclusion\n\nThis architecture provides a solid foundation for building the Planity Clone with clear separation of concerns, scalability, and maintainability.
