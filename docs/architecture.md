  "content": "# Planity Clone - System Architecture\n\n## 1. Overview\nPlanity Clone is a mobile-first salon and spa booking platform. The system is built as an Nx pnpm monorepo with an Expo React Native mobile client and a NestJS backend API. Data is stored in PostgreSQL with PostGIS for geospatial queries, and Redis powers caching and BullMQ background jobs. Supabase provides managed Postgres/PostGIS and media storage. CI/CD uses GitHub Actions, EAS Build for mobile, and Docker Compose for local infra.\n\n## 2. Service Boundaries (NestJS Modules)\n- **Auth**: Registration, login, JWT issuance, social (Google/Apple), password reset. Depends on Users.\n- **Users**: Profile, addresses, payment method refs, notification prefs. RBAC (customer/provider/admin).\n- **Businesses**: CRUD, geolocation (PostGIS), hours, approval status. Uses Supabase Storage for photos.\n- **Categories**: Seeded taxonomy, sub-categories. Admin-managed.\n- **Services**: Offered services per business, linked to category, price, duration.\n- **Staff**: Business employees, shift patterns.\n- **Availability**: Computes slots from business hours, staff shifts, service duration, existing bookings, timezone. Caches in Redis.\n- **Bookings**: Multi-step booking, status lifecycle, reschedule/cancel. Calls Payments, Notifications, Availability.\n- **Payments**: Stripe/PayPal integration, deposits, webhooks, refunds. PCI via tokenization.\n- **Reviews**: Ratings, comments, owner responses.\n- **Favorites**: User-business links, synced via API.\n- **Notifications**: Email/SMS/push triggers, opt-out, i18n. Enqueues jobs.\n- **Jobs**: BullMQ workers for notifications, reminders, slot warm-up. Idempotent.\n- **Analytics**: Provider revenue/utilization metrics.\n\n## 5. Mobile Architecture\n- **Expo Router** file-based routing; route groups for auth, tabs, provider, admin.\n- **TanStack React Query** manages server state, pagination, mutations; used via `api-client` package.\n- **React Native Reanimated** for map preview cards, screen transitions, micro-interactions.\n- **Auth** token stored in SecureStore; attached by api-client interceptor.\n- **Map** uses react-native-maps; viewport changes debounced and sent to `/businesses/search?bounds=...`.\n- **Design System** from `@planity/ui` ensures consistent components.\n- **Offline** via React Query persistence (optional).\n\n## 6. Data Model (Prisma + PostGIS)\nKey models: `User`, `Business` (location `PostGIS Point`), `Category`, `Service`, `Staff`, `Booking`, `Review`, `Favorite`, `Payment`, `Notification`. Prisma schema connects to Supabase Postgres with `postgis` extension. Spatial queries (distance sort, viewport) use raw SQL or `@prisma/client` extensions.\n\n## 7. Infrastructure & DevOps\n- **Docker Compose**: local `postgres` (with postgis) and `redis`.\n- **Supabase**: production Postgres+PostGIS and Storage buckets for images.\n- **Nx**: task orchestration, affected builds/tests.\n- **pnpm**: workspace package management.\n- **GitHub Actions**: on PR run lint, typecheck, Jest unit tests, build API, and trigger EAS Build for mobile preview.\n- **EAS Build**: managed build profiles for iOS/Android.\n- **Jest**: unit tests in packages and apps; NestJS e2e with test DB.\n\n## 8. Cross-Cutting Concerns\n- **Security**: JWT auth guard, role decorators, input validation pipes.\n- **i18n**: notification localization, client strings.\n- **Idempotency**: payment intents and job handlers keyed by id.\n- **Logging**: structured logs in NestJS; Sentry optional.\n\n## 9. Booking Flow (Sequence)\n1. Customer selects service & staff on business detail.\n2. Mobile queries `availability` for slots (Redis-cached).\n3. Customer confirms slot -> `bookings.create` mutation.\n4. API checks real-time slot, creates pending booking, calls `payments` for charge.\n5. On success, booking confirmed; `notifications` enqueues confirmation email/push.\n6. Background job sends reminder before appointment.\n\n## 10.
