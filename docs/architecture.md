Service Boundaries\n\n### 5.1 Mobile App (Expo)\n- **Responsibilities**: UI rendering, navigation, local state, optimistic updates, push notification handling, deep linking.\n- **Key Libraries**: Expo Router (file-based routing), TanStack React Query (server state, caching, pagination), React Native Reanimated (animations), React Native Maps (map view).\n- **State Management**: React Query for server state; Zustand for complex client state (e.g., booking wizard steps).\n- **Authentication**: Supabase client SDK handles sign-up/in, token refresh, and session persistence. API calls attach the access token.\n\n### 5.2 API Gateway (NestJS)\n- **Responsibilities**: Business logic, data validation, authorization, integration with database and external services.\n- **Modules**:\n  - **Auth Module**: Validates Supabase JWT, manages user roles, issues application-specific claims if needed.\n  - **User Module**: Profile CRUD, avatar upload (delegates to Supabase Storage), notification preferences.\n  - **Business Module**: Business CRUD, services, staff, working hours, gallery.\n  - **Booking Module**: Slot computation, booking creation, reschedule/cancel, slot locking with Redis.\n  - **Search Module**: Full-text search (PostgreSQL tsvector), geospatial queries (PostGIS), autocomplete.\n  - **Payment Module**: Integration with Stripe (or similar) for payment processing, saved payment methods.\n  - **Notification Module**: Push notifications (Expo Push), email (SendGrid), SMS (Twilio) – triggered by events.\n  - **Admin Module**: Category management, business approval, platform analytics.\n- **Database Access**: Prisma ORM with a single PostgreSQL instance. PostGIS for location queries.\n- **Caching**: Redis for frequently accessed data (business details, categories) and slot locks.\n\n### 5.3 Admin Dashboard\n- **Responsibilities**: Web interface for administrators to manage platform content.\n- **Tech**: React with Vite, TanStack React Query for API calls, shared types from `libs/shared-types`.\n- **Authentication**: Same Supabase auth flow, restricted to admin role.\n\n### 5.4 Database & Caching\n- **PostgreSQL + PostGIS**: Stores all relational data. PostGIS enables `ST_DWithin` for distance-based search and geolocation indexing.\n- **Redis**:\n  - **Cache**: Key-value store with TTL for business details, categories, search results.\n  - **Slot Locking**: Temporary locks on time slots during booking (5-minute TTL).\n  - **Session Store**: For rate limiting and token blacklisting.\n\n### 5.5 External Services\n- **Supabase**:\n  - **Auth**: Email/password, Google, Apple, magic link. User metadata stores role (`customer` or `provider`).\n  - **Storage**: Avatars, business images, service photos.\n  - **Real-time**: (Optional) For live availability updates, but not critical for MVP.\n- **Stripe**: Payment processing, saved cards.\n- **Expo Push Notifications**: For mobile push.\n- **SendGrid/Twilio**: Email and SMS notifications.\n\n## 6. Data Flow & Key Interactions\n\n### 6.1 Authentication Flow\n1. User signs up/in via Supabase client in the mobile app.\n2. Supabase returns JWT access/refresh tokens.\n3. On first login, user selects role; the app calls `POST /api/users/role` to store role in our database (synced with Supabase user metadata).\n4. All subsequent API requests include `Authorization: Bearer <access_token>`.\n5. NestJS `AuthGuard` verifies the JWT using Supabase's JWKS endpoint and extracts user ID and role.\n6. Refresh token rotation is handled transparently by the Supabase client.\n\n### 6.2 Booking Flow with Slot Locking\n1. Customer selects service, staff (optional), and date.\n2. App requests available slots: `GET /api/businesses/:id/slots?serviceId=...&date=...&staffId=...`.\n3. Backend computes slots based on:\n   - Provider working hours (with breaks).\n   - Service duration.\n   - Existing bookings (confirmed).\n   - Active locks (from Redis).\n4. Customer picks a slot; app calls `POST /api/bookings/lock` with slot details.\n5. Backend attempts to acquire a lock in Redis: `SET lock:business:{id}:staff:{staffId}:slot:{start} <userId> NX EX 300`.\n   - If successful, returns a `lockId` (UUID) and starts a 5-minute timer.\n   - If lock exists, returns conflict.\n6. Customer proceeds to payment/review. If payment is required, the lock is held during the payment process.\n7. On successful payment, app calls `POST /api/bookings` with `lockId`. Backend:\n   - Validates lock ownership.\n   - Creates booking in a transaction.\n   - Releases lock.\n   - Sends confirmation notifications.\n8. If user abandons or timer expires, the lock is automatically released by Redis TTL. A cleanup job can also handle edge cases.\n\n### 6.3 Search & Discovery\n- **Full-text search**: PostgreSQL `tsvector` on business name, description, category names. Query: `plainto_tsquery('english', :query)`.\n- **Geospatial**: `ST_DWithin` with user coordinates and radius filter. Indexed with GiST.\n- **Autocomplete**: `SELECT name FROM businesses WHERE name ILIKE :prefix% LIMIT 5`.\n- **Caching**: Search results for common queries cached in Redis for 5 minutes.\n- **Map view**: App sends bounding box; backend queries businesses within the box using `ST_MakeEnvelope`.\n\n## 7. Database Schema (High-Level Entities)\n\n```prisma\n// Prisma schema (simplified)\nmodel User {\n  id        String   @id @default(uuid())\n  supabaseId String @unique\n  email     String   @unique\n  role      Role     // CUSTOMER, PROVIDER, ADMIN\n  profile   UserProfile?\n  // ...\n}\n\nmodel Business {\n  id          String   @id @default(uuid())\n  ownerId     String\n  owner       User     @relation(fields: [ownerId], references: [id])\n  name        String\n  description String?\n  address     String\n  location    Unsupported(\"geometry(Point, 4326)\")?    // [{start, end}]\n}\n```\n\n## 8. API Design\n\nRESTful API with versioning (`/api/v1/`). Key endpoints:\n\n- **Auth**: `POST /auth/role` (set role after signup)\n- **Users**: `GET /users/me`, `PATCH /users/me`, `DELETE /users/me`\n- **Businesses**: `GET /businesses`, `GET /businesses/:id`, `GET /businesses/:id/slots`\n- **Bookings**: `POST /bookings/lock`, `POST /bookings`, `GET /bookings`, `PATCH /bookings/:id/cancel`, `PATCH /bookings/:id/reschedule`\n- **Search**: `GET /search?q=&lat=&lng=&radius=&category=&...`\n- **Categories**: `GET /categories`\n- **Favorites**: `POST /favorites`, `DELETE /favorites/:businessId`, `GET /favorites`\n- **Admin**: `GET /admin/businesses/pending`, `PATCH /admin/businesses/:id/approve`\n\nAll endpoints are protected by JWT guard except public ones (business listing, search). Role-based guards restrict provider/admin actions.\n\n## 9. Caching Strategy\n\n- **Business details**: Cache `GET /businesses/:id` with TTL 10 minutes. Invalidate on update.\n- **Categories**: Cache full tree with TTL 1 hour; invalidate on admin changes.\n- **Search results**: Cache per query hash for 5 minutes.\n- **Slot availability**: Not cached; computed live due to real-time nature.\n- **User sessions**: Redis for blacklisting revoked tokens (short TTL matching token expiry).\n\n## 10. Real-time Slot Locking Mechanism\n\n- **Lock Key**: `lock:business:{bizId}:staff:{staffId}:slot:{isoStart}`\n- **Value**: `userId:lockId`\n- **TTL**: 300 seconds (5 minutes).\n- **Acquisition**: `SET NX` ensures atomicity.\n- **Release**: On successful booking, `DEL` the key. On expiry, Redis auto-deletes.\n- **Cleanup**: A scheduled job every minute checks for locks older than 5 minutes (fallback).\n- **Concurrency**: Handled by Redis single-threaded nature; no race conditions.\n\n## 11. Deployment & CI/CD\n\n- **Mobile App**:\n  - EAS Build for development, preview, and production builds.\n  - GitHub Actions triggers EAS builds on push to `main` or `release/*`.\n  - Over-the-air updates via EAS Update.\n- **API & Admin**:\n  - Docker Compose for local development (PostgreSQL, Redis, API, Admin).\n  - Production deployment on a cloud VM or Kubernetes (not specified).\n  - GitHub Actions builds Docker images, runs tests, and deploys.\n- **Database**: Managed PostgreSQL with PostGIS (e.g., Supabase, AWS RDS).\n- **Redis**: Managed instance (ElastiCache, Upstash).\n- **Supabase**: Used as a service; migrations managed via Supabase CLI.\n\n## 12. Testing Strategy\n\n- **Unit Tests**: Jest for utility functions, hooks, NestJS services.\n- **Integration Tests**: Supertest for API endpoints with a test database.\n- **E2E Tests**: Detox or Maestro for critical mobile flows (booking, auth).\n- **Admin Dashboard**: React Testing Library + Cypress for critical admin workflows.\n- **CI**: All tests run on pull requests; coverage thresholds enforced.\n\n## 13. Security Considerations\n\n- **Authentication**: Supabase handles secure token issuance; NestJS validates JWTs.\n- **Authorization**: Role-based guards; resource ownership checks (e.g., user can only cancel own bookings).\n- **Data Validation**: Class-validator DTOs in NestJS; input sanitization.\n- **Rate Limiting**: Redis-backed rate limiter on sensitive endpoints (login, booking).\n- **HTTPS**: Enforced everywhere.\n- **Secrets**: Stored in GitHub Secrets, injected via environment variables.\n- **GDPR**: Account deletion cascades to remove personal data; data export endpoint.\n\n## 14. Conclusion\nThis architecture leverages a modern, scalable stack with clear separation of concerns. The monorepo structure ensures code sharing and consistency, while the chosen technologies provide a robust foundation for real-time booking, location-based search, and a smooth mobile experience.
