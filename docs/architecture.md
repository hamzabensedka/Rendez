  "content": "# System Architecture for Planity Clone\n\n## Overview\nThis document outlines the architecture of the Planity Clone mobile application, covering the client, server, data layer, infrastructure, and development practices. The design follows clean architecture principles, clear separation of concerns, and leverages the specified technology stack to ensure scalability, maintainability, and developer productivity.\n\n---\n\n## Technology Stack\n- **Mobile Client**: Expo, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Server**: NestJS (Node.js), Prisma ORM, PostgreSQL with PostGIS extension, Redis, BullMQ\n- **Infrastructure**: Docker Compose, Nx monorepo, pnpm, GitHub Actions, EAS Build, Supabase (optional auth/storage)\n- **Testing**: Jest, React Native Testing Library, Supertest\n- **Design System**: Shared UI library with Reanimated animations\n\n---\n\n## High-Level Components\n\n1. **Mobile Client** (`apps/mobile`)\n   - Expo managed workflow\n   - Navigation via Expo Router (file‑based)\n   - State management & data fetching with TanStack React Query\n   - Animations & gestures with React Native Reanimated\n   - UI components sourced from a shared design system library\n\n2. **API Gateway** (`apps/api`)\n   - NestJS modular monolith (can be split into microservices later)\n   - REST endpoints consumed by the mobile client\n   - WebSocket gateway (Socket.io) for real‑time slot updates\n   - Authentication (JWT + refresh token rotation)\n   - Validation, error handling, logging middleware\n   - Integration with Prisma for DB access\n   - BullMQ modules for background job processing\n\n3. **Data Layer**\n   - PostgreSQL (managed or self‑hosted) with PostGIS for geo‑queries\n   - Prisma ORM generates Type‑safe client\n   - Redis for caching, rate‑limiting, session store, and BullMQ job queue\n\n4. **Background Jobs**\n   - BullMQ queues (booking‑confirmation, reminder, review‑request, cleanup‑expired‑slots)\n   - Workers defined as NestJS processors\n\n5. **Infrastructure & DevOps**\n   - Docker Compose for local development (postgres, redis, api)\n   - Nx workspace to manage apps/libs, task caching, and affected commands\n   - pnpm workspaces for deterministic installs\n   - GitHub Actions CI: lint, unit/test, build, Docker image push, deploy to staging/prod\n   - EAS Build for Expo iOS/Android builds\n   - Optional Supabase for authentication (social login) and file storage (avatars, business images)\n\n---\n\n## Folder Structure (Nx Monorepo)\n\n```\nplanity-clone/\n├── apps/\n│   ├── mobile/          # Expo React Native app\n│   ├── api/             # NestJS server\n│   └── web-admin/       # (Optional) admin dashboard SPA\n├── infra/\n│   ├── feature-admin/\n│   │   ├── ui/\n│   │   ├── data/\n│   │   ├── domain/\n│   │   └── infra/\n│   ├── core/\n│   │   ├── auth/          # Auth guards, JWT strategies, refresh token Registration, login, social auth, JWT issuance/refresh | `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh` |\n| **User** | Profile CRUD, preferences | `GET /users/me`, `PATCH /users/me` |\n| **Business** | CRUD, search, geo‑search, categories | `GET /businesses`, `GET /businesses/:id`, `POST /businesses` (provider), `GET /businesses/search?lat=&lng=&radius=` |\n| **ServiceCategory** | Category management | `GET /categories` |\n| \n| **Availability** | Compute open slots based on business hours & existing bookings | `GET /businesses/:id/availability?date=YYYY-MM-DD` (real‑time updates via WS) |\n| **Booking** | Create, reschedule, cancel, payment initiation | `POST /bookings`, `PATCH /bookings/:id`, `DELETE /bookings/:id` |\n| **Payment** | Wrapper around payment gateway (Stripe/Paypal) | `POST /payments/webhook` (gateway callbacks) |\n| **Notification** | Send email/SMS/push, schedule reminders | Triggered by job workers; internal API for testing |\n| **Provider Portal** | Business owner specific endpoints (manage listings, hours, services) | `GET /provider/business`, `PATCH /provider/business/:id` |\n| **Admin** | Admin‑only user/business moderation | `GET /admin/users`, `DELETE /admin/businesses/:id` |\n| **Jobs** (BullMQ) | Async processing: booking confirmations, reminders, review requests, slot cleanup | Queues: `booking-confirmation`, `reminder`, `review-request`, `cleanup-slots` |\n\nAll services are implemented as NestJS modules inside `apps/api`. Communication between modules uses NestJS events or a lightweight CQRS pattern; cross‑service calls are avoided to keep boundaries clear.\n\n---\n\n## Data Model (Prisma)\n\n```prisma\nmodel User {\n **Redis** stores:\n  - Search result caches (keyed by query params, TTL 5 min)\n  - Rate‑limit counters (IP‑based)\n  - Session store for refresh tokens (optional)\n  - BullMQ job queue backend\n- **WebSocket** (NestJS Gateway) pushes slot availability updates to subscribed clients when a booking is created/cancelled.\n\n---\n\n## Background Job Processing\n\nUsing `@nestjs/bullmq`:\n\n```ts\n// booking-confirmation.queue.ts\nimport { Processor, WorkerHost } from '@nestjs/bullmq';\n@Processor('booking-confirmation')\nexport class BookingConfirmationProcessor extends WorkerHost {\n  async process(job): Promise<void> {\n    // send email/push, update booking status, etc.\n  }\n}\n```\nQueues are defined in `apps/api/src/queues`. Workers are auto‑registered on app startup.\n\n---\n\n## DevOps & CI/CD\n\n**Local Development**\n```bash\ndocker compose up -d   # postgres, redis\nnx run api:start:dev   # NestJS in watch mode\nnx run mobile:start    # Expo dev client\n```\n\n**CI (GitHub Actions)**\n1. Install pnpm dependencies (cached via nx)\n2. Lint (eslint) & type‑check (tsc)\n3. Run unit tests (jest) for libs & apps\n4. Build Docker image for API\n5. Push image to registry (on tag)\n6. Trigger EAS Build for mobile (separate workflow)\n\n**Production**\n- API runs as Docker containers behind a load balancer (ECS/EKS or Docker‑Swarm).\n- Database: managed PostgreSQL with PostGIS extension (AWS RDS, Supabase, or Crunchy Data).\n- Redis: Elasticache or Redis Labs.\n- CDN for static assets (Expo uploads, Supabase storage).\n- Monitoring: Prometheus + Grafana (optional), Sentry for error tracking.\n\n---\n\n## Security Considerations\n\n- Authentication: JWT access token (short‑lived, stored in memory) + refresh token (HTTP‑Only cookie, rotated).\n- Role‑based access control (RBAC) via NestJS guards (`@Roles`, `@UseGuards`).\n- Input validation using `class-validator` and custom pipes.\n- Rate limiting via `nestjs-throttler` backed by Redis.\n- Data protection: TLS everywhere, DB encryption at rest, sensitive fields hashed (passwords).\n- CORS restricted to allowed origins (expo dev, production domains).\n- Dependency scanning via `npm audit` in CI.\n\n---\n\n## Testing Strategy\n\n| Layer | Tool | Scope |\n|-------|------|-------|\n| Unit | Jest | Pure functions, utilities, React hooks, NestJS services |\n| Component | React Native Testing Library | UI components in isolation |\n| Integration | Supertest | API endpoints, API contract & DB interactions |\n| E2E (Mobile) | Detox (or Jest + React Native Testing Library with Expo) | Critical user flows (login, search, booking) |\n| Load | k6 (optional) | Simulate peak booking traffic |\n\nTest coverage threshold: 80% for libs, 70% for apps (enforced in CI).\n\n---\n\n## Design System & UI\n\n- Shared `libs/shared/ui` provides:\n  - Theme context (light/dark) using React Context values consumed via `useTheme`.\n  - Reusable primitives: Button, Input, Card, Avatar, Loader.\n  - Animation helpers using React Native Reanimated 2 (shared values, worklets).\n- Consistency enforced via lint rule (`eslint-plugin-react-native`) and storybook‑like visual regression (optional).\n\n---\n\n## Summary\n\nThe architecture divides the system into a React Native mobile client (Expo) and a NestJS API monolith, with clear module boundaries for each business domain, a PostgreSQL+PostGIS data layer accessed via Prisma, Redis for caching and job queuing, and an Nx/pnpm monorepo toolchain supporting CI/CD, testing, and scalable deployment.\n\n"
