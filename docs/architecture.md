  "content": "# Planity Clone — System Architecture\n\n## 1. Overview\n\nThis document describes the complete system architecture for the Planity Clone mobile-first marketplace. It defines the folder structure, service boundaries, technology choices, and key design decisions. The architecture prioritizes clean separation of concerns, maintainability, and scalability.\n\n## 2. Technology Stack\n\n### 2.1 Mobile App (Consumer)\n- **Framework**: Expo (managed workflow) with React Native\n- **Language**: TypeScript\n- **Routing**: Expo Router (file-based routing)\n- **State Management**: TanStack React Query (server state), Zustand (client state)\n- **Animations**: React Native Reanimated\n- **Maps**: react-native-maps (Google Maps on iOS/Android)\n- **Authentication**: Supabase Auth (OAuth, email/password, phone)\n- **Biometrics**: expo-local-authentication\n- **Secure Storage**: expo-secure-store\n- **Push Notifications**: Expo Notifications\n- **Build & Deploy**: EAS Build (iOS/Android), EAS Submit\n\n### 2.2 Web App (Business Owner Portal & Admin Dashboard)\n- **Framework**: Next.js (React) with TypeScript\n- **Routing**: Next.js App Router\n- **UI Library**: Tailwind CSS + shadcn/ui\n- **State Management**: TanStack React Query\n- **Authentication**: Supabase Auth (same backend)\n\n### 2.3 Backend\n- **Framework**: NestJS (Node.js)\n- **Language**: TypeScript\n- **ORM**: Prisma (with PostgreSQL)\n- **Geospatial**: PostGIS extension\n- **Caching**: Redis (session, rate limiting, search cache)\n- **Real-time**: WebSockets (NestJS Gateway) for booking updates\n- **Background Jobs**: Bull (Redis-backed queue)\n- **API Style**: RESTful (primary) + GraphQL (for complex queries)\n\n### 2.4 Infrastructure\n- **Database**: PostgreSQL (Supabase managed)\n- **File Storage**: Supabase Storage (images, documents)\n- **CI/CD**: GitHub Actions\n- **Containerization**: Docker Compose (local development)\n- **Hosting**: Vercel (Next.js web), EAS (mobile), Supabase (backend)\n\n### 2.5 Monorepo Management\n- **Tool**: Nx (with pnpm workspaces)\n- **Package Manager**: pnpm\n\n## 3. Service Boundaries & Modules\n\n### 4.1 Auth Module (NestJS)\n- **Responsibilities**: User registration, login, OAuth, JWT management, password reset, account deletion, rate limiting.\n- **Key Entities**: User, Session, RefreshToken\n- **Endpoints**: POST /auth/register, POST /auth/login, POST /auth/refresh, POST /auth/logout, POST /auth/password-reset, DELETE /auth/account\n- **Dependencies**: Supabase Auth (for OAuth), Redis (rate limiting, refresh token blacklist)\n\n### 4.2 Users Module\n- **Responsibilities**: Profile CRUD, preferences, favorites, search history, guest-to-user merge.\n- **Key Entities**: User, UserPreference, Favorite, SearchHistory\n- **Endpoints**: GET /users/me, PATCH /users/me, GET /users/me/favorites, POST /users/me/favorites, DELETE /users/me/favorites/:id\n\n### 4.3 Businesses Module\n- **Responsibilities**: Business CRUD, verification, categories, staff management, hours, images.\n- **Key Entities**: Business, BusinessCategory, Staff, BusinessHour, BusinessImage\n- **Endpoints**: GET /businesses, GET /businesses/:id, POST /businesses (owner), PATCH /businesses/:id, POST /businesses/:id/staff, etc.\n- **Geospatial**: PostGIS for location queries (ST_DWithin, ST_DistanceSphere)\n\n### 4.4 Services Module\n- **Responsibilities**: Service CRUD, pricing, duration, category association.\n- **Key Entities**: Service, ServiceCategory, ServicePricing\n- **Endpoints**: GET /businesses/:id/services, POST /businesses/:id/services, PATCH /services/:id\n\n### 4.5 Bookings Module\n- **Responsibilities**: Slot availability computation, booking creation, modification, cancellation, waitlist, check-in.\n- **Key Entities**: Booking, BookingSlot, BookingStatus, WaitlistEntry\n- **Endpoints**: GET /businesses/:id/slots?date=, POST /bookings, GET /bookings, PATCH /bookings/:id, DELETE /bookings/:id, POST /bookings/:id/check-in\n- **Real-time**: WebSocket for slot availability updates during booking flow.\n- **Background Jobs**: Bull queue for reminders, expiry handling.\n\n### 4.6 Reviews Module\n- **Responsibilities**: Review creation, moderation, rating aggregation.\n- **Key Entities**: Review, ReviewResponse, RatingAggregate\n- **Endpoints**: GET /businesses/:id/reviews, POST /businesses/:id/reviews, PATCH /reviews/:id, DELETE /reviews/:id\n- **Business Rules**: Review allowed only after completed booking; one review per booking.\n\n### 4.7 Search Module\n- **Responsibilities**: Full-text search, autocomplete, filtering, sorting, map clustering.\n- **Key Entities**: SearchIndex (materialized view or Elasticsearch)\n- **Endpoints**: GET /search?q=&lat=&lng=&radius=&category=&sort=\n- **Caching**: Redis for autocomplete suggestions, popular searches.\n- **Performance**: Use PostgreSQL full-text search (tsvector) with GIN index; consider Elasticsearch for scale.\n\n### 4.8 Notifications Module\n- **Responsibilities**: Push notifications, email, SMS.\n- **Key Entities**: Notification, NotificationTemplate\n- **Endpoints**: POST /notifications/send (admin), GET /users/me/notifications\n- **Integration**: Expo Push API, SendGrid (email), Twilio (SMS)\n\n### 4.9 Admin Module\n- **Responsibilities**: Platform oversight, user management, business approval, analytics.\n- **Endpoints**: GET /admin/dashboard, GET /admin/users, PATCH /admin/businesses/:id/verify, etc.\n\n## 5. Key Architectural Decisions\n\n### 6.1 Monorepo with Nx\n- **Why**: Shared types, utilities, and API client between mobile and web apps. Consistent build and test commands.\n- **Trade-off**: Increased complexity in dependency management, but Nx handles caching and dependency graphs.\n\n### 6.2 NestJS Modular Architecture\n- **Why**: Clear separation of concerns, dependency injection, built-in support for WebSockets and queues.\n- **Module per feature**: Each module has its own controllers, services, DTOs, and tests.\n\n### 6.3 TanStack React Query for Server State\n- **Why**: Automatic caching, background refetching, optimistic updates for booking flow.\n- **Caching Strategy**: Stale time of 30 seconds for search results, 5 minutes for business details, infinite for static data.\n\n### 6.4 PostGIS for Geospatial Queries\n- **Why**: Efficient location-based search and clustering. Native PostgreSQL support.\n- **Index**: GIST index on Business.location for fast ST_DWithin queries.\n\n### 6.5 Redis for Caching and Rate Limiting\n- **Use Cases**:\n  - Search autocomplete suggestions (TTL 1 hour)\n  - Popular businesses cache (TTL 5 minutes)\n  - Rate limiting counters (sliding window)\n  - Session store for refresh tokens\n  - Bull queue for background jobs\n\n### 6.6 Supabase for Auth and Storage\n- **Why**: Managed service reduces operational overhead. Supports OAuth providers out of the box. Storage for images with CDN.\n- **Integration**: Supabase Auth SDK on mobile, Supabase JS client on web. Backend uses Supabase Admin SDK for user management.\n\n### 6.7 Expo Router for Navigation\n- **Why**: File-based routing simplifies navigation structure. Supports deep linking and web compatibility.\n- **Structure**: (auth) group for login/register, (tabs) for main app, dynamic routes for business/[id].\n\n## 7. API Design Principles\n\n- **RESTful**: Resources map to entities (e.g., /businesses, /bookings).\n- **Pagination**: Cursor-based for lists (bookings, reviews) to avoid offset issues.\n- **Versioning**: URL prefix /v1/ for future compatibility.\n- **Error Handling**: Consistent error response format: { error: { code: string, message: string, details?: any } }.\n- **Validation**: DTOs with class-validator decorators in NestJS.\n\n## 8. Security Considerations\n\n- **Authentication**: JWT access tokens (15 min) + refresh tokens (7 days) stored in secure storage (Keychain/Keystore).\n- **Rate Limiting**: 5 failed login attempts -> 30 min lockout. Global rate limit per IP: 100 requests/min.\n- **Data Encryption**: HTTPS everywhere. Sensitive data (phone, email) encrypted at rest.\n- **Input Sanitization**: Prevent SQL injection via Prisma parameterized queries. XSS protection via React Native's default handling.\n- **Authorization**: Role-based access control (RBAC) with NestJS guards (Consumer, BusinessOwner, Admin).\n\n## 9. Performance Targets\n\n- **Search Results**: < 500ms for 95th percentile.\n- **Booking Creation**: < 1s end-to-end.\n- **App Launch**: < 2s cold start (with Hermes engine).\n- **Image Loading**: Lazy loading with progressive JPEGs, CDN caching.\n- **API Response**: < 200ms for 90th percentile (excluding search).\n\n## 10. Testing Strategy\n\n- **Unit Tests**: Jest for NestJS services, React Native components (React Native Testing Library).\n- **Integration Tests**: Supertest for API endpoints, Prisma test database.\n- **E2E Tests**: Detox for mobile (future), Playwright for web.\n- **CI Pipeline**: GitHub Actions runs lint, type check, unit tests, integration tests on every PR.\n\n## 11. Deployment\n\n- **Mobile**: EAS Build -> EAS Submit to App Store / Google Play. Staging via EAS Update (OTA).\n- **Web (Next.js)**: Vercel with automatic deployments from main branch.\n- **Backend (NestJS)**: Docker container deployed to a cloud provider (AWS ECS / Google Cloud Run).\n- **Database**: Supabase managed PostgreSQL with point-in-time recovery.\n\n## 12. Monitoring & Observability\n\n- **Logging**: Structured logging with Winston (NestJS), console.log for mobile (filtered in production).\n- **Error Tracking**: Sentry for both mobile and backend.\n- **Performance**: React Native Performance Monitor, Lighthouse for web.\n- **Analytics**: PostHog (self-hosted) for user events.\n\n## 13. Future Considerations\n\n- **Elasticsearch**: Replace PostgreSQL full-text search for better performance at scale.\n- **Microservices**: Split NestJS monolith into independent services (auth, bookings, search) if needed.\n- **Offline Support**: Use WatermelonDB for local-first data on mobile.\n- **Multi-language**: i18n with expo-localization and next-intl.\n\n---\n\n*This architecture document is a living artifact and should be updated as the system evolves.*",
