  "content": "# Planity Clone System Architecture\n\n## 1. Overview\nThis document describes the complete system architecture for the Planity Clone mobile application. The solution follows a clean‑architecture approach with clear separation of concerns, leveraging a monorepo managed by Nx, Expo for the mobile client, NestJS for the API server, and a robust data stack (PostgreSQL + PostGIS, Prisma, Redis). The architecture supports guest browsing, authenticated user flows, business owner portal, admin dashboard, and background job processing.\n\n## 2. Technology Stack\n- **Mobile**: Expo (SDK 49), React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **API**: NestJS (Node.js), TypeScript, Prisma ORM, PostgreSQL + PostGIS extension, Redis (caching & pub/sub), BullMQ (job queues)\n- **DevOps**: Docker Compose (local), GitHub Actions (CI), EAS Build (Expo builds), Supabase (optional auth/storage fallback for auth (optional), Nx (monorepo tooling), pnpm (package manager)\n- **Testing**: Jest (unit/integration), React Native Testing Library, Supertest (API)\n- **Design System**: Shared UI library built with React Native Reanimated & Expo Vector Icons\n\n## 3. High-Level Architecture\n```\n+-------------------+        +---------------------+        +-------------------+\n|   Mobile App      | <----> |   API Gateway (Nest) | <----> |   PostgreSQL/PostGIS|\n| (Expo/React)      |        |   - Auth Service    |        |   - Prisma ORM    |\n|   - Expo Router   |        |   - Business Service|        |   - Redis Cache   |\n|   - TanStack Query|        |   - Booking Service |        +-------------------+\n|   - Reanimated    |        |   - Notification Svc|            ^          ^\n+-------------------+        |   - Admin Service   |            |          |\n                              |   - Provider Portal |            |          |\n                              +---------------------+            |          |\n                                                                     |          |\n                                                            +--------+  +--------+\n                                                            | BullMQ |  | Supabase|\n                                                            +--------+  +--------+\n```\n- **Mobile App** consumes REST/GraphQL‑like endpoints via NestJS controllers. State is managed with TanStack React Query; UI animations use Reanimated.\n- **API Layer** is split into domain‑oriented NestJS modules (Auth, Business, Booking, Provider, Admin, Notification). Each module owns its DTOs, services, guards, and Prisma repositories.\n- **Data Layer**: PostgreSQL stores core entities; PostGIS enables geo‑queries for map‑based search. Redis caches frequently accessed read‑only data (business listings, categories) and supports pub/sub for real‑time notifications.\n- **Background Jobs**: BullMQ workers (running as separate NestJS processes) handle email/SMS notifications, payment webhook processing, and periodic slot recomputation.\n- **DevOps**: Docker Compose defines services for API, PostgreSQL, Redis, and BullMQ worker. GitHub Actions runs lint, test, build, and pushes Docker images to a registry. EAS Build creates iOS/Android binaries from the Expo workflow.\n\n## 4. Service Boundaries\n| Boundary | Responsibility | Technologies\n|----------|----------------|--------------|
| **Mobile Client** | UI, navigation, local state, offline caching, push token handling | Expo, React Native, TanStack Query, Reanimated, Expo Router |
| **API Gateway** | HTTP routing, authentication, validation, request/response transformation | NestJS, Class Validator, Passport (JWT, OAuth2) |
| **Auth Service** | User registration, login, social OAuth, JWT issuance, refresh token flow | NestJS, Prisma User entity, Redis for refresh token store |
| **Business Service** | CRUD for businesses, categories, geo‑search, service catalog | NestJS, Prisma, PostGIS (ST_DWithin), Redis cache for listings |
| **Booking Service** | Slot computation, availability checks, booking creation, rescheduling, cancellation | NestJS, Prisma, custom slot algorithm (business hours + existing bookings), BullMQ for confirmation emails |
| **Provider Portal** | Business owner CRUD for services, staff, schedule, viewing bookings | NestJS, same entities as Business/Booking, role‑based guard (provider) |
| **Admin Service** | Global settings, user moderation, analytics, system configuration | NestJS, Prisma, role‑based guard (admin) |
| **Notification Service** | Template rendering, dispatch via email/SMS/push, handling inbound webhooks | NestJS, BullMQ workers, Redis pub/sub for real‑time push to mobile via Expo push service |
| **Background Workers** | Notification dispatch, payment webhook processing, nightly slot recomputation, cleanup jobs | BullMQ (Node workers), optionally separate NestJS microservice or same API with queue processors |
| **Data Layer** | Persistent storage, geo‑indexing, caching | PostgreSQL + PostGIS, Prisma ORM, Redis (LRU cache, pub/sub) |
| **Shared Types** | DTOs, interfaces, enums used across mobile & api to ensure contract safety | TypeScript lib, generated via Nx library, consumed by both apps\n\n## 6. Data Layer Details\n- **Entities** (Prisma schema): `User`, `Business`, `Category`, `Service`, `BusinessHours`, `Slot`, `Booking`, `Review`, `Payment`, `NotificationToken`, `RefreshToken`.\n- **Geo Index**: `Business.location` is a `Point` with PostGIS GIST index; enables radius queries (`ST_DWithin`).\n- **Caching Strategy**:\n  - *Listing Cache*: Redis key `business:list:{lat},{lng},{radius},{category}` TTL 5 min.\n  - *Detail Cache*: `business:detail:{id}` TTL 10 min.\n  - *Slot Cache*: `slot:{serviceId}:{date}` TTL 1 min (computed on‑demand).\n- **Cache Invalidation**: After any mutation (create/update/delete) on a business or service, related keys are deleted via Prisma `afterSave`/`afterDelete` hooks or via BullMQ job that publishes a Redis `INVALIDATE` channel.\n- **Transactions**: Booking creation uses Prisma transaction to lock business hours, check availability, insert booking, and update slot cache.\n\n## 7. API Design\n- **RESTful endpoints** with JSON bodies; versioned under `/api/v1`.\n- **Authentication**: JWT in Authorization header; refresh token rotated via `/auth/refresh`.\n- **Error Handling**: Centralized exception filter returns `{ error: string, message?: string, code?: number }`.\n- **Pagination**: Cursor‑based (`limit`, `cursor`) for lists; total count optional.\n- **Filtering & Sorting**: Query params (`category`, `minRating`, `maxDistance`, `sortBy`).\n- **Webhooks**: Payment provider (Stripe) posts to `/payments/webhook`; verified via signature, then enqueues a BullMQ job to update booking status.\n- **OpenAPI**: Auto‑generated via `@nestjs/swagger`; available at `/api/docs`.\n\n## 8. State Management & Client‑Side\n- **TanStack React Query** handles server state: queries for businesses, slots, bookings; mutations for auth, booking, profile updates.\n- **Query Keys**: Structured as arrays, e.g., `['businesses', {lat, lng, radius, category}]`.\n- **Mutation Optimistic Updates**: For booking creation, UI optimistically adds booking; on error, rollback via query invalidation.\n- **Reanimated** used for gesture‑based bottom sheet (filter) and animated transitions between tabs.\n- **Auth Context**: Lightweight React Context exposing `user`, `login`, `logout`, `refreshToken`; persisted via `expo-secure-store`.\n- **Offline**: Query results cached; read‑only screens (business list, detail) show stale data while refetching in background.\n\n## 9. Caching & Background Jobs\n- **Redis** used as:\n  1. LRU cache for API responses (via wrapper).\n  2. Pub/sub channel `cache:invalidate` for real‑time invalidation across API instances.\n  3. Store for refresh tokens (hashed).\n  4. Temporary storage for OTP during social login flow.\n- **BullMQ** queues:\n  - `email-notifications` (send welcome, booking confirmation, reminder).\n  - `sms-notifications` (OTP, booking reminders).\n  - `payment-processing` (handle webhook, update booking status).\n  - `slot-recalc` (nightly recompute slots for all services).\n  Workers are defined in `apps/api/src/queues/`; each queue has a `processor.ts` implementing the job logic.\n\n## 10. DevOps & CI/CD\n- **Local Development**: `docker compose up` starts API, PostgreSQL, Redis, and a BullMQ worker. Expo dev server runs via `nx run mobile:start`.\n- **GitHub Actions Workflow**:\n  1. `lint` – runs ESLint & Prettier.\n  2. `test` – runs Jest for both mobile and API (coverage threshold 80%).\n  3. `build` – builds Docker images (`api`, `worker`) and pushes to GitHub Packages.\n  4. `eas-build` – triggers EAS build for iOS/Android using the latest Expo commit.\n  5. `deploy` – (optional) runs `docker compose pull && docker compose up -d` on staging server.\n- **Environment Variables**: Managed via `.env` files; separate for `development`, `staging`, `production`. Secrets (DB URL, Redis password, JWT secret, Stripe keys) injected via GitHub Secrets.\n- **Monitoring**: Health endpoint `/health` returns DB, Redis, and queue status. Logs shipped to stdout; in production captured by a logging service (e.g., Loki).\n\n## 11. Testing Strategy\n- **Unit Tests**: Jest + `@testing-library/react-native` for components; NestJS testing utilities for services/controllers.\n- **Integration Tests**:\n  - API: Supertest against a temporary PostgreSQL (via Docker) and Redis.\n  - Mobile: Detox for critical flows (login → search → book → confirmation).\n- **Contract Tests**: Shared types library ensures DTOs match between client and server; CI fails if mismatch.\n- **Test Coverage**: Aim for ≥80% statements, branches, and functions.\n\n## 12. Security Considerations\n- **Authentication**: JWT with short‑lived access token (15 min) and refresh token stored HTTP‑only cookie equivalent via secure store; refresh token rotated on use.\n- **Authorization**: NestJS guards (`AuthGuard`, `RolesGuard`) enforce scopes (`user`, `provider`, `admin`).\n- **Data Validation**: DTOs validated with class‑validator & class‑transformer; sanitization to prevent injection.\n- **Geo‑Safety**: Radius queries limited to max 50 km to prevent abusive scans.\n- **Rate Limiting**: NestJS `ThrottlerGuard` per‑IP and per‑user.\n- **Secrets**: Never committed; accessed via `process.env`.\n- **Transport**: All API calls over HTTPS; Expo enforces TLS.\n- **Payment**: PCI‑DSS compliance delegated to Stripe; webhook signature verification.\n- **Audit Logs**: Admin actions logged to `AuditLog` entity.\n\n## 13.
