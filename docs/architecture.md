Overview\n\nThis document describes the complete system architecture for the Planity Clone mobile application, a marketplace for beauty and wellness appointment booking. Service Boundaries\n\n### 4.1 API Gateway (NestJS)\n\nHandles all client requests, authentication, and routing to internal services. Exposes RESTful endpoints with GraphQL for complex queries.\n\n**Modules:**\n- **Auth Module**: JWT issuance, refresh token rotation, OAuth callbacks, password reset\n- **Users Module**: CRUD for consumers and providers, profile management\n- **Businesses Module**: Business registration, hours, images, staff management\n- **Services Module**: Service catalog, categories, pricing\n- **Bookings Module**: Slot computation, booking creation, cancellation, waitlist\n- **Search Module**: Full-text search, geospatial queries, auto-complete\n- **Reviews Module**: Review creation, moderation, rating aggregation\n- **Notifications Module**: Push notifications, email (SendGrid), SMS (Twilio)\n\n### 4.2 Background Worker (NestJS + Bull)\n\nProcesses asynchronous tasks:\n- Appointment reminders (24h, 2h before)\n- No-show auto-marking\n- Email/SMS dispatch\n- Data aggregation for analytics\n- Cache warming\n\n### 4.3 Mobile App (Expo)\n\nThin client with offline-first capabilities. Uses TanStack React Query for server state management and caching.\n\n**Key Libraries:**\n- `expo-router` for navigation\n- `@tanstack/react-query` for data fetching\n- `react-native-reanimated` for animations\n- `react-native-maps` or `mapbox-gl` for map\n- `expo-notifications` for push\n- `expo-auth-session` for OAuth\n\n## 5. // PostGIS point\n  address     String\n  phone       String?\n  email       String?\n  hours       BusinessHour[]\n  services    Service[]\n  staff       Staff[]\n  reviews     Review[]\n  images      Image[]\n  ownerId     String\n  owner       User     @relation(fields: [ownerId], references: [id])\n  createdAt   DateTime @default(now())\n  updatedAt   DateTime @updatedAt\n}\n\nmodel Service {\n  id          String   @id @default(cuid())\n  name        String\n  description String?\n  price       Decimal  @db.Decimal(10, 2)\n  duration    Int      // minutes\n  category    Category\n  business    Business @relation(fields: [businessId], references: [id])\n  businessId  String\n  staff       Staff[]\n  bookings    Booking[]\n}\n\nmodel Booking {\n  id          String   @id @default(cuid())\n  user        User     @relation(fields: [userId], references: [id])\n  userId      String\n  service     Service  @relation(fields: [serviceId], references: [id])\n  serviceId   String\n  staff       Staff? API Design\n\n### 6.1 Authentication\n\n- `POST /auth/register` — Create account\n- `POST /auth/login` — Get JWT + refresh token\n- `POST /auth/refresh` — Rotate refresh token\n- `POST /auth/logout` — Invalidate session\n- `POST /auth/forgot-password` — Send reset email\n- `POST /auth/reset-password` — Set new password\n\n### 6.2 Businesses\n\n- `GET /businesses` — Search with filters (query params: q, lat, lng, radius, category, etc.)\n- `GET /businesses/:id` — Full detail with services, staff, reviews\n- `POST /businesses` — Create business (provider only)\n- `PUT /businesses/:id` — Update business\n- `GET /businesses/:id/availability` — Slots for next 7 days\n\n### 6.3 Bookings\n\n- `GET /bookings` — User's bookings (consumer or provider)\n- `POST /bookings` — Create booking\n- `PUT /bookings/:id/reschedule` — Change time\n- `PUT /bookings/:id/cancel` — Cancel\n- `POST /bookings/:id/check-in` — QR code check-in\n\n### 6.4 Search\n\n- `GET /search/autocomplete` — Returns business names, services, locations\n- `GET /search/nearby` — Geospatial search with map bounds\n\n## 7. Data Flow\n\n### 7.1 Booking Flow\n\n1. User selects service and staff (optional)\n2. Mobile app calls `GET /businesses/:id/availability?serviceId=X&staffId=Y`\n3. Backend computes available slots using business hours, existing bookings, and staff schedule\n4. User selects a slot\n5. Mobile app calls `POST /bookings` with service, staff, startTime\n6. Backend uses pessimistic locking (Redis lock on slot) to prevent double-booking\n7. On success, booking is created with status `PENDING`\n8. Background worker sends confirmation email/SMS\n9. Payment is processed (if required) via Stripe integration\n\n### 7.2 Search Flow\n\n1. User types query or moves map\n2. Mobile app debounces and calls `GET /businesses` with query params\n3. API Gateway routes to Search Module\n4. Search Module queries PostgreSQL with full-text search and PostGIS distance calculation\n5. Results are cached in Redis (TTL: 5 minutes)\n6. Response includes business cards with next available slot (computed on the fly)\n\n## 8. Caching Strategy\n\n| Data | Cache Type | TTL | Invalidation |\n|------|------------|-----|--------------|\n| Business list (search) | Redis | 5 min | On business update |\n| Business detail | Redis | 10 min | On business/service update |\n| User session | Redis | 30 days | On logout |\n| Availability slots | Redis | 1 min | On booking creation |\n| Category tree | In-memory (Redis) | 1 hour | On admin update |\n| Static assets (images) | CDN (Cloudinary) | 30 days | On upload |\n\n## 9. Security Considerations\n\n- **JWT with refresh token rotation**: Access token expires in 15 minutes, refresh token in 30 days\n- **Rate limiting**: 100 requests/min per IP, 10 booking creations/min per user\n- **Input validation**: Zod schemas on all endpoints\n- **SQL injection**: Prevented by Prisma parameterized queries\n- **XSS**: Output encoding, Content Security Policy headers\n- **CORS**: Whitelist only mobile app origins\n- **Data encryption**: Sensitive fields encrypted at rest (AES-256)\n- **Audit logging**: All booking modifications logged\n\n## 10. Testing Strategy\n\n| Layer | Tool | Scope |\n|-------|------|-------|\n| Unit (mobile) | Jest + RNTL | Components, hooks, utils |\n| Unit (backend) | Jest | Services, controllers, guards |\n| Integration | Supertest + Testcontainers | API endpoints with real DB |\n| E2E | Detox (mobile) | Critical user flows |\n| Load | k6 | Booking and search endpoints |\n\n## 11. Deployment\n\n### 11.1 Mobile\n\n- **Development**: EAS Build with Expo Go\n- **Staging**: EAS Build + internal distribution\n- **Production**: EAS Build → App Store / Google Play\n\n### 11.2 Backend\n\n- **Infrastructure**: Docker Compose → AWS ECS / Google Cloud Run\n- **Database**: Managed PostgreSQL (RDS / Cloud SQL)\n- **Redis**: Managed Redis (ElastiCache / Memorystore)\n- **CI/CD**: GitHub Actions → build Docker images → deploy to staging → manual approval for production\n\n## 12. Monitoring & Observability\n\n- **Logging**: Winston (backend), Sentry (mobile + backend)\n- **Metrics**: Prometheus + Grafana (request latency, error rates, booking success rate)\n- **Alerting**: PagerDuty for critical errors\n- **Analytics**: Mixpanel / Amplitude for user behavior\n\n## 13. Performance Targets\n\n| Metric | Target |\n|--------|--------|\n| API response time (p95) | <200ms |\n| Search results | <500ms |\n| Booking creation | <1s |\n| Mobile app cold start | <3s |\n| Map load | <3s on 4G |\n| Concurrent users | 10,000 |\n\n## 14.
