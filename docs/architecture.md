Overview\nPlanity Clone is a mobile-first appointment booking platform built with a monorepo (Nx) that shares code between Expo (React Native) mobile apps and a NestJS backend. The system follows clean architecture principles: separation of concerns, domainвЂ‘driven boundaries, and dependency inversion. Core concerns are split into independent services (Auth, Business, Booking, Payment, Notification, Admin) that communicate via REST/GraphQL APIs and eventвЂ‘driven messages (Redis/BullMQ).\n\n## 2. Technology Stack\n- **Mobile**: Expo SDK, React Native, TypeScript, Expo Router, TanStack React Query, React Native Reanimated\n- **Web/Admin**: Same React Native codebase (Expo Web) or separate Next.js app (out of scope)\n- **Backend**: NestJS (Node.js), TypeORM replaced by Prisma ORM, PostgreSQL + PostGIS for geoвЂ‘queries, Redis for caching & pub/sub, BullMQ for background jobs\n- **DevOps**: Docker Compose (local), GitHub Actions (CI/CD), EAS Build (Expo), Supabase (optional auth/storage fallback), Nx (monorepo tooling), pnpm (package manager)\n- **Testing**: Jest (unit/integration), React Native Testing Library, Supertest (API)\n\n## 3. HighвЂ‘Level Architecture\n```\n+-------------------+      +-------------------+      +-------------------+\n|   Expo Mobile     |<---->|   API Gateway     |<---->|   NestJS Services  |\n| (Expo Router)    |      | (NestJS Gateway) |      |  - Auth Service   |\n+-------------------+      +-------------------+      |  - Business Svc   |\n        ^                         ^                |  - Booking Svc    |\n        |                         |                |  - Payment Svc    |\n        |                         |                |  - Notification   |\n        |                         |                |  - Admin Svc      |\n        |                         |                +-------------------+\n        |                         |                         |\n        |                         |                         v\n        |                 +-------------------+      +-------------------+\n        |                 |   Redis (Cache)   |      |   PostgreSQL/PostGIS|\n        |                 +-------------------+      +-------------------+\n        |                         ^                         |\n        |                         |                         |\n        +-------------------------+-------------------------+\n                                 |                     \n                                 v                     \n                        +-------------------+\n                        |   BullMQ Workers  |\n                        +-------------------+\n```\n\n* **API Gateway** вЂ“ Thin NestJS module that routes to domain services, handles global middleware (authentication, validation, rateвЂ‘limiting).\n* **Domain Services** вЂ“ Each service owns its data (via Prisma), business logic, and exposes a REST/GraphQL faГ§ade. Services communicate asynchronously via Redis pub/sub for events (e.g., `booking.created`).\n* **Redis** вЂ“ Caching (LRU) for frequent reads (business listings, slot computation), session store, and pub/sub for interвЂ‘service events.\n* **PostgreSQL + PostGIS** вЂ“ Persistent storage; PostGIS enables geoвЂ‘search for mapвЂ‘based discovery.\n* **BullMQ** вЂ“ Background workers for email/SMS notifications, payment webhook processing, slot recomputation, and cleanup jobs.\n\n## 4. API Contracts\n- **REST** (JSON) for simplicity; each service exposes `/api/<resource>` endpoints.\n- **OpenAPI 3.0** generated via NestJS `@nestjs/swagger`; shared via `libs/api` for mobile consumption.\n- **GraphQL** optional fallback (Apollo Server) if realвЂ‘time queries needed; can be enabled per service.\n- **Versioning**: URL prefix `/v1/`.\n- **Authentication**: Bearer JWT verified in Gateway middleware; scopes per role.\n- **Error Format**: `{ error: { code: string, message: string, details?: any } }`.\n\n## 8.
