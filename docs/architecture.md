API Design Principles\n- **ResourceвЂ‘oriented REST** (`/businesses`, `/bookings`, `/users`, `/reviews`).\n- **Versioning** via URL prefix `/api/v1`.\n- **DTOs** for request validation and response shaping.\n- **Pagination** using cursorвЂ‘based approach for large lists.\n- **GeoвЂ‘search**: `/businesses?near=[lat,lng]&distance=5km` leveraging PostGIS `<->` operator.\n- **Error handling**: centralized exception filter returning `{ error, message, statusCode }`.\n- **Authentication middleware** attaches `req.user` (JWT payload).\n- **Rate limiting** perвЂ‘IP via Redis.\n\n## 8. State Management & Data Flow (Mobile)\n1. **React Query** hooks (`useBusinesses`, `useBookingDetails`) encapsulate API calls, caching, background refetch, and optimistic updates.\n2. **Query keys** are structured: [`businesses`, filters, page], [`booking`, id].\n3. **Mutations** for create/update/delete use `mutationFn` with optimistic UI updates and rollback on error.\n4. **Auth state** persisted in SecureStore; React Query automatically attaches Authorization header via axios instance.\n5. **Reanimated** used for gestureвЂ‘driven UI (e.g., swipeвЂ‘toвЂ‘cancel booking, animated map markers).\n6. **Shared types** (`libs/types`) generated from Prisma schema via `prisma generate` ensuring typeвЂ‘safety across client and server.\n\n## 9. Authentication & Authorization\n- **Register/Login**: endpoint `/auth/register` & `/auth/login` returns `accessToken` (shortвЂ‘lived JWT) and `refreshToken` (httpOnly cookie or SecureStore).\n- **Refresh**: `/auth/refresh` issues new access token.\n- **Roles**: `CUSTOMER`, `PROVIDER`, `ADMIN`. Guard decorators (`@Roles`, `@Auth`) protect routes.\n- **OAuth**: `passport-google-oauth20`, `passport-apple` strategies; redirect to `/auth/google/callback`.\n- **Permissions**: Provider can only modify own business; Admin can manage all.\n\n## 10. Booking Flow & Slot Computation\n1. **Availability Service** (NestJS) computes free slots:\n   - Retrieve business hours (opening/closing per weekday).\n   - Fetch existing bookings for the target date.\n   - Subtract booked intervals, respecting service duration and buffer time.\n   - Return array of `{ start, end }` slots.\n2. **Mobile**: `useAvailableSlots` query calls `/businesses/:id/availability?date=YYYY-MM-DD&serviceId=xxx`.\n3. **Booking Creation**: optimistic UI adds pending booking; on success, server creates booking, triggers `booking-confirmed` BullMQ job (sends email/SMS, updates Redis cache).\n4. **Rescheduling/Cancel**: similar mutation flow; slot recomputation triggered on change.\n\n## 11. Payments\n- Integration with Stripe (or PayPal) via webhook.\n- Endpoint `/payments/create-intent` returns client secret.\n- Webhook `/payments/webhook` verifies signature, updates `booking.paymentId`, emits `payment-succeeded` job for confirmation.\n- PCIвЂ‘SAQвЂ‘A compliance: no raw card data touches our servers.\n\n## 12. Notifications\n- **Channels**: Email (SendGrid), SMS (Twilio), Push (Expo Notifications).\n- **Jobs**: `send-booking-confirmation`, `send-reminder-1h`, `send-review-request`.\n- **Scheduler**: BullMQ repeatable jobs for daily reminders; also triggered on booking CRUD.\n\n## 13. Admin & Provider Portals\n- Built as separate ReactвЂ‘web apps (could be under `apps/web`) sharing `libs/ui` and `libs/types`.\n- Authenticated via same JWT; roleвЂ‘based routing.\n- Provider dashboard: manage business info, services, hours, view bookings, respond to reviews.\n- Admin dashboard: CRUD over users, businesses, system settings, view analytics.\n\n## 14. Background Jobs (BullMQ)\n- Queues: `notifications`, `slot-recalc`, `payment-webhook`, `email-verification`.\n- Workers: separate Node processes (`npm run start:worker`) scaling horizontally.\n- Retry policy: exponential backoff, max 3 attempts.\n- Monitoring: BullMQ UI (`@bull-board/express`) mounted at `/admin/queues`.\n\n## 15.
